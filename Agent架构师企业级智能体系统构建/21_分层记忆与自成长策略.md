## 一、课程导入与记忆分层概览

### 1.1 课程导入：从 Workspace 到 Session 再到分层记忆

#### 🧠 直观理解

如果把构建 AI Agent 比作装修一间房子，那么第 19 课解决的是「东西放哪」的问题（Workspace 作为储物间），第 20 课解决的是「对话怎么记」的问题（Session 作为录音笔），而本节要解决的是「记录多了怎么整理」的问题——当录音笔里的内容堆积如山，你不可能每次都从头听起，你需要一套分层整理的方法。

#### 📖 详细解释

本节是「AI 应用开发课程」第 21 课的开篇。回顾最近三节课的演进脉络，每一步都自然地引出了下一步必须面对的问题：

| 课时 | 核心主题 | 解决的问题 | 引出的新问题 |
|---|---|---|---|
| 第 19 课 | **Workspace 工作空间** | 智能体运行中产生大量文档和中间产物，需要统一存放和检索 | 有一个最需要存放的东西——会话（Session）——该怎么管？ |
| 第 20 课 | **Session 会话管理** | 把会话信息记录下来，区分完整记录（full_context）和模型可见窗口（context_window） | 对话越来越长，简单裁剪远远不够，这些信息该如何分层整理？ |
| 第 21 课 | **分层记忆与自成长策略** | 对长期积累的信息按使用周期、影响范围、使用场景进行分层管理 | —— |

上节课我们建立了 Session 管理的核心概念：`full_context` 存储完整的对话历史——不裁剪、不丢弃；而 `context_window` 则是模型在本次请求中实际能够"看到"的那部分内容。二者分工明确：full_context 是档案馆，context_window 是案头的当前卷宗。

但这里有一个关键细节经常被忽视：对于处理复杂任务的智能体来说，需要记录下来的远不止「用户原始输入 → 智能体最终输出」这两端。真正有价值的记录是**整个处理过程**——中间推理、工具调用、子任务分解、临时决策——这些过程信息的量级远远超过表面的问答对。一个深度任务的单次处理过程，可能产生几十甚至上百条中间记录。

这就自然引出了本节课的核心追问：**一个长期运行的智能体，面对不断膨胀的会话历史和过程记录，到底应该如何管理这些信息？**

> 🔗 **前置知识**：本节的讨论建立在第 20 课 Session 管理的 full_context / context_window / memo 三个核心概念之上。

### 1.2 为什么需要记忆分层：Context Window 的隐喻

#### 🧠 直观理解

想象你站在一间屋里，窗外是一片广阔的天地——山川、河流、城市、行人，万物尽在。但此刻你只能透过窗户看到其中的一角。这一角虽然只是天地的一小部分，但已经足够让你判断天气、观察近处的动静、做出当下的决策。对于模型来说，`context_window` 就是这扇窗户——窗外是完整的对话历史（full_context），窗内是模型这次请求中能"看到"的那一角。

#### 📖 详细解释

老师课上用一幅图生动地解释了这个概念。下面是这幅图的 ASCII 还原：

```
┌─────────────────────────────────────────────────────────────────────┐
│                   窗外广阔的天地（Full Context）                       │
│                   完整的对话历史 · 全部中间过程 · 历次工具调用             │
│                                                                     │
│   [T1] user: 帮我规划上海三日亲子游                                     │
│   [T2] assistant: 先确认住宿区域和必去景点...                            │
│   [T3] user: 迪士尼一定要去                                            │
│   [T4] assistant: 我先按住市中心估算路线...                              │
│   [T5] assistant: 正在查询迪士尼周边酒店信息...  ← 工具调用              │
│   [T6] tool: 返回酒店列表 [{...}, {...}]                               │
│   [T7] user: 孩子走不了太多路，下午最好午休                                │
│   [T8] assistant: 路线检查：迪士尼和自然博物馆排在同一天，通勤过长           │
│   [T9] user: 重新安排，远距离项目不要和市区景点挤一天                        │
│                            ┌────────────────────────────────┐       │
│   [T10] assistant: 调整后路线...│   Context Window（窗口一角）    │       │
│                            │   模型本次请求"透过窗户看到的"     │       │
│   [T11] assistant: 最终确认...│                                │       │
│                            │   [T7] 孩子走不了太多路...       │       │
│   [T12] user: 住宿定好了     │   [T8] 路线检查...              │       │
│                            │   [T9] 重新安排...              │       │
│   [T13] assistant: 好的...  │   [T10] 调整后路线...           │       │
│                            │   [T11] 最终确认...             │       │
│   [T14] user: 天气怎样？     │                                │       │
│                            └────────────────────────────────┘       │
│                                                                     │
│   注：T1-T6 虽在 full_context 中保留，但本次请求不注入窗口              │
│   注：T12-T14 是新发生的对话，尚未超出窗口容量                           │
└─────────────────────────────────────────────────────────────────────┘
```

这张图的关键洞察在于：**context_window 只要能覆盖模型当前处理所需的信息就足够了**。我们不需要（也不可能）让模型每次请求都"看到"完整的对话历史。窗口的设计目标是「够用」而非「完整」。

然而，随着对话不断推进，两个尖锐的问题很快浮出水面：

**问题一：简单裁剪不够用**

最朴素的做法是「滑动窗口」——始终保留最近 N 条消息，旧的消息直接丢弃。这是上节课 `keep_recent` 策略的思路。但很快你会发现：

- 用户在对话开头提到的偏好（如"孩子走不了太多路"），到了第 50 轮对话时早已被滑出窗口。模型"忘记"了这个关键约束，后续的规划就失去了准星。
- 跨多轮协商确定的决策依据（如"住宿选在浦东，因为离迪士尼近"），如果被裁剪掉，后续所有讨论的基础都会动摇。

**问题二：简单总结也不够用**

另一个思路是定期对历史做摘要——把之前的对话压缩成一段总结文本。但问题在于：

- 一次 summarization 会丢失细节。第二次 summarization 是在第一次摘要的基础上再做摘要，信息衰减逐层加剧，最终只剩一个模糊的轮廓。
- 并非所有信息都适合压缩到一起。对话中的事实信息、用户偏好、任务中间产物、全局规则——它们的生命周期和召回条件完全不同，揉在一起只会让检索和注入都变得困难。

> 💡 **核心洞见**：记忆管理的真正难题不是"记不下来"，而是"不知道什么时候该记什么、什么时候该忘什么"。把不该长期记住的东西记住，和该记住的东西没记住，是同一枚硬币的两面。

**答案：分层管理**

因此我们需要的不是更聪明的裁剪算法，而是一套**分层记忆体系**——根据信息的不同属性进行分类存储和管理：

| 分类维度 | 含义 | 示例 |
|---|---|---|
| **使用周期** | 这条信息需要存活多久？ | 当前对话中 / 当前任务中 / 永久保留 |
| **影响范围** | 这条信息对谁可见？ | 仅当前对话 / 跨 Session 任务 / 全局 namespace |
| **使用场景** | 这条信息在什么时候被召回？ | 语义匹配 / 规则匹配 / 主动注入 |

老师课上引入了一个术语来描述分层后的信息组装单元——**Context Package**。每次模型请求时，Context Package 不是简单的一串消息列表，而是从不同层级、不同来源拼装出来的一个结构化上下文包。它可能包含：当前 context_window 中的活跃消息、从 Session memo 中提取的摘要要点、从 Workspace 中按需加载的文件引用、从 Namespace 中匹配到的长期记忆条目、以及当前生效的行为规则。

### 1.3 三层记忆架构全景预览

#### 🧠 直观理解

如果说 Context Window 解决的是「模型这次能看到什么」，那三层记忆架构解决的就是「信息该放在哪一层」。你可以把这三层理解为时间尺度上的三个抽屉：**当下抽屉**（这次对话中随手要用的东西）、**项目抽屉**（完成一个任务期间积累的材料）、**档案柜**（跨项目、跨时间都需要保留的知识和经验）。

#### 📖 详细解释

老师在课上展示了一张关键架构图，将记忆分为三个层次。下面是这张图的还原：

```
┌──────────────────────────────────────────────────────────────────────┐
│                    ③ NAMESPACE 长期记忆（Agent/全局级）                 │
│  生命周期：跨 Session · 跨 Task · 跨 Workspace                         │
│  影响范围：同一 namespace 下的所有 Agent 实例                            │
│                                                                      │
│  ┌─────────────────────────┐  ┌─────────────────────────┐            │
│  │ user-42:family          │  │ user-42:work            │            │
│  │ "孩子需要午休，迪士尼必去"│  │ "上午适合处理核心任务"     │            │
│  │ "家庭出行偏好宽松节奏"    │  │ "使用 Python 3.12+ 规范"  │            │
│  │ "孩子对海鲜过敏"          │  │ "部署前必须过 Code Review" │            │
│  └─────────────────────────┘  └─────────────────────────┘            │
│                                                                      │
│  存储内容：用户偏好 · 领域知识 · 跨任务经验 · 全局约束                    │
│  召回方式：语义匹配（embedding → cosine 排序）+ scope 过滤               │
├──────────────────────────────────────────────────────────────────────┤
│                    ② TASK WORKSPACE 任务记忆（任务级）                   │
│  生命周期：一个任务的多个 Session 之间                                   │
│  影响范围：同一个 Workspace 关联的所有 Session                           │
│                                                                      │
│  Session-1 ───→ Session-2 ───→ Session-N                             │
│      │               │              │                                 │
│  制品落盘(ref)   中间产物(ref)   检查报告(ref)                           │
│                                                                      │
│  读取方式：渐进式披露（ref 引用 → 目录列表 → 打开具体文件 → 段落按需加载）  │
│  存储内容：任务规划书 · 中间产物 · 校验结果 · 证据链                       │
├──────────────────────────────────────────────────────────────────────┤
│                    ① SESSION 短期记忆（对话级）                          │
│  生命周期：单次对话期间                                                 │
│  影响范围：当前 Session 实例                                            │
│                                                                      │
│  full_context  ──→ [T1][T2][T3]...[T15]  （完整记录，不投喂模型）       │
│  context_window ──→      [T11][T12]...[T15]  （模型可见，5~20 条）      │
│  memo          ──→ {"迪士尼必去", "孩子需要午休", "住宿选浦东"}  （压缩背包）│
│                                                                      │
│  存储内容：本轮对话 · 压缩摘要 · 未落盘的过程信息                          │
│  更新时机：每轮对话追加 full_context；resize 时更新 context_window 和 memo │
└──────────────────────────────────────────────────────────────────────┘
```

> 💡 **核心洞见**：三层之间不是「越上层越高级」的等级关系，而是**使用范围和生命周期不同**。选哪一层取决于信息的存活时间和共享范围，而不是信息本身的重要性。

三层的关键区别：

| 层级 | 生命周期 | 典型存活时间 | 共享范围 | 核心操作 |
|---|---|---|---|---|
| **Session 短期记忆** | 单次对话 | 几分钟到几小时 | 当前 Session | resize → 将旧消息压缩为 memo |
| **Task Workspace 任务记忆** | 一个任务周期 | 几小时到几天 | 同任务的多个 Session | 渐进式披露 → 按需读取文件 |
| **Namespace 长期记忆** | 永久（或手动清理前） | 几周到永久 | 同一 namespace 下的所有实例 | 语义检索 → scope 过滤 → 按相关性注入 |

**五种 Layer 类型预览**

在后续的代码实现中（MemoryRecord 数据类），这三层被进一步细分为五种具体的 layer 类型：

```
                    ┌────────────────────────┐
                    │     MemoryRecord        │
                    │  统一的记忆/治理载体     │
                    └───────────┬────────────┘
                                │
              ┌─────────────────┴─────────────────┐
              │                                   │
    ┌─────────┴─────────┐               ┌─────────┴─────────┐
    │    数据面（记忆）   │               │   治理面（规则/自省）│
    ├───────────────────┤               ├───────────────────┤
    │ session_short_term│ ← Session 级  │ rule_record       │ ← 已发布规则
    │ task_workspace    │ ← Task 级     │ reflection_candidate│ ← 待确认经验
    │ namespace_long_term│ ← 全局级     │                   │
    └───────────────────┘               └───────────────────┘
```

这五种 layer 类型贯穿了后续所有模块的代码实现，是理解本课代码体系的核心索引。其中前三类属于「数据面」，是记忆管理的范畴；后两类属于「治理面」，是行为规则与自成长策略的范畴。

> ⚠️ **注意事项**：这三种记忆层级并不要求全部同时启用。对于简单的 Agent 场景，可能只需要 Session 短期记忆就够了。随着 Agent 处理的任务复杂度提升，再逐步引入 Task Workspace 和 Namespace 长期记忆。这是一个「按需迭加」的设计，而非「全有或全无」。

### 1.4 数据面与治理面：记忆 vs 规则/自成长

#### 🧠 直观理解

如果把一个智能体比作一名员工，那么「数据面」就是这名员工的**工作笔记**——记录了和谁聊了什么、做了什么任务、客户有什么偏好。而「治理面」则是这名员工从工作中总结出来的**做事方法**——「上午处理核心任务效率更高」「和这个客户沟通要避免技术术语」「每次提交前要过一遍检查清单」。笔记记的是「发生了什么」，方法记的是「怎么做更好」。

#### 📖 详细解释

到目前为止我们讨论的记忆（Session memo、Workspace 文件、Namespace 知识），本质上都是在回答一个问题：**发生过什么？** 它们存储的是和会话相关的、和用户相关的数据——对话历史、任务产物、用户偏好。这些统称为**数据面**（Data Plane）。

但老师在课上特别指出，还有一类信息性质完全不同。它不是关于「发生了什么」，而是关于「应该怎么做」。它来自模型在大量执行过程中对自己的行为进行**自省和反思**——哪类回答用户给了正向反馈？哪种处理路径在类似任务中效果更好？什么情况下应该避免某种操作？这些经验沉淀下来，形成了一套**规则和规范**，我们称之为**治理面**（Governance Plane）。

```
                        智能体运行产生的信息
                                │
              ┌─────────────────┴─────────────────┐
              ▼                                   ▼
    ┌──────────────────┐               ┌──────────────────────┐
    │    数据面（记忆）   │               │   治理面（规则/自省）    │
    │                  │               │                      │
    │  "发生了什么？"    │               │  "应该怎么做？"         │
    │                  │               │                      │
    │  对话历史          │               │  行为规则              │
    │  任务产物          │               │  经验总结              │
    │  用户偏好          │               │  操作规范              │
    │  领域知识          │               │  反思洞察              │
    │                  │               │                      │
    │  来源：外部输入     │               │  来源：模型自省          │
    │  注入方式：被动匹配  │               │  注入方式：主动注入       │
    └──────────────────┘               └──────────────────────┘
```

| 对比维度 | 数据面（记忆） | 治理面（规则/自省） |
|---|---|---|
| **核心问题** | 发生过什么？ | 应该怎么做？ |
| **信息来源** | 对话、任务产物、外部输入 | 模型对自身行为的反思和归纳 |
| **典型内容** | 用户偏好、对话摘要、任务中间结果 | 行为策略、操作规范、经验教训 |
| **注入上下文的方式** | 按需检索（语义匹配 + scope 过滤） | 按规则匹配后主动注入 |
| **更新机制** | 对话完成后批量整理 | 执行反馈 → 候选生成 → 验证 → 发布 |
| **在 MemoryRecord 中的对应** | `session_short_term` / `task_workspace` / `namespace_long_term` | `rule_record` / `reflection_candidate` |

**治理面的两种对象**

治理面在代码实现中对应两种具体类型：

| 治理对象 | 含义 | 状态 | 注入方式 |
|---|---|---|---|
| **Rule Record**（规则记录） | 已确认发布的行为策略、阈值、禁用项。可以自上而下（管理员发布）产生，也可以由 Reflection Candidate 验证后自下而上晋升 | `active` / `versioned` | scope + 版本 + 有效期匹配后自动注入 |
| **Reflection Candidate**（反思候选） | 模型在执行过程中自己提出的经验或规则建议。例如「在处理家庭旅行规划时，应该优先考虑儿童友好型酒店」 | `pending`（待审核） | 默认不注入，需要经过验证确认后才能晋升为 Rule Record |

> 💡 **核心洞见**：数据面和治理面的分离，是让 Agent 从「只会执行」进化到「会自我改进」的关键分界线。没有治理面，Agent 每次面对相似任务都像是在第一次处理——它记住了过去的事实（数据面），但没有从这些事实中提炼出行为模式（治理面）。有了治理面，Agent 的行为可以随着运行经验的积累而逐步优化，这就是「自成长」的起点。

> ⚠️ **注意事项**：Reflection Candidate 在确认之前**甚至不是正式的记忆**。它被存储但不会注入到模型请求中，只有通过验证流程（人工审核或自动校验）后，才被晋升为正式的 Rule Record 并对后续行为产生影响。这个「候选→验证→发布」的流程，是防止模型从单次偶然经验中过度泛化的关键安全机制。

> 🔗 **延伸阅读**：治理面的代码实现——Rule Record 和 Reflection Candidate 的完整生命周期管理——将在后续模块中详细展开。

**Step 3: 注册 analysis handler——判断要不要裁剪、用什么策略**

#### 🧠 直观理解

analysis handler 就像一个"安检员"。每次有新的消息进入会话时，安检员看一眼当前的消息列表，判断：是否需要裁剪？如果需要，走哪条裁剪通道（策略名）？如果不需要，挥手放行（返回 `None`）。

#### 📖 详细解释

analysis handler 的函数签名：

```python
async def handler_name(full_context, context_window, memo, settings) -> str | None:
```

四个参数的含义：

| 参数 | 类型 | 含义 |
|---|---|---|
| `full_context` | `list[ChatMessage]` | 完整的对话历史，从未被裁剪 |
| `context_window` | `list[ChatMessage]` | 当前模型可见的消息窗口，裁剪发生在这里 |
| `memo` | `str` | 压缩摘要，初始为空字符串 |
| `settings` | `dict` | 创建 Session 时传入的自定义配置 |

返回值是一个**策略名字符串**或 `None`。策略名将在下一步被 route 到对应的 resize handler。

为什么设计成返回策略名而不是直接裁剪？因为**判断逻辑**和**执行逻辑**是两类本质上不同的操作——判断可能很简单（比如只检查窗口长度），但执行可能很复杂（比如调模型压缩）。分离后你可以独立替换或测试每一个。

#### 💻 代码示例

```python
async def choose_recent_window(full_context, context_window, memo, settings):
    """analysis handler: 当窗口长度 >= 7 时触发 keep_recent 策略"""
    handler_calls.append(("analysis", len(context_window)))
    return "keep_recent" if len(context_window) >= 7 else None
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 1 | `async def choose_recent_window(...)` | 必须定义为 async 函数，四个参数缺一不可 |
| 2 | `handler_calls.append(...)` | 记录调用（仅用于调试演示，生产环境可去掉） |
| 3 | `return "keep_recent" if len(context_window) >= 7 else None` | 核心逻辑：窗口 >= 7 条则路由到 `"keep_recent"` 策略；否则 `None` 表示无需裁剪 |

#### 🔀 策略路由关系图

分析和裁剪通过**策略名**关联——analysis 返回的策略名必须与 resize 注册时的名称**完全一致**：

```
┌──────────────────────────────────────────────────────────┐
│                   Session 消息流                          │
│                                                          │
│   新消息 add ──▶ analysis handler                        │
│                      │                                   │
│                      │ 返回 "keep_recent"                 │
│                      ▼                                   │
│              ┌───────────────┐                           │
│              │  策略名路由表   │                           │
│              │               │                           │
│              │ "keep_recent" │────▶ keep_recent handler  │
│              │ "keep_one"    │────▶ keep_one handler     │
│              │  ...          │     ...                   │
│              └───────────────┘                           │
│                      │                                   │
│                      │ 返回 None（不需要裁剪）              │
│                      ▼                                   │
│                 跳过裁剪，消息直接入库                      │
└──────────────────────────────────────────────────────────┘
```

> ⚠️ **注意事项**：策略名必须精确匹配。如果你在 analysis 中返回 `"keep_recent"`，但 resize 注册时用的是 `"keep-recent"`（连字符差异），路由会失败，裁剪不会执行。

---

**Step 4: 注册 resize handler——执行具体的裁剪动作**

#### 🧠 直观理解

resize handler 是"执行者"。安检员（analysis handler）告诉你走哪条通道，resize handler 就按照那条通道的规则执行裁剪。

#### 📖 详细解释

resize handler 的函数签名与 analysis handler 完全相同，但返回值不同——它返回一个**三元组**：

```python
async def handler_name(full_context, context_window, memo, settings) -> tuple:
    return (新的_full_context, 新的_context_window, 新的_memo)
```

| 返回值位置 | 含义 | 常见做法 |
|---|---|---|
| 第 1 个 | `full_context` | 返回 `None` 表示不修改完整记录（通常不删历史） |
| 第 2 个 | `context_window` | 返回裁剪后的消息列表，这是核心输出 |
| 第 3 个 | `memo` | 返回更新后的 memo，不变则返回原值 |

#### 💻 代码示例

```python
async def keep_recent(full_context, context_window, memo, settings):
    """resize handler: 保留最近 5 条消息"""
    handler_calls.append(("resize", len(context_window)))
    return None, list(context_window[-5:]), memo
    #  ↑ full_context 不变    ↑ 裁剪后的窗口  ↑ memo 不变
```

**为什么 `full_context` 返回 `None`？** 因为 `full_context` 是完整历史记录，我们通常不希望删除它——即使窗口裁剪了，完整记录依然保留，供后续摘要压缩等操作使用。返回 `None` 意味着"不修改"。

**为什么 `memo` 返回原值？** 因为 `keep_recent` 是一个"保留最近 N 条"的简单裁剪策略，不涉及摘要生成。memo 的更新由压缩策略（后续章节会讲）来负责。

**Step 5: 创建 Session 并注入 Agent**

注册完钩子后，创建 Session 实例，将消息列表加入会话，然后将裁剪后的 `context_window` 注入 Agent 进行推理。

```python
# 创建 Session 实例
review_session = Session(id="lesson-21-review")

# 注册钩子：analysis handler 无需指定策略名，resize handler 必须指定

review_session.register_resize_handler("keep_recent", keep_recent)

# 添加消息：auto_resize 默认 True，add_chat_history 时自动触发 analysis → resize

    .input("当前还有哪项住宿信息没有确认？")
    .instruct("只根据 chat_history 回答；不知道就明确说不知道。")
    .async_start()

# 输出: [('analysis', 7), ('resize', 7)]

# 输出: 最近 5 条消息的 content（因为 keep_recent 保留了最近 5 条）
```

**Step 6: 理解 auto_resize 机制——何时自动、何时手动**

#### 🧠 直观理解

auto_resize 就像手机的自动亮度调节。默认开启时，每次 `add_chat_history`（往会话里加一条消息）都会自动触发一次 analysis → resize 流程。如果你觉得"每次加消息都要检查一次"太频繁——特别是在 analysis 中调了模型的情况下——你可以关掉它，手动控制何时裁剪。

#### 📖 详细解释

##### 推导链：auto_resize 的开关决策

```
起点: auto_resize 默认为 True
  │
  ├── 第 1 步: 每次 add_chat_history 触发一次 analysis
  │   为什么？Session 框架设计为"消息变更即检查"，保证窗口始终处于裁剪后状态
  │
  ├── 第 2 步: 判断你的 analysis handler 是否调用了大模型
  │   为什么？如果 analysis 只是检查 `len(context_window)`，成本几乎为零，开着没问题
  │          但如果 analysis 中调了 LLM 来判断"是否需要裁剪"，每次 add 都调用就太贵了
  │
  ├── 第 3 步: 判断你是否需要批量添加后统一裁剪
  │   为什么？批量添加时，中间状态的裁剪没有意义——你可能刚裁掉的消息在下一轮又被加回来
  │
  └── 结论:
      - analysis 轻量 → 保持 True（几乎无性能损耗）
      - analysis 调模型 → 设为 False + 手动 async_resize()
      - 批量添加 → 设为 False + 添加完毕后手动 async_resize()
```

##### 场景对比表

| 场景 | auto_resize 建议 | 原因 |
|---|---|---|
| 裁剪策略简单轻量（如只检查窗口长度） | 保持 `True` | 几乎无性能损耗，无需手动管理 |
| analysis 中需要调大模型来判断 | 设为 `False` | 避免每次 add 都触发模型调用，成本失控 |
| 批量添加消息后统一裁剪 | 设为 `False` | 避免中间状态无效裁剪，减少不必要的 resize 次数 |
| 需要精确控制裁剪时机 | 设为 `False` | 由你决定何时触发，框架不干预 |

##### auto_resize=False 时的手动裁剪

```python
# 关闭 auto_resize
review_session_no_auto = Session(id="lesson-21-no-auto", auto_resize=False)
review_session_no_auto.register_analysis_handler(choose_recent_window)
review_session_no_auto.register_resize_handler("keep_recent", keep_recent)

# 批量添加消息——此时不会触发裁剪
await review_session_no_auto.async_add_chat_history(TRAVEL_MESSAGES)

# 手动触发裁剪
await review_session_no_auto.async_resize()
```

#### 🐛 排错实录：auto_resize=False 时手动 resize 仍先 analysis 再 resize

这是本课中老师现场强调的一个"反直觉"设计点。

**❌ 直觉预期**
```python
# 很多人会以为：关闭 auto_resize 后，手动 async_resize() 是"跳过分析、直接裁剪"
# 即只执行 resize handler，不执行 analysis handler
```

**🚨 实际行为**
```python
# 即使 auto_resize=False，调用 async_resize() 时：
# handler_calls 的输出仍然是 [('analysis', 7), ('resize', 7)]
# 说明 analysis handler 被调用了！
```

**🔍 原因分析**

这是框架的**有意设计**。`async_resize()` 触发的是一个**完整的检查+裁剪流程**，不是单纯的裁剪动作。设计意图：

1. **语义一致性**：`async_resize()` 的语义是"执行一次 resize 流程"，而 Session 中"resize 流程"的定义就是 `analysis → resize`。如果 `async_resize()` 跳过 analysis，就变成了另一个语义不同的操作。

2. **避免心智负担**：如果 auto_resize 关闭后 `async_resize()` 只执行裁剪不执行分析，开发者需要理解两种不同的 resize 行为（带分析的自动 resize vs 不带分析的手动 resize），框架设计选择保持流程一致——无论自动还是手动，走的都是同一套 analysis → resize 管线。

3. **auto_resize 到底控制什么？** 它控制的不是 analysis 和 resize 之间的关联关系，而是控制"谁来触发这个流程"——auto_resize=True 是框架自动帮你触发，auto_resize=False 是你自己手动触发。但无论谁触发，流程本身不变。

**✅ 正确理解**

```
auto_resize = True:
    每次 add_chat_history → 框架自动调用 analysis → resize

auto_resize = False:
    每次 add_chat_history → 框架什么都不做
    你手动调用 async_resize() → 仍然执行 analysis → resize（流程不变）
```

> 💡 **核心洞见**：把 auto_resize 理解为"自动触发开关"而非"流程简化开关"。关闭它意味着你接管触发时机，但不改变触发后的执行流程。

---

**Step 7: 多策略模式——注册多个策略并理解优先级**

#### 🧠 直观理解

一个会话可能需要多种裁剪策略。比如：用户说了"重新安排"——此时只需要保留最后一条消息（上下文切换）；正常情况下——保留最近 5 条。analysis handler 就像一个多层判断的调度中心，匹配到哪个条件就走哪条策略通道。

#### 📖 详细解释

analysis handler 中，策略的选择是**按代码执行顺序**的——先判断的条件如果命中，直接 return，后续判断不再执行。这是一种**覆盖关系**（不是并行匹配）。

##### 🔗 推导链：多策略优先级的覆盖机制

```
起点: analysis handler 被调用
  │
  ├── 第 1 步: 检查最后一条消息是否包含"重新安排"
  │   为什么先检查？因为这是更高优先级的信号——用户明确要求重新规划
  │   如果命中 → return "keep_one"，后续代码不执行（覆盖关系）
  │
  ├── 第 2 步: 检查窗口长度是否 >= 7
  │   为什么后检查？因为这是兜底策略——没有特殊信号时按长度裁剪
  │   如果命中 → return "keep_recent"
  │
  └── 第 3 步: 都不满足 → return None（不需要裁剪）
```

#### 💻 代码示例

首先注册第二个 resize handler——`keep_one`，只保留最后 1 条消息：

```python
async def keep_one(full_context, context_window, memo, settings):
    """resize handler: 只保留最后 1 条消息（用于上下文切换场景）"""
    handler_calls.append(("resize-keep_one", len(context_window)))
    return None, [context_window[-1]], memo
    #                      ↑ 只取最后一条
```

然后升级 analysis handler，加入关键字匹配逻辑：

```python
async def choose_strategy_by_context(full_context, context_window, memo, settings):
    """
    analysis handler: 多策略路由
    优先级: 关键字匹配 > 窗口长度判断 > 无需裁剪
    """
    handler_calls.append(("analysis", len(context_window)))

    # 策略 1: 高优先级——用户明确要求重新安排
    # 检查最后一条消息的 content 中是否包含"重新安排"
    last_message = context_window[-1]
    if "重新安排" in last_message.content:
        return "keep_one"  # ← 命中后直接 return，后续代码不执行

    # 策略 2: 兜底策略——窗口过长时保留最近 5 条
    if len(context_window) >= 7:
        return "keep_recent"

    # 无需裁剪
    return None
```

全部注册到 Session：

```python
multi_session = Session(id="lesson-21-multi")
multi_session.register_analysis_handler(choose_strategy_by_context)
multi_session.register_resize_handler("keep_recent", keep_recent)
multi_session.register_resize_handler("keep_one", keep_one)

await multi_session.async_add_chat_history(TRAVEL_MESSAGES)

# 由于最后一条消息包含"重新安排"，命中 keep_one 策略
# context_window 只剩 1 条消息
print("窗口剩余消息数:", len(multi_session.context_window))  # 输出: 1
```

#### 🔀 多策略执行流程

```
TRAVEL_MESSAGES（7 条）加入 Session
    │
    ▼
analysis handler: choose_strategy_by_context
    │
    ├── 检查: 最后一条消息包含"重新安排"？
    │   ├── 是 → return "keep_one" ──▶ resize: keep_one handler
    │   │                                   │
    │   │                           保留最后 1 条 ← 窗口 = 1
    │   │
    │   └── 否 → 继续往下
    │       │
    │       ├── 检查: len(context_window) >= 7？
    │       │   ├── 是 → return "keep_recent" ──▶ resize: keep_recent handler
    │       │   │                                     │
    │       │   │                             保留最近 5 条 ← 窗口 = 5
    │       │   │
    │       │   └── 否 → return None（跳过裁剪）
    │       │
    │       └── 策略永远不会"冲突"——先命中的先返回，后续策略被覆盖
    │
    └── 结论: 多策略之间是覆盖关系，不是并行关系
```

##### 验证覆盖关系

如果把最后一条消息中的"重新安排"改成"继续看看"，会怎样？

```python
# 修改最后一条消息
TRAVEL_MESSAGES_MODIFIED = TRAVEL_MESSAGES.copy()
TRAVEL_MESSAGES_MODIFIED[-1] = {
    "role": "user",
    "content": "继续看看还有什么需要注意的。"
}

multi_session2 = Session(id="lesson-21-multi-2")
multi_session2.register_analysis_handler(choose_strategy_by_context)
multi_session2.register_resize_handler("keep_recent", keep_recent)
multi_session2.register_resize_handler("keep_one", keep_one)

await multi_session2.async_add_chat_history(TRAVEL_MESSAGES_MODIFIED)

# "重新安排"不在最后一条消息中 → 策略 1 未命中
# 窗口长度 >= 7 → 命中策略 2，走 keep_recent
print("窗口剩余消息数:", len(multi_session2.context_window))  # 输出: 5
```

> 💡 **核心洞见**：多策略之间不会产生冲突，因为是**覆盖关系**。`return` 语句决定了哪个策略生效——谁先被判断、先 return，谁就胜出。框架不会把所有策略并行执行一遍然后选一个。

> ❓ **常见疑问**：有必要搞这么多策略吗？全部写在一个 analysis handler 里自己处理不行吗？
>
> **可以，但有代价。** 你可以把所有逻辑写在一个函数里，直接在内部完成判断+裁剪。但这样做的代价是**丧失了策略的可复用性**。当你把"keep_recent"注册为独立的 resize handler 时，你可以在多个 analysis handler 中引用它——不同的判断条件可以路由到同一个裁剪动作。策略名路由机制的本质是**解耦判断和执行**。

**Step 8: ChatMessage 数据模型——不是普通的字典**

#### 🧠 直观理解

Session 中存储的消息不是普通的 Python 字典，而是 `ChatMessage` 数据模型（dataclass）。这就像你拿到的是一个"封装好的快递盒"而不是"散装的物品清单"——你不能用 `message["key"]` 的方式访问，而要用 `message.key`（属性访问）。

#### 📖 详细解释

为什么 Session 要用 ChatMessage dataclass 而不是普通字典？

1. **类型安全**：dataclass 有明确的字段定义（`role`、`content` 等），IDE 可以自动补全，拼写错误在编码阶段就能发现
2. **与模型请求的数据结构对齐**：各大 LLM API 的消息格式都要求类似 `{"role": "...", "content": "..."}` 的结构，ChatMessage 是这种结构的规范化表达
3. **序列化与校验**：dataclass 天然支持序列化，内部可以加校验逻辑

但在构造消息列表时，你可以用普通字典——框架会自动将字典转换为 ChatMessage 实例。**输出时（从 context_window 取出）就是 ChatMessage 实例了**，这就是容易踩坑的地方：你构造时用的是字典，取出时以为还是字典。

#### 🐛 排错实录：ChatMessage 取不到 content

**❌ 错误写法**
```python
# 很多人会习惯性地用字典方式访问
for message in review_session.context_window:
    print(message["content"])  # ← 报错！
```

**🚨 报错信息**
```
TypeError: 'ChatMessage' object is not subscriptable
```

**🔍 原因分析**

`ChatMessage` 是 dataclass，不是 `dict` 的子类。它不支持 `[]` 下标访问。你只能通过 `.` 属性访问。

为什么构造时可以写 `{"role": "user", "content": "..."}`？因为 `async_add_chat_history` 接收的类型是 `ChatMessageDict`（TypedDict），框架内部做了转换：输入时接受字典 → 内部存储为 ChatMessage dataclass → 输出时返回 ChatMessage 实例。

**✅ 正确写法**
```python
# 用属性访问
for message in review_session.context_window:
    print(f"{message.role}: {message.content}")

# 如果需要转成字典（比如传给其他不接受 ChatMessage 的工具）
message_dict = {"role": message.role, "content": message.content}
```

**📝 经验总结**

> 一个简单的心理模型：**输入用字典，输出用属性**。当你从 `session.context_window` 取出消息时，它们已经是 ChatMessage 实例了，忘掉字典语法。IDE 的自动补全是你最好的朋友——输入 `message.` 后看看有哪些属性可用。

#### 🐛 排错指南

以下是本模块实操中最常见的坑：

##### 坑 1：策略名不匹配导致 resize 不执行

**现象**：analysis handler 返回了策略名，但窗口没有变化，`handler_calls` 中也只有 analysis 记录。

**原因**：analysis 返回的策略名与 resize 注册时的名称不一致（比如一个是 `"keep_recent"`，另一个是 `"keep-recent"`）。

**解决**：检查 `register_resize_handler` 的第一个参数与 analysis handler 的 `return` 值是否字符级完全一致。

##### 坑 2：忘记 await async_add_chat_history

**现象**：消息没有加入 Session，`context_window` 为空。

**原因**：`async_add_chat_history` 是异步方法，必须 `await`。直接调用不 `await` 会得到一个 coroutine 对象，消息实际上没有加入。

**解决**：确保 `await review_session.async_add_chat_history(...)`。

##### 坑 3：ChatMessage 用下标访问

**现象**：`TypeError: 'ChatMessage' object is not subscriptable`

**原因**：从 `context_window` 取出的消息是 ChatMessage dataclass，不是字典。

**解决**：用 `message.content` 而非 `message["content"]`。详见 Step 8 的排错实录。

##### 坑 4：auto_resize=False 后忘记手动 resize

**现象**：关闭 auto_resize 后，窗口长度等于全部消息数，裁剪未生效。

**原因**：关闭 auto_resize 后框架不再自动触发 analysis → resize，必须手动调用 `await session.async_resize()`。

**解决**：在所有 `async_add_chat_history` 调用完毕后，手动执行 `await session.async_resize()`。

## 三、压缩触发策略设计

在上一章我们建立了 Session 钩子机制，其中 `analysis handler` 负责判断"是否该压缩"以及"用什么策略压缩"。本章深入剖析"触发条件"——什么时候该执行压缩？这不是一个简单的 if-else，而是一个由四种触发策略组合而成的决策体系。

### 3.1 策略设计总览

四种触发策略各有侧重，按成本从低到高排列：

| 策略 | 触发条件 | 单次成本 | 精度 | 典型阈值 | 适用场景 |
|---|---|---|---|---|---|
| 轮次触发 | 对话轮次超过 N | 极低（计数） | 低 | 20~50 轮 | 简单对话、FAQ |
| 长度触发 | 上下文 Token 超限 | 极低（算长度） | 中 | 窗口上限 × 0.85 | 通用场景，需精准控预算 |
| 主动标记 | 检测到特殊标记文本 | 极低（字符串匹配） | 高（人为意图） | 无阈值 | 用户/开发者显式触发 |
| 话题漂移 | 语义检测话题已切换 | 高（需调模型） | 中 | 每 3~5 轮检查一次 | 长对话、多任务场景 |

> 🔗 **前置知识**：钩子机制中 `analysis handler` 的返回值决定了走哪种策略。详见 [二、Session 钩子机制回顾与深入](#二session-钩子机制回顾与深入)

---

### 3.2 轮次触发策略

#### 🧠 直观理解

就像整理书桌——约定每用完 20 张草稿纸就统一收拾一次。不看纸上的内容有多长、写的什么，只看用了多少张。

#### 📖 详细解释

轮次触发是所有策略中最简单的一种：统计当前会话中用户发言的次数，达到预设阈值就触发压缩。

**为什么用轮次？** 每一轮 user 发言是一个天然的逻辑切分点，计数成本几乎为零。对于交互模式固定的场景（如 FAQ 机器人），轮次是最直观的衡量标准。

**为什么不是每轮都压？** 频繁压缩浪费计算资源，且每次压缩都有信息损失的风险。需要在"让上下文保持干净"和"尽量减少模型调用"之间取得平衡。

**N 取多大？** 这完全取决于业务场景：客服对话 10 轮可能就够（用户问题简单），深度推理场景可能需要 50 轮以上（每轮包含大量中间推理）。N 的值应该在业务上线前通过实际对话数据分析来确定。

#### 🔀 对比辨析

| 维度 | 轮次触发 | 长度触发 |
|---|---|---|
| 衡量对象 | 对话交互次数 | 文本信息量 |
| 精确度 | 低——"好的"和一大段需求描述都算 1 轮 | 高——按实际 Token 数判断 |
| 极端场景 | 一句话问答 100 轮可能还没超预算 | 一段超长需求 3 轮就可能爆窗口 |
| 实现难度 | 极低 | 低 |

> ❓ **常见误解**："轮次越多信息越多"——不一定。有的轮次只有"好的""收到"几个字，有的轮次包含几百行的需求文档粘贴。轮次衡量的是交互频次，不是信息量。

#### 💻 代码示例

```python
# 文件名: trigger_rounds.py
# 功能: 轮次触发策略——统计 user 发言轮次，超出阈值则触发压缩

def should_compress_by_rounds(context_window: list[dict], max_rounds: int = 20) -> bool:
    """
    按轮次判断是否需要压缩。

    Args:
        context_window: 当前会话的消息列表，每条消息含 "role" 字段
        max_rounds: 最大允许轮次，超过此值触发压缩

    Returns:
        True 表示应触发压缩，False 表示继续正常对话
    """
    # 统计 user 角色的发言次数（通常一轮 = 一次 user 发言 + 一次 assistant 回应）
    round_count = sum(1 for msg in context_window if msg.get("role") == "user")
    return round_count >= max_rounds
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 4 | `def should_compress_by_rounds(...)` | 核心判断函数，返回布尔值供 `analysis handler` 使用 |
| 14 | `msg.get("role") == "user"` | 只统计 user 角色——以用户发言为"轮"的边界 |
| 15 | `round_count >= max_rounds` | 达到阈值即返回 True，简单直接 |

---

### 3.3 长度触发策略

#### 🧠 直观理解

就像书包有容量限制——不是按"放了几本书"算，而是按"还剩多少空间"算。两本厚字典和十本薄练习册，占的空间完全不同。

#### 📖 详细解释

长度触发按上下文的实际 Token 数来决定是否压缩。相比轮次触发，它更精准地反映上下文窗口的实际占用情况。

**为什么比轮次精准？** 因为信息量和轮次不是线性关系。一段超长需求描述 3 轮可能消耗 5000 Token，而 20 轮简单问答可能只有 2000 Token。用 Token 数做判断，才能真正防止窗口溢出。

**为什么需要安全边距（safe_margin）？**

这是长度触发策略中最容易被忽视的关键设计。假设你的模型窗口上限是 128K Token，如果你在上下文达到 128K 时才触发压缩，会发生什么？此时 system prompt 本身已经占了几千 Token，压缩指令（告诉模型"请压缩前面内容"）又要占用几百 Token，再加上模型输出的压缩结果——三者叠加必然超出 128K，导致 API 调用失败。

安全边距就是提前预留这段空间：`触发阈值 = 窗口上限 × safe_margin`。通常 safe_margin 取值 0.80~0.90，即预留 10%~20% 的空间给压缩指令和模型输出。

**为什么 safe_margin 不能太小也不能太大？**
- 太小（如 0.95）：预留空间不够，压缩指令仍可能溢出
- 太大（如 0.50）：浪费一半窗口，压缩过于频繁

#### 🔀 对比辨析

| 维度 | 固定阈值 | 动态安全边距 |
|---|---|---|
| 定义方式 | `max_tokens=8000`，写死 | `max_tokens × safe_margin`，按比例计算 |
| 模型切换时 | 需手动调参数 | 自动适应新窗口大小 |
| 风险 | 小模型溢出，大模型浪费 | 始终留出合理缓冲 |

#### 💻 代码示例

```python
# 文件名: trigger_length.py
# 功能: 长度触发策略——按上下文 Token 数触发压缩，含安全边距

def should_compress_by_length(
    context_window: list[dict],
    max_tokens: int = 8000,
    safe_margin: float = 0.85
) -> bool:
    """
    按上下文 Token 数判断是否需要压缩。

    Args:
        context_window: 当前会话的消息列表
        max_tokens: 模型上下文窗口上限（Token 数）
        safe_margin: 安全边距系数（0~1），预留压缩指令和输出的空间

    Returns:
        True 表示应触发压缩
    """
    # 实际触发阈值 = 窗口上限 × 安全边距
    # 例如 128K × 0.85 = 108.8K，在 108.8K 时触发，预留 19.2K 给压缩流程
    threshold = int(max_tokens * safe_margin)

    # 简易 Token 估算：英文约 4 字符/Token，中文约 1.5 字符/Token
    # 这里用 2.5 作为混合文本的经验系数，生产环境应使用 tiktoken 等精确 Tokenizer
    current_tokens = sum(
        len(msg.get("content", "")) // 2.5
        for msg in context_window
    )

    return current_tokens > threshold
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 9 | `safe_margin: float = 0.85` | 默认预留 15% 空间——兼顾安全和利用率 |
| 20 | `threshold = int(max_tokens * safe_margin)` | 计算实际触发阈值，乘以 margin 而非直接使用 max_tokens |
| 24 | `len(msg.get("content", "")) // 2.5` | 简易 Token 估算，生产环境应替换为 `tiktoken` 精确计算 |

> ⚠️ **注意事项**：safe_margin（通常 0.8~0.9）用于预留压缩指令本身消耗的 Token。如果窗口 128K，在 128K 时才触发压缩，system prompt + 压缩指令 + 模型输出必将超出限制。这看起来是细节，但线上出问题时往往是这里先炸。

---

### 3.4 主动标记触发策略

#### 🧠 直观理解

就像遥控器上的"归位"按钮——扫地机器人平时自动清扫，但如果用户觉得该回去了，按一下按钮它就强制回充。不需要等电量不足，不需要等清扫完成。

#### 📖 详细解释

主动标记触发策略允许在对话中插入特殊标记（如 `[COMPACT]`、`整理记忆`、`/compact`），当系统检测到这些标记时，跳过所有其他判断条件，立即执行压缩。

**为什么需要主动标记？** 轮次和长度都是"被动"策略——只有当条件满足时才触发。但有时用户或开发者确切地知道"该压缩了"，而被动条件尚未满足。例如：用户在长对话中完成了一个大任务，即将开始新任务，这时前面的上下文已经不再需要，但轮次和长度都没超限。主动标记给了人"显式控制"的入口。

**标记的设计原则：**
- 不干扰正常对话：标记内容应该是自然语言或约定俗成的指令，如 `整理记忆` 比 `/compact` 更自然
- 容易被检测：从 context_window 末尾向前扫描，找到最近的标记即触发
- 可被移除：压缩完成后标记不应该留在压缩结果中

**为什么从后往前扫描？** 对话中可能出现多个标记（用户多次要求压缩），只有最近的那次有意义——后面的对话内容不应该被压缩掉。

#### 🔀 对比辨析

| 维度 | 被动触发（轮次/长度） | 主动标记触发 |
|---|---|---|
| 触发者 | 系统自动 | 用户/开发者显式指定 |
| 时机 | 达到阈值时 | 任意时刻 |
| 可控性 | 低（取决于预设阈值） | 高（人决定何时触发） |
| 适用场景 | 系统自主管理 | 任务边界明确、需人为干预 |

#### 💻 代码示例

```python
# 文件名: trigger_marker.py
# 功能: 主动标记触发策略——检测对话中的特殊标记，显式触发压缩

# 可配置的触发标记列表，支持中文自然语言和英文简写
TRIGGER_MARKERS = ["[COMPACT]", "整理记忆", "/compact"]

def should_compress_by_marker(context_window: list[dict]) -> bool:
    """
    检测对话中是否存在主动压缩标记。

    从后往前扫描，找到最近的标记即返回 True。
    如果用户最近的标记是"整理记忆"，说明用户明确希望压缩。

    Args:
        context_window: 当前会话的消息列表

    Returns:
        True 表示检测到主动标记，应立即压缩
    """
    # reversed: 从最新消息往前扫描
    # 只有最近一次的标记有意义——后续对话不应该被压缩
    for msg in reversed(context_window):
        content = msg.get("content", "")
        # 任意一个标记出现在消息内容中即触发
        if any(marker in content for marker in TRIGGER_MARKERS):
            return True
    return False
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 5 | `TRIGGER_MARKERS = [...]` | 集中管理标记列表，方便扩展——新增语言只需加一条 |
| 18 | `for msg in reversed(context_window)` | 从最新往最旧扫描，最近标记优先 |
| 21 | `any(marker in content for marker in TRIGGER_MARKERS)` | 子串匹配，不要求整条消息等于标记 |

### 3.5 话题漂移触发策略

#### 🧠 直观理解

就像聊天时发现朋友突然换了话题——从"晚上吃什么"跳到"下周的项目计划"，你需要意识到"刚才聊的可以总结了"。用户不会主动说"我要切换话题了"，话题切换发生在无形之中。

#### 📖 详细解释

话题漂移触发策略解决了一个容易被忽视的场景：用户在长对话中悄然切换话题，此时轮次未超标、长度未超标、也没有主动标记——但前面关于旧话题的上下文已经失效，继续保留只会增加噪声。更关键的是，用户自己也不一定有"换话题就该新开窗口"的意识，他们往往在一个窗口里连续聊多个不相关的事情。

**为什么需要打点计数器？**

话题漂移检测需要调用模型来判断"当前话题和前面几轮是否还是同一个话题"，这是一次额外的模型调用，成本远高于计数和字符串比较。如果每轮都做一次语义判断，成本将线性增长。

打点计数器的设计思路是：每 N 轮（如 3 轮或 5 轮）才做一次话题检查。这样在对话流畅（话题未切换）时不会频繁浪费模型调用，而在话题切换时最多延迟 N-1 轮就能捕获。

**打点计数器的实现细节：**
1. 在 settings 中维护一个计数器 `_counter`，初始值为 0
2. 每次 `tick()` 被调用（即每轮对话后），计数器 +1
3. 当计数器达到 `check_interval`（如 3）时，重置计数器为 0，执行语义检查
4. 语义检查：取最近 N 条消息（当前话题）vs 稍早的 N 条消息（历史话题），用模型判断是否仍在同一话题
5. 如果模型判断话题已漂移 → 触发压缩；否则继续

**为什么 check_interval 选 3~5 而不是更大？**
- 太小（如 1~2）：成本接近每轮都查，浪费
- 太大（如 10）：话题可能已经漂移了 10 轮才发现，前面的旧信息已经污染了多轮对话

#### 🔀 对比辨析

| 维度 | 轮次/长度触发 | 话题漂移触发 |
|---|---|---|
| 判断依据 | 数量指标（计数/Token） | 语义指标（话题是否一致） |
| 成本 | 极低 | 高（需调模型） |
| 能发现的问题 | 上下文"太长了" | 上下文"失效了" |
| 漏检风险 | 3 轮超长需求但轮次很少（漏检） | 模型误判话题未切换（漏检） |

#### 💻 代码示例

```python
# 文件名: trigger_topic_drift.py
# 功能: 话题漂移检测——用打点计数器降低语义检查的频率开销

class TopicDriftTrigger:
    """
    话题漂移触发器。

    设计思路：
    1. 不是每轮都检查（成本太高），而是用 counter 控制检查频率
    2. 每 check_interval 轮，取当前话题和历史话题的样本，交给模型判断
    3. 如果模型判断话题已漂移，返回 True，由上层执行压缩
    """

    def __init__(self, check_interval: int = 3):
        """
        Args:
            check_interval: 每隔多少轮做一次话题漂移检查
                            默认 3——既不频繁浪费调用，也能及时捕获话题切换
        """
        self.check_interval = check_interval
        self._counter = 0  # 打点计数器，每轮对话后 +1

    def tick(self, context_window: list[dict]) -> bool:
        """
        每轮对话后调用，由 Session 钩子中的 analysis handler 驱动。

        Args:
            context_window: 当前完整的上下文窗口

        Returns:
            True 表示检测到话题漂移，应触发压缩
            False 表示话题连续或未到检查时机
        """
        self._counter += 1

        # 未达到检查间隔，跳过——这是降本的关键
        if self._counter < self.check_interval:
            return False

        # 到达检查点，重置计数器
        self._counter = 0

        # 将当前话题样本和历史话题样本交给模型做语义判断
        # 模型返回 True（已漂移）或 False（仍在同一话题）
        return self._check_semantic_drift(context_window)

    def _check_semantic_drift(self, context_window: list[dict]) -> bool:
        """
        语义漂移判断——调用模型比较当前话题与历史话题。

        注意：
        - 这不是每轮都执行的昂贵操作
        - 如果模型判断"不确定"（概率在中间地带），保守返回 False（不要强行压缩）
        """
        # 取最近 N 条用户消息作为「当前话题」样本
        recent_msgs = [
            msg for msg in context_window[-3:]
            if msg.get("role") == "user"
        ]
        # 取更早的 N 条用户消息作为「历史话题」样本
        earlier_msgs = [
            msg for msg in context_window[:-3]
            if msg.get("role") == "user"
        ][-3:]

        # 如果没有足够的历史消息，说明对话刚开始，无需压缩
        if len(earlier_msgs) < 2:
            return False

        # 将两组消息交给模型判断（伪代码，实际需调用 LLM）
        # 模型 prompt 示例：
        # "以下两段对话是否在讨论同一个话题？只回答 YES 或 NO。"
        return self._call_model_for_drift_check(recent_msgs, earlier_msgs)

    def _call_model_for_drift_check(
        self, recent: list[dict], earlier: list[dict]
    ) -> bool:
        """
        实际调用模型进行话题漂移判断。

        这里展示的是简化版实现，生产环境中应：
        1. 使用轻量/便宜的模型（不需要最强模型）
        2. 严格控制 prompt 长度和输出格式（YES/NO 二元判断）
        3. 设置 timeout 防止模型调用卡住主流程
        """
        # 构造简洁的二元判断 prompt
        prompt = (
            "比较以下两段对话，判断它们是否在讨论同一个话题：\n\n"
            f"对话A（历史）:\n{earlier}\n\n"
            f"对话B（当前）:\n{recent}\n\n"
            "只回答 YES 或 NO。"
        )
        # response = some_llm_call(prompt)  # 实际项目中替换为真实调用
        # return "NO" in response.upper()
        return False  # 默认保守——不确定就不触发
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 10 | `class TopicDriftTrigger` | 封装话题漂移检测的状态和逻辑 |
| 18 | `self._counter = 0` | 打点计数器，核心降本机制 |
| 27 | `self._counter += 1` | 每轮 tick 递增，不区分角色 |
| 30 | `if self._counter < self.check_interval` | 未到检查点直接跳过，避免每轮调模型 |
| 33 | `self._counter = 0` | 到达检查点后重置，开始新一轮计数 |
| 37 | `return self._check_semantic_drift(...)` | 只在检查点执行昂贵的语义判断 |
| 55 | `if len(earlier_msgs) < 2` | 对话刚开始，历史消息不够，不检查 |
| 79 | `return False` | 默认保守：模型不确定时，不触发压缩 |

### 3.6 组合策略与决策链

四种策略不是互斥的替代关系，而是按成本和优先级组成的**级联决策链**。每一轮对话，系统按以下顺序逐关判断：

```
每一次 Agent 调用完成后
    │
    ├── 第1关: 主动标记？
    │   ├── 是 → 🔴 立即压缩（最高优先级，尊重显式意图）
    │   └── 否 ↓
    │
    ├── 第2关: 长度超限？
    │   ├── 是 → 🔴 立即压缩（硬约束，不压系统就崩了）
    │   └── 否 ↓
    │
    ├── 第3关: 轮次超限？
    │   ├── 是 → 🟡 触发压缩（兜底策略，防止无限堆积）
    │   └── 否 ↓
    │
    └── 第4关: 话题漂移？
        ├── 是 → 🟡 触发压缩（柔性策略，清除失效上下文）
        └── 否 → 🟢 跳过（一切正常，继续对话）
```

**优先级排序的逻辑：**

1. **第1关——主动标记**：成本极低（字符串匹配），且反映用户/开发者的明确意图。无论其他指标如何，显式意图应被优先响应。🔴 最高优先级。

2. **第2关——长度超限**：这是硬约束。不压缩的结果不是"上下文不够好"，而是"API 调用直接失败"。同样成本极低，必须在轮次判断之前就拦截。🔴 硬约束。

3. **第3关——轮次超限**：兜底策略。它不解决"马上要炸"的问题（那是长度触发的事），而是防止上下文无限堆积导致的长期隐患。成本极低，🧵 黄色级别。

4. **第4关——话题漂移**：柔性策略，成本最高。它只在前面三关都没触发时才被检查——因为如果前面已经决定压缩了，再做话题判断没有意义。🧵 黄色级别，且是可选策略（可根据业务场景决定是否启用）。

> 💡 **核心洞见**：这个优先级排序的底层逻辑是——**便宜的先查，硬约束优先于软约束，显式意图优先于隐式推断**。话题漂移放最后不是因为不重要，而是因为它最贵，应该只在"其他便宜策略都没触发"时才调用。

### 3.7 策略选择的业务考量

#### 🧠 直观理解

压缩触发策略没有"通解"。就像你不能用同一套健身计划适用所有人——健身教练要根据学员的体能、目标、时间灵活调整。压缩策略也要根据业务场景的具体需求来定制。

#### 📖 详细解释

在录音中老师反复强调了一个关键观点：**策略和业务场景是强相关的，没有银弹**。

**为什么没有银弹？** 因为不同业务对"什么时候该压缩"的定义完全不同：
- 客服机器人：对话短、话题单一，轮次触发足够，连话题漂移都不需要
- 代码助手：单轮可能包含数百行代码粘贴，长度触发优先，轮次触发作为兜底
- 多任务 Agent：用户在一个窗口中先写文章、再查资料、又改代码，话题漂移必须开启
- 企业内部审批系统：安全合规要求极高，可能需要特殊的审计触发策略

**老师强调的保守原则：**

> ⚠️ **不要强行压缩，语义判断不靠谱就保守**

这是本模块最关键的设计原则。具体含义是：

1. **话题漂移检测不可靠时，宁可跳过**：如果模型的语义判断返回"不确定"或概率在中间地带（如 40%~60%），不要强行认为话题已切换。压缩后的信息不可逆——一旦错误压缩，用户可能失去关键上下文。

2. **阈值不要设得太激进**：safe_margin 设 0.95 看起来"窗口利用率高"，但一旦溢出就是生产事故。设 0.85 虽然"浪费"了 15% 的窗口，但换来的是系统稳定。

3. **先上线简单的，再逐步加入复杂的**：一上来就把四种策略全打开是不明智的。建议先上线轮次 + 长度这两种几乎零成本的策略，观察实际对话数据；再根据数据决定是否引入主动标记和话题漂移。

4. **压缩是可逆的吗？** 不是。压缩后的摘要一旦替换了原文，原始细节就不可恢复。所以触发压缩的决策必须慎重——"该压不压"只是多消耗一些 Token 预算，"不该压却压了"可能丢失关键信息。

#### 🔀 不同业务场景的策略组合建议

| 业务场景 | 建议开启的策略 | 不建议开启的策略 | 原因 |
|---|---|---|---|
| 简单 FAQ 机器人 | 轮次触发 | 话题漂移 | 对话短、话题单一，语义检查浪费成本 |
| 代码助手/AI 编程 | 长度触发 + 轮次触发（兜底） | — | 单轮可能粘贴大量代码，长度更关键 |
| 多任务个人助理 | 全部四种 + 话题漂移 | — | 用户在同一窗口切换任务，需要话题感知 |
| 合规敏感系统 | 轮次 + 长度 + 主动标记 | 话题漂移（自动压缩有风险） | 宁可冗余也不可丢失，手动控制更稳妥 |

> 💡 **核心洞见**：策略与业务强相关——没有银弹。如果语义判断拿不准，保守一点，不要强行压缩。压缩的代价是不可逆的，而"该压不压"只是多用一点预算，孰轻孰重，一目了然。

> 🔗 **延伸阅读**：本章的策略判定逻辑在后续章节中被进一步抽象为"策略判定层"，成为三层管控架构的中间层。详见 [五、Session 级压缩与三层管控架构](#五session-级压缩与三层管控架构)

### 📝 本节要点

| 要点 | 一句话总结 |
|---|---|
| 四种触发策略 | 轮次、长度、主动标记、话题漂移，按成本从低到高排列 |
| 安全边距 | 预留 10%~20% 空间给压缩指令和模型输出，防止溢出 |
| 打点计数器 | 每 N 轮才做一次语义判断，将话题漂移检测成本控制在可接受范围 |
| 决策链优先级 | 便宜的先查 → 硬约束优先 → 显式意图优先 → 柔性策略兜底 |
| 保守原则 | 语义判断不靠谱就别压，压缩不可逆，宁可浪费预算也不要丢失信息 |
| 没有银弹 | 策略选择和参数调优必须结合具体业务场景，先简后繁逐步上线 |

## 四、三层记忆架构详解

### 4.1 为什么需要记忆分层

#### 🧠 直观理解

想象你是一个在帮用户规划旅行路线的助手。在整个对话过程中，你会同时面对三类信息：用户刚刚说的"迪士尼一定要去"（当前约束）、之前几次对话中生成的路线检查报告（任务产出）、以及用户一直以来的"孩子需要午休"（长期偏好）。如果你把这三类信息全部塞进一个不断追加的文本列表里，很快会出现两个致命的矛盾：**不该长期记住的琐碎信息占据了上下文空间，而真正需要长期记住的偏好却被淹没在对话洪流中**。

这就引出了记忆分层的核心问题：**不同信息有不同的生命周期和共享范围**。用一个单一列表管理所有信息，就像用同一个抽屉同时放今天的便签纸、本周的项目文件和一辈子的房产证——既低效又危险。

#### 📖 详细解释

在 AI Agent 的实际运行中，三类信息的生命周期差异非常明显：

| 信息类型 | 典型内容 | 生命周期 | 共享范围 |
|---|---|---|---|
| 当前对话约束 | "帮我查一下今天天气"、"迪士尼一定要去" | 当前 Session 内 | 仅当前 Session |
| 任务中间产物 | 路线检查报告、价格对比表 | 任务期间（可能跨多个 Session） | 当前 Task Workspace |
| 长期偏好与规则 | "孩子需要午休"、"输出前必须附上检查结论" | 永久或长期 | 跨 Session、跨 Task、跨 Workspace |

如果强行用单层列表管理，你会遇到三个具体问题：

1. **上下文污染**：当前对话窗口被大量历史偏好占据，模型注意力分散，回答质量下降
2. **记忆丢失**：真正需要长期保留的偏好被当成普通对话消息，在窗口裁剪时一并丢弃
3. **无法跨 Session 共享**：单层列表天然绑定在当前 Session 内，下一次对话从零开始，用户需要重复表达相同的偏好

> 💡 **核心洞见**：三层之间不是"越上层越高级"，而是**使用范围和生命周期不同**。选哪一层取决于信息的存活时间和共享范围——不该让今天查天气的对话记录影响三年后的旅行规划。

#### 🧠 三层架构全景图

老师用一张层级嵌套的架构图直观展示了三层记忆的空间关系：

### 4.2 Session 短期记忆详解

#### 🧠 直观理解

Session 短期记忆是三层架构中最贴近模型请求的一层。老师用了一个生动的比喻来解释它的三个组成部分：

想象你在看一本很厚的对话记录本。**full_context** 是这本记录本的全部内容——从第一页到现在，一字不落。但你不可能把整本记录本塞进模型的眼睛里（有上下文长度限制），所以你需要选择最近几页投喂给它——这就是 **context_window**，相当于你翻开记录本、透过一个**窗户**只能看到当前的一角。而 **memo** 则是你随身携带的**补充背包**——你把窗户之外的重要内容（比如"迪士尼必去"、"孩子需要午休"）压缩提炼后放进背包里，和窗户里的信息一起交给模型。

#### 📖 详细解释

Session 短期记忆的三个要素各有明确的职责和边界：

```
                    full_context (完整记录本)
    ┌─────────────────────────────────────────────────────────┐
    │  T1  T2  T3  T4  T5  T6  T7  T8  T9  T10  T11        │
    │                                                        │
    │              ┌──────────────────┐                      │
    │              │  context_window  │   ← 窗户一角          │
    │              │  [T9][T10][T11]  │   模型可见            │
    │              └──────────────────┘                      │
    │                                                        │
    │   memo (补充背包):                                      │
    │   ┌──────────────────────────────────────────┐        │
    │   │ {"迪士尼必去", "孩子需要午休",             │        │
    │   │  "远距离项目不和市区景点挤同一天"}         │        │
    │   └──────────────────────────────────────────┘        │
    └─────────────────────────────────────────────────────────┘

    模型请求时的实际注入：
    ┌─────────────────────────────────────────────────────────┐
    │  [memo 作为背包信息前置]                                  │
    │  + [context_window 作为最近对话]                         │
    │  = 本次投喂给模型的有效信息                                │
    └─────────────────────────────────────────────────────────┘
```

三要素的详细对比：

| 概念 | 内容 | 更新时机 | 是否投喂模型 | 用途 |
|---|---|---|---|---|
| **full_context** | 完整对话历史，不裁剪 | 每轮对话后追加 | 否 | 为 memo 压缩提供完整原材料 |
| **context_window** | 当前投放的对话片段（最近 N 轮） | resize 触发后更新 | 是 | 模型当前可见的对话上下文 |
| **memo** | 压缩后的结构化摘要 | resize 时同步生成 | 是（与 context_window 一起） | 补充被窗口裁剪掉的长期关键信息 |

**full_context 为什么重要？**

如果你只保留 context_window 而丢弃完整记录，那么每次窗口滑动时被挤出的信息就永久丢失了。full_context 的价值在于：它是 memo 生成的**完整原材料**。当模型执行 `model_compress` 策略时，它需要看到完整的对话历史才能做出准确的压缩——哪些约束已被回答、哪些仍然开放、哪些是用户反复强调的长期偏好。如果没有 full_context，模型只能基于窗口内的片段做压缩，压缩质量会大打折扣。

**context_window 的"窗户一角"比喻**

context_window 的大小受模型上下文长度和成本限制。它通常只包含最近的 5-10 轮对话。这就好比你只能透过一个固定大小的窗户看外面——窗户外的世界你暂时看不到，但它们并没有消失，而是被记录在 full_context 中。当窗口向前滑动时，新的消息进来，旧的消息退出窗口，退出的关键信息被 memo 接住。

**memo 的"补充背包"比喻**

memo 是压缩背包——它不替代 context_window，而是作为补充并行注入。模型同时看到最近的对话（窗户）和压缩后的历史关键信息（背包）。这两者配合使用，让模型既能保持对当前对话的精确感知，又不丢失历史中的重要约束。

#### 📖 memo 的管理细节与灵活性

memo 的设计哲学是**框架提供钩子，管理逻辑由开发者自行决定**。具体体现在以下几点：

1. **memo 类型为 `any`**：可以是字符串、字典、列表或任意自定义结构。框架不做格式约束，完全由开发者在 `resize_handler` 中管理。

2. **格式一致性由开发者自行保证**：如果你的第一次 resize 将 memo 设为字典 `{"constraints": [...]}`，第二次 resize 时就不应该把它当列表读取——否则会直接报错。这是一个常见的坑。

3. **增量更新 vs 全量覆盖**：memo 的更新策略完全由你决定。你可以选择每次 resize 时全量替换 memo（适合 memo 内容完全重算的场景），也可以选择增量合并（适合逐步积累约束的场景）。

4. **memo 背后可以挂向量库**：有同学问能不能给 memo 加向量检索能力？完全可以。但你需要自行管理向量库的读写逻辑。resize 函数只是一个钩子——它调用你注册的处理函数，你在函数内部怎么存储、怎么检索，框架不干涉。

```python
# memo 管理的三种典型形态

# 形态1：纯文本字符串
memo = "迪士尼必去。孩子需要午休。远距离项目不要和市区景点挤在同一天。"

# 形态2：结构化字典
memo = {
    "confirmed_constraints": [
        {"text": "迪士尼必去", "source_turn": 0},
        {"text": "孩子需要午休", "source_turn": 4},
    ],
    "open_questions": [
        {"text": "住哪个区还没定", "source_turn": 2},
    ],
}

# 形态3：列表形式
memo = [
    {"type": "constraint", "content": "迪士尼必去", "turn": 0},
    {"type": "constraint", "content": "孩子需要午休", "turn": 4},
    {"type": "question", "content": "住哪个区还没定", "turn": 2},
]
```

> ⚠️ **注意事项**：memo 格式一旦选定，所有 resize 处理函数都必须遵守同一格式——这是你作为开发者需要自行保证的契约。框架不校验 memo 的数据结构，不正确的格式会在运行时以难以排查的方式出错。

> ❓ **常见疑问**：同一个 Session 下 memo 会不会合并？
>
> **会的。** memo 的合并行为取决于你注册的 `resize_handler` 的实现。你可以在 handler 中设计增量合并逻辑（将新的约束追加到已有 memo 中），也可以设计全量覆盖逻辑（每次压缩重新生成 memo）。这是你作为开发者自行控制的策略选择。

### 4.3 Task Workspace 任务记忆

#### 🧠 直观理解

如果说 Session 短期记忆是你当下对话的"工作台面"，那么 Task Workspace 任务记忆就是你的"文件柜"。工作台面上只有最近在用的一小部分信息（窗户里的对话 + 背包里的 memo），但文件柜里存放着整个任务推进过程中产生的所有中间产物——路线检查报告、价格对比表、住宿筛选结果。你不需要一次性把所有文件摊在台面上，而是**按需打开**——这就是渐进式披露。

#### 📖 详细解释

Workspace 对应本地的一个实际目录。在 Agent 执行任务的过程中，所有中间产物以文件形式落盘到这个目录中。关键的设计在于：**模型不会自动读取 Workspace 中的所有文件**。它看到的只是一个 ref 引用索引——一个轻量级的目录摘要，告诉模型"这里有些什么文件"。模型根据当前需要，决定要深入查看哪个文件——这就触发了渐进式披露。

**冷热存储的物理模型：**

```
TASK WORKSPACE (本地目录: /workspace/shanghai-family-trip-7/)
│
├── ref 引用索引 (热存储，始终在 Agent 视野内)
│   ├── 📄 route-check-v1.txt  →  "迪士尼和自然博物馆同日通勤过长"
│   ├── 📄 hotel-compare.txt   →  "浦东亲子酒店价格对比"
│   ├── 📄 itinerary-draft.md  →  "三日行程草案"
│   └── 📄 price-table.csv     →  "各景点门票汇总"
│
└── 实际文件 (冷存储，按需加载)
    ├── route-check-v1.txt     ← 只在需要时打开
    │   └── 子引用（如文件很大）
    │       ├── 第1段：检查方法
    │       ├── 第2段：具体结论   ← 可能只需要这一段
    │       └── 第3段：建议
    │
    ├── hotel-compare.txt      ← 只在需要时打开
    └── ...
```

**渐进式披露的执行流程：**

```
模型收到请求："帮我检查一下路线有没有问题"
    │
    ▼
模型查看 ref 索引 → 发现 "route-check-v1.txt: 迪士尼和自然博物馆同日通勤过长"
    │
    ▼
模型判断：需要更多细节 → 请求打开 route-check-v1.txt
    │
    ▼
模型读取文件内容 → 发现文件很大，有多个段落
    │
    ▼
模型判断：只需要第2段"具体结论" → 按需加载该段落
    │
    ▼
回到主流程 → 基于读取的信息生成回复
```

> ⚠️ **关键前提**：渐进式披露需要 Agent 具备 ReAct Loop 能力（思考→行动→观察→再思考）。如果你用的是最简方案——纯 prompt 拼接、单次模型请求——Workspace 根本不会被触发。因为单次模型请求下，Agent 没有"回来看文件"的机会。你必须显式告诉 Agent "你有一个 Workspace"，它才会在 ReAct Loop 中主动查询。

**最简方案 vs Workspace 方案的能力边界差异：**

| 维度 | 最简方案（纯 prompt 拼接） | Workspace 方案 |
|---|---|---|
| 信息来源 | 仅 prompt input 传入的信息 | prompt input + Workspace 中的文件 |
| 执行模式 | 单次模型请求 | ReAct Loop（多步推理） |
| 能否读取中间产物 | 否 | 是（渐进式披露） |
| 跨 Session 共享 | 否 | 是（同一 Workspace 目录） |
| 适用场景 | 简单问答、单轮信息查询 | 复杂多步任务、需要文件操作的场景 |

**跨 Session 的 Workspace 共享：**

Workspace 的一个重要特性是它**不绑定单个 Session**。同一个 Workspace 目录可以在多个 Session 中共享。只要 ref 引用信息被正确提供，下一个 Session 可以基于前一个 Session 落盘的文件继续工作——这实现了任务的跨 Session 延续。

```
Session-1                     Session-2
    │                             │
    │  执行路线检查                 │  "上次的检查报告在哪？"
    │  落盘: route-check-v1.txt    │
    │                             │  查看 ref 索引
    │                             │  找到 route-check-v1.txt
    │                             │  渐进式披露读取
    │                             │  基于报告继续工作
    ▼                             ▼
        同一个 Workspace 目录
        /workspace/shanghai-family-trip-7/
```

### 4.4 Namespace 长期记忆

#### 🧠 直观理解

Namespace 长期记忆是三层架构中共享范围最广、生命周期最长的一层。它回答了一个关键问题：如果用户换了新电脑、开了新对话、甚至换了一个完全不同的任务类型，Agent 还能不能记得"这个人喜欢什么"？

答案是能——通过 Namespace。如果你用过 Claude Code 或 Codex，你会注意到项目根目录下有一个 `CLAUDE.md` 或 `agents.md` 文件。这些文件就是 Namespace 长期记忆的典型形态——它们是跨 Session、跨 Task、跨 Workspace 的全局信息，定义了 Agent 的行为规则和用户的长期偏好。

#### 📖 详细解释

Namespace 的设计核心是**写入管控最严格**。原因很简单：Session 记错了东西影响一次对话，Workspace 记错了东西影响一个任务，但 Namespace 记错了东西会影响用户所有的后续交互——破坏面最大。

因此 Namespace 的写入需要满足三个条件：

| 条件 | 说明 | 反面例子 |
|---|---|---|
| **已确认** | 经过多次交互验证，或经人工确认 | 用户说了一次"试一下方案A"就被记成长期偏好 |
| **去重合并** | 同一条偏好多处出现时应合并为一条 | 同一个"孩子需要午休"在三次对话中记录了三条重复记录 |
| **带证据链** | 每条 claim 绑定 evidence_refs，可追溯来源 | 没有来源标注的记忆无法验证真伪 |

**Namespace 的两种典型形态：**

```
形态1：用户长期偏好（跨任务共享）
┌─────────────────────────────────────────────┐
│ Namespace: user-42:family-travel            │
│                                             │
│ ✅ "孩子需要午休，下午2-4点不能安排行程"      │
│    证据: evt_a05 (第3次对话), evt_b01 (第7次) │
│                                             │
│ ✅ "偏好亲子酒店，预算每晚800以内"            │
│    证据: evt_a02 (第1次对话), evt_c03 (第5次) │
│                                             │
│ ❌ 不记录: "这次想试试民宿"（仅一次提及）      │
└─────────────────────────────────────────────┘

形态2：行为规范规则（类似 CLAUDE.md）
┌─────────────────────────────────────────────┐
│ Namespace: agent-rules:family-travel        │
│                                             │
│ 📋 "发布最终行程前必须附上路线检查结论"       │
│ 📋 "涉及儿童的活动需标注适龄范围和注意事项"    │
│ 📋 "价格信息需附带货币单位和查询时间戳"        │
└─────────────────────────────────────────────┘
```

> 🔗 **关联知识**：行为规范规则的管理可以有两种方式——你可以把它们放在 Namespace 长期记忆中作为记忆记录管理，也可以把它们作为独立的 `rules.md` 文件放在 Workspace 中。两种方式的选择取决于：规则是否需要跨 Workspace 共享？是的话放 Namespace，否则放 Workspace 即可。

### 4.5 两层治理对象

除了三层记忆架构本身，Agent 还需要管理两类与"行为治理"相关的对象。它们本质上也是"记忆"——只不过记录的不是事实，而是规则和经验。

#### 📖 Rule record（已发布规则）vs Reflection candidate（待验证候选）

| 治理对象 | 含义 | 默认状态 | 如何生效 | 写入方式 |
|---|---|---|---|---|
| **Rule record** | 已发布的政策、阈值、禁用项 | `active` / `versioned` | scope + 版本 + 有效期匹配后自动注入 | 自上而下发布（人工设定）或自下而上晋升（candidate 验证通过后） |
| **Reflection candidate** | 模型从失败中提出的经验/规则建议 | `pending` | 默认不进入请求，需验证确认后方可生效 | 自下而上：模型在任务执行中发现失败模式，提出改进建议 |

**Rule record 和 Reflection candidate 的关系：**

```
              人工发布（自上而下）
                    │
                    ▼
            ┌───────────────┐
            │  Rule record   │  ← active, 自动注入到请求中
            │  (已发布规则)   │
            └───────┬───────┘
                    ▲
                    │ 验证通过后晋升
                    │
            ┌───────┴───────┐
            │  Reflection    │  ← pending, 默认不注入
            │  candidate     │
            │  (待验证候选)   │
            └───────────────┘
                    ▲
                    │ 模型从失败中学习
                    │
            ┌───────────────┐
            │  任务执行失败   │
            │  或发现改进点   │
            └───────────────┘
```

> 💡 **核心洞见**：Rule record 可以自上而下发布（比如团队规范），也可以由 candidate 经确认后自下而上晋升（比如模型发现"把远距离大项拆到不同日期"是一条可复用的经验，经人工验证后升级为规则）。candidate 在确认前甚至不算正式记忆——它是一种"待审批的经验提案"。这套机制实现了 Agent 的**自成长闭环**：执行 → 失败 → 反思 → 候选 → 验证 → 晋升为规则。

### 4.6 MemoryRecord 数据类详解

为了统一管理三层记忆和两类治理对象，我们设计了一个 `MemoryRecord` 数据类。它用五个 `layer` 类型承载所有记忆/治理记录，用统一的字段结构消除类型差异带来的管理复杂度。

#### 💻 完整代码

```python
# 文件名: scripts/s01_memory_layers.py
# 功能: 统一的 MemoryRecord 承载五种记忆/治理类型

from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Any

@dataclass
class MemoryRecord:
    id: str                          # 唯一标识
    layer: str                       # 所属层: session_short_term / task_workspace /
                                     #   namespace_long_term / rule_record / reflection_candidate
    collection: str                  # 存储集合名
    kind: str                        # 类型细分: session_memo / evidence / long_term_memory
    scope: dict[str, str]            # 可见范围: tenant_id + namespace
    summary: str                     # 一句话摘要
    body: dict[str, Any]             # 完整信息体
    vector: list[float] | None = None  # embedding 向量（可选）
    priority: int = 50               # 排序优先级（默认 50）
    created_at: str = field(         # 创建时间戳
        default_factory=lambda: datetime.now(timezone.utc).isoformat()
    )
```

#### 📖 五种 layer 类型详解

| layer 类型 | 对应架构层 | kind 典型值 | 生命周期 | 典型 priority |
|---|---|---|---|---|
| `session_short_term` | Session 短期记忆 | `session_memo` | 当前 Session | 60 |
| `task_workspace` | Task Workspace 任务记忆 | `evidence` | 任务期间 | 55 |
| `namespace_long_term` | Namespace 长期记忆 | `long_term_memory` | 长期/永久 | 40 |
| `rule_record` | 治理对象：已发布规则 | `rule_record` | 版本化管理 | 95 |
| `reflection_candidate` | 治理对象：待验证候选 | `reflection_candidate` | 待验证 | 20 |

#### 📖 关键字段设计说明

| 字段 | 类型 | 说明 | 设计意图 |
|---|---|---|---|
| `id` | `str` | 全局唯一标识 | 支持跨系统引用和去重 |
| `layer` | `str` | 五种 layer 之一 | 决定记忆的存储位置和生命周期策略 |
| `collection` | `str` | 存储集合名（如 `"sessions"`, `"memory"`, `"rules"`） | 物理存储分组 |
| `kind` | `str` | 类型细分 | 同层内不同类型记忆的检索区分 |
| `scope` | `dict[str, str]` | 可见范围控制 | **最关键字段之一**：决定谁能看到这条记忆。`{"tenant_id": "a", "session_id": "s-001"}` 表示仅 Session s-001 可见；`{"tenant_id": "a", "namespace": "user-42:family"}` 表示该 Namespace 下所有 Session 可见 |
| `summary` | `str` | 一句话摘要 | 用于列表展示和快速匹配，不承载完整信息 |
| `body` | `dict[str, Any]` | 完整信息体 | 承载实际的记忆内容、证据链、版本号等 |
| `vector` | `list[float] \| None` | embedding 向量 | 可选，用于语义检索。不强制要求每条记忆都有向量 |
| `priority` | `int` | 排序优先级 | 决定在上下文窗口有限时，哪些记忆优先注入。规则类最高（95），候选经验最低（20） |
| `created_at` | `str` | ISO 格式创建时间戳 | 支持时间衰减策略 |

#### 📖 scope 的边界控制机制

`scope` 字典是 MemoryRecord 中最重要的设计之一。它通过键值对实现多级边界控制：

```python
# Session 级：仅当前 Session 可见
scope={"tenant_id": "tenant-a", "session_id": "s-001"}

# Task 级：同一 Task 的所有 Session 可见
scope={"tenant_id": "tenant-a", "task_id": "shanghai-family-trip-7"}

# Namespace 级：跨 Task 的偏好信息
scope={"tenant_id": "tenant-a", "namespace": "user-42:family-travel"}

# 规则级：按任务类型生效
scope={"tenant_id": "tenant-a", "task_type": "family_travel"}

# 全局级：所有租户可见（谨慎使用）
scope={"tenant_id": "*"}
```

> 🔗 **前置知识**：scope 的设计继承了第 18 课中 Session 的多级 scope 概念。Session 的 scope 决定了"这个会话属于谁"，MemoryRecord 的 scope 则进一步决定了"这条记忆能被谁看到"。

#### 📖 priority 优先级设计逻辑

| priority 区间 | 适用类型 | 设计原因 |
|---|---|---|
| 90-100 | `rule_record` | 规则必须优先注入，否则行为失控 |
| 55-65 | `session_short_term` (session_memo) | 当前会话的压缩信息，时效性最高 |
| 50-60 | `task_workspace` (evidence) | 任务证据，与当前任务强相关 |
| 35-45 | `namespace_long_term` | 长期偏好，稳定但不需要抢占窗口 |
| 15-25 | `reflection_candidate` | 待验证候选，最低优先级，默认不注入 |

> ⚠️ **注意事项**：priority 不是越高越好。priority=95 的 rule 如果写得太长，会挤占模型处理当前任务的注意力。优先级设计的原则是：**时效性高 + 确定性高 = 高优先级，时效性低 + 待验证 = 低优先级**。

### 4.7 LangGraph 对比与市场方案分析

#### 📖 LangGraph 的记忆管理模型

LangGraph 作为 LangChain 生态的底层运行时（runtime）框架，对记忆的立场与 Agentic Session 完全不同。它以 **graph runtime** 为中心，将记忆视为 graph 执行过程中的状态数据：

- **短期记忆**：放在 agent state 中，由 **checkpointer** 按 thread 持久化。checkpointer 的核心功能是在 graph 执行过程中保存运行快照，使中断后的恢复成为可能。你可以认为它提供了一个"运行时的缓存 + 持久化"能力。
- **长期记忆**：通过 **Store + custom namespace** 实现，支持 semantic search。Store 更接近于一个带命名空间的键值数据库。

LangGraph 支持两个关键的工程判断——短期与长期需要不同的 scope，执行快照和跨会话知识需要不同的接口——但它始终是**底层基础设施**。它不给出企业应用层的候选审批、规则生效或技能发布流程。记忆治理的业务逻辑（比如"候选经验需要经过几次验证才能晋升为规则"）需要应用层自行实现。

#### 📖 三方对比表

| 对比维度 | LangGraph | Agently Session / Workspace | TriggerFlow（工作流引擎） |
|---|---|---|---|
| 设计中心 | 长运行、状态化 graph | 请求侧会话连续性 | 工作流执行恢复 |
| 记忆定位 | 记忆 = graph state + store | 记忆 = Session 生命周期内的结构化信息 | 记忆 = 流程节点间的数据传递 |
| 当前对话 | state + checkpointer 按 thread 持久化 | Session 原生保存 full_context + context_window + memo | 无专用对话记忆语义 |
| 长期信息 | Store + namespace，应用层自行管理 | MemoryRecord + Namespace scope 边界控制 | 调用外部存储（如数据库），无内置记忆层级 |
| 压缩扩展点 | 自实现 trim/summarize（不内置） | analysis_handler + resize_handler 钩子 | 显式工作流步骤（需要在流程中手动插入压缩节点） |
| 候选治理 | 应用自行设计（LangGraph 不提供） | candidate 状态管理 + 验证→晋升流程 | 可承载审批流程（工作流天然适合审批节点） |
| 渐进式披露 | 不内置，需自行实现 tool-use 模式 | Workspace ref 引用 + 目录→文件→段落逐层加载 | 不内置 |
| 适用场景 | 需要精确控制 graph 执行流的复杂 Agent 系统 | 以对话为核心的 Agent 应用，需要开箱即用的记忆管理 | 结构化审批流程、多步骤业务编排 |

> 💡 **核心洞见**：LangGraph 的哲学是"我给你存储接口，业务逻辑你自己做"。Agently Session 的哲学是"大部分 Agent 应用都需要在对话推进时触发记忆管理，我帮你把这个触发逻辑封装好，你只需要写处理函数"。TriggerFlow 的哲学是"记忆只是流程中的一个节点，你需要显式地在流程中插入读写步骤"。三者没有绝对的优劣——选择取决于你是否需要一个**以对话推进为触发主轴的记忆自动化管理**。

#### 📖 记忆 ≠ Session 强绑定

一个常见的误解是：记忆管理一定和 Session 的生命周期绑定。老师重点纠正了这一点：

```
误解：Session 结束时记忆就失效了
真相：记忆是外部存储，Session 只是触发记忆管理的最常见时机

记忆的触发时机可以是：
├── Session 相关触发（最常见）
│   ├── 每条消息变更 → 触发 resize 检查
│   ├── Session 结束 → 触发最终压缩
│   └── Session 跨窗口滑动 → 触发 memo 更新
│
└── 非 Session 触发（同样有效）
    ├── 定时批量整理 → 每晚 2 点对所有候选经验做批量评估
    ├── 跨 Session 偏好沉淀 → 同一用户的三个独立 Session 中发现了重复偏好
    ├── 人工触发 → 管理员手动发布新规则
    └── 事件驱动 → 任务关闭时触发经验总结
```

Session 通过钩子（`analysis_handler`、`resize_handler`）提供了**最常见的触发时机**，但记忆与 Session 是**松耦合**关系。你可以完全脱离 Session 来管理记忆——只要有办法把记忆内容注入到下一次模型请求的上下文中。

#### 📖 记忆 ≠ RAG

另一个需要纠正的误解是：记忆就是 RAG（检索增强生成）。

```
误解：记忆 = 向量库 + 语义检索 + 拼接到 prompt
真相：RAG 是记忆的一种实现方式，但不是全部

记忆的实现方式包括但不限于：
├── 向量检索（RAG 方式）
│   └── 适用范围：大规模、需要语义匹配的记忆
│
├── 结构化注入（规则/偏好）
│   └── 适用范围：确定性规则、已知 scope 的偏好
│
├── 渐进式披露（Workspace 文件）
│   └── 适用范围：任务中间产物、大文件分段读取
│
├── Skills 技能记忆
│   └── 通过声明式注册 + 渐进式披露被模型使用
│   └── 示例：一个"路线检查"技能放在 Workspace 中
│   └── Agent 在执行任务时自动发现并使用它
│
└── 上下文窗口直接注入（memo + context_window）
    └── 适用范围：当前 Session 的压缩信息
```

Skills 就是一种"技能级记忆"——它被放在 Workspace 中，通过明确的声明（如"执行旅行规划任务时必须使用路线检查技能"）被 Agent 发现和使用。它不依赖向量检索，而是依赖 Workspace 的渐进式披露机制。这证明了记忆的实现方式远不止 RAG 一种。

### 4.8 记忆的本质

#### 🧠 直观理解

所有关于"分层"、"压缩"、"治理"的讨论，最终都指向同一个核心事实：**大语言模型本身没有记忆**。模型每次调用都是一个无状态的函数——输入一段文本，输出一段文本。它不会"记得"上一轮对话说了什么，不会"记得"用户的偏好，也不会"记得"之前犯过的错误。

那我们感受到的"记忆"是从哪来的？就是把外部存储的信息，在模型请求时找出来，拼到输入文本里——**伪装出来的**。

#### 📖 推导链：记忆的完整生命周期

```
外部存储中有一条信息："用户的孩子需要午休，下午2-4点不能安排行程"
    │
    ├── 第1步：何时存储？
    │   为什么？用户在多次对话中反复表达了这一偏好，经过确认和去重后
    │   写入 Namespace 长期记忆，绑定 evidence_refs
    │
    ├── 第2步：如何召回？
    │   为什么？当用户发起新的旅行规划请求时，系统根据 scope 匹配
    │   找到这条偏好。如果是语义检索还需计算向量相似度
    │
    ├── 第3步：如何注入？
    │   为什么？将召回的记忆信息拼接到 context_window 之前，作为
    │   system prompt 或前置用户消息注入。格式可能为 JSON 结构化数据
    │   或自然语言描述，取决于模型对该格式的处理能力
    │
    └── 结论：模型看到了"孩子需要午休"这个信息，在行程规划中主动
        为空出了 2-4pm 的时间段。用户感受到"Agent 记得我的偏好"
```

> 💡 **核心洞见**：**记忆的本质 = 外部存储 + 按需召回 + 上下文注入**。三层架构的本质是对"外部存储"做分层管理（不同生命周期放在不同层），记忆治理的本质是对"按需召回"做策略优化（哪些该召回、哪些不该召回），上下文注入的本质是对"注入格式和时机"做工程优化（怎么拼、拼多少、拼在哪）。

#### 📖 记忆管理的完整公式

```
记忆管理的全流程：

Step 1: 写入
  信息产生 → 判断生命周期（属于哪一层？）→ 格式化存储（MemoryRecord）
  → 写入对应 collection（sessions / workspace / memory / rules / candidates）

Step 2: 召回
  新的模型请求到来 → 触发记忆检索 →
  ├── Session 层：直接读取当前 Session 的 memo + full_context
  ├── Workspace 层：提供 ref 索引，由 Agent 自行渐进式披露
  ├── Namespace 层：scope 匹配 + 可选语义检索
  ├── Rule 层：scope 匹配 + 版本匹配 + 有效期检查
  └── Candidate 层：默认不召回（除非显式触发验证流程）

Step 3: 注入
  召回的记忆 → 按 priority 排序 → 拼接到 context_window 之前 →
  控制总 token 数不超过模型限制 → 投喂给模型

Step 4: 更新
  模型生成回复 → 新一轮对话 → 触发 resize 检查 →
  ├── 是否需要压缩？（窗口满了？）
  ├── 是否需要更新 memo？（新增了约束？）
  ├── 是否需要沉淀长期偏好？（多次确认的偏好？）
  └── 是否需要生成候选经验？（任务失败？发现了可复用模式？）
```

> ⚠️ **注意事项**：至此，你应该对"记忆"这个概念有了一个脱胎换骨的理解——它不是模型的某种内置能力，而是一套运行在模型外部的、完整的读写管线的上层抽象。理解了这一点，你就能明白为什么"记忆分层"不是锦上添花，而是**让 Agent 从单次问答工具进化为可成长的智能体的基础设施**。

### 4.9 三层记忆架构的决策指南

#### 🤔 信息放哪一层？决策树

```
遇到一条需要记录的信息时，问自己三个问题：

信息需要存多久？
    │
    ├── 仅当前对话 → Session 短期记忆
    │   └── 再判断：是完整记录还是压缩摘要？
    │       ├── 完整记录 → full_context
    │       ├── 模型需要现在看到 → context_window
    │       └── 窗口外的关键信息 → memo
    │
    ├── 任务期间（可能跨多次对话）→ Task Workspace 任务记忆
    │   └── 再判断：需要模型主动读取还是被动引用？
    │       ├── 模型需要主动查询 → ref 索引 + 渐进式披露
    │       └── 仅作为存档 → 直接落盘为文件
    │
    └── 长期或永久 → 需要设定共享范围
        │
        ├── 同任务类型内共享 → Namespace 长期记忆（task_type scope）
        ├── 用户级全局共享 → Namespace 长期记忆（user namespace）
        └── 所有场景都必须遵守 → Rule record（tenant 级或全局 scope）

同时判断：这是事实还是规则/经验？
    │
    ├── 事实信息 → 三层记忆架构中的某一层
    └── 规则或经验 →
        ├── 已确认 → Rule record（active, priority=95）
        └── 待验证 → Reflection candidate（pending, priority=20）
```

### 4.10 常见疑问与纠正

> ❓ **疑问1**：Session 短期记忆里的 memo 是不是比 full_context 更"高级"？
>
> **不是。** memo 和 full_context 解决的是不同问题：full_context 是完整原材料，供 memo 压缩使用；memo 是压缩后的结构化管理信息，供模型请求注入使用。它们不是替代关系，而是**协同关系**——没有 full_context，memo 生成没有材料；没有 memo，窗口外的历史信息无法传递给模型。

> ❓ **疑问2**：Workspace 是不是就是 Session 的"文件版"？
>
> **不是。** 关键区别在于：Session 的信息是**每次请求自动注入**的（memo + context_window 拼到 prompt 里），Workspace 的信息是**模型主动按需读取**的（渐进式披露）。Session 是"推送"模式，Workspace 是"拉取"模式。这也解释了为什么 Workspace 需要 ReAct Loop——模型必须有"主动发起读取"的能力。

> ❓ **疑问3**：Rule record 和 Namespace 长期记忆有什么区别？不都是长期有效的吗？
>
> **关键区别在于注入方式**：Namespace 长期记忆是"背景信息"——告诉模型"用户有这个偏好"，模型可以考虑采纳也可以灵活处理。Rule record 是"硬约束"——"发布最终行程前必须附上路线检查结论"，模型没有商量的余地。所以 Rule record 的 priority 最高（95），而长期偏好的 priority 较低（40）。

> ❓ **常见误解**：记忆管理 = RAG
>
> **纠正**：RAG 是记忆管理的**一种实现方式**，主要用于大规模记忆的语义检索场景。但记忆管理还包括：结构化 scope 匹配、渐进式披露、规则版本管控、Skills 注册、候选验证流程等。把记忆管理等同于 RAG，会忽略掉大量非检索式的记忆使用模式。详见 4.7 节的完整辨析。

### 5.1 压缩的本质：从数据到记忆的四步流程

#### 🧠 直观理解

Session 级压缩的本质很简单：**从一段冗长的对话历史中，提取出对未来对话有价值的关键信息，丢弃已经失效的细节**。就像你开完一个小时的会议后，不会把每一句话都记住，而是在笔记本上写下三条决议和两个待办事项——这就是压缩。

#### 📖 详细解释

在 Agently 的 Session 框架中，压缩不是随意的文本摘要，而是一个有明确输入和输出的**模型请求任务**。整个压缩过程可以拆解为四个步骤，每一步都有明确的输入输出和设计考量：

**第一步：拿到数据来源。** 压缩需要有东西可压缩。数据来源是 Session 当前上下文窗口 `context_window` 中的消息列表，每条消息包含 role（user/assistant）、content（消息文本）和它在窗口中的位置信息。

**第二步：给出压缩策略。** 光有数据不够，还需要告诉模型"怎么压缩"。压缩策略包含三部分：一是告诉模型这段对话的上下文（通过构造结构化的模型输入），二是给出压缩指令（instruct），三是定义输出格式（output schema）。后面会详细展开，策略的灵活注入正是三层管控架构的核心。

**第三步：模型执行压缩。** 将构造好的模型输入和压缩指令发给大模型（如 DeepSeek），让模型完成语义理解、信息筛选和结构化输出。这一步是一次标准的模型调用——这恰好呼应了老师反复强调的观点：**只要能控制好单次模型请求的输入和输出，就能在更大的执行架子中灵活嵌入业务表达**。

**第四步：存到目标位置。** 模型返回压缩结果后，需要决定存在哪里、怎么存。在 Session 框架中，压缩结果通常存入 `memo` 字段，同时被压缩过的原始消息可以从 `context_window` 中裁剪掉——两个动作同时完成，既浓缩了信息、又释放了上下文空间。

```
                 压缩本质：四步流程
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
    ▼                   ▼                   ▼
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  第一步   │     │   第二步      │     │   第三步      │
│ 拿到数据  │────▶│ 给出压缩策略  │────▶│ 模型执行压缩  │
│ 来源     │     │              │     │              │
│          │     │ 构造模型输入  │     │ DeepSeek/    │
│ context_ │     │ +instruct    │     │ GPT 模型调用 │
│ window   │     │ +output      │     │ 语义浓缩     │
│ 消息列表 │     │ schema       │     │              │
└──────────┘     └──────────────┘     └──────┬───────┘
                                             │
                                             │ 压缩结果
                                             ▼
                                    ┌──────────────┐
                                    │   第四步      │
                                    │ 存到目标位置  │
                                    │              │
                                    │ memo 字段    │
                                    │ + 裁剪后的   │
                                    │ context_     │
                                    │ window[-3:]  │
                                    └──────────────┘
```

> 💡 **核心洞见**：压缩不是一个独立的算法，而是一个**以模型为核心的业务流程**。模型是压缩的执行引擎，策略是压缩的说明书，Session 框架是压缩的调度器。理解这一点，你就不会把压缩局限在"写一个摘要函数"的思维里——你可以灵活替换模型、调整策略、甚至让业务人员来写压缩指令，这正是后面三层管控架构要做的事。

### 5.2 三层管控架构：让开发与业务各司其职

#### 🧠 直观理解

想象一家公司的组织结构：老板决定"什么时候开会"（策略判定），项目经理决定"开会用什么流程"（压缩策略组成），业务专家决定"会上讨论什么议题"（压缩指令指导）。三层互不越界，但协同运转——这就是三层管控架构。

#### 📖 详细解释

在 Session 压缩的完整链路中，老师识别出了三个可以被解耦的决策层。每一层有不同的维护者、不同的变更频率、不同的抽象级别。将它们分层之后，业务人员可以独立修改"压缩什么内容"而不需要碰代码，开发人员可以独立升级"如何执行压缩"而不影响业务逻辑。

**第一层：策略判定层（开发维护）。** 这一层负责回答"什么情况下触发压缩？触发哪种压缩策略？"。在代码中体现为 `register_analysis_handler` 注册的函数，它的输入是 Session 当前状态（full_context, context_window, memo, settings），输出是一个压缩策略名（如 `"model_compress"`）或 `None`（不触发压缩）。判定条件可以是条目数量（如 ≥ 7 条）、累计 token 数、时间间隔等——纯逻辑判断，与具体压缩内容无关。

**第二层：压缩策略组成层（开发维护）。** 这一层负责回答"压缩策略具体怎么执行？"。在代码中体现为 `register_resize_handler` 注册的处理函数。它决定模型输入的构造方式（是否拼接 turn、role、content）、是否注入外部 skill、是否需要预处理消息、如何解析模型返回结果、如何将结果写回 Session。开发人员可以在此层做信息补充、格式校验、默认值回填等工作。

**第三层：压缩指令指导层（业务维护）。** 这一层负责回答"压缩的时候，模型应该关注什么？输出什么格式？"。在代码中体现为 `instruct([...])` 中的提示词内容，以及 `output({...})` 中定义的结构化 schema。这一层是纯粹的业务表达——旅行助手场景下压缩"已确认约束和待确认问题"，拟人陪聊场景下压缩"用户偏好和聊天习惯"。业务人员可以独立修改这些提示词，甚至使用外部的 yaml 配置或 skill 文件来管理。

```
┌──────────────────────────────────────────────────────────────────┐
│                    策略判定层（开发维护）                           │
│                                                                  │
│  choose_compression(full_context, context_window, memo, settings) │
│                                                                  │
│  输入: Session 当前状态快照                                        │
│  输出: 压缩策略名（如 "model_compress"）或 None（不触发）           │
│                                                                  │
│  典型实现: return "model_compress" if len(context_window) >= 7    │
│           else None                                               │
│                                                                  │
│  可判定条件: 条目数 / token 数 / 时间间隔 / 业务状态               │
└──────────────────────────────┬───────────────────────────────────┘
                               │ 返回策略名 "model_compress"
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│                  压缩策略组成层（开发维护）                          │
│                                                                  │

│                         settings)                                 │
│                                                                  │
│  职责:                                                            │
│  ├── 构造模型输入（turn + role + content 三元组）                  │
│  ├── 注入压缩指令（从配置/外部文件读取 prompt）                     │

│  ├── 调用模型执行压缩                                              │
│  └── 处理返回值，写回 Session                                      │
│                                                                  │
│  register_resize_handler("model_compress", compress_with_deepseek)│
└──────────────────────────────┬───────────────────────────────────┘
                               │ 注入 instruct + output schema
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│                  压缩指令指导层（业务维护）                          │
│                                                                  │
│  instruct([                                                       │
│      "把 Session 当前窗口压缩成任务 memo。",                       │
│      "只保留用户明确提供的约束和仍需回答的问题，不推导长期偏好。",   │

│                                                                  │
│  output({                                                         │
│      "confirmed_constraints": [...],                              │

│  })                                                               │
│                                                                  │
│  业务人员可维护的内容:                                              │
│  ├── 压缩的目标和侧重点（instruct 中的自然语言描述）                │
│  ├── 输出字段的名称和含义                                          │
│  ├── 每条字段的约束条件（not_null、类型等）                         │
│  └── 外部配置文件: compress_prompt.yaml / compress_skill.md       │
└──────────────────────────────────────────────────────────────────┘
```

**三层责任分离表：**

| 层级 | 维护者 | 核心职责 | 可变更内容 | 不可涉及 |
|---|---|---|---|---|
| 策略判定层 | 开发人员 | 决定何时触发、触发哪种策略 | 触发阈值（条目数/token数）、策略路由逻辑 | 压缩的具体内容、输出格式 |
| 压缩策略组成层 | 开发人员 | 组装模型输入、注入指令、处理返回值 | 模型输入构造方式、skill 注入逻辑、返回值解析 | 压缩的业务语义、输出字段含义 |
| 压缩指令指导层 | 业务人员 | 定义压缩目标和输出格式 | instruct 提示词、output schema 字段、外部 prompt 文件 | 模型调用代码、Session 状态管理 |

> 💡 **核心洞见**：三层管控架构的意义不在于技术复杂度，而在于**责任分离**。这和我们在 MCP、ACP 等大型架构设计中看到的模式一脉相承——当系统规模变大，让正确的人维护正确的层，比追求技术上的精巧更重要。业务人员最擅长写 prompt 和 skill，那就让他们维护压缩指令指导层；开发人员最擅长控制模型请求的输入输出，那就让他们维护策略判定层和策略组成层。三层各司其职，系统才能灵活演化。

> 🔗 **关联知识**：这种分层思想在第 3 课中就已经出现——当时我们讲配置化和模板化，把 prompt 从代码中抽离出来放入外部文件。三层管控架构正是这种思想的自然延伸：不仅 prompt 可以外部化，整个压缩策略的决策链路都可以分层解耦。**第三课的知识到现在依然适用**。

### 5.3 代码实操：Session 压缩完整实现

#### 🏗️ 前置准备

在开始编写 Session 压缩代码之前，确保以下条件已满足：

1. **环境依赖**：已安装 `agently>=4.1.4.1`，可通过 `pip show agently` 确认版本
2. **模型 Agent 已就绪**：需要一个可用的 chat agent（本示例使用 DeepSeek），配置好 base_url、model、auth
3. **Session 基础概念已理解**：了解 Session 的三要素（full_context、context_window、memo）以及 analysis_handler 和 resize_handler 的注册机制。详见本课第四章「Session 短期记忆架构」
4. **对话数据已准备**：准备一组模拟的旅行规划对话消息，用于触发压缩

以下是创建模型 Agent 的辅助函数，本实操代码依赖它：

```python
# 文件名: 模型 Agent 初始化（来自 s01_memory_layers.py）
# 功能: 创建 DeepSeek chat agent，用于后续压缩任务

from agently import Agent, Agently
from dotenv import find_dotenv, load_dotenv
import os

def create_model_agents() -> tuple[Agent, Agent]:
    load_dotenv(find_dotenv(usecwd=True))
    required = [
        "DEEPSEEK_BASE_URL",
        "DEEPSEEK_DEFAULT_MODEL",
        "DEEPSEEK_API_KEY",
    ]
    missing = [name for name in required if not os.environ.get(name, "").strip()]
    if missing:
        raise RuntimeError(f"Missing environment variables: {', '.join(missing)}")

    chat_agent = Agently.create_agent()
    chat_agent.set_settings(
        "OpenAICompatible",
        {
            "base_url": os.environ["DEEPSEEK_BASE_URL"],
            "model": os.environ["DEEPSEEK_DEFAULT_MODEL"],
            "auth": os.environ["DEEPSEEK_API_KEY"],
            "model_type": "chat",
            "request_options": {"temperature": 0},
        },
    )
    return chat_agent, None  # embedding_agent 在后续模块使用，此处省略

chat_agent, _ = create_model_agents()
```

#### 💻 Step-by-Step

##### Step 1: 准备对话数据 —— 模拟旅行规划场景

压缩需要输入一段对话历史。我们以"上海三日亲子游规划"为例，构造 7 条消息，涵盖用户的约束表达和助手的确认追问。这 7 条消息在添加进 Session 后将触发压缩。

```python
# 文件名: 对话数据准备
# 功能: 构造 7 条模拟的旅行规划对话，用于触发 Session 压缩

from agently.types.data.prompt import ChatMessageDict

messages: list[ChatMessageDict] = [
    {"role": "user", "content": "帮我规划上海三日亲子游，带 5 岁的女儿，行程要真正走得通。"},
    {"role": "assistant", "content": "先确认住宿区域和一定要去的景点。"},
    {"role": "user", "content": "迪士尼一定要去，住哪个区还没定。"},
    {"role": "assistant", "content": "我先按住市中心估算路线。"},
    {"role": "user", "content": "孩子走不了太多路，下午两点到四点最好午休。"},
    {"role": "assistant", "content": "路线检查发现迪士尼和上海自然博物馆排在同一天，通勤过长。"},
    {"role": "user", "content": "请重新安排，远距离项目不要和市区景点挤在同一天。"},
]
```

> 这 7 条消息包含了三个关键约束：迪士尼必去、孩子午休需求、远距离项目不扎堆。压缩的目标就是从这些对话中提取出 `confirmed_constraints` 和 `open_questions`。

##### Step 2: 实现策略判定 —— choose_compression

策略判定层的代码极其简洁，但它在整个架构中的角色是"调度中心"。它的唯一职责是根据 Session 当前状态，返回一个压缩策略名或 `None`。

```python
# 文件名: 策略判定层 —— 分析处理器
# 功能: 当上下文窗口消息数量 ≥ 7 时，触发 "model_compress" 压缩策略

async def choose_compression(full_context, context_window, memo, settings):
    """决定是否触发压缩以及触发哪种压缩策略。
    
    参数:
        full_context: list[ChatMessageDict] —— 完整会话历史
        context_window: list[ChatMessageDict] —— 当前上下文窗口
        memo: dict | None —— 当前 memo 内容
        settings: dict —— Session 配置
    
    返回:
        str —— 压缩策略名（匹配 register_resize_handler 注册的名称）
        None —— 不触发压缩
    """
    return "model_compress" if len(context_window) >= 7 else None
```

> **为什么阈值是 7？** 这不是一个固定的魔法数字，而是一个示例配置。在实际业务中，阈值可以是 10 条消息、5000 个 token、或者"用户的约束变更超过 2 次"。这个值应该根据你的模型上下文窗口大小和压缩成本来动态调整。更重要的是——这个阈值可以通过 `settings` 参数从外部配置中读取，让业务人员也能调整触发时机。

##### Step 3: 构造模型输入 —— turn、role、content 三元组

进入压缩策略组成层，第一步是把 context_window 中的消息列表转换为模型可理解的输入格式。这里的关键设计是**给每条消息加上 turn 序号**——这是模型理解对话顺序和追溯信息来源的基础。

```python
# 文件名: 模型输入构造
# 功能: 将 context_window 消息列表转换为带 turn 序号的模型输入

# 在 compress_with_deepseek 函数内部:
model_input = [
    {"turn": index, "role": message.role, "content": message.content}
    for index, message in enumerate(context_window)
]
```

> **为什么需要 turn 序号？** 如果不加 turn 序号，模型看到的只是一堆无序的 user 和 assistant 发言，无法判断哪条在前、哪条在后。加上 turn 后，模型能够理解对话的时序关系，在输出时也可以用 `source_turn` 追朔每条约束或问题的来源——这对后续的验证和追溯至关重要。

三元组每部分的含义：

| 字段 | 含义 | 示例值 | 作用 |
|---|---|---|---|
| `turn` | 消息在窗口中的序号，从 0 开始 | `0`, `1`, `2`... | 标识对话时序，支持 source_turn 追溯 |
| `role` | 消息发送者角色 | `"user"` 或 `"assistant"` | 让模型区分谁说了什么 |
| `content` | 消息文本内容 | `"迪士尼一定要去"` | 压缩的原始素材 |

##### Step 4: 设计压缩指令 —— instruct 的灵活注入

有了模型输入，接下来要告诉模型"怎么压缩"。这一步使用 `agent.instruct([...])` 方法，将压缩指令以自然语言的形式注入。指令的内容直接决定了压缩的质量——这就是压缩指令指导层。

```python
# 文件名: 压缩指令设计
# 功能: 通过 instruct 注入压缩目标、范围和约束

compressed = await (
    agent.input({"messages": model_input})
    .instruct([
        "把 Session 当前窗口压缩成任务 memo，只保留用户明确提供的约束和仍需回答的问题。",
        "每条约束或问题都绑定自己的 source_turn，序号使用输入中的 turn。",
    ])
    # ... output schema 将在下一步定义
    .async_start()
)
```

> **压缩指令要写什么？** 三条原则：第一，明确压缩目标（"压缩成任务 memo"）；第二，明确保留什么、丢弃什么（"只保留约束和问题，不推导长期偏好"）；第三，明确格式约束（"每条绑定 source_turn"）。指令越具体，压缩结果越可控。

> 💡 **核心洞见**：这些 instruct 内容完全可以放到外部配置文件或 skill 文件中。老师强调"你可以把它写成 yaml，用 `load_yaml_prompt` 的方式直接拿来用"——这就是配置化的力量。业务人员只需要修改外部文件中的提示词文本，不需要接触任何 Python 代码。这也解释了为什么第三课的配置化/模板化思想到现在依然适用：只要控制好模型请求的输入输出边界，就可以把业务表达交给业务人员。

##### Step 5: 定义输出结构 —— output schema

与 instruct 配合的是 `output({...})` 方法。它定义了模型输出的结构化 schema，确保压缩结果不会是一个自由格式的文本段落，而是一个字段明确、类型确定、可程序化读取的字典。

```python
# 文件名: 输出结构定义
# 功能: 使用 Agently output schema 约束模型输出格式

.output({
    "confirmed_constraints": [
        {
            "text": (str, "已经确认的约束", "not_null"),
            "source_turn": (int, "约束来自哪条消息", "not_null"),
        }
    ],
    "open_questions": [
        {
            "text": (str, "仍需回答的问题", "not_null"),
            "source_turn": (int, "问题来自哪条消息", "not_null"),
        }
    ],
})
```

output schema 的字段解析：

| 字段 | 类型 | 含义 | 业务价值 |
|---|---|---|---|
| `confirmed_constraints` | 列表 | 用户已明确确认的约束条件 | 后续规划不需再向用户确认，直接遵守 |
| `confirmed_constraints[].text` | str | 约束的具体文本描述 | 可读的约束内容 |
| `confirmed_constraints[].source_turn` | int | 该约束来源于第几条消息 | 可追溯、可验证、可回查原文 |
| `open_questions` | 列表 | 仍需向用户确认的问题 | 后续对话需优先追问的事项 |
| `open_questions[].text` | str | 问题的具体文本描述 | 可读的问题内容 |
| `open_questions[].source_turn` | int | 该问题来源于第几条消息 | 可追溯到问题出现的上下文 |

> **为什么用结构化 schema 而不是让模型自由输出？** 自由文本在后续处理中很难解析——你无法确定模型会不会漏掉某条约束，也无法程序化地读取 `source_turn` 做追溯。结构化 schema 让模型的输出变成"可计算的数据"，而不是"需要再次理解的文本"。

##### Step 6: Resize 返回值 —— 三层信息的取舍

`register_resize_handler` 注册的处理函数需要一个**特定格式的返回值**：一个三元组 `(full_context, context_window, memo)`。这个返回值直接决定了 Session 压缩后的状态。

```python
# 文件名: Resize 返回值处理
# 功能: 返回压缩后的 Session 状态三元组

# compress_with_deepseek 函数的 return 语句:
return None, list(context_window[-3:]), compressed
```

三元组各元素含义：

| 位置 | 参数 | 传入值 | 含义 |
|---|---|---|---|
| 第 1 位 | `full_context` | `None` | 不修改完整会话历史（保留全量记录） |
| 第 2 位 | `context_window` | `list(context_window[-3:])` | 上下文窗口裁剪为最后 3 条消息 |
| 第 3 位 | `memo` | `compressed`（模型返回的压缩结果） | 将压缩结果写入 memo 字段 |

> ⚠️ **注意事项**：这里同时完成了两个动作——**压缩**（把 7 条消息提炼成结构化的 memo）和**裁剪**（上下文窗口从 7 条缩减到 3 条）。这两个动作是绑定的：压缩让重要的信息不丢失（存入 memo），裁剪让上下文窗口不超限（只保留最近 3 条）。如果你使用 Session 内嵌方案，压缩后的 memo 会被框架自动放入上下文；如果你独立使用 Session 模块，则需要自己决定是否把 memo 放回对话中。

##### Step 7: 注册处理器并执行压缩

完成了策略判定和压缩处理两个函数的定义后，需要将它们注册到 Session 实例上，然后添加对话历史以自动触发压缩。

```python
# 文件名: 处理器注册与压缩执行
# 功能: 注册 analysis_handler 和 resize_handler，添加对话触发压缩

from agently.core import Session

# 创建 Session 实例
session = Session(id="lesson-21-memory")

# 注册分析处理器（策略判定层）
# 当 async_add_chat_history 被调用时，框架自动调用此函数
session.register_analysis_handler(choose_compression)

# 注册压缩处理器（压缩策略组成层）
# 第一个参数 "model_compress" 必须与 choose_compression 的返回值匹配
session.register_resize_handler("model_compress", compress_with_deepseek)

# 添加对话历史 —— 自动触发：分析 → 判定 → 压缩
await session.async_add_chat_history(messages)

# 读取压缩结果
memo = session.memo
print(memo)
```

**完整可运行代码** —— 将以上 Step 1~7 组装为一个函数：

```python
# 文件名: s01_memory_layers.py —— create_session_memo 函数（节选）
# 功能: Session 压缩完整链路：数据准备 → 策略判定 → 模型压缩 → 结果验证

from __future__ import annotations
from typing import Any, cast
from agently.types.data.prompt import ChatMessageDict
from agently.core import Session

async def create_session_memo(agent) -> dict[str, Any]:
    # Step 1: 准备对话数据
    messages: list[ChatMessageDict] = [
        {"role": "user", "content": "帮我规划上海三日亲子游，带 5 岁的女儿，行程要真正走得通。"},
        {"role": "assistant", "content": "先确认住宿区域和一定要去的景点。"},
        {"role": "user", "content": "迪士尼一定要去，住哪个区还没定。"},
        {"role": "assistant", "content": "我先按住市中心估算路线。"},
        {"role": "user", "content": "孩子走不了太多路，下午两点到四点最好午休。"},
        {"role": "assistant", "content": "路线检查发现迪士尼和上海自然博物馆排在同一天，通勤过长。"},
        {"role": "user", "content": "请重新安排，远距离项目不要和市区景点挤在同一天。"},
    ]

    # Step 2: 策略判定 —— 7 条及以上触发压缩
    async def choose_compression(full_context, context_window, memo, settings):
        return "model_compress" if len(context_window) >= 7 else None

    # Step 3~6: 压缩策略组成 —— 构造输入、注入指令、定义输出、处理返回值
    async def compress_with_deepseek(full_context, context_window, memo, settings):
        # Step 3: 构造模型输入（turn + role + content 三元组）
        model_input = [
            {"turn": index, "role": message.role, "content": message.content}
            for index, message in enumerate(context_window)
        ]

        # Step 4 & 5: 注入压缩指令 + 定义输出结构
        compressed = await (
            agent.input({"messages": model_input})
            .instruct([
                "把 Session 当前窗口压缩成任务 memo，只保留用户明确提供的约束和仍需回答的问题。",
                "每条约束或问题都绑定自己的 source_turn，序号使用输入中的 turn。",
            ])
            .output({
                "confirmed_constraints": [{
                    "text": (str, "已经确认的约束", "not_null"),
                    "source_turn": (int, "约束来自哪条消息", "not_null"),
                }],
                "open_questions": [{
                    "text": (str, "仍需回答的问题", "not_null"),
                    "source_turn": (int, "问题来自哪条消息", "not_null"),
                }],
            })
            .async_start()
        )

        # Step 6: 返回三元组 —— 压缩 memo + 裁剪窗口
        return None, list(context_window[-3:]), compressed

    # Step 7: 注册处理器并触发压缩
    session = Session(id="lesson-21-memory")
    session.register_analysis_handler(choose_compression)
    session.register_resize_handler("model_compress", compress_with_deepseek)
    await session.async_add_chat_history(messages)

    memo = cast(dict[str, Any], session.memo)
    return memo
```

#### 5.3.1 文本长度门控裁剪：超越按条目裁剪的灵活策略

上面的示例使用 `list(context_window[-3:])` 按条目数量裁剪——固定保留最后 3 条。但在实际场景中，消息长度差异极大：一条"迪士尼一定要去"只有 7 个字，而一条详细的路线分析可能 500 字。按条目裁剪不够精细。

老师提出了**按文本长度门控裁剪**的思路：从后往前逐条累加消息，计算累计文本长度，超过门控阈值时停止，只保留阈值内的消息。

```python
# 文件名: 文本长度门控裁剪（思路代码，未实际运行）
# 功能: 从后往前堆叠消息，按累计文本长度门控决定保留哪些消息

def resize_by_text_length(context_window, max_length=5000):
    """按文本长度门控裁剪上下文窗口。
    
    从后往前逐条累加消息，一旦累计长度超过 max_length 就停止。
    只保留下来的及更近的消息。
    
    参数:
        context_window: 当前上下文窗口消息列表
        max_length: 文本长度门控阈值（字符数，默认 5000）
    
    返回:
        裁剪后的消息列表（保留最靠近当前的消息）
    """
    kept_messages = []
    total_length = 0

    # 从后往前遍历（索引从 -1 到 -len(context_window)）
    for i in range(len(context_window) - 1, -1, -1):
        message = context_window[i]
        message_length = len(message.content)

        # 累加文本长度
        total_length += message_length

        # 门控判定：超过阈值则停止
        if total_length > max_length:
            break

        # 未超阈值，将消息加入保留列表
        kept_messages.insert(0, message)  # 头部插入，保持时间顺序

    return kept_messages
```

> **门控阈值如何设定？** 5000 只是一个示例值。实际应该根据你的模型上下文窗口总大小减去 system prompt、memo 和预留回复空间来动态计算。例如：模型上下文 128K tokens，你预留 4K 给 system prompt、2K 给 memo、2K 给回复，那么裁剪门控可以设在大约 120K tokens 的位置。

**按条目裁剪 vs 按长度门控裁剪：**

| 维度 | 按条目裁剪（固定 N 条） | 按长度门控裁剪 |
|---|---|---|
| 实现复杂度 | 极低（一行代码） | 中等（需遍历计算） |
| 对长短消息的适应性 | 差——短消息浪费空间，长消息可能截断 | 好——按实际占用空间决定 |
| 确定性 | 高（保留数量固定） | 中（保留数量随消息长度浮动） |
| 适用场景 | 消息长度均匀的场景 | 消息长度差异大的场景 |

#### 🐛 排错指南

##### 🐛 排错实录 1：压缩策略名不匹配导致压缩不触发

**❌ 错误写法**
```python
# analysis_handler 返回 "deepseek_compress"
async def choose_compression(full_context, context_window, memo, settings):
    return "deepseek_compress" if len(context_window) >= 7 else None

# resize_handler 注册的是 "model_compress"
session.register_resize_handler("model_compress", compress_with_deepseek)
```

**🚨 现象**
添加 7 条消息后，`session.memo` 仍为 `None`，压缩没有被触发。

**🔍 原因分析**
`register_analysis_handler` 注册的函数返回的策略名（`"deepseek_compress"`）与 `register_resize_handler` 注册的策略名（`"model_compress"`）不一致。Session 框架内部通过策略名做路由匹配，名称不匹配会导致找不到对应的 resize_handler，压缩静默跳过。

**✅ 正确写法**
```python
# 策略名必须一致
async def choose_compression(full_context, context_window, memo, settings):
    return "model_compress" if len(context_window) >= 7 else None

session.register_resize_handler("model_compress", compress_with_deepseek)
```

**📝 经验总结**
策略名是 analysis_handler 和 resize_handler 之间的唯一关联键，建议将策略名定义为常量，避免手写字符串导致的不一致。

---

##### 🐛 排错实录 2：source_turn 超出合法范围

**❌ 错误现象**
验证代码抛出 `ValueError: DeepSeek memo contains an invalid Session source-turn binding.`。

**🚨 可能原因**
模型在压缩时可能编造了一个不存在的 turn 序号（幻觉），或者错误地计算了 turn 值。

**🔍 原因分析**
模型不是确定性系统。即使 instruct 中明确写了"序号使用输入中的 turn"，模型仍可能在某些情况下输出错误的序号。这是大模型压缩的固有不确定性。

**✅ 解决方案**
在验证代码中捕获异常后，不要直接丢弃整份 memo，而是过滤掉无效的条目：

```python
# 防御性过滤：只保留 source_turn 合法的条目
valid_memo = {
    key: [
        item for item in memo[key]
        if 0 <= item.get("source_turn", -1) < len(messages)
    ]
    for key in ("confirmed_constraints", "open_questions")
}
```

**📝 经验总结**
永远不要无条件信任模型的输出。对于结构化字段（如 source_turn），必须做后置校验。这也是为什么我们在 output schema 中定义了类型和约束——不是为了约束模型（模型可能不遵守），而是为了在验证阶段能快速定位问题。

##### 🐛 排错实录 3：压缩后上下文窗口包含了已被压缩的内容

**❌ 错误写法**
```python
# 返回值保留了全部 context_window，仅更新 memo
return None, context_window, compressed
```

**🚨 现象**
压缩后上下文窗口长度没有减少，下一次对话继续累积，很快超出模型上下文限制。

**🔍 原因分析**
压缩的目的是"提炼信息到 memo + 释放上下文空间"。如果 context_window 没有裁剪，信息既存在 memo 里（浓缩版）又存在于 context_window 里（原始版），造成了冗余和空间浪费。

**✅ 正确写法**
```python
# 只保留最后 3 条原始消息，其余以 memo 形式存在
return None, list(context_window[-3:]), compressed
```

**📝 经验总结**
压缩的三个返回值（full_context, context_window, memo）需要联动设计。如果只更新 memo 而不裁剪 context_window，压缩就失去了"释放上下文空间"的意义。但如果你独立使用 Session 模块（不依赖框架的上下文组装），你可以自己决定是否把 memo 放回对话——这是松耦合关系。

### 5.4 Memo 设计实战：从通用摘要到场景化结构

#### 5.4.1 旅行助手场景：访谈表单式 memo

前面代码中定义的 `confirmed_constraints` + `open_questions` 结构并非偶然。在旅行助手场景中，助理的核心任务是帮用户完成一个旅行计划——这本质上是一个**信息收集和渐进确认**的过程。因此，memo 的设计模仿了一张访谈表单：

```python
# 旅行助手 memo 结构 —— 访谈表单式
memo = {
    "confirmed_constraints": [
        {"text": "迪士尼一定要去", "source_turn": 2},
        {"text": "孩子下午两点到四点需午休", "source_turn": 4},
    ],
    "open_questions": [
        {"text": "住宿区域尚未确定", "source_turn": 2},
    ]
}
```

**为什么是这两个字段？**

- `confirmed_constraints`（已确认约束）：用户在对话中明确表达的硬性要求，后续规划时直接遵守，不需要再次向用户确认。每压缩一次，这个列表会更完整——相当于"需求在不断被澄清"。
- `open_questions`（待确认问题）：对话中暴露出的信息缺口。这些是助手在下一轮对话中应该优先追问的事项。每压缩一次，未解决的问题会被重新整理——相当于"待办清单在不断被刷新"。

> 💡 **核心洞见**：旅行助手场景下的 memo 不是通用的对话摘要，而是**带有明确业务倾向的信息整理**。它的两个字段分别对应"已确认的事情"和"还要问的事情"——这个设计本身就是业务策略的体现。正如老师所说："memo 就是原信息加工概括后的信息，但我们的加工信息是有业务倾向的。"

#### 5.4.2 场景对比：旅行助手 vs 拟人陪聊

如果把场景从旅行助手换成拟人陪聊（角色扮演、虚拟人陪伴），memo 的设计会发生根本性变化。两种场景对"什么信息值得记住"的定义完全不同：

| 维度 | 旅行助手 memo | 拟人陪聊 memo |
|---|---|---|
| **记录重点** | 任务约束（必去景点、时间限制）、待确认信息（住宿区域未定） | 用户偏好（性格、喜好、敏感话题）、聊天习惯（称呼偏好、互动风格） |
| **数据结构** | 访谈表单式：`confirmed_constraints` + `open_questions` | 偏好列表：`user_preferences` + `known_topics` + `behavior_notes` |
| **初始人设** | 旅行助手的角色设定放 system prompt，不放入 memo | 虚拟人的初始人设（霸道总裁、温柔学姐等）**必须放入 system prompt**，不在 memo 中管理 |
| **动态人设调整** | 通常不涉及 | 如果虚拟人可以在聊天中调整自己的表达行为（如用户要求改变称呼），这种**可接受的行为调整**可以写入 memo |
| **核心逻辑** | 核实 → 确认 → 追问——帮助完成任务 | 记录 → 适应 → 个性化——让聊天更贴合用户 |
| **memo 的更新频率** | 每次压缩都可能刷新约束和问题列表 | 随着对话深入，逐步累积偏好和话题记录 |

**拟人陪聊 memo 示例结构：**

```python
# 拟人陪聊场景的 memo 设计示例
memo = {
    "user_preferences": [
        {"category": "称呼", "value": "用户不喜欢被叫'亲'", "source_turn": 5},
        {"category": "话题", "value": "用户对科幻电影兴趣浓厚", "source_turn": 12},
    ],
    "known_topics": [
        {"topic": "上周聊到的《三体》剧情讨论", "source_turn": 8},
        {"topic": "用户提到最近工作压力大", "source_turn": 15},
    ],
    "behavior_notes": [
        {"note": "用户对语气过于热情表示不适", "source_turn": 20},
    ],
}
```

> ⚠️ **注意事项**：初始人设一定要放在 **system prompt** 中，不要放在 memo 里。原因很简单——memo 管理的是对话中**动态产生**的信息，而初始人设是对话开始前就已确定的**静态配置**。如果把初始人设放进 memo，每次压缩时模型可能会错误地修改它。这个问题老师在课上特别强调了。

#### 5.4.3 通用 memo vs 定制化 memo：如何取舍？

很多教程会告诉你，记忆压缩就是"把对话历史总结成一段摘要"——这是一个**通用列表式 memo**：

```python
# 通用列表式 memo —— 无业务倾向
memo = {
    "notes": [
        "用户计划上海三日亲子游，带5岁女儿。",
        "迪士尼一定要去。",
        "孩子下午两点到四点需要午休。",
        "住宿区域未确定。",
    ]
}
```

这种格式的优点是**通用**——不依赖任何业务场景，任何对话都能套用。缺点也很明显：它是一个"扁平的信息堆"，没有结构化的分类，下游使用时需要再次解析和理解。

**定制化结构化 memo**（如前面的 `confirmed_constraints` + `open_questions`）优点是**精准**——下游代码可以直接按字段取用，不需要再理解语义。缺点是不通用——换一个场景（如从旅行助手变成陪聊机器人）就需要重新设计 memo 结构。

| 维度 | 通用列表式 memo | 定制化结构化 memo |
|---|---|---|
| 通用性 | 高——任何场景都能用 | 低——每个场景需单独设计 |
| 下游使用成本 | 高——需要再次解析文本 | 低——按字段直接取用 |
| 业务语义 | 无——所有信息平铺 | 有——字段名本身就承载业务含义 |
| 压缩指令复杂度 | 低——"列出要点"即可 | 中——需要描述每个字段的含义 |
| 适用场景 | 快速原型、多场景复用 | 生产环境、深度业务集成 |

> 💡 **核心洞见**：不存在"放之四海而皆准"的最佳记忆压缩方案。正如老师在课上反复强调的——**记忆的本质是重要信息的存储和按需召回**。"重要"的定义由业务场景决定，"存储"的格式由下游使用方式决定，"召回"的时机由对话流程决定。网上随便找一个通用记忆方案，它可能在你特定的业务场景中效果很差。**适合你的记忆方案，一定是根据你的场景定制的**。

---

> 🔗 **关联知识**：配置化/模板化的思想贯穿了从第三课到现在的全部课程。在三层管控架构中，压缩指令指导层（instruct + output schema）就是配置化的具体应用——业务人员通过修改外部 prompt 文件来控制压缩行为，无需接触代码。在第 3 课中我们学到 `load_yaml_prompt` 把整个 prompt 的配置权交给业务人员，这一思想在本课的压缩指令注入中同样适用。**第三课的知识到现在依然适用**——只要控制好模型请求的输入输出边界，业务表达就可以从代码中解耦。

### 6.1 Task Workspace 压缩：反向查找与 source_ref 保留

#### 🧠 直观理解

想象你在整理一个项目的资料。桌面上散落着十几份文件——有的跟当前项目有关，有的完全是其他项目的东西。你的目标不是从头到尾读一遍，而是**反过来**：拿着当前项目的目标，去每份文件中"检索"看它是否相关。相关就摘录要点，不相关就标记跳过。这就是 Task Workspace 压缩的核心：**反向查找**。

#### 📖 详细解释

在 Agent 记忆体系中，Task Workspace 是单个任务的工作空间，里面可能积累了多个来源（source）的信息。当我们需要压缩这些信息为任务记忆时，不能简单地一股脑全塞给模型做总结——那样容易引入无关信息，也会让模型在跨来源之间产生混淆。

正确的做法是**逐条独立判断**：

1. **反向查找**：以任务目标（task_goal）为锚点，逐条询问模型"当前这条 source 和目标相关吗？"，而不是反过来让模型自己挑相关信息。
2. **source_ref 外部绑定**：每条压缩后的事实必须绑定 `source_ref`（来源引用），且这个 `source_ref` 由循环变量直接赋值，**不依赖模型输出**——这是防止幻觉的关键设计。如果让模型自己生成引用，它可能编造一个不存在的来源。
3. **被排除的来源也要记录**：所有不相关的来源放入 `omitted_sources`，保持全程可见。静默丢弃会导致后续排查时不知道"为什么这个信息没被用上"。

#### 🔗 推导链：反向查找的设计理由

```
起点: 任务工作区有多条来源信息，需要压缩为任务记忆
  │
  ├── 第1步: 为什么不能批量输入？
  │   为什么？跨来源信息会互相污染，模型可能把来源A的内容误判为来源B的证据。
  │
  ├── 第2步: 为什么要外部绑定 source_ref？
  │   为什么？模型输出是不可控的，由代码层面绑定引用可以100%保证引用的准确性。
  │
  ├── 第3步: 为什么要记录 omitted_sources？
  │   为什么？透明性。后续审查时可以确认"这条信息确实被判断过，且不相关"。
  │
  └── 结论: 逐条独立判断 + 外部引用绑定 + 全量记录 = 可审计的压缩流水线
```

#### 💻 代码示例

```python
# 文件名: task_workspace_compress.py
# 功能: Task workspace 压缩——反向查找 + source_ref 保留

task_sources = [
    {"source_ref": "evt_s01", "content": "上海三日亲子游，同行儿童 5 岁，行程需要可执行。"},
    {"source_ref": "evt_s02", "content": "路线检查失败：迪士尼与上海自然博物馆安排在同一天，通勤过长。"},
    {"source_ref": "evt_s03", "content": "孩子下午两点到四点需要午休，不能连续长距离步行。"},
    {"source_ref": "evt_x01", "content": "另一项成都员工团建正在收集 20 人的餐饮发票，尚未开始路线规划。"},
]

task_goal = "修正上海三日亲子游初稿，形成下一版路线安排所需的任务记忆。"
task_memory = {"facts": [], "omitted_sources": []}

for source in task_sources:
    source_result = await (
        memory_agent
        .input({"task": task_goal, "source_content": source["content"]})
        .instruct([
            "判断 source_content 是否能直接帮助完成 task，不因它是一条事实就默认相关。",
            "相关时压缩为不超出来源内容的事实；无关时说明判断理由。",
        ])
        .output({
            "is_relevant": (bool, "是否和当前任务直接相关", "not_null"),
            "compressed_text": (str, "相关时返回压缩后的来源内容"),
            "reason": (str, "相关性判断依据", "not_null"),
        })
        .async_start()
    )
    if source_result["is_relevant"]:
        compressed_text = source_result.get("compressed_text", "").strip()
        assert compressed_text, "相关来源必须返回 compressed_text"
        task_memory["facts"].append({
            "text": compressed_text,
            "source_ref": source["source_ref"],  # 由循环变量绑定，不依赖模型
        })
    else:
        task_memory["omitted_sources"].append({
            "source_ref": source["source_ref"],
            "reason": source_result["reason"],
        })
```

| 关键点 | 说明 |
|---|---|
| `source_ref` 绑定 | 由循环变量直接赋值，不依赖模型输出——防止幻觉 |
| 独立判断 | 每条 source 独立送入模型，避免跨来源污染 |
| `omitted_sources` | 被排除的来源也保持可见，防止静默丢弃 |
| `assert compressed_text` | 安全断言：标记为相关的条目必须有压缩文本，防止模型给出空结果 |

> ⚠️ **注意事项**：这 4 条来源中，`evt_x01`（成都团建）和上海亲子游任务无关，模型应该正确判断为不相关并给出理由。如果模型错误地将它标记为相关，这说明你的 instruct 提示词需要更明确地强调"不因它是事实就默认相关"。

### 6.2 Namespace 压缩：去重合并与 evidence_refs 校验

#### 🧠 直观理解

Task 级别的压缩解决的是"一个任务内，哪些信息有用"的问题。Namespace 级别的压缩解决的是"跨多个任务，哪些重复信息可以合并"的问题。

举个例子：用户在"北京旅行规划"任务里说过"孩子下午要午休"，在"西安旅行规划"任务里又说过"孩子下午两点到四点需要午休"。这两条说的是同一件事，Namespace 压缩就是把它们合并为一条长期记忆。

#### 📖 详细解释

Namespace 压缩与 Task 压缩有几个关键区别：

1. **输入性质不同**：Task 压缩的输入是原始 source（可能存在无关信息），Namespace 压缩的输入必须是**已确认的跨任务记录**（confirmed_records），模型不再做相关性判断，只做**去重合并**。
2. **evidence_refs 必须可追溯**：合并后的每条 claim 需要保留 `evidence_refs` 列表，说明"这条合并结论来源于哪些原始记录"。而且代码层面必须校验这些 `evidence_refs` 都在允许的输入范围内——不允许模型编造引用。
3. **不引入新事实**：Namespace 压缩的 instruct 需要明确约束"不引入新事实"，模型只能对已有内容进行去重和精炼，不能扩展或推理出新信息。

#### 🔀 对比：Task 压缩 vs Namespace 压缩

| 维度 | Task Workspace 压缩 | Namespace 压缩 |
|---|---|---|
| 输入来源 | 任务工作区的原始 source（含无关信息） | 跨任务的已确认记录 |
| 模型职责 | 判断相关性 + 压缩 | 仅去重合并，不引入新事实 |
| 引用追溯 | source_ref（单条回溯） | evidence_refs（可能多个来源合并） |
| 校验重点 | 相关条目必须有 compressed_text | evidence_refs 必须在允许范围内 |
| 存储位置 | task_memory | namespace 长期记忆 |

#### 💻 代码示例

```python
# 文件名: namespace_compress.py
# 功能: Namespace 压缩——去重合并已确认的跨任务信息

namespace_input = {
    "scope": {"tenant_id": "tenant-a", "namespace": "user-42:family-travel"},
    "confirmed_records": [
        {
            "evidence_ref": "evt_a05",
            "text": "北京任务中，用户说明 5 岁孩子走不了太多路，下午两点到四点最好回酒店午睡。",
        },
        {
            "evidence_ref": "evt_b01",
            "text": "西安任务中，用户再次强调孩子不能走太多路，下午两点到四点需要午休。",
        },
    ],
}

namespace_result = await (
    memory_agent
    .input(namespace_input)
    .instruct([
        "压缩 confirmed_records 中重复或重叠的信息，不引入新事实。",
        "输出的每条 claim 保留实际支持它的 evidence_refs。",
    ])
    .output({
        "claims": [{
            "text": (str, "去重后的单条长期信息", "not_null"),
            "evidence_refs": [(str, "支持该信息的 evidence_ref", "not_null")],
        }],
    })
    .async_start()
)

# 校验 evidence_refs 都在允许范围内
allowed_evidence_refs = {item["evidence_ref"] for item in namespace_input["confirmed_records"]}
assert all(
    set(claim["evidence_refs"]) <= allowed_evidence_refs
    for claim in namespace_result["claims"]
)
```

> ⚠️ **注意事项**：上面的校验逻辑 `set(claim["evidence_refs"]) <= allowed_evidence_refs` 使用了集合的子集判断。这确保了模型输出的每个 evidence_ref 都来自输入——如果模型编造了 `evt_z99` 这种不存在的引用，校验会直接抛异常，防止脏数据进入长期记忆。

### 6.3 压缩共性总结：四步统一方法论

无论在哪一层做压缩（Session、Task、Namespace），本质上的逻辑链条是相同的。我们可以抽象出一个统一的四步框架：

```
数据来源
    │
    ▼
压缩策略（定义"压什么、怎么压、压成什么样"）
    │
    ▼
压缩结果（经过模型处理后的结构化输出）
    │
    ▼
存储位置（结果写入哪一层记忆存储）
```

| 压缩层级 | 数据来源 | 压缩策略 | 压缩结果 | 存储位置 |
|---|---|---|---|---|
| Session 级 | 对话历史（第 17-20 课） | 分段摘要 + 关键事件抽取 | Session Memo | session_memory |
| Task Workspace 级 | 任务工作区的原始 source | 反向查找 + 逐条独立判断相关性 | compressed_text + source_ref | task_memory["facts"] |
| Namespace 级 | 跨任务的 confirmed_records | 去重合并，不引入新事实 | claims + evidence_refs | namespace 长期记忆 |

> 💡 **核心洞见**：理解了这四步框架，你在任何场景下设计压缩系统时只需要回答四个问题——数据从哪来？用什么策略压缩？输出的结构是什么？压缩结果存到哪去？答案自然就有了。

### 6.4 Embedding 模型配置：SiliconFlow 接入

#### 🎯 目标

配置一个专门用于文本向量化的 Agent，使用 SiliconFlow 的 Embedding 模型。关键在于 `model_type` 必须指定为 `"embeddings"`，否则 SDK 可能会按聊天补全（chat completion）的方式调用，导致报错或返回格式异常。

#### 💻 Step-by-Step

```python
# 文件名: embedding_config.py
# 功能: SiliconFlow embedding 模型接入

import os
from agently import Agently
from dotenv import find_dotenv, load_dotenv

def create_embedding_agent():
    """创建并配置 SiliconFlow embedding 模型 Agent"""
    load_dotenv(find_dotenv(usecwd=True))
    required = ["SILICON_BASE_URL", "SILICON_EMBEDDING_MODEL", "SILICON_API_KEY"]
    missing = [name for name in required if not os.environ.get(name, "").strip()]
    if missing:
        raise RuntimeError(f"Missing environment variables: {', '.join(missing)}")

    embedding_agent = Agently.create_agent()
    embedding_agent.set_settings("OpenAICompatible", {
        "base_url": os.environ["SILICON_BASE_URL"],
        "model": os.environ["SILICON_EMBEDDING_MODEL"],
        "auth": os.environ["SILICON_API_KEY"],
        "model_type": "embeddings",  # 关键：必须指定为 embeddings 类型
    })
    return embedding_agent
```

| 行号 | 关键点 | 解释 |
|---|---|---|
| `model_type: "embeddings"` | 类型声明 | 告诉 SDK 此 Agent 走 embedding 接口而非 chat 接口，决定了请求体和响应体的解析方式 |
| `base_url` | API 端点 | SiliconFlow 兼容 OpenAI 接口格式，使用 OpenAICompatible 设置即可 |
| 环境变量校验 | 启动检查 | 三个环境变量缺一不可，启动时明确报错比运行时才报错更容易排查 |

> 🐞 **常见坑**：忘记设置 `model_type: "embeddings"` 是最高频的错误。如果不加这个配置，SDK 会按聊天模型的格式构造请求，SiliconFlow 返回的 embedding 数据会被错误解析，通常表现为 `TypeError` 或结果类型不匹配。

### 6.5 向量写入与语义召回

#### 🧠 直观理解

向量化的本质是给每段文字一个"坐标"。语义相近的文字，坐标就靠近。当用户提出一个问题（query），我们把问题也变成坐标，然后在所有已存储的坐标中找距离最近的几个——这就是语义召回。

#### 📖 详细解释

整个流程分为两个阶段：**写入阶段**和**召回阶段**。

老师课上画了向量库的工作流程图，下面用 ASCII 还原：

```

│  【写入阶段】                                                         │
│                                                                     │
│  原始信息 ──▶ record text ──▶ embedding 模型 ──▶ 向量 ──▶ memory_store │
│  (多条记忆)    (提取文本)      (SiliconFlow)     (list[float])  (存储)   │
│                                                                     │
│  ═══════════════════════════════════════════════════════════════════ │
│                                                                     │
│  【召回阶段】                                                         │
│                                                                     │
│  用户 query ──▶ embedding ──▶ query 向量                             │
│                              │                                      │
│                              ▼                                      │
│                      ┌───────────────┐                              │
│                      │  Scope 过滤    │  ← 先按租户/命名空间过滤        │
│                      └───────┬───────┘                              │
│                              │                                      │
│                              ▼                                      │

```

#### 💻 代码示例：写入阶段

```python
# 文件名: vector_write.py
# 功能: 批量向量化写入——record text → embedding → memory_store

# 假设 namespace_memories 是已压缩好的记忆列表
# 每项包含 id, text, scope 等字段

# 批量向量化：一次请求处理全部 texts，避免逐条调用
record_vectors = embedding_agent.input(
    [record["text"] for record in namespace_memories]
).start()
assert len(record_vectors) == len(namespace_memories)

# 组装 memory_store：将向量附加到原始记录上
memory_store = [
    {
        **record,
        "vector": vector,
        "embedding_model": os.environ["SILICON_EMBEDDING_MODEL"],
    }
    for record, vector in zip(namespace_memories, record_vectors)
]
```

> 💡 **核心洞见**：记录 `embedding_model` 是很重要的工程实践。如果将来更换了 Embedding 模型（比如从 BAAI/bge-large-zh 换成其他模型），新旧向量不在同一个向量空间中，混用会导致召回结果完全失效。记录模型信息可以在迁移时识别出需要重新向量化的条目。

#### 💻 代码示例：召回阶段

```python
# 文件名: vector_recall.py
# 功能: 语义召回完整流程——scope 过滤 → cosine 排序 → Top-K

import math

def cosine(left: list[float], right: list[float]) -> float:
    """余弦相似度计算——两个向量的夹角余弦值，值域 [-1, 1]，越接近 1 越相似"""
    dot = sum(a * b for a, b in zip(left, right))
    left_norm = math.sqrt(sum(value * value for value in left))
    right_norm = math.sqrt(sum(value * value for value in right))
    return dot / (left_norm * right_norm)

recall_scope = {"tenant_id": "tenant-a", "namespace": "user-42:family-travel"}
recall_query = "为这位用户和 5 岁孩子规划新的上海亲子游，需要召回哪些步行与午休偏好？"
query_vector = embedding_agent.input([recall_query]).start()[0]

# 步骤1: 先 scope 过滤
eligible_memories = [
    record for record in memory_store
    if record["scope"] == recall_scope
]

# 步骤2: 再 cosine 排序
ranked_memories = sorted(
    [{**record, "score": cosine(query_vector, record["vector"])}
     for record in eligible_memories],
    key=lambda record: record["score"], reverse=True,
)

# 步骤3: 取 Top-K
recalled_memories = ranked_memories[:2]

# 构造 context_package（只含正文和分数，不含向量）
context_package = {
    "query": recall_query,
    "scope": recall_scope,
    "namespace_memories": [
        {"id": item["id"], "text": item["text"], "score": round(item["score"], 4)}
        for item in recalled_memories
    ],
}
```

> ⚠️ **注意事项**：scope 过滤必须在 cosine 排序**之前**。如果顺序反过来——先对所有记录做 cosine 排序再 scope 过滤——会出现严重问题：一条语义上高度匹配但 scope 不属于当前租户的记录会被排到前面，如果 scope 过滤代码有疏漏，这条记录就可能绕过数据隔离进入召回结果。**先过滤再排序**从逻辑上确保了数据隔离这一硬约束不会被语义相似度"盖过"。

#### 📖 余弦相似度公式详解

余弦相似度衡量的是两个向量在方向上的接近程度，不受向量长度（模）的影响：

```
cosine(A, B) = (A · B) / (||A|| × ||B||)

其中：
  A · B      = Σ(Aᵢ × Bᵢ)       点积（逐元素相乘再求和）
  ||A||      = √(Σ Aᵢ²)          向量 A 的模（欧几里得长度）
  ||B||      = √(Σ Bᵢ²)          向量 B 的模
```

- 完全同向：cosine = 1（最相似）
- 正交无关：cosine = 0
- 完全反向：cosine = -1（最不相似）

Embedding 模型的输出向量一般在正象限，所以实际召回中的 cosine 值通常在 0.5~0.95 之间。

### 6.6 content vs meta 分离设计：向量化粒度的关键

#### 🧠 直观理解

不要把一整篇文章直接向量化。就像图书馆的检索系统——你搜索时匹配的是**书名和摘要**（精简、聚焦），而不是把整本书的内容拿去做全文比对。命中之后，你再根据索书号去书架上取**完整的书**（完整信息）。

#### 📖 详细解释

在向量化存储中，每条记录拆分为两部分：

```
┌─────────────────────────────────────┐
│          content（参与向量化）         │
│  ┌───────────────────────────────┐  │
│  │ "孩子下午两点到四点需要午休"      │  │  ← 精简、聚焦的摘要文本
│  │ "用户不希望行程过满"             │  │    用于与 query 计算语义相似度
│  │ "孩子不能连续长距离步行"          │  │
│  └───────────────────────────────┘  │
│                                     │
│          meta（不参与向量化）          │
│  ┌───────────────────────────────┐  │
│  │ 完整原文段落                    │  │  ← 完整的上下文、证据引用、
│  │ evidence_refs 引用列表          │  │    scope 信息、优先级等
│  │ scope 租户/命名空间             │  │    不参与向量匹配
│  │ priority 优先级                │  │
│  │ created_at 创建时间             │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

**为什么要分离？**

1. **匹配精度**：向量化的内容越聚焦，语义匹配越准确。如果把大段原文向量化，噪声信息（如"此外"、"顺便说一下"等过渡语）会稀释真正的语义信号。
2. **召回后可用性**：匹配到相关条目后，meta 中的完整原文可以让模型获得充分上下文，而不是只能看到一个干巴巴的摘要。
3. **场景化分裂索引**：同一篇文章可以生成多个 content 条目（不同角度的摘要），每条都指向同一个 meta（完整原文）。这样用一个 query 可以从不同角度命中同一篇文章，大幅提高召回率。

#### 🔀 场景化分裂索引示意

```
同一篇原始文章：
"用户在北京任务中说明 5 岁孩子走不了太多路，下午两点到四点最好回酒店午睡。
 此外用户在西安任务中再次强调了这一点，并表示不能安排连续长距离步行。"

           ┌──────────────────────────────┐
           │  原始文章（meta 中完整保存）     │
           └──────────────┬───────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
   ┌────────────┐  ┌────────────┐  ┌────────────┐
   │ content 1  │  │ content 2  │  │ content 3  │
   │ "孩子午休   │  │ "5岁孩子    │  │ "避免长距离 │
   │  两点到四点"│  │  体力有限"  │  │  步行"      │
   │ → 向量化   │  │ → 向量化    │  │ → 向量化    │
   └────────────┘  └────────────┘  └────────────┘
        │               │               │
        └───────────────┴───────────────┘
                        │
              三个索引都指向同一篇 meta

用户 query "午休时间" → 命中 content 1 → 返回完整原文
用户 query "小孩子走不动" → 命中 content 2 或 3 → 返回同一篇完整原文
```

> 💡 **核心洞见**：场景化分裂索引的本质是用**多个精炼的"入口"**指向同一个**完整的信息源**。这比把原文直接向量化更精准，也比只建一个索引更容易命中。

### 6.7 简化版向量库原理实现

#### 🧠 直观理解

专业的向量数据库（如 Milvus、Pinecone、Qdrant）核心解决的是：当你有 100 万条记录时，如何**快速**找到与新 query 最相似的 Top-K 条？它们使用近似最近邻（ANN）算法和索引结构（如 HNSW、IVF）来避免全量比对。

但在教学场景中，我们的记忆条目最多也就几百条，完全可以用一个**列表存储 + 全量 cosine 排序**来演示向量库的核心原理。

#### 📖 详细解释

简化版向量库的本质：

1. **存储层**：一个 Python 列表 `memory_store`，每项是一个字典，包含原始记录和它的向量
2. **查询层**：query 向量化后，对列表中**所有**记录逐一计算 cosine 相似度
3. **排序层**：按相似度降序排列，取 Top-K

这个实现没有索引加速，但它完整展示了向量库的核心工作流：**向量化存储 -> 相似度计算 -> 排序召回**。

#### 💻 代码示例

```python
# 文件名: simple_vector_store.py
# 功能: 简化版向量库——列表存储 + 全量 cosine 排序（教学演示）

import math
from dataclasses import dataclass
from typing import Any

@dataclass
class Record:
    """记忆条目"""
    id: str
    kind: str           # evidence / long_term_memory / session_memo
    scope: dict[str, str]
    summary: str        # 简短摘要（对应 content 角色）
    content: str        # 完整内容（对应 meta 角色）
    priority: int       # 业务优先级 0-100
    vector: list[float] # 预计算的向量


class SimpleMemoryStore:
    """简化版向量记忆库"""

    def __init__(self):
        self._records: list[Record] = []

    def add(self, record: Record) -> None:
        """写入一条记忆"""
        self._records.append(record)

    def add_batch(self, records: list[Record]) -> None:
        """批量写入"""
        self._records.extend(records)

    def _cosine(self, left: list[float], right: list[float]) -> float:
        """余弦相似度"""
        dot = sum(a * b for a, b in zip(left, right))
        left_norm = math.sqrt(sum(v * v for v in left))
        right_norm = math.sqrt(sum(v * v for v in right))
        return dot / (left_norm * right_norm)

    def _scope_match(self, record: Record, scope: dict[str, str]) -> bool:
        """检查 record 的 scope 是否与目标 scope 完全匹配"""
        return all(record.scope.get(key) == value for key, value in scope.items())

    def query(
        self,
        *,
        query_vector: list[float],
        scope: dict[str, str],
        top_k: int = 5,
    ) -> list[dict[str, Any]]:
        """
        召回查询：
        1. scope 过滤
        2. 全量 cosine 排序
        3. 返回 Top-K
        """
        # 步骤1: scope 过滤
        candidates = [r for r in self._records if self._scope_match(r, scope)]

        # 步骤2: 计算相似度并排序
        scored = [
            {**r.__dict__, "score": round(self._cosine(query_vector, r.vector), 4)}
            for r in candidates
        ]
        ranked = sorted(scored, key=lambda x: x["score"], reverse=True)

        # 步骤3: 返回 Top-K
        return ranked[:top_k]


# ============ 使用示例 ============

# 初始化
store = SimpleMemoryStore()

# 假设已有 embedding_agent 和向量化后的数据
raw_records = [
    ("r1", "evidence", "上海三日亲子游，同行儿童 5 岁。", 55),
    ("r2", "evidence", "路线检查发现迪士尼和上海自然博物馆同日通勤过长。", 65),
    ("r3", "long_term_memory", "用户不希望行程过满，下午两点到四点需要午休。", 40),
    ("r4", "session_memo", "住宿区域尚未确认，迪士尼必须保留。", 60),
    ("r5", "evidence", "孩子不能连续长距离步行。", 58),
]

# 批量向量化后写入
texts = [content for _, _, content, _ in raw_records]
vectors = embed_texts(embedding_agent, texts)
for (rid, kind, content, priority), vector in zip(raw_records, vectors):
    store.add(Record(
        id=rid,
        kind=kind,
        scope={"tenant_id": "tenant-a", "task_id": "shanghai-family-trip-7"},
        summary=kind,
        content=content,
        priority=priority,
        vector=vector,
    ))

# 查询
query_text = "为 5 岁孩子规划上海亲子游，需要召回步行和午休偏好"
query_vector = embedding_agent.input([query_text]).start()[0]
results = store.query(
    query_vector=query_vector,
    scope={"tenant_id": "tenant-a", "task_id": "shanghai-family-trip-7"},
    top_k=3,
)

for item in results:
    print(f"  [{item['id']}] {item['kind']} | score={item['score']} | {item['content']}")
```

> 💡 **核心洞见**：这个简化版实现完整演示了向量库的核心原理——它不是靠什么黑魔法，就是存储向量 + 逐条计算距离 + 排序取 Top-K。真正的向量库在此基础上加了**索引结构**（避免全量扫描）和**近似算法**（牺牲微小精度换取巨大速度提升），但底层逻辑完全一样。

### 6.8 Budget 控制与优先级融合

#### 🧠 直观理解

只靠语义相似度排序有一个问题：一条"语义上很匹配"但"业务上不重要"的记忆，可能挤掉一条"语义上稍差但业务上高优先级"的记忆。我们需要把**语义相似度**和**业务优先级**融合在一起排序，同时受**预算（字符数上限）**的约束。

#### 📖 详细解释

融合公式为：

```
recall_score = 0.7 × cosine(query_vector, record.vector) + 0.3 × (record.priority / 100)
```

- **0.7 权重给语义相似度**：这是召回的主要驱动力。余弦值在 0~1 之间。
- **0.3 权重给业务优先级**：priority 归一化到 0~1（除以 100），确保量纲一致。
- **Budget 控制**：按融合得分排序后，从高到低逐条取，累计字符数不超过 `budget_chars`。一旦超出，后续条目不再纳入（即使得分再高）。

> ⚠️ **注意事项**：Budget 控制中跳过的是"超出预算后"的条目，而不是直接丢弃。在 `diagnostics` 中记录 `candidate_count`（候选总数）和 `selected_count`（实际选中数），方便后续分析是否因为预算过小导致有效信息被截断。

#### 💻 代码示例

```python
# 文件名: recall_with_budget.py
# 功能: Build context 函数——语义 + 优先级 + 预算控制的混合召回

from __future__ import annotations

import math
import os
from dataclasses import dataclass
from importlib.metadata import version
from typing import Any

from agently import Agent, Agently
from dotenv import find_dotenv, load_dotenv


AGENTLY_VERSION = "4.1.4.1"
if version("agently") != AGENTLY_VERSION:
    raise RuntimeError(f"This example requires Agently {AGENTLY_VERSION}.")


@dataclass
class Record:
    id: str
    kind: str
    scope: dict[str, str]
    summary: str
    content: str
    priority: int
    vector: list[float]


def create_embedding_agent() -> Agent:
    """创建并校验 embedding Agent"""
    load_dotenv(find_dotenv(usecwd=True))
    required = ["SILICON_BASE_URL", "SILICON_EMBEDDING_MODEL", "SILICON_API_KEY"]
    missing = [name for name in required if not os.environ.get(name, "").strip()]
    if missing:
        raise RuntimeError(f"Missing environment variables: {', '.join(missing)}")

    agent = Agently.create_agent()
    agent.set_settings(
        "OpenAICompatible",
        {
            "base_url": os.environ["SILICON_BASE_URL"],
            "model": os.environ["SILICON_EMBEDDING_MODEL"],
            "auth": os.environ["SILICON_API_KEY"],
            "model_type": "embeddings",
        },
    )
    return agent


def embed_texts(agent: Agent, texts: list[str]) -> list[list[float]]:
    """批量文本向量化，返回 list[list[float]]"""
    result = agent.input(texts).start()
    if not isinstance(result, list) or len(result) != len(texts):
        raise TypeError("SiliconFlow embedding result has an unexpected shape.")
    return [[float(value) for value in vector] for vector in result]


def cosine(left: list[float], right: list[float]) -> float:
    """余弦相似度"""
    dot = sum(a * b for a, b in zip(left, right))
    left_norm = math.sqrt(sum(value * value for value in left))
    right_norm = math.sqrt(sum(value * value for value in right))
    return dot / (left_norm * right_norm)


def scope_match(record: Record, scope: dict[str, str]) -> bool:
    """检查 record 的 scope 是否完全覆盖目标 scope 的所有约束"""
    return all(record.scope.get(key) == value for key, value in scope.items())


def build_context(
    *,
    goal: str,
    scope: dict[str, str],
    budget_chars: int,
    records: list[Record],
    embedding_agent: Agent,
) -> dict[str, Any]:
    """
    构建上下文包：语义 + 优先级 + 预算控制

    流程：
    1. goal 向量化
    2. scope 过滤 -> 得到候选集
    3. 逐条计算融合得分: 0.7 * semantic + 0.3 * priority
    4. 按得分降序排列
    5. 按 budget_chars 截断，累计字符数超预算后停止
    6. 返回结构化 context_package
    """
    goal_vector = embed_texts(embedding_agent, [goal])[0]
    candidates = [record for record in records if scope_match(record, scope)]

    def recall_score(record: Record) -> float:
        semantic_score = cosine(goal_vector, record.vector)
        priority_score = record.priority / 100
        return 0.7 * semantic_score + 0.3 * priority_score

    ranked = sorted(candidates, key=recall_score, reverse=True)
    items: list[dict[str, Any]] = []
    used_chars = 0
    omitted = 0
    for record in ranked:
        text = f"{record.summary}: {record.content}"
        if used_chars + len(text) > budget_chars and items:
            omitted += 1
            continue
        items.append(
            {
                "id": record.id,
                "kind": record.kind,
                "content": text[:160],      # 截断展示，防止单条过长
                "score": round(recall_score(record), 4),
            }
        )
        used_chars += min(len(text), 160)

    return {
        "goal": goal,
        "scope": scope,
        "items": items,
        "omitted": [{"reason": "budget", "count": omitted}] if omitted else [],
        "diagnostics": {
            "candidate_count": len(candidates),
            "selected_count": len(items),
            "used_chars": used_chars,
            "budget_chars": budget_chars,
            "embedding_dimensions": len(goal_vector),
        },
    }


def main() -> None:
    """完整演示：写入 -> 融合排序 -> Budget 控制 -> 输出"""
    embedding_agent = create_embedding_agent()
    raw_records = [
        ("r1", "evidence", "上海三日亲子游，同行儿童 5 岁。", 55),
        ("r2", "evidence", "路线检查发现迪士尼和上海自然博物馆同日通勤过长。", 65),
        ("r3", "long_term_memory", "用户不希望行程过满，下午两点到四点需要午休。", 40),
        ("r4", "session_memo", "住宿区域尚未确认，迪士尼必须保留。", 60),
        ("r5", "evidence", "孩子不能连续长距离步行。", 58),
    ]
    vectors = embed_texts(embedding_agent, [content for _, _, content, _ in raw_records])
    records = [
        Record(
            id=record_id,
            kind=kind,
            scope={"tenant_id": "tenant-a", "task_id": "shanghai-family-trip-7"},
            summary=kind,
            content=content,
            priority=priority,
            vector=vector,
        )
        for (record_id, kind, content, priority), vector in zip(raw_records, vectors)
    ]

    package = build_context(
        goal="修正上海三日亲子游初稿并保留必要约束",
        scope={"tenant_id": "tenant-a", "task_id": "shanghai-family-trip-7"},
        budget_chars=95,
        records=records,
        embedding_agent=embedding_agent,
    )
    print("selected items:")
    for item in package["items"]:
        print(f"- {item['id']} [{item['kind']}] score={item['score']} {item['content']}")
    print("omitted:", package["omitted"])
    print("diagnostics:", package["diagnostics"])


if __name__ == "__main__":
    main()
```

| 关键设计 | 说明 |
|---|---|
| `recall_score` 融合权重 | `0.7 * semantic + 0.3 * priority`：语义主驱动，优先级辅助调整 |
| `priority / 100` 归一化 | 将优先级从 0-100 映射到 0-1，与 cosine 值量纲对齐 |
| `budget_chars` 截断逻辑 | `used_chars + len(text) > budget_chars and items`：至少保证有一条结果，之后超出预算才截断 |
| `content[:160]` 截断展示 | 每条记录最多占用 160 字符，防止单条过长挤占其他候选的预算 |
| `diagnostics` 诊断信息 | 返回候选总数、选中数、实际使用字符数、向量维度，方便调优 |

> ❓ **常见疑问**：为什么融合公式是 0.7 和 0.3，而不是 0.5 和 0.5？
>
> 这两个权重不是绝对的。选择 0.7/0.3 是基于教学场景的经验值——语义匹配是召回的主体，优先级是"微调"因子。如果你的业务场景中优先级极度重要（比如合规类信息无论如何都要排在前面），可以调高优先级的权重。关键是理解**两个分数的量纲要一致**——cosine 在 0~1 之间，priority 除以 100 后也在 0~1 之间，这样才能公平加权。

## 七、Rule as Record 与自成长策略

### 7.1 Rule as Record：规则即带有"必须遵守"语义的记忆记录

#### 🧠 直观理解

想象你家书房里有一本书。放在书架上的时候，它是"库存知识"——你需要的时候才去翻。但如果把同一本书摊开放在案头、用荧光笔标出关键段落，它就变成了"执行手册"——你每次做事都会参照它。书的内容没有变，变的是**放置位置**和**翻开时机**。

Rule as Record 的核心思想与此完全相同：一条规则，技术上就是一条 MemoryRecord，唯一的区别在于它被存放在特定位置、以特定的注入方式在模型请求时被引用。用一句话概括：

> **规则 = 记忆记录 + "必须遵守"的使用语义。**

#### 🔗 推导链：规则如何"退化"为一条记忆记录

```
起点：我们需要约束 Agent 的行为（如"发布前必须检查路线"）
  │
  ├── 第1步：规则需要一个存储载体 → 用已有的 MemoryRecord 数据结构
  │   为什么？复用已有基础设施（存储、索引、scope 匹配、优先级排序），
  │   不需要为"规则"单独造一套存储引擎。
  │
  ├── 第2步：rule_record 与 evidence 使用完全相同的 MemoryRecord 结构
  │   为什么？两种记录共享的字段包 id / layer / collection / kind /
  │   scope / summary / body / vector / priority 完全相同。
  │   唯一差异点在 kind 字段取值："rule_record" vs "evidence"。
  │
  ├── 第3步：两类记录的唯一差异出现在召回（recall）阶段
  │   为什么？存储时一视同仁，召回时才区分对待——
  │   规则被优先注入且不受 budget 裁剪，证据按 budget 注入。
  │
  └── 结论：kind="rule_record" 在整个系统中只是一个 if 分支判断。
     同一套存储、同一套召回逻辑——规则即记录。
```

#### 💡 关键设计决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 是否为规则单独建表 | 否，复用 MemoryRecord | 避免重复建设；scope 匹配、优先级排序、向量索引均可复用 |
| 规则是否允许被 budget 裁剪 | 否，全部注入 | 规则是硬约束，不能因为"上下文不够了"被牺牲 |
| 规则注入的标记 | `"use": "must_follow"` | 下游 prompt 组装时可根据标记区别对待规则和证据 |

> ⚠️ **注意事项**：虽然规则在存储层面只是记忆的一种，但这不意味着规则可以随意修改。规则承载的是行为约束，修改规则等同于修改 Agent 的"宪法"——必须经过审慎的验证流程（见 7.7 节）。

### 7.2 recall_with_rules：规则优先注入，证据按预算填充

#### 💻 代码示例

```python
# 文件名: scripts/s03_rule_as_record.py
# 功能: Rule as Record——规则优先、证据按预算注入
# 核心逻辑: rule_record 全部注入（must_follow），evidence 按 budget_chars 裁剪

from __future__ import annotations

from dataclasses import dataclass
from typing import Any


@dataclass
class Record:
    id: str
    kind: str            # "rule_record" 或 "evidence"
    scope: dict[str, str]
    summary: str
    content: str
    priority: int


def matches(record: Record, scope: dict[str, str]) -> bool:
    """检查 record 的 scope 是否完全匹配传入的 filter scope"""
    return all(record.scope.get(key) == value for key, value in scope.items())


def recall_with_rules(records: list[Record], scope: dict[str, str],
                       budget_chars: int) -> dict[str, Any]:
    # Step 1: scope 过滤——只保留匹配当前租户/任务的记录
    scoped = [record for record in records if matches(record, scope)]

    # Step 2: 按 kind 分拣，各自按 priority 降序排列
    rules = sorted(
        [record for record in scoped if record.kind == "rule_record"],
        key=lambda item: item.priority, reverse=True
    )
    evidence = sorted(
        [record for record in scoped if record.kind != "rule_record"],
        key=lambda item: item.priority, reverse=True
    )

    # Step 3: 规则优先——全部注入，不受 budget 限制
    items: list[dict[str, str]] = []
    used = 0

    for record in rules:
        text = f"RULE {record.summary}: {record.content}"
        items.append({"id": record.id, "use": "must_follow", "content": text})
        used += len(text)

    # Step 4: 证据按预算注入——超出预算则省略
    omitted = 0
    for record in evidence:
        text = f"{record.summary}: {record.content}"
        if used + len(text) > budget_chars and items:
            omitted += 1
            continue
        items.append({"id": record.id, "use": "evidence", "content": text})
        used += len(text)

    return {
        "items": items,
        "omitted": [{"reason": "budget_after_rules", "count": omitted}] if omitted else [],
        "used_chars": used,
    }


def main() -> None:
    scope = {"tenant_id": "tenant-a", "task_type": "family_travel"}
    records = [
        Record(
            id="rule-route-check",
            kind="rule_record",
            scope=scope,
            summary="路线检查输出",
            content="发布最终行程前必须附上路线检查结论。",
            priority=100,
        ),
        Record(
            id="rule-rest-window",
            kind="rule_record",
            scope=scope,
            summary="儿童午休时段",
            content="下午两点到四点不得安排连续游览和长距离转场。",
            priority=95,
        ),
        Record(
            id="evidence-route",
            kind="evidence",
            scope=scope,
            summary="路线失败",
            content="迪士尼和上海自然博物馆排在同一天时，路线检查判定通勤过长。",
            priority=70,
        ),
        Record(
            id="evidence-child",
            kind="evidence",
            scope=scope,
            summary="同行儿童约束",
            content="同行儿童 5 岁，不能连续长距离步行。",
            priority=68,
        ),
    ]

    package = recall_with_rules(records, scope, budget_chars=100)
    for item in package["items"]:
        print(f"- {item['id']} | {item['use']} | {item['content']}")
    print("omitted:", package["omitted"])


if __name__ == "__main__":
    main()
```

| 行号 | 关键逻辑 | 解释 |
|---|---|---|
| 29-32 | `rules = sorted(...)` | 按 kind 分拣：只取 `kind="rule_record"` 的记录，按 priority 降序 |
| 33-36 | `evidence = sorted(...)` | 剩余记录全部归为证据 |
| 41-43 | `items.append({"use": "must_follow", ...})` | 规则标记为 `must_follow`，下游 prompt 组装时会无条件包含 |
| 46-52 | `if used + len(text) > budget_chars` | 证据受 budget 约束：超出则省略并计数 |
| 49-50 | `omitted += 1; continue` | 策略：规则占满后，证据一个也不加，而不是溢出截断 |

> 💡 **核心洞见**：规则不受 `budget_chars` 限制——规则是必须遵守的硬约束，不能因为"预算不够"被省略。`"use": "must_follow"` vs `"use": "evidence"` 的标记在下游 prompt 组装时会被区别对待。`omitted` 字段用于上报：有多少条证据因"规则先占满预算"而被略过，方便运维和监控。

### 7.3 四种注入位置对比：同一份规则，放对地方才有用

规则的本质虽然是记录，但放到不同位置，其生效时机、影响范围和更新成本完全不同。老师给出了四种典型的注入位置：

| 注入位置 | 注入频率 | 触发时机 | 适用内容 | 典型例子 |
|---|---|---|---|---|
| **system prompt** | 每轮请求 | 每次模型调用 | 操作约束、行为底线 | "发布前必须检查路线"、"禁止生成价格信息" |
| **agents.md / rules.md** | Agent 初始化时 | Agent 生命周期启动 | 人设、风格、全局偏好 | 人格描述（soul.md）、输出格式要求 |
| **rules memo** | 条件触发 | 按轮次或阈值触发读取 | 领域规则、任务级约束 | 亲子游午休规则、公司差旅标准 |
| **skills** | 按需触发 | 工具调用时加载 | 可复用的专项方法 | 路线检查 skill、发票校验 skill |

> 🔗 **关联知识**：agents.md / rules.md 的机制源自 Anthropic 的 `claude.md` 和 OpenAI 的 `soul.md` 设计思路——在 Agent 初始化时将全局行为准则注入，形成 Agent 的"人类设"。本质上它只是 system prompt 的一种结构化表达方式。

同一个"发布前必须检查路线"的约束：
- 放在 **system prompt** 里 → 每轮请求都看到，占用每一轮上下文
- 放在 **rules memo** 里 → 每三五个轮次提醒模型去读取一次，按需加载
- 放在 **skills** 里 → 只有调用路线检查 skill 时才关联生效

> ⚠️ **注意事项**：注入位置的选择直接影响上下文成本和响应延迟。高频注入（如 system prompt）适合简短、绝对的硬约束；低频注入（如 rules memo）适合较长、条件性的领域规范。做错选择会导致：上下文浪费 OR 规则未被及时读取。

### 7.4 自成长与记忆：触发时机的根本差异

#### 📖 详细解释

记忆管理（Memory）和自成长（Self-Growth）在技术底层共享同一套存储和召回基础设施，但它们的**触发时机**完全不同：

| 维度 | 记忆管理（Memory） | 自成长（Self-Growth） |
|---|---|---|
| **触发时机** | 跟着会话走：Session 初始化时录入 memo，对话中通过分析函数判断并召回/改写 | 可脱离会话：定时启动 OR 基于计数器（轮次/工具迭代次数） |
| **更新对象** | memo 存储（上下文窗口内的记忆条目） | 全局行为规则、可复用技能 |
| **生效范围** | 当前 Session（甚至当前对话轮次） | 全局或租户级（影响后续所有任务） |
| **典型场景** | 用户说"我孩子 5 岁"→ 立即写入当前 Session 记忆，后续规划时召回 | 系统发现两次亲子游都出现"未考虑午休"→ 提炼为全局规则候选 |
| **更新频率** | 高频（每轮可能更新） | 低频（定时 batch 或达阈值后触发） |

#### 🔀 为什么自成长不能跟着会话走？

如果自成长跟着每一轮会话实时触发，每个对话轮次都可能产生新的行为规则——这在理论上是可行的，但在企业场景中极不现实。原因有二：

1. **模型可能产生偏差**：模型根据少量 observation 生成的规则候选，未必代表业务真正需要的规范。一次失误被模型当成"教训"写入全局规则，将污染所有后续任务。
2. **需要业务验证**：规则的生效往往涉及责任人确认、版本管理、scope 划定、生效时间——这些不是模型自己能决定的。

因此，企业落地的最佳实践是：**模型负责生成候选（candidate），人负责批准生效**。

### 7.5 自成长的两种载体：沉淀的方式决定使用的方式

自成长的核心产出不是泛化的"经验"，而是有明确结构、可直接被 Agent 消费的两种长期资产：

| 候选类型 | 模型总结什么 | 结构化字段 | 成为正式资产前还缺什么 |
|---|---|---|---|
| **Rule candidate** | 必须遵守的约束、阈值、禁用项、行为底线 | `constraint_type` / `threshold` / `scope` / `effective_from` | 责任人确认、版本号、生效时间、冲突检测 |
| **Skill candidate** | 可复用的方法、步骤序列、工具组合、校验逻辑 | `method_steps` / `input_schema` / `output_schema` / `dependencies` | 接口契约、实现代码、权限声明、测试用例、发布流程 |

```
Rule candidate 的典型产出：
  "根据 obs-001 和 obs-002，亲子游行程发布前必须先用 route_checker 验证，
   且若存在跨区通勤失败则需调整后再发布。"

Skill candidate 的典型产出：
  "发现 validate_itinerary 工具在两次运行中都缺失了'午休时段检查'，
   建议新增 validate_rest_window 子步骤并集成进主 skill。"
```

> 💡 **核心洞见**：两种载体对应两种注入路径——Rule candidate 走 system prompt 或 rules memo 注入线（行为约束），Skill candidate 走 workspace 工具目录加载线（可执行方法）。这不是随意选择的，而是由"什么信息在什么时机被 Agent 消费"决定的。

### 7.6 候选生成管线：从观察到候选的完整 Pipeline

#### 🧠 直观理解

自成长的起点不是"模型灵光一闪"，而是一条**结构化的管线**：系统持续收集运行观察（observations），定期将一批观察送入分析 Agent，分析 Agent 产出候选（candidates），所有候选初始状态为 pending——等待人类决策。

```
运行观察 ──▶ 批量分析 ──▶ 候选生成 ──▶ pending 队列 ──▶ 人工审批 ──▶ 写回生效
```

#### 💻 完整代码：observations → analysis → candidates → pending

```python
# 文件名: 参考.md 第5节（本课内容）
# 功能: 从运行反馈中生成规则/技能候选——完整管线代码

growth_observations = [
    {"observation_id": "obs-001",
     "text": "北京任务运行记录：先用 route_checker 检查初稿，"
            "发现故宫和环球影城同日跨区通勤失败；把环球影城移到单独日期后通过。"},
    {"observation_id": "obs-002",
     "text": "西安任务运行记录：先用 route_checker 检查初稿，"
            "发现兵马俑和市区景点同日通勤失败；把兵马俑移到单独日期后通过。"},
    {"observation_id": "obs-003",
     "text": "北京任务结束时，用户明确要求以后输出行程时附上路线检查结论。"},
    {"observation_id": "obs-004",
     "text": "西安任务中，用户再次要求最终行程附上路线检查结论。"},
    {"observation_id": "obs-005",
     "text": "另一项目里，用户询问 86 元出租车发票是否可以报销。"},
]

# Step 1: 将观察送入分析 Agent，生成候选
growth_result = await (
    memory_agent
    .input({"observations": growth_observations})
    .instruct([
        "分析 observations 中是否存在值得沉淀的长期资产候选。",
        "反复出现且必须遵守的要求可归为 rule；"
        "由多个可执行步骤组成且结果可验证的方法可归为 skill。",
        "只返回有输入 observation_id 支持的候选，不判断候选是否生效。",
    ])
    .output({
        "candidates": [{
            "candidate_type": (str, "rule 或 skill", "not_null"),
            "summary": (str, "候选长期资产的紧凑描述", "not_null"),
            "evidence_refs": [(str, "支持候选的 observation_id", "not_null")],
            "validation_needed": [(str, "正式采用前需要的验证", "not_null")],
        }],
    })
    .async_start()
)

# Step 2: 所有候选初始状态统一标为 pending——模型无权让候选直接生效
available_growth_refs = {item["observation_id"] for item in growth_observations}
growth_candidates = [
    {**candidate, "status": "pending"}
    for candidate in growth_result["candidates"]
]

# Step 3: 校验：确保模型输出的 candidate_type 合法、status 全部为 pending
assert all(item["candidate_type"] in {"rule", "skill"}
           for item in growth_candidates)
assert all(item["status"] == "pending"
           for item in growth_candidates)
```

> ⚠️ **核心原则**：**模型生成 candidate 不等于规则已生效**。上述代码到 candidate 为止——`status: "pending"`，没有写入 memory_store。确认、测试、存储、召回全部由业务系统在人工审批之后负责。

| Pipeline 阶段 | 谁负责 | 产出 | 关键约束 |
|---|---|---|---|
| 收集观察 | 运行系统 | `observations[]` | 记录完整 task_id + 上下文 |
| 分析候选 | 分析 Agent | `candidates[]` | 每个候选必须挂载 evidence_refs |
| 标记 pending | 管线逻辑 | `status: "pending"` | 硬编码，不允许模型输出非 pending 状态 |
| 人工审批 | 责任人 | `approved` / `rejected` | 见 7.7 节 verify() |
| 写回存储 | 业务系统 | 正式写入 memory_store | 包含 conflict check（见 7.8 节） |

### 7.7 verify() 三态决策：自动写入的边界在哪里

#### 📖 详细解释

从 candidate 到正式规则，中间隔着一条关键的决策函数：`verify()`。它接收一个候选，返回三种决策之一：

- **write_back**：候选通过校验，可自动写入 memory_store
- **human_review**：候选需要人工审批（不通过时最常见的状态）
- **reject**：候选被拒绝（证据不足或逻辑不成立）

三种决策的核心边界是**候选类型**。preference 和 experience 类型在满足条件时可自动写回，但 rule 类型有一条铁律：

> 🚫 **硬约束：rule 类型绝对不能自动写入，无论 confidence 多高——必须同时满足 human_approved + evidence + test_passed 才允许 write_back。**

#### 💻 代码：三态决策 + 写回

```python
# 文件名: scripts/s04_verified_reflection_writeback.py
# 功能: 反思候选验证——三态决策函数 + 写回转换

from __future__ import annotations

from dataclasses import dataclass
from typing import Literal

CandidateType = Literal["preference", "experience", "rule"]
Decision = Literal["write_back", "human_review", "reject"]


@dataclass
class ReflectionCandidate:
    id: str
    type: CandidateType
    text: str
    scope: dict[str, str]
    evidence_refs: list[str]
    confidence: float
    test_passed: bool
    human_approved: bool = False


def verify(candidate: ReflectionCandidate) -> Decision:
    """三态决策：write_back / human_review / reject"""
    has_evidence = bool(candidate.evidence_refs)

    # rule 类型的硬约束：三者必须同时满足
    if candidate.type == "rule":
        if candidate.human_approved and has_evidence and candidate.test_passed:
            return "write_back"
        # rule 类型缺任何一个条件 → 强制人工审批，永不自动 reject
        return "human_review"

    # preference / experience 类型
    if not has_evidence:
        return "reject"           # 无证据 → 直接拒绝
    if candidate.confidence >= 0.75 and candidate.test_passed:
        return "write_back"       # 高置信 + 测试通过 → 可自动写回
    return "human_review"         # 其余情况 → 人工审批


def to_memory_record(candidate: ReflectionCandidate) -> dict[str, object]:
    """将通过验证的候选转换为正式 MemoryRecord"""
    return {
        "collection": "memory",
        "kind": f"verified_{candidate.type}",
        "scope": candidate.scope,
        "summary": candidate.text[:40],
        "body": {
            "text": candidate.text,
            "evidence_refs": candidate.evidence_refs,
            "confidence": candidate.confidence,
            "status": "active",
        },
    }


def main() -> None:
    candidates = [
        ReflectionCandidate(
            id="c-pref",
            type="preference",
            text="用户的亲子游需要控制步行量，"
                 "并在下午两点到四点安排午休。",
            scope={"tenant_id": "tenant-a",
                   "namespace": "user-42:family-travel"},
            evidence_refs=["evt_a05", "evt_b01"],
            confidence=0.86,
            test_passed=True,
        ),
        ReflectionCandidate(
            id="c-rule",
            type="rule",
            text="发布最终行程前必须附上路线检查结论。",
            scope={"tenant_id": "tenant-a",
                   "task_type": "family_travel"},
            evidence_refs=["evt_a12", "evt_b12"],
            confidence=0.91,
            test_passed=True,
            human_approved=False,        # ← 未审批的 rule，即使其他条件全满分
        ),
        ReflectionCandidate(
            id="c-no-evidence",
            type="experience",
            text="所有 5 岁儿童都不喜欢博物馆。",
            scope={"tenant_id": "tenant-a",
                   "task_type": "family_travel"},
            evidence_refs=[],             # ← 零证据，直接拒绝
            confidence=0.82,
            test_passed=True,
        ),
    ]

    for candidate in candidates:
        decision = verify(candidate)
        print(f"{candidate.id}: {decision}")
        if decision == "write_back":
            print("  ->", to_memory_record(candidate))


if __name__ == "__main__":
    main()

# 预期输出：
# c-pref: write_back
#   -> {'collection': 'memory', 'kind': 'verified_preference', ...}
# c-rule: human_review
# c-no-evidence: reject
```

#### 🤔 决策树：verify() 的完整决策路径

```
输入的 ReflectionCandidate
    │
    ├── type == "rule" ？
    │   ├── YES ──▶ human_approved AND has_evidence AND test_passed ？
    │   │           ├── YES ──▶ write_back       （三者缺一不可）
    │   │           └── NO  ──▶ human_review     （强制人工，永不自动 reject）
    │   │
    │   └── NO (preference / experience)
    │       │
    │       ├── has_evidence ？
    │       │   ├── NO  ──▶ reject               （无证据 = 不可信）
    │       │   └── YES
    │       │       ├── confidence >= 0.75 AND test_passed ？
    │       │       │   ├── YES ──▶ write_back   （高置信 + 测试通过）
    │       │       │   └── NO  ──▶ human_review （不足，需人工复核）
```

三条关键设计原则：

1. **rule 类型永不自动 reject**：即使没有证据，也不拒绝——因为否定一条潜在规则本身也是业务决策，应交给人。
2. **rule 类型必须三方验证**：human_approved（人）、has_evidence（数据）、test_passed（系统）三者缺一不可。
3. **preference/experience 不存在于真空**：没有 evidence_refs 的直接拒绝——confidence 再高也不行，因为置信度可能是模型的幻觉。

> ❓ **常见疑问**：为什么 rule 类型即使 confidence=1.0 也不能自动写回？
>
> **回答**：因为 rule 改变的是 Agent 的行为边界——一次错误的自动写回可能让 Agent 在此后的所有任务中做错事。preference 写错了影响单个用户的体验偏好，但 rule 写错了影响的是整个租户的所有任务。这是**影响面评估**驱动的设计决策，不是技术能力问题。模型完全有能力在技术上自动写回——但我们选择不这样做。

### 7.8 案例分析：Hermes Agent 的节流机制与审批门

#### 📋 背景

Hermes Agent 是一个实际落地了自成长管线的开源 Agent 框架。它的自成长系统并非"每轮都触发"，而是采用**可配置的节流机制**来避免过度触发和噪声干扰。

#### ⚠️ 核心设计：双层节流

| 触发类型 | 配置项 | 默认间隔 | 触发逻辑 |
|---|---|---|---|
| **Memory review** | `memory.nudge_interval` | 每 10 轮对话 | 会话钩子统计轮次达阈值 → 触发记忆回顾与分析 |
| **Skill review** | `skills.creation_nudge_interval` | 累计 10 次工具迭代 | 工具调用计数器达阈值 → 触发技能提炼分析 |

两种触发方式对应 7.4 节中的两种自成长模式：
- **会话级推动**：Memory review 通过 session 的会话钩子直接推动（轮次达阈值）
- **计数器驱动**：Skill review 通过工具迭代计数器触发（类似 amos 的做法）

#### ✅ 审批门机制

节流保证了"不频繁打扰"，审批门保证了"不错写入"：

| 审批配置 | 作用 | 效果 |
|---|---|---|
| `memory.write_approval` | 开启后，memory 修改先进入 pending 状态 | 所有记忆修改需要人工确认后才生效 |
| `skills.write_approval` | 开启后，可先查看 skill diff 再批准 | 技能变更可预览差异，逐项审批 |

#### 📊 审批门与 verify() 的对应关系

```
Hermes 审批门                   本课程 verify() 决策
─────────────────────────────────────────────────────
memory.write_approval     →    human_review 路径
  (pending 状态)                 (返回 human_review 决策)

skills.write_approval     →    write_back 后置校验
  (查看 diff 再批准)             (write_back + 审计日志)

无审批门（关闭）           →    write_back 直接生效
  (自动写入)                     (仅 preference/experience 且满足条件)
```

> 💡 **迁移启示**：自动发现候选与人工筛查不是二选一的对立关系，而是可以**按风险配置的两个串联阶段**。风险低的偏好类（preference）可放松审批门，风险高的规则类（rule）必须走人工审批——用 Hermes 的配置项 + verify() 函数即可实现精确到类型的审批策略。

### 7.9 "四个特定"框架：整节课的理论顶峰

#### 🧠 直观理解

老师用一个极其凝练的总结收束了整节课的所有内容——记忆、规则、自成长、Skills、三层压缩、向量召回——所有这些看似独立的技术点，本质上是同一个环形流程在不同环节的具体实现：

> **特定的触发时机 → 特定的信息压缩策略 → 修改特定的信息存储位置 → 在特定的使用时机反向取出信息 → 进入下一次模型请求。**

这不是四条独立的经验，而是一条**不可分割的闭环**。缺任何一环，"记忆"就不是记忆，只是"存了一份数据"。

### 7.10 最终洞见：万变不离其宗

#### 💡 信息压缩是一切的核心

老师反复强调整节课的终极结论：

> **万变不离其宗——不管是叫"自成长"也好，叫"任务记忆"也好，叫"人格记忆"也好，叫"行为模式"也好，本质就是信息压缩。然后把压缩后的信息放到 Agent 行为的特定路径上作为信息补充，给它回写回去。**

这个洞见分两个层面：

**第一层：压缩策略决定上限。** 自成长做得好不好，不取决于"管道搭得多漂亮"，而取决于"压缩策略好不好"——你怎么总结观察、怎么提炼规则、怎么去重冲突信息。压缩策略是所有这些技术的**能力瓶颈**。

**第二层：回写路径决定效果。** 压缩后的信息放在哪里、在什么时机被读取，决定了这个信息能否真正影响 Agent 的行为。放在 session memo 里（每轮读取）和放在 namespace memo 里（新任务初始化），影响范围和生效频率完全不同。

```

                          │                          │
                          │      信息压缩策略         │
                          │   (决定信息质量上限)       │

                          └────────────┬─────────────┘
                                       │
                                       │ 产生压缩后的信息
                                       ▼
                          ┌──────────────────────────┐
                          │                          │
                          │     特定路径回写          │
                          │   (决定信息实际效果)       │
                          │                          │
                          └────────────┬─────────────┘
                                       │
                    ┌──────────────────┼──────────────────┐
                    ▼                  ▼                  ▼
              ┌──────────┐      ┌──────────┐      ┌──────────┐
              │ session  │      │  rules   │      │  skills  │
              │   memo   │      │   memo   │      │   目录   │
              └──────────┘      └──────────┘      └──────────┘
              每轮读取           初始化读取          工具调用时读取
```

> 💡 **核心洞见**：所以，真正有壁垒的不是"我们做了记忆管理"或"我们做了自成长"——这些概念谁都可以提。真正的壁垒是你**怎么压缩的、怎么存储的、怎么召回的**。这三个问题的答案，构成了一个 Agent 的记忆系统与另一个之间的本质差距。

#### 🔀 Skills 自成长的具体做法

老师用 spec 管理的类比解释了 Skills 自成长的实际工作流：

1. **行为分析**：对用户的当前行为进行技能性总结（哪些步骤可复用？哪些判断可标准化？）
2. **用户确认**：询问用户哪些技能值得进入技能池
3. **冲突校验**：检查新技能与历史技能之间是否存在冲突（can_edit 判断）
4. **分支处理**：
   - 如果 `can_edit = True`（无冲突、可直接覆盖）→ 输出 `new_skill_memo`
   - 如果 `can_edit = False`（存在冲突）→ 返回 `question`，要求用户明确处理方式

这个流程本质上就是一个小型 Pipeline：`[origin_skill + new_content] → can_edit 判断 → merge or question`。它跟 spec 更新的逻辑完全一致——读一下当前版本、判断新要求是对原有内容的修改/覆盖/删除还是新建、发现冲突就提问。

> ⚠️ **注意事项**：Skills 自成长不等于"模型学会了新技能"——模型参数没有改变。Skills 自成长的本质是把好的执行轨迹（trace）压缩提炼成结构化指令，存入 skills 目录，下一次模型调用时通过 tools/skills 机制按需加载。模型本身还是那个模型，变化的是它可用的**信息武器库**。

## 八、答疑与课程总结

### 8.1 答疑：Skills 如何随使用次数变好

---

**🙋 问题 1**：把能力提炼为 Skills，而且随着使用次数增加，Skills 越来越好是如何做到的？

> 📖 **背景**：在「自成长策略」章节中，我们讨论了如何让 Agent 从任务执行过程中提炼经验并沉淀为可复用的行为规范。学员很自然地问出这个问题——沉淀之后，Skills 本身的质量如何持续提升？这条"越来越好"的路径是什么？

**💬 老师回答**：

本质是一个**小型 pipeline**——一个固定的工作流程，分成三个阶段：

**阶段一：行为分析**

对用户在任务执行过程中的行为进行"技能性总结"。模型观察当前任务的执行轨迹，从中提取出有复价值的模式——比如"每次遇到 A 类输入时，你做了 B 类处理，效果很好"。这些观察被提炼为粗略的候选技能描述。

**阶段二：询问筛选**

将候选技能呈现给用户：「以下技能值得放入技能池，是否确认？」这一步是**人工校验**的体现。在企业场景中，不建议让模型全自动地将自己的观察直接写入技能池，因为模型可能产生偏差——它认为有价值的东西，不一定符合业务实际。询问用户，相当于给技能录入加了一道质量门禁。

**阶段三：冲突校验与写回**

当新技能准备进入技能池时，需要校验它与历史技能是否冲突。这里可以类比 **spec 管理**的思维模式：

> 💡 **类比**：用过 Coding Agent（如 WebCoding）的同学会知道 spec 文件。spec 本身就是一个技能文档。当你说"修改一下 spec"，模型会怎么做？它打开 spec 文件读取内容，对比你的修改请求，判断这属于**覆盖、删除还是新建**。如果有冲突——"你现在说的跟以前我看到的 spec 不一样"——模型会反问你来确认。

Skills 更新跟 spec 更新没有本质区别。你可以设计一个专门的 **memory agent**：

- **input**：原始的 origin_skill（旧技能）+ 待更新内容（新的候选 skill）
- **output**：核心是一个 `can_edit` 字段
  - 如果 `can_edit = True`：说明新内容可以直接覆盖原始 skill，无冲突、不影响现有行为、无历史逻辑冲突 → 输出 `new_skill_memo`
  - 如果 `can_edit = False`：存在冲突，返回 `question`，列出冲突点，等待用户确认

每次用户使用系统，新的行为模式被捕获、新的技能描述被提出，经过这条 pipeline 后，技能池不断被**填补和完善**。久而久之，Skills 自然越来越好。

> 💡 **延伸**：`can_edit` 的判断逻辑可以进一步细化。比如对比新旧 skill 的 `scope` 是否重合、评估覆盖后是否会导致已有任务的执行路径发生变化。在更复杂的系统中，你甚至可以维护一个 `skill_version` 链，类似代码的版本管理，让 Skills 的进化过程完全可追溯。

### 8.2 答疑：长期记忆的遗忘策略

---

**🙋 问题 2**：长期记忆的遗忘策略是什么？如何处理过期和冲突的记忆？

> 📖 **背景**：在讨论了三层记忆架构和向量召回之后，学员自然想知道记忆如何"过期"。不遗忘的记忆系统会无限膨胀，不仅存储压力上升，召回精度也会因为噪音增加而下降。

**💬 老师回答**：

> ⚠️ **注意事项**：记忆遗忘这件事，业界目前**还没有统一标准**，仍在争论中。不像 Session 管理或 Task 压缩那样有成熟的范式，记忆的生命周期管理是一个开放问题。

尽管如此，实践中通常采用两个互补的策略：

**策略一：日期冷却（时间衰减）**

离当前时间越远的记忆，权重越低，越值得丢弃。这对应于遗忘曲线的基本直觉：一个月前的某个偏好，和今天的偏好发生冲突时，你当然会以新的偏好为更高权重。时间本身在记忆遗忘中作为一个**权重因子**。

具体实现上，通过 `expires_on` 字段给每条记忆指定一个过期日期。超过这个日期的记忆在召回时被过滤掉。

**策略二：任务隔离（Scope 过滤即遗忘）**

当你执行特定任务时，只需要加载与该任务 scope 匹配的记忆。那些 scope 不匹配的记忆虽然物理上还存在，但在本次任务上下文中不会被召回——这本质上就是一种**软遗忘**。你不需要在任何工作区召回所有信息，无关信息自然就"被遗忘"了。

| 遗忘策略 | 机制 | 本质 | 适用场景 |
|---|---|---|---|
| **日期冷却** | `expires_on` 字段，超过日期即过滤 | 硬遗忘，物理排除 | 临时性偏好、时效性事实 |
| **任务隔离** | `scope` 字段，不匹配 scope 不召回 | 软遗忘，逻辑隔离 | 领域性知识、任务专属偏好 |

> 🔗 **关联知识**：日期冷却和任务隔离并非互斥，两者通常组合使用。一条记忆先通过 scope 过滤（任务级），再通过日期过滤（时间级），两层筛选后才是真正参与本轮推理的记忆集合。

### 8.3 答疑：角色信息如何传给模型

---

**🙋 问题 3**：角色信息一般怎么传给模型？角色设定有什么用途？

> 📖 **背景**：在构建 Agent 时，学员需要知道角色信息（人设、语气、回答风格）应该放在请求的哪个位置。这个问题看似简单，但在实践中如果放错位置（比如放在 user message 而不是 system prompt），会导致角色遵循度下降。

**💬 老师回答**：

**最简单的传法：直接放在 system prompt 里。**

角色设定本质上就是一个普通的 system prompt。不管你用的是什么框架或 API：

- 可以直接设置 `system` 字段
- 可以使用 `role prompt`
- 也可以用最原始的写法：`set_agentic_prompt`，给 Agent 设定一个 agent prompt

这些写法本质上做的事情是相同的——在模型请求的最顶部注入一段角色描述。

**角色设定一般用来做什么？**

- **角色扮演**：你是一个资深的旅行规划师 / 严谨的代码审查员 / 幽默的生活助手
- **语气控制**：回答要正式还是口语化？要简洁还是详细？
- **回答风格**：先给结论再解释？偏好列表格式还是段落格式？是否主动追问？

> 💡 **延伸**：如果你对角色设定有很高的遵循度要求（比如法律合规场景中角色的行为边界不能有任何偏离），除了放在 system prompt 中，还可以将角色描述放入 **库（Workspace 文件）**中作为当次提示的一部分。双重注入能显著提高模型对角色约束的遵守程度。

### 8.4 答疑：Session 存储位置

---

**🙋 问题 4**：Session 的存储内容放在内存中吗？还是存在文件里？

> 📖 **背景**：在 Session 钩子机制中，我们讨论了 Session 的创建、销毁、resize。学员关心数据持久化的具体落点——如果服务重启，Session 数据还在吗？

**💬 老师回答**：

Session 的存储主要放在**内存**中。每次消息请求前，Session 管理模块从内存中取出当前会话的完整上下文，准备好后注入给模型。这个设计的目的是**速度**——会话信息需要高频读写，内存访问的延迟远低于磁盘 I/O。

但有一个重要的区分：**Session 级别 vs Task 级别**。

- **Session 级别的内容**（当前对话窗口、memo、历史消息）→ 主要在**内存**
- **Task 级别的内容**（任务中间产物、工作文件、阶段性输出）→ 可以存在 **Workspace 文件**里

Agent 能否读写 Workspace 取决于它是否拥有 workspace 权限。如果有权限，Agent 可以在 Workspace 中存取 Task 级别的数据；如果没有权限，就只能依赖内存中的 Session 数据。

> 🔗 **关联知识**：这跟下节课要讲的内容紧密相关——在企业长任务场景中，我们往往不满足于只把状态存在内存里。内存中的状态一旦进程重启就丢失了，所以我们还需要**运行时状态持久化**的方案，支持将任务进度写入外部存储，并在恢复时从持久化位置召回。详见第 22 课「主动任务发起与状态 Resume」。

### 8.5 答疑：冲突处理策略

---

**🙋 问题 5**：有冲突了，询问用户的逻辑是写在程序里的吗？如何判断是否可以直接覆盖？

> 📖 **背景**：在 Skills 更新和记忆写回的流程中，都会遇到"新内容和旧内容冲突"的场景。学员想知道这个判断逻辑是硬编码在程序中的，还是由模型来判断的。

**💬 老师回答**：

冲突处理逻辑既可以**写在程序里**，也可以**交给模型判断**——更准确地说，是程序定义判断框架，模型执行具体的冲突分析。

流程如下：

1. 当记忆更新 / Skills 更新被触发时，系统调用一个专门的 agent（可以叫 memory agent 或 skill update agent）
2. 这个 agent 接收两个 input：
   - **原始内容**（origin_skill / origin_memory）
   - **待更新内容**（new content）
3. Agent 的核心输出是一个 `can_edit` 字段：
   - `can_edit = True`：新内容可以直接覆盖旧内容。条件是无冲突、不影响现有行为、无历史逻辑矛盾
   - `can_edit = False`：存在冲突，返回 `question`，列出冲突详情和候选方案，等待用户确认
4. 程序根据 `can_edit` 的值做分支：
   - `True` → 直接执行写入
   - `False` → 将 `question` 呈现给用户，进入人工确认流程

> 💡 **核心洞见**：冲突判断的"智能"部分交给模型——它能理解语义层面的冲突（"你之前说偏好简洁回答，现在又说要详细解释"），单纯的字符串比对做不到这一点。而"流程控制"部分留在程序里——收到 question 后是弹窗询问还是写入待审队列，由业务逻辑决定。

| 策略 | 触发条件 | 处理方式 | 适用场景 |
|---|---|---|---|
| **新覆盖旧** | confidence 差值大、新信息明确、can_edit=True | 自动用新内容替代旧内容 | 偏好更新、事实修正 |
| **用户确认** | 两条内容 confidence 都高且存在语义冲突、can_edit=False | 返回 question，等待用户反馈 | 矛盾偏好、角色边界争议 |

### 8.6 记忆遗忘与更新的完整实现

#### 🧠 直观理解

每条记忆有三个"开关"控制它是否应该被丢弃：一个**过期日期**（到时间就淘汰）、一个**版本链**（有更好的版本了，旧版本自动失效）、一个**墓碑标记**（这条记忆本身就是错的，直接删除）。系统在召回记忆时，按固定顺序检查这三个开关，活下来的记忆才参与推理。

#### 📖 详细解释

**三字段详解：**

| 字段 | 类型 | 语义 | 生活类比 |
|---|---|---|---|
| `expires_on` | `date \| None` | 过期日期。超过此日期则不再召回 | "食品保质期到了，下架" |
| `supersedes` | `str \| None` | 被替代。指向被本记录替代的旧记录的 ID | "v2.0 发布了，v1.0 停止维护" |
| `tombstone` | `bool` | 墓碑标记。标记此记录为无效/错误 | "该商品存在质量问题，永久下架" |

**过滤管线的正确顺序：tombstone → supersedes → expires_on**

> ⚠️ **注意事项**：过滤顺序有明确优先级，不可调换。tombstone 是最强的排除信号——一条被标记为错误的记忆，不论是否过期、是否被替代，都不应该被召回。因此它必须最先判断。

为什么是这个顺序？

1. **先判断 tombstone**：如果记忆本身就是错的（标记为 tombstone），那后面的检查都没有意义。这是最强排除，一步否决。
2. **再判断 supersedes**：在剩下的有效记忆中，检查是否有旧版本被新版本替代。被替代的旧记忆不应出现。
3. **最后判断 expires_on**：前两步通过后，再检查日期是否过期。这是最弱的过滤条件，放在最后。

如果顺序错了——比如先检查 expires_on 再检查 tombstone——就会出现一条被标记为错误的记忆因为还没"过期"而被召回的问题。

#### 💻 完整代码

```python
# 文件名: scripts/s05_update_forget_policy.py
# 功能: 记忆遗忘的完整过滤管线，演示三字段过滤机制

from __future__ import annotations

from dataclasses import dataclass
from datetime import date, timedelta


@dataclass
class MemoryRecord:
    """单条记忆记录的数据结构。

    每条记忆由三个生命周期字段控制其有效性：
    - expires_on:  时间生命周期（过期日期）
    - supersedes:  版本生命周期（替代关系）
    - tombstone:   质量生命周期（有效性标记）
    """
    id: str
    text: str
    scope: dict[str, str]
    confidence: float
    expires_on: date | None = None      # 超过此日期即过滤
    supersedes: str | None = None       # 指向被本记录替代的旧记录 ID
    tombstone: bool = False             # True 表示该记忆已失效


def active_records(records: list[MemoryRecord], today: date) -> list[MemoryRecord]:
    """从全部记忆记录中过滤出当前有效的记录。

    过滤顺序（不可调换）：
    1. tombstone 过滤 —— 标记为无效的记录直接排除
    2. supersedes 过滤 —— 被其他记录替代的旧版本排除
    3. expires_on 过滤 —— 已过期的记录排除
    4. 输出按 confidence 降序排列
    """
    # 步骤1：收集所有被替代的记录 ID
    superseded_ids = {record.supersedes for record in records if record.supersedes}

    active: list[MemoryRecord] = []
    for record in records:
        # 过滤层1：墓碑标记 —— 最强排除，一步否决
        if record.tombstone:
            continue
        # 过滤层2：版本替代 —— 被新版本替代的旧记录排除
        if record.id in superseded_ids:
            continue
        # 过滤层3：日期过期 —— 超过 expires_on 的记录排除
        if record.expires_on is not None and record.expires_on < today:
            continue
        active.append(record)

    # 按置信度降序排列，置信度越高的记忆排在越前面
    return sorted(active, key=lambda item: item.confidence, reverse=True)


def main() -> None:
    """演示 active_records 的过滤效果。

    测试数据包含四种典型场景：
    - pref-v1: 被 pref-v2 替代的旧偏好
    - pref-v2: 替代 pref-v1 的新偏好（应保留）
    - summer-rule: 已过期的季节性规则（应过滤）
    - bad-memory: 标记为 tombstone 的错误记忆（应过滤）
    """
    today = date(2026, 7, 9)

    records = [
        MemoryRecord(
            id="pref-v1",
            text="亲子游不要安排得太满。",
            scope={"tenant_id": "tenant-a", "namespace": "user-42:family-travel"},
            confidence=0.80,
        ),
        MemoryRecord(
            id="pref-v2",
            text="亲子游要控制步行量，并在下午两点到四点安排午休。",
            scope={"tenant_id": "tenant-a", "namespace": "user-42:family-travel"},
            confidence=0.92,
            supersedes="pref-v1",  # v2 替代 v1
        ),
        MemoryRecord(
            id="summer-rule",
            text="7 月高温期间，中午十二点到三点不安排户外景点。",
            scope={"tenant_id": "tenant-a", "task_type": "family_travel"},
            confidence=0.75,
            expires_on=today - timedelta(days=1),  # 已过期
        ),
        MemoryRecord(
            id="bad-memory",
            text="所有 5 岁儿童都不喜欢博物馆。",
            scope={"tenant_id": "tenant-a", "task_type": "family_travel"},
            confidence=0.40,
            tombstone=True,  # 标记为无效
        ),
    ]

    print("active records for recall:")
    for record in active_records(records, today):
        print(f"- {record.id} | confidence={record.confidence} | {record.text}")


if __name__ == "__main__":
    main()
```

**运行结果：**

```
active records for recall:
- pref-v2 | confidence=0.92 | 亲子游要控制步行量，并在下午两点到四点安排午休。
```

四条记录中只有 `pref-v2` 通过全部过滤层：
- `bad-memory` → tombstone 过滤，排除
- `pref-v1` → 被 `pref-v2` 替代（supersedes 过滤），排除
- `summer-rule` → 已过期（expires_on 过滤），排除
- `pref-v2` → 通过所有检查，按 confidence=0.92 保留

#### 🔀 过滤管线流程图

```
全部 MemoryRecord 列表
        │
        ▼
  ┌─────────────────┐
  │ 过滤层1: tombstone │ ── True ──▶ 丢弃（记忆是错的）
  └────────┬────────┘
           │ False
           ▼
  ┌─────────────────┐
  │ 过滤层2: supersedes│ ── 被替代 ──▶ 丢弃（有更好版本）
  └────────┬────────┘
           │ 未被替代
           ▼
  ┌─────────────────┐
  │ 过滤层3: expires_on│ ── 已过期 ──▶ 丢弃（过了保质期）
  └────────┬────────┘
           │ 未过期
           ▼
  ┌─────────────────┐
  │ 按 confidence 排序 │
  └────────┬────────┘
           ▼
      active_records（参与召回）
```

> 💡 **核心洞见**：三层过滤的设计不是随意的。每一层对应记忆生命周期的一个维度——质量（tombstone）、版本（supersedes）、时间（expires_on）。它们按排除强度从强到弱排列，确保最强的排除条件最先生效，避免弱条件提前放过不应召回的记忆。

**题目**：为一段模拟对话实现 Session 压缩（保留 confirmed_constraints + open_questions）、Task 压缩（保留 source_ref）和 Namespace 压缩（去重合并 + evidence_refs）。

**验收标准**：
- [ ] analysis handler 正确判断压缩触发条件
- [ ] resize handler 返回正确的 (full_context, context_window, memo) 三元组
- [ ] Task 压缩中每条 fact 绑定 source_ref 且来自循环变量而非模型输出
- [ ] Namespace 压缩的 evidence_refs 全部在输入范围内

### 练习 2：实现向量存储与语义召回

**题目**：使用 SiliconFlow embedding 模型，实现写入（生成 record embedding）和召回（scope 过滤 + cosine 排序）的完整链路。

**验收标准**：
- [ ] memory_store 中所有记录向量维度一致
- [ ] 召回时 scope 过滤在 cosine 排序之前执行
- [ ] context_package 只含正文和分数，不含完整向量
- [ ] 不同 scope 的记录不会被错误召回

### 练习 3：实现记忆的遗忘过滤

**题目**：实现 `active_records` 函数，正确处理 `tombstone`、`supersedes`、`expires_on` 三种过滤条件，并在测试数据上验证过滤效果。

**验收标准**：
- [ ] 过滤顺序正确（tombstone → supersedes → expires_on），不可调换
- [ ] 被替代的旧记忆被正确排除
- [ ] 过期记忆被正确排除
- [ ] 输出按 confidence 降序排列
- [ ] 测试数据覆盖全部四种场景（正常保留、版本替代、过期、墓碑标记）
