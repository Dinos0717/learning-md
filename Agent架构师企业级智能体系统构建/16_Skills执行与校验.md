### 1.1 Skills系列课程的定位：从结构到规划再到执行

#### 🧠 直观理解

如果把Skills三节课比作"建造一座房子"：

- **第14课（Skill结构）**：认识砖瓦、钢筋、水泥——Skill由哪些组件构成（`SKILL.md`、`resources/`、`references/` 等）。
- **第15课（选择与规划）**：拿到建筑图纸，决定用哪块砖、从哪开始——在一堆候选Skill中命中合适的那个，并画出外层DAG规划图。
- **第16课（执行与校验）**：真正开工，把砖砌上去，每砌一层检查一次——选定Skill之后，怎么把任务完整地执行下来并验证每一步都有据可查。

今天这节课是Skills三部曲的最后一课，也是最"落地"的一课：上一课画的图，这一课要跑起来。

#### 📖 详细解释

三节Skills课解决的是同一个核心问题的三个不同侧面：**如何让系统真正"用"一个Skill**。

**第14课**回答了"Skill长什么样"。按照Anthropic官方的Skill标准，一个Skill包含核心文件 `SKILL.md`（定义Skill的元数据、触发条件、执行标准）、资源目录 `resources/`（脚本、工具、模板等）、参考目录 `references/`（知识文档、案例等）。这一课建立了对Skill物理结构的完整认知。

**第15课**回答了"用哪个Skill"。当一个用户请求到达时，系统面对的可能不止一个候选Skill。第15课的核心是：如何在一堆Skill中做选择（命中机制），以及当选中的任务需要跨越多个Skill时，如何在外层用DAG（有向无环图）规划出多个Skill之间的执行顺序。第15课的核心贡献是"选择"与"任务级规划（外层DAG）"。

**第16课**回答的是"选中了之后怎么办"。这才是执行的核心战场。一个Skill被命中之后，**不是把 `SKILL.md` 整段文本塞给模型就完事了**——那只叫"读了说明书"，不叫"执行了任务"。真正的执行要做七件事：渐进式发现和加载、准备运行时上下文、能力门控、制定执行计划（step-DAG）、按计划执行并取证、证据验收、交付最终结果。

> 💡 **核心洞见**：第15课画图（选中哪个Skill、多Skill之间怎么排），第16课跑图（单Skill内部怎么一步一步执行并校验）。两节课合在一起才构成一个完整的"选中→规划→执行→校验"闭环。如果只有选择没有执行，图就只是图；如果只有执行没有选择，你连该跑哪个Skill都不知道。

### 1.2 单Skill执行 vs 多Skill外层DAG：两种执行范式

#### 🧠 直观理解

想象你要做一顿晚餐：

- **单Skill场景**：你说"我想做一道红烧排骨"。你找到了红烧排骨的菜谱（Skill命中），接下来只需要照着菜谱一步一步执行就行。按顺序备料、焯水、炒糖色、炖煮、收汁——每一步都在一本菜谱内部完成，不需要去查阅米饭的菜谱或汤的菜谱。需求本身就是单一道菜。

- **多Skill场景**：你说"我想做一桌宴席，四菜一汤"。这时候你不能再闷头盯着一本菜谱了——你需要先规划：哪道菜耗时最长先炖上、哪些可以并行（凉菜先拌好放着）、最后统一上桌时热菜和冷菜怎么配合。你需要在所有菜谱之上画一张"宴席调度图"——这就是外层DAG。

两种场景都是合理的，它们的区别不在于谁"更高级"，而在于**任务本身的复杂度**。你不需要为煮一碗面画DAG，但为十人宴席画DAG就是必要的。

#### 📖 详细解释

从系统设计角度看，单Skill执行和多Skill外层DAG代表了两种不同的架构思路，它们回答的是不同层级的问题。

**单Skill执行（第16课主线）**：用户请求 → 系统命中一个Skill → 系统将Skill与可用工具**扁平化一体看待** → 进入Skill内部的执行链路。这种思路的核心假设是：大多数用户请求只需要一个核心Skill就能完成，复杂度集中在Skill内部（如何把一件事做对、做透），而非Skill之间（如何协调多件事）。

**多Skill外层DAG（第15课重点）**：用户请求 → 系统识别出多个子任务 → 每个子任务对应不同的Skill → 外层DAG规划Skill之间的依赖和顺序 → 逐个Skill内部执行。这种思路的核心假设是：复杂任务天然需要多个专业Skill协作，规划的重点在Skill之间。

> ⚠️ **重要纠正：上节课DAG方案没有问题**

上节课讲的DAG规划方案**本身没有问题**——它是一个面向复杂多任务场景的优化方案。老师在录音中也专门做了澄清：DAG方案更适合去做更复杂流程的规划，比如用户同时要求"规划行程 + 发送通知 + 订酒店"这种跨多个Skill的场景。但如果是"只规划一个杭州一日游"这样的单Skill场景，我们可能直接选择把Skill和工具扁平化地看作一体，深入Skill内部去执行——这是另一种思路，不代表DAG方案有问题。

这个纠正的核心在于：**两种方案适用于不同的任务复杂度，它们不是对错关系，而是场景选择关系**。

#### 🔀 对比辨析

| 维度 | 单Skill执行（第16课） | 多Skill外层DAG（第15课） |
|---|---|---|
| **适用场景** | 单一明确任务（如"规划杭州一日游"） | 复合多步骤任务（如"规划行程 + 发通知 + 订酒店"） |
| **Skill数量** | 1 个 | 多个 |
| **规划层级** | Skill内部 step-DAG（内层） | 任务级 DAG（外层）+ 各 Skill 内部 step-DAG（内层） |
| **核心挑战** | 怎么把单个Skill执行透彻、每一步有据可查 | 怎么把多个Skill串联起来、不冲突不遗漏 |
| **课程焦点** | 执行深度：一条链路上每一步的"怎么做"与"怎么验" | 规划广度：多个Skill之间的依赖关系与调度顺序 |

两种范式不是互斥的，而是互补的。一个成熟系统往往需要同时支持两者：简单任务走单Skill快速通道，复杂任务走外层DAG规划通道。今天的课聚焦前者——把一条单Skill执行链路做到极致。

#### 🔗 推导链：为什么在单Skill场景下选择"扁平化"思路

```
起点: 上节课提出外层DAG方案用于多Skill协同规划
  │
  ├── 第1步: 观察大多数实际用户请求的形态
  │   为什么？日常对话中，用户倾向于一次表达一个核心意图。
  │   像"帮我规划杭州一日游"这种请求，一个 travel-planner Skill
  │   就能完整覆盖。复合意图（如"规划 + 通知 + 订酒店"）通常需要
  │   多轮对话才能澄清，是少数场景。
  │
  ├── 第2步: 单Skill场景才是高频场景
  │   为什么？因为Skill本身就是按"完成一类任务"的粒度封装的。
  │   如果一个请求跨越多个Skill，要么是用户一次性说了太多事
  │   （可以在对话层拆解），要么是Skill粒度定义得过细
  │   （可以在设计层合并）。在单Skill场景中，不需要外层DAG。
  │
  ├── 第3步: 把Skill和工具"扁平化一体看待"
  │   为什么？在单Skill场景中，执行的所有步骤都在同一个Skill的
  │   语义空间内。SKILL.md 本身就会描述"做这件事需要哪些步骤"，
  │   模型只需要根据这些描述和可用工具生成一张 step-DAG（内层DAG）
  │   即可，不需要在多个Skill之间再做一层规划。把精力集中在
  │   "执行深度"而非"规划广度"上，是更务实的选择。
  │
  └── 结论: 单Skill场景下，Skill和工具扁平化看待，执行链路是
       "发现→准备→门控→规划→执行取证→验收→交付"；复杂多Skill
       场景下，外层DAG串联多个Skill，每个Skill再走自己的内层链路。
```

> ❓ **常见疑问**："那今天的课是不是就不讲DAG了？"
>
> **不是。** 今天的课会讲DAG，但指的是**内层DAG（步骤级）**——模型根据 `SKILL.md` 为当前任务生成一张步骤间的有向无环图，然后交给 TaskDAG 引擎按拓扑序执行。这与上节课的外层DAG（任务级）是不同层级的规划工具：
> - **外层DAG**：协调多个Skill（如 travel-planner → notifier → booking），是"Skill之间的图"
> - **内层DAG**：协调单个Skill内的多个步骤（如 查天气 → 算预算 → 排景点 → 生成方案），是"步骤之间的图"
>
> 本课后续会仔细展示内层DAG的生成与执行。

### 1.4 单Skill执行链全景：七步链路总览

#### 🧠 直观理解

Skill 执行不是把 `SKILL.md` 整段文本塞给模型然后等模型写答案。那样只是"读了说明书"——模型输出一段看似合理但**没有证据链**的文本，你不知道它哪些地方是认真算的、哪些地方是随口编的。

真正的 Skill 执行，应该像一次**有检查清单的手术**：

- **术前**：读说明书（`SKILL.md`）、确认器械齐不齐（能力门控）、画手术路线图（step-DAG）
- **术中**：按步骤操作，每步记录证据——工具的返回、计算的结果、模型的分析判断
- **术后**：对照检查清单逐条核验（证据验收），没通过的地方回去补

这七步链路，就是单Skill执行的"手术流程"。它确保最终交给用户的不是一段"看起来不错"的文字，而是一份**可以溯源、可以复查、不通过可以自动修正**的任务完成报告。

#### 📖 详细解释：核心定位

本节课的核心定位用一句概括：

> **Skill 执行不是让模型背说明书，而是让系统按标准 Skill 格式组织一次有证据的任务完成过程。**

拆解这句话的三个关键信息：

1. **"不是让模型背说明书"**：把 `SKILL.md` 全量塞给模型让它"理解并输出"是脆弱的做法。模型可能忽略关键约束、可能幻觉、可能跳过复杂步骤。说明书是给人（或系统）参照的，不是让模型背诵的。
2. **"有证据的任务完成过程"**：执行链路上的每一步都要产出**可查验的证据**。天气数据来自API调用（可溯源）、预算来自脚本计算（可复现）、景点推荐标注了选择理由（可审视）。没有证据的陈述就是猜想。
3. **"按标准Skill格式组织"**：利用 `SKILL.md` 中定义的 review 标准、执行约束、资源清单，让系统按 Skill 自己的标准来规范执行过程，而不是用一个通用模板套在所有 Skill 上。

```
                                   ▼ (验收通过)
┌──────────────────────────────────────────────────────────────────────┐
│  第7步: 交付结果 (Deliver)                                           │
│  ──────────────────────────────────────────                          │
│  • 将验证通过的结果 + 证据摘要 + 假设/风险标注整合为最终输出           │
│  • 按用户友好的格式呈现: 行程表、预算明细、注意事项                    │
│  → 详见 M08 模块                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

> ⚠️ **关键理解**：这七步链路不是一条单向流水线，而是一条**带闭环回路的校验链路**。第6步验收不通过时，会把反馈注入第5步重新执行，直到所有 review 标准通过为止。这是"执行-验证-修正-再验证"的循环，与软件工程中的"测试-修复-重测"是同一个逻辑。

#### 📖 模型与工具的分工

在这条七步链路中，有一条贯穿始终的设计原则：**模型和工具各司其职，交替协作**。

| 角色 | 负责什么 | 不负责什么 |
|---|---|---|
| **模型（LLM）** | 理解用户意图、阅读 `SKILL.md`、制定执行计划（step-DAG）、判断下一步做什么、组织最终交付文本、标注不确定内容 | 给出精确计算、获取实时数据、执行确定性操作 |
| **工具/脚本** | 查询天气API、计算预算、验证日期是否合理、执行网络请求、返回结构化数据 | 理解语义、做主观判断、组织最终交付文本 |

**为什么要这样分工？**

模型擅长"理解"但容易"幻觉"——你让它估计人均预算，它可能拍一个数（"差不多200吧"），但如果让脚本根据门票+交通+餐饮的实际价格去算，得到的就是一个可复现的精确值。反过来，工具擅长"精确"但不理解上下文——天气API返回"周六小雨"，它不会自动推断"雨天应该多排室内景点"。模型拿到天气数据后做这个推断，才是正确的分工。

> ⚠️ **注意事项**：模型和工具不是"你干你的我干我的"并行关系，而是在整条链路中**交替协作**：模型读完 `SKILL.md` 后决定"第3步需要查天气"→ 工具返回天气数据 → 模型根据天气数据调整行程 → 工具再次验证调整后的计划是否符合预算 → 模型根据验证结果做最终判断。这是一种"模型决策 → 工具验证 → 模型再决策"的循环协作模式。

## 二、渐进式披露——三层加载与按需选择资源

> 🎯 **本节目标**
>
> 学完本节后，你将能够：
> - [ ] 理解渐进式披露（Progressive Disclosure）为什么是 Skill 区别于 Function Calling 的核心特征
> - [ ] 掌握三层加载模型（元数据 → SKILL.md → 按需选资源）的每一层职责
> - [ ] 编写 `pick_resources()` 让模型按任务智能选择支持资源，而非用关键词匹配
> - [ ] 判断企业场景下"通用 Skill 消费逻辑"与"SOP Workflow"的适用边界

---

### 🧠 直观理解

渐进式披露（Progressive Disclosure）的核心思想可以用一个生活类比来理解：

> **你走进一座图书馆，不会一进门就把所有书架上的书全部搬到自己桌上。**
>
> 你首先会看**目录索引**（哪些书在这个馆里），然后找到感兴趣的**书架区域**，翻看**书的目录和简介**，最后才挑出真正需要的**那几本**带到阅读桌上仔细读。

Skill 系统的渐进式披露正是这个模式。它和 Function Calling、MCP Tool 的根本区别在于：Skill 提供的是**一整套信息**——不仅有"这个能力能做什么"的接口描述，还有"怎么用、什么时候用、用什么辅助材料"的行为指导。如果一次性把所有内容塞进上下文，不仅浪费 token，还会让模型被无关信息干扰。

> 💡 **核心洞见**：渐进式披露不只是"省 token"的优化手段，它本质上是为模型**减少噪音**。模型看到的无关材料越少，误用的机会就越少。

---

### 📖 详细解释

#### 三层加载模型

一个标准的 Skill 系统分三层加载，每一层有明确的职责边界：

```
┌──────────────────────────────────────────────────────────────────┐
│                     Skill 三层渐进式加载模型                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  第①层：元数据层（Metadata Layer）                                 │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ 存储内容：skill_id + name + description                     │  │
│  │ 触发时机：在所有 Skill 中做索引和匹配                         │  │
│  │ 核心动作：【索引阶段】用轻量元数据快速判断该 Skill 是否相关     │  │
│  │ 成本：极低（几十到几百 tokens）                               │  │
│  └────────────────────────────────────────────────────────────┘  │
│        │                                                         │
│        │ ✅ 命中：进入第②层                                       │
│        │ ❌ 未命中：跳过，不消耗后续资源                           │
│        ▼                                                         │
│  第②层：说明书层（SKILL.md Body Layer）                           │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ 存储内容：SKILL.md 正文（唯一必读说明书）                      │  │
│  │ 触发时机：元数据命中后                                        │  │
│  │ 核心动作：【加载阶段】读取完整 SKILL.md，理解行为指导           │  │
│  │ 内容包含：workflow、when to use、分支引导、reference 列表      │  │
│  │ 成本：中等（几千到上万 tokens）                                │  │
│  └────────────────────────────────────────────────────────────┘  │
│        │                                                         │
│        │ 从 SKILL.md 正文中发现 references / 可选资源列表          │
│        ▼                                                         │
│  第③层：支持资源层（Supporting Resources Layer）                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ 存储内容：references/、templates/、examples/、scripts/ 等     │  │
│  │ 触发时机：模型根据当前任务 + SKILL.md + 资源清单做判断          │  │
│  │ 核心动作：【按需选择】模型挑出本次任务真正需要的支持文件         │  │
│  │ 关键原则：AI 的判断力，不由代码写死规则                         │  │
│  │ 成本：可变（只加载选中的文件）                                 │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

#### 每一层的"为什么"

**第①层为什么只需要元数据？**
Skill 的数量可能很多（社区可能有几十上百个），如果每个 Skill 都先读 SKILL.md 再判断是否相关，开销太大。用 name + description 做第一道筛选，成本极低且足够有效——这一点在上一节课「Skill 如何被选中」中已经详细展开了。

**第②层为什么 SKILL.md 是"唯一必读说明书"？**
标准 Skill 格式里，`SKILL.md` 是唯一必须存在的文件。其他所有文件（`references/`、`templates/`、`examples/`、`scripts/`）都是可选支持资源。目录名只是提示（hint），不是协议字段——没有任何规范强制要求 references/ 里的文件必须被加载。真正决定"该读哪些"的是 SKILL.md 正文中的行为指导。

**第③层为什么必须交给模型判断？**
这是整节课最关键的设计决策。用户任务的措辞千变万化——"帮团队规划杭州一日游"和"查一下杭州有什么好玩的"虽然看似相近，但需要加载的支持资源可能完全不同。关键词匹配（比如"看到 references/ 目录就全读"）在简单场景也许够用，但面对真实世界的复杂任务，**理解"这次任务到底需要什么"是一个语义理解问题，不是字符串匹配问题**。这个活只能交给模型。

---

#### travel-planner Skill 的真实结构

以课程中使用的 `travel-planner` Skill 为例（这是一个社区真实贡献的 Skill，MIT License 开源），它的目录结构如下：

```
travel-planner/
├── SKILL.md                          ← 第②层：唯一必读说明书
├── LICENSE                           ← 非功能性文件
├── README.md                         ← 非功能性文件
└── references/                       ← 第③层：可选支持资源
    ├── planning-criteria.md          ← 规划标准（评分维度）
    ├── itinerary-style.md            ← 行程风格指南
    ├── output-template.md            ← 输出模板
    ├── travel-guidelines.md          ← 出行指南（注意事项）
    └── wanderlog-workflow.md         ← Wanderlog 工具专用流程
```

在这个结构里，`SKILL.md` 正文包含了行为指导的分支逻辑：什么时候使用 Wanderlog（immediately）、什么时候留在对话里（chat-only）。模型读了 SKILL.md 之后，会根据这些指导，结合当前任务，决定第三层要加载哪些 references。

> ⚠️ **注意事项**：`references/` 目录下的 wanderlog-workflow.md 虽然也在 references/ 中，但本次任务是"先看计划，先不用写进 Wanderlog"，模型正确地跳过了它。这恰好证明了**第③层不能靠"在 references/ 目录里就全读"的简单规则**——需要语义理解。

---

#### Reference 的双层判断嵌套

SKILL.md 正文中的 Reference 描述通常包含两层信息：

| 组成部分 | 内容 | 作用 |
|---|---|---|
| **前半段** | 文件路径（如 `references/planning-criteria.md`） | 告诉 Host 去哪里找到这个文件 |
| **后半段** | 描述（如"使用条件/何时引用"） | 告诉模型什么情况下应该读这个文件 |

这两层信息构成了一次"判断嵌套"：
1. **外层判断**（第①层）：元数据匹配 → 决定是否加载这个 Skill
2. **内层判断**（第③层）：Reference 描述匹配当前任务 → 决定是否加载这个具体文件

每一层都在用自己的判断逻辑缩小信息范围——这就是"渐进式"的本质。

### 🔀 对比辨析

#### 渐进式披露 vs 一次性全量注入

| 维度 | ❌ 一次性全量注入 | ✅ 渐进式披露 |
|---|---|---|
| **Token 消耗** | 所有 Skill 相关文件全部塞入上下文，不管任务是否需要 | 分层加载，只读与当前任务相关的部分 |
| **模型干扰** | 无关信息过多，模型容易被带偏，产生幻觉或误用 | 模型只看到任务相关的材料，判断更精准 |
| **扩展性** | Skill 包越大，上下文成本越高，天花板明显 | 支持超大 Skill 包（几十个 references），成本仅随选中文件线性增长 |
| **维护性** | 新增 resources/ 文件后，所有调用都自动加载（可能引入副作用） | 新增文件不影响已有调用，模型按需挑选 |
| **对 Skill 作者的要求** | 低——反正全塞进去 | 高——需要写好 description 和 reference 描述，模型才能正确判断 |
| **实现复杂度** | 低——读目录全部塞进去即可 | 中——需要实现分层加载逻辑和模型调用 |

> 🔗 **关联知识**：这与我们在 MCP 协议探讨中讨论的"Host 负责智能决策、资源端负责提供能力"的架构原则一脉相承。Skill 在这里只是"给 Host 做能力工业"，真正的智能判断在 Host 端。

---

#### SKILL.md 中的行为分支引导

`travel-planner` 的 SKILL.md 中包含类似这样的分支引导：

```
when to use:
  - immediately: 需要直接操作 Wanderlog 时 → 加载 wanderlog-workflow.md
  - chat-only: 只需要规划建议、不操作工具时 → 不加载 wanderlog-workflow.md
```

这种分支引导相当于 SKILL.md 的作者在告诉模型："我有这些能力，但你要根据实际情况选择用哪条路径。"模型读了这段之后，结合用户说的"先不用写进 Wanderlog"，自然就不会选 wanderlog-workflow.md。

> 💡 **核心洞见**：SKILL.md 不只是一份"说明书"，更是一套**决策树**。好的 Skill 作者会在 SKILL.md 中给出明确的分支条件，帮助模型做出正确的披露选择。

---

#### 企业场景讨论：通用 Skill 消费逻辑 vs SOP Workflow

录音中老师提出了一个值得深思的权衡问题：

| 方案 | 通用 Skill 消费逻辑 | SOP Workflow |
|---|---|---|
| **核心思路** | 做一套通用 Host，各个业务团队用自然语言编写 Skill.md 来表达业务逻辑 | 搭建流程相对固定的 SOP Workflow（agentic 的，节点内调用模型做判断） |
| **优点** | 灵活，业务团队可以自主用自然语言表达；新业务接入快 | 流程可控、可预测；每个节点的输入输出明确；不容易出现解析偏差 |
| **缺点** | 自然语言表达不规范，解析无法 100% 可靠；依赖 Skill 作者遵循标准格式 | 灵活性较低；新业务流程需要开发新节点 |
| **适用场景** | 业务逻辑变化频繁、团队需要自主表达的场景 | 工序严谨、对执行可靠性要求极高的场景 |
| **当前实操判断** | 如果 Skill 作者基本按官方文档格式写，可以尽最大努力解析 | 如果企业对流程一致性要求高，SOP Workflow 效果更稳定 |

关键洞察：**Skill 的设计哲学鼓励用自然语言表达**，这带来了极大的灵活性，但也带来了不可控性。我们能做到的是："如果这个 Skill 基本按照官方文档约束的格式结构去表达，我们尽量从里面解析出所需信息——但不能保证 100%。"

### 💻 代码示例

下面展示课程中实现的完整 `pick_resources()` 代码。这段代码演示了：在第①层（元数据）和第②层（SKILL.md）已经加载完成后，如何让模型完成第③层——按任务选择支持资源。

```python
# 文件名: pick_resources.py
# 功能: Skill 渐进式披露第③层——让模型按当前任务从可选资源中挑选真正需要加载的文件

import sys
sys.path.insert(0, "scripts")

from agently import Agently
from skill_runtime import configure_model, load_standard_skill, TRAVEL_PLANNER_DIR as SKILL_DIR

print("model_provider :", configure_model())
agent = Agently.create_agent()
# load_standard_skill() 完成两件事：
#   1. 读取 SKILL.md 正文（第②层）
#   2. 枚举 Skill 目录中的可选支持资源，构建 resource_catalog
package = load_standard_skill(SKILL_DIR)

task = "帮团队规划一个杭州一日游（本周六、非周一），约 10 人，人均 300 元以内含门票午餐市内交通，含午餐、尽量户外；先看计划，先不用写进 Wanderlog。"

# ── 核心代码（展示用）：第 3 层"该读哪些支持资源"是理解任务的活，交给模型挑（不靠目录名/关键词猜）──
def pick_resources(agent, task, package):
    r = (agent.create_request()
        .input(task)
        .info({"skill_md": package.skill_md,
               "available_resources": package.resource_catalog()})       # 目录名只是提示，不是协议字段
        .instruct(["按这次任务 + SKILL.md 选真正要进一步读取的支持文件",
                   "不要因为文件在 references/ 就机械全选；与本次任务无关的不要选"])
        .output({"needed": ([str], "这次要读取的支持资源路径", True)})
        .start())
    # 安全校验：只保留 resource_catalog 中确实存在的路径，防止幻觉
    catalog = {x["path"] for x in package.resource_catalog()}
    return [p for p in r["needed"] if p in catalog]

# 渐进式披露三层：① 元数据先索引 → ② 命中后才读 SKILL.md → ③ 用到才让模型挑支持资源
print("① 元数据   :", package.skill_id, "—", package.description[:38], "…")
print("② SKILL.md :", f"已读，正文 {len(package.skill_md)} 字（唯一必读说明书）")
all_resources = [r["path"] for r in package.resource_catalog()]
needed = pick_resources(agent, task, package)
print("③ 支持资源 :", f"共 {len(all_resources)} 个可选；模型这次只挑要读的 →", needed)
print("   这次跳过 :", [p for p in all_resources if p not in needed])
```

### ❓ 常见疑问

**🙋 问题 1：为什么不直接在代码里解析 SKILL.md 中的 reference 关键词来决定加载哪些文件？**

> 📖 **背景**：有同学可能会想——SKILL.md 里不是已经写了 reference 的描述吗？用正则匹配"planning"、"itinerary"等关键词，对应到文件路径不就行了？

**💬 回答**：

关键词匹配有四个致命缺陷：
1. **无法理解语义组合**：任务说"先看计划，先不用写进 Wanderlog"，关键词匹配能看到"plan"和"wanderlog"两个词，但它无法理解"不用"这个否定语义——结果就是仍然会加载 wanderlog-workflow.md。
2. **任务措辞千变万化**：同一个需求，用户可能说"帮我规划行程"，也可能说"周末想去杭州转转，帮我看看怎么安排"——靠关键词穷举不可能覆盖所有表达方式。
3. **自然语言表达不规范**：Skill 作者写 reference 描述时可能用不同的措辞，同义词、近义词、缩写——正则规则会越来越臃肿且不可维护。
4. **需要跨文件理解**：模型需要同时理解"任务想要什么"和"SKILL.md 指导了什么"，这是两个语义空间的映射——不是字符串匹配能做到的。

> 💡 **延伸**：这个问题触及了 Skill 系统设计的核心哲学——**Skill 鼓励自然语言表达，而自然语言的解析本身就应该是 AI 的强项**。把解析工作交给模型，而不是写死正则规则，是这个系统设计的应有之义。

---

**🙋 问题 2：模型会不会产生幻觉，选了一个不存在的文件路径？**

**💬 回答**：

会，而且这是需要防御的。代码中已经包含了防御措施：

```python
catalog = {x["path"] for x in package.resource_catalog()}
return [p for p in r["needed"] if p in catalog]  # 只保留确实存在的路径
```

模型返回的路径列表会经过一次"白名单过滤"——只有 `resource_catalog()` 中确实存在的路径才会被保留。这个做法和 MCP 中校验工具调用的思路一致：**用代码做最后的边界守卫**。

> 💡 **延伸**：这其实引出了下一模块的内容——"能力门控"（Capability Gating）。加载了正确的资源只是第一步，系统还需要验证这些资源在当前运行时环境中是否真的可用。详见 M03。

---

**🙋 问题 3：如果 SKILL.md 的作者没有按标准格式写 reference 描述，解析不出来怎么办？**

**💬 回答**：

这正是录音中老师花了不少篇幅讨论的"格式依赖"问题。坦白说，**没有银弹**。

当前的约定是：
- 如果 Skill 作者基本遵循官方文档的格式约束，我们尽最大努力解析出所需信息
- 但不能保证 100%——因为自然语言表达天然存在不规则性

这也是为什么在企业场景中，老师提出了"通用 Skill 消费逻辑 vs SOP Workflow"的权衡——如果你的业务对执行可靠性有极高要求，可能 SOP Workflow 是更稳妥的选择。Skill 的设计哲学是"灵活优先"，它牺牲了一定的确定性来换取表达自由。这是设计取舍，不是缺陷。

## 三、能力门控——运行时能力与Skill场景的匹配

> 🎯 **本节目标**

### 3.1 能力门控（Capability Gate）是什么

#### 🧠 直观理解

想象你要自驾去一个从来没去过的地方。出发前你会做什么？不是直接踩油门，而是打开 Checklist 检查一遍：GPS 导航装好了吗？油箱加满了吗？轮胎气压够吗？

**能力门控就是"出发前的装备检查清单"**。Skill 选好了、说明书读完了，但在真正执行之前，必须回答一个关键问题：这个 Host（宿主环境）手上到底有没有 Skill 声称需要的那些能力？如果缺了关键能力就硬着头皮跑，要么返回编造的结果（幻觉），要么跑到一半撞墙报错。

#### 📖 详细解释

在 Skill 执行体系里，有一条基本原则需要反复强调：

> 💡 **核心洞见**：Skill 提供的是"指引"（guidance），实际能力由宿主提供。

举个例子：一个 travel-planner Skill 在它的 SKILL.md 里写着"用实时检索核实航班信息"。这句话只是告诉你"这次任务可能需要搜索能力"，但宿主到底能不能搜，Skill 自己说了不算——那是宿主环境的事实。

如果宿主没接 web_search，模型还照着 Skill 的指引去"实时检索"，结果只能是：要么编一个假结果（幻觉），要么坦承"我没法查"但已经浪费了一轮对话。门控机制就是要在这一切发生之前拦住它。

### 3.2 门控回答的两个关键问题

```
┌─────────────────────────────────────────────────────────────┐
│                     能力门控的两个维度                        │
│                                                             │
│  问题①：Skill 声明了需要什么能力？                              │
│     ↓                                                       │
│   从 SKILL.md + 当前任务中，由模型推理得出                      │
│                                                             │
│  问题②：宿主环境实际有什么能力？                                │
│     ↓                                                       │
│   从 host_capability_catalog（宿主能力目录）中读取               │
│   这是宿主"真实暴露"的事实，不靠推理，不靠猜测                    │
└─────────────────────────────────────────────────────────────┘
```

这两个问题必须分别回答，因为它们的来源完全不同：

- **问题①** 的答案藏在 SKILL.md 的正文和当前任务描述里，需要模型去阅读理解
- **问题②** 的答案来自宿主系统的 Action 管理中心，是已注册能力的硬事实

#### 能力从哪里来？——课堂互动问答

> 🙋 **学员提问**：`host_capability_catalog` 这个能力目录到底是怎么产生的？是从 Skill 里面推导出来的吗？

> 💬 **老师回答**：完全不是。`host_capability_catalog` 是宿主"真实暴露"的能力目录。它来自 Action 管理中心的已注册能力列表，跟 Skill 没有任何推导关系。举个例子：你的宿主系统接入了 `web_search` API、`wanderlog_mcp` 插件、`file_system` 读写权限，那么目录里就有这三项。如果你今天没接 wanderlog_mcp，那目录里就不应该有它。Skill 只是"建议"你可能需要什么，但有没有、能不能用，是宿主说了算。

> 🔗 **关联知识**：这个设计思路承接了第 12 课「受控执行环境与沙盒设计」中关于能力注册的思想——宿主的可用能力是一个明确的、可枚举的集合，不能靠模型"猜想"宿主演进出了什么新功能。

### 3.3 required vs optional：两种能力需求级别

Skill 声称需要的能力并不都是"缺了就转不动"的。门控机制要求区分两个等级：

| 级别 | 语义 | 缺了以后的后果 | 典型场景 |
|---|---|---|---|
| **required**（必需） | 没有它就不能诚实地完成本次任务 | **blocked**：终止执行，不硬跑 | travel-planner 的用户要求查实时天气，但宿主没有 web_search |
| **optional**（可选） | 有了会更好，但没有也不影响核心交付 | **degraded**：降级运行，计划照出但注明"此步骤无法执行" | 用户希望将计划写入 Wanderlog，但宿主当前未接入该 MCP |

> ⚠️ **注意事项**：`optional` 不等于"强行塞进计划只为展示 degraded"。如果用户明确说了"这次不用写入外部系统"，那该写入能力根本不应被列为需求。降级是由真实的 Skill 指引产生的、合理的"可让步"项，不是炫技用的。

### 3.4 为什么门控要设计成两层架构

这是本节最重要的设计决策。门控不是一个单一函数能完成的，它被刻意拆成两段：

```
┌──────────────────────┐     ┌──────────────────────┐
│  阶段一：能力推断       │     │  阶段二：确定性门控       │
│  infer_capability_needs │ ──▶ │  evaluate_capabilities  │
│                       │     │                       │
│  执行者：大模型          │     │  执行者：代码逻辑         │
│  输入：SKILL.md + 任务   │     │  输入：阶段一输出 +       │
│  输出：能力需求清单       │     │        host_catalog    │
│  (标好 required/opt)   │     │  输出：ready/blocked/    │
│                       │     │        degraded         │
└──────────────────────┘     └──────────────────────┘
```

#### 阶段一：模型推断需要什么能力（为什么不用写死清单）

如果你在代码里写死一个 `["web_search", "wanderlog_mcp", "image_gen"]` 的清单，那每新加一个 Skill，你就得手动改一次——这完全违背了 Skill 体系"通用解释器"的初衷。

> 💡 **核心洞见**：Skill 是动态插入的，能力清单也应该动态生成。让模型去读 SKILL.md，理解这个 Skill 的意图，从中抽取出它真正依赖的能力——这才是可扩展的做法。

模型在这一步拿到的信息有三样：
1. **Skill 的名称**（如 `travel-planner`）
2. **Skill 的描述**（如"规划旅行路线，核实行程可行性"）
3. **SKILL.md 的正文内容**（Skill 内部指引的完整文本）

有了这三样，模型就能理解这个 Skill "声称自己能做什么"，从而判断需要依赖哪些宿主能力。注意：这一阶段不告诉模型宿主当前有什么能力——那个留给阶段二。

#### 阶段二：代码确定性检查能力是否存在（为什么不用模型判断）

如果你让模型同时回答"需要什么"和"现在有什么"，会引入两个致命问题：

1. **幻觉风险**：模型可能幻想出一个宿主并不存在的能力，然后说"有"——实际上没有
2. **不一致性**：两次询问同一个问题可能得到不同答案，因为模型是概率性的

所以阶段二必须用确定性代码。规则极简：

```text
IF 某项能力的 requirement == "required" AND 宿主没有 → blocked
IF 某项能力的 requirement != "required" AND 宿主没有 → degraded
IF 所有 required 能力都存在 → ready
```

> ⚠️ **注意事项**：阶段二不调用任何大模型。它是一个纯 Python 函数，输入是结构化 JSON，输出也是结构化 JSON。老师特别强调——门控结果最好是结构化的。将来出了问题（比如"为什么没查天气"），你不需要去翻聊天记录猜原因，直接看结构化输出里的 `blocked` 字段就知道是 web_search 没接上。

### 3.5 💻 代码示例：完整的门控实现

以下是课件中的完整可运行代码，展示了从能力推断到确定性门控的完整流程。

```python
# 文件名: capability_gate.py
# 功能: 能力门控——两阶段架构的完整实现
# 依赖: agently, skill_runtime（本课程配套运行时）

from agently import Agently
from skill_runtime import CLASSROOM_HOST_CAPABILITY_CATALOG   # 宿主"真实暴露"的能力目录

# ── 核心代码①：模型从 SKILL.md + 任务抽取能力需求，标 required/optional（不写死清单）──
def infer_capability_needs(agent, task, package):
    """阶段一：让模型理解 Skill 意图，推断需要哪些宿主能力。

    输入：
        agent: 已配置好的 Agently Agent 实例
        task: 当前任务描述（字符串）
        package: 已加载的 Skill 包对象，包含 skill_md 等属性

    输出：
        结构化列表，每项含 capability / requirement / reason
    """
    r = (agent.create_request()
        .input(task)
        .info({"skill_md": package.skill_md,
               "host_capability_catalog": CLASSROOM_HOST_CAPABILITY_CATALOG})
        .instruct([
            "从 SKILL.md 与当前任务判断这次真正需要哪些宿主能力，不要写死的专用清单",
            "有等价能力就用 host_capability_catalog 里的 id；没有就用描述性名称",
            "required = 没它就不能诚实完成；optional = 可降级或用户本次没要求",
        ])
        .output({"needs": ([{"capability": (str, "能力 id 或描述名", True),
                             "requirement": (str, "required 或 optional", True),
                             "reason": (str, "为什么这次需要", True)}], "需要的宿主能力", True)})
        .start())
    return r["needs"]


# ── 核心代码②：host 确定性门控。required 缺→blocked，optional 缺→degraded（用代码，不靠模型）──
def evaluate_capabilities(needs, env):
    """阶段二：用确定性代码检查能力齐备度，不调用大模型。

    输入：
        needs: infer_capability_needs() 的输出列表
        env: 宿主当前可用的能力字典，key 为能力 id，value 为 "available" 或 "missing"

    输出：
        结构化字典 {"status": "ready"|"blocked", "blocked": [...], "degraded": [...]}
    """
    missing = lambda n: env.get(n["capability"], "missing") != "available"
    blocked  = [n["capability"] for n in needs if n["requirement"] == "required" and missing(n)]
    degraded = [n["capability"] for n in needs if n["requirement"] != "required" and missing(n)]
    return {"status": "blocked" if blocked else "ready", "blocked": blocked, "degraded": degraded}


# ── 现场跑（复用之前的 agent / task / package）──
needs = infer_capability_needs(agent, task, package)
print("模型抽取的能力需求：", [(n["capability"], n["requirement"]) for n in needs])

# 场景一：能联网，没接 Wanderlog → ready（required 能力具备）
print("能联网、没接 Wanderlog ->", evaluate_capabilities(needs, {"web_search": "available"}))

# 场景二：连实时检索都没有 → blocked（缺 required 能力，不能诚实完成任务）
print("连实时检索都没有       ->", evaluate_capabilities(needs, {}))
```

#### 运行输出

```
模型抽取的能力需求： [('web_search', 'required')]
能联网、没接 Wanderlog -> {'status': 'ready', 'blocked': [], 'degraded': []}
连实时检索都没有       -> {'status': 'blocked', 'blocked': ['web_search'], 'degraded': []}
```

#### 输出解读

| 场景 | 宿主能力状态 | 门控结果 | 含义 |
|---|---|---|---|
| 场景一 | web_search 可用 | `ready`，blocked / degraded 均为空 | required 能力具备，可正常执行 |
| 场景二 | 没有任何能力 | `blocked`，blocked 列表含 web_search | required 能力缺失，不能诚实完成"查实时事实"类任务，应终止执行 |

> 💡 **核心洞见**：场景一中即使没接 Wanderlog，degraded 列表也是空的——因为模型在这次任务中判断 Wanderlog 并非必需。degraded 列表只记录那些"被 Skill 声称需要、模型判断为 optional、且宿主确实缺失"的能力。如果模型根本没把它列为需求，它就不会出现在任何列表里。

### 3.6 🔀 关键设计决策汇总

#### 决策一：信息分离——告诉模型能力目录，但不告诉它"当前可用性"

在阶段一，`.info()` 方法里传入了 `host_capability_catalog`（有什么能力可选），但**不传当前可用状态**。为什么？

因为如果让模型在阶段一就知道"web_search 现在不能用"，它可能产生"讨好式省略"——直接把本该 required 的能力标成 optional，以此避免 blocked 状态。我们要的是诚实的评估：Skill 说需要什么，模型就如实报告需要什么。能不能用是阶段二的事。

#### 决策二：evaluate_capabilities 返回结构化输出

```python
{"status": "blocked", "blocked": ["web_search"], "degraded": []}
```

这不是随手写的字典。三个字段各有明确的消费方：

| 字段 | 含义 | 谁消费 |
|---|---|---|
| `status` | 顶层判断：`"ready"` 可继续 / `"blocked"` 应终止 | 执行引擎（决定是否进入下一步） |
| `blocked` | required 但缺失的能力列表 | 错误报告 / 日志系统（告知用户"为什么跑不了"） |
| `degraded` | optional 但缺失的能力列表 | 规划阶段（M05）生成计划时标注"此步骤因缺能力而跳过" |

> 💡 **核心洞见**：结构化输出意味着"出了问题不需要翻聊天记录"。传统的 agent 系统里，模型输出一段自然语言描述"抱歉，我无法完成这个任务，因为……"——你得手动解析这段话才能知道根因是什么。结构化门控让调用方（Host）可以直接判断、直接路由。

#### 决策三：required 语义保守设定

`required` 的判断门槛设得很高——"没它就不能**诚实**完成"。注意是"诚实"，不是"完成"。如果模型可以编一个天气数据糊弄用户，技术上也算"完成"了——但那不是我们想要的行为。门控的目的是保证 Skill 的指引被**真实地**执行，而不是被**表面地**满足。

## 四、门控实战调优——Bug发现与实时修正实录

### 🧠 直观理解

前面我们建立了门控的基本框架：模型从 Skill 说明书中抽取能力需求，宿主系统做确定性判断——required 缺了 blocked，optional 缺了 degraded。逻辑通顺，框架完整，看起来没什么问题。

直到课堂上有同学举手说了一句话，让整个设计瞬间露出了一个逻辑裂缝。

> "老师，你给模型的 capability 列表里都是已有的能力，让它从已有列表里选，那门控不就形同虚设了吗？"

这一句话，把在场所有人——包括老师——都拉回了同一个起点：**我们被自己的"完美设计"绕进去了。**

接下来要展示的，不是提前准备好的排错教案，而是一段真实发生在课堂上的即时 Bug 发现与修正过程。这段经历之所以珍贵，是因为它揭示了一个远比代码错误更深层的问题：**逻辑自洽不等于逻辑正确。**

---

### 🐛 排错实录：门控自查自证的逻辑闭环

#### ❌ 原门控设计

在修正之前，门控的工作方式是：将宿主当前已接入的能力列表（`host_capability_catalog`）原样喂给模型，让模型从其中勾选这次 Skill 需要哪些。

```python
# ── 原设计：infer_capability_needs（❌ 存在逻辑 Bug）──
def infer_capability_needs(agent, task, package):
    r = (agent.create_request()
        .input(task)
        .info({"skill_md": package.skill_md,
               "host_capability_catalog": CLASSROOM_HOST_CAPABILITY_CATALOG})
        .instruct([
            "从 SKILL.md 与当前任务判断这次真正需要哪些宿主能力，不要写死的专用清单",
            "有等价能力就用 host_capability_catalog 里的 id；没有就用描述性名称",
            "required = 没它就不能诚实完成；optional = 可降级或用户本次没要求",
        ])
        .output({"needs": ([{"capability": (str, "能力 id 或描述名", True),
                             "requirement": (str, "required 或 optional", True),
                             "reason": (str, "为什么这次需要", True)}], "需要的宿主能力", True)})
        .start())
    return r["needs"]
```

配合的确定性门控：

```python
# ── evaluate_capabilities（❌ 看似正确，但上游数据源有缺陷）──
def evaluate_capabilities(needs, env):
    missing = lambda n: env.get(n["capability"], "missing") != "available"
    blocked  = [n["capability"] for n in needs if n["requirement"] == "required" and missing(n)]
    degraded = [n["capability"] for n in needs if n["requirement"] != "required" and missing(n)]
    return {"status": "blocked" if blocked else "ready", "blocked": blocked, "degraded": degraded}
```

---

#### 🔍 根因分析：为什么这是一个逻辑 Bug

这里的 Bug 不在于 `evaluate_capabilities` 的代码写错了——它的判断逻辑完全正确。Bug 出在**上游数据流**。

我们来看数据是怎么流动的：

```
┌─────────────────────────────────────────────────────┐
│              host_capability_catalog                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │web_search│ │file_read │ │wanderlog │  ...       │
│  │available │ │available │ │ missing  │            │
│  └──────────┘ └──────────┘ └──────────┘            │
└──────────────────────┬──────────────────────────────┘
                       │ 原样喂给模型
                       ▼
┌─────────────────────────────────────────────────────┐
│              模型判断（从已有列表中选）                │
│                                                     │
│  "我需要 web_search (required)"                     │
│  "我需要 file_read  (optional)"                     │
│                                                     │
│  ⚠️ 问题是：模型只能从我们给的列表中选！              │
│  列表中全是 available 的能力！                       │
│  所以模型选出来的 → 一定是 available → blocked 永远为空│
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│         evaluate_capabilities：blocked = []          │
│                                                     │
│  "一切正常，出发吧！"                                 │
│  — 但实际上 Skill 可能需要的能力根本没在列表里        │
└─────────────────────────────────────────────────────┘
```

这就是学员指出的核心矛盾：**让模型从一个已经过筛选的集合里选东西，选出来的结果不会出现"缺失"，因为"缺失"的选项根本没有被放进这个集合。** 门控的 blocked 分支成了一段永远走不到的代码。

老师在听到这个问题后的第一反应是坦诚的——"我一样被绕进去了。"这句话不仅是对学生的尊重，也是一个重要提醒：**即使是经验丰富的开发者，在设计"模型+代码"混合系统时，也会在数据流的上下游衔接处掉进逻辑陷阱。**

---

#### 🛠️ 修正过程：三轮迭代调优

##### 第一轮：加入 missing 出口

修正的核心思路：**给模型一个"表达不满"的通道。** 不能让模型只在给定的能力列表中做选择题——它需要一个属于自己的空白处，写出"我需要但你没给我的"能力。

```python
# ── 修正：infer_capability_needs 增加 missing 出口（✅ 第一轮修正）──
def infer_capability_needs(agent, task, package):
    r = (agent.create_request()
        .input(task)
        .info({"skill_md": package.skill_md,
               "host_capability_catalog": CLASSROOM_HOST_CAPABILITY_CATALOG})
        .instruct([
            "从 SKILL.md 与当前任务判断这次真正需要哪些宿主能力",
            "有等价能力就用 host_capability_catalog 里的 id；没有就用描述性名称",
            "required = 没它就不能诚实完成；optional = 可降级或用户本次没要求",
            # ★ 新增关键指令
            "如果 SKILL.md 中描述的能力需求，在 host_capability_catalog 中找不到对应项，"
            "请在 missing 字段中列出这些能力，标注 required 或 optional",
        ])
        .output({"needs": ([{"capability": (str, "能力 id 或描述名", True),
                             "requirement": (str, "required 或 optional", True),
                             "reason": (str, "为什么这次需要", True)}], "需要的宿主能力", True),
                 "missing": ([{"capability": (str, "描述性名称", True),
                               "requirement": (str, "required 或 optional", True),
                               "reason": (str, "为什么需要但未在 catalog 中找到", True)}],
                             "Skill 需要但 catalog 中未找到的能力", True)})
        .start())
    return r
```

**为什么需要 missing？** 因为它打破了"自查自证"的闭环。有了 missing 出口，模型可以表达"这个 Skill 需要搜索能力，但你的 catalog 里没有"——这句话才有可能触发后续的 blocked 分支。

---

但在测试中马上发现了新问题：模型开始发散。它会列出一些 Skill 里根本没有提到、但模型"觉得可能需要"的能力。比如一个旅行规划 Skill，模型可能会在 missing 里填上"天气预测能力""汇率转换能力"，而这些在 Skill 的说明书里从未出现过。

这就引出了第二轮修正。

##### 第二轮：限制 missing 必须在 Skill 范围内寻找

```python
.instruct([
    # ... 前面的指令保留 ...
    # ★ 新增约束
    "missing 中列出的能力必须严格来源于 SKILL.md 中描述的需求，"
    "不允许模型自行推断或添加 Skill 说明书之外的能力需求。"
    "换句话说：missing 只是指出 'SKILL.md 说需要 X，但 catalog 里没有 X'，"
    "而不是 '我猜你可能还需要 Y'。",
])
```

这轮修正的关键在于**划定边界**。missing 的角色不是让模型自由发挥，而是让模型充当一个诚实的"对照者"：把 SKILL.md 里的要求，和 catalog 里的条目，一一对照，指出哪些对不上。

这一轮之后，missing 的输出变得精准了。但还有一个问题：当下游 `evaluate_capabilities` 需要做 blocked/degraded 判断时，它不仅要看 `needs` 还要看 `missing`，而且还要判断"missing 为空"到底意味着什么——是真的不需要额外能力，还是模型忘了写？为了让下游消费更清晰，引入了第三轮修正。

##### 第三轮：加入 no_missing_can_work flag 信号

```python
.output({"needs": (...),
         "missing": (...),
         # ★ 新增 flag 信号
         "no_missing_can_work": (bool,
             "True 表示：根据 SKILL.md 的要求和当前 catalog 的覆盖情况，"
             "即使没有任何 missing 的能力，当前任务也能诚实完成。"
             "False 表示：missing 列表中至少有一个能力是完成任务所必需的。"
             "下游可直接用此 flag 做快速判断，无需再解析 missing 列表",
             True)})
```

这个 flag 的价值在于**给下游提供了一个降维判断信号**。调用方不需要自己解析 missing 列表里的 `required`/`optional` 字段——直接看 `no_missing_can_work` 就能做出决策：

| `no_missing_can_work` | 含义 | 下游行为 |
|---|---|---|
| `True` | 现有能力足够，按 normal 流程执行 | 继续推进到执行规划 |
| `False` | 缺少关键能力，无法诚实完成 | **立即停在门控处**，返回 blocked 状态 |

---

#### 🔄 修正后的完整数据流

结合课件的原始设计，修正后的门控形成了两阶段判定：

```
                   ┌─────────────────────────┐
                   │   host_capability_catalog │
                   │   (运行时实际接入的能力)    │
                   └────────────┬────────────┘
                                │
            ┌───────────────────┼───────────────────┐
            │                   │                   │
            ▼                   ▼                   ▼
   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
   │  web_search  │   │  file_read   │   │  其他能力...  │
   │  available   │   │  available   │   │              │
   └──────────────┘   └──────────────┘   └──────────────┘
                                │
                                │ 模型对照 SKILL.md 进行判断
                                ▼
   ┌─────────────────────────────────────────────────────┐
   │                  模型输出                             │
   │                                                     │
   │  needs: [                    ← 在 catalog 中找到了   │
   │    {capability: "web_search", requirement: "required"}
   │  ]                                                   │
   │                                                     │
   │  missing: [                  ← catalog 中没有的       │
   │    {capability: "wanderlog_mcp", requirement: "optional"}
   │  ]                                                   │
   │                                                     │
   │  no_missing_can_work: True   ← missing 里的都是       │
   │                                   optional，不影响核心  │
   └──────────────────────┬──────────────────────────────┘
                          │
                          ▼
   ┌─────────────────────────────────────────────────────┐
   │             evaluate_capabilities                    │
   │                                                     │
   │  blocked  = needs中required但missing的 +             │
   │             missing中required的                       │
   │  degraded = optional但missing的                       │
   │                                                     │
   │  → status: "ready"（missing的都是optional，可降级）    │
   └─────────────────────────────────────────────────────┘
```

修正后的核心变化：**门控不再是"自查自证"的摆设，而是一个真正能喊停的守护节点。** 当 Skill 需要的能力在运行时环境中不存在时，模型会通过 `missing` 诚实表达，宿主系统据此决定：继续，还是停在这里。

### 🧪 格式依赖测试：证明模型判断优于规则引擎

在修正门控逻辑的过程中，课堂上还进行了一个重要的实验——测试模型是否对提示词中的特定关键词有隐性依赖。

实验中，老师把 prompt 中所有的 `reference` 和 `resource` 关键词全部删除——这两个词在原提示词中用于引导模型关注 Skill 包的支持资源部分。理论上，如果模型是依赖关键词匹配的分词器，删除这些词之后，模型应该找不到对应的资源信息。

实验结果是：**模型仍然正确识别了 Skill 的能力需求和资源引用。**

这就证明了一个关键点：**语言模型在做的是语义判断，而不是关键词匹配。** 老师随即给出了一个重要的开发警告：

> ⚠️ **开发警告**：如果你使用 coden agent 做辅助开发，它经常会尝试用分词器（tokenizer）来做关键词匹配，而不是真正让模型去做语义分析。它会假设你的 Skill 结构是标准格式，然后自己写一套分词逻辑来检索内容——这种情况下，你对格式的任何微调都可能导致它"找不到"信息。**在写 Skill 执行器时，一定要防止 agent 用分词器替代模型判断。**

这也是为什么整个门控设计的核心是"把判断权交给模型"：**模型理解语义，分词器只认识字符串。** 面对灵活多变的 Skill 说明书内容，只有语义判断能扛得住格式变化。

---

### 📊 修正后的运行验证

修正完成后，用一个真实的 Skill 场景（旅行规划 Skill，需要 web_search 和 wanderlog_mcp）做了验证：

```python
# 模拟：宿主当前只接入了 web_search，没接 wanderlog_mcp
catalog = {
    "web_search": "available",
    "file_read": "available",
}
needs_result = infer_capability_needs(agent, task, package)
print("needs:", [(n["capability"], n["requirement"]) for n in needs_result["needs"]])
print("missing:", [(m["capability"], m["requirement"]) for m in needs_result["missing"]])
print("no_missing_can_work:", needs_result["no_missing_can_work"])
```

**预期输出**：

```
needs: [('web_search', 'required')]
missing: [('wanderlog_mcp', 'optional')]
no_missing_can_work: True
```

**解读**：

- `web_search` 被正确识别为 `required`，且在 catalog 中存在 → 在 `needs` 中
- `wanderlog_mcp` 在 Skill 中被提到（用于写入旅行计划），但 catalog 中没有 → 在 `missing` 中，且标注为 `optional`（因为没有它，核心的搜索和规划仍可完成）
- `no_missing_can_work: True` → 下游可直接继续推进

如果将 `web_search` 也从 catalog 中移除，模型会将其放入 `missing` 并标为 `required`，此时 `no_missing_can_work` 应为 `False`，整个请求应被 blocked。

### 📝 经验总结

从这次真实的课堂 Debug 中，可以提炼出三条关键教训：

**教训一：模型+代码混合系统，Bug 常常出现在数据流衔接处。**

`evaluate_capabilities` 的代码一个字都没写错，但上游 `infer_capability_needs` 给它的数据本身就是"被筛选过的"——所有条目都来自已有的能力列表。门控不是代码逻辑错了，而是**信息来源的设计**错了。检查混合系统时，不要只盯着单个函数的正确性，要看数据从产生到消费的整条链路。

**教训二：给模型一个"说不"的出口。**

设计 prompt 时，很容易不自觉地把模型的输出空间限制在一个"你想听到的答案"范围内。但当模型的角色是一个 honest broker（诚实的信息对照者）时，**必须给它表达"你给的和我需要的不匹配"的能力**。`missing` 字段就是这个"不匹配"的出口——没有它，门控永远看不到问题。

**教训三：missing 需要有边界约束，否则模型会发散。**

给了模型表达"缺失"的自由后，必须立即加上约束：缺失的范围应该限定在 Skill 说明书内。否则模型出于"善意"，会替你想出各种可能需要但实际无关的能力，导致门控信号失真。`missing` 是精准的工具，不是自由发挥的画布。

> 💡 **核心洞见**：门控的本质不是"让模型判断能不能做"，而是"让模型诚实报告需求和现实的差距，由宿主做最终判定"。模型负责"看到差距"，代码负责"决定行动"——这是一个各司其职的协作模式，而不是把所有判断权都交给模型或代码中的任何一方。

## 五、执行规划——从Skill指引生成step-DAG

### 🧠 直观理解

想象你是一个旅行规划师。老板刚刚通过「门控检查」确认：你的工位有电脑、有网络、有浏览器——"这个活你能干"。

现在你需要做的是：**画一张今天的工作地图**。不是写一篇散文描述"我要怎么规划旅行"，而是画一张精确的、排好顺序的任务清单：

> 1. 查景点开门时间（不需要等任何人，现在就查）
> 2. 查门票价格（也不需要等，和1同时查）
> 3. 查交通路线（同样独立，同时查）
> 4. 查天气情况（独立，同时查）
> 5. 等上面四项全查完，把结果汇总成一份旅行计划

前三项之间没有依赖关系——这就是「根步骤」，可以并行。最后一步必须等前面全完成——这就是「汇聚节点」。这张图，就是本模块要生成的 **step-DAG**（有向无环图，Directed Acyclic Graph）。

> 💡 **核心洞见**：门控回答了"能不能跑"（环境能力齐备度），执行规划回答了"怎么跑"（先做什么后做什么）。两步共同构成 Skill 执行的安全护栏——缺了任何一个，Agent 要么跑不起来，要么跑乱了。

### 📖 详细解释

#### 执行规划的本质：从"能做什么"到"这次怎么做"

上一步门控给了一份「可用能力清单」——比如 `web_search`、`model_request`。但这只是一个**名词列表**，不是一份**行动计划**。就好比厨房里有刀、有锅、有调料，但你还需要一张菜谱告诉你先切菜还是先开火。

执行规划的输入是四个东西，缺一不可：

1. **任务描述**：用户说了什么，本次要解决什么问题。
2. **SKILL.md 的行为指引**：Skill 作者写下的自然语言指令——"你需要做信息收集并生成报告"——这是模型的"工作说明"。
3. **已选中的资源**：支材门控阶段筛选出的 Reference 文件内容——本次任务需要用到的参考资料。
4. **门控后的可用能力**（`allowed_capabilities`）：已经过滤掉环境不支持的能力——这是模型的"工具包"。

输出是一张 **step-DAG**：一个有向无环图，每个节点是一个独立的、可被确定性执行的步骤。

#### step-DAG 的节点结构

每个节点包含四个字段，构成一个自包含的执行单元：

| 字段 | 类型 | 含义 | 示例 |
|---|---|---|---|
| `step_id` | `str` | 稳定标识符，全局唯一。建议用 `research_` 或 `plan_` 前缀标识节点类型 | `research_opening_hours` |
| `capability` | `str` | 本步骤使用的能力。必须来自 `allowed_capabilities`，生成步骤固定用 `model_request` | `web_search` |
| `topic` | `str` | 仅 `web_search` 步骤需要，指定搜索什么事实项。非搜索步骤强制为空字符串 | `opening_hours` |
| `depends_on` | `[str]` | 前置依赖的 `step_id` 列表。空列表 = 根步骤（无前置依赖，可并行执行） | `[]` 或 `["research_weather"]` |

> ⚠️ **注意事项**：`topic` 不能由模型自由发挥。宿主维护了一份 `web_search_topics()` 清单——这是事实项的权威来源。模型只能从这个清单中选取 topic，越界的 topic 会被 host 直接 `assert` 拒绝，不执行。

#### 为什么是 DAG 而不是线性列表？

线性列表意味着必须一条接一条顺序执行。但"查天气"和"查价格"之间没有任何依赖关系——如果顺序执行，就是在浪费时间。DAG 的**图结构**天然支持并行：所有根步骤（`depends_on` 为空）可以同时启动，执行器只需保证汇聚节点等待所有上游完成即可。这是 DAG 相比线性列表的核心优势：**用拓扑结构编码依赖，让无依赖的任务最大化并行**。

#### 两种节点类型：取证步骤 vs 生成步骤

本模块的 step-DAG 中只有两类节点，职责清晰分离：

| 节点类型 | capability | 特征 | 典型数量 |
|---|---|---|---|
| **取证步骤** | `web_search` | `depends_on` 为空（根步骤），彼此独立可并行 | N 个（取决于事实项数量） |
| **生成步骤** | `model_request` | `depends_on` 指向全部取证步骤，汇聚所有上游结果 | 恰好 1 个 |

> 💡 **核心洞见**：取证步骤"收集事实"，生成步骤"综合输出"。这种分离使得执行器可以并行跑 N 个搜索，最后才调用一次模型做汇总——而不是每搜一个结果就调一次模型。效率上，这是 N+1 次调用 vs 2N 次调用的区别。

### 📋 标准备选动作模板 → 实际执行动作规范

老师在白板上画了一个关键的转换过程——这也是整个执行规划中最需要理解的设计思想：**Capability 提供的是"标准备选动作模板"（你能做什么的选项池），DAG 节点产出的是"实际执行动作规范"（这次具体怎么做）。**

下面用 ASCII 图还原这个转换逻辑。左边是 capability 的能力清单（备选模板），右边是经过模型规划后产出的 step-DAG 节点（执行规范）：

关键理解：**同一个 Capability 可以产生多个 DAG 节点**。`web_search` 只是一个工具名，「查天气」和「查价格」用的都是同一个 `web_search` capability，但因为 `topic` 不同，它们是两个独立的节点。这就是"备选模板"和"实际执行规范"的区别——模板是通用的，执行规范是具体化的。

### 🔀 关键对比：内层DAG（本课） vs 外层DAG（15课）

这是第 16 课与第 15 课最本质的区别点。两者都叫"图"，都用了 DAG 的概念，但**抽象层次完全不同**——如果混淆了，后面的执行逻辑会完全理解错。

先用一张对照图直观感受两者的粒度差异：

```
外层 DAG（15课 — 任务级编排）                        内层 DAG（16课 — 操作级执行）

  ┌──────────────┐                                   ┌──────────────────────┐
  │   Skill A    │────┐                              │   research_weather   │ (web_search)
  │  "订机票"     │    │                              │   research_prices    │ (web_search)
  └──────────────┘    │    ┌──────────────┐          │   research_transport │ (web_search)
                      ├───▶│   Skill C    │          │   research_hours     │ (web_search)
  ┌──────────────┐    │    │ "生成旅行方案" │          └──────────┬───────────┘
  │   Skill B    │────┘    └──────────────┘                     │
  │  "订酒店"     │                                              ▼
  └──────────────┘                                   ┌──────────────────────┐
                                                     │   plan_generation    │ (model_request)
  节点 = 一个完整 Skill                                 └──────────────────────┘
  依赖 = 业务逻辑依赖
                                                     节点 = 一次能力调用
                                                     依赖 = 数据流依赖
```

再用表格精确对比七个维度：

| 对比维度 | 外层DAG（15课：编排） | 内层DAG（16课：执行） |
|---|---|---|
| **抽象层级** | 任务级：决定"用哪个 Skill、以什么顺序" | 操作级：决定"单 Skill 内各步骤怎么跑" |
| **节点粒度** | 一个节点 = 一个完整的 Skill 或子任务 | 一个节点 = 一次具体的能力（capability）调用 |
| **节点数量级** | 3~10 个 Skill 节点 | 取决于事实项数量，通常 5~20 个节点 |
| **节点间关系** | Skill 间有业务逻辑依赖，"先订机票才能规划路线" | 取证步骤彼此完全独立，"先查天气还是先查价格无所谓" |
| **图结构特征** | 复杂多变，可能有条件分支、循环、子图 | 典型的「汇聚结构」：多个并行根节点 → 单个生成节点 |
| **生成方式** | 模型语义规划 + TriggerFlow 编译为可执行 DAG | 模型生成 step 列表 + Host 确定性边界校验（`assert`） |
| **关注的核心问题** | 多 Skill 的编排逻辑、资源分配、冲突处理 | 单 Skill 的执行细节、capability 合法性、参数正确性 |

> 💡 **核心洞见**：外层 DAG 决定的是"战略"（用什么 Skill、按什么顺序上），内层 DAG 决定的是"战术"（每个 Skill 内部具体分几步跑、每步用什么工具）。两层 DAG 叠加，构成了 Agent 从宏观任务接收到微观工具调用的**完整执行路径**。15 课回答的问题至多到"这个 Skill 该不该入选"，16 课的回答是"入选之后，怎么一步步跑完"。

> ❓ **常见误解**："那是不是外层 DAG 的每个 Skill 节点展开后就是一张内层 DAG？"——是的，正是如此。外层 DAG 是 Skill 级别的拓扑，内层 DAG 是 capability 级别的拓扑。二者是**层级嵌套**关系，不是平级替代关系。

### 🔗 推导链：从 SKILL.md 行为指引到可执行 step-DAG

下面完整展示从 Skill 作者写下的自然语言行为指引，到最终可交给执行器跑的可执行 step-DAG 的每一步推理过程。这个推导链是理解"模型如何规划"的核心。

```
起点: SKILL.md 中写下的行为指引（自然语言，人看的）
  示例内容: "你需要收集目标景点的开放时间、门票价格、交通方式、天气情况，
            然后综合这些信息生成一份旅行建议报告。"

  │
  ├── 第1步: 模型解读行为指引的语义，拆解为可执行的动作类型
  │   为什么？自然语言的行为指引是给人看的纲要，不是机器可执行的指令。
  │   模型需要识别出：「收集信息」→ 需要多次 web_search 调用；
  │   「综合生成报告」→ 需要一次 model_request 调用。
  │   这一步在做什么：从"做什么"的语义描述中，提取出"用什么工具"的动作类型。
  │
  ├── 第2步: 将动作类型映射到 allowed_capabilities 白名单
  │   为什么？模型不能使用门控未放行的能力——这是 host 的安全边界。
  │   如果 SKILL.md 提到了"向用户提问确认"（capability: ask），
  │   但 ask 不在 allowed_capabilities 中，这一步就必须跳过或标记为缺失。
  │   门控的存在保证了：模型规划出来的图，宿主一定有能力去执行。
  │   如果系统检测到必要的能力缺失，应该触发能力的动态加载流程，
  │   而不是继续往下做一份不完整的计划。
  │
  ├── 第3步: 确定每个动作之间的依赖关系（这是 DAG 形状的关键）
  │   为什么？不是所有步骤都可以随意排列。依赖关系决定了图的拓扑结构。
  │   - web_search 步骤之间：搜索"天气"不需要等搜索"价格"的结果
  │     → depends_on 为空 → 标记为根步骤，执行器可以并行启动
  │   - model_request 生成步骤：需要所有取证结果作为上下文
  │     → depends_on 指向全部 research_* 节点 → 标记为汇聚节点
  │   这个依赖关系直接决定了 DAG 的形状：多条并行的根边 → 汇聚到单个叶子节点。
  │
  ├── 第4步: 填充每个节点的具体参数（从模板到实例）
  │   为什么？Capability 只给了参数模板（如 web_search 接受 query、lang），
  │   但这次具体搜什么、用什么语言，需要模型根据任务语义和话题清单填充。
  │   - "收集开门时间" → topic: "opening_hours"（从 web_search_topics() 清单中选取）
  │   - "收集门票价格" → topic: "prices"
  │   - "收集交通方式" → topic: "transport"
  │   - "收集天气情况" → topic: "weather"
  │   注意：topic 不能由模型凭空编造——host 在边界检查中会验证它是否在合法清单中。
  │
  └── 结论: 产出一张结构化的 step-DAG，可以直接喂给执行器
      [
        { step_id: "research_opening_hours", capability: "web_search", topic: "opening_hours", depends_on: [] },
        { step_id: "research_prices",        capability: "web_search", topic: "prices",        depends_on: [] },
        { step_id: "research_transport",     capability: "web_search", topic: "transport",     depends_on: [] },
        { step_id: "research_weather",       capability: "web_search", topic: "weather",       depends_on: [] },
        { step_id: "plan_generation", capability: "model_request", topic: "",
          depends_on: ["research_opening_hours","research_prices","research_transport","research_weather"] }
      ]
```

> ⚠️ **注意事项**：老师在录音中强调，他对这个 skill 读了很多遍，确认 **skill 本身没有传参要求**。skill 需要做的是信息收集，而信息收集的能力（如 `web_search`）必须在宿主的 capability 数组中被提供。如果宿主没有提供信息收集能力，门控阶段就会标记为缺失——根本走不到执行规划这一步。

### 💻 代码示例：plan_execution() 完整实现

下面是融合自课件第 3 节的完整可运行代码。它展示了模型如何将上下文规划成一张 step-DAG，以及 host 如何在模型产出后再做一轮**确定性边界检查**。

```python
# 文件名: plan_execution.py
# 功能: 将门控后的可用能力规划为一张可执行的 step-DAG

from skill_runtime import web_search_topics, load_resource_texts

# ── 核心代码：让模型把上下文规划成一张 step-DAG，host 再规整 + 守边界 ──
def plan_execution(agent, task, package, selected_resources, allowed_capabilities):
    """
    将 任务 + SKILL.md + 已选资源 + 门控后能力 → 规划为可执行的 step-DAG

    Args:
        agent: Agently Agent 实例（负责语义规划）
        task: 用户任务描述（自然语言）
        package: Skill Package（含 SKILL.md 和 Reference 文件）
        selected_resources: 支材门控选中的资源 ID 列表
        allowed_capabilities: 门控确认可用的能力列表（如 ["web_search"]）

    Returns:
        list[dict]: step-DAG，每个节点含 step_id / capability / topic / depends_on
    """
    # ── 第一阶段：模型做语义规划，生成原始 step 列表 ──
    r = (agent.create_request()
        .input(task)
        .info({"skill_md": package.skill_md,
               "selected_resource_texts": load_resource_texts(package, selected_resources),
               "allowed_capabilities": allowed_capabilities,
               "available_web_search_topics": web_search_topics()})        # 工具能查的事实项清单
        .instruct([
            "把执行拆成一张 step-DAG：每步给 capability / topic / depends_on",
            "capability 只能来自 allowed_capabilities；最终生成步骤用 model_request",
            "给每个 available_web_search_topics 各排一条独立取证步骤（depends_on 为空）",
            "再排恰好一个 model_request 生成步骤，depends_on 指向全部取证步骤（汇聚节点）",
        ])
        .output({"steps": [{"step_id": (str, "稳定 id，如 research_weather", True),
                            "capability": (str, "allowed 能力或 model_request", True),
                            "topic": (str, "web_search 步骤的事实项；生成步骤留空", True),
                            "depends_on": ([str], "上游 step_id；根步骤为空", True)}]})
        .start(max_retries=3))

    # ── 第二阶段：host 确定性边界检查（模型说了不算的，host 说了才算）──
    allowed = set(allowed_capabilities) | {"model_request"}               # model_request 总是允许
    topics = set(web_search_topics())                                     # 宿主支持的事实项清单

    for s in r["steps"]:
        # 边界检查1：capability 必须在白名单内——越界就拦，不默默执行
        assert s["capability"] in allowed, (
            f"未授权能力被 host 拒绝: {s['capability']}。"
            f"允许的能力: {allowed}。模型规划的图越界了，host 不会执行。"
        )
        # 边界检查2：非 web_search 步骤不应有 topic，强制清空
        s["topic"] = s["topic"] if s["capability"] == "web_search" else ""
        # 边界检查3：web_search 的 topic 必须是宿主支持的事实项——不能由模型编造
        assert not s["topic"] or s["topic"] in topics, (
            f"不支持的 topic: {s['topic']}。"
            f"宿主支持的 topic: {topics}。topic 不能由模型自由编造。"
        )
    return r["steps"]
```

#### host 确定性边界的三道防线

上面代码中的三个 `assert` 不是可选的"建议"，而是**硬约束**——它们共同构成了 host 对模型输出的三道防线：

| 防线 | 检查内容 | 为什么需要 | 触发后行为 |
|---|---|---|---|
| 第一道 | `capability` 是否在 `allowed` 白名单中 | 模型可能"幻觉"出一个不在白名单里的 capability | `assert` 拒绝，不执行，抛出明确错误信息 |
| 第二道 | 非搜索步骤是否被错误赋予了 `topic` | 模型可能给生成步骤也塞一个 topic，污染节点语义 | 静默清空：`topic = ""` |
| 第三道 | `topic` 是否在宿主支持的 `web_search_topics()` 清单中 | 模型不能凭空编造 topic | `assert` 拒绝，不执行 |

> 💡 **核心洞见**：**模型负责语义规划（"这次该查什么"），host 负责能力边界（"你能用什么查"）。** 模型说了不算的，host 说了才算。这是将 LLM 的不确定性输出，转化为确定性可执行结构的关键设计——不是在模型 prompt 里反复"请求"它遵守规则，而是用代码 `assert` 硬拦截。如果门控发现某些能力缺失，在更完整的系统中应该触发**能力的动态加载和检索**，而不是无视缺失继续向下执行。

#### 运行输出：step-DAG 表格

调用 `plan_execution()` 后，产出的不是一段自然语言散文，而是一张结构化的 step-DAG：

```python
# ── 调用示例 ──
allowed_capabilities = ["web_search"]
execution_steps = plan_execution(agent, task, package, needed, allowed_capabilities)

# 计划不是一条直线，而是一张 step-DAG：取证步骤彼此独立（根步骤），生成步骤依赖它们
print("plan_status=ready ｜ 计划是一张 step-DAG：\n")
for step in execution_steps:
    cap = step["capability"] + (f"({step['topic']})" if step["topic"] else "")
    deps = "、".join(step["depends_on"]) or "—（根步骤，可先跑）"
    print(f"  {step['step_id']:<26} | {cap:<22} | 依赖: {deps}")
```

实际运行输出：

```
plan_status=ready ｜ 计划是一张 step-DAG：

  research_opening_hours     | web_search(opening_hours) | 依赖: —（根步骤，可先跑）
  research_prices            | web_search(prices)        | 依赖: —（根步骤，可先跑）
  research_transport         | web_search(transport)     | 依赖: —（根步骤，可先跑）
  research_weather           | web_search(weather)       | 依赖: —（根步骤，可先跑）
  plan_generation            | model_request             | 依赖: research_opening_hours、research_prices、research_transport、research_weather
```

#### 观察这张 step-DAG 的四个关键特征

1. **前四个 `research_*` 步骤全是根步骤**：`depends_on` 为空，它们之间没有任何依赖关系。谁先跑谁后跑无所谓——执行器可以并行启动四个搜索，最大化吞吐量。
2. **最后一个 `plan_generation` 是汇聚节点**：它的 `depends_on` 指向全部四个 research 步骤。只有当所有取证结果都收集齐了，才能开始生成。这是 DAG 结构的"扇入"形态。
3. **取证节点用 `web_search`，生成节点用 `model_request`**：两种节点类型泾渭分明——取证节点调工具，生成节点调模型。这种分离使得资源调度和错误处理可以按类型分别对待。
4. **`topic` 字段只在 `web_search` 步骤中有值**：生成步骤的 `topic` 为空字符串——这是 host 第二道防线的结果：非搜索步骤的 topic 被强制清空。

> 🔗 **延伸阅读**：这张 step-DAG 的下一步就是真正执行它——详见本文「第六章：DAG 执行」（← 合并时替换为锚点链接）。执行器的本质是按拓扑排序跑图：先并行启动四个取证节点，等它们全部完成后再跑生成节点。执行器不需要理解"旅行规划"是什么，它只需要看懂 `depends_on` 字段即可。

#### step-DAG 各字段的权责归属

将每个字段的"谁决定、谁校验"关系整理为下表，这是理解模型与 host 分工的关键：

| 节点字段 | 由谁决定？ | 由谁校验？ | 校验规则 |
|---|---|---|---|
| `step_id` | 模型自由生成 | host 不做校验（仅要求唯一性） | 建议用 `research_` / `plan_` 前缀标识节点类型 |
| `capability` | 模型从 `allowed_capabilities` 中选择 | host `assert` 硬校验 | 必须 ∈ `allowed` 白名单；生成步骤固定 `model_request` |
| `topic` | 模型从 `web_search_topics()` 中选取 | host `assert` 硬校验 | 必须 ∈ `topics` 合法清单；非搜索步骤强制清空为 `""` |
| `depends_on` | 模型根据语义依赖确定 | host 不做校验（DAG 执行器会校验拓扑合法性） | 根步骤为空数组；生成步骤指向全部取证节点 |

> ⚠️ **注意事项**：计划产出的不是一段自由文本描述，而是一张可以被**确定性执行**的结构化图。图的每个节点都有明确的能力调用指令、具体参数和前置条件。正是这种结构化，使得下一节的 DAG 执行器能够**无需理解任务语义**，仅凭图的拓扑结构就能正确调度执行——这是将"Agent 智能"和"执行器确定性"解耦的关键设计。

### 🧠 直观理解：从"备选材料"到"可执行图"再到"实际结果"

在动手写代码之前，先理解一个核心转换过程。老师在网络卡顿的情况下，靠画图"推"完了整个执行流程的逻辑——这恰好是理解 DAG 执行器最好的方式：不看代码，先看图。

整个 Skill 执行链路中，有三个关键形态的转换：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Skill 执行的三态转换                           │
│                                                                     │
│  ┌──────────────────┐      模型生成执行计划      ┌──────────────────┐ │
│  │   备选材料（Block） │ ──────────────────────▶ │  可执行图（step-DAG）│ │
│  │                  │                           │                  │ │
│  │ · web_search     │   把 block 变成             │ research_* (并行) │ │
│  │ · model_request  │   可执行图                  │      ↓           │ │
│  │ · wanderlog_mcp  │                           │ plan_generation   │ │
│  │ · ...            │   标准的备选动作模板         │   (汇聚节点)       │ │
│  └──────────────────┘                           └────────┬─────────┘ │
│                                                          │          │
│                                                 执行器消费规划        │
│                                                          │          │
│                                                          ▼          │
│                                                  ┌──────────────────┐ │
│                                                  │  实际结果 + 证据   │ │
│                                                  │                  │ │
│                                                  │ · evidence_id_1  │ │
│                                                  │ · evidence_id_2  │ │
│                                                  │ · 行程标题+stops │ │
│                                                  └──────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

> 💡 **核心洞见**：能力门控（capability gate）做完之后，我们得到的只是"备选的动作块"（block）——比如 `web_search` 能查什么、`model_request` 能生成什么。这些 block 只是**标准的备选动作模板**，还没有形成"这一次具体要怎么跑"的计划。模型生成执行计划这一步，就是把 block 翻译成**可执行图**——每一步有了具体的 topic、明确的 depends_on 关系。然后执行器消费这张可执行图，产出**实际的执行结果**，并在过程中留下 evidence（证据）。这就是"备选材料→可执行图→实际结果"的完整转换。

> ⚠️ **注意事项**：这里有两层 DAG，含义不同。上一节课（第 15 课）讲的是**外层 DAG**，回答的是"要从北京去杭州，该用哪个 Skill（查机票？做规划？）"——这是任务级别的。本模块讲的是**内层 DAG**，回答的是"travel-planner 这个 Skill 内部，先查什么、后查什么、最后怎么生成"——这是执行级别的。外层 DAG 选 Skill，内层 DAG 执行 Skill。

### 💻 Step 1：MVP 执行器原型——手搓 20 行看清调度器本质

#### Step 1.1 为什么先手搓一个？

直接调 `Agently.create_dynamic_task()` 当然可以，但那样你看到的只是一个黑盒。DAG 执行器的核心逻辑其实非常简单——不到 30 行代码就能写出来。先手搓一遍，你就知道：

- 拓扑序是怎么决定的
- 上游产物的传递机制是什么
- 不同节点类型的分发逻辑（取证 vs 生成）
- 下游解锁的条件判断

这些东西理解了，再用 TaskDAG 框架时，你就能明白它替你做了什么、为什么比手搓好。

> 💡 **核心洞见**：真正工程不会自己手搓调度器。MVP 是教学原型，目的是让你"看清调度器在干嘛"；TaskDAG 是生产方案，加上校验、并发、证据收集——我们最后要用的就是后者。先看清骨架，再交给成品。

#### Step 1.2 完整代码

```python
# 文件名: mvp_executor.py
# 功能: 最小 DAG 执行器原型——把"执行器到底在干嘛"摊开成四步
#       真实工程再交给 Agently TaskDAG

from collections import deque
from skill_runtime import web_search, build_draft_request

def mvp_execute_dag(steps):
    # ── 第零步：构建辅助数据结构 ──
    by_id = {s["step_id"]: s for s in steps}            # step_id → step 快速查找
    indegree = {sid: 0 for sid in by_id}                 # 每个节点的入度（还有几个上游没跑完）
    children = {sid: [] for sid in by_id}                # 每个节点的下游子节点列表

    # 根据 depends_on 填充 indegree 和 children
    for s in steps:
        for dep in s["depends_on"]:
            indegree[s["step_id"]] += 1                  # 当前节点依赖 dep，入度 +1
            children[dep].append(s["step_id"])           # dep 是当前节点的上游

    # ── 第一步：拓扑序 —— 入度为 0 的根节点先入队 ──
    queue = deque(sorted(sid for sid, n in indegree.items() if n == 0))
    order, results = [], {}

    while queue:
        sid = queue.popleft()
        order.append(sid)                                # 记录执行顺序
        step = by_id[sid]

        # ── 第二步：收齐上游产物 ──
        upstream = [results[d] for d in step["depends_on"]]

        # ── 第三步：按节点类型分发 ──
        if step["capability"] == "web_search":
            # 取证节点 → 调工具，返回 evidence（含 evidence_id + summary）
            results[sid] = web_search(step["topic"]).as_dict()
        else:
            # 生成节点 → 调模型，先收集上游的 evidence 作为依据
            evidence = [e for e in upstream if isinstance(e, dict) and "evidence_id" in e]
            results[sid] = build_draft_request(
                agent, package.skill_md, task, evidence
            ).start(max_retries=3)

        # ── 第四步：解锁下游 ──
        for child in children[sid]:
            indegree[child] -= 1                         # 当前节点跑完，子节点入度 -1
            if indegree[child] == 0:                     # 入度归零 = 所有上游都跑完了
                queue.append(child)                      # 可以入队执行了

    # ── 第五步：环检测 ──
    if len(order) != len(by_id):
        raise ValueError("step-DAG 存在依赖环，无法执行")
    return order, results
```

#### Step 1.3 行级解析表

MVP 执行器的每一行都在做什么？下面逐行拆解，重点理解 `deque`、`indegree`、`children` 三个数据结构的协作关系。

| 行号 | 代码关键部分 | 解释 |
|---|---|---|
| 8 | `by_id = {s["step_id"]: s for s in steps}` | 建立 step_id 到 step 字典的快速查找。O(1) 取到任意节点的详细信息。为什么需要？因为后面循环里拿到的是 step_id 字符串，要通过它查 capability、topic、depends_on。 |
| 9 | `indegree = {sid: 0 for sid in by_id}` | 初始化每个节点的**入度**（还有几个上游依赖没跑完）。入度为 0 表示这个节点所有上游都已跑完或没有上游（根节点），可以立即执行。这是拓扑排序的核心计数器。 |
| 10 | `children = {sid: [] for sid in by_id}` | 初始化每个节点的**下游子节点列表**。当前节点跑完后，需要通知它的下游："我跑完了，你的入度可以减 1 了"。children 就是用来做这个通知用的邻接表。 |
| 13-15 | `for dep in s["depends_on"]: indegree[...] += 1; children[dep].append(...)` | 遍历每个节点的 `depends_on` 列表，同时更新两个结构：被依赖者（dep）的 children 列表里加入当前节点；当前节点的 indegree 累加。一趟遍历同时建好入度表和邻接表。 |
| 18 | `deque(sorted(...))` | 使用 `collections.deque` 双端队列作为执行队列。`popleft()` 是 O(1)，比 list 的 `pop(0)` 的 O(n) 高效得多。`sorted()` 保证同批根节点按字母序出队，可复现。 |
| 23 | `sid = queue.popleft()` | 从队首取出一个就绪节点。先进先出（FIFO），保证先就绪的先执行。 |
| 27 | `upstream = [results[d] for d in step["depends_on"]]` | 收集当前节点所有上游依赖的执行结果。例如 `plan_generation` 依赖三个 research_* 节点，这里就会拿到三个 evidence 字典。这是 DAG 执行器的"产物传递"机制——每个节点只能拿到它声明依赖的上游结果。 |
| 30-31 | `if step["capability"] == "web_search": results[sid] = web_search(...)` | **取证节点分发**：capability 是 `web_search`，说明这是一个取证步骤，调用 `web_search` 工具按 topic 查实时信息。返回的字典包含 `evidence_id`（唯一标识）和 `summary`（摘要）。这是 evidence trace 的**唯一来源**——工具返回的才是证据，模型回复的不是。 |
| 34-35 | `else: evidence = [e for e in upstream if ...]; results[sid] = build_draft_request(...)` | **生成节点分发**：capability 不是 `web_search`（实际是 `model_request`），说明这是生成步骤。先从上游产物中筛选出真正的 evidence（有 `evidence_id` 的才是证据），再交给模型生成行程。模型只能基于这些证据来生成，不能编造事实。 |
| 37-40 | `for child in children[sid]: indegree[child] -= 1; if indegree[child] == 0: queue.append(child)` | **解锁下游**：当前节点执行完毕，遍历它的所有子节点，每个子节点的入度减 1。当某个子节点的入度降到 0 时，说明它的所有上游都已完成，可以加入就绪队列。这就是 DAG 执行的"解锁"机制——不是主动推，而是被动等上游都完成。 |
| 42-43 | `if len(order) != len(by_id): raise ValueError(...)` | **环检测**：如果执行完所有能跑的节点后，order 的长度小于节点总数，说明有节点永远等不到入度为 0——即存在循环依赖。fail-fast，抛出明确错误，不默默产生不完整结果。 |

#### Step 1.4 运行 MVP 执行器

```python
mvp_order, mvp_results = mvp_execute_dag(execution_steps)

# 找到生成节点的 id
gen_id = next(s["step_id"] for s in execution_steps if s["capability"] == "model_request")

print("MVP 执行顺序:", " -> ".join(mvp_order))
print("取证节点产出:", [r["evidence_id"] for r in mvp_results.values()
                     if isinstance(r, dict) and "evidence_id" in r])
print("生成节点产出:", mvp_results[gen_id]["title"])
```

**MVP 运行输出：**

```
MVP 执行顺序: research_opening_hours -> research_prices -> research_transport -> plan_generation
取证节点产出: ['research_opening_hours_1', 'research_prices_1', 'research_transport_1']
生成节点产出: 杭州西湖经典一日游（周末轻户外）
```

**输出解读：**

- **执行顺序**：三个取证节点按字母序依次执行（MVP 是单线程串行），最后执行生成节点。注意 MVP 里三个 research_* 是**串行**的——因为 `deque` 一次只能 `popleft()` 一个，这是 MVP 的限制。
- **取证节点产出**：返回了三个 `evidence_id`。每个 ID 格式为 `research_<topic>_<序号>`，由 `web_search` 工具生成，代表一条可追溯的事实记录。这三个 ID 就是所谓的 **evidence trace**——工具返回的事实证据，不是模型编的。
- **生成节点产出**：模型基于三条证据生成了行程标题 `"杭州西湖经典一日游（周末轻户外）"`。模型没有凭空编造"杭州有什么好玩的"——它读的是 web_search 返回的开放时间、价格、交通信息。

> ⚠️ **注意事项**：MVP 执行器只做串行调度——即使三个 research_* 之间没有依赖关系（`depends_on` 都为空），它也是一个一个跑的。真正的生产环境里，这三个步骤**可以并行**。这就是为什么我们需要 TaskDAG。

### 💻 Step 2：TaskDAG 生产实现——把同一张图交给框架

#### Step 2.1 为什么不再手搓了？

MVP 的 30 行代码让你看清了调度器的四步本质。但真实工程里，你还要面对：

- **校验**：这张图有没有环？有没有缺了依赖？handler 是不是都注册了？
- **并发**：互不依赖的步骤能不能同时跑？串联跑浪费时间。
- **证据收集**：执行过程中每一步的产物怎么统一收集、怎么跟上游关联？
- **可观察性**：每一步的执行状态怎么追踪？出问题了怎么回查？

这些不是 DAG 执行器的"核心逻辑"，但却是生产环境绕不开的工程问题。与其自己在 MVP 上加补丁，不如把同一张 step-DAG 交给 Agently 的 **TaskDAG** 基座能力——它把校验、并发、证据收集、执行底座全部封装好了。

> 💡 **核心洞见**：TaskDAG 不是"帮你执行 DAG"的黑盒，而是在 MVP 那四步的基础上，增加了**校验层**（`TaskDAGValidator` fail-closed 查环/缺依赖/未知 handler）、**并发调度层**（互不依赖的节点并行）、**结果收集层**（`snap["task_results"]` 统一拿所有节点产物），并且底层落到 **TriggerFlow**（阶段可观察、可回查）。

#### Step 2.2 完整代码：三步把图交给 TaskDAG

整个 TaskDAG 生产实现分三步：**格式翻译 → handler 注册 → 校验 + 执行**。

```python
# 文件名: taskdag_executor.py
# 功能: 将同一张 step-DAG 交给 Agently TaskDAG 基座能力执行
#       三步：① to_task_dag() 格式翻译 → ② 注册 handler → ③ 校验 + 执行

from agently import Agently
from agently.core.orchestration.TaskDAG import TaskDAGResolver, TaskDAGValidator
from skill_runtime import web_search, build_draft_request

# ── 第一步：把计划翻成 TaskDAG 的提交格式（task_dag/v1）──
# create_dynamic_task 要求：handler 名以 _handler 结尾
# 取证步 → research_handler，生成步 → generate_handler
def to_task_dag(steps):
    tasks, gen_id = [], ""
    for s in steps:
        node = {
            "id": s["step_id"],
            "kind": "local",                    # 本地 handler（不是远程调用）
            "depends_on": s["depends_on"],      # 依赖关系原样传递
        }
        if s["capability"] == "web_search":
            # 取证节点：绑定 research_handler，传入搜索 topic
            node["binding"] = "research_handler"
            node["inputs"] = {"topic": s["topic"]}
        else:
            # 生成节点：绑定 generate_handler，记录 gen_id 用于后续取结果
            node["binding"] = "generate_handler"
            gen_id = s["step_id"]
        tasks.append(node)

    return {
        "graph_id": "single-skill",              # 图的标识
        "task_schema_version": "task_dag/v1",    # 协议版本
        "tasks": tasks,
        "semantic_outputs": {"itinerary": gen_id} # 声明哪个节点的输出是"最终行程"
    }, gen_id

# ── 第二步：注册 handler 函数 ──
# 取证 handler：调 web_search 工具，返回 evidence 字典
async def research_handler(ctx):
    """ctx.task.inputs 拿到 to_task_dag 里配的 inputs（这里是 {"topic": "xxx"}）"""
    return web_search(ctx.task.inputs["topic"]).as_dict()

# 生成 handler：从 ctx.dependency_results 拿上游证据，调模型生成行程
async def generate_handler(ctx):
    """ctx.dependency_results 里是所有上游节点的执行结果"""
    ev = [r for r in ctx.dependency_results.values()
          if isinstance(r, dict) and "evidence_id" in r]
    return await build_draft_request(
        agent, package.skill_md, task, ev
    ).async_start(max_retries=3)

# handler 注册表：handler 名 → 函数
handlers = {
    "research_handler": research_handler,
    "generate_handler": generate_handler,
}

# ── 第三步：校验 + 执行 ──
# 3a. 翻译计划
plan, gen_id = to_task_dag(execution_steps)

# 3b. TaskDAGValidator 校验：查环、查缺依赖、查未知 handler，全部 fail-closed
#     通过 TaskDAGResolver 将 handler 注册表与 plan 中的 binding 对应起来
order = (TaskDAGValidator(TaskDAGResolver(handlers))
         .validate(plan, strict_schema_version=True)
         .topological_task_ids)

print("TaskDAGValidator 校验后的拓扑序:", " -> ".join(order), "\n")

# 3c. 提交执行：Agently.create_dynamic_task() 底层落到 TriggerFlow 执行底座
snap = await Agently.create_dynamic_task(
    target=task,
    plan=plan,
    handlers=handlers
).async_start(timeout=180)

# 3d. 收集结果
results = snap["task_results"]          # 每个节点的产物都在这里
evidence_trace = [
    results[s["step_id"]]
    for s in execution_steps
    if s["capability"] == "web_search"
]
execution_result = {
    "answer": results[gen_id],
    "evidence_trace": evidence_trace,
}
```

#### Step 2.3 关键组件解析

| 组件 | 作用 | 为什么需要 |
|---|---|---|
| `to_task_dag()` | 将 step-DAG 翻译为 `task_dag/v1` 格式 | TaskDAG 有自己的协议格式（`graph_id`、`task_schema_version`、`tasks` 列表）。step-DAG 是模型产出的"语义计划"，TaskDAG 格式是框架能消费的"机器格式"。翻译层解耦两者。 |
| `binding` | 每个 task 节点绑定一个 handler 名 | TaskDAG 通过 binding 名字找到对应的 handler 函数。binding 必须以 `_handler` 结尾——这是 `create_dynamic_task` 的约定。 |
| `ctx.task.inputs` | handler 获取节点级输入 | 在 `to_task_dag` 中通过 `inputs` 字段传入的 `{"topic": "xxx"}`，在 handler 里通过 `ctx.task.inputs["topic"]` 拿到。每个节点的 inputs 是独立隔离的。 |
| `ctx.dependency_results` | handler 获取上游产物 | 这是一个 `dict`，key 是上游节点的 step_id，value 是那个节点的执行结果。生成 handler 通过它拿到所有取证节点的 evidence，作为模型生成行程的事实依据。 |
| `TaskDAGResolver` | 将 handler 注册表注入 TaskDAG | 在执行前确保 plan 中每个 binding 名字都能找到对应的 handler 函数。如果 binding 了 `"unknown_handler"` 但没注册，Resolver 会发现并报错。 |
| `TaskDAGValidator` | fail-closed 校验 | 执行前做三道检查：①图中有没有环（拓扑序能算出吗）②有没有节点依赖了不存在的节点③有没有 binding 了未注册的 handler。任何一项不通过，直接拒绝执行——不允许带着隐患跑。 |
| `topological_task_ids` | 输出校验后的拓扑序 | 跟 MVP 的手算拓扑序完全一致，但这里是通过框架的正式校验流程产出的。 |
| `create_dynamic_task()` | 提交动态任务图 | Agently 提供的便捷入口，接收 `target`（任务描述）、`plan`（图定义）、`handlers`（执行函数），底层落到 **TriggerFlow** 执行底座——每一步可观察、可回查。 |
| `snap["task_results"]` | 统一的节点产物字典 | key 是 step_id，value 是该节点的返回值。不管是取证节点的 evidence 还是生成节点的行程，都统一存放在这里，便于后续验收环节取用。 |

#### Step 2.4 运行 TaskDAG 执行器

```python

for ev in evidence_trace:
    print(f"  [{ev['evidence_id']}] {ev['topic']}: {ev['summary']}")

answer = execution_result["answer"]
print(f"\n据证据生成的行程：{answer['title']}")
for stop in answer["stops"]:
    print(f"  - {stop['name']}（依据 {stop['evidence_ids']}）")
```

| 维度 | MVP 执行器（教学原型） | TaskDAG（生产方案） |
|---|---|---|
| **代码量** | ~30 行 | 三步共 ~60 行（含 handler、格式翻译、校验） |
| **核心逻辑** | 拓扑序 + 收产物 + 分发 + 解锁下游 | 同样四步，但封装在框架内部 |

| **handler 注册** | 无——分发逻辑硬编码在 `if/else` 里 | `handlers` 字典显式注册，`TaskDAGResolver` 注入——新增节点类型只需加 handler，不改调度器 |

### 🐛 排错指南

#### 坑 1：MVP 执行器报 `ValueError: step-DAG 存在依赖环`

**🚨 报错信息**
```
ValueError: step-DAG 存在依赖环，无法执行
```

**🔍 原因分析**
`depends_on` 形成了循环引用。例如 A 依赖 B、B 依赖 A，两个节点的入度永远不等于 0，永远无法入队。MVP 在最后检查 `len(order) != len(by_id)` 时发现有些节点没被执行。

**✅ 修复**
检查 `execution_steps` 中每个步骤的 `depends_on` 字段，确保没有循环引用。一个快速检查方法：把所有依赖关系画成箭头，看有没有路径能走回自己。

#### 坑 2：TaskDAGValidator 报未知 handler

**🚨 报错信息**
```
（TaskDAGValidator 抛出异常，提示某个 binding 名称未注册）
```

**🔍 原因分析**
`to_task_dag()` 中 `node["binding"]` 设置的名字，在 `handlers` 字典中找不到对应的函数。常见原因：handler 名没以 `_handler` 结尾，或者拼写不一致。

**✅ 修复**
确保 `to_task_dag()` 中的 `binding` 字符串与 `handlers` 字典的 key 完全一致，且都以 `_handler` 结尾。例如 `"research_handler"` 不能写成 `"research"` 或 `"researchHandler"`。

#### 坑 3：生成节点的行程中出现模型编造的事实

**🚨 现象**
行程中某个 stop 的描述包含具体价格或开放时间，但对应的 evidence 中并没有这条信息。

**🔍 原因分析**
这是模型在生成时"自由发挥"了——它没有严格遵守"只能基于 evidence 生成"的约束。在 MVP 和 TaskDAG 的代码中，我们都通过 `evidence = [e for e in upstream if ... "evidence_id" in e]` 筛选了上游证据传入模型，但模型本身可能不听话。

**✅ 修复**
- 在 `build_draft_request` 的 instruct 中强化约束："只能依据提供的 evidence 生成行程，不要引入 evidence 中不存在的事实"。
- 在验收环节（下一模块 M07）做确定性溯源：检查每个 stop 引用的 `evidence_ids` 是否都在 evidence_trace 中。
- 验收不过不是终点——把不达标的点当反馈，让模型修一版再验（validate→repair 闭环）。

## 七、证据验收——两层校验与修复闭环

### 7.1 🧠 直观理解：执行完了不算完，验收通过才算完

想象餐厅厨房：大厨把菜炒出来了，但厨师长还要看一眼，尝一口，确认色香味都在标准之内，才让服务员端上桌。炒完不等于出菜，验收通过才算。

Skill 执行也是同样的道理。执行链（上一节 DAG 执行）跑完了，execution_result 也拿到了——但这就够了吗？模型可能在某一步做出了不符合 Skill 质量要求的判断，也可能引用了不存在的证据。所以执行完成之后，必须有一个**独立的验收环节**，用 Skill 自己定义的标准，逐条检查执行结果。

这个验收不是程序员事后另外写的 check list，而是 Skill 发布时自带的审查标准。travel-planner 的 SKILL.md 里写着 "Run a quick plan review" 的若干条规则——验收时就拿这些规则来判。也就是说，**验收用的是 Skill 自己的标准，不是程序员另立的规则**。

如果验收不通过怎么办？不是直接报错退出，而是触发「反攻」——把不达标的点当反馈，让模型根据同一批证据修一遍，修完再验。最多两轮。这就是 **validate → repair → re-validate** 的闭环。

> ⚠️ **注意事项**：验收不是在执行完成之后才"想起来要做"的。从执行链一开始，每一步工具调用就在攒 evidence_trace；到了验收这一步，只是把攒好的证据拿出来对账。**验收是设计时就嵌入流程的，不是事后补的。**

### 7.2 📖 详细解释：两层校验体系的设计思想

验收分成两层，各自解决不同的问题：

#### 第一层：确定性溯源（代码做，不靠模型）

```
每个 stop 声称引用了某些 evidence_id
        │
        ▼
把这些 evidence_id 和 evidence_trace 里实际存在的做集合判断
        │
        ├── 全都在 → 溯源通过
        └── 有不在的 → 报告"stop X 引用了不存在的证据 {missing_ids}"
```

溯源的核心问题是：**执行结果里声称的引用，到底有没有对应的证据？** 这是一个集合包含判断（`set(s["evidence_ids"]) <= known`），跟模型的措辞、语气、推理过程完全无关。答案是确定的、二值的——要么在，要么不在。

这类问题**不应该交给模型来判断**。原因很简单：模型可能"猜"一个答案——说在其实不在，说不在其实在。对于事实性校验，模型的概率输出是不可靠的。

> 💡 **核心洞见**：溯源只做「存在性检查」，不做「质量判断」。像"几个景点算过载"这类质量阈值，不能用写死的规则（如 `3 <= len(stops) <= 5`）来判断——模型输出本就不定，行程合不合适也依任务和偏好而定。溯源代码里绝对不能塞这种质量阈值，那是第二层的事。

#### 第二层：语义验收（模型做，按 Skill 标准判）

```
执行结果 + evidence_trace + SKILL.md 的 review 标准
        │
        ▼
模型逐条对照 Skill 自己的审查标准
        │
        ├── 每条给 passed + reason
        └── overall_passed: True/False
```

语义验收要回答的问题是：**执行结果在质量上好不好？** 这不是集合运算能解决的——需要理解 Skill 的语义标准，判断内容是否诚实、是否完整、是否实用。

比如 travel-planner 里有一条标准："Include at least one practical note after each place or stop cluster"。这个判断涉及对行程结构、stop 内容的语义理解——代码写死规则很难准确判断"什么是 practical note"，但模型可以。

> ⚠️ **注意事项**：语义验收里，**模型自然语言不是证据**。模型可以说"天气很好适合出游"，但这不能当证据。只有工具返回（web_search 的 JSON）、脚本输出、外部系统回执才是可验收的证据。如果模型把一个未核实的事实当作既定结论，在 review 中按 Skill 的诚实性要求就应当被判 FAIL。

#### 为什么必须分两层？

| 维度 | 第一层：确定性溯源 | 第二层：语义验收 |
|---|---|---|
| 谁来做 | 确定性代码 | 模型 |
| 判断内容 | evidence_id 是否存在于 trace 中 | 执行结果质量是否达标 |
| 判断方式 | 集合包含运算（`set <= known`） | 理解语义标准，逐条评判 |
| 答案性质 | 确定的、二值的 | 需要语义理解，每次可能不同 |
| 为什么不能反过来 | 模型不应被信任做事实核查，可能"猜"错 | 代码无法做语义质量判断，写死规则会漏判/误判 |

**一句话总结**：模型能理解语义但不应被信任做事实核查；代码做事实核查精确可靠但不该塞质量阈值。两层各司其职，组合起来才构成完整验收。

### 7.3 🔀 模型与工具的边界划分

验收环节是 Skill 执行链的最后一环，但它对边界的坚守和前几个环节一脉相承。下面的表格从「理解用户需求」到「验收行程」，完整展示了每个环节应该由谁来做、以及为什么：

| 环节 | 更适合谁来做 | 原因 |
|---|---|---|
| 理解用户要什么 | 模型 | 输入是自然语言，可能有隐含偏好 |
| 形成执行计划 | 模型 + host 边界 | Skill 给流程，模型按当前任务取舍；host 拒绝未授权能力 |
| 查天气 / 开放时间 / 交通 | web_search 工具 | 这是实时事实，不该让模型编 |
| 生成行程说明 | 模型 | 需要组织语言、解释取舍 |
| 检查证据是否存在（溯源） | 确定性代码 | evidence_id 在不在 trace 里，集合判断，不需要模型猜 |
| 判断行程好不好（验收） | 模型按 SKILL.md 的 review 标准 | 质量判断依任务而定，用 Skill 自己的标准，不写死规则 |

> 💡 **核心洞见**：注意最后两行——同样是「检查」性质的工作，却分给了两个不同的执行者。证据存在性是事实问题，代码做；行程好坏是质量判断，模型做。边界不按「是不是检查」来划，而按「问题的本质」来划。

### 7.4 💻 第一层校验：确定性溯源 `provenance_issues()`

验收的对象是上一节 DAG 执行产出的真实结果，不是写死的样例数据：

```python
# 验收对象 = 上一节"执行"出来的真实结果，不是写死的样例。
itinerary = execution_result["answer"]
evidence_trace = execution_result["evidence_trace"]
```

第一层校验函数 `provenance_issues()` 做的是确定性溯源——用代码检查每个 stop 引用的 evidence_id 是否真实存在于 evidence_trace 中：

```python
# ── 核心代码：确定性溯源 ──
# 每个 stop 引用的 evidence_id 必须真实存在（用代码，不靠模型）
def provenance_issues(stops, trace):
    """检查每个 stop 的 evidence_ids 是否都存在于 trace 中。

    Args:
        stops: 行程中的 stop 列表，每个 stop 包含 evidence_ids 字段
        trace: 执行过程中积累的证据列表，每条包含 evidence_id

    Returns:
        问题列表——哪些 stop 引用了不存在的证据。空列表表示溯源全部通过。
    """
    # 收集所有已知的 evidence_id，做成集合
    known = {e["evidence_id"] for e in trace}

    # 对每个 stop，检查其 evidence_ids 是否是 known 的子集
    # set <= known 就是集合包含判断——纯粹、确定、与模型措辞无关
    return [
        f"{s['name']} 引用了不存在的证据 {set(s['evidence_ids']) - known}"
        for s in stops
        if not set(s["evidence_ids"]) <= known   # 关键：集合包含判断
    ]
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 10 | `known = {e["evidence_id"] for e in trace}` | 将 trace 中所有 evidence_id 收集为集合，后续做 O(1) 成员判断 |
| 12-15 | `return [...]` | 列表推导式：遍历每个 stop，找出其 evidence_ids 不在 known 中的 |
| 14 | `set(s["evidence_ids"]) - known` | 集合差集运算——直接算出哪些 id 不存在，与模型措辞完全无关 |
| 15 | `not set(s["evidence_ids"]) <= known` | `<=` 是子集判断。只有全部包含才为 True，有任何一个不在就触发 |

> ⚠️ **注意事项**：这个函数**只做存在性检查**，绝对不塞质量阈值。不要在这里写 `if 3 <= len(stops) <= 5` 之类的判断。行程合不合适、几个点算过载——那是第二层语义验收的事，而且是模型按任务上下文灵活判断，不能写死数字。

### 7.5 💻 第二层校验：语义验收 `review_by_skill()`

第二层函数 `review_by_skill()` 让模型按照 SKILL.md 中 Skill 自己定义的 review 标准，逐条判断执行结果是否达标：

```python
# ── 核心代码：语义验收 ──
# 用模型按 SKILL.md 里 Skill 自己的 review 标准逐条判（不写死规则）
def review_by_skill(agent, skill_md, task, itinerary, trace):
    """让模型按照 SKILL.md 中的审查标准逐条验收行程。

    Args:
        agent: 模型 agent 实例
        skill_md: SKILL.md 全文——包含 Skill 自己定义的 review 标准
        task: 用户原始任务描述
        itinerary: 执行结果（行程）
        trace: 执行过程中积累的证据列表

    Returns:
        {"checks": [...], "overall_passed": bool}
        checks 中每条包含 criterion（标准原文）、passed（是否通过）、reason（判断理由）
    """
    return (agent.create_request()
        # 把 Skill 的审查标准、原始任务、证据链全部提供给模型
        .info({
            "skill_review_guide": skill_md,   # 标准现读现用，不是写死的
            "task": task,
            "evidence_trace": trace
        })
        .input(itinerary)
        .instruct([
            "只依据 SKILL.md 里这个 Skill 自己给的 review / quality 标准判，不要自立标准、不套写死阈值",
            "逐条给 passed + 理由；没被工具核实却当既定事实的，按 Skill 的诚实性要求判不通过",
        ])
        .output({
            "checks": [{
                "criterion": (str, "来自 SKILL.md 的一条标准", True),
                "passed": (bool, "这条是否通过", True),
                "reason": (str, "依据标准与证据的简短说明", True)
            }],
            "overall_passed": (bool, "是否达到该 Skill 的交付标准", True)
        })
        .start(max_retries=3))
```

> 💡 **核心洞见**：注意 `.info({"skill_review_guide": skill_md, ...})`——审查标准是**每次执行时现读现用**的，不是硬编码在代码里的。这意味着当 Skill 更新了它的 review 标准，验收逻辑自动跟随，程序员不需要改代码。这也再次印证了「验收用 Skill 自己的标准」。

`.instruct()` 中的两条指令是验收的灵魂：

1. **"只依据 SKILL.md 里这个 Skill 自己给的 review / quality 标准判"**——防止模型自己发明标准。travel-planner 的审查规则（天气 fallback、预算、用餐安排、每站加实用提示等）全都来自 SKILL.md，模型不能另立规则。

2. **"没被工具核实却当既定事实的，按 Skill 的诚实性要求判不通过"**——区分「模型自己说的」和「工具返回的」。比如模型说"门票 45 元"但没有 research_prices 证据支撑，就是 FAIL。

### 7.6 📊 完整运行输出：串联两层校验

将两层串联调用，验收结果全部展示：

```python
# 第 1 层：确定性溯源（集合判断，不塞质量阈值）
prov = provenance_issues(itinerary["stops"], evidence_trace)
print("溯源通过:", not prov, prov or "")

# 第 2 层：语义验收（模型按 travel-planner 自己的 "Run a quick plan review" 标准逐条判）
review = review_by_skill(agent, package.skill_md, task, itinerary, evidence_trace)
for c in review["checks"]:
    print(f"[{'PASS' if c['passed'] else 'FAIL'}] {c['criterion']} —— {c['reason']}")

# 两层都过才算成功
print("overall_passed:", (not prov) and bool(review["overall_passed"]))
```

运行输出如下：

```
溯源通过: True

[PASS] Use live web research for any fact that can change over time. —— 引用了 research_opening_hours_1（开放时间）、research_prices_1（价格）、research_transport_1（交通）三个实时搜索证据，符合对时间敏感信息的实时核实要求。
[PASS] When you infer from the research, say so. —— 在 assumptions 中明确标注了未核实的内容，例如博物馆名额、餐厅可容纳人数、灵隐寺特殊活动、交通票价等，诚实标注了推断而非当作既定事实。
[PASS] Do not introduce unsupported certainty for dynamic facts; research them. —— 对动态事实（门票价格、开馆状态、交通时间）均标注了来源或说明了未核实状态，没有将未核实的信息当作确定性结论。
[PASS] Include as relevant: paid-attraction costs, entrance fees, and reservation notes for any ticketed stops in the plan. —— 在 assumptions 中明确标注灵隐寺门票约45元/人，并对博物馆免费但需预约给出了标注和提醒，符合要求。
```

**关键观察**：

- 第一层溯源全部通过——所有 stop 引用的 evidence_id 都真实存在于 trace 中，不存在"无中生有"的引用。
- 第二层 7 条标准中 6 条 PASS、1 条 FAIL。FAIL 的这条——"未在每个 stop 后添加实用提示"——揭示了模型在生成行程时的具体短板：它记得研究开放时间、价格、交通，但忘了在每个打卡点后附加实用提示。
- **overall_passed: False**——两层都过才算成功，有一条 FAIL 就是整体不通过。这正是验收的设计意图：不因为大部分对了就放行，不达标的必须修。

> ❓ **常见疑问**：为什么"未添加实用提示"用代码写正则判断不出来？因为"实用提示"是语义概念——它可以是预约链接、可以是交通提醒、可以是注意事项，形态不固定。写死规则（如检查 stop 下是否有 `notes` 字段）只能做结构校验，判断不了内容是否真的"实用"。这类质量判断必须由模型来做。

### 7.7 🔄 validate → repair 闭环：不过就修，修完再验

验收不通过不是终点，而是新一轮修正的起点。整个流程构成一个闭环：

```
执行结果产出
      │
      ▼
┌──────────────┐
│  第一层：溯源   │  ← 集合判断，代码做
└──────┬───────┘
       │ 通过
       ▼
┌──────────────┐
│  第二层：语义   │  ← 按 SKILL.md 标准，模型判
└──────┬───────┘
       │
       ├── overall_passed: True ──▶ ✅ 验收通过，进入总结输出
       │
       └── overall_passed: False ──▶ 🔄 触发「反攻」机制
                                              │
                                              ▼
                                     ┌─────────────────┐
                                     │ 将 FAIL 条目作为   │
                                     │ 反馈交给模型，要求   │
                                     │ 用同一批证据重新修正 │
                                     └────────┬────────┘
                                              │
                                              ▼
                                     ┌─────────────────┐
                                     │ 修正后再次验收     │
                                     │ （最多 2 轮）     │
                                     └─────────────────┘
```

> ⚠️ **注意事项**：**反攻不是无限循环**。最多 2 轮。如果 2 轮后仍然不通过，说明 Skill 的标准或模型当前能力之间存在系统性的差距，需要人工介入判断——是标准太严了还是模型确实做不到。

反攻机制的核心思路在录音稿中有明确体现：老师讲到流程走到校验之后，如果校验不通过，就会"反攻到这边"（回退到执行环节），让模型根据反馈重新处理。这个设计很务实——不是在验收失败时报错退出，而是把验收当反馈，驱动修正。修正的依据是同一批 evidence_trace，不会重新调工具，所以修正成本很低。

> 💡 **核心洞见**：验收不通过时，正确的反攻方式是把 **FAIL 条目作为反馈给模型**，而不是让程序员手动改输出。比如上面那条 FAIL——"缺少每站实用提示"——反馈给模型后，模型会逐站补充 practical notes，然后再跑一次验收。整个过程不需要人插手，完全自动化。

### 7.8 📊 两类 Skill 的验收方式对比

两层校验正好对应两种不同类型的 Skill——它们给验收的方式不同，但共同点是：**都用 Skill 自己定义的 validate 方法**。

| 维度 | 可执行校验脚本型 | review 标准型 |
|---|---|---|
| 典型 Skill | xlsx 计算（`recalc.py`） | travel-planner（行程审查标准） |
| Skill 提供什么 | 可执行的校验脚本 | SKILL.md 中的 review 规则列表 |
| 验收执行者 | host 直接运行脚本 | 模型按标准逐条判 |
| 验收结果 | 确定性的 pass/fail | 每条标准有 passed + reason |
| 为什么这样设计 | xlsx 的公式正确性可以精确计算，不需要模型"理解" | 行程质量涉及语义判断，只能由模型做 |
| 对应第几层 | 相当于第一层（确定性） | 相当于第二层（语义） |

```
┌──────────────────────────────────────────────────────┐
│                  Skill 验收方式决策树                    │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Skill 产出结果能否被确定性校验？                         │
│       │                                              │
│       ├── 能（如公式计算结果）                            │
│       │   └──▶ 提供可执行校验脚本（recalc.py）             │
│       │        Host 直接跑 → 确定性 pass/fail            │
│       │                                              │
│       └── 不能（如行程质量、文案好坏）                      │
│           └──▶ 在 SKILL.md 中提供 review 标准             │
│                模型按标准逐条判 → passed + reason         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

无论哪种方式，验收的核心原则不变：**用 Skill 自己定义的 validate 方法**。程序员不替 Skill 发明校验规则，Skill 的发布者最清楚自己的产出应该达到什么标准。

>

## 八、完整执行链——TriggerFlow五阶段串联

### 8.1 目标：把分散环节串成一条完整流水线

#### 🧠 直观理解

前面七个模块（M01-M07）分别拆解了 Skill 执行链上的每一个独立环节：渐进式信息披露、Host 资源决策、能力门控、执行计划 DAG、证据记录、生成校验、反攻修订。这就好比你把一辆车的发动机、变速箱、刹车、悬挂全部拆开研究了一遍——每个零件都看懂了，但还没把它们装回去让车跑起来。

本节的任务就是**把零件装回去，拧紧每一颗螺丝，发动引擎，看整条流水线跑通**。

#### 📖 详细解释

TriggerFlow 五阶段串联的本质是：**用 Agently 的 TriggerFlow 基座能力，把 load_skill（加载技能）、capability_gate（能力门控）、plan_dag（规划执行图）、execute_dag（执行图）、verify_and_finalize（验收与定稿）这五个环节编排成一条显式、可观察的阶段流**。

这条链上有三个关键设计原则：

1. **阶段不必理解彼此的内部逻辑**：每个阶段只关心自己的输入是什么、产出是什么、事件往 runtime stream 写什么。阶段之间通过统一的数据上下文传递，不相互调用内部函数。
2. **模型先给结构化判断，不是先写最终答案**：`plan_dag` 阶段让模型输出一张 step-DAG（JSON），`verify_and_finalize` 阶段让模型按 review 标准逐条判（JSON）。结构化了，后面的工具调用、证据记录、验收才接得住。
3. **可观察性优先于执行效率**：每个阶段都往 runtime stream 写一条阶段事件，外部可以通过 `get_async_runtime_stream()` 实时观察执行状态——计划了几步、取了哪些证据、修了几版、过没过。

> 💡 **核心洞见**：TriggerFlow 五阶段不是简单的"顺序调用五个函数"。它是一套**有责任的阶段编排**——每个阶段有明确的输入契约、输出契约和可观察的事件协议。这套协议的工程意义在于：你可以单独替换任一阶段（换一个 planner、换一个 executor），只要它遵守输入/输出契约，整条链不受影响。

### 8.2 四层通用架构：工程责任的分离

在把五阶段落到代码里之前，先看这条链在工程上是如何分层承担责任的。以下四层通用架构是这节课的工程底座——它不是 Agently 特有的，换成任何框架都要回答这些责任问题：

```
┌─────────────────────────────────────────────────────────────────┐
│                      📋 模型请求层（ModelRequest）                │
│  职责：把模型的判断变成结构化输出（JSON，不是散文）              │
│  产出：step-DAG 计划、review 判定、stop 列表                     │
│  关键："让模型先给结构化判断" —— 后续所有步骤都依赖这个 JSON    │
├─────────────────────────────────────────────────────────────────┤
│                      🔄 流程编排层（TriggerFlow）                │
│  职责：把五阶段拆成显式、可观察的阶段流                          │
│  产出：每个阶段的 runtime event（load/capability/plan/exec/verify)│
│  关键：每步可观察，外部可实时知道"现在在哪一步、什么状态"        │
├─────────────────────────────────────────────────────────────────┤
│                      🔧 工具执行层（Tools / web_search）          │
│  职责：获取时效事实（天气、交通、价格、开放时间）               │
│  产出：按计划逐步攒下来的 evidence trace                         │
│  关键：工具只给事实，不给意图——"想去哪"由模型据知识提出         │
├─────────────────────────────────────────────────────────────────┤
│                      📊 证据记录层（Evidence Trace）             │
│  职责：每一步留下可追溯的证据记录                                │
│  产出：最终 verify 阶段的溯源依据                                │
│  关键：每个 stop 必须指回真实 evidence，否则验收不通过           │
└─────────────────────────────────────────────────────────────────┘
```

这四层和 Agently 三个基座能力的映射关系：

| 四层通用架构 | Agently 基座能力 | 在本课中的具体使用 |
|---|---|---|
| 模型请求层 | **ModelRequest** | planner 生成 step-DAG 计划、verifier 按 review 标准逐条判、drafter 生成最终 itinerary |
| 流程编排层 | **TriggerFlow** | 把 5 个阶段编排成显式阶段流，每步写 runtime event |
| 工具执行层 | **TaskDAG** | executor 按拓扑序执行 step-DAG 中的每个任务节点 |
| 证据记录层 | 跨层贯穿 | evidence trace 在 execute 阶段攒入，在 verify 阶段作为溯源依据 |

> 🔗 **关联知识**：关于 ModelRequest 的结构化输出设计，详见 [第三章：规划——把意图翻译成执行计划]（← 合并时替换为锚点链接）。关于 TaskDAG 的执行机制，详见 [第四章：执行——按计划图取证据]（← 合并时替换为锚点链接）。

### 8.3 TriggerFlow 五阶段流程：每一步的输入、输出与事件

#### 🔄 执行流程

```
用户请求（city/start/duration/budget/preference）
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│ 阶段 1: load_skill（加载技能）                                    │
│ 输入：用户请求                                                    │
│ 输出：选中的 skill_id + selected_resources（渐进式披露）            │
│ 事件：{'stage': 'load_skill', 'skill_id': 'travel-planner',        │
│        'selected_resources': ['itinerary-style.md',               │
│        'output-template.md', 'planning-criteria.md']}             │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 2: capability_gate（能力门控）                                │
│ 输入：skill 所需的全部能力列表                                     │
│ 输出：allowed=True/False + degraded 能力列表                       │
│ 事件：{'stage': 'capability_gate', 'allowed': True,               │
│        'degraded': ['wanderlog_mcp']}                             │
│ 关键：wanderlog_mcp 降级不影响主流程，allowed 仍为 True            │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 3: plan_dag（规划执行图）                                     │
│ 输入：用户上下文 + skill 资源（itinerary-style, criteria）         │
│ 输出：一张 step-DAG（节点=研究任务，边=依赖关系）                   │
│ 事件：{'stage': 'plan_dag', 'planned_steps': 5,                   │
│        'research_topics': ['weather','transport','prices',        │
│        'opening_hours']}                                          │
│ 关键：ModelRequest 让模型输出结构化 JSON 计划，不是散文           │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 4: execute_dag（执行计划图）                                  │
│ 输入：阶段 3 产出的 step-DAG                                       │
│ 输出：按拓扑序执行，逐步攒下 evidence trace                        │
│ 事件：{'stage': 'execute_dag',                                    │
│        'order': ['research_weather','research_transport',         │
│        'research_prices','research_opening_hours',                │
│        'generate_itinerary'],                                     │
│        'evidence_count': 4}                                       │
│ 关键：TaskDAG 按拓扑序执行，plan 和 execute 不再是各跑各的         │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 5: verify_and_finalize（验收与定稿）                          │
│ 输入：阶段 4 的 evidence trace + 阶段 3 的 DAG + 生成的 itinerary │
│ 输出：flow_status=passed/failed + repair_count + final answer     │
│ 事件：{'stage': 'verify_and_finalize', 'status': 'passed',        │
│        'repair_count': 0, 'review_criteria': 7}                   │
│ 关键：先确定性溯源（每个 stop 指回 evidence），再让模型逐条判     │
│       首版不过 → validate→repair 闭环兜住（最多 2 轮）             │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
最终结果（flow_status / repair_count / evidence_trace / stops）
```

#### 📖 每阶段的详细职责

**阶段 1: load_skill** —— 不是简单地把 SKILL.md 读进来。它做的是**渐进式信息披露**：skill 资源被分成多份文件（`itinerary-style.md`、`output-template.md`、`planning-criteria.md`），Host 根据用户任务选择暴露哪些资源给 Agent。这个阶段回答了"这个 skill 有哪些资源可用，当前任务需要哪几个"。

**阶段 2: capability_gate** —— 执行前先做**环境能力齐备度检查**。skill 声明了它需要的全部能力（如 `web_search`、`wanderlog_mcp`），Host 逐一检查当前环境是否具备这些能力。不具备的能力标记为 `degraded` 但不一定阻断流程——只要核心能力具备，`allowed` 仍为 `True`。本示例中 `wanderlog_mcp` 被降级，但因为用户说了"先看计划先不写 Wanderlog"，不影响流程继续。

**阶段 3: plan_dag** —— 这是**结构化判断的起点**。用 ModelRequest 让模型把用户需求翻译成一张 step-DAG：哪些研究任务需要先做（天气、交通、价格、开放时间），哪些任务依赖哪些任务的结果（生成行程依赖所有研究结果）。这一步产出的 JSON 就是下一阶段 TaskDAG 的执行对象。

**阶段 4: execute_dag** —— TaskDAG 按拓扑序执行阶段 3 的计划图。`web_search` 是显式 mock 的外部工具，只负责给时效事实。地点由模型据知识提出，不是工具给的。每完成一个研究节点就把结果攒到 evidence trace 里。这一步的关键是**计划和执行不再各跑各的**——plan_dag 产出的那张 step-DAG，正是 execute_dag 真正按拓扑序执行的对象。

**阶段 5: verify_and_finalize** —— 两段式验收。第一段是**确定性溯源**：检查每个 stop 有没有对应的 evidence，没有的直接判定为不合格。第二段是**模型 review**：让模型按 SKILL.md 自己的 review 标准（如"节奏别太赶""人均 300 以内"）逐条判。首版不过时，**用同一批证据（不重新取证）按模型反馈修一版再验**，最多 2 轮。最终 `flow_status=passed` 等价于"溯源通过 AND 模型 review 通过"。

> ⚠️ **注意事项**：verify_and_finalize 的 repair 循环有硬上限（2 轮），防止陷入无限修版。如果 2 轮都没过，`flow_status=failed`，但已经提供的信息和证据仍然会输出。

- 模型配置默认 `auto`：自动读取项目父级 `.env`，优先 DeepSeek，其次 Qwen

#### 💻 完整执行代码

```python
# 文件名: s06_full_skill_execution_flow.py（调用入口）
# 功能: 把 TriggerFlow 五个阶段串联成完整执行链并运行

import asyncio
from skill_runtime import configure_model
from s06_full_skill_execution_flow import flow

# 完整执行链 = 5 个显式阶段，每步都往 runtime stream 写一条事件，便于观察和回查：
#   load_skill -> capability_gate -> plan_dag -> execute_dag -> verify_and_finalize
# plan_dag / execute_dag 用 skill_runtime 的 planner / executor（即前面几节"展示用"的同一套逻辑的完整版）；
# verify_and_finalize 先确定性溯源，再让模型按 SKILL.md 的 review 标准逐条判，不过就修一版再验（最多 2 次）。
configure_model()
flow_task = {
    "city": "杭州", "start": "龙翔桥", "duration": "一日",
    "dates": "本周六（非周一，博物馆不闭馆）",
    "budget": "人均 300 元以内，含门票/午餐/市内交通，不含大交通与住宿",
    "preference": "团队约 10 人、含午餐、节奏别太赶、3 个点、以户外为主、适合第一次到杭州的人，先看计划先不写 Wanderlog",
}

execution = flow.create_execution(auto_close=False)
await execution.async_start(flow_task)
close_task = asyncio.create_task(execution.async_close())
async for event in execution.get_async_runtime_stream(timeout=None):
    print("阶段事件:", event)                 # 计划步数、取证条数、修了几版、验收状态都在这条运行态流里
snapshot = await close_task

final = snapshot["final"]
answer = final["answer"]
print("-" * 48)
print("验收状态  :", final["status"])                                  # passed = 溯源通过 且 模型 review 通过
print("修订轮数  :", final["repair_count"])                            # 首版没过会按反馈修订再验，最多 2 轮
print("证据条数  :", len(final["evidence_trace"]))                     # 执行时按计划逐步攒下来的时效事实
print("最终 stop :", " -> ".join(s["name"] for s in answer["stops"]))  # 地点由模型据知识提出，每次可能不同
```

### 8.5 Agently 三个基座能力的协同分工

整条执行链的核心竞争力不在单个"写得好"的 phase，而在于 Agently 三个基座能力的分工配合：

```
┌──────────────────────────────────────────────────────────────┐
│                   TriggerFlow（流程编排层）                     │
│          "管什么时候做什么、谁先谁后、什么时候收工"             │
│                                                              │
│   把 5 个阶段拆成显式阶段流，每步可观察                       │
│   load_skill → capability_gate → plan_dag                    │
│             → execute_dag → verify_and_finalize             │
│                                                              │
│   ┌─────────────────────┐   ┌─────────────────────┐          │
│   │  ModelRequest       │   │  TaskDAG             │          │
│   │  （模型请求层）      │   │  （工具执行层）       │          │
│   │                     │   │                      │          │
│   │  "管想什么"         │   │  "管做什么"          │          │
│   │  规划 step-DAG      │   │  按拓扑序执行 DAG    │          │
│   │  生成 itinerary     │   │  逐步攒 evidence     │          │
│   │  逐条 review 判定   │   │  每个节点独立可重试  │          │
│   │                     │   │                      │          │
│   │  使用于：           │   │  使用于：            │          │
│   │  plan_dag           │   │  execute_dag         │          │
│   │  verify_and_finalize│   │                      │          │
│   └─────────────────────┘   └─────────────────────┘          │
└──────────────────────────────────────────────────────────────┘
```

三个基座能力的职责边界：

| 基座能力 | 核心职责 | 在五阶段链中的角色 | 为什么不能互换 |
|---|---|---|---|
| **TriggerFlow** | 编排可观察的阶段流 | 外层驱动器，定义 5 个阶段的先后顺序、数据传递、事件协议 | 它不负责"想"也不负责"做"，只管"什么时候该做什么" |
| **ModelRequest** | 结构化模型请求 | plan_dag 中生成 step-DAG、verify_and_finalize 中逐条 review、最终生成 itinerary | 它不负责执行——它给出计划，但不跑计划 |
| **TaskDAG** | 执行计划图 | execute_dag 中按拓扑序执行 step-DAG 的每个节点 | 它不负责规划——它只执行别人给它的 DAG，不自己生成 DAG |

> ⚠️ **注意事项**：不要把 TriggerFlow 理解成"一个大函数里顺序调了 5 个小函数"。它们之间的本质区别是：TriggerFlow 管理的是**阶段流**（每个阶段有自己的生命周期、可观察事件、错误处理），TaskDAG 管理的是**任务图**（节点之间的依赖关系、拓扑排序、并行执行）。这两者在抽象层级上是不同的——TriggerFlow 是外层编排，TaskDAG 是内层执行引擎。

### 8.6 动态与稳定：哪些会变，哪些不变

这是理解 TriggerFlow 执行链可靠性的关键。我们区分两类属性：

#### 📊 动态属性——每次跑都可能不同

| 动态属性 | 谁决定 | 为什么变 | 本例实际值 |
|---|---|---|---|
| `skill_id` 与 `selected_resources` | load_skill（Host 决策） | 不同任务匹配不同 skill 和资源组合 | `travel-planner` + 3 个资源 |
| `planned_steps` | ModelRequest（模型规划） | 模型根据任务复杂度规划不同步数 | 5 |
| `research_topics` | ModelRequest（模型规划） | 模型决定要查哪些时效事实 | weather/transport/prices/opening_hours |
| `evidence_count` | TaskDAG（执行结果） | 执行中实际取到的证据条数 | 4 |
| `review_criteria_count` | ModelRequest（模型生成） | 模型按 SKILL.md 生成了几条 review 标准 | 7 |
| `stops` 及 `stop` 个数 | ModelRequest（模型知识） | 地点由模型据知识提出，每次可能不同 | 断桥→白堤→孤山→花港观鱼→灵隐寺 |
| `repair_count` | verify_and_finalize（验收判定） | 首版不过时修订轮数，0-2 | 0 |
| 模型选择 | 环境配置 | 可用模型不同 | auto（DeepSeek 或 Qwen） |

#### 🔒 稳定属性——跑 100 次也不变

| 稳定属性 | 为什么稳定 | 含义 |
|---|---|---|
| **5 个阶段全部走完** | 这是 TriggerFlow 的编排契约——只要不抛致命异常，5 个阶段必然依次执行 | 流程完整性保证 |
| **每个 stop 指回真实 evidence** | 确定性溯源的硬性要求——不满足就不过 | 输出可追溯性保证 |
| **`flow_status=passed`** 的含义 | 恒等于"溯源通过 AND 模型 review 通过"——这是 verifier 的硬性契约，不随模型、stop、证据变化而改变 | 验收标准一致性保证 |
| **plan 和 execute 的对应关系** | `plan_dag` 产出的 step-DAG 正是 `execute_dag` 执行的对象——不是各跑各的 | 计划-执行一致性保证 |
| **repair 硬上限** | 最多 2 轮——防止无限修版 | 执行时间可控性保证 |

> 💡 **核心洞见**：**动态属性是"结果"层面的变化——同样的输入每次跑出来的 stop 可能不同、修了几版可能不同。但稳定属性是"过程"层面的不变——五个阶段一定会走完、每个 stop 一定指回证据、flow_status 的含义永远不会变。** 这就是为什么这条链是可靠的：它在过程层面建立了不变量，让结果层面的变化被约束在一个可信的框架内。

---

### 8.7 核心共识回顾：结构化判断先于最终答案

贯穿整条执行链的一个核心设计共识是：**让模型先给结构化判断，而不是直接写最终答案**。

```
❌ 错误路径（传统 chatbot 模式）：
   "帮我规划杭州一日游" → 模型直接输出一篇散文式行程 → 无法验证、无法追溯

✅ 正确路径（TriggerFlow 模式）：
   "帮我规划杭州一日游"
     → plan_dag: 模型输出结构化 JSON（step-DAG） → 程序拿着计划去执行
     → execute_dag: 按计划取证，攒 evidence trace → 程序记录每一步的来源
     → verify_and_finalize: 模型逐条 review（JSON）→ 程序拿着 review 结果判过不过
     → 最终生成: 消费 evidence + 通过 review → 输出 itinerary
```

这个共识在三个环节中体现：

1. **plan_dag**：输出的是结构化 step-DAG（JSON 包含 steps、dependencies、topics），不是"我先查天气，再查交通..."的散文
2. **verify_and_finalize**：模型输出的是逐条 review 判定（JSON 包含 criterion、passed、reason），不是"这个计划不错"的散文
3. **evidence trace**：每条 evidence 有结构化的 source、timestamp、content，不是散落在段落里的引用

> ⚠️ **注意事项**：如果模型在 plan_dag 阶段就直接输出旅行散文，那后续的 execute_dag 就不知道该执行什么、verify_and_finalize 就不知道验证什么、evidence trace 就没法建立溯源关系。结构化了，后面的工具调用、证据记录、验收才接得住。

---

### 8.8 Plan 与 Execute 的闭环：不是各跑各的

这一版执行链和传统方案最大的不同是：**计划和执行不再各跑各的**。

```
传统方案（各跑各的）：
  plan("杭州一日游") → 模型输出一段计划文本
  execute("杭州一日游") → 工具独立查询，不参照计划
  问题：计划说查天气，执行可能在查美食——计划和执行脱节

TriggerFlow 方案（闭环）：
  plan_dag → 产出一张 step-DAG（JSON）
           ↓
  execute_dag → 按这张 DAG 的拓扑序逐一执行每个节点
  保证：执行的是什么 = 计划说了什么
```

`web_search` 在这个闭环中的角色是明确的：它是显式 mock 的外部工具，只负责给时效事实（天气数据、交通信息、价格、开放时间）。地点由模型据知识提出，不是工具给的。这个设计保证了"意图归模型，事实归工具"的职责清晰分离。

## 九、方法通用性证明——xlsx Skill的同一套方法

### 📋 背景

在前八个模块中，我们围绕 travel-planner 这个课堂样例，完整演示了一套标准 Skill 的执行与验证链：从 `load_standard_skill()` 加载 Skill 包、自动枚举可选资源、模型按需自选资源、宿主执行通用探针，到最终产出可溯源的确定性结论。整套方法在 travel-planner 上跑通了——但一个关键问题自然浮现：**这套方法是否绑定于 travel-planner 这个特定场景？**

回顾 travel-planner 的特征：
- 拥有 `references/` 目录（内含 `REFERENCES.md`、`itinerary-style.md` 等参考文档）
- 关键支持资源全部集中在 `references/` 目录下
- 验证方式是**模型 review 标准**：模型读取参考文档中的规则，对照执行输出给出一致性评估

如果这套方法只适用于"有 references 目录 + 模型 review 验证"这种特定形态的 Skill，那它的通用性就大打折扣——它本质上只是 travel-planner 的专用工具链，而非一套真正的"标准 Skill 解析方法"。要证伪这个担忧，我们需要一个形态完全不同的 Skill 来检验。

xlsx Skill 正是这样一个完美的对照案例。它与 travel-planner 的唯一共同点是**都遵循标准 Skill 包的结构**——这正是我们想要证明的方法边界。

### ⚠️ 问题

将同一套方法应用于 xlsx Skill，本质上在检验以下四个假设：

1. **资源枚举假设**：如果 Skill 没有 `references/` 目录，资源全部在 `scripts/` 下，`resource_catalog()` 的通用枚举还能正常工作吗？
2. **模型自主选择假设**：模型在不被提示具体资源名（如"必须选 `recalc.py`"）且未收到 xlsx 专用能力表的情况下，能从 14 个资源中自行选出正确的验证资源吗？
3. **验证方式适配假设**：如果 Skill 的验证方式不是模型 review，而是可执行脚本的确定性结论，宿主的通用执行链能平滑适配吗？
4. **通用探针假设**：四项通用执行探针（包可读、临时文件可写、已选资源可读、Python 子进程可用）在完全不同的 Skill 上仍然有效吗？

这四个假设如果全部成立，方法就证明了自身的边界是"标准 Skill 包"而非"travel-planner 场景"或"references 目录布局"。

### 🔍 分析过程

#### xlsx Skill 的背景与来源

xlsx Skill 来自 Anthropic 官方 Skills 仓库（[anthropics/skills](https://github.com/anthropics/skills)），随课收录在 `materials/xlsx/` 目录下，**原样收录、只引用不修改**（来源与许可详见 `materials/README.md`）。这是一个生产级的真实 Skill，专门用于创建和操作 Excel `.xlsx` 工作簿——与 travel-planner 的"旅行规划"场景毫无关联。

选择 xlsx Skill 作为对照案例，有三个理由：
- **来源权威**：来自 Anthropic 官方的生产级 Skill，不是课堂构造的样例，说服力更强
- **结构对立**：没有 `references/` 目录，资源全在 `scripts/`——与 travel-planner 形成完美对照
- **验证方式不同**：验证逻辑是可执行的 Python 脚本（`recalc.py`），输出确定性的数值结论，而非模型的解释性 review——代表了另一种验证范式

#### xlsx Skill 与 travel-planner 的结构差异

两个 Skill 的目录结构截然不同，但都遵循标准 Skill 包的基本约定：

```
travel-planner/                   xlsx/
├── SKILL.md                      ├── SKILL.md
├── references/                   ├── scripts/
│   ├── REFERENCES.md             │   ├── recalc.py          ← 核心验证脚本
│   ├── itinerary-style.md       │   ├── office/
│   ├── output-template.md       │   │   ├── pack.py
│   └── planning-criteria.md     │   │   ├── soffice.py
│                                │   │   └── helpers/
│                                │   │       ├── __init__.py
│                                │   │       ├── merge_runs.py
│                                │   │       └── simplify_redlines.py
│                                │   └── ...
│                                └── (其他辅助文件)
```

关键差异总结如下：

| 维度 | travel-planner | xlsx |
|---|---|---|
| 是否有 `references/` 目录 | **有** | **无**（`has_references_dir=False`） |
| 关键资源位置 | `references/` 目录 | `scripts/` 目录 |
| 资源类型 | 参考文档（kind=text） | 可执行脚本（kind=script）+ 辅助文本（kind=text） |
| 资源数量 | 3 个参考文档 | 14 个资源（script + text 混合） |
| 验证方式 | 模型 review 标准（读取文档规则，对照输出给出一致性评估） | **可执行校验脚本**（宿主执行 `recalc.py`，脚本返回确定性数值结论） |
| 结论类型 | 解释性（模型推理 + 文档对照） | 确定性（`recalc_total_errors=0` 是脚本的计算结果，不依赖模型判断） |

两个 Skill 的唯一共同点是**都遵循标准 Skill 包的结构**——都有 `SKILL.md`、都有可选的资源目录、都有清晰的元数据入口。而我们的方法恰恰建立在"标准 Skill 包"这个抽象之上，不是建立在"references 目录"或"travel-planner 场景"的具体假设之上。

#### 同一套验证方法：复用全流程

尽管两 Skill 形态迥异，验证脚本仍复用完全相同的核心流程，不做任何领域特化：

```
                    load_standard_skill(skill_dir)
                              │
                              ▼
               自动枚举可选支持资源（resource_catalog()）
                              │
                              ▼
              构建 Prompt：当前任务 + SKILL.md 全文 + 资源目录清单
                              │
                              ▼
              模型阅读后自行选择需要的资源
                   ❌ 不提示具体资源名（如"必须选 recalc.py"）
                   ❌ 不给 xlsx 专用能力表
                              │
                              ▼
              模型以自然语言列出运行时前提条件
                              │
                              ▼
              宿主执行通用探针（4 项检查，全 Skill 通用）
                              │
                              ▼
              若模型选中可执行资源 → 宿主提取 SKILL.md 中的命令并真实执行
                              │
                              ▼
              脚本返回确定性结论（recalc_status / total_errors / proof_status）
```

这个流程中有三个关键设计原则，它们共同保证了方法的通用性：

**原则一：不给模型提示具体资源名。** 验证脚本不会在 prompt 中说"你必须选 `scripts/recalc.py`"。模型在收到 xlsx 的 `SKILL.md` 全文和自动枚举的资源目录清单后，需要自己理解当前验证任务、阅读 Skill 的说明文档，然后判断哪个资源对验证任务有用。如果模型选错了资源，那说明要么枚举不够完整，要么模型的理解能力不足以匹配资源——这两种情况都是真实的边界，不应被人为的提示所掩盖。

**原则二：不给 xlsx 专用能力表。** 宿主的能力声明是通用的——"我能读取 Skill 包内的文件、我能写临时文件、我能读你选中的资源、我能启动 Python 子进程"——这些能力不是为 xlsx 定制的，travel-planner 验证时用的也是同一套声明。验证脚本没有为 xlsx 额外增加"我能操作 xlsx 工作簿"、"我安装了 openpyxl"之类的专用能力声明——模型在了解通用能力后，自己判断当前 Skill 需要什么运行时环境，并由宿主去确认这些前提。

**原则三：验证方式由 Skill 自身定义。** travel-planner 的验证方式是"模型按 review 标准给出评估"——这是 travel-planner 自身的 `SKILL.md` 决定的。xlsx 的验证方式是"执行 `recalc.py` 脚本并检查返回值"——这是 xlsx 自身的 `SKILL.md` 决定的。宿主不需要知道"这个 Skill 应该怎么验证"——它只需要知道如何按 `SKILL.md` 中的描述去执行资源。如果缺 `soffice`/`openpyxl`，脚本会优雅跳过而不报错——这也是通用性的一部分：方法不假设环境一定完备。

| **has_references_dir** | `True` | `False` |
| **资源枚举结果** | `resource_catalog()` 枚举出 3 个参考文档 | `resource_catalog()` 枚举出 14 个资源（kind 为 script 和 text） |
| **模型自主选择** | 选中 `references/` 下的参考文档 | 选中 `scripts/recalc.py`（未被提示具体名） |
| **验证方式** | **模型 review 标准**：模型读取参考文档中的规则，对照执行输出给出一致性评估 | **可执行校验脚本**：宿主执行 `scripts/recalc.py`，由脚本返回确定性数值结论 |
| **结论类型** | 解释性（模型推理 + 文档规则对照） | 确定性（`recalc_total_errors=0` 是脚本的数值计算结果） |
| **宿主能力声明** | 通用 4 项探针 | **完全相同**的通用 4 项探针 |
| **专用能力表** | 无 | 无（与 travel-planner 一致，未为 xlsx 增加任何专用能力声明） |
| **proof_status** | `passed`（经模型 review + 确定性溯源） | `passed`（经脚本数值验证） |
| **失败处理** | validate→repair 修订（最多 2 轮） | 脚本返回 `recalc_total_errors > 0` → `proof_status=failed` |

> 💡 **核心洞见**：两类 Skill 的验证方式虽然不同（模型 review vs 可执行脚本），但验证基础设施——`load_standard_skill()` → `resource_catalog()` 自动枚举 → 模型自主选择 → 通用执行探针 → 宿主执行——是完全相同的。这说明方法不绑定于任何一种验证范式，而是提供了一套**可适配不同验证需求的通用执行框架**。Skill 自身的 `SKILL.md` 定义了验证方式，宿主的职责只是按指示执行，不需要知道"这个 Skill 应该怎么验证"。

> 💡 **补充洞见**：xlsx 的验证逻辑（`recalc.py`）返回的是确定性结论——`recalc_total_errors=0` 是一个脚本计算出的数值，不依赖模型判断。这与 travel-planner 的模型 review——模型对比执行结果和参考文档规则后给出评估——形成了两种完全不同的验收范式。但这两种范式在我们的执行链中无缝共存，因为执行链只关心"如何加载 Skill、枚举资源、执行资源"，不关心"验证结论由谁做出"。

### ✅ 验证结论

xlsx Skill 验证的通过（`proof_status=passed`，`recalc_total_errors=0`），逐一回答了开篇提出的四个假设：

**假设 1：没有 `references/` 目录能否正常工作？** —— 能。`resource_catalog()` 正确枚举出 `scripts/` 下的全部 14 个资源（`resource_kinds=['script', 'text']`），枚举逻辑与资源所在的目录名无关。`has_references_dir=False` 不影响任何流程。

**假设 2：模型能自行选中正确资源吗？** —— 能。在未提示"必须选 `recalc.py`"且未给 xlsx 专用能力表的情况下，模型从 14 个资源中自行选中了 `selected_resources=['scripts/recalc.py']`。模型还列出了 5 项运行时需求（openpyxl、pandas、LibreOffice、子进程调用、recalc.py 可用），这些都是它基于 `SKILL.md` 的理解自行推断的。

**假设 3：验证方式不是模型 review 能适配吗？** —— 能。宿主从 `SKILL.md` 中提取命令格式（`python scripts/recalc.py output.xlsx`），在真实环境中创建含公式的工作簿并执行校验脚本。脚本返回 `recalc_status=success` 和 `recalc_total_errors=0`，`proof_status=passed`。整个过程不需要模型参与判断——结论由脚本的数值计算确定。

**假设 4：通用执行探针在完全不同 Skill 上有效吗？** —— 有效。四项探针全部通过：
```
host_probes=skill_package_readable:available,
            local_temp_file_rw:available,
            selected_resources_readable:available,
            python_process_available:available
host_execution_gate=ready
```
没有为 xlsx 增加任何专用探针，四项通用检查足以覆盖执行前提。

### 💡 迁移启示

这个案例的核心启示可以总结为三层，从设计哲学到实践指导逐层展开：

**第一层：抽象层级决定了方法的边界。** 我们盯住的不是"references 目录"这种实现细节，也不是"travel-planner 领域"这种业务假设，而是"标准 Skill 包"这个更高层级的抽象。只要一个 Skill 符合标准包结构——有 `SKILL.md` 作为入口、有可选的资源目录、有清晰的元数据——这套方法就能工作，无论是 Anthropic 官方仓库的生产级 xlsx Skill，还是课堂构造的学习用 travel-planner Skill。抽象层级的选择直接定义了方法的适用范围：选的太低，方法变成特化工具；选的太高，方法变成空洞框架。"标准 Skill 包"这个抽象层级恰到好处——它既有足够的具体约束（SKILL.md + 资源目录），又有足够的通用空间（不限制资源类型、不限制验证方式）。

**第二层：通用枚举 + 模型自选的组合是通用性的关键。** `resource_catalog()` 做的是"机械式枚举"——走文件系统，不关心 Skill 语义，只看目录和文件类型。模型做的是"语义选择"——理解当前任务、阅读 `SKILL.md`，在枚举结果中挑出需要的资源。这种分工意味着：枚举逻辑永远不需要为不同 Skill 定制（加新 Skill 不碰枚举代码），选择逻辑始终由模型的理解能力驱动（模型能力提升自动改善选择质量）。验证 xlsx 时不加任何"xlsx 专属选择规则"，这正是这个设计原则的体现。

**第三层：验证方式可以多样，但执行链保持统一。** travel-planner 用模型 review（解释性结论），xlsx 用可执行脚本（确定性结论）。宿主不需要实现"模型 review 执行器"或"脚本校验执行器"——它只需要一个通用的"资源执行器"：按 `SKILL.md` 中的描述调用资源，把结果返回给调用方。验证结论由 Skill 自身定义（travel-planner 的 review 标准在 `REFERENCES.md` 中，xlsx 的校验逻辑在 `recalc.py` 中），宿主不参与判断。这种设计使得方法可以轻松扩展到第三种、第四种验证范式的 Skill，无需修改执行链核心代码。

> 🔗 **关联知识**：关于通用执行探针的设计原理和 4 项检查的具体实现，详见第 8 章「完整执行链」中 travel-planner 的探针流程。xlsx 验证使用的探针代码与 travel-planner 完全一致，没有针对 xlsx 增加任何特殊处理。

> 🔗 **延伸阅读**：附录「单个 Skill 执行链——外部系统事实摘记」中的证据强弱表，从多维度对比了 travel-planner 和 xlsx 的验证证据强度（模型 review 的解释性证据 vs 脚本计算的确性证据），可作为本案例的补充阅读材料。

> 🔗 **前置知识**：xlsx 验证中复用的 `load_standard_skill()` 解析和 `resource_catalog()` 枚举逻辑，详见第 2 章「标准 Skill 包的结构与解析」。

## 十、自进化Skill——经验沉淀与生命周期治理

### 10.1 🧠 直观理解

想象你经营一家旅行社。每天，你会接到上百个"杭州一日游"的订单。头几天，你每来一个订单都要从头开始规划：查景点、排路线、订餐厅、算时间。做着做着，你发现几乎所有"杭州一日游"的客人需求都差不多——西湖+灵隐寺+河坊街，中午吃杭帮菜。

这时候，你不会在第 101 个订单打进来的时候现场发明一个流程，而是会在**下班之后**，坐下来把过去 100 次的执行记录摊开，总结出一套标准化的"杭州一日游 SOP"。第二天早上，你把这份 SOP 写到操作手册里。从这以后，新员工接到杭州一日游订单，直接翻手册第 3 章就行了。

这就是"自进化 Skill"的真实含义——**不是在执行中改写规则，而是在执行后沉淀经验、经过人工审核后写进 Skill 库**。

> 💡 **核心洞见**：老师反复强调——"所有的所谓自进化，一定不是发生在 Skill 在调用的那一个瞬间。Skill 这个东西它在被使用的时候，它是个静态的东西。"

### 10.2 📖 详细解释：自进化的真实含义

很多同学第一次听到"自进化 Skill"这个词，脑海里浮现的画面是：一个 Skill 在运行过程中，发现自己的某条指令写得不够好，于是当场把自己改好，下一次就变聪明了。

**这是一个致命的误解。** 工程上，"自进化 Skill"的准确描述是：**执行后的经验沉淀和 Skill 库治理**，它改变的是 Skill 文件，不是模型权重，也不是运行时的行为。

#### 10.2.1 关键分界：执行时 vs 执行后

整个 Skill 系统的设计有一条底线原则：**Skill 在被加载执行的那一刻是一个静态的快照**。加载什么就是什么，它不会在执行过程中偷偷变化，不会在调用工具的同时改写自己的指令。

```
        ╔═══════════════════════╗
        ║   Skill 执行过程       ║
        ║   （完全静态）         ║
        ║                     ║
        ║  引用 SKILL.md       ║
        ║  按指令调用工具       ║
        ║  返回执行结果         ║
        ║                     ║
        ║  ❌ 绝不改写 Skill   ║
        ║  ❌ 绝不杜撰新内容   ║
        ╚═══════╤═════════════╝
                │
                │  任务完成
                ▼
┌───────────────────────────────────┐
│         执行之后                   │
│    （经验沉淀与 Skill 库治理）      │
│                                   │
│   ✅ 分析执行记录                  │
│   ✅ 识别复用模式                  │
│   ✅ 提炼候选 Skill                │
│   ✅ 人工审核                      │
│   ✅ 写入 Skill 库                 │
└───────────────────────────────────┘
```

这就是老师在录音中反复强调的核心逻辑：

> "我们不会在 Skill 使用的过程中间对 Skill 本身进行改写。但是我们可以在事后通过这个分析和汇总去对 Skill 进行对应的这样的一个处理。"

为什么这是正确的工程实践？原因有三：

1. **安全性**：如果允许 Skill 在执行中自行改写，你无法预测它下一步会做什么。一个正在执行任务的 Skill 如果突然修改了自己的工具调用规则，可能带来不可控的风险。
2. **可审计性**：每一次 Skill 的变更都应该有明确的审批记录。执行中的自动修改会让问题排查变得几乎不可能——谁改的？什么时候改的？为什么改？
3. **分工清晰**：执行和执行后的治理是两个不同阶段的工作，应由不同的流程和角色负责。执行阶段只管"按指令干活"，治理阶段才管"指令本身怎么优化"。

#### 10.2.2 自进化不是"模型越用越聪明"

另一个容易混淆的概念是：把"自进化 Skill"理解为模型在训练自己。这里必须澄清——**自进化改变的是 Skill 文件（SKILL.md 等说明书文件），不是模型的权重参数**。

```
❌ 错误理解：
   模型每次执行 Skill 后，权重被更新，下次"更聪明"

✅ 正确理解：
   执行记录中发现模式 → 提炼成新的 Skill 文件 → 人工审核 → 放进 Skill 库
   下次加载时，Agent 读到的是一个更完善的指令文件，行为因此更准确
```

Agent 变"聪明"了，不是因为它的大脑（模型）被训练了，而是因为它**拿到了一份更好的说明书**。

### 10.3 🔄 执行后沉淀流程：Hermes 的完整机制

Hermes 是当前公开系统中，在 Skill 生命周期治理方面做得较为完整的一个案例。它以 `~/.hermes/skills/` 目录存放 agent 自建的 Skill，并配套写入审批、危险内容扫描、Curator 维护和归档等机制。

> ⚠️ **证据定位**：自进化 Skill 在证据强弱表中被标为"Hermes 案例"而非"通用标准"。它**不是**标准 Skill 执行链的必要环节，不同系统对 Skill 生命周期治理的实现差异很大。以下描述的流程是 Hermes 的实现方式，用于说明"执行后沉淀"的设计思想。

Hermes 的自进化触发条件是：**同一个行为模式被检测到重复执行 5 次以上**。

```
一次任务完成
    │
    ▼
┌─────────────────────────────────────────┐
│  Step 1: 模式识别                        │
│  系统检查执行记录，发现同一个"问题类型     │
│  + 工具调用链"反复出现 ≥ 5 次             │
│  老师原话："我已经检查到你做了 5 次       │
│  同样的工作"                             │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│  Step 2: 行为提炼                        │
│  将重复出现的问题和对应的工具调用链进行   │
│  抽象，提炼出一个行为模式。               │
│  老师原话："尝试把你的这个问题进行一个     │
│  提炼……变成一个可能的 Skill"              │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│  Step 3: 生成候选 Skill                  │
│  系统根据提炼出的行为模式，生成一份       │
│  Skill 候选稿（通常是 SKILL.md 的草稿），  │
│  包含核心指令、参考资源、工具映射等。     │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│  Step 4: 去重与范围检查                  │
│  检查候选 Skill 是否与已有的 Skill 重复；  │
│  确认适用范围边界——它到底覆盖哪些场景，   │
│  不覆盖哪些场景。                        │
└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│  Step 5: 危险内容扫描                    │
│  Hermes 在候选 Skill 写入前会进行安全     │

└───────────────┬─────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────┐
│  Step 6: 人工审核（关键环节）             │
│  由人对候选 Skill 进行最终评审。          │
│  老师原话："通常是要做人工审核来确定       │
│  人写进一个 Skill 库的"                   │
│  评审要点：                              │
│  - 这个模式确实有价值吗？                │
│  - 提炼出的指令准确吗？                  │
│  - 有没有遗漏关键步骤？                  │
│  - 会不会和现有 Skill 冲突？             │

```

老师特别强调了一个工程共识：

> "Amos 这个系统如果它说'我自己完全不需要人去介入，就自己去成长出一个 Skill'，这个首先它是，在我们实际落地的时候是不推荐的。"

**人工审核不是可选的附加步骤，而是工程落地的必要条件。** 原因很简单：模型提炼出的模式可能看起来合理，但在实际业务场景中可能存在盲区——比如遗漏了某个关键的合规步骤，或者误将两个应该分开的场景合并了。

### 10.4 📊 两种 Skill 创建思路对比

在 Skill 系统的实际设计中，有两种截然不同的创建 Skill 的思路。老师在课堂上对这两种思路做了明确的评析：

| 维度 | 方式 A：用户一句话创建 | 方式 B：从行为模式中抽象 |
|---|---|---|
| **做法** | 用户描述需求，模型直接生成 Skill 文件 | 系统从用户的实际使用行为中识别模式，提炼成候选 Skill |
| **输入来源** | 用户的**意图**（"我想要一个做 XX 的 Skill"） | 用户的**行为**（用户实际怎么用工具完成任务的） |
| **准确性** | 较低——用户不一定能准确描述自己的真实需求 | 较高——行为记录是用户真实工作方式的直接反映 |
| **覆盖度** | 可能遗漏用户没有意识到的步骤 | 覆盖用户实际上会做的所有步骤 |
| **用户负担** | 用户需要主动描述需求 | 用户无需额外操作，在正常使用中自动积累 |
| **典型案例** | "帮我写一个每天自动检查邮件的 Skill" | 用户连续 5 次在收到客户邮件后都执行了"查历史→核实需求→发报价"这个链路 |
| **老师评价** | 不推荐作为主要方式 | **更推荐**——"更真实、更可靠" |

老师的原话清晰地表达了这一判断：

> "沈泉同学之前问过一个对应的问题，就是你们公司其实在做一个 Skill 的系统，是希望能够用模型一句话生成 Skill。我觉得与其做那个事情，不如做我们这种——在用户使用的过程中间发现一些常见的行为模式，把工具调用的链条整理成一个更实际的、模式化的 Skill。我觉得这个可能会更合理。"

> "你让用户去不断尝试用一句话去创造一个 Skill，它其实准确度反而不如用户通过行为的方式告诉你——我这个 Skill 是如何完成的，我实际上就是如何通过你给我提供的工具完成的。那个是一个更真实的东西。"

**核心差别在于：行为不会说谎。** 用户在真实场景中反复执行的链路，比用户口头描述的"我认为我应该怎么做"更能反映实际需求。方式 B 的本质是把用户"已经在做的事"固化为可复用的 Skill，而不是让用户去"猜"需要什么样的 Skill。

### 10.5 🔄 Curator：Skill 的生命周期管理员

Hermes 的 Curator 是一个后台维护进程——它不参与 Skill 的执行，但负责 Skill 库的"新陈代谢"：哪些 Skill 是活跃的、哪些已经很少用了、哪些该归档了。

技能生命周期状态流转：

```
    ┌──────────┐     高频使用     ┌──────────┐     长期不用（~90天）   ┌──────────┐
    │  active  │ ───────────────▶ │  stale   │ ──────────────────▶ │ archived │
    │  活跃     │ ◀─────────────── │  停滞     │                       │  归档     │
    └──────────┘     重新使用      └──────────┘                       └──────────┘
```

- **active（活跃）**：近期被频繁使用的 Skill，保持在 Skill 库的显眼位置，优先被推荐和加载。
- **stale（停滞）**：一段时间未被使用的 Skill，仍在库中但优先级降低。如果重新被使用，可以回到 active 状态。
- **archived（归档）**：长期未使用（默认约 90 天）的 Skill，从活跃库中移除但保留存档。归档操作不是删除——归档的 Skill 在特殊情况下仍可以被恢复，但不再出现在常规的 Skill 列表中。

> ⚠️ **重要限制**：Curator 只管理 agent 自己创建的 Skill（存放在 `~/.hermes/skills/` 的），**不管理**捆绑 Skill 或从 Hub 安装的 Skill。你从 Skill 市场下载的、系统预装的 Skill，不会因为不使用就被 Curator 归档。

Curator 的存在说明了 Skill 生态中的一个关键理念：**Skill 库不是无限膨胀的仓库，而是需要定期打理的花园。** 随着 agent 的使用，自建的 Skill 会越来越多，其中很多可能是一次性的、过时的或已被更好的 Skill 替代的。如果没有 Curator 这样的机制，Skill 库会变成一个"垃圾场"——大量冗余 Skill 干扰 Agent 的选择和加载效率。

### 10.6 ❓ 常见误解

> ❓ **误解一："自进化就是 Skill 跑着跑着自己变聪明了"**

**纠正**：不是。自进化不是发生在执行过程中，而是发生在执行完成之后。Skill 运行的时候是静态的——它不会在执行过程中修改自己。所谓的"变聪明"，是指经过执行后的模式分析，生成了更完善的 Skill 文件，**下一次**加载时 Agent 读到的指令更好，因此行为更准确。

> ❓ **误解二："自进化就是模型在自己训练自己"**

**纠正**：不是。自进化改变的是 Skill 文件（SKILL.md 等说明书），不涉及模型权重的任何更新。模型本身没有任何变化——"进化"的是指令文件的内容和质量。理解这一点对评估 Skill 系统的实际能力至关重要。

> ❓ **误解三："所有 Skill 系统都应该有自进化"**

**纠正**：不是。自进化 Skill 是 Hermes 这个特定系统的特性，属于 Skill 生命周期治理的范畴，**不是**标准 Skill 执行链的必要组成部分。不同系统对 Skill 管理的实现差异很大。如果你的系统没有自进化机制，Skill 照样可以正常被发现、加载、执行、验证——这些才是 Skill 执行链的核心步骤。

> ❓ **误解四："自动化创建 Skill 比人工写更高效"**

**纠正**：不完全对。全自动创建的 Skill 可能看似高效，但实际工程落地时不推荐跳过人工审核。原因在于：模型提炼行为模式时可能"看起来合理但实际有坑"——比如遗漏合规步骤、合并了不该合并的场景、或者生成的指令存在歧义。人工审核确保了 Skill 的质量和安全性。效率的提升不在于去掉人，而在于让人的工作从"从零编写"变为"审核和微调"。

### 10.7 🔗 关联知识与定位

**在本课中的定位**：本节属于 Skill 生命周期治理的范畴，与 M01-M08 描述的 Skill 执行链是**两个不同的阶段**。

```
Skill 执行链（M01-M08）          Skill 生命周期治理（本节）
─────────────────────────       ─────────────────────────
• 发现 Skill                     • 分析执行记录
• 加载说明书                     • 识别复用模式
• 检查能力和权限                 • 提炼候选 Skill
• 执行工具调用                   • 人工审核
• 验证结果                       • 写入 Skill 库
• 交付最终结果                   • Curator 维护（active → stale → archived）

       ↑                                    ↑
  执行阶段（同步、在线）              治理阶段（异步、离线）
  Skill 是静态的                      修改 Skill 文件
```

**证据层级**：根据附录事实摘记中的证据强弱表，本节内容属于"Hermes 案例"级别——它展示了一种可行的设计思路，但不是跨系统的通用标准。在讲述或设计 Skill 系统时，应将其作为"可以参考的模式"而非"必须实现的规范"。

**工程底线**：无论是否引入自进化机制，Skill 系统都必须遵守一条底线——**执行中的 Skill 是静态快照，不被改写、不被篡改、不被"偷偷进化"。** 这条原则确保了 Skill 执行的可预测性、可审计性和安全性。自进化（如果有）只发生在执行之后，由专门的治理流程负责。

### 11.1 答疑：ReAct Loop vs DAG —— 不走 DAG 的流程是什么样？

---

**🙋 问题**（五班同学提问）：

> 如果不走 DAG 的方式，整个执行流程大概是什么样的？

> 📖 **背景**：在前面的课程中，老师重点讲解了 DAG（Directed Acyclic Graph，有向无环图）动态规划作为 Skill 执行的编排方式。同学由此产生一个自然的追问：如果我不做这种前置规划，那系统还能怎么跑？这就引出了 Skill 执行的另一条路径——ReAct Loop。

**💬 老师回答**：

做不做 DAG，本质上面影响的是我们**怎么去真正地执行这个 Skill**。如果你不做 DAG 动态规划，那么通常走的是一条 **ReAct Loop** 的路径。

**ReAct Loop 的完整流程**如下：

```
┌─────────────────────────────────────────────────────────────┐
│                     ReAct Loop 路径                           │
│                                                               │
│  ① 渐进式披露                                                 │
│     系统向模型逐步披露 Skill 的完整信息                        │
│     （技能描述、能力清单、约束条件等）                           │
│                         │                                     │
│                         ▼                                     │
│  ② 模型理解与判断                                              │
│     模型拿到完整信息后，基于当前状态进行判断                    │
│     决定：下一步应该做什么？                                   │
│                         │                                     │
│                         ▼                                     │
│  ③ 生成行动指令                                                │
│     模型输出一个具体的、可执行的下一步行动指令                  │
│     （调用哪个工具、传什么参数）                                │
│                         │                                     │
│                         ▼                                     │
│  ④ 执行行动                                                    │
│     系统执行该行动指令，得到结果                                │
│                         │                                     │
│                         ▼                                     │
│  ⑤ 观察结果 + 反馈                                             │
│     将执行结果（成功或失败）作为新一轮的输入                    │
│     ├── 任务未完成 ──▶ 回到 ②，继续下一轮判断                 │
│     └── 任务完成 / 配额用尽 ──▶ 结束                          │
│                                                               │
│  ⚡ Loop 配额：通常设置上限（如 10 轮），防止无限循环           │
└─────────────────────────────────────────────────────────────┘
```

对比 DAG 的路径，两者的差异非常清晰：

```
┌─────────────────────────────────────────────────────────────┐
│                      DAG 路径                                │
│                                                               │
│  ① 输入 Skill 描述 + 用户目标                                 │
│                         │                                     │
│                         ▼                                     │
│  ② 门控判断（前置）                                           │
│     检查能力是否具备：需要什么？缺少什么？                      │
│     ├── 缺能力且不可替代 ──▶ 终止，告知用户                   │
│     └── 能力齐全或可替代 ──▶ 继续                             │
│                         │                                     │
│                         ▼                                     │
│  ③ DAG 动态规划                                               │
│     根据依赖关系生成有向无环图                                 │
│     确定各节点的执行顺序                                       │
│                         │                                     │
│                         ▼                                     │
│  ④ 按图编排执行                                               │
│     依照 DAG 的拓扑顺序，依次或并行执行各节点                  │
│                         │                                     │
│                         ▼                                     │
│  ⑤ 完成                                                       │
│     每一步都是确定性的，不需要在执行中反复判断                 │
└─────────────────────────────────────────────────────────────┘
```

**两条路径的核心差异**，可以用下面这张对比表来总结：

| 维度 | ReAct Loop | DAG |
|---|---|---|
| 门控时机 | **不做前置门控**。能力缺失在执行中才会暴露，靠 loop 配额兜底 | **前置门控**。在编排之前就判断能力是否齐备，缺了就终止 |
| 执行策略 | **在做中试错**。每轮给一个下一步行动指令，根据结果动态调整 | **前置假设可行**。假定每个环节都能被完成，按计划执行 |
| 编排方式 | 边做边编，动态生成下一步 | 先编排后执行，一次性规划 |
| 失败处理 | 尝试失败会作为输入反馈到下一轮，驱动模型换一种方式再试 | 依赖前置判断质量，执行中遇到失败需要重新规划或报错 |
| 核心依赖 | Loop 配额（如 10 轮限制）和失败反馈迭代 | 前置门控判断的准确性 |
| 可控性 | 灵活但路径不确定，可能消耗较多步骤 | 确定性强但灵活性低，依赖前置假设 |

> 💡 **核心洞见**：老师特别强调——**这两种方式没有孰优孰劣**。ReAct Loop 的好处是它一定会不断地去尝试，尤其在能力池中提供了类似 Dash 这样的灵活指令时，所有的尝试失败也会作为输入，成为下一轮行动的指引。DAG 则更加前置假设，它强假设每个环节都可以被完成，因此更吃前置门控的判断质量。选哪种方式，取决于你的场景：是需要灵活试探，还是可以确定性地规划。

> 🔗 **关联知识**：ReAct Loop 的思想在前面的课程中贯穿始终——从最初的 Agent 设计到 Skill 执行框架，都是这种"观察-思考-行动"循环的延伸。DAG 动态规划则是在此基础上引入的前置编排优化。

### 11.2 答疑：提示词撰写技巧 —— Input / Instruct / Output 三层结构

---

**🙋 问题**（陈赫同学提问）：

> 提示词怎么写比较好？有什么技巧？

> 📖 **背景**：在 Skill 执行框架中，几乎每个环节都需要调用模型——门控判断、DAG 规划、行动生成、结果评估等。每个调用都需要精心设计提示词。同学直接问"怎么写 prompt 好"，这是一个非常实际且高频的问题。

**💬 老师回答**：

这个问题问得很好，但老师首先给出了一个**思维上的扭转**——调用模型的时候，我们首先要去明确的**不是怎么写提示词**，而是**先明确模型调用的目标**。

这个出发点的转变非常关键。写 prompt 很容易陷入"我要怎么措辞才能让模型听我的"这种思维，但更好的出发点是先回答三个问题：

1. **上游给了我们什么？**（输入信息是什么）
2. **我们要让模型做什么？**（行为目标是什么）
3. **下游需要模型产出什么？**（输出结构是什么）

这三个问题拆解开来，就构成了提示词的三层结构：

```
┌──────────────────────────────────────────────────────────────────┐
│                    提示词三层结构                                  │
│                                                                    │
│  Layer 1: INPUT（输入信息）                                        │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │ • 用户目标 / 任务描述                                         │ │
│  │ • Skill 的技能指引（能力清单、使用说明）                       │ │
│  │ • 已知的上下文信息（历史记录、当前状态）                       │ │
│  │                                                               │ │
│  │ 作用：为模型提供完整的决策依据                                 │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                              │                                     │
│                              ▼                                     │
│  Layer 2: INSTRUCT（行动规则）                                     │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │ • 模型需要按什么逻辑来处理输入？                               │ │
│  │ • 判断规则是什么？步骤是什么？                                 │ │
│  │ • 注意事项、约束条件                                           │ │
│  │                                                               │ │
│  │ 作用：定义模型的行为准则，告诉它「怎么想、怎么做」              │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                              │                                     │
│                              ▼                                     │
│  Layer 3: OUTPUT（结构化输出）                                     │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │ • 需要输出什么样的数据结构？                                   │ │
│  │ • 字段定义、类型约束、枚举值                                   │ │
│  │ • 输出格式规范（JSON Schema、Markdown 等）                    │ │
│  │                                                               │ │
│  │ 作用：面向下游程序 / 工程接口消费，确保输出可直接被解析使用    │ │
│  └──────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

> 💡 **核心洞见**：老师在讲解 Output 层时，点出了一个非常关键的技巧——**在定义 output 的时候，要一边想下游怎么消费，一边设计数据结构**。这不是先定义好输出再给下游用，而是一个双向迭代的过程：你设计字段的时候脑子里已经在同步想"下游拿到这个字段会怎么用"。

**以门控判断为例，看 Output 如何跟着下游需求演化：**

在 Skill 执行框架中，门控判断需要回答"当前 Skill 能不能执行"这个问题。它的 output 设计经历了如下演化：

| 阶段 | 原始想法 | 下游需要知道什么？ | Output 字段设计 |
|---|---|---|---|
| 初始 | 判断能不能执行 | 能还是不能？ | `can_execute: true/false` |
| 细化 | 不能执行的原因是什么？ | 缺了哪些能力？有没有替代方案？ | 拆分为 `needs`（需要的能力列表）、`missing`（缺失的能力列表） |
| 最终 | 如果缺了能力但可以工作，下游怎么区分？ | 三种情况：能力齐全 / 缺能力但可替代 / 缺能力且不可替代 | `needs` / `missing` / `no_missing_can_work`（缺了能力但可通过替代方案完成） |

```json
// 门控判断的最终 output 结构示例
{
  "needs": ["文件读取能力", "Python代码执行能力", "网络请求能力"],
  "missing": ["网络请求能力"],
  "no_missing_can_work": true
}
// 下游消费逻辑：
// if missing.length === 0 → 直接执行
// else if no_missing_can_work === true → 用已有能力替代执行
// else → 告知用户能力不足，终止流程
```

可以看到，`no_missing_can_work` 这个字段的出现，正是因为写 prompt 的时候一边在考虑"下游怎么使用这个判断结果"，一边决定了需要多一个字段来表达"缺了但没关系"这种微妙的状态。如果只是闷头写 output，没有同步想下游怎么消费，很容易漏掉这种关键字段。

> 💡 **延伸**：这套三层结构（input / instruct / output）是一个**可复用的方法论**，不局限于门控判断这一个场景。任何需要调用模型的环节——无论是 ReAct Loop 中的每轮行动指令生成，还是 DAG 的节点编排，都可以用这个框架来组织 prompt。核心始终是：上游给了什么 → 要做什么判断/行为 → 下游需要什么结构的数据。

### 11.3 答疑：DAG 嵌套 —— 节点执行时能再拆分为子 DAG 吗？

---

**🙋 问题**（陈涛同学提问）：

> DAG 是语言模型通过分析 Skill 的描述文件来输出的，那在执行 DAG 的某个节点时，这个节点是不是需要再细分为另外一个 DAG？类似于 DAG 的嵌套？

> 📖 **背景**：这个问题触及了 Skill 执行框架设计中的深层问题——从"任务 → Skill 拆解"到"Skill → 步骤编排"，再到"步骤 → 子步骤编排"，这个层级关系能否用 DAG 嵌套来表达？

**💬 老师回答**：

老师首先肯定了这个想法的合理性——**DAG 嵌套在理论上是可行的**。在实际的框架执行器设计中，确实已经考虑过是否存在嵌套的情况。

但是，老师紧接着给出了一个重要的**实践警告**：如果允许无限嵌套，很容易出现**递归问题**。每一层都可以再嵌套一层 DAG，那理论上会无限递归下去，既难以调试，也难以控制执行成本。

因此，目前建议的最佳实践是：

```
┌──────────────────────────────────────────────────┐
│                任务层级规划                        │
│                                                    │
│  最大任务层（用户目标）                             │
│  ┌──────────────────────────────────────────────┐ │
│  │  完成对 Skill 的拆解                          │ │
│  │  （这一层决定：需要哪些 Skill？先后顺序？）    │ │
│  └──────────────────┬───────────────────────────┘ │
│                     │                              │
│                     ▼                              │
│  Skill 层（单个 Skill 内部）                       │
│  ┌──────────────────────────────────────────────┐ │
│  │  Skill 内部只做自己的步骤编排                 │ │
│  │  （这一层决定：Skill 内部的步骤怎么做？）     │ │
│  │  不建议再嵌套子 DAG                           │ │
│  └──────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────┘
```

换句话说，**在最大的任务层完成 Skill 的拆解，在 Skill 层内部只做自己的步骤编排**，不要再往下拆出子 DAG。这样两层结构已经足够覆盖绝大多数场景。

> ⚠️ **注意事项**：老师也坦诚地提到，关于嵌套几层是最佳实践，目前还没有确定的结论，**这个领域的最佳实践还在探索中**。如果你在自己的项目中尝试了不同的层级策略，可以根据实际效果来判断什么深度适合你的场景。

### 11.4 答疑：Reference 内容压缩 —— 长文档怎么处理？

---

**🙋 问题**（同学提问）：

> Skill 的 Reference 文档内容如果很长，怎么处理？全部塞进 prompt 里 Token 消耗太大了。

> 📖 **背景**：Skill 的描述文件中可能会引用 external reference 文档作为知识补充或操作指南。但如果 reference 文档本身很长（比如完整的 API 文档、几十页的操作手册），直接把全文放进 prompt 会导致 Token 消耗飙升，既不经济也可能超出模型的上下文窗口限制。

**💬 老师回答**：

老师给出的方案非常直接——**在中间加一次模型请求做压缩**。

具体方式是：

```
┌──────────────────────────────────────────────────────┐
│              Reference 压缩流程                       │
│                                                        │
│  ① 读取 Reference 原文                                 │
│     获得完整的 reference 文档内容                       │
│                         │                              │
│                         ▼                              │
│  ② 中间模型请求（压缩层）                               │
│     Prompt: 「请从以下文档中，提取与当前任务              │
│      [任务描述] 相关的所有要点，忽略无关内容」            │
│                         │                              │
│                         ▼                              │
│  ③ 获得压缩结果                                         │
│     模型输出：与任务相关的要点摘要                       │
│     （大幅缩减原文字数，保留关键信息）                   │
│                         │                              │
│                         ▼                              │
│  ④ 将压缩结果替代原文                                    │
│     拼入后续的 prompt 中，作为上下文使用                 │
│     后续的模型调用不再携带 reference 全文                │
└──────────────────────────────────────────────────────┘
```

> 💡 **核心洞见**：这个方案的巧妙之处在于——**用一次小成本的模型调用来换取后续大成本的节省**。一次压缩请求的 Token 消耗远小于后续多轮调用每次都携带完整 reference 的消耗。而且，压缩本身也是由模型完成的，它能理解语义，比简单的文本截断或关键词匹配要精准得多。

> 💡 **延伸**：这个思路本质上是一个通用的"信息压缩"模式，不仅限于 reference 文档。任何需要在多轮调用中反复使用的长文本，都可以在第一轮做一次语义压缩，后续用压缩版替代原文。这在 Agent 的多轮对话、多步执行场景中尤其有用。

### 11.5 答疑：Tools 工具生命周期 —— 需要销毁吗？

---

**🙋 问题**（同学提问）：

> Tools 工具是在 Skill 调用的时候注册并初始化的，那当每个 Skill 执行完毕之后，Tools 工具是不是就销毁了？

> 📖 **背景**：这个问题来自于对工具生命周期管理的困惑。同学可能受到 MCP（Model Context Protocol）连接管理的影响，或者习惯于传统的"初始化-使用-销毁"资源管理模式，想确认 Tools 是否也需要类似的显式生命周期管理。

**💬 老师回答**：

老师的回答非常明确——**Tools 工具不需要销毁**。

这里需要区分两种不同类型的连接方式：

| 类型 | 特点 | 典型例子 | 是否需要销毁 |
|---|---|---|---|
| 长连接（Persistent） | 建立后持续保持，复用连接 | 数据库连接池、WebSocket | 需要管理生命周期，适时释放 |
| 按需连接（On-Demand） | 每次使用时建立，用完即断 | HTTP API 调用、函数执行 | 不需要销毁，调用结束即释放 |

Tools 工具的执行模式属于**按需连接**——工具内部的函数在执行时，"连接-查询-返回"这三个动作是连在一起、一次调用完成的。因此它的生命周期天然控制在**每一次行为调用之内**，不需要额外的销毁逻辑。

更重要的是，**模型的行为频度决定了不需要长连接**。模型调用 Tool 的频率（几秒到几十秒一次）远低于传统程序中"毫秒级"的高频调用场景。如果是一秒几百次的数据库查询，长连接是必要的；但模型每轮对话可能只调用一两次工具，每次工具调用都是独立的、可自包含的，所以按需连接完全足够。

> ⚠️ **注意事项**：老师特别指出，同学问的那种"注册-初始化-销毁"的模式，更像是在描述 **MCP 连接**的行为——MCP 涉及 Client 与 Server 之间的长连接管理，确实需要考虑生命周期。但普通的 Tools 工具不是这个模式，不需要混淆。

> 💡 **延伸**：这个设计选择也体现了 Agent 框架的一个重要设计哲学——**工具应该是无状态的、可随时调用的**。无状态的工具不需要管理生命周期，不持有持久连接，每个调用都自包含。这让系统的复杂度大大降低，也更容易做错误处理和重试。

### 11.6 答疑：代码部署 —— 如何让能力被大家使用？

---

**🙋 问题**（同学提问）：

> 写好的代码、做好的能力，怎么部署出去让大家使用？

> 📖 **背景**：课程中讲了 Agent 的设计、Skill 的编排、工具的注册等大量"怎么写"的内容，但同学关心的是最终一步——"怎么让别人能用上"。这是从开发到交付的关键一跳。

**💬 老师回答**：

老师的回答简洁有力，直指核心——**最核心的方式就是把它服务化**。

```
┌──────────────────────────────────────────────────────┐
│                   服务化部署模式                       │
│                                                        │
│  ┌──────────┐      接口调用       ┌──────────────────┐ │
│  │  调用方   │ ─────────────────▶ │   你的服务         │ │
│  │ (任何人)  │                    │  (Agent / Skill)  │ │
│  │           │ ◀───────────────── │                   │ │
│  └──────────┘      返回结果       └──────────────────┘ │
│                                                        │
│  关键点：                                              │
│  • 调用方只看到接口，看不到内部实现                     │
│  • 接口定义了输入/输出的契约                           │
│  • 内部怎么实现（DAG 还是 ReAct Loop）对调用方透明      │
└──────────────────────────────────────────────────────┘
```

服务化之后，你的能力就变成了一个**接口**。调用方不需要知道你内部是怎么实现的——用的是 DAG 还是 ReAct Loop，Skill 内部有几个步骤，Tools 是怎么连接的——这些全部被封装在接口后面。调用方只需要按接口约定的格式发请求、收结果。

老师在这里引用了**第 5 课**的内容——在那节课中已经详细讲过服务化的设计理念和工程实践，本课的核心是在前面课程基础上的延伸。

> 💡 **延伸**：服务化不仅是部署手段，更是一种**架构思维方式**。当你把自己的 Agent 或 Skill 封装成一个服务，你就强制自己定义了清晰的边界——输入是什么、输出是什么、接口契约是什么。这个过程本身就会反过来促进你对 Skill 设计质量的审视：是不是输入输出足够清晰？是不是内部逻辑足够内聚？这些在写代码时可能被忽略的问题，在做服务化封装时都会变得显眼。

> 🔗 **前置知识**：关于服务化部署的工程实践和接口设计细节，详见 [第 5 课：服务化与接口设计]（← 合并时替换为锚点链接）。

### 💡 一句话总结

> **Skill 执行不是让模型背说明书，而是让系统按标准 Skill 格式组织一次有证据的任务完成过程。**

这句话点出了本节课最核心的设计哲学：模型负责「理解与规划」，系统负责「执行与取证」，Skill 文件负责「定义标准与验收规则」。三者各司其职，模型自身的自然语言输出不作为可验收证据——工具返回、脚本输出、外部系统回执才是真正的 evidence trace。

### ✅ 六大要点清单

本节课的全部知识点，可以凝练为以下六个要点：

| # | 要点 | 一句话解释 |
|---|------|-----------|
| **1** | **Skill 执行主线** | 发现加载 → 能力门控 → 执行规划 → 按计划执行（取证+生成）→ 证据验收 → 交付，六步构成完整闭环 |
| **2** | **标准格式不等于固定目录约定** | `SKILL.md` 是必读说明书，支持资源由运行时枚举后交给模型按需选择，目录结构灵活但不失规范 |
| **3** | **规划产出是可执行 step-DAG** | 执行器的本质就四件事：拓扑序 + 按节点类型分发 + 收集结果 + 沿 `depends_on` 传产物。本课用 MVP 原型讲清原理，生产环境交给 Agently **TaskDAG** 基座 |
| **4** | **说明书给流程，工具给事实** | 说明书（SKILL.md）告诉系统「应该怎么做」，工具返回和脚本输出提供「实际发生了什么」。模型自然语言不是证据，evidence trace / EvidenceEnvelope 才是 |
| **5** | **验收用 Skill 自己的 validate 方法** | 确定性规则用代码自动溯源校验，主观标准由模型按 SKILL.md 的 review 条目逐条判；不通过就把不达标点当反馈，修复后重验（validate → repair 循环） |
| **6** | **三个基座能力支撑整条链** | **ModelRequest**（规划/生成/验收）、**TaskDAG**（执行计划图引擎）、**TriggerFlow**（编排可观察的阶段流）——三者构成 Agently 框架执行 Skill 的底层能力三角 |

> ⚠️ **注意事项**：自进化 Skill（系统从单次执行中提取经验、沉淀为通用规则并修改 SKILL.md 文件）是本课作为概念拓展介绍的内容，**不是标准 Skill 执行链的必备步骤**，也不涉及模型权重训练。它发生在一次完整执行之后，由评审者人工确认后写入 Skill 库——执行过程本身只引用、不篡改 SKILL.md。

#### 练习 2：设计一个 validate 方法

**题目**：为你自己定义的一个 Skill（可以是虚拟的）设计一套 validate 方法，要求同时包含「确定性校验」和「模型判断」两类规则。

示例场景：假设你有一个「代码审查 Skill」，它的任务是审查一段代码并给出改进建议。请写出它的 validate 规则，至少包含：
- 3 条确定性校验（用代码就能判断是否通过的规则）
- 2 条模型判断（需要模型按 review 标准主观判定的规则）
- 对每条规则，说明不通过时应该如何触发 repair

**验收标准**：
- [ ] 确定性校验规则具备可编程性（能用 if/正则/断言表达）
- [ ] 模型判断的 review 标准具有可操作性（不是「做得好」这种模糊表述）
- [ ] 每条规则都给出了不通过时的 repair 建议
- [ ] 说明了 validate → repair 循环的终止条件（什么时候可以结束循环）

### 🔮 下节课预告：多 Agent 任务分派

到这里，我们已经完整走完了 Skills 能力体系的「结构 → 选择 → 执行」三节课。加上前面课程中学到的 MCP 协议和 Tools 工具调用能力，我们手中已经有了一个相当完整的智能 Agent 构建工具箱。

但是——还有一个拼图没有补上：

> 我们的 Agent 系统目前只能在「自己的能力边界内」完成任务。如果遇到一个超出当前系统能力的任务——比如需要用到本地安装的 Codex、Claude Code、或者云端部署的 Open Cloud Agent——我们能不能把子任务**分包出去**，让外部 Agent 系统替我们完成？

这就是下节课要探讨的主题：**多 Agent 任务分派**。

```
┌──────────────────────────────────────────────┐
│              你的 Agent Host                  │
│                                              │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│   │   MCP    │  │  Tools   │  │  Skills  │  │
│   │  工具接入 │  │  工具调用 │  │  能力封装 │  │
│   └──────────┘  └──────────┘  └──────────┘  │
│                                              │
│   ┌──────────────────────────────────────┐   │
│   │        多 Agent 任务分派  ← NEW       │   │
│   │                                      │   │
│   │  ┌──────┐  ┌──────────┐  ┌───────┐  │   │
│   │  │Codex │  │Claude    │  │Open   │  │   │
│   │  │      │  │Code      │  │Cloud  │  │   │
│   │  └──────┘  └──────────┘  └───────┘  │   │
│   │     外部 Agent 系统（本地 / 云端）     │   │
│   └──────────────────────────────────────┘   │
│                                              │
└──────────────────────────────────────────────┘
```

我们会讨论的核心问题包括：

- 什么任务适合分包给外部 Agent？分包的粒度如何把握？
- 外部 Agent 系统之间如何通信？用统一的协议还是各自适配？
- 分包出去的任务如何追踪状态、收集 evidence、做最终验收？
- 多系统设计的可靠性考量：任务失败、超时、结果冲突怎么处理？
