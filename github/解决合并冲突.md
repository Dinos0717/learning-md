## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经掌握了：

- **分支操作** —— 创建分支、切换分支、删除分支
- **分支合并** —— Merge、Squash、Rebase 三种合并策略
- **四个分区** —— 工作区、暂存区、本地存储库、远端存储库
- **进阶操作** —— Stash、Reset、Revert、Amend、Tag、Cherry Pick

在这些操作中，合并是最容易出问题的环节。当你和同事（或你自己在不同分支上）**同时修改了同一个文件的同一行代码**，Git 就无法自动决定该保留谁的修改——这就是**冲突（Conflict）**。

> 冲突不是 Bug，而是 Git 在说："你们两个都改了这里，我不知道该听谁的，你来决定吧。"

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 冲突是怎么产生的？ | 两个分支同时修改同一个文件的同一行，Git 无法自动合并 |
| 冲突标记长什么样？ | `<<<<<<<`、`=======`、`>>>>>>>` 分别代表什么 |
| 如何在 Desktop 中解决冲突？ | 分支合并冲突、远端 Pull 冲突的完整解决流程 |
| 如何用命令行解决冲突？ | `git merge` → 编辑文件 → `git add` → `git commit` |
| 文件被重命名了怎么解决冲突？ | 两边把同一个文件改成了不同名字的冲突处理方法 |
| Stash 在解决冲突中起什么作用？ | Pull 冲突时用 `stash` 暂存本地修改，手动合并后还原 |

## 二、冲突的本质——Git 为什么无法自动合并

### 2.1 正常合并 vs 冲突合并

Git 在合并两个分支时，会比较两个分支相对于它们共同祖先的变化。结果可能是：

```
┌─────────────────────────────────────────────────────────┐
│            Git 合并的三种结果                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  情况1：Fast-Forward（快进合并）                          │
│  ├── 一个分支没有任何新提交，另一个分支有                  │
│  ├── Git 直接把指针移到最新提交                          │
│  └── ✅ 无冲突，完全自动                                  │
│                                                         │
│  情况2：自动合并（Auto Merge）                            │
│  ├── 两个分支都有新提交，但修改的是不同的文件/不同行       │
│  ├── Git 自动把两边的修改合并到一起                       │
│  └── ✅ 无冲突，完全自动                                  │
│                                                         │
│  情况3：冲突（Conflict）                                  │
│  ├── 两个分支修改了同一个文件的同一行                     │
│  ├── Git 无法判断该保留哪一个修改                         │
│  └── ⚠️ 需要人工介入，手动解决                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**一个直观的例子：**

```
共同的祖先版本（两个分支从这里分叉）：
    colors.txt 内容：
    red
    green
    blue

main 分支的修改：              feature 分支的修改：
    red                            red
    green                          green
    blue                           blue
    black          ← 第4行         yellow         ← 第4行
    pink           ← 第5行         purple         ← 第5行

当 main 合并 feature 时：
    Git 发现第 4 行: main 写了 "black"，feature 写了 "yellow"
    Git 发现第 5 行: main 写了 "pink"，feature 写了 "purple"
    → Git 不知道该保留谁 → 触发冲突！
```

### 2.2 冲突标记（Conflict Markers）详解

当冲突发生时，Git 会在冲突文件中插入特殊的标记。这些标记是理解冲突的关键：

```
<<<<<<< HEAD
这是当前分支（你正在操作的分支）的内容
=======
这是要合并进来的分支的内容
>>>>>>> feature
```

```
┌─────────────────────────────────────────────────────────┐
│              冲突标记的详细含义                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  <<<<<<< HEAD                                           │
│  （或 <<<<<<< current-branch-name）                       │
│  ├── "从这里开始，是当前分支的代码"                        │
│  ├── HEAD = 你当前所在的分支（接收合并的分支）            │
│  └── 这是版本 A                                          │
│                                                         │
│  =======                                                │
│  ├── "这是分隔线，上面是版本A，下面是版本B"                │
│  └── 这是"分界线"                                        │
│                                                         │
│  >>>>>>> feature                                        │
│  （或 >>>>>>> incoming-branch-name）                      │
│  ├── "到这里结束，这是要合并进来的分支的代码"              │
│  ├── feature = 即将合并进来的分支名                       │
│  └── 这是版本 B                                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**实际冲突文件的样子：**

```
red
green
blue
<<<<<<< HEAD
black          ← main 分支写的
pink           ← main 分支写的
=======
yellow         ← feature 分支写的
purple         ← feature 分支写的
>>>>>>> feature
```

### 2.3 解决冲突的核心思想

> **解决冲突 = 编辑这个文件，把它修改成你希望它最终成为的样子，然后删掉所有冲突标记。**

```
冲突文件（修改前）：              解决后的文件（修改后）：
red                                red
green                              green
blue                               blue
<<<<<<< HEAD                       black
black                              yellow
pink                               pink
=======                            purple
yellow
purple
>>>>>>> feature

↑ 有冲突标记，Git 不知道该保留谁    ↑ 冲突标记已删除，保留了所有想要的内容
```

**解决冲突的四个步骤（无论用什么工具，本质都一样）：**

```
Step 1: 打开冲突文件
    ↓
Step 2: 手动编辑，决定保留哪些内容
    ├── 保留版本A（HEAD）的内容 → 删掉版本B和所有标记
    ├── 保留版本B（incoming）的内容 → 删掉版本A和所有标记
    ├── 两个版本都保留（手动合并）→ 删掉所有标记
    └── 两个版本都不要，写全新的内容 → 删掉所有标记，写新的
    ↓
Step 3: 删除所有冲突标记（<<<<<<< ======= >>>>>>>）
    ↓
Step 4: 保存文件 → 标记为已解决 → 提交
    ↓
🎉 冲突解决完毕！
```

## 三、场景一：分支合并冲突（GitHub Desktop）

### 3.1 场景搭建

**准备工作**：以 `hello-world` 项目为例，项目中有一个 `colors.txt` 文件。

```
初始状态（main 和 feature 分支内容相同）：

colors.txt:
    red
    green
    blue
```

**开始制造冲突：**

```
┌─────────────────────────────────────────────────────────┐
│  步骤1：在 feature 分支上修改                             │
├─────────────────────────────────────────────────────────┤
│  $ git checkout feature                                │
│  编辑 colors.txt，在第4、5行添加：                       │
│                                                         │
│  red                                                    │
│  green                                                  │
│  blue                                                   │
│  yellow     ← 新增                                      │
│  black      ← 新增                                      │
│                                                         │
│  提交：git commit -m "feature: add yellow and black"    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  步骤2：切换到 main 分支，也在同一位置修改                │
├─────────────────────────────────────────────────────────┤
│  $ git checkout main                                   │
│  编辑 colors.txt，在第4、5行添加：                       │
│                                                         │
│  red                                                    │
│  green                                                  │
│  blue                                                   │
│  black      ← 新增（和 feature 同一行！）               │
│  pink       ← 新增（和 feature 同一行！）               │
│                                                         │
│  提交：git commit -m "main: add black and pink"         │
└─────────────────────────────────────────────────────────┘
```

```
现在的分支状态：

main 分支：         feature 分支：
red                 red
green               green
blue                blue
black    ← 冲突！    yellow   ← 冲突！
pink     ← 冲突！    black    ← 冲突！
```

### 3.2 触发冲突

在 GitHub Desktop 中，将 feature 分支合并到 main 分支：

```
┌─────────────────────────────────────────────────────────┐
│  操作：在 Desktop 中点击 "Choose a branch to merge       │
│        into main" → 选择 feature                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  点击后 Desktop 提示：                                   │
│  ┌──────────────────────────────────────────────┐       │
│  │  ⚠️ There will be 1 conflicted file           │       │
│  │     colors.txt                                │       │
│  │                                              │       │
│  │  [Create a merge commit]                      │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  点击 "Create a merge commit" 后：                       │
│  ┌──────────────────────────────────────────────┐       │
│  │  ⚠️ 1 conflicted file                        │       │
│  │     colors.txt                                │       │
│  │                                              │       │
│  │  [Open in VS Code]  [Open in Finder]         │       │
│  │                                              │       │
│  │  [Abort Merge]    [Continue Merge] (灰色)     │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  关键信息：                                              │
│  ├── Continue Merge 按钮目前是灰色的（不能点）           │
│  ├── 必须先解决所有冲突文件，按钮才会变绿                │
│  └── Abort Merge 可以取消本次合并，回到合并前的状态      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 在 VS Code 中解决冲突

点击 "Open in VS Code"，打开冲突文件：

```
┌─────────────────────────────────────────────────────────┐
│  VS Code 中看到的冲突文件内容：                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  red                                                    │
│  green                                                  │
│  blue                                                   │
│  <<<<<<< HEAD                     ← 当前分支(main)的内容 │
│  black                                                   │
│  pink                                                    │
│  =======                          ← 分隔线              │
│  yellow                           ← feature 分支的内容  │
│  black                                                   │
│  >>>>>>> feature                  ← 合并来源分支        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**手动解决冲突——逐行编辑：**

```
你的目标：让 colors.txt 包含所有想要的颜色

编辑步骤：

1. 分析：HEAD(main) 有 black 和 pink
         feature 有 yellow 和 black
         → black 重复了，保留一份即可
         → 四个颜色都要：black, yellow, pink

2. 删除冲突标记行：
   删除 <<<<<<< HEAD
   删除 =======
   删除 >>>>>>> feature

3. 保留你想要的内容：
   保留 black（只保留一份）
   保留 yellow
   保留 pink

4. 最终文件内容：
   red
   green
   blue
   black
   yellow
   pink
```

```
┌─────────────────────────────────────────────────────────┐
│           VS Code 中冲突标记的可视化                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  VS Code 会用颜色高亮冲突区域：                           │
│  ├── 绿色背景 = 当前分支（HEAD）的修改                   │
│  ├── 蓝色背景 = 合并进来分支的修改                       │
│  └── 上方有快捷按钮（见第七章）                          │
│                                                         │
│  按 Ctrl+S（Mac: Cmd+S）保存文件                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.4 完成合并并推送

```
解决冲突后，回到 GitHub Desktop：

┌─────────────────────────────────────────────────────────┐
│  Desktop 中的变化：                                      │
│                                                         │
│  ┌──────────────────────────────────────────────┐       │
│  │  ✅ 1 conflicted file  → 前面出现绿色对勾      │       │
│  │     colors.txt                                │       │
│  │                                              │       │
│  │  [Abort Merge]    [Continue Merge] (绿色!)    │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  1. 点击绿色的 "Continue Merge" 按钮                     │
│     → 合并完成！历史中多了一条 "Merge branch feature"    │
│                                                         │
│  2. 点击 "Push origin" 推送到远端                        │
│     → 远端也同步了合并结果                               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
整个流程回顾：

feature 分支修改          main 分支修改
     │                        │
     └────────┬───────────────┘
              │
         git merge feature
              │
         ⚠️ 冲突出现！
              │
         打开冲突文件
              │
         手动编辑 → 删除冲突标记 → 保留想要的内容
              │
         保存文件
              │
         ✅ 冲突已解决 → Continue Merge
              │
         git push → 完成！
```

## 四、场景二：远端同步冲突（Pull 冲突）

### 4.1 场景说明

> 这是多人协作中最容易遇到的冲突场景。你的同事写完代码并推送到了远端，而你不知情，在本地修改了同一个文件的同一行。当你 `git pull` 拉取远端更新时，冲突就产生了。

```
┌─────────────────────────────────────────────────────────┐
│           Pull 冲突的产生过程                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  远端仓库（GitHub）：                                    │
│  colors.txt 第4、5行：                                   │
│  yellow                                                  │
│  pink                                                    │
│  （同事已经 push 上去了）                                │
│                                                         │
│  你本地（feature 分支）：                                │
│  colors.txt 第4、5行：                                   │
│  black                                                   │
│  purple                                                  │
│  （你在本地也改了同样的位置）                             │
│                                                         │
│  当你 git pull：                                         │
│  ├── Git 尝试把远端的 yellow/pink 合并到你的本地         │
│  ├── 发现你本地已经改了第4、5行为 black/purple           │
│  └── Git 不知道该保留谁的 → ⚠️ 冲突！                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 Stash and Continue 流程

在 GitHub Desktop 中拉取远端更新时，如果本地有未提交的修改，Desktop 会提供 `Stash and Continue` 选项：

```
┌─────────────────────────────────────────────────────────┐
│  Desktop 操作流程：                                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: 点击 "Pull origin" 或 "Fetch origin"           │
│           Desktop 检测到本地有未提交的修改               │
│                                                         │
│  Step 2: Desktop 提示：                                  │
│  ┌──────────────────────────────────────────────┐       │
│  │  ⚠️ Pulling will overwrite your local changes │       │
│  │                                              │       │
│  │  [Stash and Continue]                        │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  Step 3: 点击 "Stash and Continue"                      │
│          ├── Desktop 把你的本地修改暂存（stash）起来     │
│          ├── 先从远端拉取最新代码                        │
│          └── 然后尝试还原你的本地修改                    │
│                                                         │
│  Step 4: 还原 stash 时发现冲突！                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │  ⚠️ 1 conflicted file                            │   │
│  │     colors.txt                                    │   │
│  │                                                  │   │
│  │  左侧边栏出现 "Stash" 区域，显示你被暂存的修改     │   │
│  └──────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Stash 是什么？

Stash（暂存）= 把你的本地修改临时"藏起来"

git stash 做的事情：
┌─────────────────────────────────────┐
│  1. 把你工作区的修改打包保存          │
│  2. 把你的工作区恢复成干净状态        │
│  3. 你就可以安全地 pull 远端代码      │
│  4. pull 完后再把 stash 还原回来      │
│     （还原时可能触发冲突）            │
└─────────────────────────────────────┘

Stash 就像一个"草稿箱"：
先把未完成的工作放进去 → 处理紧急事情 → 再取出来继续
```

### 4.3 解决冲突并推送

```
还原 stash 后出现冲突，解决方式和场景一相同：

Step 1: 点击冲突文件，Open in VS Code

Step 2: 在 VS Code 中编辑冲突文件
        ← 方法和场景一完全一样：
        ← 删除 <<<<<<< ======= >>>>>>> 标记
        ← 保留你想要的所有颜色

Step 3: 保存文件

Step 4: 回到 Desktop，填写 commit message
        例如："解决远端同步冲突——合并 colors.txt"

Step 5: 点击 "Commit to feature"

Step 6: 点击 "Push origin" 推送到远端

✅ 远端同步冲突解决完毕！
```

```
⚠️ 关键认知：

Pull 冲突的本质和 Merge 冲突是一样的——
都是 Git 发现两边改了同一行，不会自动合并。

区别只在于触发方式：
├── Merge 冲突：你主动执行 git merge 时触发
└── Pull 冲突：你执行 git pull（= fetch + merge）时触发
```

## 五、场景三：文件名冲突

### 5.1 场景搭建

> 当两个分支都把同一个文件重命名成了不同的名字，Git 无法判断最终该保留哪个文件名。

```
┌─────────────────────────────────────────────────────────┐
│  场景：两个分支修改了同一个文件的文件名                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  初始状态：项目中有一个 colors.txt 文件                  │
│                                                         │
│  main 分支：                                            │
│  ├── 把 colors.txt 重命名为 colors-m.txt                │
│  └── 提交："rename to colors-m.txt"                     │
│                                                         │
│  feature 分支：                                         │
│  ├── 把 colors.txt 重命名为 colors-f.txt                │
│  └── 提交："rename to colors-f.txt"                     │
│                                                         │
│  当合并时：                                             │
│  ├── main 说：这个文件叫 colors-m.txt                   │
│  ├── feature 说：这个文件叫 colors-f.txt                │
│  └── Git 无法决定保留哪个名字 → ⚠️ 冲突！              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 GitHub Desktop 解决方案

> ⚠️ 文件名冲突在 GitHub Desktop 中的 UI 提示比较"迷惑"，不太容易理解。老师建议采用更直接的方式。

```
┌─────────────────────────────────────────────────────────┐
│  Desktop 操作（推荐方式）：                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: Desktop 提示有 3 个冲突文件（很迷惑）           │
│          此时不要纠结 Desktop 的提示                     │
│                                                         │
│  Step 2: 把所有冲突文件的更改撤销（Discard）             │
│          ├── 右键 → Discard changes                     │
│          └── 目的是清掉 Git 的冲突标记                   │
│                                                         │
│  Step 3: 直接在文件系统中手动调整                        │
│          ├── 打开项目文件夹                              │
│          ├── 决定保留哪个文件名                          │
│          │   例如：决定保留 colors-m.txt                 │
│          ├── 删除不需要的文件（colors-f.txt）            │
│          └── 确保最终目录是你想要的样子                  │
│                                                         │
│  Step 4: 回到 Desktop                                   │
│          ├── Desktop 检测到文件变化                      │
│          └── 冲突标记自动打上绿色对勾 ✅                 │
│                                                         │
│  Step 5: 点击 "Continue Merge" → 合并完成               │
│                                                         │
│  Step 6: Push 到远端                                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.3 命令行解决方案

文件名冲突在命令行中的提示更加清晰：

```bash
# ── 当前在 feature 分支，准备合并 main ──

# Step 1: 执行合并
$ git merge main
# 输出：
# CONFLICT (rename/delete): colors.txt deleted in main and renamed to colors-f.txt in feature.
# CONFLICT (rename/rename): colors.txt renamed to colors-m.txt in main and to colors-f.txt in feature.
# Automatic merge failed; fix conflicts and then commit the result.


# Step 2: 查看冲突状态
$ git status
# 输出关键信息（红字部分）：
#   deleted by us:   colors.txt          ← feature 分支"删除"了原始文件名
#   added by them:   colors-m.txt        ← main 分支添加了这个文件
#   added by us:     colors-f.txt        ← feature 分支添加了这个文件


# Step 3: 按照 git status 的提示操作
# 先处理"两个分支都删除"的原始文件
$ git rm colors.txt

# 决定保留哪个文件 → 假设保留 colors-m.txt
# 删除不想要的文件
$ git rm colors-f.txt

# 确保想要的文件在目录中
# （colors-m.txt 应该已经在目录中了，来自 main 分支）


# Step 4: 再次查看状态，确认无冲突
$ git status
# 输出应该是绿色的，没有冲突标记


# Step 5: 提交解决结果
$ git commit -m "resolve file name conflict - keep colors-m.txt"


# Step 6: 推送到远端
$ git push
```

```
│          文件名冲突的命令行处理要点                        │
├─────────────────────────────────────────────────────────┤
│                                                         │

│      它会告诉你具体哪些文件冲突了                         │
│                                                         │
│  2️⃣  git rm 用于删除冲突中不想要的文件                   │
│      ├── git rm <被两个分支都删除的原始文件>              │
│      └── git rm <你不想要的那个重命名文件>                │
│                                                         │
│  3️⃣  最终目标：                                          │
│      文件系统中只保留你想要的那一个文件                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 六、纯命令行解决冲突

### 6.1 完整命令流程

不依赖 GitHub Desktop，完全用命令行来解决合并冲突：

```bash
# ── 场景：当前在 main 分支，合并 feature 分支时产生冲突 ──


# ══════════════════════════════════════════════════════
# Step 1: 执行合并，触发冲突
# ══════════════════════════════════════════════════════

$ git merge feature

# 输出：
# Auto-merging colors.txt
# CONFLICT (content): Merge conflict in colors.txt
# Automatic merge failed; fix conflicts and then commit the result.
#
# ↑ 关键信息：
#   - "CONFLICT (content)" → 内容冲突
#   - "Merge conflict in colors.txt" → 冲突文件是 colors.txt
#   - "fix conflicts and then commit" → 先解决冲突，再提交


# ══════════════════════════════════════════════════════
# Step 2: 查看冲突状态
# ══════════════════════════════════════════════════════

$ git status

# 输出：
# On branch main
# You have unmerged paths.
#   (fix conflicts and run "git commit")
#
# Unmerged paths:
#   (use "git add <file>..." to mark resolution)
#         both modified:   colors.txt
#
# ↑ "both modified" = 两个分支都修改了这个文件


# ══════════════════════════════════════════════════════
# Step 3: 手动编辑冲突文件
# ══════════════════════════════════════════════════════

# 用任意编辑器打开冲突文件
$ code colors.txt        # VS Code
# 或
$ vim colors.txt         # Vim
# 或
$ nano colors.txt        # Nano
# 或直接在文件管理器中双击打开

# 文件内容如下：
# red
# green
# blue
# <<<<<<< HEAD
# black
# pink
# =======
# yellow
# black
# >>>>>>> feature

# 手动编辑：删除冲突标记，保留想要的内容
# 编辑后：
# red
# green
# blue
# black
# yellow
# pink
# purple

# 保存并关闭文件


# ══════════════════════════════════════════════════════
# Step 4: 标记冲突已解决
# ══════════════════════════════════════════════════════

$ git add colors.txt

# git add 有两个含义：
# 1. 正常情况：把文件添加到暂存区
# 2. 冲突解决时：告诉 Git "这个文件的冲突我已经解决了"


# ══════════════════════════════════════════════════════
# Step 5: 检查状态（确认无残留冲突）
# ══════════════════════════════════════════════════════

$ git status

# 输出：
# On branch main
# All conflicts fixed but you are still merging.
#   (use "git commit" to conclude merge)
#
# Changes to be committed:
#         modified:   colors.txt
#
# ↑ "All conflicts fixed" = 所有冲突已解决 ✅


# ══════════════════════════════════════════════════════
# Step 6: 提交合并结果
# ══════════════════════════════════════════════════════

$ git commit -m "resolve color file conflict"

# 不需要额外参数，Git 知道这是一个合并提交


# ══════════════════════════════════════════════════════
# Step 7: 推送到远端
# ══════════════════════════════════════════════════════

$ git push

# 🎉 冲突解决完毕！
```

### 6.2 命令行 vs Desktop 对比

| 维度 | 命令行 | GitHub Desktop |
|---|---|---|
| **冲突提示** | `CONFLICT (content): Merge conflict in colors.txt` | 红色叹号 + "1 conflicted file" |
| **查看冲突文件** | `git status` | 左侧文件列表直接显示 |
| **编辑冲突** | 手动打开文件编辑 | 点击 "Open in VS Code" |
| **标记已解决** | `git add <文件名>` | 文件前的勾自动变绿 |
| **取消合并** | `git merge --abort` | 点击 "Abort Merge" 按钮 |
| **提交** | `git commit -m "..."` | 点击 "Continue Merge" |
| **优势** | 精确控制每一步，适合复杂场景 | 可视化直观，适合日常场景 |
| **劣势** | 需要记忆命令 | 复杂冲突（如文件名冲突）提示不够清晰 |

## 九、常见问题与排错指南

### 9.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| **Continue Merge 按钮灰色，点不了** | 还有冲突文件没有解决（没有 `git add` 标记） | 检查所有冲突文件是否都已编辑保存，确保 Desktop 中冲突文件前显示绿色对勾 | 🔴 高 |
| **git status 显示 "You are still merging"** | 合并尚未完成，需要提交或放弃 | 解决完冲突后执行 `git commit`；如果要取消，执行 `git merge --abort` | 🔴 高 |
| **冲突文件中出现看不懂的标记** | 这是 Git 的标准冲突标记 | `<<<<<<<` 上面是当前分支，`=======` 下面是合并分支，`>>>>>>>` 是结束 | 🟡 中 |
| **解决了冲突但提交后代码不对** | 解决冲突时误删了重要代码 | 用 `git log` 查看历史，用 `git revert` 回退，或重新合并解决 | 🟡 中 |
| **Abort Merge 后文件还是冲突状态** | Abort 应该能完全还原，如果不行可能是 Desktop 的 bug | 命令行执行 `git merge --abort` 强制还原 | 🟢 低 |
| **Pull 时提示 "overwrite local changes"** | 本地有未提交的修改 | 先 `git stash` 暂存本地修改 → pull → `git stash pop` 还原 | 🟡 中 |
| **文件名冲突 Desktop 显示 3 个冲突文件** | Desktop 对重命名冲突的展示方式 | 用命令行 `git status` 查看更清晰的提示；或直接 Discard 后手动调整文件系统 | 🟡 中 |
| **解决冲突后文件里残留了 >>>>>>> 标记** | 手动编辑时漏删了 | 全局搜索 `<<<<<<<` 或 `>>>>>>>`，确保所有标记都被删除 | 🔴 高 |

### 9.2 排查流程

```
遇到冲突，不知道怎么解决？
    │
    ├── Step 1: 确认冲突范围
    │   git status 或 Desktop 左侧面板
    │   到底有几个文件冲突了？
    │   ├── 1个 → 直接打开编辑
    │   └── 多个 → 逐个处理
    │
    ├── Step 2: 理解冲突内容
    │   打开冲突文件
    │   ├── <<<<<<< HEAD 和 ======= 之间 → 当前分支的内容
    │   ├── ======= 和 >>>>>>> branch 之间 → 合并分支的内容
    │   └── 看不懂？→ 用 VS Code 的颜色高亮辅助理解
    │
    ├── Step 3: 决定保留策略
    │   ├── 全要当前分支 → Accept Current Change
    │   ├── 全要合并分支 → Accept Incoming Change
    │   ├── 两个都要 → Accept Both Changes 或手动合并
    │   └── 都不对，写新的 → 手动编辑
    │
    ├── Step 4: 检查结果
    │   保存文件，审阅一遍最终内容
    │   ├── 代码能运行吗？
    │   ├── 逻辑正确吗？
    │   └── 有没有残留的冲突标记？
    │
    ├── Step 5: 标记并提交
    │   命令行：git add → git commit
    │   Desktop：看到绿色对勾 → Continue Merge
    │
    └── Step 6: 推送
        git push 或 Desktop Push origin
```

### 练习二：进阶——模拟多人协作的 Pull 冲突

**题目**：模拟"同事推了代码，你本地也改了同一行"的场景，用 `git stash` 机制解决冲突。

**要求**：
- 使用 GitHub 上的一个真实仓库（或创建一个测试远端仓库）
- 在 GitHub 网页端直接修改一个文件并提交（模拟同事的操作）
- 在本地也修改同一个文件的同一行（不提交）
- 用 `git pull` 或 Desktop 的 `Stash and Continue` 触发冲突
- 解决冲突并推送

<details>
<summary>操作步骤</summary>

```
场景设置：
1. GitHub 上有一个仓库，main 分支有 colors.txt
2. 在 GitHub 网页端把第4行改成 "orange"（模拟同事的提交）
3. 你本地把第4行改成 "purple"（未提交）
4. 你在 Desktop 中 Pull → 触发冲突

解决：
1. Desktop 提示 Stash and Continue → 点击
2. 还原 stash 时出现冲突
3. 用 VS Code 打开冲突文件
4. 决定最终内容（比如两个颜色都要）
5. 保存 → Commit → Push
```
</details>

### 练习三：综合实战——文件名冲突

**题目**：分别在两个分支上把同一个文件重命名为不同的名字，练习解决文件名冲突。

**要求**：
- 使用命令行完成（因为 Desktop 的文件名冲突提示不够清晰）
- 注意观察 `git status` 的输出信息
- 使用 `git rm` 命令删除冲突中不需要的文件
- 最终保留一个你想要的文件名

<details>
<summary>参考答案（点击展开）</summary>

```bash
# 1. 准备
mkdir rename-conflict-test
cd rename-conflict-test
git init
echo "some content" > original.txt
git add original.txt
git commit -m "add original.txt"

# 2. main 分支重命名为 main-version.txt
git checkout -b temp-branch  # 先建个临时分支
git mv original.txt main-version.txt
git commit -m "rename to main-version.txt"
git checkout main
git merge temp-branch        # 把重命名合并到 main
git branch -d temp-branch

# 3. feature 分支重命名为 feature-version.txt
git checkout -b feature
git mv original.txt feature-version.txt
git commit -m "rename to feature-version.txt"

# 4. 合并 → 触发文件名冲突
git checkout main
git merge feature

# 5. 查看冲突详情
git status
# 关键输出：
#   deleted by us:   original.txt
#   added by them:   feature-version.txt

# 6. 假设保留 main-version.txt
git rm original.txt           # 删除原始文件名（两边都删了）
git rm feature-version.txt    # 删除不想要的重命名文件
# main-version.txt 自动保留（它是 main 分支的文件）

# 7. 确认
git status
# 应该没有冲突标记了

# 8. 提交
git commit -m "resolve file rename conflict - keep main-version.txt"

echo "✅ 文件名冲突解决完毕！"
```
</details>
