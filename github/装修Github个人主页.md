## 一、课程背景与目标

### 1.1 前情回顾

在前面的课程中，你已经学会了：

- **注册 GitHub 账号** —— 拥有了自己在开源世界的"身份证"
- **GitHub 网站的基本操作** —— 了解首页、仓库、Issue 等核心页面
- **浏览和发现开源项目** —— 通过 Explore、Trending 找到感兴趣的项目
- **理解 Git 与 GitHub 的关系** —— Git 是版本控制工具，GitHub 是代码托管平台

现在你有了自己的 GitHub 账号，但打开个人主页一看——空空荡荡，毫无个性。这节课就来解决这个问题。

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 你的 GitHub 主页长什么样？ | 默认只有用户名，没有任何个性化内容 |
| 别人点进你的主页能看到什么？ | 一片空白——没有自我介绍、没有项目展示 |
| 为什么有些大佬的主页那么炫酷？ | 他们使用了 GitHub 的特殊仓库 + Markdown 语法装修 |
| 如何让你的主页也成为一张"名片"？ | 简装修（基础设置）+ 精装修（README 自定义） |

> ⚠️ **关键认知**：GitHub 个人主页不只是展示你写了多少代码——它是你在开源世界的**个人名片**。面试官、合作者、潜在雇主都可能点进你的主页。一个精心装修的主页能让人一秒记住你。

## 二、个人首页简装修

> 简装修 = 使用 GitHub 自带的功能，不用写代码，纯图形化操作，5 分钟搞定。

### 2.1 编辑个人资料（Edit Profile）

打开你的 GitHub 个人首页，地址是：

```
github.com/你的英文用户名
```

例如老师的用户名是 `techr`，地址就是 `github.com/techr`。

进入后在头像下方找到 **Edit profile** 按钮，点击进入编辑页面。以下是你可以修改的每一项：

| 字段 | 说明 | 示例 |
|---|---|---|
| **Name** | 显示名称（可填中文名） | 陈泽鹏 |
| **Bio** | 个人简介，一句话介绍自己 | 前端开发者 / 开源爱好者 |
| **Pronouns** | 性别代词（可选） | he/him |
| **Company** | 所在公司 | @alibaba |
| **Location** | 所在地点 | 杭州，中国 |
| **Timezone** | 个人时区 | Asia/Shanghai (GMT+8) |
| **Website / Blog** | 个人博客或网站链接 | https://myblog.com |
| **Social accounts** | 最多 4 个社交账号链接 | 微博、知乎、小红书等 |

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 打开 github.com/你的用户名                            │
│      ↓                                                       │
│  Step 2: 点击头像下方的 Edit profile 按钮                       │
│      ↓                                                       │
│  Step 3: 填写中文名、简介、性别、公司、地点、博客链接              │
│      ↓                                                       │
│  Step 4: 添加 4 个社交账号（微博、知乎、小红书等）                │
│      ↓                                                       │
│  Step 5: 点击页面底部的 Save 保存                               │
│      ↓                                                       │
│  🎉 你的个人资料已更新！                                        │
└─────────────────────────────────────────────────────────────┘
```

> ⚠️ **注意**：社交账号最多填 4 个。把最重要的放在前面，比如技术博客、LinkedIn、掘金主页等。这些链接会成为你主页上的快捷入口。

### 2.2 置顶你最得意的项目

在个人首页的中间区域，GitHub 允许你**置顶（Pin）最多 6 个项目**。被置顶的项目会以大卡片的形式展示，是别人最先看到的内容。

**操作方式有两种：**

**方式一：拖拽排序**

点击已置顶项目上的 6 个点图标（⋮⋮），直接拖拽调整前后顺序。

**方式二：自定义选择**

```
点击 "Customize your pins" 
    → 在弹出的窗口中勾选你想展示的项目
    → 点击 Save pins 保存
```

```
你的项目列表
    ├── 勾选 ✅ → 展示在主页
    └── 取消勾选 ❌ → 隐藏
```

> 💡 **类比**：就像 B 站 / 抖音上置顶自己最好的视频——把你的"代表作"放在最显眼的位置。建议选择：
> - 代码质量最高的项目
> - 有完整 README 的项目
> - 技术栈最能代表你方向的项目
> - Star 数较多的项目

### 2.3 读懂你的贡献活跃度图表

GitHub 会在你的主页自动展示**贡献热力图（Contribution Graph）**，这是一个由绿色方块组成的日历图。

```
贡献热力图解读：

🟩🟩🟩🟩  深绿色 → 这一天提交次数很多（高活跃度）
🟩🟩     浅绿色 → 这一天有少量提交（低活跃度）
⬜      灰色/白色 → 这一天没有提交记录
```

| 颜色 | 含义 | 示例 |
|---|---|---|
| 🟩 深绿色 | 提交次数多 | 6月9日有10次贡献（10次代码提交） |
| 🟩 浅绿色 | 提交次数少 | 当日只有1-2次提交 |
| ⬜ 灰色 | 完全没有活跃 | 这一天没有在 GitHub 上操作 |

> ⚠️ **重要认知**：热力图会成为别人评估你"是否活跃"的第一印象。持续活跃的热力图（即使每天只有少许提交）比"偶尔爆发一次然后消失"要好得多。面试官经常会看这个。

热力图下方是**历史活动记录**，按时间倒序显示：

```
你对某个仓库进行了某次提交 → 显示提交信息 → 显示大约时间
```

## 三、Markdown 语法精讲

> 精装修需要编写 README.md 文件，而 README.md 使用的是 Markdown 语法。这一节系统讲解 Markdown，**建议边看边在 GitHub 上实操**。

### 3.1 标题（Headers）

使用 `#` 号标记标题层级，`#` 后面**必须有一个空格**。

```markdown
# 一级标题
## 二级标题
### 三级标题
#### 四级标题
##### 五级标题
###### 六级标题
```

**渲染效果预览：**

GitHub 会根据标题层级**自动区分字体大小和粗细**，一级标题最大最粗，层级越深字体越小。

| 语法 | 层级 | 典型用途 |
|---|---|---|
| `# 文字` | 一级标题 | 文档总标题 |
| `## 文字` | 二级标题 | 大章节标题 |
| `### 文字` | 三级标题 | 小节标题 |
| `#### 文字` | 四级标题 | 子小节 |

### 3.2 文本格式化

Markdown 使用 `*` 星号和 `_` 下划线来标记文本样式：

```markdown
*斜体文本*          → 用一个星号包裹
_斜体文本_          → 用一个下划线包裹（效果相同）

**粗体文本**        → 用两个星号包裹
__粗体文本__        → 用两个下划线包裹（效果相同）

***粗斜体文本***    → 用三个星号包裹
___粗斜体文本___    → 用三个下划线包裹（效果相同）
```

**效果对照表：**

| 语法 | 渲染效果 |
|---|---|
| `*斜体*` | *斜体* |
| `**粗体**` | **粗体** |
| `***粗斜体***` | ***粗斜体*** |
| `_斜体_` | *斜体* |
| `__粗体__` | **粗体** |

> 💡 **建议**：`*` 和 `_` 效果相同，但在实际使用中，大多数开发者更习惯用 `*` 星号。保持一种风格即可，不要混用。

### 3.3 换行与分割线

**换行有两种方式：**

```markdown
# 方式一：插入一个空行（段落间距较大）

这是第一段。

这是第二段。（上面有空行，形成了段落分隔）

# 方式二：使用强制换行符（无空行，紧挨着换行）
这是第一行。<br>这是第二行。
```

| 方式 | 写法 | 效果 |
|---|---|---|
| 空行换行 | 两段之间空一行 | 段落间距较大，有明显的段间距 |
| 强制换行 | `<br>` | 紧挨着换行，没有额外间距 |

**分割线有五种写法：**

```markdown
---     ← 三个短横线
***     ← 三个星号
___     ← 三个下划线
- - -   ← 空格分隔的三个短横线
* * *   ← 空格分隔的三个星号
```

五种写法渲染效果完全相同，都是一条横贯页面的分割线。**推荐使用 `---`**，最简洁通用。

### 3.4 列表

**有序列表：** 数字 + 英文句点 + 空格

```markdown
1. 第一步
2. 第二步
3. 第三步
```

**无序列表：** 使用 `*`、`+` 或 `-` + 空格

```markdown
* 项目一
* 项目二
* 项目三

- 也可以
- 用减号

+ 还可以
+ 用加号
```

| 符号 | 效果 |
|---|---|
| `*` 星号 | • 项目 |
| `-` 减号 | • 项目（渲染相同） |
| `+` 加号 | • 项目（渲染相同） |

**嵌套列表：** 在子列表项前面加两个空格（或一个 Tab）缩进：

```markdown
1. 大类别一
   * 子项目 A
   * 子项目 B
2. 大类别二
   * 子项目 C
      - 孙子项目
      - 孙子项目
```

**渲染效果：**
```
1. 大类别一
   •  子项目 A
   •  子项目 B
2. 大类别二
   •  子项目 C
      ◦  孙子项目
      ◦  孙子项目
```

> ⚠️ **注意**：缩进必须一致。混用空格和 Tab 可能导致缩进层级错误。建议只使用空格缩进。

### 3.5 引用

在每行开头打一个 `>` 大于号：

```markdown
> 这是一段引用文字。
```

**嵌套引用：** 使用多个 `>`：

```markdown
> 第一层引用
>> 第二层引用
>>> 第三层引用
```

**渲染效果：**

> 第一层引用
> > 第二层引用
> > > 第三层引用

> 💡 **典型用途**：在 README 中用引用块突出显示重要提示、安装说明、或引用外部文档。

### 3.6 代码块

**行内代码：** 用两个反引号 `` ` `` 包裹

```markdown
安装命令是 `npm install`，然后运行 `npm start`。
```

**渲染效果：** 安装命令是 `npm install`，然后运行 `npm start`。

**多行代码块：** 用三个反引号包裹

````markdown
```
const name = "GitHub"
console.log(`Hello, ${name}!`)
```
````

**指定语言（推荐！）：** 在开头的三个反引号后面写上语言名称

````markdown
```javascript
function greet(name) {
    console.log(`Hello, ${name}!`)
}
greet("GitHub")
```
````

**效果：** 指定语言后，GitHub 会自动进行**语法高亮**（Syntax Highlighting）——不同语法元素会显示不同颜色：

```javascript
function greet(name) {       // ← 关键字 function 会变色
    console.log(`Hello, ${name}!`)  // ← 字符串会变色
}
```

> ⚠️ **一定要指定语言！** 不写语言名称代码也能显示，但没有语法高亮，可读性大打折扣。

**常用语言标识：**

| 语言 | 标识符 |
|---|---|
| JavaScript | `javascript` 或 `js` |
| Python | `python` 或 `py` |
| HTML | `html` |
| CSS | `css` |
| TypeScript | `typescript` 或 `ts` |
| JSON | `json` |
| Bash / Shell | `bash` 或 `sh` |
| Markdown | `markdown` 或 `md` |
| SQL | `sql` |

### 3.7 超链接

```markdown
[显示文字](实际网址)
```

**示例：**

```markdown
[百度](https://www.baidu.com)
```

**渲染效果：** [百度](https://www.baidu.com) —— 只显示"百度"两个字，点击后跳转到百度首页。

> 💡 **原理**：方括号 `[]` 里的内容是用户看到的**显示文字**，圆括号 `()` 里的是**实际链接地址**。

### 3.8 图片

图片语法和超链接很像，区别是在最前面加一个**英文叹号** `!`：

```markdown
![图片描述](图片地址)
```

```
超链接：[文字](网址)
图  片：![描述](网址)
             ↑ 这里多了个叹号
```

**图片地址有两种写法：**

| 方式 | 写法 | 适用场景 |
|---|---|---|
| **网络地址** | `![图](https://example.com/img.png)` | 图片托管在图床（如 SM.MS、Imgur） |
| **相对路径** | `![图](./avatar.png)` | 图片上传到了当前仓库的根目录 |

**上传图片到仓库的方法：**

```
Step 1: 进入你的同名仓库首页
    ↓
Step 2: 点击 Add file → Upload files
    ↓
Step 3: 选择要上传的图片文件
    ↓
Step 4: 点击 Commit changes 提交
    ↓
Step 5: 在 README 中用相对路径引用：
        ![我的头像](./avatar.png)
    ↓
🎉 图片出现在你的主页上了！
```

> 💡 **建议**：小图片直接上传到仓库，简单方便。大图片或多图片建议使用免费图床（如 SM.MS、GitHub Issues 做图床等），避免仓库体积过大。

### 3.9 表格

Markdown 表格由三部分组成：**表头** + **分隔线** + **内容行**。

```markdown
| 列一 | 列二 | 列三 |
|------|------|------|
| A    | B    | C    |
| 数据1 | 数据2 | 数据3 |
```

**各部分说明：**

| 部分 | 写法 | 说明 |
|---|---|---|
| 表头 | `\| 列一 \| 列二 \|` | 第一行，竖线分隔各列 |
| 分隔线 | `\|------\|------\|` | 第二行，至少三个短横线，**必须有** |
| 内容 | `\| 数据 \| 数据 \|` | 从第三行开始，填写实际内容 |

**实操示例：**

```markdown
| 姓名 | 技能 | 年限 |
|------|------|------|
| 张三 | JavaScript | 3年 |
| 李四 | Python | 5年 |
```

**渲染效果：**

| 姓名 | 技能 | 年限 |
|------|------|------|
| 张三 | JavaScript | 3年 |
| 李四 | Python | 5年 |

> 💡 **列数不限**，但每一行的竖线数量必须一致。分隔线的横线数量不要求与表头宽度完全对齐——GitHub 会自动调整列宽。

│ 引用          │ > 文字                                       │
│ 行内代码      │ `代码`                                       │
│ 代码块        │ ```语言 代码 ```                              │
│ 超链接        │ [文字](网址)                                  │
│ 图片          │ ![描述](地址)                                 │
│ 表格          │ \| 列 \| 列 \| + \|---\|---\| + 内容         │
│ 强制换行      │ <br>                                         │
└──────────────┴──────────────────────────────────────────────┘
```

## 四、个人首页精装修——特殊仓库 README

> 精装修 = 利用 GitHub 特殊仓库机制 + Markdown 语法 + 第三方小组件，打造独一无二的个人主页。

### 4.1 创建同名仓库

这是精装修的**关键第一步**。GitHub 有一个特殊机制：当你创建一个**与自己用户名完全相同的仓库**时，该仓库的 README.md 文件内容会**直接展示在你的个人首页上**。

**操作步骤：**

```
Step 1: 回到你的 GitHub 首页
    ↓
Step 2: 点击 Repositories 标签旁边的 New 按钮（绿色）
    ↓
Step 3: 在 Repository name 中填你的英文用户名（必须一模一样！）
    ↓
Step 4: GitHub 会弹出一个提示：
        "You have discovered a special repository!"
        这表示你创建的是特殊仓库 ✓
    ↓
Step 5: 往下拉，找到 "Add a README file" → 勾选上 ✅
    ↓
Step 6: 拉到最底下，点击 "Create repository" 绿色按钮
    ↓
🎉 特殊仓库创建成功！
```

> ⚠️ **关键**：仓库名称必须和你的用户名**一模一样**（包括大小写）。如果用户名是 `TechR`，仓库名也必须是 `TechR`，不能是 `techr`。

**GitHub 的提示信息解读：**

```
🎉 You have discovered a special repository!

This repository's README.md will appear on your public profile.
```

翻译：这个仓库的 README.md 内容会直接展示在你的**公开个人主页**上。

### 4.2 编写你的第一个 README

仓库创建好后，点击 README.md 文件右上角的**小铅笔图标**（Edit this file），进入编辑模式。

现在把你学到的 Markdown 语法用起来，写一份自我介绍：

```markdown
# 👋 Hi, I'm 陈泽鹏

> 前端开发者 | 开源爱好者 | 终身学习者

---

## 🧑 关于我

- 🔭 目前正在学习：**React + TypeScript 全栈开发**
- 🌱 最近在研究：大语言模型应用开发（LangChain）
- 💬 可以问我：JavaScript、Vue.js、前端工程化
- 📫 联系我：techr@example.com

---

## 🛠 技术栈

| 前端 | 后端 | 工具 |
|------|------|------|
| JavaScript | Node.js | Git |
| TypeScript | Python | VS Code |
| Vue.js | Express | Docker |
| React | | GitHub Actions |

---

## 📊 GitHub 统计

<!-- 这里放小组件，下一步会讲 -->
```

点击 **Preview** 可以预览效果，满意后点击 **Commit changes** 提交。

> 💡 **提交后**，回到你的个人首页 `github.com/你的用户名`，就能看到 README 内容展示在头像和项目列表**上方**。

### 4.3 上传图片到仓库

在 README 中展示图片，需要先把图片上传到仓库。

**实操步骤：**

```
Step 1: 进入同名仓库首页
    ↓
Step 2: 点击 "Add file" 按钮 → 选择 "Upload files"
    ↓
Step 3: 拖拽或选择本地图片文件（如 avatar.png）
    ↓
Step 4: 下拉点击 "Commit changes"
    ↓
Step 5: 回到 README.md，编辑时使用相对路径引用：
        ![我的头像](./avatar.png)
    ↓
🎉 图片展示成功！
```

> 💡 **两种图片引用方式对比：**

| 方式 | 示例 | 优点 | 缺点 |
|---|---|---|---|
| 上传到仓库 | `![图](./docker.png)` | 稳定、跟着项目走 | 占用仓库空间 |
| 外链图床 | `![图](https://imgur.com/xxx.png)` | 不占仓库空间 | 依赖第三方服务稳定性 |

### 4.4 添加第三方小组件

除了标准 Markdown，GitHub 社区开发了很多**动态小组件**来增强主页效果。你只需在 README 中插入一小段代码即可。

**组件一：仓库状态统计（GitHub Stats Card）**

```markdown
![GitHub Stats](https://github-readme-stats.vercel.app/api?username=你的英文用户名)
```

把 `你的英文用户名` 替换成自己的，就能显示：
- 总 Star 数
- 总提交数
- PR（Pull Request）数量
- 贡献量统计

**组件二：主页访问量统计（Visitor Count）**

```markdown
![Visitor Count](https://visitor-badge.laobi.icu/badge?page_id=你的英文用户名.你的英文用户名)
```

**效果：** 每次有人访问你的主页，计数自动 +1。

> 💡 **原理**：这些组件本质上是**动态生成图片的服务**。GitHub README 加载时，会请求第三方服务的 API，服务根据你的数据生成一张统计图返回并展示。所以你不需要维护任何后端代码。

**添加小组件后提交：**

```
编辑 README.md → 粘贴组件代码 → 替换用户名 → Commit changes
    ↓
回到个人首页 → 刷新页面
    ↓
🎉 统计数据和访问量计数出现了！
```

> ⚠️ **注意**：第三方组件依赖外部服务。如果某天服务挂了，图片会显示为"裂图"。选择成熟稳定、社区广泛使用的组件（如 `github-readme-stats`）。

### 4.5 向大佬学习——阅读别人的主页代码

GitHub 上每位大佬的炫酷主页，背后都是一个**同名仓库的 README.md 文件**。你可以直接查看和学习。

**如何找到大佬的装修代码：**

```
Step 1: 逛 GitHub 时，发现某大佬的主页很炫酷
    ↓
Step 2: 在头像右侧区域找到他的同名仓库（通常有星标/高亮）
    ↓
Step 3: 点击进入仓库
    ↓
Step 4: 打开 README.md 文件
    ↓
Step 5: 点击右上角的 "Raw" 按钮或直接看代码视图
    ↓
Step 6: 学习他的 Markdown 写法、组件选择、排版技巧
    ↓
🎉 把好用的技巧用到自己的主页上！
```

```
别人的主页 → 找到同名仓库 → 查看 README.md 源码 → 学习 + 借鉴
```

> 💡 **推荐学习方向**：
> - 排版布局：别人如何组织章节顺序
> - 小组件：别人用了哪些你没见过的组件
> - 配色和图标：Emoji 的使用技巧
> - 技术栈展示：如何用 Badge（徽章）展示技能

## 五、简装修 vs 精装修——对比总结

| 维度 | 简装修 | 精装修 |
|---|---|---|
| **操作方式** | 纯图形化操作，点击填写 | 编写 Markdown 代码 |
| **所需技能** | 无需编程 | 需要 Markdown 基础 |
| **耗时** | 约 5 分钟 | 30 分钟 ~ 数小时（取决于复杂程度） |
| **自定义程度** | 有限（只能填固定字段） | 无限（任意排版、图片、组件） |
| **效果** | 整洁、规范的简历感 | 炫酷、个性化、科技范 |
| **修改入口** | Edit Profile 按钮 | 同名仓库 → 编辑 README.md |
| **适用场景** | 快速拥有基本信息展示 | 打造深入人心的个人品牌 |

```
       简装修                          精装修
   ┌──────────┐                ┌──────────────────┐
   │ 基础名片   │  ────────→    │   完整个人主页      │
   │ 姓名、简介  │   升级！      │   README 自定义     │
   │ 置顶项目   │              │   Markdown 排版     │
   │ 热力图    │              │   小组件、图标        │
   └──────────┘                └──────────────────┘
     必需项                         加分项
```

## 六、常见问题与排错指南

### 6.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| 创建同名仓库后主页没变化 | 没有编辑 README.md 文件 | 点击 README.md → 铅笔图标编辑 → 写内容 → 提交 | 🔴 高 |
| README 内容不显示在主页 | 仓库名和用户名不完全一致 | 检查大小写：`TechR` ≠ `techr`，必须一模一样 | 🔴 高 |
| 图片显示为裂图 | 图片路径错误或图片未上传 | 检查路径是否正确（相对路径以仓库根目录为基准） | 🟡 中 |
| 表格渲染不出来 | 缺少分隔线行 | 表头下面必须有一行 `\|---\|---\|` 样式的分隔线 | 🟡 中 |
| Markdown 格式不生效 | 语法符号后面缺少空格 | `#标题` ❌ → `# 标题` ✅（井号后面要空格） | 🟡 中 |
| 第三方组件不显示 | 小组件服务不可用或用户名错误 | 检查 URL 中的用户名是否正确；换一个稳定的组件服务 | 🟢 低 |
| 社交链接保存后不显示 | 格式不正确 | 需要填写完整 URL（含 https://） | 🟢 低 |
| 主页访问量计数不动 | 浏览器缓存 | 硬刷新页面（Ctrl+Shift+R 或 Cmd+Shift+R） | 🟢 低 |

### 6.2 排查流程

```
主页装修出问题了？
    │
    ├── 简装修问题？
    │       ├── 个人资料没更新？ → 检查是否点了 Save 保存
    │       ├── 置顶项目不对？ → Customize your pins 重新勾选
    │       └── 社交链接不显示？ → 检查是否填了完整 URL
    │
    └── 精装修问题？
            ├── README 不显示在主页？
            │       ├── 仓库名和用户名一样吗？（大小写！）
            │       └── README.md 文件存在吗？
            │
            ├── Markdown 语法不生效？
            │       ├── 空格写了吗？（# 和文字之间）
            │       ├── 分隔线写了吗？（表格第二行）
            │       └── 有空行吗？（段落之间）
            │
            └── 小组件/图片不显示？
                    ├── 检查图片路径（是相对路径还是网络地址？）
                    ├── 检查用户名是否替换正确
                    └── 刷新页面（可能是缓存）
```
