## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经学会了：

- **创建仓库** —— 选择公开/私有、开源协议、编写 README
- **分支与 PR** —— 分支隔离开发、Pull Request 合并请求
- **Issue** —— 提交 Bug 报告和功能请求
- **开源协作全流程** —— Issue → Branch → PR → Code Review → Merge

到目前为止，你主要在用仓库的**代码托管功能**。但一个 GitHub 仓库远不止放代码这么简单——它还有一套完整的**附属功能**来支撑团队协作。

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 项目文档写在哪里？ | README 放不下长文档怎么办？|
| 怎么知道项目的活跃程度？ | 有多少人访问？谁在贡献代码？|
| 依赖的第三方库怎么保持更新？ | 手动检查太累了，有没有自动化方案？|
| 团队任务怎么追踪？ | Bug 修到哪了？谁在做什么？优先级怎么排？|
| 社区讨论放哪里？ | Issue 和论坛有什么区别？|

> ⚠️ **关键认知**：一个成熟的 GitHub 项目，代码只是"冰山一角"。Wiki、Insights、Projects、Discussions 四大功能构成了项目管理的完整体系。掌握它们，你才能真正管理好一个项目。

## 二、仓库不止是放代码——四大附属功能总览

一个 GitHub 仓库，除了代码托管，还有四大附属功能：

```
                    GitHub Repository
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   📦 代码托管         📊 Insights         💬 Social
   (你已学过的)        (数据分析)          (社交协作)
        │                  │                  │
   ├── 源代码          ├── Pulse         ├── Issues
   ├── Branch         ├── Contributors   ├── Pull Requests
   ├── Commit         ├── Traffic        ├── Discussions
   ├── PR             ├── Dependabot     └── Wiki
   └── 文件管理        └── Dependency Graph

        本节课重点讲解的四大附属功能：
```

| 功能 | 一句话说明 | 类比 |
|---|---|---|
| **📚 Wiki** | 项目的维基百科，存放详细文档 | 产品说明书 |
| **📊 Insights** | 项目数据分析中心，了解项目状态 | 数据仪表盘 |
| **📋 Projects** | 可视化管理看板，追踪任务进度 | Trello / Jira |
| **💬 Discussions** | 项目社区论坛，开放式讨论 | 贴吧 / 论坛 |

## 三、GitHub Wiki——项目的维基百科

### 3.1 Wiki 是什么

**GitHub Wiki** 是每个仓库自带的维基百科系统，用于存放**项目的详细文档**。

```
Wiki 适合存放：
    ├── 详细的使用指南
    ├── API 设计文档
    ├── 架构设计说明
    ├── 常见问题解答（FAQ）
    ├── 开发者上手教程
    └── 其他大型、结构化的文档内容

Wiki ≠ README：
    ├── README = 项目门面，简洁介绍（一页看完）
    └── Wiki = 详细手册，多页结构（像一本书）
```

### 3.2 创建第一个 Wiki 页面

**操作步骤：**

```
进入仓库 → 点击顶部的 "Wiki" 标签栏
    ↓
点击 "Create the first page" 按钮
    ↓
进入编辑器 → 使用 Markdown 语法编写内容
    ↓
可以在页面底部填写可选的 Edit message（修改说明）
    ↓
点击 "Save Page"
    ↓
🎉 第一个 Wiki 页面创建成功！
```

**Wiki 的便利特性：**

| 特性 | 说明 |
|---|---|
| **内置导航栏** | 自动生成侧边栏，方便浏览各个页面 |
| **搜索功能** | 可以搜索所有 Wiki 文档内容 |
| **Markdown 支持** | 与你学过的 Markdown 语法完全通用 |
| **自动目录** | 标题会自动出现在导航栏中 |

> 💡 **导航栏自动生成**：你在页面中使用的二级标题（`##`），会自动出现在 Wiki 右侧导航栏里，点击可以快速跳转。这让 Wiki 天然适合写长篇文档。

### 3.3 添加更多页面

```
Wiki 首页 → 点击右上角 "New Page" 按钮
    ↓
填写页面标题（会成为导航栏中的链接名）
    ↓
用 Markdown 编写页面内容
    ↓
点击 "Save Page"
    ↓
🎉 导航栏中出现了新页面的链接！
```

### 3.4 Wiki 是一个独立的 Git 仓库

这是一个很重要的细节：**Wiki 本身就是一个独立的 Git 仓库**。

```
右下角可以看到一个克隆链接（Clone URL）

    Wiki 仓库 ≠ 代码仓库
    ├── 代码仓库：存放源代码
    └── Wiki 仓库：存放文档

你可以：
    ├── 独立克隆 Wiki 仓库到本地
    ├── 在本地编辑文档
    └── 推送回 GitHub（和操作代码仓库一样！）
```

> 💡 **这意味着**：你可以用 Git 的所有功能来管理文档——版本历史、分支、PR，全部适用。后面的课程会讲到如何克隆仓库。

### 3.5 Wiki 的适用场景

| 项目类型 | Wiki 典型内容 |
|---|---|
| 开源库 / 框架 | API 参考文档、使用指南、迁移指南 |
| 内部工具 | 部署手册、运维文档、故障处理流程 |
| 学习项目 | 课程笔记、学习路线、资料汇总 |
| 团队项目 | 编码规范、架构决策记录（ADR）、会议纪要 |

```
什么时候用 README？          什么时候用 Wiki？

单页能说清楚                  需要多页结构
项目概述、快速开始             详细文档、API参考
给人第一印象                  深入学习时查阅
```

## 四、GitHub Insights——项目数据分析中心

> GitHub Insights 是一组数据分析功能，帮助项目维护者和贡献者**分析和理解代码库的活动状态**。

### 4.1 Pulse——项目活动概览

进入仓库 → 点击 **Insights** 标签 → 默认进入 **Pulse** 页面。

Pulse 提供了一个**简要的项目活动概要**：

```
┌─────────────────────────────────────────────────────────────┐
│  Pulse —— 项目活动概览                                       │
│                                                             │
│  📊 概览数据（可选择时间范围：1天/3天/1周/1月）                   │
│                                                             │
│  Merged Pull Requests:  X     ← 已合并的 PR 数量             │
│  Open Pull Requests:    X     ← 当前打开状态的 PR            │
│  Closed Issues:         X     ← 已关闭的 Issue              │
│  New Issues:            X     ← 当前打开状态的 Issue         │
│                                                             │
│  📝 直接提交情况（PR 之外的提交，即直接往 main 提交）           │
│      共 Y 位作者进行了 Z 次对 main 的直接提交                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 指标 | 含义 | 怎么读 |
|---|---|---|
| **Merged PRs** | 已合并的 PR 数 | 越多 = 项目越活跃 |
| **Open PRs** | 等待处理的 PR 数 | 太多 = 可能 Review 人手不够 |
| **Closed Issues** | 已解决的问题 | 越多 = 项目维护得越好 |
| **New Issues** | 新提出的问题 | 太多 = 可能 Bug 比较多 |
| **直接提交** | 不经过 PR 直接推 main 的次数 | 太多 = 流程不够规范（正式项目应该少） |

> ⚠️ **注意**：右上角可以切换时间范围——**1天、3天、24小时、1周、1个月**。选不同时间范围，数据会变化。选择合适的时间范围才能得到有意义的结论。

### 4.2 Contributors——贡献者分析

**Contributors** 页面显示每位贡献者对代码的贡献情况：

```
图表解读：

    提交次数
        ↑
        │  ▄▄
        │  ██  ▄▄
        │  ██  ██▄▄
        │  ██  ████  ▄▄
        │  ██  ████  ██
        └──────────────────→ 时间
          贡献者A   贡献者B

    横轴 = 时间
    纵轴 = 提交次数
    每个人一条颜色
```

| 信息 | 说明 |
|---|---|
| 每个贡献者一列 | 能看到每个人在什么时间段贡献了多少代码 |
| 颜色深浅 | 提交越多颜色越深 |
| 时间趋势 | 可以看出谁是最活跃的贡献者 |

### 4.3 Community Standards——社区规范清单

**Community Standards**（社区标准）是一个**清单列表**，用来检查项目的社区规范是否完善。

```
┌─────────────────────────────────────────────────────────────┐
│  Community Standards 检查清单                                 │
│                                                             │
│  ✅ Description        项目描述                               │
│  ⚠️ Code of Conduct    社区行为准则（哪些行为许可/禁止）         │
│  ⚠️ Contributing       贡献代码应遵守的规范                     │
│  ✅ License            开源许可证                             │
│  ⚠️ Security Policy    安全标准                               │
│  ⚠️ Issue Template     Issue 模板                            │
│  ⚠️ PR Template        Pull Request 模板                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

每个项目的旁边都有一个 **Add** 按钮，点击可以以**文件形式**将规范添加到项目中：

| 文件 | 作用 | 添加到项目后的文件名 |
|---|---|---|
| **Code of Conduct** | 定义社区行为准则——什么能做、什么不能做 | `CODE_OF_CONDUCT.md` |
| **Contributing** | 规定贡献代码的规范和流程 | `CONTRIBUTING.md` |
| **Security Policy** | 如何报告安全漏洞 | `SECURITY.md` |
| **Issue Template** | 提交 Issue 时使用的模板 | `.github/ISSUE_TEMPLATE/` |
| **PR Template** | 提交 PR 时使用的模板 | `.github/PULL_REQUEST_TEMPLATE.md` |

> 💡 **为什么要填满这些？** 一个"完善"的项目会给来访者信心——表明这个项目是认真维护的、有规范的。特别是对于开源项目，Community Standards 全绿的项目更容易吸引贡献者。

### 4.4 Traffic——项目受欢迎程度

**Traffic** 页面展示项目的访问数据：

| 指标 | 含义 |
|---|---|
| **Git Clones** | 项目被克隆了多少次 |
| **Visitors** | 有多少独立访客访问了项目页面 |
| **Referring sites** | 访客从哪些网站跳转过来 |
| **Popular content** | 哪些页面最受欢迎 |

> 💡 **Traffic 数据只有你自己能看到**（仓库拥有者），其他人看不到你的仓库访问数据。

### 4.5 Commits & Code Frequency——代码提交与变动

**Commits** 页面：

```
提交次数
    ↑
    │     ▄▄
    │  ▄▄ ██ ▄▄
    │  ██ ██ ██
    └──────────→ 时间（按周）

横轴 = 时间（按周）
纵轴 = 提交次数
```

**Code Frequency** 页面：

```
代码变动行数
    ↑
    │ 🟩🟩🟩  🟩🟩
    │ 🟩🟩🟩  🟩🟩🟥🟥
    │ 🟩🟩🟥  🟩🟥🟥
    └────────────────→ 时间

    🟩 绿色 = 新增代码行数
    🟥 红色 = 删除代码行数
```

| 信息 | 怎么读 |
|---|---|
| 绿色多 | 项目在快速增长 |
| 红色多 | 可能在重构，删除旧代码 |
| 绿红相当 | 正常迭代，增删平衡 |

### 4.6 Dependency Graph——依赖关系图

**Dependency Graph** 用于分析和管理项目的依赖关系，包含三个选项卡：

**① Dependencies（本项目依赖了谁）：**

```
显示本项目依赖的第三方库：

    我的项目
        └── 依赖: 汉字转拼音库 v0.5.0
            （在 requirements.txt 中声明）

从这里可以看到：
    ├── 依赖了哪些库
    ├── 依赖的版本
    └── 在哪个文件中声明的（如 requirements.txt、package.json）
```

**② Dependents（谁依赖了本项目）：**

```
显示本项目被其他人依赖的情况：

    拼音库
        └── 被 6,191 个其他项目依赖！

这个数字越大 → 项目越重要、影响力越大
```

> 💡 **老师演示的例子**：一个叫"拼音"的 Python 库，被 **6,191 个**其他项目依赖。这就是开源世界的"网络效应"——你的库被越多人用，它的影响力就越大。

### 4.7 Insights 功能速查表

```
┌─────────────────────────────────────────────────────────────┐
│              Insights 功能速查表                              │
├──────────────┬──────────────────────────────────────────────┤
│ 功能          │ 用途                                         │
├──────────────┼──────────────────────────────────────────────┤
│ Pulse        │ 简要活动概览（PR/Issue/提交 数量）              │
│ Contributors │ 每位贡献者的代码贡献量和时间分布                 │
│ Community    │ 讨论、Issue、PR 的社区活跃度统计                │
│ Standards    │ 项目规范完善度清单（行为准则、贡献指南等）        │
│ Traffic      │ 项目访问量（克隆次数、访客数）                   │
│ Commits      │ 提交次数随时间变化                             │
│ Code Freq.   │ 代码增删行数随时间变化（绿色=增，红色=删）       │
```

## 五、Dependabot——自动化依赖更新

### 5.1 Dependabot 是什么

**Dependabot** 是 GitHub 的**自动化依赖更新机器人**。

```
传统方式：
    每隔一段时间手动检查 → 发现新版本 → 手动修改配置文件 → 手动测试 → 提交

Dependabot 方式：
    自动定时扫描 → 发现新版本 → 自动创建 PR → 你只需点 Merge ✅
```

| 功能 | 说明 |
|---|---|
| **定时扫描** | 按天/周/月的频率检查依赖更新 |
| **自动创建 PR** | 发现新版本后自动创建更新 PR |
| **版本更新** | 依赖库的小版本（minor）和补丁（patch）更新 |
| **安全更新** | 发现安全漏洞后自动推送修复版本 |

### 5.2 启用 Dependabot

**方式一：通过网页引导创建（最简单）**

```
仓库 → Insights → Dependency Graph → Dependabot 选项卡
    ↓
点击 "Create config file"
    ↓
GitHub 自动创建了一个配置文件（dependabot.yml）
    ↓
修改配置 → Commit
    ↓
Dependabot 开始扫描！
```

**方式二：手动创建配置文件**

在仓库根目录创建 `.github/dependabot.yml` 文件。

### 5.3 自动创建的 PR

配置完成后，Dependabot 会自动扫描。如果发现有更新，会**自动创建 PR**：

```
Dependabot 扫描中……
    ↓
发现：拼音库有新版本 0.5.0 → 0.5.1
    ↓
自动创建 PR：
    ├── 标题："Bump 拼音库 from 0.5.0 to 0.5.1"
    ├── 自动修改了 requirements.txt 中的版本号
    └── 附带详细的更新说明
    ↓
你只需：
    ├── 检查改动是否 OK
    ├── 点击 Merge 合并
    └── 🎉 依赖更新完成！
```

> 💡 **Dependabot 的 PR 和人类创建的 PR 完全一样**——有标题、描述、代码改动、可以 Review、可以合并。唯一的区别是作者是一个机器人。

### 5.4 配置文件详解

老师演示了 Dependabot 配置的核心参数。以下是一个基础配置示例：

```yaml
# .github/dependabot.yml

version: 2
updates:
  - package-ecosystem: "pip"       # 包管理工具（必填）
    directory: "/"                  # 依赖文件所在目录（必填）
    schedule:
      interval: "weekly"            # 扫描间隔（必填）
```

**三个必填参数：**

| 参数 | 说明 | 示例 |
|---|---|---|
| **package-ecosystem** | 包管理工具类型 | `pip`、`npm`、`maven`、`gradle` |
| **directory** | 依赖声明文件的目录 | `/`（根目录）、`/backend` |
| **schedule.interval** | 扫描间隔 | `daily`、`weekly`、`monthly` |

```
schedule.interval（扫描间隔）：

    daily     → 每天检查一次
    weekly    → 每周检查一次（推荐，不会太频繁）
    monthly   → 每月检查一次
```

### 5.5 Groups——分组管理更新规则

可以使用 **Groups** 将不同类型的更新分组，每组定义不同的规则：

```yaml
# dependabot.yml 中的 groups 配置示例

updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    groups:                          # 🆕 分组管理
      angular:
        patterns:
          - "@angular/*"             # 第一组：Angular 相关包
        update-types:
          - "minor"                  # 优先更新小版本
          - "patch"                  # 然后更新补丁
      minor-and-patch:
        patterns:
          - "*"                      # 第二组：其他所有包
        update-types:
          - "patch"                  # 安全更新优先
          - "minor"
```

**分组的好处：**

```
不分组的 Dependabot：
    每个依赖更新都单独创建一个 PR
    → 10 个依赖有更新 → 10 个 PR → 太分散

分组后的 Dependabot：
    同组的更新合并到一个 PR
    → 版本更新一个 PR，安全更新一个 PR → 清晰！
```

**Groups 的关键字段：**

| 字段 | 说明 |
|---|---|
| **patterns** | 匹配哪些包（支持通配符 `*`） |
| **update-types** | 更新类型优先级：`major`（大版本）、`minor`（小版本）、`patch`（补丁） |

**update-types 三种类型：**

| 类型 | 版本变化 | 含义 | 风险 |
|---|---|---|---|
| **patch** | `0.5.0` → `0.5.1` | 安全补丁、Bug 修复 | 🟢 低 |
| **minor** | `0.5.0` → `0.6.0` | 新增功能，向后兼容 | 🟡 中 |
| **major** | `0.5.0` → `1.0.0` | 重大变更，可能不兼容 | 🔴 高 |

### 5.6 Ignore——忽略特定更新

如果不想让 Dependabot 更新某些依赖或某些版本，可以用 **Ignore** 配置：

```yaml
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    ignore:                          # 🆕 忽略配置
      - dependency-name: "angular"   # 忽略 angular 的
        versions: [">=15.0.0"]       # 15.0 以上版本
        update-types: ["major"]      # 或者忽略所有大版本更新
```

**Ignore 的使用场景：**

| 场景 | 配置 |
|---|---|
| "Angular 大版本升级风险太高，暂时不升" | `update-types: ["major"]` for angular |
| "这个库的 2.0 以上版本 API 全变了" | `versions: [">=2.0.0"]` |
| "某个库被弃用了，不更新了" | `dependency-name: "old-lib"` 完全忽略 |

### 5.7 支持的包管理器

Dependabot 支持几乎所有主流包管理工具：

```

├────────────┬────────────────────────────────────────────────┤

├────────────┼────────────────────────────────────────────────┤
```

## 七、GitHub Discussions——项目社区论坛

### 7.1 Discussions 是什么

**GitHub Discussions** 是仓库自带的**社区论坛**功能。

```
Discussions = 围绕项目的一个论坛

在这里可以：
    ├── 发帖讨论技术问题
    ├── 分享使用心得
    ├── 提出项目发展的想法
    ├── 问答互助
    └── 发布公告
```

### 7.2 Discussions vs Issues

| 维度 | Issues | Discussions |
|---|---|---|
| **定位** | 具体任务追踪 | 开放式讨论 |
| **内容** | Bug报告、功能请求、具体任务 | 想法、问题、分享、公告 |
| **有明确答案吗** | 有（Bug修好 / 功能上线） | 不一定（开放性讨论） |
| **谁在用** | 所有项目 | 通常**较大项目**才启用 |
| **类比** | 工单系统 | 论坛 / 贴吧 |

> 💡 **一般小型项目用 Issue 就够了**，Discussion 通常是较大项目（几千 Star 以上、社区活跃）才会启用的功能。

### 7.3 基本操作

**发帖：**

```
Discussions 选项卡 → 点击 "New discussion"
    ↓
选择分类（Category）
    ├── 💡 Ideas（想法建议）
    ├── 🙏 Q&A（问答求助）
    ├── 📣 Announcements（公告）
    └── 💬 General（一般讨论）
    ↓
填写标题和内容 → 点击 "Start discussion"
    ↓
🎉 帖子发布成功！
```

**浏览和参与讨论：**

```
Discussions 页面提供：
    ├── 按分类筛选
    ├── 按标签筛选
    ├── 搜索功能
    ├── 按最新/最热排序
    └── 在帖子下回复评论
```

## 九、常见问题与排错指南

### 9.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| Wiki 页面保存后导航栏没更新 | 页面标题没正确使用 Markdown 标题语法 | 确保使用了 `#` 或 `##` 作为标题 | 🟡 中 |
| Dependabot 没有自动创建 PR | 配置文件路径或格式错误 | 检查 `.github/dependabot.yml` 是否存在；YAML 缩进是否正确 | 🔴 高 |
| Dependabot 创建的 PR 太多 | 没有设置 Groups 分组 | 在配置文件中添加 Groups，合并同类更新 | 🟢 低 |
| Projects 卡片拖拽不了 | 权限不足 | 确认你有项目的 Write 权限 | 🟡 中 |
| Projects 卡片显示 Draft | 还没有关联 Issue | 点击三个点 → Convert to issue | 🟢 低 |
| Discussions 选项卡找不到 | 项目没有启用 Discussions | Settings → Features → Discussions ✅ | 🟡 中 |
| Traffic 数据看不到 | 只有仓库拥有者可以看 | 确认你是仓库 Owner | 🟢 低 |
| Wiki 克隆链接在哪 | 没注意到右下角 | Wiki 页面右下角有 Clone URL | 🟢 低 |

### 9.2 排查流程

```
仓库附属功能出问题了？
    │
    ├── Wiki？
    │       ├── 找不到？→ 看仓库顶部标签栏有没有 Wiki
    │       ├── 无法编辑？→ 检查权限
    │       └── 导航栏不更新？→ 检查是否用了正确的 Markdown 标题
    │
    ├── Dependabot？
    │       ├── 不工作？→ 检查 .github/dependabot.yml 文件是否存在且格式正确
    │       ├── PR 太多？→ 添加 Groups 分组
    │       └── 想忽略某更新？→ 添加 ignore 配置
    │
    ├── Projects？
    │       ├── 找不到？→ 检查仓库是否开启了 Projects 功能
    │       ├── 卡片不能拖？→ 检查权限
    │       └── Team Capacity 没数据？→ 确保已分配 Assignee 和 Size
    │
    └── Discussions？
            ├── 选项卡不存在？→ Settings → 启用 Discussions
            └── 没人参与？→ 小型项目用 Issue 就够了
```

## 十、课后练习

### 练习一：创建 Wiki 页面

**目标：** 为你的 hello-world 仓库创建项目 Wiki。

**要求：**
- [ ] 进入 hello-world 仓库的 Wiki 页面
- [ ] 创建首页（Home），用 Markdown 写一段项目介绍
- [ ] 创建一个新页面"使用指南"，包含至少 3 个二级标题
- [ ] 验证导航栏中出现了新页面的链接和二级标题
- [ ] 在 Wiki 页面右下角找到克隆链接并记录下来

### 练习二：配置 Community Standards

**目标：** 完善你的项目社区规范。

**要求：**
- [ ] 进入 Insights → Community Standards
- [ ] 为项目添加 Code of Conduct（可以选择 GitHub 提供的模板）
- [ ] 为项目添加 Contributing Guide
- [ ] 为项目添加 Issue Template（Bug Report 和 Feature Request 两种）
- [ ] 为项目添加 PR Template
- [ ] 检查清单中还有哪些是未完成的，记录下来

### 练习三：启用 Dependabot

**目标：** 为你的项目配置自动化依赖更新。

**要求：**
- [ ] 在 hello-world 仓库中创建一个简单的依赖文件（如 `requirements.txt` 或 `package.json`）
- [ ] 在依赖文件中声明一个具体的库版本（比如 `requests==2.28.0`）
- [ ] 创建 `.github/dependabot.yml` 配置文件
- [ ] 设置扫描间隔为 `weekly`
- [ ] 添加 Groups 配置（将安全更新和版本更新分组）
- [ ] 观察 Dependabot 是否开始扫描（可能需要等待）

### 练习四：创建项目看板

**目标：** 用 GitHub Projects 管理一个模拟项目。

**要求：**
- [ ] 创建一个新的 Project 看板（选择 Board 模板）
- [ ] 添加至少 5 个任务卡片
- [ ] 自定义状态列：Todo → In Progress → Code Review → Testing → Done
- [ ] 将 3 个卡片转为 Issue
- [ ] 给每个 Issue 分配 Assignee、设置 Priority、Size
- [ ] 在 Team Capacity 视图中查看每个成员的工作负载
- [ ] 将一张卡片从 Todo 拖到 In Progress，再拖到 Done

### 练习五：综合思考

**目标：** 理解各功能之间的关系。

**要求：**

回答以下问题：

1. 一个 Issue 可能出现在哪些页面？（提示：Issue 列表、Projects 看板、Insights……）
2. Dependabot 创建的 PR 和人类创建的 PR 有什么相同和不同？
3. Wiki 和 README 分别适合放什么内容？你会在哪里写"快速开始指南"？
4. 小团队（3 人）做一个小项目，哪些附属功能是必须的？哪些可以不用？

## 十一、课程小结

### 11.1 核心知识图谱

```
┌────────────────────────────────────────────────────────────┐
│         GitHub 仓库附属功能 核心知识体系                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1️⃣  Wiki——项目的维基百科                                    │
│      ├── 存放详细文档、指南、API 设计文档                      │
│      ├── 支持 Markdown，自动生成导航栏                       │
│      ├── 独立 Git 仓库，可独立克隆和推送                      │
│      └── 适用：多页结构化文档                                  │
│                                                            │
│  2️⃣  Insights——项目数据分析                                   │
│      ├── Pulse：PR/Issue/提交 活动概览                      │
│      ├── Contributors：贡献者分析                            │
│      ├── Community Standards：项目规范清单                  │
│      ├── Traffic：访问量和克隆统计                          │
│      ├── Code Frequency：代码增删（绿增红删）                │
│      └── Dependency Graph：依赖关系                         │
│                                                            │
│  3️⃣  Dependabot——自动化依赖更新                               │
│      ├── 定时扫描依赖 → 自动创建更新 PR                       │
│      ├── 三个必填：package-ecosystem, directory, interval   │
│      ├── Groups：分组管理，合并同类更新                      │
│      ├── Ignore：忽略特定版本或库                           │
│      └── 支持 pip/npm/maven/gradle 等主流包管理器            │
│                                                            │
│  4️⃣  Projects——可视化管理看板                                 │
│      ├── 卡片 + 列 = 任务管理                                │
│      ├── 状态流转：Todo → In Progress → ... → Done          │
│      ├── 卡片可转 Issue，关联编号                            │
│      ├── Team Capacity：团队工作量视图                       │
│      ├── Size / Priority / Iteration 专业级管理              │
│      └── 多视图：Board / Table / Roadmap                    │
│                                                            │
│  5️⃣  Discussions——项目社区论坛                                │
│      ├── 发帖讨论、问答求助、分享心得                          │
│      ├── 按分类和标签筛选                                    │
│      └── 大项目启用，小项目用 Issue 即可                      │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 11.2 一句话总结

> 代码是仓库的骨架，Wiki、Insights、Dependabot、Projects、Discussions 是仓库的血肉——它们共同构成了一套完整的项目协作基础设施。
