## 二、git diff——三区对比完全指南

### 2.1 回顾四个工作区

> 在「Git 命令行 1」中我们学过 Git 的四个分区，这里快速回顾：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  工作目录              暂存区              本地分支    远端    │
│  (Working Dir)    →   (Staging)     →   (Local)  → (Remote) │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐ ┌────────┐ │
│  │ 你正在    │     │ git add  │     │git commit│ │git push│ │
│  │ 编辑的    │ ──→ │ 之后的   │ ──→ │ 之后的   │ │   →    │ │
│  │ 文件     │     │ 文件快照  │     │ 版本记录  │ │        │ │
│  └──────────┘     └──────────┘     └──────────┘ └────────┘ │
│                                                             │
│  git diff 就是在这些区域之间做对比！                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 工作目录 vs 暂存区：`git diff`

> `git diff` 后面什么都不加，比较的是**工作目录**和**暂存区**之间的差异。

**操作演示**：

```bash
# 当前状态：在 color 文件中新增一行 yellow
echo "yellow" >> color

# 查看工作目录和暂存区的差异
git diff
```

**输出解读**：

```diff
diff --git a/color b/color
index e69de29..5c8b7e2 100644
--- a/color
+++ b/color
@@ -0,0 +1 @@
+yellow          ← 新增的内容（工作目录有，暂存区没有）
```

**接下来，把文件加入暂存区后再看**：

```bash
# 把修改加入暂存区
git add color

# 再次执行 git diff
git diff
# （无输出——说明工作目录和暂存区现在一致了！）
```

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  理解关键：                                          │
│  ├── git diff 只显示「还没 git add」的改动            │
│  ├── git add 之后，git diff 就看不到了               │
│  └── 因为此时工作目录 = 暂存区，没有差异               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.3 暂存区 vs 本地分支：`git diff --staged`

> `git diff --staged`（或 `git diff --cached`）比较的是**暂存区**和**本地分支（最新 commit）**之间的差异。

```bash
# 此时 color 文件在暂存区中（已 git add，还未 commit）
git diff --staged
```

**输出解读**：显示暂存区里有什么内容是即将被提交但还未提交的。

```bash
# 如果想把文件从暂存区拿出来（取消暂存）
git restore --staged color

# 此时再执行 git diff
git diff
# 又能看到 yellow 的改动了（因为改动回到工作目录了）
```

```
工作目录 ──git diff──→ 暂存区 ──git diff --staged──→ 本地分支
   ↑                        ↑
   └── git diff HEAD ──────→┘（直接比较工作目录和本地分支）
```

### 2.4 工作目录 vs 本地分支：`git diff HEAD`

> `git diff HEAD` 比较的是**工作目录**和**本地分支（HEAD 指向的最新 commit）**之间的差异。

```bash
git diff HEAD
```

**这个命令的实用场景**：你已经做了很多修改，想一次性看看「从上次 commit 以来，我到底改了哪些东西？」——不需要关心哪些 add 了、哪些没 add，直接看全部改动。

### 2.5 三区对比速查表

| 命令 | 对比对象 | 使用场景 |
|---|---|---|
| `git diff` | 工作目录 ↔ 暂存区 | 看哪些改动还没 add |
| `git diff --staged` | 暂存区 ↔ 本地分支 | 看即将 commit 什么 |
| `git diff HEAD` | 工作目录 ↔ 本地分支 | 看从上次 commit 以来所有改动 |

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│                    git diff 三区对比关系图                      │
│                                                             │
│                        git diff HEAD                         │
│              ┌─────────────────────────────────┐             │
│              ↓                                 ↓             │
│      ┌───────────┐    git diff     ┌───────────┐           │
│      │  工作目录   │ ←──────────→  │   暂存区    │           │
│      │ (Working)  │               │ (Staging)  │           │
│      └───────────┘               └───────────┘           │
│                                        │                    │
│                              git diff --staged              │
│                                        │                    │
│                                        ↓                    │
│                                  ┌───────────┐             │
│                                  │  本地分支   │             │
│                                  │  (Local)   │             │
│                                  └───────────┘             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、分支对比：2 点比较与 3 点比较

### 3.1 2 点比较（绝对差异）

> 2 点比较直接比较两个分支**当前最新状态**之间的差异。

**准备实验环境**：

```bash
# 基于 main 创建 feature 分支
git checkout -b feature

# 在 feature 分支上添加一行 red 并提交
echo "red" >> color
git commit -am "red"

# 切换回 main 分支
git switch main

# 在 main 分支上添加一行 green 并提交
echo "green" >> color
git commit -am "green"
```

**执行 2 点比较**：

```bash
# main 与 feature 的 2 点比较
git diff main..feature
# 显示 feature 分支相对于 main 分支的差异

# 颠倒顺序
git diff feature..main
# 显示 main 分支相对于 feature 分支的差异（结果与上面相反）
```

**2 点比较的含义**：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│           C1（共同祖先）                                     │
│          /          \                                       │
│         /            \                                      │
│    C2 (feature)    C3 (main)                                │
│     +red             +green                                 │
│                                                             │
│  git diff main..feature                                     │
│  → 直接比较 C3 和 C2 的差异                                  │
│  → 结果是两个分支各自改了什么都会显示                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 3 点比较（相对差异）

> 3 点比较先找到两个分支的**共同祖先**，再用**靠后面的那个分支**与共同祖先进行对比。

```bash
# 3 点比较
git diff main...feature
```

**3 点比较的含义**：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│           C1（共同祖先） ← 第一步：找到它！                    │
│          /          \                                       │
│         /            \                                      │
│    C2 (feature)    C3 (main)                                │
│     +red             +green                                 │
│                                                             │
│  git diff main...feature                                    │
│  → 第一步：找到共同祖先 C1                                    │
│  → 第二步：靠后面的分支（feature）与 C1 对比                   │
│  → 结果：只显示 feature 分支上的改动（red）                   │
│  → 看不到 main 分支的 green！                                │
│                                                             │
│  git diff feature...main                                    │
│  → 第一步：找到共同祖先 C1                                    │
│  → 第二步：靠后面的分支（main）与 C1 对比                      │
│  → 结果：只显示 main 分支上的改动（green）                    │
│  → 看不到 feature 分支的 red！                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 GitHub PR 中的 3 点比较

> ⚠️ **重要发现**：GitHub 上的 Pull Request 页面**默认使用 3 点比较**！

**实操演示**：

```
在 GitHub 上创建 PR（feature → main）：
├── URL 中包含三个点：.../main...feature
├── 显示的差异 = feature 与共同祖先的差异
├── 只看到 feature 分支改了 red
└── 看不到 main 分支改了什么（green）

如果想看到 2 点比较的结果：
├── 直接在 URL 中把三个点改成两个点
├── .../main..feature
└── 此时能看到两个分支分叉后各自的改动
```

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  GitHub PR 的比较逻辑（注意！）：                              │
│                                                             │
│  feature → main 的 PR                                       │
│  ├── 默认 3 点比较：feature 与共同祖先比                      │
│  │   → 只看 feature「引入了什么新东西」                       │
│  └── 手动改 2 点比较：feature 与 main 直接比                  │
│      → 看两个分支「各有什么不同」                             │
│                                                             │
│  为什么 PR 默认 3 点？                                       │
│  └── 作为 reviewer，你关心的是「这个 PR 带来了什么变更」        │
│      而不是「main 分支上别人加了什么」（你已经知道了）         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 场景对比图

| 对比方式 | 命令 | 比较逻辑 | 能看到什么 | GitHub PR 默认 |
|---|---|---|---|---|
| **2 点** | `git diff A..B` | A 最新 vs B 最新 | 两个分支各自的改动 | ❌（需手动改 URL） |
| **3 点** | `git diff A...B` | B 最新 vs 共同祖先 | 只看 B 分支引入了什么 | ✅ |

## 四、commit 与 tag 之间的对比

### 4.1 commit 之间对比

> `git diff` 后面可以直接接两个 commit ID，比较任意两次提交之间的差异。

```bash
# 先查看提交历史，获取 commit ID
git log --oneline

# 输出示例：
# a1b2c3d 第三次提交
# e4f5g6h 第二次提交
# i7j8k9l 第一次提交

# 任意选择两个 commit ID 进行比较
git diff i7j8k9l a1b2c3d
# 对比从"第一次提交"到"第三次提交"之间的所有变化
```

### 4.2 tag 参与对比

> tag、分支、commit 三者之间**可以任意组合**进行对比。

```bash
# tag 和 tag 对比
git diff v1.0.0 v1.0.2

# tag 和分支对比
git diff v1.0.0 main

# tag 和 commit 对比
git diff v1.0.0 a1b2c3d

# commit 和分支对比
git diff a1b2c3d feature
```

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  git diff 的对比对象可以是以下任意组合：                        │
│                                                             │
│       ┌─────────┐                                           │
│       │   tag   │──────┐                                    │
│       └─────────┘      │                                    │
│                         ├── 任意两两组合都可以！               │
│       ┌─────────┐      │                                    │
│       │  分支    │──────┤                                    │
│       └─────────┘      │                                    │
│                         │                                    │
│       ┌─────────┐      │                                    │
│       │ commit  │──────┘                                    │
│       └─────────┘                                           │
│                                                             │
│  本质：它们都指向某个代码快照，diff 就是比较两个快照的差异。     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五、git stash——临时保存工作现场

### 5.1 使用场景

> 你正在 feature 分支上写代码，突然需要切换到 main 分支修一个紧急 bug。但你当前的代码还没写完，不想 commit。怎么办？——**用 `git stash` 把工作现场暂存起来。**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Stash 的典型使用场景：                                       │
│  ├── 代码写了一半，需要紧急切分支                             │
│  ├── 当前改动不想 commit（未完成、实验性代码）                │
│  ├── 想把当前的改动先"存档"，之后再回来继续                   │
│  └── 在不同的分支之间搬运未提交的改动                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 基本用法：先暂存再 stash

```bash
# 当前状态：修改了 color 文件，还未 commit
git status
# modified: color

# 方式一：先 git add，再 git stash
git add .
git stash
# 保存成功！工作目录恢复干净

# 查看 stash 列表
git stash list
# 输出：stash@{0}: WIP on feature: a1b2c3d red
#       ↑ stash 的名字（自动生成）
```

> ⚠️ **注意**：默认的 `git stash` 只会保存**已经 add 到暂存区**的改动。只修改了但没 add 的文件不会被 stash！

### 5.3 进阶用法：`-a` 一键全存

> `git stash -a`（`-a` = `--all`）**同时保存工作区和暂存区的所有修改**，不要求文件先经过暂存区。

```bash
# 方式二：一步到位，不需要先 git add
git stash -a

# 查看状态
git status
# nothing to commit, working tree clean
# 所有改动（不管有没有 add）都被 stash 了！
```

| 方式 | 命令 | 前提条件 | 保存范围 |
|---|---|---|---|
| 常规 | `git add` + `git stash` | 需要先 add | 仅暂存区 |
| 一键 | `git stash -a` | 无 | 工作区 + 暂存区全部 |

### 5.4 取出 stash：apply vs pop

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  取出 stash 的两种方式：                                      │
│                                                             │
│  ┌──────────────────┬──────────────────────────────┐        │
│  │  git stash apply  │  git stash pop               │        │
│  ├──────────────────┼──────────────────────────────┤        │
│  │ 取出修改           │ 取出修改                       │        │
│  │ stash 保留在列表中  │ stash 从列表中删除              │        │
│  │ 可反复取出         │ 只能取出一次                    │        │
│  │ 需要手动删除       │ 自动清理                       │        │
│  └──────────────────┴──────────────────────────────┘        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```bash
# ── apply：取出但不删除 ──
git stash apply stash@{0}
# 修改恢复到了工作目录
git stash list
# stash@{0} 还在！可以再次取出

# ── pop：取出并删除 ──
git stash pop stash@{0}
# 修改恢复到了工作目录
git stash list
# stash@{0} 没了！已被自动清理
```

> ⚠️ **Windows PowerShell 用户注意**：花括号 `{}` 在 PowerShell 中是特殊语法，需要使用引号包裹：`git stash apply "stash@{0}"`

### 5.5 删除 stash：drop

```bash
# 手动删除某个 stash
git stash drop stash@{0}

# 验证：列表已清空
git stash list
# （无输出）
```

### 5.6 stash 完整生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│                    git stash 生命周期                          │
│                                                             │
│   工作目录有未提交的改动                                       │
│         │                                                   │
│         ├── git stash      （需先 git add）                   │
│         ├── git stash -a   （无需 add）                      │
│         ↓                                                   │
│   改动存入 stash 列表（stash@{0}, stash@{1}, ...）            │
│   工作目录恢复干净                                             │
│         │                                                   │
│         ├── git stash list   → 查看所有 stash                │
│         │                                                   │
│         ├── git stash apply  → 取出改动（stash 保留）         │
│         │         │                                         │
│         │         └── git stash drop → 手动删除              │
│         │                                                   │
│         ├── git stash pop    → 取出改动（stash 自动删除）     │
│         │                                                   │
│         └── git stash drop   → 直接删除（不取出）             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 六、git tag——给提交打标签

### 6.1 在当前提交打 tag

> tag 就像一个**不会移动的分支指针**，通常用于标记版本号（v1.0.0, v2.1.0 等）。

```bash
# 在当前 HEAD 位置打一个 tag
git tag v1.0.0

# tag 不能重名！换个名字
git tag v1.0.1
```

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  tag vs branch 的区别：                                       │
│                                                             │
│  branch：会随着新 commit 往前移动                             │
│  tag：钉死在某个 commit 上，永远不动                          │
│                                                             │
│  类比：branch 是河流（持续流动），tag 是界碑（固定不变）       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 在历史提交打 tag

> 可以给**任意一个历史提交**打 tag，不止是最新的。

```bash
# 先查看历史提交
git log --oneline

# 复制目标 commit ID，然后打 tag
git tag v1.0.2 a1b2c3d
#            ↑ tag 名  ↑ 历史 commit ID
```

**在 GitHub Desktop 中验证**：可以看到该历史提交被打上了 v1.0.2 的标签。

### 6.3 推送 tag 到远端

```bash
# 把所有本地 tag 一次性推送到远端
git push --tags
```

> ⚠️ 普通的 `git push` **不会**推送 tag，必须使用 `--tags` 参数。

### 6.4 删除 tag（本地 & 远端）

```bash
# ── 删除本地 tag ──
git tag -d v1.0.1
# 注意：这里 -d 是 delete 的缩写

# ── 删除远端 tag ──
git push origin --delete v1.0.1
# 语法与删除远端分支完全一样
```

### 6.5 tag 参与 diff 对比

> 如第 4.2 节所述，tag 可以和 tag、分支、commit 任意组合进行 diff 对比。

```bash
# tag 之间对比
git diff v1.0.0 v1.0.2

# tag 和分支对比
git diff v1.0.0 main

# 2 点 / 3 点比较同样适用
git diff v1.0.0..v1.0.2
git diff v1.0.0...main
```

## 七、合并多个提交为一次提交

> 场景：你在 feature 分支上做了 3 个小提交（"第一次"、"第二次"、"第三次"），每个提交都是修修补补。在合并到 main 之前，你想把 3 个提交**压缩成 1 个**整洁的提交。

### 7.1 方法一：reset --mixed + 重做提交

> 核心思路：**退回到 3 次提交之前，保留文件改动，然后重新做一次提交。**

```bash
# 先查看提交历史
git log --oneline
# 输出：
# c3 第三次
# c2 第二次
# c1 第一次
# c0 之前的提交  ← 我们要回到这里

# 切换到对应分支
git switch feature

# 复制 c0 的 commit id，按 Q 退出 log

# ⚠️ 必须使用 --mixed（保留文件改动在工作目录）
git reset --mixed <c0的commit-id>
```

**reset 之后的状态**：

```bash
git status
# 显示：有未暂存的修改（那三次提交改过的文件内容全都保留在磁盘上！）

git diff
# 可以看到三次提交的所有改动都还在这里
```

```bash
# 把所有改动重新提交为一次
git add .
git commit -m "three into one：合并三次提交为一个"

# 查看历史记录
git log --oneline
# 三次提交已经合并成了一次！
```

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  reset --mixed 的工作原理：                                   │
│                                                             │
│  reset 前：... ← c0 ← c1 ← c2 ← c3 (HEAD)                   │
│                                                             │
│  reset --mixed c0 后：                                       │
│  ├── HEAD 回到 c0                                           │
│  ├── c1, c2, c3 的提交记录被丢弃                              │
│  └── c1, c2, c3 的**文件改动保留在工作目录**（关键！）          │
│                                                             │
│  然后重新 git add + git commit → 产生一个全新的提交            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 方法二：rebase -i 交互式合并

> 核心思路：使用交互式 rebase，在编辑器中指定哪些提交要「squash」（压缩进上一个提交）。

```bash
# 同样先查看提交历史
git log --oneline
# c3 第三次
# c2 第二次
# c1 第一次
# c0 之前的提交  ← 我们要 rebase 到这里

# 交互式 rebase
git rebase -i <c0的commit-id>
# -i = --interactive（交互模式）
```

**进入编辑器后的操作**：

```bash
# 编辑器显示如下内容：
pick c1 第一次
pick c2 第二次
pick c3 第三次

# 需要改成：
pick c1 第一次          ← 至少保留一个 pick！
squash c2 第二次        ← s = squash，把 c2 压缩进 c1
squash c3 第三次        ← s = squash，把 c3 压缩进 c1

# 操作步骤：
# 1. 按 i 进入编辑模式
# 2. 把后两行的 pick 改成 squash（或缩写 s）
# 3. 按 ESC 退出编辑模式
# 4. 输入 :wq! 保存并退出
```

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  rebase -i 中的操作选项：                                     │
│                                                             │
│  pick   = 原样保留这次提交                                   │
│  reword = 保留提交但修改 commit message                      │
│  edit   = 保留提交但停下来让你修改内容                         │
│  squash = 把这次提交合并进上一次提交                          │
│  drop   = 删除这次提交                                       │
│  merge  = 合并相关操作                                        │
│  reset  = 重置                                               │
│                                                             │
│  ⚠️ 关键规则：至少保留一个 pick！不能把所有都改成 squash。      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**编辑器会再次打开**，让你确认最终的 commit message：

```bash
# 这是合并后的提交信息确认页
# This is a combination of 3 commits.
# 第一次
# 第二次
# 第三次

# 按 ESC，输入 :wq! 保存退出
# rebase 成功！
```

```bash
# 查看压缩后的历史
git log --oneline
# 三次提交已经被压缩成了一次！
```

### 7.3 两种方法对比

| 维度 | reset --mixed | rebase -i squash |
|---|---|---|
| **操作复杂度** | ⭐ 简单直观 | ⭐⭐⭐ 需要编辑 rebase 脚本 |
| **中间状态** | 文件改动散落在工作目录 | 每一步都可控 |
| **commit message** | 需要自己重写 | 可选择保留/合并原有信息 |
| **适用场景** | 连续的几个提交，全部合并 | 需要挑选哪些合并、哪些保留 |
| **灵活度** | 低（要么全合并，要么不合并） | 高（可以 pick 部分，squash 部分） |
| **风险** | 操作不可逆（除非 reflog） | 操作不可逆（除非 reflog） |

### 7.4 注意事项：强制推送与多人协作

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ⚠️  重写历史 = 必须强制推送                                  │
│                                                             │
│  情况一：被合并的提交没有推送过远端                            │
│  ├── 直接 git push 即可                                      │
│  └── 因为远端没有这些提交，不存在冲突                          │
│                                                             │
│  情况二：被合并的提交已经在远端了                              │
│  ├── 必须使用 git push --force 或 git push -f                │
│  ├── 因为本地历史已经被重写，和远端不一致                      │
│  └── 普通 push 会报错！                                      │
│                                                             │
│  ⛔ 绝对禁止的操作：                                          │
│  ├── 在多人协作的集成分支（main / develop）上使用强制推送      │
│  ├── rebase -i 修改已经被别人引用的提交                       │
│  └── 强制推送会覆盖别人的工作，造成严重问题！                   │
│                                                             │
│  安全原则：                                                   │
│  ├── reset / rebase 只在自己的 feature 分支上做               │
│  ├── 推到远端前确认没有其他人基于该分支工作                     │
│  └── 不确定时，用方法一（reset --mixed + 新提交）更安全        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 八、PR 冲突——先在本地解决再提交

### 8.1 正确流程

> 当你要创建 Pull Request 时，发现 GitHub 提示「Cannot automatically merge」（存在冲突），**不要直接创建 PR**。正确的做法是：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  PR 冲突的正确处理流程：                                       │
│                                                             │
│  Step 1：切换到你自己的 feature 分支                          │
│  git switch feature                                         │
│         │                                                   │
│         ↓                                                   │
│  Step 2：把目标分支（main）合并进 feature 分支                 │
│  git merge main                                             │
│  → 在本地触发冲突                                            │
│         │                                                   │
│         ↓                                                   │
│  Step 3：在本地解决冲突                                       │
│  打开冲突文件 → 编辑保留需要的代码 → 保存                      │
│         │                                                   │
│         ↓                                                   │
│  Step 4：标记冲突已解决并提交                                 │
│  git add .                                                  │
│  git commit -m "resolve conflict with main"                 │
│         │                                                   │
│         ↓                                                   │
│  Step 5：推送到远端                                          │
│  git push                                                   │
│         │                                                   │
│         ↓                                                   │
│  Step 6：在 GitHub 上创建 PR                                 │
│  → 这次不会有任何冲突！PR 可以顺利创建                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**完整命令流程**：

```bash
# 1. 切换到自己的分支
git switch feature

# 2. 把 main 合并进来（本地触发冲突）
git merge main
# 输出：CONFLICT (content): Merge conflict in color
#      Automatic merge failed; fix conflicts and then commit the result.

# 3. 编辑冲突文件，保留需要的代码
# （在文件中看到 <<<<<<< ======= >>>>>>> 标记，手动选择保留哪些内容）

# 4. 标记冲突已解决并提交
git add .
git commit -m "resolve conflict: merge main into feature"

# 5. 推送
git push

# 6. 去 GitHub 创建 PR → 没有冲突了！
```

### 8.2 错误流程 vs 正确流程对比

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ❌ 错误流程：                                               │
│  GitHub 提示冲突 → 仍然创建 PR → 在 PR 页面上看冲突 → 困惑   │
│                                                             │
│  ✅ 正确流程：                                               │
│  GitHub 提示冲突 → 取消创建 PR → 本地 merge → 解决冲突        │
│  → commit → push → 再创建 PR（零冲突）                        │
│                                                             │
│  核心原则：冲突永远在本地解决！GitHub 的 PR 页面不是           │
│  用来解决冲突的最佳场所。                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 九、完整命令速查表

### git diff

| 命令 | 作用 |
|---|---|
| `git diff` | 工作目录 ↔ 暂存区 |
| `git diff --staged` | 暂存区 ↔ 本地分支 |
| `git diff HEAD` | 工作目录 ↔ 本地分支 |
| `git diff A..B` | 分支/commit/tag 之间 2 点比较 |
| `git diff A...B` | 分支/commit/tag 之间 3 点比较 |

### git stash

| 命令 | 作用 |
|---|---|
| `git stash` | 暂存暂存区中的改动（需先 add） |
| `git stash -a` | 暂存所有改动（工作区 + 暂存区） |
| `git stash list` | 查看 stash 列表 |
| `git stash apply stash@{n}` | 取出改动（保留 stash） |
| `git stash pop stash@{n}` | 取出改动（删除 stash） |
| `git stash drop stash@{n}` | 删除 stash |

### git tag

| 命令 | 作用 |
|---|---|
| `git tag v1.0.0` | 在当前 HEAD 打 tag |
| `git tag v1.0.0 <commit-id>` | 在历史提交打 tag |
| `git push --tags` | 推送所有 tag 到远端 |
| `git tag -d v1.0.0` | 删除本地 tag |
| `git push origin --delete v1.0.0` | 删除远端 tag |

### 提交合并

| 命令 | 作用 |
|---|---|
| `git reset --mixed <commit-id>` | 回退但保留文件改动 |
| `git rebase -i <commit-id>` | 交互式 rebase（可 squash 提交） |

## 十、常见问题与排错指南

### 10.1 错误速查表

| 错误信息 / 现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| `git stash` 后文件没保存 | 文件没有先 add | 使用 `git stash -a` 代替 | ⭐⭐⭐ |
| `fatal: tag 'v1.0.0' already exists` | tag 不能重名 | 换个名字，或先删除旧 tag | ⭐⭐ |
| PowerShell 中 `stash@{0}` 报错 | `{}` 是 PowerShell 特殊语法 | 加引号：`"stash@{0}"` | ⭐⭐ |
| `git push` 后 tag 没出现在远端 | 普通 push 不推送 tag | 使用 `git push --tags` | ⭐⭐⭐ |
| reset 后 push 报错（non-fast-forward） | 历史被重写，远端有旧提交 | 使用 `git push --force`（仅限个人分支！） | ⭐⭐⭐ |
| rebase -i 后编辑器卡住了 | 进入了 Vim 编辑模式 | 按 i 编辑 → ESC → `:wq!` 退出 | ⭐⭐ |
| 3 点比较结果和预期不一样 | 混淆了 2 点和 3 点的含义 | 回顾第 3.4 节对比图 | ⭐ |

### 10.2 排查流程

```
执行 git stash 后 status 仍然有改动
    │
    ├── 文件是否先 git add 了？
    │   ├── 是 → 检查是否使用了 git stash -a
    │   └── 否 → git stash -a 可以一步搞定
    │
执行 git diff --staged 无输出
    │
    ├── 有没有先 git add？
    │   ├── 是 → 检查文件是否已经在暂存区
    │   └── 否 → 先 git add 再试
    │
reset --mixed 后 push 失败
    │
    ├── 这些提交推过远端吗？
    │   ├── 否 → 应该可以直接 push，检查是否在正确的分支
    │   └── 是 → 需要 --force，但确认这是你自己的分支！
    │
PR 页面提示冲突
    │
    └── → 不要创建 PR！先 git merge main 在本地解决
```

## 十一、课后练习

### 练习一：git diff 三区对比实操

1. 创建一个新文件 `test.txt`，写入 `line1`
2. 执行 `git diff`，观察输出
3. `git add test.txt`，再次执行 `git diff` 和 `git diff --staged`，对比两次输出的差异
4. 在 `test.txt` 中添加 `line2`（不 add），执行 `git diff HEAD`，观察输出

**验收标准**：能准确说出三种 diff 命令分别在比较哪两个区域。

### 练习二：stash 完整工作流

1. 创建一个文件 `work.txt`，写入 `task A`
2. 使用 `git stash -a` 暂存这个改动
3. 使用 `git stash list` 确认 stash 存在
4. 切换到一个新分支，修改一些内容并提交
5. 切回原分支，使用 `git stash pop` 恢复改动
6. 确认 `git stash list` 中 stash 已删除

**验收标准**：能独立完成 stash 的存储、查看、取出、自动清理全流程。

### 练习三：提交压缩 + tag + PR 模拟

1. 在 feature 分支上做 3 个小提交（内容自定）
2. 使用 `rebase -i squash` 将 3 个提交合并为 1 个
3. 在合并后的提交上打 tag `v1.0.0`
4. 模拟 PR 冲突场景：
   - 切回 main 分支，修改同一文件的同一行并提交
   - 切换到 feature 分支，执行 `git merge main`
   - 在本地解决冲突并提交
5. （可选）推送到你的远端仓库

**验收标准**：
- `git log --oneline` 显示 3 个提交被合并成了 1 个
- tag `v1.0.0` 存在并能参与 `git diff`
- 冲突已在本地解决，merge commit 存在

### 12.2 一句话总结

> **`git diff` 让你看清改了哪里，`git stash` 让你临时保存不丢失，`git tag` 让你标记里程碑，提交合并让你保持历史整洁——这四项技能组合起来，让你的 Git 操作从"能提交代码"升级到"能优雅管理代码历史"。**
