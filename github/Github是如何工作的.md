## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经掌握了：

- **创建仓库** —— 知道如何创建 GitHub 仓库、选择开源协议、编写 README
- **理解 Commit** —— Commit 是仓库状态的快照，形成历史链路，可回溯任意时间点
- **编辑和上传文件** —— 在线编辑和本地上传两种方式
- **GitHub Gist** —— 轻量级代码片段分享

现在你有了仓库，也会提交代码了。但实际工作中，你不可能直接在主干上改代码——万一改坏了怎么办？多人同时改同一个文件怎么办？这节课就来解决**"怎么安全地开发"**这个问题。

### 1.2 本节课的核心问题

| 问题 | 说明 |

| 代码写完了怎么合回去？ | Pull Request（PR）——合并请求的工作流程 |
| 什么是 Code Review？ | 为什么大厂都在做代码审查？ |

> ⚠️ **关键认知**：分支 + Pull Request 是 GitHub 协作的**核心工作流**。全世界数百万开发者每天都在用这个流程协作。学会它就等于掌握了 GitHub 最核心的用法。

## 二、什么是分支（Branch）

### 2.1 直观理解

**分支（Branch）** 简单来说，就是存储库的**不同版本**。

```
默认情况：
    每个仓库都有一个 main 分支（或 master 分支）→ 主干

创建分支：
    基于 main 创建一个 feature 分支 → 在 feature 上开发 → 不影响 main
```

**类比：游戏存档的分支剧情**

```
                    main（主线剧情）
                       │
            创建 feature 分支（支线剧情）
                       │
        ┌──────────────┴──────────────┐
        │                             │
    main 上继续主线              feature 上探索支线
    （不受支线影响）              （不影响主线）
        │                             │
        └──────────────┬──────────────┘
                       │
              支线完成，合并回主线
```

> 💡 分支的全称在 Git 命令中是 `branch`。在 GitHub 网页上，你会在仓库左上角看到一个带分支图标的小按钮，点击它就能看到所有分支。

### 2.2 分支的隔离性——核心特性

分支最核心的特性是**隔离性**：各自分支上的代码修改**不会互相影响**。

**关键规则：**

```
当你从 main 创建 feature 分支时：
    feature 分支 = main 分支当时状态的完整快照
    也就是说：刚创建出来时，feature 和 main 的代码完全一样

当你在 feature 上进行修改并提交后：
    ├── 修改只对 feature 分支有效 ✅
    ├── 只能在 feature 分支上被看到 ✅
    ├── 切换到 main 分支后，看不到 feature 的修改 ✅
    └── main 分支不受任何影响 ✅

反过来也一样：
    如果有人在 main 上做了修改，feature 上也不会自动出现这些修改
```

```
         main 分支                    feature 分支
    ┌──────────────┐            ┌──────────────┐
    │ README.md     │            │ README.md     │
    │ (原始内容)     │   互不     │ (你加了新内容)  │
    │               │ ← 影响 →  │               │
    │ 看不到feature  │            │ 看不到main上   │
    │ 上的改动       │            │ 的新改动       │
    └──────────────┘            └──────────────┘
```

> ⚠️ **一句话总结**：当你切换回 main 主干时，是看不到 feature 分支上的代码修改的。各自分支上的修改互不影响。

### 2.3 为什么需要分支

**没有分支的世界：**

```
所有人都在 main 上直接改代码
    ├── 张三改了一半的功能 → 推上线 → 系统崩了
    ├── 李四也在这时候改同一个文件 → 代码冲突 → 谁的改动丢了？
    └── 紧急 Bug 要修 → 但 main 上还有一堆未完成的代码 → 怎么上线？
```

**有了分支的世界：**

```
每个人创建自己的分支
    ├── 张三在 feature-zhang 上开发 → 不影响任何人
    ├── 李四在 feature-li 上开发 → 不影响任何人
    ├── 王五在 bugfix 上紧急修复 → 修复完直接合并，不受开发分支干扰
    └── 各人开发完成后 → 合并回 main → 代码审查 → 安全上线
```

| 没有分支 | 有分支 |
|---|---|
| 所有人都挤在一条线上改代码 | 每人有自己的独立空间 |
| 半成品代码可能被误上线 | 不合并就不会影响主干 |
| 紧急修复被未完成的功能阻塞 | 可以基于 main 快速创建修复分支 |
| 代码冲突频繁 | 各分支独立，减少冲突 |

## 三、分支工作流动画演示

> 老师使用了一个开源项目 [Learn Git Branching](https://learngitbranching.js.org/) 来以动画形式演示分支的工作效果。强烈建议你课后亲自去玩一下这个网站，非常直观。

### 3.1 场景设定

一个典型的日常工作场景：

```
你正在 feature 分支上开发新功能
    → 功能做了一半，还没做完
    → 突然领导说：生产环境有 Bug，赶紧修！
    → 你不能把做了一半的功能直接上线
    → 怎么办？
```

### 3.2 完整流程推演

**第一步：创建 feature 分支，开始开发**

```
main 分支
    │
    ├── 创建 feature 分支（feature = main 当前状态的快照）
    │       │
    │       ├── Commit 1：开发功能A的前半部分
    │       └── Commit 2：开发功能A的后半部分
    │              （功能还没做完！）
    │
    │  此时 feature 领先 main 两个 Commit
```

```
可视化：

main:     ●━━━━━━━━━━━━━━━━━━━━ (原始状态)
                      │
feature:              ●━━━━●━━━━● (领先2个Commit，功能未完成)
                     C1    C2
```

**第二步：紧急 Bug 来了！**

```
领导："生产环境有 Bug，马上修！"

你不能把 feature 直接合并（功能才做了一半）
    → 正确的做法：

1. 切换回 main 分支：git switch main
2. 基于 main 创建 bugfix 分支
3. 在 bugfix 上修复 Bug → Commit
4. 把 bugfix 合并回 main
5. 上线 main → Bug 修复完成 ✅
```

```
可视化：

main:     ●━━━━━━━━━━━━━━━━━━━━━● (bugfix合并进来，Bug修复)
                      │          │
feature:              ●━━━━●━━━━● (你的开发没受影响！)
                     C1    C2
                      
bugfix:                        ● (修复Bug的提交，已合并)
```

**第三步：回到 feature 继续开发**

```
Bug 修完了，回到 feature 分支：

1. 切换回 feature 分支
2. 继续开发 → Commit 3：功能收尾
3. Commit 4：功能测试完成
4. 功能开发完成！🎉
```

```
可视化：

main:     ●━━━━━━━━━━━━━━━━━━━━━●
                      │          │
feature:              ●━━━━●━━━━●━━━━●━━━━● (4个Commit，功能完成)
                     C1    C2    C3    C4
```

**第四步：把 feature 合并回 main**

```
feature 开发完成 → 合并回 main

    main:     ●━━━━━━━━━━━━━━━━━━━━━●━━━━━━━━━━━━━━━━●
                      │          │                    │
    feature:          ●━━━━●━━━━●━━━━●━━━━●           │
                     C1    C2    C3    C4             │
                                                      │
    bugfix:                        ●                  │
                                                      │
    最终：bugfix 和 feature 的改动都合并进了 main ✅
```

### 3.3 流程总结

```
                    ┌─────────────┐
                    │  main 分支   │ ← 主干，始终是可上线状态
                    └──────┬──────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
            ▼              ▼              ▼
    ┌───────────┐  ┌───────────┐  ┌───────────┐
    │ feature   │  │ bugfix    │  │ 张三的     │
    │ (你的功能) │  │ (紧急修复) │  │ 开发分支   │
    └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
          │              │              │
          │   开发完成    │  修复完成     │  开发完成
          ▼              ▼              ▼
          └──────────────┴──────────────┘
                         │
                    合并回 main
                         │
                         ▼
                    上线部署 🎉
```

> 💡 **这就是分支存在的意义**：你可以**同时进行**功能开发和 Bug 修复，互不干扰。每个人都在自己的分支上工作，不会踩到别人的代码。

## 四、在 GitHub 上操作分支

> 本文档先教 GitHub 网页端的操作（最推荐新手使用），命令行的操作在后续课程中会详细讲解。

### 4.1 创建分支

以之前创建的 `hello-world` 仓库为例：

```
仓库首页 → 点击左上角带分支图标的小按钮 → 输入分支名称 → 点击创建
```

**实操步骤：**

```
Step 1: 进入 hello-world 仓库首页
    ↓
Step 2: 找到左上角的分支选择器（显示 "main" 的按钮）
    ↓
Step 3: 点击它，在输入框中输入新分支名，例如 "feature"
    ↓
Step 4: 看到提示："Create branch feature from main"
         （基于 main 分支创建一个叫 feature 的分支）
    ↓
Step 5: 点击 "Create branch feature from main"
    ↓
🎉 feature 分支创建成功！当前自动切换到 feature 分支。
```

> ⚠️ **注意提示信息**："Create branch feature **from main**" —— 这意味着新分支是基于 main 创建的。创建出来的 feature 分支包含 main 当时的所有代码。

### 4.2 在分支上修改代码

现在我们切换到 feature 分支上，做一点修改：

```
确保当前在 feature 分支 → 找到要修改的文件 → 点击 ✏️ 小铅笔
    ↓
在文件开头添加一行内容
    ↓
点击 "Commit changes"
    ↓
注意：这次 Commit 是提交到 feature 分支，不是 main 分支！
    ↓
提交成功 ✅
```

**关键验证：**

```
feature 分支上 → 文件有新增的那行内容 ✅
切换到 main 分支 → 同一个文件，没有新增内容 ✅

        → 证明：feature 上的修改不会影响 main ✅
```

> 💡 提交完成后，GitHub 可能会弹出一个绿色的 **"Compare & pull request"** 按钮——这是 GitHub 在提醒你："你在这个分支上做了修改，要不要发起一个合并请求？"这个我们第五节详细讲。

### 4.3 切换分支

```
仓库首页 → 点击左上角分支选择器 → 点击你想切换到的分支名称
    ↓
页面刷新，当前分支切换完成
    ↓
你看到的是该分支上的代码状态
```

```
分支选择器：

    ┌──────────┐
    │ main  ▾  │ ← 点击展开
    └──────────┘
         │
    ┌────┴────────┐
    │ main         │ ← 主干分支
    │ feature      │ ← 你创建的功能分支
    │ bugfix       │ ← 修复分支（如果有的话）
    └──────────────┘
```

### 4.4 基于任意分支创建新分支

分支不仅可以基于 main 创建，还可以**基于任何已有分支创建新的分支**。

```
仓库首页 → 点击 Branches 标签 → 进入所有分支列表
    ↓
点击 "New branch" 按钮
    ↓
输入新分支名，例如 "feature2"
    ↓
在 "Source" 下拉菜单中，选择基于哪个分支创建
    ├── 基于 main → feature2 只能看到 main 的代码
    └── 基于 feature → feature2 能看到 feature + main 的所有代码
    ↓
点击创建
```

**不同来源的区别：**

```
基于 main 创建 feature2：
    feature2 内容 = main 的代码
    （看不到 feature 上的修改）

基于 feature 创建 feature2：
    feature2 内容 = main 的代码 + feature 的所有修改
    （包含了 feature 上的一切）
```

```
可视化：

main:     ●━━━━━━━━━━━━━━ (原始状态)
              │
feature:      ●━━━━●━━━━ (有额外修改)
              │     │
              │     └── 基于 feature 创建 → 包含所有修改
              │
              └── 基于 main 创建 → 只有原始状态
```

> 💡 **适用场景**：
> - 基于 main 创建分支：做独立的新功能
> - 基于 feature 创建分支：多人协作同一个大功能，各自负责子功能

### 4.5 删除分支

当功能分支的代码已经合并进 main 后，建议**删除该功能分支**，保持分支树的干净。

```
仓库首页 → 点击 Branches 标签 → 查看所有分支
    ↓
找到已合并的功能分支（如 feature）
    ↓
点击右侧的 🗑️ 删除按钮
    ↓
🎉 分支已删除，刷新后只剩下 main 等活跃分支
```

> ⚠️ **什么时候删除？** 分支代码已经合并进主干之后。合并前不要删——合并操作需要源分支存在。

## 五、Pull Request（合并请求）详解

### 5.1 PR 是什么

**Pull Request（PR）** 是 GitHub 上最核心的协作机制。

| 名称 | 含义 |
|---|---|
| 英文 | Pull Request（拉取请求） |
| 更好的翻译 | **合并请求**——将更改从一个分支合并进另一个分支的提案 |
| 本质 | 一个**提案**：请求仓库管理员把自己的代码合并进主干 |

```
PR = 一个提案，包含三部分：

    ① 比较两个分支之间的代码差异（改了什么？）
    ② 让管理员审核代码改动（Code Review）
    ③ 审核通过后，将代码合并进目标分支（Merge）
```

> 💡 **类比**：就像你写了一篇文章的修改稿，提交给编辑审核。编辑看了说"这里改一下"，你改完再提交，最后编辑说"OK 了"，把你的修改合并到正式版本里。

### 5.2 PR 的工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Step 1：贡献者在自己的分支上修改代码                            │
│      ↓                                                       │
│  Step 2：贡献者创建 Pull Request（合并请求提案）                 │
│      ↓                                                       │
│  Step 3：PR 页面展示两个分支的代码差异                           │
│      ↓                                                       │
│  Step 4：仓库管理员审核代码（Code Review）                      │
│      ├── 可以在特定代码行留言                                   │
│      ├── 可以提出修改建议                                      │
│      └── 最终给出审核结论 ↓                                    │
│      ↓                                                       │
│  Step 5：审核通过 → 管理员点击 Merge → 代码合并进主干            │
│      ↓                                                       │
│  Step 6：（可选）删除已合并的功能分支，保持仓库整洁               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 创建 PR

**操作步骤：**

```
仓库首页 → 点击 "Pull requests" 选项卡 → 点击 "New pull request"
    ↓
选择合并方向：
    base: main     ← 合并到哪里（目标分支）
    compare: feature ← 从哪个分支合并（源分支）
    方向 = feature → main
    ↓
GitHub 自动展示两个分支之间的代码差异：
    ├── 绿色背景 = 新增的代码行
    └── 红色背景 = 删除的代码行
    ↓
确认改动无误 → 点击 "Create pull request"
    ↓
填写 PR 信息：
    ├── 标题：简洁描述这次改了什么
    └── 描述：详细说明改动的背景、内容、测试情况
    ↓
点击 "Create pull request"
    ↓
🎉 PR 创建成功！
```

**合并方向说明：**

```
base: main     ← 这是"目的地"，代码要合到这里
compare: feature ← 这是"来源"，从这取代码

表示：把 feature 的代码合并进 main
```

**PR 描述模板示例：**

```markdown
## 改动内容
在 README.md 开头添加了课程介绍

## 改动原因
让新访客第一时间了解这个仓库的用途

## 测试情况
- [x] 本地预览通过
- [x] 格式检查通过
```

### 5.4 Code Review（代码审查）

PR 创建后，仓库管理员可以在 PR 页面进行**Code Review（代码审查）**。

**管理员操作入口：**

```
PR 页面 → 点击 "Files changed" 选项卡 → 查看所有代码改动
```

**在具体代码行上评论：**

```
把鼠标移到某一行代码的左侧 → 出现蓝色 + 号 → 点击
    ↓
弹出评论框 → 输入审查意见
    ↓
可以写：
    ├── "这里变量名可以更清晰一点"
    ├── "建议把这段逻辑抽取成独立函数"
    └── "这行有潜在的空指针风险"
    ↓
点击 "Add single comment" 或 "Start a review"
```

**最终审核结论（三种）：**

```
管理员在 PR 页面右上角 → 点击 "Review changes" → 选择结论：

┌────────────────┬──────────────────────────────────────┐
│ Comment        │ 只是评论，不决定是否合并               │
│                │ "我有些想法，先讨论一下"               │
├────────────────┼──────────────────────────────────────┤
│ Approve        │ 同意合并！代码没问题，可以合入主干      │
│                │ "LGTM! (Looks Good To Me!)"         │
├────────────────┼──────────────────────────────────────┤
│ Request changes│ 要求修改，改完再考虑合并               │
│                │ "这里有3个地方需要修改"                │
└────────────────┴──────────────────────────────────────┘
```

```
Code Review 审核流程：

贡献者提交 PR
    ↓
管理员审查代码
    │
    ├── Comment → 讨论中，等回应
    │       └── 贡献者回复 / 修改 → 管理员重新审查
    │
    ├── Request Changes → 需要修改
    │       └── 贡献者修改代码 → 提交新 Commit → PR 自动更新 → 管理员重新审查
    │
    └── Approve → 代码 OK！
            └── 进入合并步骤
```

> ⚠️ **Code Review 的意义**：一个人写代码难免有盲区——逻辑漏洞、性能问题、安全隐患。另一个人审核可以发现这些问题。这就是为什么大厂都强制要求 Code Review。

### 5.5 合并 PR

当管理员审核通过（Approve）后，就可以合并了：

```
PR 页面 → 点击 "Merge pull request" 按钮
    ↓
点击 "Confirm merge" 确认
    ↓
PR 状态变为 "Merged"（已合并）✅
    ↓
回到仓库首页 → 切换到 main 分支 → feature 的代码已经出现在 main 上了！
```

```
合并前：                             合并后：

main:     ●━━━━━━━━━━              main:     ●━━━━━━━━━━● (含feature代码)
                │                                  │
feature:        ●━━━━●━━━━          feature:       ●━━━━●━━━━
               C1    C2                           C1    C2

        两个分支，代码不同                   main 包含了所有代码
```

### 5.6 合并后清理分支

```
PR 合并成功后：
    ↓
回到 Branches 页面
    ↓
找到已合并的 feature 分支 → 点击 🗑️ 删除
    ↓
刷新 → 只剩下 main 分支
    ↓
🎉 仓库保持整洁！
```

> 💡 **为什么要删除？** 功能分支的使命已经完成（代码已合入主干），保留它只会让分支列表越来越乱。保持"用后即删"的好习惯。

## 六、完整协作流程演练

把前面学到的所有步骤串起来，一个完整的"功能开发 → 合并"流程：

```
Step 1: 基于 main 创建 feature 分支
        仓库首页 → 分支选择器 → 输入 feature → 创建
    ↓
Step 2: 在 feature 分支上开发
        切换文件 → ✏️ 编辑 → 修改内容 → Commit changes
        （可以多次 Commit，每次完成一个小步骤）
    ↓
Step 3: 发起 Pull Request
        Pull requests 选项卡 → New pull request
        base: main ← compare: feature
        填写标题和描述 → Create pull request
    ↓
Step 4: Code Review（管理员操作）
        Files changed → 逐行审查
        → Comment / Request Changes / Approve
    ↓
Step 5: 合并
        管理员点击 Merge pull request → Confirm merge
    ↓
Step 6: 清理
        删除 feature 分支 → 保持分支树干净
    ↓
🎉 一个功能开发周期完成！
```

## 八、常见问题与排错指南

### 8.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| 在 feature 上改了代码，main 上也出现了 | 可能不小心在 main 上改了或者已经合并了 | 检查是否已合并；确认当前在哪个分支上操作 | 🔴 高 |
| PR 创建后看不到代码差异 | base 和 compare 选反了 | 确认 base=目标分支(main)，compare=源分支(feature) | 🔴 高 |
| Merge 按钮是灰色的 | 存在未解决的代码冲突（Conflicts） | 需要先解决冲突才能合并（后续课程会讲） | 🔴 高 |
| 切换分支后文件内容没变 | 两个分支上该文件内容恰好相同 | 正常现象！在某个分支上做了修改后就能看出差异 | 🟡 中 |
| 创建分支时代码不对 | 选错了 Source 分支 | 删除错误分支，重新选择正确的 Source 分支创建 | 🟡 中 |
| PR 被要求修改（Request Changes） | 代码有问题需要改 | 在 feature 分支上修改 → 提交新 Commit → PR 自动更新 | 🟡 中 |
| 删除分支后后悔了 | 分支已删除 | 可以恢复（后续课程会讲），但建议合并后再删 | 🟢 低 |
| 从 feature 创建的分支看不到 main 最新代码 | 这是正常行为 | 基于 feature 创建的分支只包含 feature + main(创建feature时)的代码 | 🟢 低 |

### 8.2 排查流程

```
分支或 PR 出问题了？
    │
    ├── "我在哪个分支上？"
    │       看仓库首页左上角分支选择器显示的当前分支名
    │
    ├── "我改的东西在哪个分支上？"
    │       切到不同分支看文件内容 → 确定修改在哪个分支
    │
    ├── "PR 为什么合并不了？"
    │       ├── 有冲突？→ 页面上会提示 "This branch has conflicts"
    │       ├── 还没人审核？→ 等管理员 Review
    │       └── 被 Request Changes？→ 按要求修改代码再提交
    │
    ├── "代码怎么不见了？"
    │       ├── 切错分支了？→ 切换到正确的分支
    │       ├── 分支被删了？→ 检查 Branches 列表
    │       └── 还没 Commit？→ 确认是否点击了 Commit changes
    │
    └── "基于 feature 创建的分支没有 main 最新代码？"
            这是正常的！基于哪个分支创建，就包含那个分支的代码。
            如果需要 main 最新代码，应该基于 main 创建新分支。
```

## 九、课后练习

### 练习一：创建和切换分支

**目标：** 熟练创建和切换分支。

**要求：**
- [ ] 在 hello-world 仓库中创建一个名为 `feature` 的新分支（基于 main）
- [ ] 切换到 feature 分支
- [ ] 确认当前分支是 feature（看分支选择器）
- [ ] 切换回 main 分支
- [ ] 再切换到 feature 分支

### 练习二：分支隔离性验证

**目标：** 亲身验证分支的隔离性——各自修改互不影响。

**要求：**
- [ ] 在 feature 分支上，编辑 README.md，在开头加一行 "## Feature 分支测试"
- [ ] Commit 这次修改
- [ ] 切换回 main 分支，打开 README.md —— **确认看不到这行内容** ✅
- [ ] 切换回 feature 分支 —— **确认那行内容还在** ✅
- [ ] 得出自己的结论并写下来

### 练习三：模拟紧急修复流程

**目标：** 复现"开发中遇到紧急 Bug"的真实工作流。

**场景：** 你正在 feature 上开发，突然需要在 main 上修复一个 Bug。

**要求：**
- [ ] 确保当前在 feature 分支（模拟正在开发功能）
- [ ] 切换到 main 分支
- [ ] 基于 main 创建 `bugfix` 分支
- [ ] 在 bugfix 分支上修改 README.md（模拟修复 Bug），Commit
- [ ] 创建 PR：bugfix → main
- [ ] 自己审核并 Merge 这个 PR
- [ ] 删除 bugfix 分支
- [ ] 切换回 feature 分支 —— **确认 feature 上的开发进度不受影响** ✅

### 练习四：创建并合并 PR

**目标：** 完整走一遍"功能开发 → PR → 合并"流程。

**要求：**
- [ ] 在 feature 分支上做 2-3 次有意义的修改（编辑 README.md）
- [ ] 每次修改都单独 Commit，写清楚的 Commit Message
- [ ] 创建 PR：feature → main
- [ ] 填写 PR 标题和描述
- [ ] 在 Files changed 中查看代码差异（绿色/红色标记）
- [ ] 点击 Merge pull request → Confirm merge
- [ ] 确认 main 分支上已出现 feature 的代码
- [ ] 删除 feature 分支

### 练习五：Code Review 体验

**目标：** 体验 Code Review 中的评论功能。

**要求：**
- [ ] 创建一个新的 feature 分支并做修改
- [ ] 创建 PR
- [ ] 进入 Files changed 页面
- [ ] 在至少 2 行代码上添加评论（模拟审查者的反馈）
- [ ] 点 Review changes → 分别尝试 Comment、Approve 两种结论
- [ ] 记录 Code Review 过程中你的感受——审查别人的代码和被人审查代码有什么不同？

## 十、课程小结

### 10.1 核心知识图谱

```
┌────────────────────────────────────────────────────────────┐
│           GitHub 是如何工作的 核心知识体系                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1️⃣  分支（Branch）                                        │
│      ├── 分支 = 存储库的不同版本                              │
│      ├── main / master = 主干分支（默认存在）                 │
│      ├── 创建分支 = 创建主干当时状态的快照                     │
│      ├── 刚创建时，feature 和 main 代码完全一样               │
│      └── 分支可以基于任何已有分支创建                          │
│                                                            │
│  2️⃣  分支的隔离性——核心特性                                  │
│      ├── feature 上的修改只在 feature 可见                   │
│      ├── 切换回 main → 看不到 feature 的修改                 │
│      ├── 各自分支上的修改互不影响                             │
│      └── 隔离性 = 多人协作的基础                              │
│                                                            │
│  3️⃣  分支工作流（动画演示的核心场景）                          │
│      ├── main → 创建 feature → 开发功能                     │
│      ├── 紧急 Bug → 切回 main → 创建 bugfix → 修复 → 合并   │
│      ├── 回到 feature → 继续开发 → 完成 → 合并               │
│      └── 结果：同时完成功能开发 + Bug修复 ✅                  │
│                                                            │
│  4️⃣  Pull Request（合并请求）                               │
│      ├── 本质：将更改从源分支合并进目标分支的提案               │
│      ├── base: main ← compare: feature                     │
│      ├── 会自动比较两个分支之间的代码差异                       │
│      └── 包含标题 + 描述 + 代码差异展示                       │
│                                                            │
│  5️⃣  Code Review（代码审查）                                │
│      ├── 管理员逐行审查代码改动                               │
│      ├── 可以在具体代码行添加评论                             │
│      ├── 三种结论：Comment / Approve / Request Changes       │
│      └── 审查通过后方可合并                                  │
│                                                            │
│  6️⃣  合并后清理                                             │
│      ├── PR 合并后 → 功能分支使命完成                         │
│      └── 删除已合并分支 → 保持分支树干净                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 10.2 一句话总结

> 分支让你安全地"试错"和并行开发，Pull Request 让代码在合并前经过审查——这两个机制构成了 GitHub 协作的基石。
