## 二、冲突解决——IDEA 的三路合并利器

### 2.1 制造冲突

```
场景重现（和之前课程一样）：
├── main 分支：colors.txt 添加 yellow + pink
├── feature 分支：colors.txt 添加 yellow + blue
└── 合并时第4、5行冲突！

在 IDEA 中执行合并：
Git 面板 → main 分支 → 右键 → Merge 'main' into 'feature'
→ IDEA 提示：Conflicts（冲突）
```

### 2.2 Accept Yours vs Accept Theirs

IDEA 提供了三种冲突处理方式的快捷按钮：

```
┌─────────────────────────────────────────────────────────┐
│  冲突弹窗中的三个按钮：                                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [Accept Yours]                                         │
│  ├── 完全保留当前分支（feature）的版本                   │
│  ├── 对方的修改全部丢弃                                  │
│  └── 适用：确定自己的版本是对的                          │
│                                                         │
│  [Accept Theirs]                                        │
│  ├── 完全采用合并分支（main）的版本                      │
│  ├── 自己的修改全部丢弃                                  │
│  └── 适用：确定对方的版本是对的                          │
│                                                         │
│  [Merge...]                                             │
│  ├── 打开三路合并窗口，手动逐行选择                      │
│  ├── 两边都保留 / 只保留一边 / 自己改写                  │
│  └── 适用：不确定该用谁的，需要精细控制 ← 推荐           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.3 Merge 三路合并窗口详解

> 这是 IDEA 冲突解决中**最强大的功能**。和手动编辑 `<<<<<<< ======= >>>>>>>` 标记完全不同，IDEA 提供了可视化的三路合并窗口。

```
┌─────────────────────────────────────────────────────────┐
│          IDEA 三路合并窗口布局                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────────────┐    ┌──────────────────┐           │
│  │   左侧面板         │    │   右侧面板         │           │
│  │   feature 分支     │    │   main 分支        │           │
│  │   (你的版本)       │    │   (对方的版本)      │           │
│  │                   │    │                   │           │
│  │   yellow           │    │   yellow           │           │
│  │   blue             │    │   pink             │           │
│  └──────────────────┘    └──────────────────┘           │
│              ↓                    ↓                      │
│  ┌──────────────────────────────────────────────┐       │
│  │          中间面板（最终结果）                    │       │
│  │                                              │       │
│  │   yellow                                      │       │
│  │   blue     ← 从左侧 Accept 过来的              │       │
│  │   pink     ← 从右侧 Accept 过来的              │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**操作方式：**

```
每行冲突代码都可以独立操作：

操作按钮：
├── [Accept]（箭头 →）  = 把这行加到最终结果中
├── [Discard]（叉号 ✕） = 放弃这行，不要它
└── 手动编辑             = 在中间面板直接修改代码

常见组合：
┌─────────────────────────────────────────────────────┐
│                                                     │
│  两边都要：                                          │
│  左侧点 Accept → 右侧点 Accept → 两行都进中间        │
│                                                     │
│  只要左边：                                          │
│  左侧点 Accept → 右侧点 Discard                      │
│                                                     │
│  只要右边：                                          │
│  左侧点 Discard → 右侧点 Accept                      │
│                                                     │
│  自己改写：                                          │
│  两侧都 Discard → 在中间面板手写新代码               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

```
完整流程：

Step 1: IDE A 提示冲突 → 点击 [Merge...]

Step 2: 三路合并窗口打开
        ├── 左侧 = 你的版本（feature）
        ├── 右侧 = 对方的版本（main）
        └── 中间 = 最终结果（你决定）

Step 3: 逐行处理冲突
        对每一行选择 Accept（保留）或 Discard（放弃）

Step 4: 所有冲突行处理完毕
        中间面板出现绿色对勾 ✅

Step 5: 点击 [Apply] 按钮
        → 冲突解决完毕！

Step 6: Push 到远端
```

```
┌─────────────────────────────────────────────────────────┐
│  IDEA 三路合并 vs 手动编辑冲突标记                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  手动编辑：                                              │
│  ├── 看到 <<<<<<< ======= >>>>>>> 标记                  │
│  ├── 手动删除标记行                                      │
│  ├── 手动决定保留哪行                                    │
│  └── 容易误删、漏删标记                                  │
│                                                         │
│  IDEA 三路合并：                                         │
│  ├── 左右对比，一目了然                                  │
│  ├── 按钮操作，不会漏删标记                              │
│  ├── 可以看到两边完整的上下文                            │
│  └── 大幅降低出错概率 ✅                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 三、Update Project——Merge vs Rebase 的 Pull 策略

### 3.1 问题场景：Push Rejected

> 这是多人协作中的高频场景：你和同事都在 main 分支上开发，同事先 push 了，你后 push 时被拒绝。

```
┌─────────────────────────────────────────────────────────┐
│          Push Rejected 的场景                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  同事赵工：                                              │
│  修改代码 → Commit → Push 到远端 ✅                     │
│                                                         │
│  你：                                                    │
│  修改代码 → Commit → Push 到远端                        │
│  → ❌ Push Rejected！                                   │
│  "远端有你本地没有的新提交，请先拉取"                    │
│                                                         │
│  此时你需要先 Update Project（拉取远端更新）             │
│  然后再 Push                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

IDEA 的 Update Project 提供了两种拉取策略：

```
IDEA 菜单：Git → Update Project（或工具栏蓝色箭头 ▼）

弹出选项：
┌──────────────────────────────────────────────┐
│  Update Project                              │
│                                              │
│  ○ Merge                                     │
│    等价于：git pull                           │
│                                              │
│  ○ Rebase                                    │
│    等价于：git pull --rebase                  │
│                                              │
│  [OK]    [Cancel]                            │
└──────────────────────────────────────────────┘
```

### 3.2 Merge 方式（git pull）

```
执行 git pull（Merge 方式）：

提交历史的变化：
变前：                      变后：
远端：A → B → C（同事的）    远端：A → B → C（同事的）
本地：A → B → D（你的）      本地：A → B → C（同事的）
                                        ↘ D → M（合并提交）
                                        
产生了新的 Merge 提交 "M" ❌

结果：
├── 代码正确合并了 ✅
├── 但多了一次 Merge 提交 ❌
├── 提交历史出现分叉 ❌
└── 如果经常发生，历史树会非常杂乱 ❌
```

### 3.3 Rebase 方式（git pull --rebase）

```
执行 git pull --rebase：

提交历史的变化：
变前：                      变后：
远端：A → B → C（同事的）    远端：A → B → C（同事的）
本地：A → B → D（你的）      本地：A → B → C（同事的）→ D'（你的）
                                                    
你的提交 D 被"搬"到了 C 之后 ✅

结果：
├── 代码正确合并了 ✅
├── 没有额外的 Merge 提交 ✅
├── 提交历史是一条直线 ✅
├── 远端的 C 被你拉取到了本地 ✅
└── 你的 D 在 C 之后，顺序正确 ✅
```

```
┌─────────────────────────────────────────────────────────┐
│  关键区别：                                              │
│                                                         │
│  git pull（Merge）：                                    │
│  把你的提交和远端的提交合并 → 产生一个新的合并提交       │
│                                                         │
│  git pull --rebase（Rebase）：                          │
│  先把远端的提交拉下来，再把你的提交"搬"到最新之后       │
│  → 不产生额外的合并提交                                 │
│                                                         │
│  ⚠️ 与分支 Rebase 的重要区别：                          │
│     git pull --rebase 之后不需要 Force Push！            │
│     因为它只是把你的本地未推送提交"搬"了个位置           │
│     远端的历史没有被改变，所以普通 Push 就够             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.4 推荐选择

```
选择指南：

日常开发中 → 推荐 Rebase（git pull --rebase）
├── 保持提交历史干净
├── 不需要 Force Push
└── 不会产生无意义的 Merge 提交

多人协作同一分支 → Rebase 更合适
├── 每个人都在 main 上开发
└── 频繁 pull 如果用 Merge 会产生大量合并提交

新手入门 → 可以先用 Merge
├── 理解起来更直观
└── 遇到问题更容易排查
```

## 四、Rollback——丢弃本地修改

> Rollback = GitHub Desktop 中的 Discard。把本地文件的修改**丢弃**，恢复到与本地仓库一致的状态。

```
┌─────────────────────────────────────────────────────────┐
│  Rollback 操作步骤：                                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  场景：你改了一些代码，但改错了，想全部撤销               │
│                                                         │
│  Step 1: 打开 Commit 面板（Ctrl+K）                      │
│          Changes 区域列出了所有有改动的文件               │
│                                                         │
│  Step 2: 双击文件可查看具体改动内容                      │
│                                                         │
│  Step 3: 选中不想要的文件                               │
│          右键 → Rollback                                │
│                                                         │
│  Step 4: IDEA 再次确认                                  │
│          点击 "Rollback" → 文件的改动被丢弃             │
│                                                         │
│  结果：文件恢复到最近一次 commit 的状态 ✅                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Rollback 的本质：

git checkout -- <文件名>

→ 用本地仓库中的版本覆盖工作区中的文件
→ 未暂存的改动直接丢失
→ 已暂存的改动也直接丢失（因为 IDEA 的 Rollback 比较"暴力"）

⚠️ 注意：Rollback 不可逆！丢弃的内容无法恢复！
         确保你真的不需要这些改动了再操作
```

## 五、Reset——撤回提交的四种模式

### 5.1 操作步骤

```
┌─────────────────────────────────────────────────────────┐
│  Reset 操作：                                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: 打开 Git 面板 → Log 选项卡                     │
│          看到完整的提交历史树                             │
│                                                         │
│  Step 2: 找到你想 Reset 到的目标提交                     │
│          右键 → "Reset Current Branch to Here..."       │
│                                                         │
│  Step 3: 选择 Reset 模式                                │
│          ┌──────────────────────────────────────┐       │
│          │  Reset                                │       │
│          │                                       │       │
│          │  ○ Soft                               │       │
│          │  ○ Mixed  (默认，推荐)                 │       │
│          │  ○ Hard                               │       │
│          │  ○ Keep                               │       │
│          └──────────────────────────────────────┘       │
│                                                         │
│  Step 4: 点击 "Reset"                                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 Soft / Mixed / Hard / Keep 详解

> 四种模式的区别在于：**撤回提交后，改动保留在哪里？**

```
假设你有 3 次提交：
Commit 1 → Commit 2 → Commit 3 (HEAD)
你想 Reset 回 Commit 1

        Commit 1  →  Commit 2  →  Commit 3
           ↑                         ↑
       目标位置                   当前位置(HEAD)
```

| 模式 | 命令 | 撤回提交 | 撤回暂存区 | 撤回工作区 | 说明 |
|---|---|---|---|---|---|
| **Soft** | `git reset --soft` | ✅ 撤回 | ❌ 保留 | ❌ 保留 | 改动留在暂存区，可以立即重新 commit |
| **Mixed** | `git reset --mixed`（默认） | ✅ 撤回 | ✅ 撤回 | ❌ 保留 | 改动留在工作区（文件还在磁盘上），需要重新 add + commit |
| **Hard** | `git reset --hard` | ✅ 撤回 | ✅ 撤回 | ✅ 撤回 | **全部丢弃！**工作区恢复为目标提交的状态 |
| **Keep** | `git reset --keep` | ✅ 撤回 | ✅ 撤回 | 部分保留 | 和 Hard 的区别：未提交的本地改动会被保留 |

```
┌─────────────────────────────────────────────────────────┐
│          四种模式的直观对比                               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  原始状态：3 次提交，Commit 2 和 3 有改动               │
│  Reset 到 Commit 1 后：                                  │
│                                                         │
│  Soft：                                                 │
│  ├── 提交历史：只剩 Commit 1                             │
│  ├── 暂存区：Commit 2 和 3 的改动都在（绿色）            │
│  └── 工作区：没变化                                      │
│                                                         │
│  Mixed（默认，最推荐）：                                 │
│  ├── 提交历史：只剩 Commit 1                             │
│  ├── 暂存区：清空                                        │
│  └── 工作区：Commit 2 和 3 的改动在磁盘上（蓝色）        │
│      → 可以重新 add → commit                            │
│                                                         │
│  Hard（危险！）：                                        │
│  ├── 提交历史：只剩 Commit 1                             │
│  ├── 暂存区：清空                                        │
│  └── 工作区：恢复到 Commit 1 的状态                      │
│      → Commit 2 和 3 的改动全部丢失！                   │
│                                                         │
│  Keep（比 Hard 温和）：                                  │
│  ├── 和 Hard 一样，但会保留未提交的本地改动              │
│  └── 如果本地有未提交的修改，Hard 会丢弃，Keep 会保留   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

> ⚠️ **推荐使用 Mixed（IDEA 默认）**：它撤回提交但不丢失代码，你随时可以重新整理后再次提交。Hard 只有在确定不要那些代码时才用。

## 六、Revert——安全的"反向提交"

> Revert 和 Reset 不同：Revert **不删除旧提交**，而是创建一个"反向的新提交"来抵消改动。

```
┌─────────────────────────────────────────────────────────┐
│  Revert 操作步骤：                                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: Git 面板 → Log 选项卡                          │
│          找到你想撤销的那次提交                           │
│                                                         │
│  Step 2: 右键该提交 → "Revert Commit"                   │
│                                                         │
│  Step 3: IDEA 自动创建一个新的 Revert 提交               │
│          message: "Revert 'xxx'"                        │
│          这次提交的内容 = 你想撤销的改动的反向操作        │
│                                                         │
│  Step 4: Push 到远端                                    │
│          右键分支 → Push                                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Reset vs Revert 的核心区别：

Reset：                             Revert：
Commit 1 → Commit 2 → Commit 3     Commit 1 → Commit 2 → Commit 3
                ↓ DELETE                        ↓
         Commit 3 "消失了"                     Commit 4（反向操作）
                                               
历史被改写 ❌                       历史保留了 ✅
适合未 Push 的本地提交              适合已 Push 到远端的提交
                                   安全！不影响同事
```

> ⚠️ **黄金法则**：已 Push 的提交用 Revert，未 Push 的提交用 Reset。永远不要在共享分支上 Reset 已推送的提交。

## 七、Amend——修正最近一次提交

### 7.1 未 Push 的 Amend

> Amend 让你修改**最近一次 commit** 的内容或 message，而不产生新的提交。

```
┌─────────────────────────────────────────────────────────┐
│  Amend 操作步骤（未 Push）：                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  场景：刚提交了 "hello tech trump"                       │
│        发现 "trump" 拼错了，应该是 "trumpet"             │
│                                                         │
│  Step 1: 在编辑器中把代码改正确                          │
│          "trump" → "trumpet"                            │
│                                                         │
│  Step 2: 打开 Commit 面板（Ctrl+K）                      │
│                                                         │
│  Step 3: 勾选 ☑ Amend                                   │
│          ├── Commit Message 区域会显示上一次的 message   │
│          ├── 修改 message 为正确的内容                   │
│          └── 勾选上改动的文件                            │
│                                                         │
│  Step 4: 点击 "Amend Commit"                            │
│          → 最近一次提交被更新！没有产生新提交 ✅          │
│                                                         │
│  Step 5: Push 到远端                                    │
│          （因为从未 Push 过，所以普通 Push 就行）        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 7.2 已 Push 的 Amend 限制

```
┌─────────────────────────────────────────────────────────┐
│  ⚠️ 已 Push 到远端的 Amend 会被拒绝！                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  如果你 Amend 了一个已经 Push 到远端的提交：             │
│                                                         │
│  IDEA 本地可以 Amend ✅                                  │
│  但 Push 时 → ❌ Push Rejected！                        │
│                                                         │
│  原因：本地历史被改写，和远端不一致                       │
│                                                         │
│  要强行推送 → 需要 Force Push                           │
│  但这会覆盖远端的提交历史                                │
│                                                         │
│  ⚠️ 建议：已 Push 的提交不要 Amend！                    │
│     改用 Revert 来安全地撤销                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 八、Squash——合并多个提交

> Squash 把多个 commit 压缩成一个，让提交历史更干净。

```
┌─────────────────────────────────────────────────────────┐
│  Squash 操作步骤：                                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: Git 面板 → Log 选项卡                          │
│          看到提交历史树                                   │
│                                                         │
│  Step 2: 按住 Ctrl（Mac: Cmd）选中要合并的多个提交       │
│          例如：选中 3 个零碎的提交                        │
│                                                         │
│  Step 3: 右键 → "Squash Commits"                        │
│                                                         │
│  Step 4: 填写合并后的 Commit Message                    │
│          点击 OK                                         │
│                                                         │
│  Step 5: 3 个提交变成 1 个！✅                            │
│          → Push 到远端                                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Squash 的效果：

Squash 前：                        Squash 后：
Commit 1: "fix typo"               Commit: "实现用户登录功能"
Commit 2: "add comment"              ↑
Commit 3: "final fix"           把3个碎提交合并成了1个

⚠️ 注意事项：
├── 未 Push 的提交 → Squash 后普通 Push 即可
└── 已 Push 的提交 → 不要 Squash！（会改写历史，需要 Force Push）
```

## 九、Cherry Pick——拣选提交

> Cherry Pick 让你**只把某几个 commit 拷贝到当前分支**，而不是合并整个分支。

```
┌─────────────────────────────────────────────────────────┐
│  Cherry Pick 操作步骤：                                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: 先切换到「接收」的分支                         │
│          例如：Checkout feature 分支                     │
│                                                         │
│  Step 2: Git 面板 → Log 选项卡                          │
│          找到 main 分支的提交历史                         │
│          （在左侧分支树中双击 main）                     │
│                                                         │
│  Step 3: 选中你想要"摘"过来的提交                        │
│          Ctrl+点击可多选                                 │
│          例如：只选 Commit 4 和 Commit 5                 │
│                                                         │
│  Step 4: 右键 → "Cherry-Pick"                           │
│          → 选中的提交被拷贝到 feature 分支               │
│                                                         │
│  Step 5: 如果有冲突 → 用三路合并窗口解决                 │
│          解决后点击 Commit                               │
│                                                         │
│  Step 6: 完成！两个 commit 出现在 feature 分支 ✅        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Cherry Pick 的典型应用场景：

场景1："我只需要你的那个 bug 修复"
main 分支有 10 个新提交
但你只想要其中的 bug 修复（Commit 7）
→ Cherry Pick Commit 7 到你的分支

场景2："我做的工具函数可以给其他分支用"
feature-A 分支上写了一个通用工具函数
feature-B 分支也需要这个函数
→ Cherry Pick 那个函数的提交到 feature-B
```

## 十、Stash 与 Shelf——暂存更改

### 10.1 Stash——Git 标准暂存

> 当你代码写到一半需要切换分支时，用 Stash 把未完成的代码"藏起来"。

```
┌─────────────────────────────────────────────────────────┐
│  Stash 操作步骤：                                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  存起来：Stash                                          │
│  Step 1: 在 Commit 面板或项目文件树右键                 │
│          → Git → Stash Changes                          │
│                                                         │
│  Step 2: 填写 Stash Message                             │
│          例如："保存两个更改——用户模块开发中"            │
│                                                         │
│  Step 3: 点击 "Create Stash"                            │
│          → 工作区变干净了！可以随意切换分支              │
│                                                         │
│  取出来：Apply vs Pop                                   │
│  Step 1: Git 面板 → Stash 选项卡                        │
│          看到之前保存的 Stash                            │
│                                                         │
│  Step 2: 右键 Stash → 选择取出方式：                    │
│          ├── Apply Stash  = 取出改动，保留 Stash        │
│          └── Pop Stash    = 取出改动，删除 Stash        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Stash 的局限性：

⚠️ Git Stash 是"全量"的——一次 Stash 会把所有文件的改动都存起来
    你不能选择"只 Stash 这个文件，不 Stash 那个文件"

这就是 IDEA 发明 Shelf 的原因 ↓
```

### 10.2 Shelf——IDEA 独有的选择性暂存

> Shelf 是 IDEA 独有的功能（不是 Git 标准命令），弥补了 Stash 不能选择文件的缺陷。

```
┌─────────────────────────────────────────────────────────┐
│  Shelf 操作步骤：                                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  搁置某个文件（选择性）：                                 │
│  Step 1: Commit 面板中，选中你想 shelf 的文件            │
│          右键 → "Shelve Changes"                        │
│                                                         │
│  Step 2: Shelf 窗口打开                                  │
│          你可以勾选/取消勾选要搁置的文件                 │
│          → 这是 Stash 做不到的！                         │
│                                                         │
│  Step 3: 点击 "Shelve"                                  │
│          → 只有你选中的文件被搁置                        │
│          → 其他文件的改动保持不变                        │
│                                                         │
│  取出搁置：Unshelve                                     │
│  Step 1: Git 面板 → Shelf 选项卡                        │
│          看到之前搁置的文件                              │
│                                                         │
│  Step 2: 右键 → "Unshelve"                              │
│          → 文件改动被恢复 ✅                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
Stash vs Shelf 对比：

| 维度 | Stash | Shelf |
|------|-------|-------|
| 是否 Git 标准命令 | ✅ 是 | ❌ IDEA 独有 |
| 是否可选择文件 | ❌ 全部或全不 | ✅ 可勾选 |
| 其他 Git 工具能看到吗 | ✅ 能 | ❌ 只能在 IDEA 中 |
| 适用场景 | 快速全量暂存 | 精细选择性暂存 |
| 推荐 | 临时切换分支 | 只想藏起部分文件的改动 |
```

## 十六、常见问题与排错指南

### 16.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| **Push Rejected** | 远端有新提交，本地落后 | Update Project（选 Rebase）→ 再 Push | 🔴 高 |
| **Amend 后 Push 被拒** | 已 Push 的提交被 Amend 改写 | 不要 Amend 已 Push 的提交；如果必须改，用 Force Push | 🔴 高 |
| **Reset Hard 后代码丢了** | Hard 模式丢弃了工作区改动 | 用 `git reflog` 找回；以后优先用 Mixed | 🔴 高 |
| **Squash 后 Push 被拒** | 已 Push 的提交被 Squash | 未 Push 的才能 Squash；已 Push 的用 Force Push（慎用） | 🟡 中 |
| **Stash Apply 后冲突** | Stash 的内容和当前代码冲突 | 手动解决冲突（和 Merge 冲突一样处理） | 🟡 中 |
| **Cherry Pick 出现冲突** | 拣选的提交和当前分支代码冲突 | 用 IDEA 三路合并窗口解决 → Commit | 🟡 中 |
| **Reset/Revert 灰色不可用** | 当前分支被保护或文件只读 | 检查分支权限；确认不在 Detached HEAD 状态 | 🟢 低 |
| **Tag 推不上去** | Push 时没勾选 Push Tags | Push 面板左下角勾选 ☑ Push Tags | 🟢 低 |

### 16.2 Reset 模式选择决策

```
你需要撤回提交，该用哪个模式？

未 Push 到远端？
├── 想保留改动，重新整理后提交 → Mixed（默认，推荐）
├── 改动已经暂存好了，想直接重新提交 → Soft
└── 确定不要这些代码了 → Hard

已 Push 到远端？
└── 不要用 Reset！→ 用 Revert
    创建反向提交，安全地撤销
```

```
│  5️⃣  IDEA 独有优势                                         │
│      ├── 三路合并窗口（比手动编辑冲突标记优雅太多）         │
│      ├── Shelf 选择性暂存（Git Stash 做不到）              │
│      └── 可视化历史树 + 右键操作（比命令行直观）           │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 17.2 一句话总结

> **IDEA 的 Git 进阶操作让复杂的事情变简单：三路合并窗口让冲突解决不再痛苦，Reset/Revert/Amend/Squash 让你自由掌控提交历史，Cherry Pick 让你按需取用提交，Git Blame 和 Show History 让你随时追溯代码的来龙去脉。掌握这些，你就是团队里的 Git 高手。**
