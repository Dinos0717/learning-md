## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经掌握了：

- **分支操作** —— 创建、切换、基于任意分支创建新分支
- **PR 合并** —— 通过 Pull Request 将功能分支合并到主干
- **Git 四个分区** —— 工作区、暂存区、本地存储库、远端存储库
- **进阶操作** —— Stash、Reset、Revert、Amend、Tag

但关于"合并"这件事，你还只知道一种方式——点击 PR 的 Merge 按钮。实际上，Git 提供了**三种完全不同的合并策略**，各有优劣，适用于不同场景。选错了策略，可能导致历史混乱甚至丢失同事的代码。

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| Merge、Squash、Rebase 有什么区别？ | 三种合并策略的原理、优缺点、适用场景 |
| 什么情况下可以不用 PR，直接本地合并？ | 把主干同步到功能分支时，可以本地合并 |
| Rebase 为什么需要 Force Push？ | 变基改变了历史，远端历史不一致，必须强制覆盖 |
| 能不能只把某几个 Commit 合并到另一个分支？ | Cherry Pick——拣选你需要的提交 |
| 在 GitHub 网页端合并 PR 时该选哪个策略？ | 普通 Merge（推荐）或 Squash，**不要选 Rebase** |

> ⚠️ **关键认知**：合并策略的选择不是"哪个更好"的问题，而是"哪个更合适"。Merge 最安全但历史稍乱，Squash 最简洁但丢失细节，Rebase 最干净但要求 Force Push。理解各自的取舍，才能在不同场景做出正确选择。

### 1.3 本课要掌握的目标

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  目标一：理解并区
分三种合并策略                             │
│         ├── Merge：保留所有历史 + 生成 merge commit           │
│         ├── Squash Merge：压缩所有 commit 为一个              │
│         └── Rebase：变更根基，线性历史，需 Force Push         │
│                                                             │
│  目标二：能在正确的场景选择正确的策略                              │
│         ├── 多人协作分支 → Merge                             │
│         ├── 想保持历史干净 → Squash                          │
│         └── 个人分支整理 → Rebase                            │
│                                                             │
│  目标三：掌握 Cherry Pick 选择性合并                            │
│         ├── 理解 Cherry Pick 的应用场景                       │
│         └── 在 Desktop 中拣选特定 commit 到另一个分支          │
│                                                             │
│  目标四：知道什么时候绝对不能 Force Push                          │
│         └── 多人协作分支 = 永远不 Force Push                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、分支操作复习

### 2.1 创建分支的两种方式

**方式一：在 GitHub 网页端创建，同步回本地**

```
GitHub 仓库首页 → 点击分支选择器 → View all branches
    ↓
点击 "New branch" → 输入分支名 → 选择基于哪个分支创建
    ↓
点击 "Create new branch"
    ↓
回到 Desktop → 点击 "Fetch origin" → 新分支出现了
    ↓
点击分支名即可切换过去
```

**方式二：在 Desktop 中创建，推送到远端**

```
Desktop → 点击分支选择器 → "New branch"
    ↓
输入分支名 → 选择基于哪个分支创建
    ↓
点击 "Create branch"
    ↓
此时分支只存在于本地！→ 点击 "Publish branch" → 推送到远端
    ↓
验证：刷新网页端分支列表 → 新分支出现了 ✅
```

```
总结：
    网页建 → Fetch → 本地有
    Desktop 建 → Publish → 远端有

    两种方式等价，选哪个取决于你当时在用什么工具。
```

### 2.2 删除分支

**方式一：在网页端删除**

```
GitHub 分支列表 → 每个分支右侧的 🗑️ 按钮 → 点击删除
    ↓
⚠️ 只删除了远端！本地分支还在。
    还需要在 Desktop 中再删一次本地分支。
```

**方式二：在 Desktop 中删除（推荐）**

```
Desktop → 分支选择器 → 右键目标分支 → Delete
    ↓
弹出确认框 → 勾选 "Delete the remote branch" →
    ├── ✅ 勾选 = 同时删除本地 + 远端（推荐！）
    └── ☐ 不勾选 = 只删除本地
    ↓
点击 Delete
    ↓
🎉 本地和远端同时删除干净！
```

> 💡 **推荐 Desktop 一键删除**：省得在网页端和本地各删一次。

## 三、PR 合并复习

通过 Pull Request 合并是最正式的合并方式，之前课程已详细讲过，这里快速复习：

```
GitHub 网页 → Pull requests 选项卡 → New pull request
    ↓
选择合并方向：base: main ← compare: feature
    ↓
查看代码差异 → Create pull request
    ↓
团队 Code Review → Approve
    ↓
点击 "Merge pull request" → Confirm merge
    ↓
🎉 feature 合并进 main！
```

> 💡 PR 合并**本质上也是 Merge**——只是多了一层 Code Review 的流程。

## 四、本地合并——不通过 PR 直接合并分支

### 4.1 什么时候需要本地合并

并不是所有合并都需要走 PR 流程。以下场景适合**本地合并**：

```
场景一：把主干的更新同步到功能分支

    main 分支有新的提交（同事合入了新功能）
        ↓
    你的 feature 分支需要这些更新
        ↓
    把 main 合并进 feature（不是 feature 合进 main！）
        ↓
    这种"反向合并"不需要 PR，本地合并即可

场景二：个人分支之间合并

    你的 feature-a → feature-b
    两个都是你自己的分支，不需要审核
```

```
关键判断：

    合并方向                    需要 PR 吗？
    feature → main（功能合入主干）    ✅ 需要 PR
    main → feature（主干同步到功能）  ❌ 不需要，本地合并即可
```

### 4.2 操作步骤

**场景：把 main 分支上的新提交合并到 feature 分支。**

```
Desktop → 切换到**接受合并的分支**（这里是 feature）
    ↓
点击分支选择器 → 最下方 "Choose a branch to merge into feature"
    ↓
选择 main 分支
    ↓
Desktop 显示："这个操作将把 3 个提交从 main 合并进 feature"
    ↓
选择合并策略（三个选项）：
    ├── Create a merge commit  ← 默认（普通 Merge）
    ├── Squash and merge       ← 压缩合并
    └── Rebase                 ← 变基
    ↓
点击按钮 → 合并完成！
```

> ⚠️ **方向别搞反**：是 main 合进 feature，不是 feature 合进 main。判断标准是——**你当前在哪个分支，就合到哪个分支**。

## 五、三种合并策略详解

> 这是本节课的核心内容。我们用同一个场景来对比三种策略的效果。

### 5.1 初始化场景

```
初始状态：

    main 分支：    ●━━━━●━━━━●━━━━●
                 C1    C3    C4    C5

    feature 分支： ●━━━━●
                 C1    C2

    main 比 feature 多了 C3、C4、C5 三个提交
    现在要把 main 合并进 feature
```

```
可视化（图例）：

    main:     ●━━●━━●━━●
             C1  C3  C4  C5

    feature:  ●━━●
             C1  C2
                    ↑ 当前所在分支（接受合并）
```

### 5.2 Merge（普通合并）——最常用的方式

**命令：** `git merge main`

**原理：**

```
Merge 做了什么：
    ① 把 main 分支上的 C3、C4、C5 三个提交合并过来
    ② 创建一个新的 "merge commit"（合并提交）
    ③ 保留所有分支的完整历史
```

**合并后的历史图：**

```
    main:     ●━━●━━●━━●
             C1  C3  C4  C5
                  ↘       ↘
    feature:  ●━━●━━━━━━━━●━━●
             C1  C2          M  ← Merge Commit
                         (C3+C4+C5)

    特点：
    ├── C3、C4、C5 三个提交都保留了
    ├── 新生成一个 M（merge commit）
    ├── 可以看到是从 main 合并过来的（有线连着）
    └── 历史完整可追溯
```

**Desktop 操作：**

```
切换到 feature 分支 → 选择 main 合并 → 点击 "Create a merge commit"
    ↓
查看 History：
    ├── C3、C4、C5 三个提交都在
    ├── 多了一个 "Merge branch 'main' into feature" 的提交
    └── 分支历史保留完整 ✅
```

**评价：**

| 优点 | 缺点 |
|---|---|
| ✅ 完整保留所有 commit 历史 | ❌ 多一条 merge commit 记录 |
| ✅ 分支可溯源，清楚知道代码从哪来 | ❌ 历史树看起来可能比较"乱" |
| ✅ 不需要 Force Push | |
| ✅ 最安全，多人协作首选 | |

### 5.3 Squash Merge（压缩合并）——清爽的提交历史

**命令：** `git merge --squash main`

"Squash" 的英文意思是**挤压、压缩**。

**原理：**

```
Squash Merge 做了什么：
    ① 把 main 分支上的 C3、C4、C5 三个提交的内容全部拿出来
    ② 把它们压缩成一个提交（C6）
    ③ 把 C6 放进 feature 分支
    ④ C3、C4、C5 的原始记录不会出现在 feature 的历史中
```

**合并后的历史图：**

```
    main:     ●━━●━━●━━●
             C1  C3  C4  C5   ← C3、C4、C5 在 main 上还在

    feature:  ●━━●━━━━━━━━━━●
             C1  C2          C6  ← 只有一个压缩后的提交
                              ↑ Squash Commit
                              (C3+C4+C5 的所有改动合并为一个)
```

**Desktop 操作：**

```
切换到 feature 分支 → 选择 main 合并 → 选择 "Squash and merge"
    ↓
查看 History：
    ├── C3、C4、C5 没有出现（被压缩了）
    ├── 只有一个新的 commit："Squash commit"
    └── 历史非常干净 ✅
```

**评价：**

| 优点 | 缺点 |
|---|---|
| ✅ 历史记录非常干净 | ❌ 丢失了 C3、C4、C5 的独立记录 |
| ✅ 只用一条 commit 代表所有改动 | ❌ 不好追溯每次改动的细节 |
| ✅ 不需要 Force Push | |
| ✅ PR 合入主干时常用 | |

### 5.4 Rebase（变基）——最干净的线性历史

**命令：** `git rebase main`

"Rebase" 的中文翻译是**变基**，即**变更根基**。

**原理——两层理解：**

```
第一层理解（从效果看）：
    把 main 分支的更新"合并"进 feature

第二层理解（从本质看）：
    把 feature 分支的"根基"从原来的地方，变更到 main 分支的最新位置
```

**合并后的历史图：**

```
    Rebase 前：                          Rebase 后：

    main:  ●━━●━━●━━●           main:  ●━━●━━●━━●
          C1  C3  C4  C5              C1  C3  C4  C5
                                                 │
    feature: ●━━●                feature:        ●━━●
            C1  C2                              C2' C3' C4' C5'
            ↑ 根基在 C1                          ↑ C2 变基成了 C2'
                                                挂到了 C5 下面

    关键变化：
    ├── feature 的根基从 C1 移到了 C5
    ├── C2 被"重新播放"到 C5 之后，变成了 C2'
    ├── 没有创建 merge commit
    └── 所有提交在一条直线上（线性历史）
```

**Desktop 操作：**

```
切换到 feature 分支 → 选择 main 合并 → 选择 "Rebase"
    ↓
⚠️ Desktop 弹出警告："变基操作后必须强制推送到 GitHub"
    ↓
点击 "Begin rebase"
    ↓
查看 History：
    ├── 所有提交在一条直线上 ✅
    ├── 没有 "Merge branch..." 的提交 ✅
    └── 但必须 Force Push 才能推送到远端 ⚠️
    ↓
点击 "Force push origin" → 确认 → 推送成功
```

**评价：**

| 优点 | 缺点 |
|---|---|
| ✅ 历史完全线性，最干净 | ❌ **必须 Force Push！** |
| ✅ 没有多余的 merge commit | ❌ 不能在多人协作分支上使用 |
| ✅ 提交记录一目了然 | ❌ 改变了提交的 ID（C2 → C2'） |

> ⚠️ **Rebase 的核心取舍**：用"改变历史"换取"更干净的历史"。这个代价就是 Force Push——所以只能在自己一个人的分支上用。

### 5.5 三种策略完整对比表

```
┌─────────────────────────────────────────────────────────────┐
│           Merge vs Squash vs Rebase 完整对比                  │
├──────────────┬──────────────┬──────────────┬────────────────┤
│              │ Merge         │ Squash        │ Rebase         │
├──────────────┼──────────────┼──────────────┼────────────────┤
│ 历史记录      │ 完整保留       │ 压缩成一条     │ 线性，无分支     │
│ 产生 merge    │ ✅ 产生一个     │ ✅ 产生一个     │ ❌ 不产生        │
│ commit?      │              │              │                │
│ 原始 commit   │ ✅ 全保留       │ ❌ 被压缩消失   │ ⚠️ 重建         │
│ 是否保留      │              │              │ (C2 → C2')     │
│ 能否 Force    │ 不需要        │ 不需要        │ **必须**        │
│ Push 推送    │              │              │ Force Push ⚠️  │
│ 历史可溯源性  │ ⭐⭐⭐ 最好    │ ⭐ 最差        │ ⭐⭐ 中等       │
│ 历史干净度    │ ⭐ 最乱        │ ⭐⭐⭐ 干净    │ ⭐⭐⭐ 最干净   │
│ 安全性        │ ⭐⭐⭐ 最安全  │ ⭐⭐⭐ 安全    │ ⭐ 有风险       │
│ 适用场景      │ 多人协作、     │ PR合入主干、   │ **个人分支**     │
│              │ 主干合并       │ 清理历史       │ 整理提交历史     │
│ 多人协作分支   │ ✅ 推荐！      │ ✅ 可以用      │ ❌ 绝对不行！    │
│ PR 合入主干   │ ✅ 推荐！      │ ✅ 可以用      │ ❌ 不推荐        │
│ 个人分支整理   │ ✅ 可以用      │ ✅ 可以用      │ ✅ 最推荐        │
└──────────────┴──────────────┴──────────────┴────────────────┘
```

### 5.6 选型决策指南

```
你正在合并代码，该选哪个策略？

    是不是要合并到多人协作分支（如 main）？
        ├── 是 → 用 Merge（或多一个步骤用 Squash）
        │       ❌ 绝对不要用 Rebase
        │
        └── 否（只是整理自己的个人分支）
                ↓
            你想要什么效果？
                ├── 保留完整历史 → Merge
                ├── 只想用一个提交表示所有改动 → Squash
                └── 想要最干净的线性历史 → Rebase（需 Force Push）
```

```
一句话口诀：

    "多人用 Merge，个人用 Rebase，想简洁用 Squash"
```

## 六、Cherry Pick——选择性合并提交

### 6.1 Cherry Pick 是什么

**Cherry Pick** 的中文翻译是**拣选提交**或**优选提交**——就像从樱桃树上挑选你喜欢的那几颗樱桃。

```
Cherry Pick = 只把某几个特定的 Commit 合并到另一个分支

和 Merge 的区别：
    Merge：把整个分支的所有 Commit 都合并过去
    Cherry Pick：只挑选你想要的那几个 Commit
```

**场景示例：**

```
    main:     ●━━●
             C1  C2

    feature:  ●━━●━━●━━●
             C1  C3  C4  C5

    需求：只把 C3 和 C4 合并到 main，C5 留在 feature 上

    Merge → 会把 C3+C4+C5 全部合并 ❌
    Cherry Pick → 只挑 C3 和 C4 ✅
```

### 6.2 操作步骤

**Desktop 中的操作（注意方向！）**

```
Step 1：切换到**输出 commit 的分支**（这里是 feature）
        ⚠️ Desktop 的逻辑：你在哪个分支就从这个分支挑选
    ↓
Step 2：在 History 中，按住 Shift 键同时选中 C3 和 C4
    ↓
Step 3：右键 → "Cherry pick 2 commits"
    ↓
Step 4：选择目标分支（接受合并的分支）→ 这里选 main
    ↓
Step 5：点击 "Cherry pick 2 commits into main"
    ↓
Step 6：Desktop 自动切换到 main 分支
        C3 和 C4 的改动出现在 main 上了！
    ↓
Step 7：点击 Push origin 推送到远端
    ↓
🎉 Cherry Pick 完成！
```

**合并后的结果：**

```
    main:     ●━━●━━━━━━●━━━━━━●
             C1  C2     C3'    C4'  ← 只挑了 C3 和 C4

    feature:  ●━━●━━●━━●
             C1  C3  C4  C5           ← C5 还留在 feature 上
```

### 6.3 Desktop 与命令行的方向差异

这是一个重要的细节——Desktop 和命令行的操作逻辑**正好相反**：

| | Desktop | 命令行 |
|---|---|---|
| 当前所在分支 | 输出 commit 的分支（feature） | 接受合并的分支（main） |
| 操作 | 选 commit → 选目标分支 | 直接指定要拣选的 commit |
| 命令 | — | `git cherry-pick <commit-id>` |

```
Desktop 逻辑：
    打开 feature → 选中 C3、C4 → 告诉它"拣选到 main"

命令行逻辑：
    git switch main  →  先切换到 main
    git cherry-pick C3 C4  →  把 C3、C4 拣选进来
```

### 6.4 Cherry Pick 的适用场景

| 场景 | 说明 |
|---|---|
| **只上线部分功能** | feature 上有 3 个功能，但只想上线其中 2 个 |
| **把 Bug 修复挑到其他分支** | 在 feature 上修了一个 Bug，但这个 Bug 在 main 上也存在 |
| **跨分支复用代码** | 某个提交的改动在多个分支都需要 |
| **不想把整个分支合并** | feature 分支上有些实验性代码还不能合入 main |

## 七、在 GitHub 网页端选择合并策略

PR 合并时，GitHub 网页端同样提供了三种选项：

```
PR 页面 → Merge pull request 旁边有个下拉箭头 ▼

    ├── Create a merge commit       ← 普通合并（默认）
    ├── Squash and merge            ← 压缩合并
    └── Rebase and merge            ← 变基合并
```

```
┌─────────────────────────────────────────────────────────────┐
│  在网页端合并 PR 时推荐的策略：                                 │
│                                                             │
│  ✅ 推荐：Create a merge commit（普通合并）                      │
│     理由：保留完整历史，安全可追溯                               │
│                                                             │
│  ✅ 可以：Squash and merge（压缩合并）                          │
│     理由：如果 feature 分支上那些"WIP""fix typo"等               │
│          零碎 commit 不想出现在主干历史中，用 Squash             │
│                                                             │
│  ❌ 不推荐：Rebase and merge                                  │
│     理由：PR 合并一般是往主干合，主干是多人协作分支，              │
│           Rebase 可能搞乱历史                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐
│           分支合并策略速查卡                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │

│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ Merge（最安全）                                       │    │
│  │   效果：保留所有历史 + 生成 merge commit               │    │
│  │   推送：普通 Push                                    │    │
│  │   场景：多人协作分支、主干合并                          │    │
```

## 十、常见问题与排错指南

### 10.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| Rebase 后 Push 被拒 | Rebase 改变了历史，和远端不一致 | 必须 Force Push（个人分支），或放弃 Rebase 改用 Merge | 🔴 高 |
| Force Push 后同事的代码丢了 | 在多人协作分支上使用了 Force Push | 立刻联系同事，从备份恢复；**永不再犯** | 🔴 高 |
| Desktop 中 Cherry Pick 找不到目标分支 | 当前在目标分支上而非源分支 | Desktop 要先切换到源分支（输出 commit 的分支） | 🟡 中 |
| Squash 后想找回原来的提交记录 | Squash 压缩掉了原始 commit | 无法找回——这就是 Squash 的代价。下次确认需要历史时用 Merge | 🟡 中 |
| 不知道选哪个合并策略 | 对三种策略的区别不清楚 | 参考 5.6 决策指南；不确定就选 Merge（最安全） | 🟡 中 |
| 网页端 PR 合并不了 | 有冲突未解决 | 先解决冲突才能合并（下节课讲） | 🟡 中 |

### 10.2 排查流程

```
合并出问题了？
    │
    ├── Push 被拒（Rebase 后）？
    │       这是正常的！Rebase 改变了历史，必须 Force Push
    │       → 确认是个人分支 → Force Push
    │       → 不确定是个人分支 → 不要 Rebase，用 Merge
    │
    ├── 选哪个策略？
    │       ├── 合到 main（多人协作）？→ Merge（默认）或 Squash
    │       ├── 整理个人分支？→ Rebase（最干净）
    │       └── 不确定？→ Merge（永远安全）
    │
    ├── Cherry Pick 操作后发现 pick 错了？
    │       对被 pick 进去的 commit 做 Revert（不要 Reset！）
    │
    └── 合并后历史很乱？
            这是 Merge 的正常结果。如果介意：
            → 之后用 Squash 策略
            → 或在个人分支上用 Rebase 整理再 PR
```

## 十一、课后练习

### 练习一：三种合并策略对比实操

**目标：** 亲身体验三种合并策略的差异。

**要求：**
- [ ] 创建三个 feature 分支：`feature-merge`、`feature-squash`、`feature-rebase`
- [ ] 在 main 上做 2-3 次 Commit（模拟主干有新提交）
- [ ] 分别在三个分支上使用对应的合并策略把 main 合进来
- [ ] 观察每次合并后的 History——记录三种策略在 History 中的不同表现
- [ ] 填写对比表：

| 策略 | History 中有几个新 Commit | 有没有 "Merge branch..." 的 Commit |
|---|---|---|
| Merge | | |
| Squash | | |
| Rebase | | |

### 练习二：Rebase 全流程练习

**目标：** 完整走一遍 Rebase → Force Push 流程。

**要求：**
- [ ] 在自己的个人分支上做 2 次 Commit
- [ ] 同时在 main 上做 2 次 Commit（模拟同事的更新）
- [ ] 在个人分支上执行 Rebase main
- [ ] 观察 History 变化——确认变成了线性
- [ ] Push 时观察被拒 → 使用 Force Push
- [ ] 确认推送成功
- [ ] ⚠️ 注意：只在个人分支上做这个练习！

### 练习三：Cherry Pick 练习

**目标：** 掌握选择性合并特定提交。

**要求：**
- [ ] 在 feature 分支上做 3 次不同的 Commit（每次改不同文件或不同内容）
- [ ] 使用 Cherry Pick 只把第 1 和第 3 次 Commit 选到 main 上
- [ ] 确认第 2 次 Commit 没有被带过去
- [ ] 在 History 中对比 feature 和 main 的差异
- [ ] 注意 Desktop 的方向：**先在源分支（feature）选 commit，再选目标分支（main）**

### 练习四：Network 图分析

**目标：** 学会阅读 Network 图。

**要求：**
- [ ] 进入你的仓库 → Insights → Network
- [ ] 观察图中的分支分叉和合并关系
- [ ] 找到你最近的几次 Merge 操作在图中的体现
- [ ] 记录：你能从图中看到几个分支？几次 Merge？哪个是最新提交？

### 练习五：策略选型思考题

**目标：** 理解不同场景下的策略选择。

**要求：** 回答以下问题并解释你的理由：

1. 你要把 feature 分支合并到 main（main 有 5 个同事在协作），你选什么策略？为什么？
2. 你在自己的个人分支上做了 10 个零碎的 Commit（"fix typo""WIP""测试"），想整理一下再提交 PR。怎么做？
3. 你在 feature 上写了 3 个功能，但老板说只上线功能 A 和功能 C，功能 B 暂时不上。你怎么操作？

## 十二、课程小结

### 12.1 核心知识图谱

```
┌────────────────────────────────────────────────────────────┐
│            分支合并 核心知识体系                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1️⃣  本地合并 vs PR 合并                                    │
│      ├── feature → main：需要 PR（正式流程+Code Review）     │
│      └── main → feature：本地合并即可（同步主干更新）          │
│                                                            │
│  2️⃣  三种合并策略                                            │
│      ├── Merge：保留所有历史 + merge commit                  │
│      │   └── 最安全，多人协作首选                              │
│      ├── Squash：压缩所有 commit → 一个 commit               │
│      │   └── 最简洁，PR 合入主干可用                          │
│      └── Rebase：变更根基 → 线性历史                          │
│          └── 最干净，但必须 Force Push → 仅个人分支            │
│                                                            │
│  3️⃣  Rebase 的本质                                          │
│      ├── Rebase = 变更根基 = 改变历史                         │
│      ├── C2 → C2'（commit ID 变了）                         │
│      ├── 必须 Force Push → 远端历史被覆盖                    │
│      └── 🔴 多人协作分支绝对不能 Rebase！                     │
│                                                            │
│  4️⃣  Cherry Pick——选择性合并                                 │
│      ├── 只挑选特定的 commit 合并到另一个分支                  │
│      ├── Desktop：在源分支选 commit → 右键 → Cherry pick     │
│      ├── 命令行：switch 到目标分支 → git cherry-pick <id>    │
│      └── 场景：只上线部分功能、跨分支复用修复                  │
│                                                            │
│  5️⃣  铁律                                                    │
│      ├── "多人用 Merge，个人用 Rebase，想简洁用 Squash"      │
│      └── "Force Push 只能在个人分支"                          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 12.2 一句话总结

> Merge 求安全，Squash 求简洁，Rebase 求干净但要付出 Force Push 的代价——三种策略没有最好，只有最合适。记住铁律：**多人在用的分支，Merge 永远是你最可靠的朋友。**
