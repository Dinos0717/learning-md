## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经掌握了：

- **分支操作** —— 创建分支、切换分支、删除分支
- **PR 合并** —— 通过 Pull Request 把功能分支合并到主干
- **合并策略** —— Merge、Squash、Rebase 的区别与选择
- **解决冲突** —— 三种冲突场景的处理方法
- **开源协作** —— Issue、PR、Code Review 的基本概念

但这些操作都是在**你有写权限的仓库**里进行的。现实中的开源贡献，你面对的是一个**别人的项目**——你没有写权限，不能直接 push 代码。那你该怎么贡献呢？这就是本课要回答的问题。

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 没有写权限，怎么给别人的项目贡献代码？ | 通过 **Fork** 把项目复制到自己名下，在自己的副本上开发，然后提 PR 合并回去 |
| Fork 之后，母项目更新了怎么办？ | 使用 **Sync Fork** 从母项目同步最新代码到子项目 |
| 同步母项目代码后产生冲突怎么办？ | 解决冲突 + Rebase + Force Push 保持历史干净 |
| 贡献被合并后该怎么收尾？ | 删除功能分支，保持 Fork 同步，准备下一次贡献 |
| 整个流程总共几步？ | 8 步：Issue → Fork → Clone+Branch → 开发 → Sync → PR → Review → Merge |

### 1.3 本课要掌握的目标

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  目标一：理解 Fork 机制                                   │
│         ├── Fork 是什么：把别人的项目复制到自己名下       │
│         ├── 母项目与子项目的关系                          │
│         └── Sync Fork：从母项目同步更新到子项目           │
│                                                         │
│  目标二：掌握开源贡献的完整 8 步流程                      │
│         ├── Issue（沟通）→ Fork（复制）→ Clone（本地）   │
│         ├── Branch（分支）→ Code（编码）→ Sync（同步）   │
│         └── PR（提交流程）→ Review（审阅）→ Merge（合并）│
│                                                         │
│  目标三：掌握 Sync + Rebase 的同步技巧                    │
│         ├── 为什么 PR 前要先同步母项目                    │
│         ├── Rebase 保持历史干净                           │

```

## 二、角色设定——贡献者 vs 维护者

### 2.1 两种角色的权限差异

在开源项目中，有两种关键角色：

| 角色 | 通俗说法 | 权限 | 本课示例 |
|---|---|---|---|
| **维护者（Maintainer）** | 项目主人 | 可以直接 push 代码、合并 PR、管理 Issue | 小助理 |
| **贡献者（Contributor）** | 热心路人 | **不能**直接 push 代码，只能通过 Fork + PR | 爬爬虾 |

```
┌─────────────────────────────────────────────────────────┐
│           两种角色的操作路径对比                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  维护者（有写权限）：                                    │
│  Clone → Branch → Code → Commit → Push → (可选) PR      │
│  可以跳过 PR，直接 push 到主干                           │
│                                                         │
│  贡献者（无写权限）：                                    │
│  Fork → Clone(Fork) → Branch → Code → Commit →          │
│  Push(Fork) → Sync Upstream → PR(Fork→母项目)           │
│  必须通过 Fork + PR 才能贡献代码                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 本课的演示场景

> 本课使用**双角色模拟**的方式演示整个开源贡献流程。
>
> - **维护者**：小助理，维护一个名为 `awesome-project` 的开源项目
> - **贡献者**：爬爬虾，想给小助理的项目增加一个新功能
>
> 我们将跟随爬爬虾的视角，走完从"有想法"到"代码被合并"的完整旅程。

## 三、第一步：Issue——先沟通，再动手

### 3.1 为什么要先提 Issue

> ⚠️ **关键原则**：在对开源项目做任何代码修改之前，**先和项目维护者沟通**。

```
不提 Issue 直接提交 PR 的风险：

❌ 你花了两天写了一个功能，提了 PR
❌ 维护者说："这个功能我们不需要 / 已经在做了 / 方向不对"
❌ 你的时间白费了！

✅ 先提 Issue 说明你的想法
✅ 维护者说："这个想法很好，期待你的 PR！"
✅ 你确认方向正确后再动手写代码
```

### 3.2 Issue 怎么写

一个好的 Issue 应该包含以下内容：

```
┌─────────────────────────────────────────────────────────┐
│  Issue 模板：我想增加一个功能                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  标题（Title）：                                         │
│  简短说明你想做什么                                      │
│  例如："增加用户头像上传功能"                             │
│                                                         │
│  描述（Description）：                                   │
│  1. 这个功能是干什么的？                                  │
│     解决什么问题？给项目带来什么价值？                     │
│                                                         │
│  2. 你的大体实现思路是什么？                              │
│     技术方案、涉及的文件、是否引入新依赖                  │
│                                                         │
│  3. 实现完后你会提交 PR                                   │
│     表示你愿意动手实现这个功能                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**操作步骤：**

```
Step 1: 打开项目主页 → 点击 "Issues" 标签页

Step 2: 点击 "New Issue" 按钮

Step 3: 填写标题和描述

Step 4: 点击 "Submit new issue" 提交
```

### 3.3 等维护者回复

提交 Issue 后，等待维护者的反馈：

```
维护者可能的回复：

情况A：✅ 认可
    "这个功能很棒！期待你的 PR！"
    → 可以开始动手了！

情况B：❓ 需要讨论
    "这个方向可以，但实现方式需要调整一下..."
    → 继续讨论，达成一致后再动手

情况C：❌ 拒绝
    "这个功能目前不在项目规划中，暂时不需要"
    → 不用浪费时间去写了

情况D：🤝 已经在做了
    "这个功能我已经在开发了，你可以看看我的分支"
    → 可以协作而不是重复劳动
```

> 💡 **经验之谈**：哪怕维护者没有明确说"期待你的 PR"，只要他对你的想法表示了认可和兴趣，你就可以开始着手开发了。

## 四、第二步：Fork——把项目复制到自己名下

### 4.1 Fork 是什么

> **Fork（复刻）= 把别人的 GitHub 仓库，完整复制一份到你自己的 GitHub 账号下。**

```
Fork 之前：
┌──────────────────────────┐
│  github.com/小助理         │
│  awesome-project          │  ← 母项目（你无权写入）
│  ├── main 分支            │
│  ├── feature-a 分支       │
│  └── ...                  │
└──────────────────────────┘

Fork 之后：
┌──────────────────────────┐     ┌──────────────────────────┐
│  github.com/小助理         │     │  github.com/爬爬虾         │
│  awesome-project          │     │  awesome-project          │
│  （母项目）                │←───│  （子项目，你有完全控制权）  │
│                           │ Fork │                           │
│                           │     │  可以自由 push、创建分支   │
└──────────────────────────┘     └──────────────────────────┘
```

> ⚠️ **关键认知**：Fork 之后，你拥有自己副本的**完全控制权**——可以自由地创建分支、推送代码、修改文件。当你的代码准备好后，再通过 PR 把改动"贡献"回母项目。

### 4.2 Fork 操作步骤

```
Step 1: 打开你想贡献的项目主页
        例如：github.com/小助理/awesome-project

Step 2: 点击右上角的 "Fork" 按钮
        ┌─────────────────────┐
        │  ⑂ Fork  ⭐ Star    │
        └─────────────────────┘

Step 3: 在弹出窗口中，选择 Fork 到哪个账号
        Owner: 爬爬虾  ← 你自己的 GitHub 账号

Step 4: 点击 "Create fork" 按钮

Step 5: 几秒钟后，Fork 完成！
        你会被带到自己的副本：
        github.com/爬爬虾/awesome-project
        
        页面顶部会显示：
        "forked from 小助理/awesome-project"
        ← 这标注了母项目是谁
```

### 4.3 Fork 之后的仓库关系

```
┌─────────────────────────────────────────────────────────┐
│          母项目与子项目的关系                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  母项目（Upstream）：                                    │
│  github.com/小助理/awesome-project                       │
│  ├── 这是"根"                                           │
│  ├── 你无法直接 push                                    │
│  └── 代码的权威来源                                      │
│                                                         │
│  子项目（Fork / Origin）：                               │
│  github.com/爬爬虾/awesome-project                       │
│  ├── 这是你的"副本"                                     │
│  ├── 你有完全的控制权                                    │
│  └── 可以自由修改                                       │
│                                                         │
│  关系：                                                 │
│  ├── Fork 之后，子项目自动记住母项目是谁                 │
│  ├── 子项目页面有 "Sync fork" 按钮来同步母项目的更新     │
│  └── PR 从子项目发起，目标指向母项目                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 五、第三步：Clone + Branch——本地开发准备

### 5.1 Clone 到本地

Fork 之后，你在 GitHub 上有了自己的副本。接下来把它克隆到本地：

```
⚠️ 注意：Clone 的是你自己的 Fork（子项目），不是母项目！

Step 1: 在你的 Fork 页面，点击绿色 "Code" 按钮

Step 2: 复制 HTTPS 或 SSH 链接
        例如：https://github.com/爬爬虾/awesome-project.git

Step 3: 在 GitHub Desktop 中：
        点击左上角小箭头 → Add → Clone repository
        → URL → 粘贴链接 → Clone
```

```
Clone 哪个仓库？
    ├── ❌ Clone 母项目 → 你没有 push 权限，代码推不上去
    └── ✅ Clone 你的 Fork → 你有完全权限，可以自由 push
```

### 5.2 创建功能分支

> ⚠️ **好习惯：永远不要在 main 分支上直接开发。** 为每个功能创建独立的分支。

```
在 GitHub Desktop 中：
Step 1: 确保当前在 main 分支

Step 2: 点击 "Current Branch" → "New Branch"

Step 3: 给分支取一个有意义的名字
        格式：feature-功能描述
        例如：feature-user-avatar-upload

Step 4: 点击 "Create Branch"
```

```
为什么要建分支而不直接在 main 上开发？

main 分支的作用：
├── 保持和母项目 main 的同步（后面会用到）
├── 作为"干净基线"，随时可以基于它创建新功能分支
└── 不要在 main 上做任何修改！

功能分支的作用：
├── 隔离本次开发的修改
├── 多个功能可以并行开发（各自独立的分支）
└── 开发完成后只把这个分支提 PR
```

### 5.3 写代码 + Commit

```
Step 1: 用 VS Code（或任意编辑器）打开项目

Step 2: 编写代码，实现你的功能

Step 3: 写一个有意义的 Commit Message
        例如："实现用户头像上传功能"
        而不是："修改了一些东西" ❌

Step 4: 点击 "Commit to feature-xxx"

Step 5: 点击 "Publish branch" 推送到你的 Fork
```

```bash
# ── 命令行等效操作 ──

# 克隆你的 Fork
git clone https://github.com/爬爬虾/awesome-project.git
cd awesome-project

# 创建功能分支
git checkout -b feature-user-avatar-upload

# 写代码...
# 修改文件...

# 提交
git add .
git commit -m "实现用户头像上传功能"

# 推送到你的 Fork
git push origin feature-user-avatar-upload
```

## 六、第四步：Sync Upstream——同步母项目的更新

### 6.1 为什么要先同步

> ⚠️ 这是很多新手会跳过的关键步骤！**在创建 PR 之前，一定要先把母项目的最新代码同步到你的分支里。**

```
为什么必须同步？

时间线：
10:00  你 Fork 了母项目，母项目 main 有 100 行代码
10:30  你开始开发功能，在你的 feature 分支上改了第 50 行
11:00  与此同时，母项目的维护者也在开发，
        他在 main 分支上也改了第 50 行 ← 冲突隐患！
12:00  你开发完成，直接提 PR
        → ⚠️ 第 50 行冲突！PR 无法自动合并！
        → 维护者要求你先解决冲突

正确做法：
12:00  你开发完成
        → 先把母项目的最新代码同步到你的分支
        → 解决可能出现的冲突
        → 确保能正常合并后再提交 PR
```

```
Sync 的完整逻辑：

母项目 main ──→ 你的 Fork 的 main ──→ 你的 feature 分支
  (upstream)      (origin/main)       (feature-xxx)

步骤：
1. 从母项目拉取最新代码到你的 Fork 的 main
2. 把你的 Fork 的 main 合并（Rebase）到你的 feature 分支
3. 解决冲突（如果有）
4. Force Push 你的 feature 分支
```

### 6.2 同步操作步骤

```
┌─────────────────────────────────────────────────────────┐
│  Step 1: 在 GitHub 网页端同步 Fork 的 main 分支          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 打开你的 Fork 页面                                   │
│     github.com/爬爬虾/awesome-project                    │
│                                                         │
│  2. 点击 "Sync fork" 按钮                               │
│     ┌──────────────────────────────┐                    │
│     │ 🔄 Sync fork                 │                    │
│     │ This branch is behind ...    │                    │
│     └──────────────────────────────┘                    │
│                                                         │
│  3. 点击 "Update branch"                                │
│     → 母项目的最新代码被拉到你的 Fork 的 main 分支        │
│     → 因为我们从未改过 main 分支，所以不会冲突           │
│                                                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Step 2: 在 Desktop 中把 main 合并到 feature 分支        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 在 Desktop 中切换到 main 分支                        │
│     点击 "Fetch origin" 拉取最新的 main 代码             │
│     （因为刚刚 Sync fork 了，所以 main 有新代码）        │
│                                                         │
│  2. 切换到 feature 分支                                  │
│     Current Branch → feature-xxx                        │
│                                                         │
│  3. 把 main 合并进 feature                               │
│     Choose a branch to merge into feature               │
│     → 选择 main 分支                                    │
│                                                         │
│  4. 选择合并方式：Rebase                                 │
│     （推荐使用 Rebase，原因见 6.4）                      │
│                                                         │
│  5. 点击 "Begin rebase"                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.3 合并时出现冲突怎么办

如果在 Rebase 过程中出现冲突（因为你改的代码和母项目新代码有重叠）：

```
┌─────────────────────────────────────────────────────────┐
│  Rebase 冲突解决流程：                                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Desktop 提示有冲突文件（红色叹号）                    │
│                                                         │
│  2. 点击 "Open in VS Code" 打开冲突文件                  │
│                                                         │
│  3. 决定保留哪些代码：                                   │
│     ├── <<<<<<< 上面 = 母项目的新代码                    │
│     ├── ======= 分隔线                                   │
│     └── >>>>>>> 下面 = 你的代码                          │
│                                                         │
│  4. 编辑文件，保留想要的内容（通常两边都要）              │
│     删除冲突标记（<<<<<<< ======= >>>>>>>）              │
│                                                         │
│  5. 保存文件                                             │
│                                                         │
│  6. 回到 Desktop → 看到绿色对勾 ✅                       │
│     → 点击 "Continue rebase"                            │
│                                                         │
│  7. Rebase 完成后，点击 "Force push origin"             │
│     ⚠️ 因为 Rebase 重写了提交历史，所以必须 Force Push   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.4 为什么选择 Rebase 而不是 Merge

```
两种同步方式对比：

方式A：Merge（普通合并）
    main:   A → B → C → D（母项目的新提交）
              ↘       ↗
    feature:   E → F → M（产生一个 Merge 提交）
    
    结果：提交历史中出现一个 "Merge branch main into feature" 的提交
    问题：这个合并提交对母项目来说没有意义，会让历史变乱

方式B：Rebase（变基）
    main:   A → B → C → D（母项目的新提交）
    
    feature:                     E' → F'（你的提交被"搬"到了最新 main 之后）
    
    结果：提交历史是一条直线，干净清爽
    优势：母项目维护者看到的是一个干净的 PR，没有无意义的合并提交

结论：feature 分支是你一个人用的 → 用 Rebase 保持历史干净
      main 分支是多人共用的 → 用 Merge 保证安全
```

> ⚠️ **Force Push 提醒**：Rebase 之后必须 Force Push，因为你的本地提交历史被重写了。这个操作在你的 feature 分支上是安全的（因为只有你一个人在用这个分支）。

## 七、第五步：创建 Pull Request

### 7.1 PR 的合并方向

> ⚠️ **关键**：PR 的方向是 **你的 Fork → 母项目**，不是反过来！

```
正确的 PR 方向：

github.com/爬爬虾/awesome-project  (你的 Fork)
    分支: feature-xxx
              │
              │ Pull Request
              ▼
github.com/小助理/awesome-project  (母项目)
    分支: main

❌ 常见错误：
   把母项目合并到自己的 Fork → 方向反了！
   PR 的 "base" 应该是母项目，"compare" 应该是你的 Fork
```

```
Step 4: 确认合并方向
        ┌─────────────────────────────────────────┐
        │ base repository: 小助理/awesome-project  │ ← 母项目
        │ base: main                               │ ← 合入母项目的 main
        │                                         │
        │ head repository: 爬爬虾/awesome-project  │ ← 你的 Fork
        │ compare: feature-xxx                     │ ← 你的功能分支
        └─────────────────────────────────────────┘

Step 5: 检查代码差异
        确认你修改的内容是正确的

Step 6: 点击 "Create pull request"

Step 7: 填写 PR 标题和描述（见 7.3）

Step 8: 点击 "Create pull request" 提交
```

## 十、第八步：收尾——清理与保持同步

### 10.1 删除功能分支

功能已经合并，本地的功能分支可以删掉了：

```
在 GitHub Desktop 中：

Step 1: 切换到 main 分支

Step 2: 右键点击 feature-xxx 分支 → Delete

Step 3: 选择 "Delete on remote as well" 同时删除远端分支

（在 Fork 的 GitHub 页面也可以手动删除分支）
```

```bash
# ── 命令行 ──
git checkout main
git branch -d feature-user-avatar-upload    # 删除本地分支
git push origin --delete feature-user-avatar-upload  # 删除远端分支
```

### 10.2 保持 Fork 与母项目同步

你的 Fork 不会自动同步母项目的更新。要定期手动同步：

```
定期同步的操作：

1. 打开你的 Fork 页面
   github.com/爬爬虾/awesome-project

2. 点击 "Sync fork" → "Update branch"
   （把母项目的最新代码同步到你的 Fork 的 main 分支）

3. 在 Desktop 中 Fetch origin → 拉取最新 main 代码

建议频率：
├── 每次准备开发新功能之前 → 必须同步
├── 定期（如每周）→ 保持不落后太多
└── 母项目有重大更新时 → 及时同步
```

### 10.3 准备下一次贡献

```
下一次想贡献新功能时的完整流程：

Step 1: Sync Fork → 更新你的 main 分支
Step 2: 从 main 创建新的 feature 分支
Step 3: 写代码 + Commit + Push
Step 4: Sync Upstream（PR 前再同步一次）
Step 5: 创建新的 PR
Step 6: 等待 Review → Merge

循环往复！
```

```
┌─────────────────────────────────────────────────────────┐
│          Fork 的生命周期                                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  你的 Fork 是长期持有的：                                 │
│  ├── 不要每次都重新 Fork                                 │
│  ├── 保持 main 分支和母项目同步                          │
│  ├── 每次新功能从最新的 main 创建新分支                  │
│  └── 功能合并后删除对应的功能分支                        │

│  ├── main = 母项目的镜像（保持同步）                      │
```

## 十二、常见问题与排错指南

### 12.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| **PR 无法自动合并（有冲突）** | 没有在提 PR 前同步母项目的最新代码 | 在本地 Rebase 母项目的 main 到你的 feature 分支，解决冲突后 Force Push | 🔴 高 |
| **Push 被拒绝（rejected）** | Rebase 后没有使用 Force Push | 执行 Force Push（`git push --force-with-lease`） | 🔴 高 |
| **Fork 的 main 落后母项目很多** | 长期没有 Sync Fork | 点击 "Sync fork" → "Update branch" | 🟡 中 |
| **Clone 了母项目而不是 Fork** | Clone 时用了母项目的链接 | 删除本地仓库，重新 Clone 你 Fork 的链接 | 🟡 中 |
| **PR 方向反了（base 和 compare 搞错）** | 把母项目设为了 compare | 在 PR 页面使用下拉菜单交换 base 和 compare | 🟡 中 |
| **Force Push 后 PR 显示很多不相关的提交** | Rebase 时把母项目的提交也包含进来了 | 确保 Rebase 时用的是 `git rebase main`（在 feature 分支上执行） | 🟡 中 |
| **Issue 提了没人回复** | 维护者可能很忙，或你的 Issue 不够清晰 | 耐心等待几天；检查 Issue 描述是否清晰完整 | 🟢 低 |
| **维护者要求 Squash 提交** | 你的 PR 有太多零碎的提交记录 | 使用 `git rebase -i` 交互式变基来合并提交 | 🟢 低 |

### 12.2 排查流程

```
PR 出现问题？
    │
    ├── PR 显示 "Can't automatically merge"？
    │   └── 你的分支落后母项目了
    │       → Sync Fork → Rebase main 到 feature → 解决冲突 → Force Push
    │
    ├── Push 报错 "failed to push some refs"？
    │   └── 本地历史与远端不一致（Rebase 后常见）
    │       → git push --force-with-lease origin feature-xxx
    │
    ├── PR 包含了别人的提交？
    │   └── Rebase 方向错了，或者 Merge 了不该合并的东西
    │       → 检查 git log，确认提交历史
    │
    ├── CI/CD 检查失败了（如果有）？
    │   └── 代码风格、测试、构建出了问题
    │       → 查看 CI 日志，修复对应问题，Push 更新
    │
    └── 维护者一直没回复？
        └── 等 3-5 天 → 在 PR 下礼貌地留言提醒
            → 如果长期无回复，可能是项目不活跃了
```

```
   - 下方小字 "forked from 原作者/项目名"

4. 在 GitHub Desktop 中 Clone 你 Fork 的仓库
```
</details>

### 练习二：进阶——完整模拟开源贡献（双账号）

  5. 创建 PR

- 账号B（贡献者）：删除功能分支

## 十四、课程小结

### 14.1 核心知识图谱

```
┌────────────────────────────────────────────────────────────┐
│         开源贡献流程——核心知识体系                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1️⃣  两种角色                                               │
│      ├── 维护者：有写权限，可以 Merge PR                    │
│      └── 贡献者：无写权限，必须通过 Fork + PR               │
│                                                            │
│  2️⃣  Fork 机制                                              │
│      ├── Fork = 把别人的项目复制到自己名下                   │
│      ├── 母项目（Upstream）vs 子项目（Fork/Origin）         │
│      └── Sync Fork：从母项目同步更新到子项目                │
│                                                            │
│  3️⃣  八步流程                                               │
│      ├── Issue → Fork → Clone+Branch → Code+Push          │
│      ├── → Sync Upstream → Rebase → Force Push            │
│      └── → Create PR → Review → Merge → Cleanup           │
│                                                            │
│  4️⃣  Sync + Rebase 技巧                                     │
│      ├── PR 前必须同步母项目的最新代码                       │
│      ├── feature 分支用 Rebase（保持历史干净）              │
│      └── Rebase 后必须 Force Push                          │
│                                                            │
│  5️⃣  Fork 的长期维护                                        │
│      ├── 不要反复 Fork，一个 Fork 长期使用                   │
│      ├── 定期 Sync Fork 保持 main 同步                     │
│      └── 每次新功能从最新 main 创建新分支                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 14.2 一句话总结

> **开源贡献的核心流程：先 Issue 沟通 → Fork 到自己名下 → 在功能分支上开发 → 同步母项目避免冲突 → 提 PR → 根据 Review 修改 → 维护者 Merge。关键认知是：Fork 给了你在自己地盘上自由开发的能力，PR 是把你的成果贡献回母项目的通道。**
