## 一、课程背景与目标

### 1.1 前情回顾

在前面的课程中，你已经学会了：

- **创建仓库** —— 选择公开/私有、开源协议、编写 README
- **理解 Commit** —— 仓库状态的快照，形成历史链路
- **掌握分支** —— 分支的隔离性，在各自分支上开发互不影响
- **创建和合并 PR** —— Pull Request 的完整流程、Code Review 三种结论

现在你已经掌握了所有"零件"，但这节课要回答一个更重要的问题：**这些零件是怎么组装起来，让全世界的人一起协作的？**

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 一个开源功能从"想法"到"上线"要经历哪几步？ | Issue → Branch → Commit → PR → Review → Merge → Release |
| 不同角色在协作中各自做什么？ | 使用者提需求、开发者写代码、管理员审核合并 |
| 为什么 GitHub 能支撑全球协作？ | 每个人有自己的一份代码拷贝，通过 PR 汇合 |
| 为什么要学这套流程？ | 这是全世界数百万开发者每天都在用的工作方式 |

> ⚠️ **关键认知**：本节课用一个**完整的动画故事**把前面学过的所有概念串起来。你会发现，Issue、Branch、Commit、PR、Code Review、Merge 不是孤立的知识点，而是一条**环环相扣的协作链路**。

### 1.3 本课要掌握的目标

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  目标一：理解开源协作的完整五步流程                                │
│         ├── Issue → Branch → PR → Code Review → Merge       │
│         ├── 知道每一步的目的是什么                              │
│         └── 能用自己的话讲出整个流程                            │
│                                                             │
│  目标二：掌握 Issue 的创建方法                                   │
│         ├── 理解 Issue 的三种用途（Bug/功能/讨论）              │
│         ├── 学会使用 Issue 模板                                │
│         └── 能独立提交一份规范的功能请求                         │
│                                                             │
│  目标三：体验完整的协作角色                                      │
│         ├── 理解使用者（提需求）、开发者（写代码）、               │
│         │   管理员（审核合并）各自的职责                          │
│         └── 能模拟走完一次完整的协作流程                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、一个真实的协作故事

> 本节课老师使用了一个 GitHub 官方的动画短片，并加上了中英双语字幕。这个故事生动展示了 GitHub 开源协作的完整流程。我们跟着故事一步一步来理解。

### 2.1 角色介绍

故事中有两组角色，代表了一个开源项目中的所有参与者：

**开发团队（在 GitHub 上协作）：**

| 角色 | 身份 | 职责 |
|---|---|---|
| **Sam** | 项目负责人 | 分配任务，协调团队 |
| **PJ** | 开发者 | 写代码，创建 PR |
| **Melinda** | 开发者 | Code Review，提修改意见 |
| **Mike** | 开发者 | 参与协作 |

**使用者（提出需求的人）：**

| 角色 | 身份 | 需求 |
|---|---|---|
| **Eddie** | 拖拉机驾驶员 | 想让农民分享收割机的诊断数据，改进收成 |

**关键背景：**
- Sam 和她的团队开发的是**拖拉机软件**（让拖拉机更高效）
- 他们虽然做同一个项目，但**分散在世界各地**
- Eddie 不是开发者——他是真正开拖拉机的人，每天和 Harvester L700 拖拉机打交道
- Harvester L700 是公司最新的旗舰拖拉机，有大量传感器和处理器，产生海量诊断数据

> 💡 **这个设定的巧妙之处**：Eddie 不是程序员，但他有**真实的需求**。开源项目的使用者往往不是写代码的人——他们是一线用户，最知道"痛点"在哪。GitHub 让使用者和开发者可以直接沟通。

### 2.2 问题的起点——Eddie 的需求

Eddie 每天开着 Harvester L700 在田里工作。他发现：

```
Harvester L700 有大量传感器和处理器
    ↓
产生海量诊断数据
    ↓
Eddie 有个想法：农民之间能不能分享这些数据？
    ↓
如果大家共享数据，每个人都能更好地改进收成！
    ↓
但 Eddie 不会写代码——他需要一个功能
    ↓
他打开了 GitHub…… 🚀
```

### 2.3 故事全景图

在深入每一步之前，先看一下协作的**全貌**：

```
Eddie（使用者）                    Sam的团队（开发者）
    │                                  │
    │  ① 提交 Issue（功能请求）           │
    ├─────────────────────────────────→│
    │                                  │  ② Sam 看到 Issue
    │                                  │     指派 PJ 开发
    │                                  │
    │                                  │  ③ PJ 创建分支
    │                                  │     在分支上写代码
    │                                  │     Commit 保存快照
    │                                  │
    │                                  │  ④ PJ 发起 PR
    │                                  │     请求合并代码
    │                                  │
    │                                  │  ⑤ Melinda 等人
    │                                  │     Code Review
    │                                  │     审查代码质量
    │                                  │
    │                                  │  ⑥ ✅ 审核通过
    │                                  │     合并到主干
    │                                  │
    │  ⑦ 🎉 新功能上线！                 │
    │     全世界的农民都能用了              │
    │                                  │
    └──────────────────────────────────┘
```

## 三、第一步：Issue——一切从提出问题开始

### 3.1 Issue 是什么

**Issue** 是 GitHub 上的**讨论线程**。它是一切协作的起点。

```
Issue = 一个讨论话题

可以包含：
    ├── 标题和描述
    ├── 标签（Bug、Feature、Help Wanted 等）
    ├── 评论区（多人讨论）
    ├── 指派负责人
    └── 关联的 PR（追踪谁在解决这个问题）
```

> 💡 **"Issue" 不只是"问题"**：虽然叫 Issue，但它不只是用来报 Bug 的。任何需要讨论、追踪、协作的事情都可以开一个 Issue。

### 3.2 Issue 的三种典型用途

| 用途 | 说明 | 谁发起的 | 示例 |
|---|---|---|---|
| **🐛 Bug 报告** | 发现了软件的某个缺陷 | 使用者 | "点击登录按钮后页面白屏" |
| **💡 功能请求** | 希望软件增加某个新功能 | 使用者 | "希望能支持暗色模式" |
| **💬 讨论** | 关于项目的任何话题 | 任何人 | "下个版本要不要升级到 Vue 3？" |

**Eddie 的场景属于第二种——功能请求：**

```
Eddie 的需求：
    "农民之间能不能分享拖拉机的诊断数据？"
    
    → 这是一个 Feature Request
    → 他不是开发者，但他是最了解"痛点"的人
    → 通过 Issue，他可以直接把需求告诉开发团队
```

### 3.3 实操：提交一个功能请求

**操作步骤：**

```
Step 1: 打开一个你喜欢的开源项目
    ↓
Step 2: 点击顶部的 "Issues" 选项卡
    ↓
Step 3: 点击绿色的 "New issue" 按钮
    ↓
Step 4: 如果项目提供了 Issue 模板，选择 "功能请求"（Feature Request）
    ↓
Step 5: 点击 "Get started"
    ↓
Step 6: 根据模板填写内容：
    ├── 你想要什么功能？
    ├── 为什么需要这个功能？
    ├── 你设想的实现方式？
    └── 有没有替代方案？
    ↓
Step 7: 点击 "Submit new issue"
    ↓
🎉 Issue 提交成功！开发者会看到你的请求。
```

> ⚠️ **给真实项目提 Issue 的礼仪**：
> 1. 先搜索——看看有没有人提过相同的问题
> 2. 按模板填——不要删掉模板内容
> 3. 描述清晰——附上截图、日志、复现步骤
> 4. 保持礼貌——开源维护者大多是用爱发电的志愿者

### 3.4 Issue 模板——开源项目的沟通规范

很多成熟的开源项目会提供 Issue 模板，帮助使用者规范地提交问题：

```
┌─────────────────────────────────────────────┐
│  选择一个 Issue 类型：                        │
│                                             │
│  🐛 Bug Report                              │
│     描述你遇到的 Bug                          │
│                                             │
│  💡 Feature Request          ← Eddie选这个  │
│     描述你希望增加的功能                       │
│                                             │
│  ❓ Question                                │
│     提出你的疑问                              │
│                                             │
│  每种类型点进去都有对应的填写模板               │
└─────────────────────────────────────────────┘
```

**功能请求模板示例：**

```markdown
## 你希望增加什么功能？
[清晰描述你想要的功能]

## 为什么需要这个功能？
[描述你的使用场景和痛点]

## 你设想的实现方式？
[如果有想法可以写，没有也可以留空]

## 有没有替代方案？
[当前你是怎么解决这个问题的？]
```

> 💡 **故事中**：Sam 是第一个看到 Eddie 的 Issue 的人。她认为 **PJ 是最适合做这个功能的人**，于是把任务分派给了 PJ。这就是真实协作中的"任务分配"——不是随机找人，而是根据每个人的专长来分派。

## 四、第二步：Branch——创建独立的开发空间

### 4.1 故事中的分支操作

PJ 接到任务后，开始写代码。但他**不会直接在 main 主干上修改**——他创建了一个分支。

```
故事中的核心逻辑：

"PJ isn't the only person working on this project.
People all over the world are also working on it."

（PJ 不是这个项目唯一的开发者，全世界还有其他人也在贡献代码。）

    → 不能在主干上直接改
    → 必须创建自己的分支
    → 在分支上的修改不会影响别人
    → 这就是上一课学的"分支隔离性"
```

### 4.2 实操复习：创建分支

```
进入任意一个开源项目
    ↓
点击左上角的分支选择器（显示 main 或 master）
    ↓
点击 "View all branches" 查看所有分支
    ↓
点击右上角的 "New branch" 按钮
    ↓
填写分支名（例如 feature-share-data）
    ↓
选择基于哪个分支创建（通常选 main 主干）
    ↓
点击 "Create branch"
    ↓
🎉 分支创建完成！可以开始在上面写代码了。
```

### 4.3 Commit——在分支上追踪变更

PJ 在分支上写代码，每完成一个阶段性工作就做一次 Commit：

```
分支上的 Commit 链路：

feature-share-data 分支：
    ● ──── ● ──── ● ──── ●
   C1      C2      C3      C4
  开始    数据    分享    测试
         模型    接口    通过

每次 Commit = 保存一次仓库状态的快照
每次 Commit = 写一个 Commit Message 说明改了什么
```

> 💡 **回顾关键概念**：Commit 不仅仅是"把文件提交到 GitHub 网站"。每次 Commit，Git 都会保存一份仓库此时所有文件的状态快照。随着 Commit 越来越多，形成一条可回溯的历史链路。

## 五、第三步：Pull Request——发起合并提案

### 5.1 故事中的 PR

PJ 写完了代码，但他不能自己直接把代码合并进主干——他需要发起一个 **Pull Request**。

```
故事原文：

"PJ opens a Pull Request on GitHub.
Pull requests let you show others the changes you're proposing,
so they can review and discuss."

翻译：PJ 提交了一个 PR，也就是将他的分支代码合并回主干的请求。
接下来他的团队会审核这个请求中的代码，对不足做出指正或修改。
```

**PR 的本质：**

```
PR ≠ "我宣布代码合并了"
PR = "我提议把代码合并，请大家审查"

区别：
    ├── "我宣布" → 不经过审查，风险自负
    └── "我提议" → 团队审查 → 通过后才合并 ✅
```

### 5.2 实操复习：创建 PR

```
Step 1: 在 feature 分支上完成代码修改并 Commit
    ↓
Step 2: 点击仓库顶部的 "Pull requests" 选项卡
    ↓
Step 3: 点击绿色的 "New pull request" 按钮
    ↓
Step 4: 选择合并方向（注意箭头的方向！）
         base: main  ← 目标分支（合到哪里）
         compare: feature ← 源分支（从哪里合）
         箭头方向 = feature → main
    ↓
Step 5: 查看列出的修改记录，确认无误
    ↓
Step 6: 点击 "Create pull request"
    ↓
Step 7: 填写 PR 标题和描述
    ↓
Step 8: 点击 "Create pull request"
    ↓
🎉 PR 创建成功！
```

### 5.3 PR 列表——团队协作的"任务看板"

PR 创建后，它会出现在项目的 **Pull requests 列表**中，团队的同事都能看到：

```
Pull requests 列表：

┌────────────────────────────────────────────────┐
│  Open PRs                                      │
│                                                │
│  #42 添加数据分享功能    PJ        待审核 🟡     │
│  #41 修复登录页样式     Melinda   待审核 🟡     │
│  #40 升级依赖版本       Mike      已合并 🟣     │
│                                                │
└────────────────────────────────────────────────┘

点击任意 PR → 进入详情页 → 可以查看代码 → 可以 Code Review
```

## 六、第四步：Code Review——团队审查代码

### 6.1 为什么每个人都要 Review

故事中有一个关键细节：

```
"On GitHub, everyone has a copy of the project."

（在 GitHub 上，每个人都有自己的项目拷贝。）

    → Melinda 可以看到 PJ 的代码
    → Mike 也可以看到 PJ 的代码
    → 每个人都可以提出自己的审查意见
    → 这意味着：代码质量不是一个人说了算的
```

**Code Review 的价值：**

| 价值 | 说明 |
|---|---|
| **发现 Bug** | 第二双眼睛更容易发现逻辑错误 |
| **统一风格** | 保持代码风格和架构的一致性 |
| **知识共享** | 审查者和被审查者都在学习 |
| **安全保障** | 防止恶意代码或后门被合入 |
| **团队成长** | 资深开发者通过 Review 指导新人 |

### 6.2 Review 的过程

```
Melinda 看到 PJ 的 PR
    ↓
进入 "Files changed" 选项卡 → 逐行查看代码改动
    ↓
如果发现问题或有改进建议：
    在具体代码行左侧点 + → 写评论
    ↓
全部看完后 → 点击 "Review changes" → 选择结论：

    ┌──────────────────────────────────────┐
    │ Comment         只是评论，不决定合并    │
    │                  "有个想法，讨论一下"   │
    │                                      │
    │ Approve         同意合并 ✅            │
    │                  "代码没问题！"         │
    │                                      │
    │ Request Changes 需要修改 ❌            │
    │                  "这里有3个问题要改"    │
    └──────────────────────────────────────┘
```

**故事中的关键台词：**

```
"The team has signed off on the changes."
（团队已经批准了这些更改。）

"Signed off" = 审核通过，签字确认。
这表示 Code Review 环节结束，代码获得了团队的认可。
```

### 6.3 Git 记录所有人的贡献

```
"Git keeps a record of everyone's contributions,
alongside relevant messages from the team's other systems."

（Git 记录了每个人的贡献，以及团队其他系统中的相关消息。）

含义：
    ├── 谁写了哪行代码 → Git 记录
    ├── 谁审查了代码 → GitHub 记录
    ├── 谁提出了什么意见 → PR 评论区记录
    └── 整个协作过程可追溯
```

## 七、第五步：Merge——代码合入主干

### 7.1 合并操作

Code Review 通过后，PJ 就可以把代码合并进主干：

```
Step 1: PR 页面 → 点击 "Merge pull request"
    ↓
Step 2: 点击 "Confirm merge" 确认
    ↓
Step 3: 出现紫色 Merged 图标 🟣
    ↓
🎉 合并完成！feature 分支的代码现在在 main 主干上了。
```

> 💡 **回忆上节课**：合并后建议删除已合并的功能分支，保持分支树干净。主干上代码就是最新的完整代码。

### 7.2 发布上线

合并完代码后，故事讲到了最后一步：

```
"The new feature will be available to everyone
as soon as it's merged in."

（一旦合并，新功能就立刻对所有人可用。）

    main 主干 = 可发布状态
    → 合并完成
    → 对主干进行一次发布（Release）
    → 软件的新功能上线 🎉
    → 全世界的农民都能受益！
```

## 八、完整协作流程总览

### 8.1 五步协作流程

把故事中的每一步串起来，就是 GitHub 开源协作的完整流程：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ① Issue（提出问题）                                         │
│     Eddie 提交功能请求 → Sam 看到 → 指派 PJ 开发              │
│     工具：Issue 页面，选择模板，填写描述                       │
│      ↓                                                       │
│  ② Branch（创建分支）                                        │
│     PJ 基于 main 创建 feature 分支 → 获得独立开发空间         │
│     工具：分支选择器 → New branch → 选 Source → 创建          │
│      ↓                                                       │
│  ③ Commit + PR（写代码 + 发起合并请求）                       │
│     PJ 在分支上写代码 → 多次 Commit 保存快照                   │
│     → 发起 PR：compare feature → base main                  │
│     工具：编辑文件 → Commit changes → New pull request       │
│      ↓                                                       │
│  ④ Code Review（代码审查）                                   │
│     Melinda / Mike 等人审查代码 → 提意见                      │
│     → 修改 → 再审查 → Approve（批准）                         │
│     工具：Files changed → 逐行评论 → Review changes          │
│      ↓                                                       │
│  ⑤ Merge + Release（合并 + 发布）                            │
│     PJ 点击 Merge → 代码合入主干 → 发布 → 功能上线 🎉         │
│     工具：Merge pull request → Confirm merge                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 角色分工表

| 角色 | 代表人物 | 在流程中做什么 | 使用的 GitHub 功能 |
|---|---|---|---|
| **使用者** | Eddie | 提出需求（Bug 或新功能） | Issue |
| **项目负责人** | Sam | 分派任务、协调资源 | Issue 指派、项目管理 |
| **开发者** | PJ | 写代码、创建 PR | Branch、Commit、PR |
| **审查者** | Melinda, Mike | 审查代码质量、提修改意见 | Code Review、Files changed |
| **所有人** | All | 讨论、提建议、贡献代码 | Issue 评论区、PR 评论区 |

### 8.3 一套流程串起所有学过的概念

```
前面课程学过的概念            在本课协作流程中的位置

   仓库（Repository）    →    整个协作的发生地
   README               →    项目介绍，帮助 Eddie 了解这个项目
   Commit               →    PJ 在分支上保存每次改动的快照
   Branch（分支）        →    PJ 的独立开发空间，不影响主干
   Pull Request（PR）    →    PJ 发起合并请求，请求审查
   Code Review          →    Melinda 等人审查代码
   Merge（合并）         →    审查通过，合入主干
   Issue                →    🆕 Eddie 提交功能请求，一切从这里开始
```

> 💡 **现在你看到了全貌**：之前每节课学的是一个"零件"，这节课把这些零件**组装成了一个完整的协作机器**。这就是全世界数百万开发者每天都在用的工作流。

## 十、常见问题与排错指南

### 10.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| Issue 提交后没人回复 | 描述不够清晰或项目维护者太忙 | 补充更多细节；耐心等待；可以礼貌地在评论区 @ 维护者 | 🟡 中 |
| PR 被拒绝（Request Changes） | 代码有需要改进的地方 | 仔细阅读审查意见 → 修改代码 → 提交新 Commit → PR 自动更新 | 🟡 中 |
| PR 长时间没人 Review | 团队可能没注意到或人手不足 | 在 PR 评论区 @ 相关同事请求 Review | 🟡 中 |
| 不知道 PR 该往哪个分支合 | 不熟悉项目的分支策略 | 查看项目的 CONTRIBUTING.md 文件，通常有说明 | 🟡 中 |
| 创建分支时选错了 Source | 代码基准不对 | 删除错误分支，重新基于正确的分支创建 | 🟢 低 |
| Issue 模板看不懂（英文） | 很多国际项目用英文 | 用翻译工具辅助理解，按模板逐项填写 | 🟢 低 |

### 10.2 排查流程

```
协作流程中卡住了？
    │
    ├── Issue 阶段？
    │       ├── 不知道该不该提 Issue？→ 先搜索是否已有类似问题
    │       ├── 不知道怎么写？→ 按项目提供的模板填
    │       └── 没人理？→ 检查描述是否清晰，必要时补充信息
    │
    ├── 开发阶段？
    │       ├── 分给谁做？→ 在 Issue 中 @ 负责人或自己认领
    │       ├── 不会创建分支？→ 回顾第八课：分支操作
    │       └── Commit Message 怎么写？→ 描述本次改了什么即可
    │
    ├── PR 阶段？
    │       ├── PR 没反应？→ @ 同事请求 Review
    │       ├── 被要求修改 → 按要求改完后再 Commit
    │       └── 有冲突？→ 先把 main 最新代码合并到自己的分支再解决冲突
    │
    └── 合并后？
            ├── 功能没生效？→ 确认是否发布了新版本
            └── 分支太多？→ 删除已合并的功能分支
```

## 十一、课后练习

### 练习一：提交 Issue

**目标：** 学会规范地提交一个 Issue。

**要求：**
- [ ] 在你自己的 hello-world 仓库中，开启 Issues 功能（Settings → Features → Issues ✅）
- [ ] 提交一个 **Bug Report** 类型的 Issue（模拟发现了一个 Bug）
- [ ] 提交一个 **Feature Request** 类型的 Issue（模拟想要一个新功能）
- [ ] 每条 Issue 都包含清晰的标题和描述
- [ ] 观察两个 Issue 出现在 Issue 列表中

### 练习二：完整协作模拟（单人分饰多角）

**目标：** 一个人模拟整个团队的协作流程。

**场景：** 你自己同时扮演 Eddie（使用者）和开发者。

**要求：**
- [ ] **扮演 Eddie**：在你自己的仓库中提交一个 Issue（功能请求）
- [ ] **扮演开发者**：基于 main 创建一个 feature 分支
- [ ] 在 feature 分支上做出代码修改（实现 Eddie 请求的"功能"）
- [ ] 至少做 2 次 Commit，每次写清晰的 Commit Message
- [ ] 创建 PR：feature → main
- [ ] 在 PR 描述中关联 Issue（写 `Closes #1`）
- [ ] **扮演审查者**：进入 Files changed，给自己提至少 1 条评论
- [ ] 审查通过 → Merge PR
- [ ] 确认 Issue 自动关闭
- [ ] 删除 feature 分支

**验收标准：**
- [ ] Issue 被创建并关闭
- [ ] PR 被创建并合并
- [ ] 至少 2 次 Commit 记录可查
- [ ] 功能分支已删除

### 练习三：给真实开源项目提 Issue

**目标：** 体验真实的开源协作。

**要求：**
- [ ] 找到一个你日常使用的开源项目（工具、库、框架都可以）
- [ ] 浏览它的 Issue 列表，了解大家是怎么提 Issue 的
- [ ] 找一个真实的需求或 Bug（哪怕是文档拼写错误也可以！）
- [ ] 按照项目的 Issue 模板提交
- [ ] 截图记录你的 Issue 链接

> 💡 **提示**：如果你实在找不到 Bug，可以从这些入手：
> - 文档中的拼写错误
> - 某个功能的中文翻译不准确
> - README 中的示例代码过时了

### 练习四：画出协作流程图

**目标：** 用自己的话和图示理解协作流程。

**要求：**
- [ ] 在白纸或画图工具上画出完整的五步协作流程
- [ ] 标注每一步使用的 GitHub 功能
- [ ] 标注每一步涉及的角色
- [ ] 用自己的话给一个完全不懂 GitHub 的人解释清楚
