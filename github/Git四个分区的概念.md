### 1.1 前情回顾

在前面课程中，你已经学会了：

- **网页端操作** —— 创建仓库、编辑文件、分支、PR
- **GitHub Desktop** —— 克隆仓库、本地编辑、Fork、将文件夹初始化为仓库

到目前我们已经讲了不少 Git 术语了：Clone、Commit、Push、Pull、Fetch……但它们之间的关系是什么？每个命令到底在操作什么？本节课把这些概念**串联起来**，给你一张完整的 Git 工作流地图。

### 1.2 本节课的核心问题

## 二、Git 四个分区总览

### 2.1 四个分区是什么

Git 把代码的管理分成了**四个独立的区域**，大多数 Git 操作的本质，就是**在这四个分区之间同步文件**。

| 分区 | 英文名 | 位置 | 一句话理解 |
|---|---|---|---|
| **工作区** | Working Directory | 你电脑磁盘上的本地文件夹 | 你正在编辑的代码 |
| **暂存区** | Staging Area / Index | `.git/index` 文件中 | Commit 前的"候车室" |
| **本地存储库** | Local Repository | `.git/objects/` 中 | Commit 的永久存储地 |
| **远端存储库** | Remote Repository | GitHub / GitLab 等服务器 | 共享和备份代码的地方 |

### 2.2 四个分区的关系图

```
       你的电脑（本地）                        互联网上（远端）
  ┌─────────────────────────┐          ┌──────────────────┐
  │                         │          │                  │
  │  ┌───────────┐          │          │  Remote Repo     │
  │  │ Working   │          │          │  (GitHub 网站)    │
  │  │ Directory │          │          │                  │
  │  │ (工作区)   │          │          └────────┬─────────┘
  │  └─────┬─────┘          │                   │
  │        │                │        ┌──────────┴─────────┐
  │        │  git add       │        │    git push ↑      │
  │        ▼                │        │    git fetch ↓     │
  │  ┌───────────┐          │        │    git pull ↓      │
  │  │ Staging   │          │        │    git clone ↓     │
  │  │ Area      │          │        │                   │
  │  │ (暂存区)   │          │        └───────────────────┘
  │  └─────┬─────┘          │
  │        │                │
  │        │  git commit    │
  │        ▼                │
  │  ┌───────────┐          │
  │  │ Local     │          │
  │  │ Repository│          │
  │  │ (本地存储库)│          │
  │  └───────────┘          │
  │                         │
  └─────────────────────────┘
```

### 2.3 一个比喻帮你记住

```
把 Git 四个分区想象成"写信-寄信"的过程：

    📝 工作区       = 你正在写的草稿（可以随便改）
    📋 暂存区       = 把信装进信封，写好地址（准备寄出）
    📮 本地存储库   = 邮局的发件箱（信已经投递，但还在本地邮局）
    ✈️ 远端存储库   = 对方收到了信（信送到了远方）

    写草稿 → 装信封 → 投递到邮局 → 送达对方
    Working → Staging → Local Repo → Remote Repo
      ↑         ↑          ↑            ↑
    编辑代码  git add   git commit   git push
```

## 三、工作区（Working Directory）

### 3.1 工作区是什么

**工作区** 就是你电脑磁盘上的**本地文件夹**——你用 VS Code 打开的那个目录。

```
工作区 = 你正在编辑的代码的所在地

特点：
    ├── 你可以随意增删改文件
    ├── 改错了 Ctrl+Z 就行
    ├── 还没有被 Git 正式记录
    └── 这是你的"开发环境"
```

### 3.2 工作区中的文件状态

在 GitHub Desktop 中，工作区的文件改动会显示为不同状态：

```
文件状态        Desktop 显示              含义
────────────────────────────────────────────────────
🆕 New         + 号，文件名旁有绿色标记      新创建的文件
✏️ Modified    修改图标                     已有文件的内容被修改
🗑️ Deleted    删除图标                     文件被删除
```

**老师演示的例子：**

```
git-test 项目中的改动：

    test4.py    🆕 New      ← 新创建的文件（打印本月日历的小程序）
    test.py     🗑️ Deleted  ← 被删除的文件
    test3.py    ✏️ Modified  ← 修改了内容（把"拼音"改成"爬爬虾"）

这三个文件的改动都发生在工作区。
```

## 四、暂存区（Staging Area / Index）

### 4.1 暂存区是什么

**暂存区（Staging Area）** 也叫 **Index（索引区）**，是 `git commit` 之前的**临时保存点**。

```
暂存区 = Commit 前的"候车室"

作用：
    ├── 只有进入暂存区的文件，才能被 Commit 正式记录
    ├── 让你可以"挑选"哪些改动要提交，哪些不要
    └── 是一个可逆的中间步骤
```

### 4.2 为什么需要暂存区

暂存区的存在解决了一个核心问题：**不是所有改动都必须一起 Commit**。

```
场景：你同时改了 3 个文件

    test4.py   → 新功能（已完成，可以提交）
    test3.py   → 改了个参数（已完成，可以提交）
    test.py    → 修复了一个 Bug（还没测完，不想提交！）

没有暂存区：
    所有改动打包成一次 Commit → test.py 的半成品也被提交了 ❌

有暂存区：
    git add test4.py  → 暂存区里有 test4.py
    git add test3.py  → 暂存区里有 test3.py
    （test.py 不 add，留在工作区）
    git commit → 只提交了 test4.py 和 test3.py ✅
```

> 💡 **暂存区 = 选择权**：你可以把一次修改的多个文件分成多次 Commit，每次 Commit 只包含相关的改动。这让 Commit 历史更加清晰。

### 4.3 GitHub Desktop 隐藏了暂存区

上节课使用 GitHub Desktop 时，你可能注意到了一个细节：**没有看到暂存区这个概念**。

```
GitHub Desktop 的设计：

    命令行 Git：                          GitHub Desktop：
    git add 文件        ← 第一步          点击 Commit
    git commit -m "..."  ← 第二步          ↓
                                         Desktop 自动执行了 git add + git commit
                                         （两个命令合并成一个操作）
```

| 方式 | 暂存区 | 操作步骤 |
|---|---|---|
| **Git 命令行** | 显式存在 | `git add` → `git commit`（两步） |
| **GitHub Desktop** | **隐藏了** | 点击 Commit（自动完成 Add + Commit） |

> 💡 **Desktop 简化了体验**：对于日常开发来说，大部分时候你确实想把所有改动一起提交。Desktop 把 Add + Commit 合并成一步，让操作更简单。等学到 Git 命令行时，你会重新见到暂存区。

## 五、本地存储库（Local Repository）

### 5.1 本地存储库是什么

**本地存储库** 是存储在你电脑上的 Git 仓库——具体来说，就是 `.git` 文件夹里的内容。

```
本地存储库 = .git 文件夹

位置：项目文件夹下的隐藏 .git 目录
内容：所有 Commit 的永久记录（objects/）、分支引用（refs/）、HEAD 等
```

### 5.2 Commit 与本地存储库的关系

```
git commit 做的事情：

    暂存区的内容  ──Commit──→  本地存储库

    Commit 完成后：
        ├── 暂存区的内容被永久保存到 .git/objects/
        ├── 生成一个 Commit 对象（包含作者、时间、信息、指向 tree）
        ├── 当前分支的引用更新（refs/heads/xxx 指向新的 Commit ID）
        └── 这次改动被正式记录，成为历史链路的一部分
```

**Commit 后的状态：**

```
Commit 只在本地，还没有同步到 GitHub：

    ✅ 改动已保存到本地存储库（安全，不会丢）
    ✅ 可以查看历史记录
    ❌ 其他人还看不到你的代码
    ❌ 如果电脑坏了，代码就没了
    → 需要 git push 推送到远端存储库
```

> 💡 **本地存储库 = 离线也能工作**：所有 Commit 都先在本地存储。没有网络也能 Commit，等有网络了再 Push。这就是 Git 分布式设计的优势。

## 六、远端存储库（Remote Repository）

### 6.1 远端存储库是什么

**远端存储库** 是托管在互联网服务器上的 Git 仓库。本课程中特指 **GitHub 网站**。

```
常见的远端存储库托管平台：

    ├── GitHub（本课程使用）
    ├── GitLab
    ├── Bitbucket（Stash）
    ├── Gitee（码云）
    └── 自建的 Git 服务器
```

### 6.2 远端存储库的作用

| 作用 | 说明 |
|---|---|
| **共享代码** | 团队成员都能看到和拉取你的代码 |
| **远程备份** | 电脑坏了？代码在远端一份都不会丢 |
| **协作枢纽** | PR、Code Review、Issue 都在远端进行 |
| **公开展示** | 开源项目让全世界的人看到 |

> ⚠️ **只有 Push 到远端的代码，才能被其他人看到**。只 Commit 到本地存储库的代码，只有你自己能看到。

## 七、四大命令——在分区之间移动代码

### 7.1 命令全景图

```
                git clone (远端 → 本地，一次性操作)
                     ↓
    ┌──────────────────────────────────────────────┐
    │                                              │
    │   工作区  ←──────── 本地存储库  ←────── 远端存储库 │
    │          git merge      │     git fetch       │
    │                         │     git pull        │
    │   工作区  ──git add──→ 暂存区                  │
    │   暂存区  ──git commit──→ 本地存储库            │
    │   本地存储库 ──git push──→ 远端存储库           │
    │                                              │
    └──────────────────────────────────────────────┘
```

### 7.2 git clone——远端 → 本地

```
git clone = 把远端存储库完整复制到本地

操作：git clone <仓库URL> <本地路径>

在 Desktop 中：Clone a repository → 选仓库 → 选路径 → Clone

做三件事：
    ① 在本地创建文件夹（工作区）
    ② 下载所有文件和完整历史（本地存储库）
    ③ 自动关联远端仓库地址

方向：Remote → Working Directory + Local Repository
```

> 💡 **Clone 是第一次使用的操作**——一个项目只需 Clone 一次。之后就用 Push / Pull 同步。

### 7.3 git add——工作区 → 暂存区

```
git add = 把工作区的改动保存到暂存区

操作：git add <文件名>          ← 添加指定文件
      git add .                ← 添加所有改动

方向：Working Directory → Staging Area

含义：告诉 Git "这些改动我准备好了，下次 Commit 要把它们记录进去"

注意：
    ├── git add 是可逆的（可以用 git reset 撤回）
    ├── 没有 add 的文件 = Git 不关心它
    └── Desktop 中此操作被隐藏（Commit 时自动执行）
```

### 7.4 git commit——暂存区 → 本地存储库

```
git commit = 将暂存区的所有改动保存到本地存储库

操作：git commit -m "提交信息"

方向：Staging Area → Local Repository

做三件事：
    ① 把暂存区的内容生成 Commit 对象
    ② 存储到 .git/objects/ 中
    ③ 更新当前分支的引用（refs/heads/xxx）

重要：
    ├── 每次 Commit = 仓库的一个完整快照
    ├── Commit 形成历史链路，可回溯
    └── Commit 只保存到本地，还没上 GitHub
```

### 7.5 git push——本地存储库 → 远端存储库

```
git push = 将本地存储库的 Commit 推送到远端存储库

操作：git push origin main

方向：Local Repository → Remote Repository

在 Desktop 中：点击 "Push origin" 按钮

含义：把本地的 Commit 发送到 GitHub，让其他人能看到

注意：
    ├── Push 前确保本地有最新的远端代码（否则可能被拒）
    └── Desktop 中会显示 "N↑" 表示有 N 个本地 Commit 未推送
```

### 7.6 git fetch——远端存储库 → 本地存储库

```
git fetch = 获取远端的最新更新（但不合并到工作区）

操作：git fetch

方向：Remote Repository → Local Repository（仅更新本地存储库）

在 Desktop 中：点击 "Fetch origin"

做一件事：
    把远端的新 Commit 下载到本地存储库
    但不会修改你的工作区文件

含义：先"看一看"远端有什么新东西，但先不合并
```

### 7.7 git merge——本地存储库 → 工作区

```
git merge = 将本地存储库中 Fetch 下来的远端更新合并到工作区

操作：git merge origin/main

方向：Local Repository → Working Directory

做一件事：
    把 Fetch 下来的远端 Commit 合并到当前工作区的文件中

含义：把远端代码真正"融入"你的本地文件
```

### 7.8 git pull = git fetch + git merge

这是一个非常关键的认知：

```
git pull = git fetch + git merge

分解：
    git fetch   ←  第一步：获取远端更新（存到本地存储库）
    git merge   ←  第二步：合并到工作区

    git pull    ←  一步完成上面两步
```

| 命令 | 做了什么 | 分几步 | 适用场景 |
|---|---|---|---|
| **git fetch** | 只下载远端更新到本地存储库 | 1 步 | 想先看看远端改了什么，不急着合并 |
| **git pull** | 下载远端更新 + 直接合并到工作区 | 1 步（内部 2 步） | 确定要同步远端代码 |

```
为什么有时候要分两步？

    git fetch                  → "让我看看远端有什么新东西"
    查看远端改动                   → "哦，同事改了 test5.py，跟我改的不冲突"
    git merge                  → "好，合并进来"

    vs

    git pull                   → 直接下载 + 合并，不给你中间"看看"的机会
```

> 💡 **一句话总结**：`git pull` 同时更新了**本地存储库**和**本地工作区**。`git fetch` 只更新本地存储库，工作区不变。

```
修改 test3.py：
```

**在 Desktop 中查看改动：**

```
Desktop 显示的改动：

    test.py     🗑️  Deleted   ← 删除
    test3.py    ✏️  Modified   ← 修改（第3行从"拼音"变成"爬爬虾"）
```

### 8.3 提交并推送到远端

```
Step 1：在 Desktop 左下角填写 Commit Message：
        "修改三个 test 文件"
    ↓
Step 2：点击 "Commit to main"
        （此时改动保存到了本地存储库）
    ↓
Step 3：Desktop 顶部出现 "Push origin" 按钮
        旁边显示 "1↑" = 本地有 1 个 Commit 还没推送
    ↓
Step 4：点击 "Push origin"
    ↓
Step 5：在 GitHub 网页端刷新 → 看到新的 Commit 和文件改动
        Commit Message："修改三个 test 文件"
        时间："1 minute ago"
    ↓
🎉 本地改动已同步到 GitHub！
```

```
Desktop 中的箭头标记：

    1↑  = 本地有 1 个 Commit 未推送到远端
    1↓  = 远端有 1 个 Commit 未拉取到本地
    0   = 本地和远端完全同步
```

### 8.4 从远端拉取更新

模拟场景：同事在 GitHub 网页端直接修改了代码。

```
Step 1：在 GitHub 网页端 → 进入 git-test 仓库
        → Add file → Create new file
        → 文件名：test5.py
        → 内容：print("远端新增的文件")
        → Commit changes
    ↓
    现在远端比本地多了一个 Commit！
    ↓
Step 2：回到 Desktop → 点击 "Fetch origin"
        （获取远端最新状态）
    ↓
Step 3：按钮变成 "Pull origin 1↓"
        意思是远端有 1 个 Commit 需要拉取
    ↓
Step 4：点击 "Pull origin"
    ↓
Step 5：Show in Explorer → test5.py 出现在本地文件中了！
    ↓
🎉 远端改动已同步到本地！
```

### 8.5 远端和本地同时有改动时

这是一个**非常常见的场景**：你在本地改了代码，同事也在远端（网页端）改了代码——双方都不知道对方改了。

```
场景设置：

    远端（GitHub 网页）：修改了 test5.py，添加了一行内容，Commit
    本地（你的电脑）：  修改了 test3.py，也做了改动

时间线：
    ① 远端先 Commit（你不知道）
    ② 你在本地修改 test3.py → Commit → ✅ 成功（本地存储库）
    ③ 你点击 Push → ❌ 报错！
```

**为什么会报错？**

```
Push 被拒绝的原因：

    远端 test5.py 有新的 Commit
        ↓
    你的本地存储库没有这个新 Commit
        ↓
    如果强行 Push，远端的 Commit 链会出问题
        ↓
    Git 拒绝 Push，要求你先同步远端更新
```

**Desktop 中的错误提示：**

```
GitHub Desktop 不能推送这个更新到远端分支，
因为远端有一个更新的提交，但是那个提交还没有出现在你的本地。

解决步骤：
    ① 先 Fetch → 把远端更新下载到本地存储库
    ② 再 Pull → 把远端更新合并到工作区
    ③ 最后 Push → 把你的本地 Commit 推上去
```

**实操解决：**

```
Step 1：点击 "Fetch" → 获取远端更新
    ↓
Step 2：按钮变成 "Pull origin"（1↑ 1↓）
        意思：本地有 1 个 Commit 要推送，远端有 1 个 Commit 要拉取
    ↓
Step 3：点击 "Pull origin"
        远端改动合并到本地工作区 ✅
    ↓
Step 4：现在再点击 "Push origin"
    ↓
🎉 推送成功！
    远端和本地都包含了对方的最新改动。
```

```
整个过程：

    本地：test3.py 的改动       远端：test5.py 的改动
         │                          │
         ├── Commit ✅               ├── Commit ✅
         │                          │
         ├── Push ❌ 被拒            │
         │                          │
         ├── Fetch ←────────────────┤
         │                          │
         ├── Pull（合并）            │
         │                          │
         └── Push ─────────────────→ ✅ 成功！
```

> ⚠️ **注意**：老师在这个例子中特意强调——**本地改的是 test3，远端改的是 test5，是不同的文件**，所以没有冲突。如果改的是**同一个文件的同一行**，会出现**冲突（Conflict）**，需要单独一节课来讲解。

```
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 十、常见问题与排错指南

### 10.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| Push 被拒："remote has newer commits" | 远端有本地没有的新 Commit | Fetch → Pull → Push（三步走） | 🔴 高 |
| Desktop 中看不到暂存区 | Desktop 故意隐藏了暂存区 | Commit 时自动执行了 Add，不需要手动操作暂存区 | 🟢 低 |
| Pull 后本地改动被覆盖 | 远端和本地改了同一文件的同一行 | 这会产生冲突，需要手动解决（后续课程会讲） | 🔴 高 |
| Fetch 后文件没变化 | Fetch 只更新本地存储库，不更新工作区 | 还需要 Pull（即 Merge）才能让工作区文件变化 | 🟡 中 |
| Desktop 显示 3↑ 是什么意思 | 本地有 3 个 Commit 还没推送到远端 | 点击 Push origin | 🟢 低 |

### 10.2 排查流程

```
代码同步出问题了？
    │
    ├── 不知道怎么同步？
    │       记住口诀：
    │       "有网就用 Push/Pull，没网先 Commit 存本地"
    │
    ├── Push 被拒绝？
    │       ├── 先 Fetch → 看看远端有什么新东西
    │       ├── 再 Pull → 把远端更新合并到本地
    │       └── 最后 Push → 把你的改动推上去
    │
    ├── 不确定本地和远端是否同步？
    │       看 Desktop 顶部的箭头标记：
    │       ↑N = 本地 N 个 Commit 未推送
    │       ↓N = 远端 N 个 Commit 未拉取
    │       0 = 完全同步 ✅
    │
    └── Pull 后出现冲突提示？
            这是"同一文件同一行"被双方同时修改了
            → 需要手动解决冲突（后续课程会详细讲解）
```

## 十二、课程小结

### 12.1 核心知识图谱

```
┌────────────────────────────────────────────────────────────┐
│              Git 四个分区 核心知识体系                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1️⃣  四个分区——代码的四种"存放位置"                             │
│      ├── 工作区（Working Directory）= 本地文件夹，你编辑代码的地方│
│      ├── 暂存区（Staging Area）= Commit 前的候车室            │
│      │   ⚠️ Desktop 隐藏了暂存区，Commit 时自动 Add           │
│      ├── 本地存储库（Local Repository）= .git 文件夹，存 Commit│
│      └── 远端存储库（Remote Repository）= GitHub 网站         │
│                                                            │
│  2️⃣  七大操作——在分区之间移动代码                                │
│      ├── clone  ：远端 → 工作区 + 本地存储库（首次）           │
│      ├── add    ：工作区 → 暂存区（挑选要提交的改动）           │
│      ├── commit ：暂存区 → 本地存储库（保存快照）              │
│      ├── push   ：本地存储库 → 远端存储库（共享/备份）          │
│      ├── fetch  ：远端存储库 → 本地存储库（下载不看）           │
│      ├── merge  ：本地存储库 → 工作区（合并到文件）             │
│      └── pull   ：远端 → 本地存储库 + 工作区（fetch+merge）    │
│                                                            │
│  3️⃣  Desktop 中的简化                                        │
│      ├── Commit 按钮 = git add + git commit（两步合一）       │
│      ├── ↑N = 本地 N 个 Commit 未推送                        │
│      ├── ↓N = 远端 N 个 Commit 未拉取                        │
│      └── Push 被拒 → Fetch → Pull → Push（三步解决）         │
│                                                            │
│  4️⃣  关键口诀                                                │
│      ├── 编辑在工作区 → Add 到暂存区 → Commit 到本地           │
│      ├── Push 上远端 → Pull 下本地                           │
│      └── Pull = Fetch + Merge                               │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 12.2 一句话总结

> Git 的本质就是四个分区之间的文件同步——编辑在工作区，挑选进暂存区，Commit 存本地，Push 上远端，Pull 拉回来；理解了这张地图，Git 中的每个操作都有了方向和意义。
