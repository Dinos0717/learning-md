## 一、课程背景与目标

### 1.1 前情回顾

在前面课程中，你已经掌握了：

- **Git 四个分区** —— 工作区 → 暂存区 → 本地存储库 → 远端存储库
- **基本操作** —— Clone、Commit、Push、Pull、Fetch
- **GitHub Desktop 基础** —— 克隆、分支管理、Fork、本地文件夹初始化

但你有没有遇到过这些情况：
- 改了 3 个地方，只想提交其中 1 个？
- 改了一半代码，突然要切换分支修 Bug？
- Commit 完了才后悔，想撤销？
- Commit Message 写错了想改？

这节课就是你的**后悔药大全**。

### 1.2 本节课的核心问题

| 问题 | 说明 |
|---|---|
| 只想提交文件中的部分改动怎么做？ | 在 Desktop 中勾选/取消勾选具体行 |
| 代码写了一半要切换分支怎么办？ | Stash 暂存、或 Commit、或 Discard、或直接带过去 |
| Commit 完后悔了能撤销吗？ | Reset（撤回）、Revert（反向抵消）、Amend（修正） |
| 已 Push 的 Commit 怎么撤销？ | Revert（推荐）或 Reset + Force Push（危险） |
| 怎么给版本打标签？ | 用 Tag 标记里程碑（如 v1.0.0） |

> ⚠️ **关键认知**：这节课的操作很多是"不可逆"或"有风险"的。我会标注每个操作的安全等级。多人协作项目尤其要注意——有些操作只能在你自己一个人的分支上用。

### 1.3 本课要掌握的目标

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  目标一：掌握日常开发中的灵活操作                                 │
│         ├── 部分提交（勾选/取消具体改动行）                       │
│         ├── 放弃更改（Discard changes）                      │
│         └── 临时存储（Stash）并还原                           │
│                                                             │
│  目标二：理解四种"后悔药"的区别和使用场景                          │
│         ├── Discard：放弃未提交的改动                          │
│         ├── Reset：撤回 Commit（可能需要 Force Push）          │
│         ├── Revert：用新 Commit 抵消旧 Commit（最安全）         │
│         └── Amend：修正最新一次 Commit                        │
│                                                             │
│  目标三：知道什么时候能 Force Push，什么时候绝对不能               │
│         ├── 个人分支 → 可以 Force Push                        │
│         └── 多人协作分支 → 绝对不能 Force Push                 │
│                                                             │
│  目标四：学会使用 Tag 标记版本                                  │
│         ├── 创建 Tag（如 v1.0.0）                            │
│         └── 推送 Tag 到远端                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、部分提交——一个文件只提交一部分改动

### 2.1 场景说明

你修改了一个文件，里面有三处改动，但你只想提交其中一处。另外两处还没准备好。

```
修改前：
    print("开始")
    print("结束")

修改后：
    print("开始")
    print("第一部分")    ← 想提交
    print("第二部分")    ← 不想提交（还没写完）
    print("第三部分")    ← 不想提交（还在调试）
    print("结束")
```

### 2.2 操作步骤

```
在 Desktop 中，点击有改动的文件
    ↓
左侧显示代码 Diff（差异对比）
    ↓
每一处改动旁边都有 ☑️ 勾选框
    ├── ☑ 第一部分   ← 保留勾选 → 会提交
    ├── ☐ 第二部分   ← 取消勾选 → 不提交
    └── ☐ 第三部分   ← 取消勾选 → 不提交
    ↓
填写 Commit Message → 点击 Commit
    ↓
🎉 只有"第一部分"被提交了！
    "第二部分"和"第三部分"还留在你的本地文件中
    （可以之后勾上再提交，或继续修改）
```

### 2.3 部分提交 vs 全部提交

```
全部提交（默认）：
    ├── Desktop 中所有改动默认勾选
    └── Commit = 所有改动一起提交

部分提交（进阶技巧）：
    ├── 手动取消某些改动的勾选
    ├── Commit = 只提交勾选的改动
    └── 未勾选的改动留在工作区，下次再处理
```

| 场景 | 推荐方式 |
|---|---|
| 功能 A 和功能 B 都写完了 | 全部提交 |
| 功能 A 写完，功能 B 写了一半 | **部分提交**：只提交功能 A 的代码 |
| 修了两个 Bug，一个验证通过一个还没测 | **部分提交**：只提交已验证的 Bug 修复 |

> 💡 **部分提交是最佳实践**：一次 Commit 只做一件事。不要把"修 Bug"和"新功能"混在一个 Commit 里。部分提交让你能精确控制每次 Commit 的内容。

## 三、Discard——放弃更改

### 3.1 场景说明

你改了一堆代码，突然发现："不对，思路全错了，我想回到改动之前的状态。"

```
你把中文注释改成了英文版 → 转念一想：还是中文好 → 想全部撤销

Desk 显示：文件有改动
你想要：回到改动之前的状态（即回到本地分支最新 Commit 的状态）
```

### 3.2 操作步骤

```
在 Desktop 中 → 右键点击有改动的文件 → 选择 "Discard changes"
    ↓
弹出确认框 → 点击 "Discard changes" 确认
    ↓
🎉 文件恢复到了本地分支最新 Commit 的状态！

文件内容：
    之前：英文注释
    之后：中文注释（恢复了）
```

**Discard 的本质：**

```
Discard = 把工作区的文件强制同步成本地分支最新 Commit 的状态

方向：Local Repository → Working Directory（覆盖）
效果：你在工作区做的所有修改 → 全部丢弃 → 文件回到 Commit 时的样子

对应 Git 命令：
    单个文件：git restore <文件名>
    所有文件：git reset --hard
```

### 3.3 回收站保护机制

GitHub Desktop 有一个贴心设计：放弃更改后，**被放弃的内容会放入回收站**。

```
Discard 后：
    ├── 文件内容恢复到 Commit 时的状态 ✅
    └── 被丢弃的改动进入了回收站 🗑️

如果反悔了 → 打开回收站 → 找回文件 → 恢复改动 ✅
```

> 💡 **回收站 = 最后的保险**：Discard 不是彻底删除，你还有一次从回收站抢救的机会。但不要依赖它——如果清空了回收站或过了一段时间，就真的找不回来了。

## 四、Stash——临时存储未完成的代码

### 4.1 场景说明

这是开发者每天都会遇到的情况：

```
你正在 feature 分支上开发新功能
    → 代码写了一半，还没到能 Commit 的程度
    → 突然！生产环境出问题了，要立刻修 Bug
    → 你必须切换到 main 分支去修复
    → 但你的代码还没提交，直接切换可能会丢……
```

**四种处理方案：**

| 方案 | 操作 | 推荐度 | 说明 |
|---|---|---|---|
| ✅ **Commit** | git add . → git commit | ⭐⭐⭐ 最推荐 | 最简单、最安全 |
| ✅ **Stash** | git stash | ⭐⭐⭐ 推荐 | 暂存起来，修完 Bug 再取回 |
| ⚠️ **Discard** | git reset --hard | ⭐⭐ 谨慎 | 放弃改动，从头再来 |
| ❌ **直接切换** | 把改动带到新分支 | ⭐ 不推荐 | 会把半成品代码带到 main |

### 4.2 操作步骤

**使用 Stash：**

```
Desktop 中 → 文件有改动但不想提交
    ↓
右键点击有改动的文件 → 选择 "Stash All Changes"
    ↓
改动被存储到 Stash 中
    ↓
本地文件恢复到改动前的状态
    ↓
现在可以放心切换分支了！
    ↓
修完 Bug 后 → 切回 feature 分支
    ↓
Desktop 中显示 "Stashed Changes" → 点击展开
    └── 两个选项：
        ├── Restore → 还原改动（继续刚才的开发）
        └── Discard → 丢弃存储（不想要了）
    选择 Restore
    ↓
🎉 代码回来了！继续开发。
```

### 4.3 Desktop 的 Stash 局限性

GitHub Desktop 的 Stash 有一个重要限制：

```
GitHub Desktop：最多只能存储 1 次 Stash
    └── 新的 Stash 会覆盖旧的

Git 命令行：可以存储无限次 Stash
    └── 用 git stash list 查看所有存储
```

> 💡 **如果需要多次 Stash**：后面课程讲到 Git 命令行时会详细介绍。目前先用 Desktop，一次 Stash 足够覆盖大部分场景。

### 4.4 不 Stash 直接切换分支会怎样

如果你试图**不处理改动直接切换分支**，Desktop 会弹出提示，给你两个选择：

```
┌─────────────────────────────────────────────────────────────┐
│  你有未提交的改动，如何处理？                                    │
│                                                             │
│  选项一：Leave your changes on feature                      │
│         = 把改动 Stash 到 feature 分支（推荐）                 │
│         和手动 Stash 的效果一样                               │
│                                                             │
│  选项二：Bring my changes to main                           │
│         = 把改动带到 main 分支                                │
│         ⚠️ 不推荐！会把半成品代码带到主干                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> ⚠️ **选"带到新分支"的后果**：你的半成品代码会出现在 main 分支上。如果你不小心在 main 上 Commit 了……那就等于把没写完的功能合入了主干。所以**不推荐这个选项**。

### 4.5 四种处理方式总结

```
开发了一半，需要切换分支 → 四种选择：

    ┌─ ① Commit（最推荐）──────────────────────────────┐
    │   git add . + git commit -m "WIP: 开发中"         │
    │   优点：最安全，代码被永久保存                        │
    │   缺点：多了一条不完整的 Commit 记录                  │
    │   适用：随时可以 Commit 的情况                       │
    ├───────────────────────────────────────────────────┤
    │                                                    │
    ├─ ② Stash（推荐）─────────────────────────────────┤
    │   git stash                                        │
    │   优点：Commit 历史保持干净，随时取回                  │
    │   缺点：Desktop 只能存 1 个 Stash                    │
    │   适用：代码还没到能 Commit 的程度                    │
    ├───────────────────────────────────────────────────┤
    │                                                    │
    ├─ ③ Discard（谨慎）───────────────────────────────┤
    │   git reset --hard                                 │
    │   优点：最彻底                                       │
    │   缺点：改动彻底消失（回收站可抢救一时）               │
    │   适用：确定这些改动完全不需要了                       │
    ├───────────────────────────────────────────────────┤
    │                                                    │
    └─ ④ 直接切换（不推荐）────────────────────────────┤
       git switch <分支名>                                │
       优点：最快                                         │
       缺点：半成品代码污染目标分支                          │
       适用：几乎没有适用场景                               │
```

## 五、Reset——撤回提交

### 5.1 场景说明

Commit 做完了，突然发现："不该提交这个！"或者"Commit Message 写错了！"

Reset 就是用来**撤回 Commit** 的操作。

### 5.2 撤回到上一个 Commit

**未 Push 到远端时——Desktop 可以直接操作：**

```
Desktop → History 选项卡 → 找到你想撤销的那次 Commit 的"上一次" Commit
    ↓
右键点击 → 选择 "Reset to commit"
    ↓
含义：把本地分支还原到选中的那个 Commit 的状态
    效果：撤销了这个 Commit 之后的所有 Commit
    ↓
结果：
    ✅ Commit 记录被撤销（History 中消失了）
    ✅ 文件改动还在（代码没有被删！）
    ↓
🎉 你可以重新修改代码，再 Commit 一次。
```

```
示例：

    Commit 3 ← 最新（想撤销这次）
    Commit 2 ← 在这次的 Commit 上右键 → Reset to commit
    Commit 1 ←

    Reset 后：
    Commit 2 ← 现在最新的 Commit
    Commit 1 ←
    （Commit 3 消失了，但 Commit 3 的文件改动还保留在工作区）
```

### 5.3 已 Push 的 Commit 能 Reset 吗

**如果 Commit 已经 Push 到远端了——Desktop 不允许 Reset：**

```
已 Push 的 Commit → 右键 → "Reset to commit" 按钮是灰色的
    ↓
Desktop 故意禁用了这个操作 → 防止你误操作导致远端代码混乱
```

> 💡 **Desktop 在保护你**：对已 Push 的 Commit 做 Reset，意味着你本地的历史链路和远端的不一致了——这是高危操作。

### 5.4 命令行强制 Reset + Force Push

如果确实需要撤销已 Push 的 Commit（比如：这个分支只有你一个人用），可以用命令行：

```
Step 1：右键仓库 → "Open in Command Prompt"（打开命令提示符）
    ↓
Step 2：查看 Commit 历史
    git log
    （找到你想回退到的那个 Commit 的 ID）
    ↓
Step 3：执行 Reset
    git reset --mixed <commit-id>
   （--mixed = 撤回 Commit，但保留文件改动）
    ↓
Step 4：此时 Desktop 显示本地落后于远端（1↓）
    因为远端有那个 Commit，你本地撤回了 → 不一致！
    不能点 Pull（那样会把撤销的 Commit 又拉回来）
    ↓
Step 5：强制推送
    git push -f
    （-f = force，强制覆盖远端）
    ↓
🎉 远端也被重置了！
```

> ⚠️ **再次强调**：`git push -f` 是极其危险的操作。

### 5.5 ⚠️ Force Push 的禁忌

```
┌─────────────────────────────────────────────────────────────┐
│              ⚠️  Force Push 铁律                              │
│                                                             │
│  ✅ 可以 Force Push 的情况：                                  │
│      ├── 这个分支只有你一个人在开发                            │
│      ├── 你刚推上去的代码，发现有问题，立刻撤回                 │
│      └── 你确认不会有其他人基于你的 Commit 继续开发             │
│                                                             │
│  ❌ 绝对不能 Force Push 的情况：                               │
│      ├── main / master 主干分支                             │
│      ├── 多人协作的开发分支                                   │
│      └── 任何其他人可能基于此分支开发的情况                     │
│                                                             │
│  Force Push 的后果：                                         │
│      你强行覆盖远端 → 别人基于旧 Commit 做的改动               │
│      → 历史链路断裂 → 别人的代码可能丢失！😱                    │
│                                                             │
│  结论：多人协作分支 = 绝对不能用 Force Push                     │
│        个人分支 = 可以用，但要小心                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 六、Revert——安全的"后悔药"

### 6.1 Revert 是什么

**Revert** 和 Reset 目的相同（撤销改动），但方式完全不同：

```
Reset：直接删除 Commit 记录（改变历史）
Revert：创建一个"反向"的新 Commit 来抵消旧 Commit（不改变历史）

类比：
    Reset = 把写错的这一页从日记本上撕掉
    Revert = 在下一页写上"上一页作废，改成这样……"
```

### 6.2 操作步骤

```
Desktop → History 选项卡 → 找到想撤销的那次 Commit
    ↓
右键点击 → 选择 "Revert changes in commit"
    ↓
Desktop 自动创建一个新的 Commit：
    标题："Revert '原来的 Commit Message'"
    内容：把那次 Commit 的所有改动反向操作
        ├── 原来的新增 → 变成删除
        ├── 原来的删除 → 变成新增
        └── 原来的修改 → 反向修改回去
    ↓
结果：
    新增了一条 Revert Commit
    原来的 Commit 还在（历史没有被改变）
    代码回到了原来的状态 ✅
    ↓
点击 Push → 推送到远端（不需要 Force Push！）
    ↓
🎉 远端也安全地撤销了改动！
```

```
Revert 后的历史记录：

    Commit 3: Revert "更新 test5"  ← 新生成的，抵消了 Commit 2
    Commit 2: 更新 test5           ← 还在！没有被删除
    Commit 1: 初始提交

    效果：代码回到了 Commit 1 的状态
    但历史中保留了完整的记录（Commit 2 + 它的 Revert）
```

### 6.3 Reset vs Revert 对比

```
┌─────────────────────────────────────────────────────────────┐
│              Reset vs Revert 对比                            │
├──────────────┬──────────────────┬───────────────────────────┤
│              │ Reset             │ Revert                    │
├──────────────┼──────────────────┼───────────────────────────┤
│ 原理          │ 删除 Commit 记录   │ 新建反向 Commit 抵消       │
│ 历史是否被改   │ ✅ 改变了历史      │ ❌ 历史完整保留            │
│ 是否产生新 Commit│ ❌ 不产生        │ ✅ 产生一个新 Commit       │
│ 已Push能否用   │ 需要 Force Push   │ 可以直接 Push            │
│ 多人协作分支    │ ❌ 不能用          │ ✅ 可以用（推荐！）        │
│ 安全性         │ ⚠️ 危险           │ ✅ 安全                   │
│ 使用场景       │ 个人分支、未Push   │ 多人协作、已Push          │
└──────────────┴──────────────────┴───────────────────────────┘
```

> 💡 **一句话总结**：**多人协作用 Revert，个人分支用 Reset**。Revert 永远是最安全的选择。

## 七、Amend——修正最新一次提交

### 7.1 场景说明

Commit 刚做完，发现有个小问题：

```
刚提交的 Commit：
    代码：print("helllo word")  ← "hello" 拼错了，"world" 也拼错了
    Message："更新 test5"       ← "更新" 写成了"更新"

你想要：
    修改代码（改拼写）+ 修改 Commit Message
    但不想要产生新的 Commit 记录
```

**Amend = 修正最近一次 Commit 的内容或 Message，不产生新 Commit。**

### 7.2 修改未 Push 的 Commit

```
Desktop → History 选项卡 → 最新 Commit 上右键 → 选择 "Amend commit"
    ↓
进入 Amend 编辑模式：
    ├── 可以直接修改 Commit Message（例如："更新" → "更新"）
    └── 可以直接修改代码文件（例如：修正拼写错误）
    ↓
修改完成后 → 点击 "Amend last commit"
    ↓
结果：
    ✅ 最后一次 Commit 的内容和 Message 都被修正了
    ✅ 没有产生新的 Commit
    ✅ History 中还是那一条记录
```

### 7.3 修改已 Push 的 Commit

如果 Amend 一个已经 Push 的 Commit：

```
Desktop 会弹出警告：
    "这次修改完成后，你必须使用 Force Push 才能推送到远端"

    → 同样的规则：个人分支可以，多人协作分支不行

如果你确认可以 Force Push → 点击 "Begin amend"
    ↓
修改 Message 和代码
    ↓
点击 "Amend last commit"
    ↓
Desktop 显示 "Force push origin"
    ↓
点击 → 确认警告 "I'm sure"
    ↓
🎉 修正已推送到远端！
```

> ⚠️ **同样的 Force Push 规则**：已 Push 的 Commit 做 Amend 后，必须 Force Push。所以只在自己一个人的分支上这么做。

### 7.4 Amend 的适用范围

| 情况 | Amend 能用吗 |
|---|---|
| 修正**最新一次** Commit | ✅ 可以 |
| 修正**历史** Commit（非最新） | ❌ 不可以 |
| 最新的 Commit **未 Push** | ✅ 可以，不需要 Force Push |
| 最新的 Commit **已 Push** | ⚠️ 可以，但需要 Force Push |
| 多人协作分支 | ❌ 不能用（涉及 Force Push） |

## 九、Tag——为里程碑打标签

### 9.1 Tag 是什么

**Tag（标签）** 是把某一次 Commit **标记为项目的重要事件或里程碑**。

```
Tag 的作用：
    ├── 标记版本发布（v1.0.0、v2.0.0）
    ├── 标记重要节点（"上线前最后版本"）
    └── 方便以后快速定位某个历史状态
```

### 9.2 创建 Tag

```
Desktop → History 选项卡 → 找到想打标签的那次 Commit
    ↓
右键点击 → 选择 "Create tag"
    ↓
输入 Tag 名称（例如 v1.0.0）
    ↓
点击 "Create tag"
    ↓
🎉 Tag 创建完成！这次 Commit 旁边多了一个标签标记。
```

**Tag 命名规范：**

```
语义化版本号（Semantic Versioning）：

    v1.0.0
    │ │ │
    │ │ └── Patch（补丁）：Bug 修复
    │ └──── Minor（小版本）：新增功能，向后兼容
    └────── Major（大版本）：重大变更，可能不兼容

示例：
    v1.0.0  → 第一个正式版本
    v1.0.1  → 修了一个 Bug
    v1.1.0  → 新增了一个功能
    v2.0.0  → 重大更新，API 不兼容旧版
```

### 9.3 推送 Tag 到远端

Tag 创建后只在本地——需要推送到远端：

```
Desktop → 点击 "Push origin"
    ↓
Tag 和 Commit 一起推送到 GitHub
    ↓
在 GitHub 网页端 → 仓库的 Releases 页面 → 能看到这个 Tag
    ↓
🎉 可以基于这个 Tag 创建正式的 Release（发布版本）！
```

### 9.4 Tag 的典型用途

| 用途 | 说明 |
|---|---|
| **版本发布** | `v1.0.0` → `v1.1.0` → `v2.0.0` |
| **快速回溯** | 在 100 个 Commit 中快速找到"v1.0.0 发布"的那个 |
| **Release 页面** | GitHub 的 Releases 功能基于 Tag 来创建 |
| **下载特定版本** | 用户可以通过 Tag 下载特定版本的代码 |

```
Tag 和 Branch 的区别：

    Branch = 指针会随着新 Commit 自动移动（main 永远指向最新）
    Tag    = 指针固定不动，永远指向那个特定的 Commit

    想回到 v1.0.0 发布时的代码？找 v1.0.0 Tag 就行了！
```

## 十一、常见问题与排错指南

### 11.1 错误速查表

| 错误现象 | 原因 | 解决方法 | 优先级 |
|---|---|---|---|
| Desktop 中 Reset 按钮是灰色的 | Commit 已经 Push 到远端了 | 用 Revert 代替，或命令行 Reset + Force Push | 🟡 中 |
| Stash 后找不到了 | Desktop 只存一个 Stash，可能被覆盖了 | 在 Desktop 中查看 "Stashed Changes" 区域 | 🔴 高 |
| Force Push 后别人的代码丢了 | 在多人协作分支上用了 Force Push | 立即联系团队，从备份恢复；**记住教训：永远不要在共享分支 Force Push** | 🔴 高 |
| Discard 后后悔了 | 想恢复被放弃的改动 | 检查回收站！Desktop 会把放弃的改动放入回收站 | 🟢 低 |
| Amend 后不能 Push | Commit 已 Push，需要 Force Push | 确认分支是否只有你一个人用 → Force Push；否则用 Revert | 🟡 中 |
| Revert 后又想撤销 Revert | Revert 本身也是一个 Commit | 再对 Revert Commit 做一次 Revert（负负得正） | 🟢 低 |

### 11.2 排查流程

```
想撤销什么东西？
    │
    ├── 还没 Commit 的改动？
    │       ├── 只想撤销一个文件？→ 右键 → Discard changes
    │       ├── 想暂存起来？→ Stash All Changes
    │       └── 想全部丢弃？→ Discard changes（回收站有备份）
    │
    ├── 已经 Commit 但还没 Push？
    │       ├── 想完全撤销？→ Reset to commit
    │       ├── 只想改 Message？→ Amend commit
    │       └── 想保留历史记录？→ Revert（虽然没 Push 也可以用）
    │
    ├── 已经 Commit 且已 Push？
    │       ├── 多人协作分支？→ Revert（唯一安全的选择）
    │       ├── 个人分支？→ Revert（推荐）或 Reset + Force Push
    │       └── 只改最新 Commit？→ Amend（需 Force Push，仅个人分支）
    │
    └── 打标签？
            ├── 想标记版本？→ 右键 Commit → Create tag
            └── 创建后远端看不到？→ 点击 Push origin
```

## 十二、课后练习

### 练习一：部分提交练习

**目标：** 掌握只提交文件中部分改动的方法。

**要求：**
- [ ] 在本地仓库中，编辑一个文件，添加 3 处不同的改动
- [ ] 在 Desktop 中观察 Diff 视图，确认 3 处改动都有 ☑️ 勾选框
- [ ] 取消其中 2 处的勾选，只保留 1 处
- [ ] 填写 Commit Message 并提交
- [ ] 确认提交后，另外 2 处改动还保留在本地文件中
- [ ] 将另外 2 处改动也提交（或 Discard）

### 练习二：Stash 练习

**目标：** 掌握临时存储和恢复未完成代码的方法。

**要求：**
- [ ] 在 feature 分支上编辑 README.md（模拟开发了一半的功能）
- [ ] 使用 Stash All Changes 暂存改动
- [ ] 切换到 main 分支，确认 README.md 没有改动
- [ ] 切回 feature 分支，Restore Stash
- [ ] 确认改动已恢复
- [ ] 再试一次：这次不 Stash，直接切换分支，观察 Desktop 的提示

### 练习三：Reset 和 Revert 对比

**目标：** 亲身体验 Reset 和 Revert 的区别。

**要求：**
- [ ] 在个人分支上做一次 Commit（Commit 1），Push 到远端
- [ ] 再做一次 Commit（Commit 2），**不 Push**
- [ ] 对 Commit 2 使用 Reset to commit → 观察：Commit 2 消失了，但文件改动还在
- [ ] 再做 Commit 2，这次 Push 到远端
- [ ] 对 Commit 2 使用 Revert changes in commit
- [ ] 观察：新增了一个 Revert Commit，Push 后远端也被撤销了
- [ ] 在 History 中对比两种操作的差别并写下来

### 练习四：Amend 练习

**目标：** 掌握修正最新 Commit 的方法。

**要求：**
- [ ] 在个人分支上做一次 Commit（故意把 Message 写错，代码也写错）
- [ ] **不 Push**，使用 Amend commit 修正 Message 和代码
- [ ] 确认 History 中只有一条记录，Message 和代码都已被修正
- [ ] Push 后，再做一次 Amend → 观察 Force Push 警告
- [ ] 不做 Force Push，而是用 Revert 来撤销（比较两种方式的体验）

### 练习五：创建版本 Tag

**目标：** 学会用 Tag 标记项目里程碑。

**要求：**
- [ ] 在 main 分支的最新 Commit 上创建 Tag：`v0.1.0`
- [ ] Push origin（包括 Tag）
- [ ] 在 GitHub 网页端的 Releases 页面确认 Tag 存在
- [ ] 再做一次 Commit → 创建 Tag：`v0.2.0`
- [ ] Push origin
- [ ] 在 History 中观察两个 Tag 分别指向不同的 Commit
