## 一、课程引入与前情回顾——从工程化控制到工具管理

### 1.1 课堂开场与直播花絮

课程开始时，老师正在调试直播推流工具 OBS（Open Broadcaster Software），发现画面配置与预期有出入。老师简单说明了当前的技术调整状态后，迅速切换到正式的授课内容。

> ⚠️ **注意事项**：这类技术调试环节在真实工程教学中不可避免——老师采用的是"先跑起来、再逐步优化"的策略，这与我们后续讲到的工程化迭代思维一脉相承：不必追求一步到位，但要确保每一步都可运行、可观察。

这一个小插曲也暗示了本课的主题：**当系统中的组件（工具、配置、流程节点）越来越多时，靠手工、靠记忆、靠约定俗成来管理，迟早会碰到天花板。** OBS 的调试只是一个人面对一个小工具的配置问题，而我们要讨论的，是几十上百个工具在大模型 Agent 体系中的系统化管理。

### 1.2 本课命题的提出——"工具多了，字典式管理还够用吗？"

老师用一连串追问切入今天的核心命题：

> "管理的这个过程里面，我是不是可以有更优的一些方法去把工具能够管理起来？比如说，我是一定要使用字典式的这样一个管理方式吗？工具一多了之后，它能不能用更优的一种管理方式，能够更好地让我们的模型根据自己的实际需要去动态地选择？"

这段话包含了三个层层递进的追问：

1. **字典式管理是唯一解吗？** ——前几课中，工具以 Python 字典（`dict`）形式存储，`"工具名" → {desc, args, func}` 的映射简单直接，但这不是唯一的管理方式，也不一定是最优的。
2. **工具多了之后怎么办？** ——当工具数量从 3-5 个增长到 20+ 个，字典的维护成本、可发现性、复用性都会暴露出结构性问题。
3. **能否让模型动态选择？** ——不是每次把所有工具都告诉模型（全量注入），而是根据当前场景和意图，让模型只看到它需要的那几个。

这三个追问共同指向一个主题：**从"能用"到"系统化管理"的跃迁。** 前几课解决的是"如何让模型调用工具"，本课要解决的是"当工具多到管不过来时，如何建立一套体系让工具可注册、可查询、可按需激活、可替换执行"。

> 💡 **核心洞见**：工具管理的问题不是"工具写错了"，而是**管理层面缺少基础设施**。就像代码多了需要模块化、数据库多了需要 ORM、服务多了需要注册中心——工具多了，也需要一套相应的管理体系。

### 1.3 前课知识体系回顾——从字典到四阶段演进

在提出本课命题后，老师带领学员系统回顾了第 6 课到第 8 课的知识演进路径。这不是简单的"上一课讲了什么"的罗列，而是一条清晰的**工程化能力递进链条**。

#### 1.3.1 第 6 课：字典管理 + Function Calling

第 6 课的核心成果是让模型能够调用外部工具（Function Calling）。当时的实现方式是**手动字典管理工具**——在实际的 s05 代码中，关键结构如下：

```python
# 第 6 课 s05 的实际写法：手动字典管理工具
REACT_TOOLS = {
    "get_weather": {
        "desc": "查询指定城市的当前天气状况...",
        "args": {"city": "城市名称"},
        "func": _call_weather,
    },
    "get_exchange_rate": {
        "desc": "查询两种货币之间的实时汇率",
        "args": {"from_currency": "源货币代码", "to_currency": "目标货币代码"},
        "func": _call_exchange_rate,
    },
}

# 全部工具一次性转为 schema，塞进 prompt
tools_schema = [{"name": k, "desc": v["desc"], "args": v["args"]}
                for k, v in REACT_TOOLS.items()]

# TriggerFlow ReAct Loop：reason → act → reason → ...
flow = TriggerFlow(name="react-loop")
flow.to(reason)                       # PLAN: 模型决定调哪个工具
flow.when("Act").to(act_and_observe)  # EXECUTE: 执行工具
flow.when("Reason").to(reason)        # 回填结果，继续循环
```

> ❓ **常见疑问**：字典管理在当时只有 3-5 个工具时完全够用，为什么现在要"小题大做"？
>
> 这恰恰是本课的逻辑起点——**字典管理本身没有错，但它不具备扩展性。** 当工具数量增长一个数量级时，"字典定义 + 全量注入"的写法会暴露出三个结构性问题（详见下一章）。

#### 📖 第 6 课的关键认知

老师特别强调：第 6 课不仅仅是"教会模型调用工具"这么简单。模型调用工具的背后，其实是一个多步骤的认知过程：

```
模型收到用户问题
      │
      ▼
┌─────────────────┐
│ 1. 观察工具信息   │  ← 模型需要理解每个工具能做什么
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 判断选择工具   │  ← 结合当前问题，从工具集中选出最合适的
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 生成调用参数   │  ← 按照工具的 input schema 构造参数
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. 执行并反馈     │  ← 拿到工具返回结果，整合后回应用户
└─────────────────┘
```

> 💡 **核心洞见**：Function Calling 不是"有一个工具就让模型去用"这么简单，中间的观察、判断、选择、执行每一步都是工程化的控制点。理解了这一点，才能理解为什么后续课程要引入 ReAct Loop、路由、并发等模式。

#### 1.3.2 第 7 课：ReAct Loop + 路由 + 并发 + Plan 模式

第 6 课解决的是"一次调用一个工具"的问题，但真实场景中：

- **复杂问题需要多步推理**：查天气后判断是否需要提醒穿衣 → 需要查穿衣指数 → 最终综合给出建议。这不是一次 Function Calling 能完成的。
- **不同问题需要不同处理路径**：查天气走查询路径，写报告走生成路径——需要路由分发的机制。
- **多个独立工具可以同时调用**：查询天气和查询汇率互不依赖，可以并发提升效率。

第 7 课用 TriggerFlow 在第 6 课的骨架上叠加了这些模式：

| 模式 | 解决的问题 | 核心机制 |
|---|---|---|
| **ReAct Loop**（Reason-Act 循环） | 单次推理不够，需要"观察→行动→再观察"的串行推演 | `reason → act → reason → ...` 循环，每次根据上一步的结果决定下一步 |
| **路由（Router）** | 不同意图需要分发到不同处理模块 | 模型判断意图 → 分发到对应的处理分支 |
| **并发（Concurrency）** | 多个无依赖的工具可以同时启动 | 一个复杂任务中，同时调用 3 个独立工具 |
| **Plan 模式** | 用户只说概要需求，模型需要自主规划步骤 | 模型将概要问题展开为多步行动计划，过程中可与用户交互确认 |

> 🔗 **关联知识**：ReAct Loop 是后续所有工程化模式的"循环骨架"。无论 Plan Handler 怎么替换、Execution Handler 怎么切换，PLAN → EXECUTE → 回填的循环结构是不变的——这个洞察在后续章节 4.1 中会有详细展开。

#### 🔗 推导链：从单次调用到复杂推理

```
起点: 第 6 课的 Function Calling
  模型看到问题 → 选一个工具 → 调用 → 返回结果
  │
  ├── 问题：复杂问题需要多步吗？
  │   为什么？单次工具调用只能获取一个维度的信息，
  │   无法基于中间结果调整下一步策略。
  │
  ├── 引入 ReAct Loop
  │   为什么？让模型在每次行动后重新观察结果，决定下一步做什么。
  │   这是串行的、逐步推演的推理过程。
  │
  ├── 叠加路由
  │   为什么？不同问题的最佳处理流程不同，不能都走同一个 ReAct Loop。
  │
  ├── 叠加并发
  │   为什么？无依赖的工具调用串行执行是浪费，并发可以缩短总耗时。
  │
  └── 结论: TriggerFlow 将这些模式编排在一起，
      形成了一套比"裸调 Function Calling"强大得多的工程化体系。
```

#### 1.3.3 第 8 课：四阶段演进与文档审查实战

第 8 课将前两课的所有模式整合到一个完整的**文档审查助手**项目中，展示了从"最简可用"到"工程化完整方案"的四阶段演进过程：

```
阶段一：最简可用（基础 Function Calling）
    │
    ▼
阶段二：引入 ReAct Loop（多步推理、串行推演）
    │
    ▼
阶段三：叠加路由 + 并发（场景分流、并行提效）
    │
    ▼
阶段四：工程化完整方案（注册中心、动态注入、可替换 Handler）
```

> 💡 **核心洞见**：四阶段演进不是四个独立的方案，而是**同一系统在不同工程化程度下的形态**。每个阶段都建立在上一阶段的基础上，这是工程化迭代的典型思维：不追求一步到位，但每一步都让系统更可控、更可扩展。

老师指出：到第 8 课结束时，大家已经看到了**工程化控制的全貌**——不是让模型"裸跑"，而是在模型外围构建了一套能让模型被控制、被引导、被优化的"轨道"。

### 1.4 核心认知：模型智能 vs 工程化轨道

这是本课引入阶段最重要的认知锚点。老师用大量篇幅反复强调一个看似矛盾的观点：

> "虽然看上去模型的能力是非常强的，就比如说现在能力已经非常强，但是我们依然需要去使用一些工程化的方式，在让模型解决复杂问题的时候，能够被控制起来。"

#### 🧠 直观理解

想象一匹千里马——它跑得很快、耐力很强，但如果没有任何缰绳、马鞍和骑手的引导，它在复杂地形中可能会跑偏、会受伤、会无法到达目的地。**工程化控制就是模型这匹"千里马"的缰绳和马鞍**——不是限制它的能力，而是确保它的能力能被精准地引导到目标上。

#### 📖 详细解释

老师用了一个关键的词来描述这个认知——**"harness"（缰绳/装置/轨道）**。这个词有三层含义：

1. **缰绳（harness as reins）**：引导模型"往哪个方向走"。比如 ReAct Loop 让模型在推理时有了"观察→行动→再观察"的结构化流程，而不是一次性输出完事。
2. **装置（harness as apparatus）**：为模型提供运行所需的工程化基础设施。注册中心、动态注入、Handler 接口——这些都是让模型"跑起来"的装置。
3. **轨道（harness as track）**：确保模型的行为在可控范围内。路由定义了"哪些问题走哪条路"，Plan Handler 定义了"以什么策略做决策"——这些都是在为模型铺设轨道。

> ⚠️ **注意事项**：这个认知非常关键——它不是"模型能力不够才需要工程化"，而是"**模型和实际业务之间，存在大量的工程化处理空间**"。有了这个基础认知，才能理解为什么本课要从"字典管理"升级到"注册中心 + 动态注入 + 可替换 Handler + 执行层归一化"的完整体系。

#### 🔀 对比辨析：没有工程化控制 vs 有工程化控制

| 维度 | 无工程化控制 | 有工程化控制 |
|---|---|---|
| 工具管理 | 字典散落各处，无处可查 | 注册中心集中管理，全局可查 |
| 工具注入 | 全量塞给模型，token 成本高 | 按场景动态激活，模型只看到需要的子集 |
| 决策策略 | 模型自主选择，无法切换 | Plan Handler 可替换：模型/规则/A-B 测试 |
| 执行方式 | 直接调用，无统一接口 | Execution Handler 可替换：真实/mock/远程 |
| 扩展性 | 加工具改多处，删工具怕遗漏 | 注册一次、按需激活、可插拔替换 |
| 可测试性 | 难以模拟不同策略 | mock Handler 轻松切换测试场景 |

### 2.2 问题一：模型关心 vs 工程关心——两类信息使用场景不同

#### 🧠 直观理解

想象你是一家餐厅的老板。你需要两份不同的清单：
- **给顾客看的菜单**：菜名、简介、价格——顾客据此决定点什么；
- **给厨师看的操作手册**：食材配方、烹饪步骤、炉灶档位——厨师据此把菜做出来。

模型就是"顾客"——它只需要知道工具叫什么、干什么用、需要什么参数，然后决定"我要用这个工具"。工程侧就是"厨师"——它需要知道这个工具对应的**函数地址**是什么，才能真正把它调用起来。

当我们用同一份字典结构同时管理这两类信息时，问题就出现了：**两类数据的角色不同，使用场景不同，却混在一起维护。**

#### 📖 详细解释

##### 2.2.1 变量名是内存地址，不能直接和字符串挂钩

这是理解问题一所需的最底层认知。我们来看老师上课时重点讲解的推导链。

> 🔗 **推导链：变量名 → 内存地址 → 为什么需要字典映射**

```
起点: 你在 Python 中定义了 def tool1(): pass

  │
  ├── 第1步: 变量名 tool1 到底是什么？
  │   为什么？编程语言中，tool1 这个标识符本质上是内存地址的一个别名。
  │   机器并不认识"tool1"这个人类可读的字符串——它只认识一串内存地址。
  │   变量名只是"方便人类阅读的标签"，对计算机没有语义含义。
  │
  ├── 第2步: 如果我们 print(tool1)，会看到什么？
  │   为什么？在 Python 中执行 print(tool1)，输出类似：
  │   <function tool1 at 0x7f8b1c002a60>
  │   它是一个 function 对象，存在内存的某个地址上。
  │   工具的本质就是"一段存放在内存中的函数地址"。
  │
  ├── 第3步: 模型能直接使用这个内存地址吗？
  │   为什么不能？模型运行在远程服务器上，它看到的只是我们通过 API
  │   发给它的文本描述。它不可能直接访问我们本机的内存地址。
  │   模型能做的是：返回一个字符串——比如 "tool1"——告诉我们它想用哪个工具。
  │
  ├── 第4步: 拿到字符串 "tool1" 之后，我们怎么找到对应的函数？
  │   为什么需要字典？因为 "tool1" 这个字符串和内存中的 tool1 函数
  │   之间没有任何语言内置的映射关系。你必须自己建立这种映射。
  │   这就是为什么需要 dict：{"tool1": tool1} ——
  │   用字符串作为 key，用函数地址作为 value。
  │
  └── 结论: 字符串与内存地址之间必须由开发者手动建立映射。
           字典结构不是"更好的选择"，而是"唯一的选择"。
```

##### 2.2.2 模型真正关心什么 vs 工程真正关心什么

现在把上面的推导链放入工具管理的语境中。老师画了一张关键对比图——直接还原：

**模型在决策时关心**：

| 字段 | 含义 | 为什么需要 |
|---|---|---|
| `name` / `id` | 工具的唯一标识 | 模型用这个字符串告诉我们它想调哪个工具 |
| `description` | 工具的功能描述 | 模型据此判断"这个任务该用哪个工具" |
| `args` (schema) | 参数名和类型约束 | 模型需要知道该传哪些参数、各是什么类型 |

**工程在调用执行时关心**：

| 字段 | 含义 | 为什么需要 |
|---|---|---|
| `func` | 执行函数的实际地址 | 拿到模型返回的工具名后，必须找到对应的可调用对象 |
| 参数传递 | 将模型给出的 args 传给 func | 类型匹配、默认值处理、合法性校验 |

> 💡 **核心洞见**：两类数据有重合但不完全重叠。`desc` 和 `args` 是两者都需要的——模型用它决策，工程用它校验。但 `func`（函数地址）只有工程需要，`name/id` 在工程侧的作用只是"字典的 key"。**同一条数据被两条完全不同的链路消费，这就是管理混乱的根因。**

##### 2.2.3 从代码看问题——第 6 课做法的回顾

回想一下第 6 课中我们是怎么做工具调用的。老师带着大家回顾了那个关键代码片段：

```python
# 第6课的典型写法：在一个脚本头部定义工具
# 工具散放在脚本文件的头部

# 定义工具函数
def tool1(param1):
    # 工具1的具体逻辑
    pass

def tool2(param2):
    # 工具2的具体逻辑
    pass

# 用字典集中管理工具信息
tool_info = {
    "tool1": {
        "desc": "工具1主要处理的任务集中在...",
        "args": {"arg1": {"type": "string", "description": "参数说明"}},
        "func": tool1,   # ← 手动建立的函数地址映射
    },
    "tool2": {
        "desc": "工具2主要处理的任务集中在...",
        "args": {"arg1": {"type": "integer", "description": "参数说明"}},
        "func": tool2,
    },
}
```

然后在 ReAct Loop 的调用环节，我们这样找到并执行工具：

```python
# 模型返回的决策结果中给出了 function_name
# 我们从 tool_info 字典中查找对应的执行函数
function_name = result["function_name"]   # 模型返回字符串 "tool1"
tool_entry = tool_info[function_name]     # 用字符串做 key 查到对应条目
func_to_call = tool_entry["func"]         # 取出函数地址
func_to_call(**result["arguments"])       # 传入参数执行
```

这个流程中，工具管理暴露了三个具体痛点：

1. **定义分散**：`tool1`、`tool2` 的函数定义和它们的 `desc`、`args` 描述信息写在同一个脚本头部。当工具跨文件、跨模块时，没有统一的管理入口。
2. **角色混淆**：同一份 `tool_info` 字典既要为模型生成 schema（只取 `desc` 和 `args`），又要为工程提供函数地址（取 `func`）。两个消费方对同一份数据有不同需求，却没有做分离。
3. **模型不关心的信息也在传递**：`func` 字段对模型毫无意义——模型不能也无需直接调用函数。但当前方案中 `func` 和其他字段耦合在一起，增加了管理负担。

##### 2.2.4 课上学员互动：为什么"tool1"不能直接对回 tool1 函数？

> ❓ **常见疑问**：很多同学看到上面的代码时会有疑问——`tool_info` 字典里不是写了 `"tool1": tool1` 吗？这难道不是已经建立了映射？为什么老师还要强调"不能直接挂钩"？

**💬 老师回答**：

这里的关键在于理解编程语言的本质。在 Python 中，`tool1` 这个变量名在字典定义时的确指向了内存中的 `tool1` 函数——`"tool1": tool1` 这句代码把**内存地址**存进了字典。但问题的核心在于：**字典的 key `"tool1"` 是一个字符串字面量，模型返回的 `"tool1"` 是另一个字符串——它们在 Python 层面是"碰巧内容相同"，而不是语言内置的映射关系。**

更具体地说：如果你把字典的 key 写成 `"函1"`，而模型返回 `"tool1"`，字典就无法查找。这说明映射是**纯手动的、基于字符串匹配的**——没有任何机制保证"模型返回的字符串"一定等于"字典的某个 key"。你必须自己维护这种一致性。

**延伸理解**：为什么其他编程语言也是一样的？
- 在编译型语言（如 Go、Rust）中，函数名在编译后变成了符号表中的地址，运行时根本不存在"函数名字符串"这个概念。
- 在 JavaScript 中，虽然可以通过 `window["funcName"]` 动态访问全局函数，但这同样依赖手动建立的命名约定。一旦函数不是全局的（模块化开发中非常常见），这种方式就失效了。

> 💡 **延伸**：这个认知是整个工具管理体系的基石。理解了"字符串必须手动映射到函数地址"，你就能理解为什么需要一个"注册中心"来集中维护这种映射——而不是在每个调用的地方都写一遍 `if tool_name == "xxx": call_xxx()`。

#### 🐛 排错实录：如果把 func 字段去掉会怎样？

这是老师课上引导大家思考的一个场景——如果不管理 `func`，只给模型提供 `desc` 和 `args`，会发生什么？

**❌ 错误写法**
```python
# 只注册了模型关心的信息，没有 func 映射
tool_info_only_desc = {
    "tool1": {"desc": "查询天气", "args": {"city": "string"}},
    "tool2": {"desc": "查询汇率", "args": {"from": "string", "to": "string"}},
}

# 模型返回：我想用 tool1，参数 city="北京"
# 然后呢？你怎么执行 tool1？
```

**🚨 问题现象**
```
模型返回了 function_name = "tool1" 和 arguments = {"city": "北京"}
但是系统无法找到 tool1 对应的执行函数——因为 func 字段被省略了。
```

**🔍 原因分析**

问题出在信息流的断裂。模型只能返回一个**字符串**（工具名），而执行需要一个**可调用对象**（函数地址）。字符串和可调用对象之间的映射必须由开发者手动建立并维护。如果省略了 `func` 字段，就断了这条映射链。

**✅ 正确做法**

必须维护两份信息，供两条链路消费：
- **模型链路**：`name + desc + args` → 生成 schema → 注入 prompt → 模型决策 → 返回 `function_name`
- **工程链路**：`function_name` → 字典查找 → 取出 `func` → 传入 `args` 执行

```python
# 正确：同时管理模型信息和工程信息
tool_info = {
    "tool1": {
        "desc": "查询天气",           # ← 模型链路使用
        "args": {"city": "string"},   # ← 模型链路使用
        "func": call_weather,         # ← 工程链路使用
    },
}
```

**📝 经验总结**

始终记住：工具管理本质上是**维护一个"字符串→函数地址"的映射表**。这个认知会贯穿后续所有模块——注册中心的核心功能之一就是集中维护这份映射。

#### 🔀 前后对比：Demo 级 vs 工程级的鸿沟

| 维度 | Demo 级（第 6 课写法） | 工程级（20+ 工具） |
|---|---|---|
| 工具数量 | 3-5 个 | 20+ 个，持续增长 |
| 定义位置 | 一个脚本头部 | 多个文件、多个模块 |
| 开发者 | 一个人 | 多人协作，各自有定义习惯 |
| 字典维护 | 手写，肉眼检查 | 需要集中管理，否则无法回答"系统有多少工具" |
| 模型 schema 生成 | 手动遍历 dict 拼接 | 需要标准化、自动化 |
| 函数地址管理 | `"tool1": tool1` 直接引用 | 跨文件后引用关系复杂，容易断链 |

### 2.3 问题二：工具散落 + 全量注入——缺乏统一的激活管理

#### 🧠 直观理解

问题一告诉我们：需要把模型关心的信息和工程关心的信息分开管理。但如果只是分开，而没有统一的管理中心，问题会进一步恶化。想象一个公司的资产管理部门：如果不建一个统一的固定资产台账，每个部门各自记录自己的设备，那么想回答"公司一共有多少台电脑"这个问题都做不到。

工具的处境完全一样。**如果没有注册中心，"系统目前到底有多少工具可用"这个看似基础的问题，根本无法回答。**

#### 📖 详细解释

##### 2.3.1 工具的"三个维度"——需要分层看待

老师在上课时画了一个非常重要的三层概念图。这是一个递进收敛的过程——随着命题越来越具体，可用的工具也随之聚焦：

```
         全量工具池（ABC）
         ┌─────────────────────────────────────────────┐
         │  query_leave   query_approval  write_report  │
         │  send_notify   exec_script     query_hr      │
         │  approve_req    ...共 20+ 个工具             │
         └──────────────┬──────────────────────────────┘
                        │
                        │ 根据场景/意图筛选
                        ▼
         场景工具集（AB）
         ┌──────────────────────────┐
         │  query_leave             │
         │  query_approval          │  ← 只读场景：只给查询工具
         │  query_hr                │
         └──────────────┬───────────┘
                        │
                        │ 根据具体命题进一步收敛
                        ▼
         实际调用工具（A）
         ┌─────────────────┐
         │  query_leave     │  ← 模型在此节点上实际选用的工具
         └─────────────────┘
```

> 💡 **核心洞见**：三层不是三个独立的列表，而是**同一套工具的三种视图**。全量工具池是一次注册的；场景工具集是动态激活的；实际调用工具是模型在激活集中自主选择的。**use_tools 的本质不是简单地限制工具列表，而是变更注入给模型的提示词信息**——模型能看到什么工具，取决于你给了它什么 schema。

##### 2.3.2 全量注入的代价——不只是 token 成本

如果我们不做分层，每次请求都把 20+ 工具的 schema 全部注入模型请求：

1. **Token 成本固定且线性增长**：每增加一个工具，system prompt 就多一段 schema 文本。全量注入意味着无论用户问什么，模型都要处理全部工具描述。一个只查假期的请求，也要带着"执行脚本"的工具描述一起入参。

2. **模型选错概率上升**：工具越多，模型的"注意力"越分散。老师特别提到了 OpenClaude 的案例——随着 skills 越装越多，工具就选不准了。"一开始给它做一个小命题好像还行，但随着用的越来越多，装了越来越多的 skills、越来越多的工具，我们发现它的工具选不准了。它解决具体问题的时候，思路可能就没有那么清晰了，甚至会受到整个信息的一个干扰。"

   这个现象的根因在于：模型的注意力机制在处理大量工具信息时，会产生"上下文干扰"。每个工具的 desc 都在争夺模型的注意力权重——一个关于"写入文件"的工具描述，可能在模型试图选择一个"只读查询"工具时产生负面的暗示效应，让模型倾向于"可能也许应该写点什么"。

3. **缺乏场景语义**：全量注入时，模型看到的是一个平面的工具列表。它不知道"当前场景只需要查询类工具"这个约束。工具之间的适用边界没有被有效传达给模型。

##### 2.3.3 工程化的挑战——多开发者 + 多文件 + 多模块

> 🔗 **推导链：Demo 级到工程级的复杂度跃迁**

```
起点: 第6课中，工具定义在单一脚本的头部
  │
  ├── 第1步: 工具数量从 3 个增长到 20+
  │   为什么？真实企业的内部助手需要覆盖查询、写入、分析等多个领域。
  │   每个领域至少有 3-5 个工具，总计轻松超过 20。
  │
  ├── 第2步: 工具定义不再局限于一个文件
  │   为什么？不同领域由不同团队成员开发，各自维护自己的工具模块。
  │   查询工具在 query_tools.py，写入工具在 write_tools.py，等等。
  │
  ├── 第3步: 每个人定义工具的方式可能不同
  │   为什么？没有统一的注册规范。小张用 desc 字段，小王用 description 字段。
  │   小李的 args 用列表，小赵的 args 用字典。调用方无法用统一的方式消费。
  │
  ├── 第4步: 问题一直接受到冲击
  │   为什么？格式不统一的工具描述传递给模型时，模型对工具的理解会出现偏差。
  │   格式不统一的 func 管理让调用链路变得脆弱。
  │
  └── 结论: 需要一个统一的注册中心，它解决两个问题：
           ① 让所有工具都"有处可查"（集中管理）
           ② 让激活层能按场景动态筛选工具子集（避免全量注入）
```

| 场景 | 信号特征 | 示例 | 需要的工具子集 |
|---|---|---|---|
| 查询场景 | 用户想"查/看/了解" | "帮我查下假期余额" | query_leave, query_approval, query_hr |
| 写入场景 | 用户想"发/改/保存" | "帮我提交请假申请" | submit_leave, send_notify, write_report |
| 分析场景 | 用户想"分析/生成报告" | "生成本月考勤报告" | exec_report, write_report, query_attendance |

### 2.4 问题三：规划逻辑固化——ReAct Loop 中的"黑盒"

#### 🧠 直观理解

如果你买了一辆车，但你只能控制两件事：给油箱加什么油，以及出发前说一句"我要去某某地方"。至于发动机怎么运转、变速箱怎么换挡、ABS 怎么介入——你完全碰不了，也看不到。这辆车平时开还行，但当你需要省油模式、需要手动降档爬坡、需要在雪地关闭 ABS 的时候，你就傻了。

ReAct Loop 中的 function calling 就是这辆车。被框架封装好后，你只能操作两个入口，中间的决策过程是一个"黑盒"。

#### 📖 详细解释

##### 2.4.1 ReAct Loop 中可干预的只有两个入口

老师课上带着大家画了一张非常关键的可干预点分析图。直接还原：

```
                          用户输入
                             │         ← 可干预点②: 用户指令 (instruct)
                             │           通常写成 "You must..."/"你必须..."
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                     ReAct Loop（封装好的黑盒）                   │
│                                                                  │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  可干预点①: 工具描述 (tool schema / desc / args)        │   │
│   │  ────────────────────────────────────────────────────    │   │
│   │  这是我们注入给模型的工具信息。                           │   │
│   │                                                          │   │
│   │         ▼                                                │   │
│   │  ┌───────────────────────────┐                           │   │
│   │  │  PLAN（how to choose?）   │  ← ✗ 无法干预            │   │
│   │  │  模型内部的规划提示词     │     这是一段内置的指令    │   │
│   │  │  如何分析工具、如何决策   │     被框架或模型服务封装  │   │
│   │  │  用哪个工具、什么顺序     │                           │   │
│   │  └───────────┬───────────────┘                           │   │
│   │              │ action command                            │   │
│   │              ▼                                           │   │
│   │  ┌───────────────────────────┐                           │   │
│   │  │  EXECUTE（action call）   │  ← ✗ 无法干预            │   │
│   │  │  直接执行，没有中间节点   │     无法插入日志          │   │
│   │  │                           │     无法做权限校验        │   │
│   │  └───────────┬───────────────┘     无法做审计包裹        │   │
│   │              │                                           │   │
│   └──────────────┼───────────────────────────────────────────┘   │
│                  │                                                │
│                  ▼                                                │
│           OBSERVE（观察结果，回填上下文）                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

两个可干预点具体是什么？

- **可干预点①：工具描述（tool schema）**。你可以在 `desc` 字段里写得更详细、更准确，来影响模型的工具选择。比如在描述中强调"这个工具只用于人力资源查询场景"。这是能做的，但效果有限——描述只是"信息"，不是"指令"。

- **可干预点②：用户指令（instruct / system prompt）**。你可以在用户的输入或系统提示中写 "You must use the query tool" / "你必须使用查询工具"。这种强指令确实能影响模型行为，但代价很大。

##### 2.4.2 强指令的问题——模型的自主性被压制

> ⚠️ **注意事项**：老师在课上明确指出，使用 "You must" 这类强指令存在两个严重问题。

**问题 A：模型可能强行贴合你的指令，放弃自主判断。**

当你写 "You must use the XXX tool to solve this problem"，模型会倾向于无论如何都使用 XXX 工具——即使它内心"判断"这个工具并不适合当前任务。模型的推理能力和指令遵从之间存在张力：遵从指令意味着放弃一部分推理的自主性。

**问题 B：强指令无法处理需要动态判断的场景。**

举一个老师课上提到的例子：根据文件类型使用不同的浏览器工具。如果当前文件是 PDF，应该用 PDF 阅读器工具；如果是网页链接，应该用网页浏览器工具。这种判断**本质上是规则能解决的**——URL 的特征是规则化的，用模型来做这种判断反而不准，因为"语义相似但规则不同"的边界情况模型处理不好。

##### 2.4.3 "how to choose"——被封装的规划层提示词

老师在课上反复强调一个关键架构细节：

> 在整个 function calling 的过程中，PLAN 环节"如何选择工具"（how to choose）其实是一组模型内置的提示词。它告诉模型：根据什么原则分析工具、根据什么逻辑做选择。但因为这组提示词被封装在框架或模型服务内部，**你无法修改它、无法替换它、甚至无法看到它。**

这说明有**一个模型请求发生在 PLAN 环节内部**——不是你把工具描述丢给模型，然后模型直接吐出结果那么简单。在工具描述和用户输入之外，还有一个隐藏的"层"在做引导：它告诉模型怎么分析工具、怎么匹配需求、怎么排序候选。这个层在框架封装好的 ReAct Loop 里是**不可达的（unreachable）**。

这意味着：

- 你想让模型在特定场景下"优先选择只读工具"——做不到，除非用强指令。
- 你想让模型在测试环境用"固定调用顺序"——做不到，model plan 的逻辑你改不了。
- 你想在模型选工具之前加一层校验（比如：当前用户没有写入权限，拦截所有 write 类工具）——做不到，没有插入校验逻辑的节点。
- 你想让模型在 debug 模式下输出"为什么选了 A 而不是 B"的详细推理——看不到，plan 内部没有暴露 profiling 信息。

##### 2.4.4 执行层的不可干预

PLAN 环节产生 action command 后，直接进入 EXECUTE 环节。在封装好的 ReAct Loop 中，这个环节同样是不可干预的。你无法在中间插入：

- **执行前校验**：模型选了 "write_report"——但这个用户真的有写入权限吗？
- **执行日志**：记录每次工具调用的输入参数和执行结果，用于审计和排错。
- **Mock 执行**：测试时不想真调外部 API，想用模拟数据替代——没有钩子可以拦截。
- **超时控制**：单次工具调用需要超时限制——框架内部如果没做，你就加不了。

##### 2.4.5 Hook 模式——解决问题的方向

> 🔗 **关联知识**：本小节的 Hook 模式只是一个引子。详细的 Plan Handler 和 Execution Handler 会在后续模块中拆解（详见 [第五章：Plan 和 Execution Handler 详解]）。

老师提到，像 OpenClaude 这样的框架已经提供了 Hook 机制——允许开发者在特定节点（如规划前、执行前、执行后）注入自定义逻辑。这本质上是**打开黑盒，提供可扩展的干预点**。

把 Hook 的思路映射到 ReAct Loop 中，我们希望在这些位置插入可定制的逻辑：

```
planning 前                → 可以注入"场景约束"（如：只允许 query 类工具）
planning 后 / execute 前   → 可以注入"权限校验"（如：拦截 write 类工具）
execute 后                 → 可以注入"审计日志"（记录调用结果）
```

这就是问题三的解决方向——不是推翻 ReAct Loop，而是在循环的关键节点上开"口子"，允许开发者插入自己的逻辑。这些"口子"的正式形态，就是 Plan Handler 和 Execution Handler。

## 三、工具注册中心——集中管理工具声明与执行

### 3.1 设计思路

#### 🧠 直观理解

想象你走进一座大型图书馆。如果没有统一的图书编目系统，每本书散落在各个角落，你不知道馆藏有多少书、书名是什么、放在哪里。查找一本书要在不同的书架、不同的楼层之间来回翻找——这就是第06-08课中工具散落在各处、字典定义、全量注入的困境。

**工具注册中心就是这套编目系统**：所有工具在这里"登记入册"，任何时候想知道系统有哪些工具可用，到注册中心查一次就够了。

#### 📖 详细解释

回顾本课开头提出的三个关键问题，注册中心直接解决了其中两个：

| 问题 | 现象 | 注册中心如何解决 |
|---|---|---|
| **问题一：工具散落各处** | 工具注册分散在 agent 创建时、TriggerFlow chunk 里、不同 Python 模块中。加一个工具要改多处，删一个工具不知道谁在引用 | **集中声明，全局可查**——一个类管理所有工具，`list_all()` 一次看清全部家底 |
| **问题二：全量注入，无法按场景组合** | 20 个工具每次请求都进 system prompt，token 成本固定，模型选错概率上升 | **出口可筛选**——`list_specs()` 提供全量声明供激活层筛选，`get_spec(id)` 按需取出 |

注册中心的本质是一个**管理器（Manager）**模式：用内部字典存储数据，通过有限的一组接口对外暴露能力，外部使用者永远不直接触碰内部存储结构。

```
                    ┌──────────────────────────────────────┐
                    │           工具注册中心                 │
                    │          ToolRegistry                 │
                    │                                      │
                    │  内部存储: _tools (dict)               │
                    │  ┌────────────────────────────────┐  │
                    │  │ "query_leave"  → {id, desc,     │  │
                    │  │                  input_schema,  │  │
                    │  │                  func,           │  │
                    │  │                  side_effect..}  │  │
                    │  │ "write_report" → { ... }         │  │
                    │  │ "send_notify"  → { ... }         │  │
                    │  │ ...                             │  │
                    │  └────────────────────────────────┘  │
                    │                                      │
                    │  对外接口:                             │
                    │  register()    → 集中注册             │
                    │  list_all()    → 全量工具清单         │
                    │  list_ids()    → 所有工具 ID          │
                    │  list_specs()  → 能力声明列表(给模型)  │
                    │  get_spec(id)  → 按 ID 查声明(不含func)│
                    │  get_func(id)  → 按 ID 取执行函数     │
                    └──────────────────────────────────────┘
```

#### 🔗 推导链：为什么需要 Manager 而非裸露的字典

```
起点: 用裸字典管理工具 { "tool_name": { desc, func, ... } }
  │
  ├── 第1步: 字典直接暴露 → 任何代码都能随意修改内部结构
  │   为什么？ 缺乏封装，数据结构零散分布在各个模块中，修改无约束，一致性无法保证
  │
  ├── 第2步: 引入 ToolRegistry 类，用 _tools 下划线标记私有属性
  │   为什么？ Python 约定以 _ 开头表示内部实现，不鼓励外部直接访问。所有读写必须通过接口方法
  │
  ├── 第3步: 通过 register() 方法统一注册入口
  │   为什么？ 注册时可以做格式校验（如 input_schema 类型检查）、去重检查、日志记录——这些逻辑如果加在裸字典赋值时，无法集中管理
  │
  └── 结论: Manager 模式提供「数据 + 行为」的封装——数据是 _tools 字典，行为是 register/list/get 等接口方法。外部只关心接口，不关心存储细节
```

> 💡 **核心洞见**：字典在底层，但不可直接触碰。注册中心的价值不在于「怎么存」（底层就是一个 dict），而在于「如何约束访问」——所有读写必须经过有限的一组接口，这是后续扩展安全校验、分类分级、日志审计等能力的**唯一锚点**。

### 3.2 注册声明规范

每一份注册到中心的工具，需要提供五个关键信息。这五个字段构成了**工具的能力声明（ActionSpec）**——它是模型与系统之间的契约。

| 字段 | 类型 | 含义 | 用途 |
|---|---|---|---|
| `tool_id` | `str` | 工具的唯一标识符 | 系统内部索引，模型通过 name 字段引用 |
| `desc` | `str` | 供模型理解的工具描述 | 进入 system prompt，模型基于此判断何时调用 |
| `input_schema` | `dict` | 输入参数的约束定义 | 模型基于此传参；也可以用来做运行前参数校验 |
| `func` | `Callable` | 实际的执行函数 | 当模型决定调用后，系统执行此函数 |
| `side_effect_level` | `str` | 副作用级别 | 执行前的策略挂钩点，决定是否需要额外审查 |

#### 3.2.1 side_effect_level 三级设计详解

`side_effect_level` 是注册中心设计中一个看似简单但极其重要的字段。它记录了工具对系统的**影响程度**，分为三个级别：

```
┌──────────────────────────────────────────────────────────┐
│              side_effect_level 三级体系                   │
│                                                          │
│   read (只读)                                             │
│   ├── 仅获取信息，不产生任何变更                           │
│   ├── 示例: query_leave, query_approval, get_weather      │
│   └── 风险: 最低。正常场景始终允许                        │
│                                                          │
│   write (写入)                                            │
│   ├── 会对数据/文件/状态产生变更                           │
│   ├── 示例: write_report(保存文件), submit_form(提交表单)  │
│   └── 风险: 中等。特定场景可能需要拦截或人工确认           │
│                                                          │
│   exec (执行级)                                           │
│   ├── 通过执行行为直接影响系统运行状态                     │
│   ├── 示例: exec_script(运行脚本), deploy_service(部署)    │
│   └── 风险: 最高。通常需要审批流程或沙箱隔离              │
└──────────────────────────────────────────────────────────┘
```

**为什么需要人工约定这个级别？**

计算机无法自动判断一个函数是"只读"还是"写入"——它只能看到一个 `Callable` 对象。`side_effect_level` 是**开发者基于业务语义人为标注的元信息**。这个标注是可靠的，因为只有写这个工具的人才最清楚它的行为。

**与第8课人工介入的关联**：

注册中心记录副作用级别，但**不直接决定如何处理**。处理策略由执行层根据级别来决定：

```
工具执行流程:

  get_func(tool_id) 获取执行函数
        │
        ▼
  检查 side_effect_level
        │
        ├── level = "read"
        │   └── 直接执行，无需额外审查
        │
        ├── level = "write"
        │   └── 根据场景策略判断：
        │       ├── 只读场景 → 拒绝执行
        │       ├── 信任场景 → 直接执行
        │       └── 敏感场景 → 人工介入确认（第8课模式）
        │
        └── level = "exec"
            └── 强制走安全流程：
                ├── 模型审查：让另一个模型评估操作合理性
                ├── 人工审批：第8课子流模式，推送审批卡片
                └── 企业风控：走内部风控扫描流程
```

> 💡 **核心洞见**：`side_effect_level` 是管理抓手。它把"这个工具危险吗"这个主观判断转化为一个明确的、可被代码读取的字段。后续所有的激活策略（第四章 动态注入）、执行层策略（第五章 Plan/Execution Handler）都依赖这个字段做出决策。**声明阶段标注级别，执行阶段依据级别行动**——声明与执行的分离是这套设计的关键。

> ⚠️ **注意事项**：`side_effect_level` 是人为约定，不是系统强制分类。在最小原型中可以不做（默认 `"read"`），但在生产环境中强烈建议对每个工具明确标注。你也可以根据自己的业务场景自定义级别名称，比如用 `low` / `medium` / `high` 表示风险等级，核心思想不变。

#### 3.2.2 input_schema 的强约束讨论

在 MVP 实现中，`input_schema` 是一个简单的 `dict`，如 `{"employee_id": "str"}`。老师在现场编码时提出了进一步的思考：

> "我们甚至可以写成这个样子——input_schema 里面我用 TypedDict 来约定，到底是什么样的一个东西。它的核心作用就是，我们进了这个东西之后，在内部把这些信息映射到我们的中央管理的数据结构里面。"

这意味着 `input_schema` 可以进一步演进为更严格的类型系统：

```python
# MVP 阶段（当前）：松散的 dict 约束
input_schema = {"employee_id": "str", "days": "int"}

# 演进方向 1：TypedDict 强类型
from typing import TypedDict

class QueryLeaveInput(TypedDict):
    employee_id: str

# 演进方向 2：Pydantic 模型校验
from pydantic import BaseModel

class QueryLeaveInput(BaseModel):
    employee_id: str
    # 可以加 validator、默认值、范围限制等
```

这种强约束的价值在于：在工具执行**之前**就能校验参数格式是否合法，避免执行到一半才发现参数类型不对。

#### 💻 Step-by-Step

**完整代码** —— 文件名 `s01_tool_registry.py`：

```python
"""
第09课 演示脚本 s01：自主实现简版工具注册中心 MVP

只用 Python 原生 dict + 函数，不依赖任何框架的 Action 系统。
展示注册中心的核心：集中声明、按 id 查询、全量清单。
"""
from typing import Callable, Any

# ── 简版注册中心 MVP ────────────────────────────────────────

class ToolRegistry:
    """工具注册中心：集中管理所有工具的能力声明和执行函数"""

    def __init__(self):
        # 内部存储器：一个私有字典，外部不可直接触碰
        # 下划线 _ 开头是 Python 约定，表示"这是内部实现，不鼓励外部访问"
        self._tools = {}  # id → { id, desc, input_schema, func, side_effect_level }

    def register(self, tool_id: str, desc: str, input_schema: dict,
                 func: Callable, side_effect_level: str = "read"):
        """注册一个工具：能力声明 + 执行函数
        
        Args:
            tool_id: 工具唯一标识符，供系统内部索引
            desc: 供模型理解的描述文本
            input_schema: 输入参数约束，如 {"employee_id": "str"}
            func: 实际执行函数，接收 input_schema 定义的参数
            side_effect_level: 副作用级别，read/write/exec，默认 "read"
        """
        self._tools[tool_id] = {
            "id": tool_id,
            "desc": desc,
            "input_schema": input_schema,
            "func": func,
            "side_effect_level": side_effect_level,
        }

    def list_all(self) -> list:
        """返回全量工具清单（包含 func，供激活层或调试使用）"""
        return list(self._tools.values())

    def list_ids(self) -> list:
        """返回所有工具 id 列表"""
        return list(self._tools.keys())

    def list_specs(self) -> list:
        """返回所有工具的能力声明列表（不含 func，专门供模型构建 tool schema 用）
        
        模型只需要知道工具叫什么、干什么、收什么参数，不需要执行函数本身。
        这个方法就是为此设计的——返回给激活层用来构建模型的 tool schema。
        """
        return [{"id": t["id"], "desc": t["desc"], "schema": t["input_schema"]}
                for t in self._tools.values()]

    def get_spec(self, tool_id: str) -> dict | None:
        """获取单个工具的能力声明（不含 func）
        
        返回 tool_id 对应的声明信息，但排除执行函数。
        这是安全的出口——调用方只能看到工具的"说明书"，拿不到执行逻辑。
        """
        t = self._tools.get(tool_id)
        if not t:
            return None
        return {k: v for k, v in t.items() if k != "func"}

    def get_func(self, tool_id: str) -> Callable | None:
        """获取工具的执行函数
        
        这是唯一能拿到 func 的出口。后续安全校验、沙箱包裹都从这个方法注入。
        如果 tool_id 不存在，返回 None 而非抛异常——让调用方自行处理容错。
        """
        t = self._tools.get(tool_id)
        return t["func"] if t else None
```

**行级解析表**：

| 行号 | 代码 | 设计意图 |
|---|---|---|
| 2 | `from typing import Callable, Any` | 引入类型注解，`Callable` 用于标注 func 参数 |
| 6-7 | `class ToolRegistry:` + 类说明 | 定义一个 Manager 类，明确职责：管理工具的声明和函数 |
| 9-10 | `def __init__(self):` + `self._tools = {}` | 初始化私有字典。`_` 前缀是 Python 约定，表示"内部实现，不鼓励外部直接访问" |
| 12-20 | `def register(self, ...)` | 注册入口。五个参数对应注册声明的五个字段。`side_effect_level` 默认 `"read"` 体现安全优先原则——不声明就只给最低权限 |
| 22-23 | `def list_all(self)` | 全量清单出口。返回 `dict` 的 `values()` 列表，包含 func |
| 25-26 | `def list_ids(self)` | 轻量出口——只返回 id 列表，适合快速检查工具是否存在 |
| 28-34 | `def list_specs(self)` | **关键出口**。构建模型所需的 tool schema，刻意排除 func 字段——模型只需要声明，不需要执行函数 |
| 36-43 | `def get_spec(self, tool_id)` | 按 id 查声明。同样排除 func。`dict.get()` 而非 `[]` 索引——避免 KeyError，返回 None 让调用方容错 |
| 45-49 | `def get_func(self, tool_id)` | 获取执行函数的唯一出口。后续安全校验、沙箱包裹都可在这条路径上注入 |

### 3.4 代码实操：使用演示

现在用注册中心管理一组企业助手工具，完整演示注册、查看、查询、调用的全流程。

#### Step 1: 定义工具函数

```python
# ── 定义工具函数（模拟企业助手的四个工具）──

def query_leave(employee_id: str) -> dict:
    """查询员工剩余年假天数 —— read 级别"""
    return {"employee_id": employee_id, "days_remaining": 12}

def query_approval(request_id: str) -> dict:
    """根据申请单号查询审批状态和审批人 —— read 级别"""
    return {"request_id": request_id, "status": "审批中", "approver": "李经理"}

def write_report(title: str, content: str) -> str:
    """生成并保存分析报告 —— write 级别，会产生文件写入"""
    return f"报告《{title}》已保存"

def send_notification(employee_id: str, message: str) -> str:
    """向员工发送通知消息 —— write 级别，会触发消息推送"""
    return f"通知已发送给 {employee_id}"
```

#### Step 2: 创建注册中心并集中注册

```python
# ── 创建注册中心，将四个工具集中注册 ──

registry = ToolRegistry()

# 注册 read 级别工具：查询类，安全无副作用
registry.register("query_leave", "查询员工剩余年假天数",
                   {"employee_id": "str"}, query_leave, "read")
registry.register("query_approval", "根据申请单号查询审批状态和审批人",
                   {"request_id": "str"}, query_approval, "read")

# 注册 write 级别工具：写入类，会产生数据变更
registry.register("write_report", "生成并保存分析报告",
                   {"title": "str", "content": "str"}, write_report, "write")
registry.register("send_notification", "向员工发送通知消息",
                   {"employee_id": "str", "message": "str"}, send_notification, "write")

print(f"注册完成！注册中心现有 {len(registry.list_ids())} 个工具")
```

> 💡 **核心洞见**：注册时用**具名参数**的写法（`tool_id="query_leave"` 而非位置参数）可以让代码更直观清晰。老师在录音中强调："我们甚至可能更直观一点，用具名参数的方式，能更清楚地看到整个注册过程，代码的角度直观清晰，而且有一个非常明确的规范，来确保不同位置、不同人员开发时使用的信息是一致的。"

print(f"  query_leave → {spec}")
# 输出: query_leave → {'id': 'query_leave', 'desc': '查询员工剩余年假天数', 
#        'input_schema': {'employee_id': 'str'}, 'side_effect_level': 'read'}
# 注意: 返回值中不含 func —— get_spec 故意排除了执行函数

# ── 调用工具 ──

print("\n── 调用工具 ──")
func = registry.get_func("query_leave")
if func is None:
    print("  错误：工具未注册")
else:
    result = func("EMP_001")
    print(f"  query_leave('EMP_001') → {result}")
    # 输出: query_leave('EMP_001') → {'employee_id': 'EMP_001', 'days_remaining': 12}

# ── 尝试查询不存在的工具 ──

print("\n── 查询不存在的工具 ──")
missing = registry.get_func("non_existent_tool")
print(f"  non_existent_tool → {missing}")
# 输出: non_existent_tool → None
# 返回 None 而非抛异常，调用方可以优雅地处理容错
```

> ⚠️ **注意事项**：`get_spec` 返回的声明中不包含 `func` 字段——这是故意为之。声明信息是给模型和策略层看的，执行函数是给执行层用的，两者应该分离。这样即使你把 `get_spec` 的返回值直接打印或传输出去，也不会泄露执行逻辑。

    desc : 生成并保存分析报告
    level: write

  send_notification
    desc : 向员工发送通知消息
    level: write

### 3.6 排错指南

#### 🐛 排错实录1：访问 `_tools` 私有属性导致数据不一致

**❌ 错误写法**
```python
registry = ToolRegistry()
registry.register("my_tool", "xxx", {}, lambda: "ok")

# 绕过接口，直接修改内部字典
registry._tools["my_tool"]["desc"] = "被篡改的描述"
registry._tools["hijack_tool"] = {"id": "hijack_tool", "func": lambda: "危险操作"}
```

**🚨 风险说明**
```
直接修改 _tools 绕过了 register() 方法中的所有校验逻辑。
虽然 Python 不会报错（_ 只是约定，不是强制），但这种写法：
1. 可能写入格式不规范的数据
2. 跳过了注册时可能存在的去重、校验逻辑
3. 后续在 register() 中增加的日志/审计功能对这笔数据无效
```

**🔍 原因分析**

Python 的 `_` 前缀是**约定**而非**强制**。它告诉使用者"这是内部实现，请不要直接访问"，但编译器不会阻止你。这是 Python 设计哲学中"我们都是成年人"（We are all consenting adults）的体现。

**✅ 正确写法**
```python
# 所有操作都通过接口方法
registry.register("my_tool", "更新后的描述", {}, lambda: "ok", "write")
spec = registry.get_spec("my_tool")
```

**📝 经验总结**

遵守 `_` 前缀约定。如果你想更严格地保护内部数据，可以使用 `__` 双下划线前缀触发 Python 的 name mangling（名称改编），但这会增加调试复杂度。在团队协作中，**代码审查比语法限制更有效**。

---

#### 🐛 排错实录2：`get_func` 返回的函数调用报 TypeError

**❌ 错误写法**
```python
# 注册时声明参数名为 employee_id
registry.register("query_leave", "...", {"employee_id": "str"}, query_leave, "read")

# 调用时用了错误的关键字参数名
func = registry.get_func("query_leave")
result = func(emp_id="EMP_001")  # 参数名不匹配！
```

**🚨 报错信息**
```
TypeError: query_leave() got an unexpected keyword argument 'emp_id'
```

**🔍 原因分析**

`input_schema` 中声明的参数名与实际函数的参数签名必须一致。不一致时，Python 会直接报 `TypeError`。注册中心本身不做参数名校验——它只是存储，不做检查。

**✅ 正确写法**
```python
# 确认 input_schema 中的参数名与函数签名一致
func = registry.get_func("query_leave")
result = func(employee_id="EMP_001")  # 参数名匹配
# 或使用位置参数
result = func("EMP_001")
```

**📝 经验总结**

这是注册中心可以进一步演进的方向之一——在 `register()` 时检查 `input_schema` 中的 key 是否与函数的参数签名匹配。用 `inspect` 模块可以做到：

```python
import inspect

def register(self, tool_id, desc, input_schema, func, side_effect_level="read"):
    # 可选：校验参数一致性
    sig = inspect.signature(func)
    for key in input_schema:
        if key not in sig.parameters:
            raise ValueError(f"参数 '{key}' 不在函数 {tool_id} 的签名中")
    # ... 继续注册
```

### 3.7 扩展方向探讨

在课程中，老师提出了一个开放式讨论：**有了注册中心之后，有哪些地方可以进一步扩展？**以下是学员讨论和老师回应的总结。

#### 3.7.1 分类分级过滤

利用已有的 `side_effect_level` 字段，在 `list_all()` 或 `list_specs()` 的出口处按级别过滤：

```python
def list_all(self, level: str = None) -> list:
    """返回全量工具清单，可按副作用级别过滤"""
    tools = list(self._tools.values())
    if level:
        tools = [t for t in tools if t.get("side_effect_level") == level]
    return tools
```

**应用场景**：在只读场景中，只暴露 `level="read"` 的工具给模型——即使注册中心有 20 个工具，模型也只能看到其中的只读子集。

#### 3.7.2 安全校验注入

在 `get_func()` 这个关键出口处注入安全检查逻辑：

```python
def get_func(self, tool_id: str) -> Callable | None:
    """获取工具的执行函数——在返回前可以做安全检查"""
    t = self._tools.get(tool_id)
    if not t:
        return None
    
    raw_func = t["func"]
    level = t.get("side_effect_level", "read")
    
    # 安全检查：如果是 write/exec 级别，包裹校验逻辑
    if level in ("write", "exec"):
        def safe_func(*args, **kwargs):
            # 这里可以加入：
            # 1. 人工审批流程（第8课模式）
            # 2. 模型二次审查
            # 3. 企业风控扫描
            # 4. 沙箱验证（见下方 3.7.5）
            print(f"[安全校验] 工具 {tool_id} 级别={level}，执行前检查...")
            return raw_func(*args, **kwargs)
        return safe_func
    
    return raw_func
```

> 💡 **核心洞见**：`get_func()` 是执行函数的唯一出口，这意味着所有安全策略都可以在这个**单一关口**处注入。不需要在每个工具函数内部写安全检查，也不需要修改调用方的代码——这是 Manager 模式给扩展性带来的最大红利。

#### 3.7.3 业务标签

除了 `side_effect_level`，还可以在注册时为工具添加业务维度的标签：

```python
def register(self, tool_id, desc, input_schema, func, 
             side_effect_level="read", tags: list = None):
    self._tools[tool_id] = {
        # ... 原有字段 ...
        "tags": tags or [],  # 如 ["人事", "假期", "查询"]
    }

def list_by_tag(self, tag: str) -> list:
    """按业务标签筛选工具"""
    return [t for t in self._tools.values() if tag in t.get("tags", [])]
```

**应用场景**：按业务领域（人事/财务/运维）或功能类型（查询/报表/通知）快速筛选工具。

#### 3.7.4 日志记录与调用次数统计

在注册中心的每个接口方法中加入日志和统计埋点：

```python
def __init__(self):
    self._tools = {}
    self._call_count = {}   # tool_id → 调用次数
    self._call_history = [] # 最近 N 次调用记录

def get_func(self, tool_id: str) -> Callable | None:
    t = self._tools.get(tool_id)
    if not t:
        return None
    
    raw_func = t["func"]
    def logged_func(*args, **kwargs):
        # 记录调用
        self._call_count[tool_id] = self._call_count.get(tool_id, 0) + 1
        self._call_history.append({
            "tool_id": tool_id,
            "args": args,
            "kwargs": kwargs,
            "timestamp": time.time(),
        })
        # 保留最近 100 条
        if len(self._call_history) > 100:
            self._call_history = self._call_history[-100:]
        return raw_func(*args, **kwargs)
    
    return logged_func
```

#### 3.7.5 沙箱验证

在 `get_func()` 中对 write/exec 级别的工具做沙箱包裹：

> "工具可以在沙箱里面去做对应的验证。它会注入在什么位置？在 get_func 这个地方。我们现在是直接把 func 拿出来的，但我们可以先拿到 func 去做 validate 和 risk 检查，然后真正 return 的时候，返回的其实是风险检查后的结果。"

#### 3.7.6 鉴权控制

注册时标注工具所需的权限级别，在 `get_func()` 中校验调用方权限：

```python
def register(self, tool_id, desc, input_schema, func,
             side_effect_level="read", required_role: str = None):
    # required_role: 调用此工具需要的最小角色，如 "admin" / "manager" / "user"
    self._tools[tool_id] = {
        # ... 原有字段 ...
        "required_role": required_role,
    }
```

#### 扩展总结对比表

| 扩展方向 | 注入位置 | 解决的问题 |
|---|---|---|
| 分类分级 | `list_all()` / `list_specs()` | 按 level 过滤，只读场景不暴露写入工具 |
| 安全校验 | `get_func()` | 执行前统一安全检查，不改工具函数本身 |
| 业务标签 | `init` + `list_by_tag()` | 按业务维度快速筛选工具 |
| 日志统计 | `get_func()` | 记录调用历史，支持审计和监控 |
| 沙箱验证 | `get_func()` | 高风险工具隔离执行 |
| 鉴权控制 | `get_func()` | 按角色控制工具访问权限 |

> 💡 **核心洞见**：所有扩展都收敛在两个位置——**入口（`register`）**和**出口（`get_func` / `list_specs`）**。这就是 Manager 模式的力量：接口有限，扩展无限。

### 4.1 设计思路：全量注册 vs 按需激活

#### 🧠 直观理解

上一节我们建好了工具注册中心——所有工具集中存放、一处可查。但**每次请求把多少工具给模型看**，是另一个决策。

打个比方：你有一个巨大的工具箱（注册中心），里面装了几百种工具。但现在你只是要修水管——你不需要把电焊机、木工刨、喷漆枪全部摊在师傅面前。你只需要从工具箱里挑出扳手、生料带、密封圈这**一小撮**，摆在师傅手边让他选。

这就是"动态注入"：**注册中心管全量，激活层管子集。注册只做一次，激活每次不同。**

#### 📖 详细解释

在中学阶段，工具少（4-5 个），直接全量塞给模型没毛病。但当我们面向企业内部助手这样的真实场景时，工具可能膨胀到 20+、100+、甚至 1000+。三个问题随之暴露：

1. **全量塞入，token 成本线性增长**：每个工具的描述信息都会挤占上下文窗口
2. **选择太多，模型失焦**：1 万个工具让模型从里面挑一个，准确率急剧下降
3. **没有场景隔离**：只读场景只想给查询工具，写入场景只想给操作工具——但没有代码位置声明这个分组规则

解决方案的核心架构如下：

```
注册中心（全量工具池）
┌──────────────────────────────────────────┐
│ query_leave  query_hr_info  write_report │
│ send_notify  query_approval  exec_script │
│ ...               共 N 个                │
└──────────────────────────────────────────┘
         │
         │  激活层：按场景筛选
         │  意图分类 / 规则 / Tag标签 / 向量检索
         ▼
激活工具池（本次请求工作集）
┌──────────────────────┐
│ query_leave          │
│ query_hr_info        │     ← 模型只看到这几个
└──────────────────────┘
```

> 💡 **核心洞见**：注册和激活是两件完全独立的事。注册在启动时一次性完成，激活在每次请求时动态决定。**注册中心不变，激活策略可随场景灵活替换。**

> 🔗 **关联知识**：这个模式和 [第 7 课路由模式] 完全同构。第 7 课把不同的 system prompt 注入不同分支，本节课把不同的工具子集注入不同分支。两个"动态注入"是同一个设计模式的不同应用。

### 4.2 意图分类模块——让模型判断意图

#### 🧠 直观理解

激活层需要一个"裁判"来判断当前用户输入属于什么场景。这个裁判不是关键词匹配（`if "查" in input`），而是用**模型本身去理解语义**。

比如用户说"帮我看看 EMP_001 还有几天假"，模型能理解这是查询意图；用户说"生成一份 3 月统计报告"，模型能理解这是写入意图。即便是更微妙的表达——"我想了解一下假期情况，然后看看要不要发通知"——模型也能推理出这是混合意图，按写入处理。

#### 📖 详细解释

为什么不用关键词匹配？

- 关键词匹配是脆弱的：用户说"帮我看下那个东西"——没有"查"字，但明显是查询
- 关键词匹配无法处理混合意图：用户说"查完假期之后通知他"——既有查又有写
- 关键词匹配无法适应表达变化：同一个意思有无数种说法

模型驱动的意图分类能理解语义本身，不依赖表面词汇。

> 🔗 **关联知识**：这是第 8 课"意图识别"模块的具体应用。当时在文档审查助手中用意图识别做路由分发，本节课把它用于工具激活层的场景筛选。

#### 💻 代码示例

**Step 1: 环境准备与注册中心初始化**

```python
# 文件名: dynamic_injection.py
# 功能: 基于意图分类 + 注册中心的动态工具注入

import os, warnings, asyncio
warnings.filterwarnings('ignore')
from dotenv import load_dotenv, find_dotenv
from typing import Callable
from agently import Agently, TriggerFlow

# 环境配置
load_dotenv(find_dotenv())
Agently.set_settings("OpenAICompatible", {
    "base_url": os.environ.get("DEEPSEEK_BASE_URL", "https://api.deepseek.com"),
    "model":    os.environ.get("DEEPSEEK_DEFAULT_MODEL", "deepseek-v3"),
    "auth":     os.environ.get("DEEPSEEK_API_KEY", ""),
})

# ── ToolRegistry（复用上一节的注册中心）─────────────────────
class ToolRegistry:
    """工具注册中心：集中管理所有工具的能力声明和执行函数"""

    def __init__(self):
        self._tools = {}  # id → { id, desc, input_schema, func, side_effect_level }

    def register(self, tool_id: str, desc: str, input_schema: dict,
                 func: Callable, side_effect_level: str = "read"):
        """注册一个工具：能力声明 + 执行函数"""
        self._tools[tool_id] = {
            "id": tool_id,
            "desc": desc,
            "input_schema": input_schema,
            "func": func,
            "side_effect_level": side_effect_level,
        }

    def list_all(self) -> list:
        """返回全量工具清单（供激活层使用）"""
        return list(self._tools.values())

    def list_ids(self) -> list:
        """返回所有工具 id"""
        return list(self._tools.keys())

    def list_specs(self) -> list:
        """返回所有工具的能力声明列表（供激活层构建 tool schema 用）"""
        return [{"id": t["id"], "desc": t["desc"], "schema": t["input_schema"]}
                for t in self._tools.values()]

    def get_spec(self, tool_id: str) -> dict | None:
        """获取单个工具的能力声明（不含 func）"""
        t = self._tools.get(tool_id)
        if not t:
            return None
        return {k: v for k, v in t.items() if k != "func"}

    def get_func(self, tool_id: str) -> Callable | None:
        """获取工具的执行函数"""
        t = self._tools.get(tool_id)
        return t["func"] if t else None


# 创建注册中心，集中注册四个工具
registry = ToolRegistry()
registry.register("query_leave",   "查询员工剩余年假天数",
                   {"employee_id": "str"},
                   lambda employee_id: f"员工 {employee_id} 剩余年假 12 天", "read")
registry.register("query_hr_info", "查询员工基本人事信息",
                   {"employee_id": "str"},
                   lambda employee_id: f"员工 {employee_id}: 张三, 技术部, 高级工程师", "read")
registry.register("write_report",  "生成并保存分析报告",

### 4.3 Tag标签扩展——让激活层更灵活

#### 🧠 直观理解

`INTENT_TOOL_MAP` 虽然能用，但有一个明显的问题：**工具的归类是硬编码的**。加一个新工具不仅要注册，还要手动更新 `INTENT_TOOL_MAP` 里的映射。当工具数量增长到几十上百个时，这种维护方式会成为瓶颈。

更好的做法：**在注册工具时直接打上 Tag 标签**，让工具自己声明"我属于哪一类"。激活层不需要维护映射表，只需要按 Tag 从注册中心拉取。

#### 📖 详细解释

这是老师现场做的代码演进——从 v1 的硬编码 `INTENT_TOOL_MAP`，逐步升级到 v2 的 Tag 标签动态筛选。核心思路：

1. **`register()` 时增加 `tag` 参数**（可选，不填也有默认值）——最小破坏原有接口
2. **新增 `get_specs_by_tag(tag)` 方法**——按标签从注册中心筛选匹配的工具
3. **激活层不再依赖 `INTENT_TOOL_MAP`**——直接用 `classify_intent` 的输出作为 `tag` 去拉工具

这个升级看似简单，但体现了架构思维的关键原则：**只在模块内的一个地方做小改动，不影响其他代码**。

#### 💻 Step-by-Step：现场编码演进

**Step 3: 扩展 register() 方法，增加 tag 参数**

### 4.4 排错实录：`get_specs_by_tag` 的数据结构错误

#### 🐛 排错实录：dict 遍历当作 list 遍历

这是老师现场写代码时真实踩的坑。在实现 `get_specs_by_tag` 时，犯了 Python 数据结构遍历的经典错误。

**❌ 错误写法**

```python
def get_specs_by_tag(self, tag: str) -> list:
    """按 tag 筛选工具——有 bug 的版本"""
    result = []
    # 错误：对 _tools 做 for-in 遍历时，拿到的是 key（字符串），不是 tool 对象
    for tool in self._tools:
        # tool 此时是字符串 "query_leave"，不是 dict
        # tool["tag"] 会报错：string indices must be integers
        if tool["tag"] == tag:
            result.append(tool)
    return result
```

**🚨 报错信息**

```
TypeError: string indices must be integers
```

**🔍 原因分析**

`self._tools` 是一个字典，结构如下：

```python
{
    "query_leave":   { "id": "query_leave", "tag": "query", ... },
    "query_hr_info": { "id": "query_hr_info", "tag": "query", ... },
    "write_report":  { "id": "write_report",  "tag": "write", ... },
    "send_notify":   { "id": "send_notify",   "tag": "write", ... },
}
```

当使用 `for tool in self._tools` 遍历一个字典时，Python 默认遍历的是字典的**键（key）**，而不是值（value）。所以 `tool` 拿到的是字符串 `"query_leave"`，尝试用 `tool["tag"]` 去索引字符串自然就报错了。

这是一类非常常见的数据结构误用：**把 dict 遍历当成了 list 遍历**。

**✅ 正确写法**

有两种修复方式：

方式 A：遍历 `.values()`（推荐，语义最清晰）

```python
def get_specs_by_tag(self, tag: str) -> list:
    """按 tag 筛选工具——正确版本"""
    result = []
    for tool in self._tools.values():  # ← 遍历字典的值
        if tool.get("tag") == tag:
            result.append(tool)
    return result
```

方式 B：遍历 `.items()`，同时拿到 key 和 value

```python
def get_specs_by_tag(self, tag: str) -> list:
    """按 tag 筛选工具——正确版本（items 方式）"""
    result = []
    for tool_id, tool in self._tools.items():
        if tool.get("tag") == tag:
            result.append(tool)
    return result
```

**📝 经验总结**

1. **Python 字典遍历的默认行为**：`for x in dict` 遍历的是 key，不是 value。要拿 value 用 `.values()`，要同时拿 key 和 value 用 `.items()`。
2. **用 `.get()` 代替 `[]`**：`tool.get("tag")` 在 tag 不存在时返回 `None`，而 `tool["tag"]` 会抛 `KeyError`。对于可选字段，`.get()` 是更安全的实践。
3. **最小改动原则**：从发现 bug 到修复只需要改一行代码（`self._tools` → `self._tools.values()`），不需要重构整个方法。

### 4.5 完整演示：TriggerFlow ReAct Loop + 动态注入

#### 🎯 目标

将意图分类 + Tag 标签筛选 + 注册中心串联起来，嵌入第 6 课的 ReAct Loop 骨架。每次请求时：模型判断意图 → 按 tag 从注册中心取出工具子集 → 只把子集注入模型 → 模型在受限的工具集内决策调用。

**Step 5: TriggerFlow ReAct Loop + 动态注入**

```python
async def demo_dynamic_injection(user_input: str):
    """每次请求根据意图从注册中心注入不同工具子集"""
    # Step 1: 模型驱动意图分类
    intent = classify_intent(user_input)
    # v1 方式：用硬编码映射表
    active_tool_ids = INTENT_TOOL_MAP.get(intent, ["query_leave"])
    # v2 方式（升级后）：用 get_specs_by_tag 动态筛选
    # active_tool_ids = [s["id"] for s in registry.get_specs_by_tag(intent)]

    tools_desc = build_tools_desc(active_tool_ids)

    print(f"意图：{intent} → 从注册中心注入工具：{active_tool_ids}")
    print(f"模型只看到这 {len(tools_desc)} 个工具（注册中心共 {len(registry.list_specs())} 个）\n")

    # Step 2: TriggerFlow ReAct Loop（复用第 6 课骨架）
    flow = TriggerFlow()

    async def reason(data):
        state = data.input
        step = state.get("step", 0) + 1
        history = state.get("history", [])
        print(f"  [Step {step}] Reason")

        resp = (
            Agently.create_agent()
            .input({"用户需求": user_input})
            .info({"可用工具": tools_desc, "已完成步骤": history})
            .instruct([
                "根据用户需求和已完成步骤判断：是否需要调用工具？",
                "如果需要，输出 type='tool' 并指定工具名和参数",
                "如果已有足够信息或不需要工具，输出 type='final' 并给出回答",
            ])
            .output({
                "type":      ("'tool' | 'final'", "下一步动作类型"),
                "reasoning": ("str", "推理过程"),
                "tool_name": ("str | null", "type='tool' 时填工具名"),
                "tool_args": ("dict | null", "type='tool' 时填参数"),
                "answer":    ("str | null", "type='final' 时填最终回答"),
            })
            .get_response()
        )
        d = resp.get_data(ensure_keys=["type", "reasoning"])
        print(f"    推理：{d.get('reasoning', '')}")
        print(f"    决策：{d['type']}")

        state["step"] = step
        if d["type"] == "tool" and d.get("tool_name"):
            state["last_decision"] = d
            data.emit_nowait("Execute", state)
        else:
            await data.async_set_state("answer", d.get("answer", "无法处理该请求"))

    async def execute(data):
        state = data.input
        d = state["last_decision"]
        tool_name = d["tool_name"]
        tool_args = d.get("tool_args") or {}
        func = registry.get_func(tool_name)
        if func is None:
            result = f"工具 {tool_name} 未注册"
        else:
            result = func(**tool_args)
        print(f"    执行：{tool_name}({tool_args}) → {result}")

        history = state.get("history", []) + [{
            "tool": tool_name, "args": tool_args, "result": result,
        }]
        state["history"] = history
        data.emit_nowait("Reason", state)

    flow.to(reason)
    flow.when("Execute").to(execute)
    flow.when("Reason").to(reason)

    result_state = flow.start({})
    return result_state.get("history", []), result_state.get("answer", "")
```

### 4.6 方案对比：Tag vs RAG vs 模型直接选择

#### 🧠 直观理解

动态注入的关键问题是：**怎么从 N 个工具中选出 K 个激活工具？** 有至少三种思路。老师花了大量时间深入讨论了这三种方案的取舍，因为这是整个动态注入过程中决策密度最高的部分。

#### 📖 详细解释——三种方案深度对比

**方案 A：Tag 标签**

在注册时为每个工具打上分类标签，激活层按意图映射到 tag，查询注册中心中匹配该 tag 的工具。

- 优点：简单直接，完全确定性（同一输入永远得到同一工具集），不消耗额外模型调用
- 缺点：标签粒度固定，语义灵活性差；非此即彼的分类；工具多了之后 tag 本身也会膨胀爆炸

**方案 B：RAG / 向量库检索**

将所有工具的 description 预先向量化存入向量库。每次请求时，将用户的原始输入（或经过模型改写后的 query）转为向量，与工具描述向量做空间距离计算，取 top-K 最接近的工具。

- 优点：语义匹配，不受 tag 粒度的限制；向量计算成本极低；天然支持大规模（1 万+ 工具）
- 缺点：额外引入向量库基础设施；用户表达的多样性和向量匹配之间有一定 gap（需要做 query 改写来对齐）

**方案 C：模型直接选择**

把全部工具交给模型，让模型自己从列表中选出要用的工具。

- 优点：4~5 个工具时准确率最高——模型不仅能理解语义，还能结合上下文做综合判断
- 缺点：工具超过 20~30 个后模型失焦，准确率显著下降；超过上百个时 token 成本爆炸，甚至超出上下文窗口

#### 🔀 决策树

```
需要从 N 个工具中激活 K 个子集给模型：
    │
    ├── N <= 10 且对准确率要求极高？
    │   ├── 是 → 直接让模型选（方案 C）
    │   │   └── 原因：模型综合判断能力在小量级时最优
    │   └── 否 → 继续判断
    │
    ├── 场景边界清晰，能明确分类？
    │   ├── 是 → 使用 Tag 标签（方案 A）
    │   │   └── 原因：实现成本最低，确定性最高
    │   └── 否 → 使用 RAG / 向量库（方案 B）
    │       └── 原因：语义匹配不依赖人工分类
    │
    ├── N > 100 或工具还在快速增长？
    │   ├── 是 → 使用 RAG / 向量库（方案 B）
    │   │   └── 原因：向量库天然支持规模扩展
    │   └── 否 → Tag 或 RAG 均可，权衡运维成本
```

#### 📊 三方案对比表

| 维度 | Tag 标签 | RAG / 向量库 | 模型直接选择 |
|---|---|---|---|
| **核心原理** | 注册时人工打标签，激活时按标签筛选 | 语义向量空间距离计算，取 top-K | 模型阅读理解工具列表后自主决策 |
| **适用量级** | 10~200 个，分类清晰的场景 | 100~10万+，无上限 | 4~10 个，小规模 |
| **准确率** | 高（确定性匹配） | 中高（依赖 embedding 质量 + query 改写） | 小量级时最高，20+ 后快速下降 |
| **token 成本** | 极低（只多一次意图分类调用） | 低（embedding 调用 + 向量计算） | 高（每个工具的描述都占上下文） |
| **基础设施** | 无需额外组件 | 需要向量数据库 | 无需额外组件 |
| **维护成本** | 中（tag 随工具增长需要管理） | 低（自动向量化，无需人工分类） | 低（无需维护映射） |
| **灵活性** | 低（非此即彼的分类粒度） | 高（语义级别的模糊匹配） | 高（模型自主判断） |
| **爆炸风险** | tag 本身会膨胀 | 向量库天然抗膨胀 | 上下文窗口限制 + 模型失焦 |

> 💡 **核心洞见**：**没有对错，只有场景的适配度。** Tag 管理容易上手，但 tag 本质上是非此即彼的选择——当你的工具场景变得模糊、难以清晰分类时，tag 就会成为瓶颈。而向量库检索不是关键词匹配——它是语义空间的距离计算，能够容忍用户表达和工具描述之间的"语义差异"。

---

### 4.7 "空间换时间"——向量库的工程哲学

> 这是老师特别强调的一个工程思维，贯穿了整个讨论。

当使用 RAG / 向量库方案时，一个容易被忽略的关键细节是：**工具的 description 不是在被检索的那一刻才向量化的**。

```
Tag 方案的流程：
  注册工具（打 tag）→ 请求时按 tag 筛选 → 拿到工具子集
  计算发生在：请求时（O(n) 遍历）

向量库方案的流程：
  注册工具 → 立即向量化 description → 存入向量库（提前完成！）
  请求时 → query 向量化 → 空间距离计算 → top-K 排序 → 拿到工具子集
  计算发生在：注册时（向量化 description）+ 请求时（向量化 query + 距离计算）
```

`向量化 description` 是一次性的、前置的工作。当你的工具注册入库的那一瞬间，向量化就已经完成了。

`向量化 query` 是每次请求时的工作，但只是把用户的一句话转成一串数字，成本极低。

`空间距离计算` 本质上是一个标准算法——你的 query 向量是一串数字，你库中每一个工具 description 的向量也是一串数字，计算它们之间的距离。这个过程不涉及大模型推理，不消耗高额 token。

这就是"**空间换时间**"的工程策略：我把工具描述提前变成空间中一组稳定的坐标点，请求时只需要计算一下你的 query 点离哪个工具点最近。而这个"提前处理"的代价，分散在每次注册工具时完成，对请求时的毫秒级响应没有任何影响。

> 🔗 **延伸阅读**：关于 RAG 的 query 改写优化，详见 [第08课：实战——智能文档审查助手] 中的相关讨论。当用户原始表达和工具描述的语义结构不一致时（比如用户说"帮我看下那个东西" vs 工具描述"查询员工剩余年假天数"），在 query 和向量库之间加一轮模型改写能显著提升匹配准确率。

### 4.8 关键工程观察

#### 🧠 观察一：工具越多，描述越膨胀

老师分享了一个非常有意思的实证观察：

当你只有 4~5 个工具的时候，每个工具的 description 可能就一句话——"查询员工剩余年假天数"。简单明了。

但当你超过 20~30 个工具，一个很有意思的现象就会出现：**每个工具的作者都会希望自己的工具在模型选择时更"抢眼"**，于是不断地扩展和细化 description。本来一句话的描述，会演变成三四段详细说明——"该工具用于查询员工的年假剩余天数，支持按员工 ID 精确查询，会返回天数、已用天数、当年总天数等字段，适用于 HR 自助查询场景，注意需要员工 ID 作为必填参数……"

这就像一个团队汇报时，每个人都想把自己的工具有多厉害说得更详细来争取被选中。结果就是：**工具越多，每个工具的描述越长，全量注入的 token 消耗呈超线性增长。**

老师建议大家去看一下 Anthropic Skills 仓库中的 skills 描述文件——即使是一个场景非常清晰的 Word 处理 skill，description 也要写非常长的内容，才能保证模型在各种场景下准确调用。

#### 🧠 观察二：架构思维的精髓——改一行 vs 改一片

当从 `INTENT_TOOL_MAP` 硬编码升级到 `get_specs_by_tag` 动态筛选时，我们做了以下改动：

1. 在 `register()` 方法中增加了一个可选的 `tag` 参数
2. 新增了 `get_specs_by_tag()` 方法
3. 注册工具时多传一个 `tag="query"` 参数

**就这么小的改动**，其余的代码——ReAct Loop 骨架、`build_tools_desc`、`execute` handler——完全不需要动。

这就是架构的精髓：**不是在一堆混乱的代码里找位置改，而是清晰地知道你要解决的是哪个模块里的哪个小行为。** 工具注册中心的职责是管理工具元数据，那么"按什么维度分类"就应该在注册中心里改——而不是在 ReAct Loop 里、在 prompt 构建里、在工具调用链里到处改。

> 💡 **核心洞见**：架构的核心价值不是"让代码看起来整齐"，而是**让你清楚地知道，要解决某个问题时，只需要改哪个模块的哪个方法，而不影响其他部分。**

### 5.1 问题回顾：为什么需要可替换的规划与执行？

#### 🧠 直观理解

你家的路由器和你的上网设备之间有一个固定的接口——网线插口。不管你把路由器换成哪个品牌，只要网线插口标准不变（RJ45），设备就能上网。Plan Handler 和 Execution Handler 就是这个"网线插口"——接口固定，内部实现随便换。

#### 📖 详细解释

回顾 第二章中提出的**问题三：规划逻辑写死，没办法切换决策模式**。在第 06 课的 ReAct Loop 中，Plan（推理决策）和 Execute（工具执行）的逻辑直接写在 TriggerFlow 的 chunk 函数里：

```
用户输入 → [reason chunk: 模型决策 + 调用哪个工具] → [execute chunk: 查字典调函数] → 回填 → 继续循环
```

这种写法的三个痛点：

1. **审计模式做不了**：如果在审计场景下只想允许"只读"工具，你得改代码才能限制模型的决策范围。
2. **测试验证做不了**：想验证决策链路是否正确，但不希望真的调用函数（可能产生副作用），没有 mock 机制。
3. **紧急切换做不了**：生产环境模型决策出问题，想临时切回固定规则——对不起，代码是写死的。

> 💡 **核心洞见**：不是 ReAct Loop 本身有问题，而是 Plan 和 Execute 的具体逻辑应该"可注入"而不是"硬编码"。循环骨架（PLAN → EXECUTE → 回填）是对的，但骨架里的"肉"应该能换。

---

### 5.2 可替换架构的设计原则

#### 🧠 直观理解

想象一个标准化的快递包裹系统：不管你寄的是什么（书、衣服、电子产品），包裹的**外箱尺寸是固定的**（输入固定），**收件人收到的就是一个包裹**（输出固定），但**箱子里装什么、怎么包装**（中间处理过程）可以千变万化。这就是"输入固定、输出固定、中间可替换"。

#### 📖 详细解释

在 Agently 的 TriggerFlow 体系里，每一个执行块（chunk）都遵循这个标准：

```
┌──────────────────────────────────────────────────────────┐
│                  TriggerFlow 执行块标准                    │
│                                                          │
│   输入：data（包含当前上下文状态、用户输入、工具历史）        │
│                          │                               │
│                          ▼                               │
│         ┌────────────────────────────┐                   │
│         │   黑盒执行单元（可替换）      │                   │
│         │                            │                   │
│         │  · 模型自主规划              │                   │
│         │  · 固定规则                 │                   │
│         │  · 关键词匹配               │                   │
│         │  · 配置中心下发             │                   │
│         │  · A/B 测试策略             │                   │
│         └────────────────────────────┘                   │
│                          │                               │
│                          ▼                               │
│   输出：action_calls[]（选中的工具 + 参数）或 final_answer  │
│         或 tool_results[]（执行结果）                      │
└──────────────────────────────────────────────────────────┘
```

#### 🔗 推导链：插件化系统设计原则

```
起点: 一个写死的 ReAct Loop chunk 函数
  │
  ├── 第1步: 观察输入和输出的结构
  │   为什么？要替换必须先知道"边界"在哪里。如果输入输出都不确定，替换无从谈起。
  │
  ├── 第2步: 确认输入是确定的
  │   为什么？TriggerFlow 的 data 是标准入参，包含用户请求和上下文状态。
  │   无论内部逻辑怎么变，拿到的东西是一样的。
  │
  ├── 第3步: 确认输出是确定的
  │   为什么？Plan 层的输出永远是 action_calls[] 或 final_answer；
  │   Execute 层的输出永远是 tool_results[]。下游不关心上游怎么算出来的。
  │
  ├── 第4步: 中间处理过程封装为独立函数
  │   为什么？一旦输入输出确定，中间就是一个可替换的"执行单元"（execution unit）。
  │   你可以完全重写，也可以在原有基础上包裹（wrap）修改。
  │
  └── 结论: 只要遵守"输入固定、输出固定"的契约，主流程完全不需要感知 Handler 内部的实现差异。
      这就是插件化系统的前提条件。
```

### 5.3 Plan Handler 工厂——规划层的可替换实现

#### 🎯 目标

实现两个 Plan Handler 的工厂函数：`make_model_planner`（模型自主规划）和 `make_fixed_planner`（固定规则规划），理解工厂函数模式的核心机制。

> ⚠️ **关键概念**：工厂函数模式**不是类继承**。它是一个普通函数，返回一个 `async function`。调用方拿到这个闭包后直接作为 callback 注入 TriggerFlow。这种模式比类继承更轻量、更灵活——不需要定义抽象基类，不需要管理实例生命周期。

#### 🏗️ 前置准备

#### 💻 Step-by-Step

**Step 1: 搭建注册中心与工具描述构建器**

注册中心是后续所有 Handler 的基础设施。Plan Handler 需要从注册中心获取工具描述列表来告知模型"有哪些工具可用"。

```python
# ── 注册中心（复用 第三章的 ToolRegistry 模式）─────────────────
from typing import Callable

class ToolRegistry:
    """工具注册中心：集中管理所有工具的能力声明和执行函数"""
    def __init__(self):
        self._tools = {}

    def register(self, tool_id: str, desc: str, input_schema: dict,
                 func: Callable, side_effect_level: str = "read"):
        """注册一个工具：能力声明 + 执行函数"""
        self._tools[tool_id] = {
            "id": tool_id, "desc": desc, "input_schema": input_schema,
            "func": func, "side_effect_level": side_effect_level,
        }

    def list_specs(self) -> list:
        """返回所有工具的能力声明列表"""
        return [{"id": t["id"], "desc": t["desc"], "schema": t["input_schema"]}
                for t in self._tools.values()]

    def get_func(self, tool_id: str) -> Callable | None:
        """获取工具的执行函数"""
        t = self._tools.get(tool_id)
        return t["func"] if t else None


# 注册两个示例工具：一个只读（query_leave），一个写入（send_notify）
registry = ToolRegistry()
registry.register("query_leave", "查询员工剩余年假天数",
                  {"employee_id": "str"},
                  lambda employee_id: f"员工 {employee_id} 剩余年假 12 天", "read")
registry.register("send_notify", "向员工发送通知消息",
                  {"employee_id": "str", "message": "str"},
                  lambda employee_id, message: f"通知已发送给 {employee_id}", "write")


def build_tools_desc():
    """从注册中心构建工具描述列表，供 Plan Handler 注入模型请求"""
    return [{"name": s["id"], "desc": s["desc"], "args": s["schema"]}
            for s in registry.list_specs()]
```

> 这里只注册了两个工具（一个 `read`、一个 `write`），目的是让你在后续四种组合演示中能清晰观察到：规则模式只允许调用 `query_leave`，而模型模式可能会选择 `send_notify`。

**Step 2: `make_model_planner`——模型自主规划**

这是默认的 Plan Handler。工厂函数接收 `user_input` 作为参数，返回一个 `async def planner(data)` 闭包。这个闭包就是注入 TriggerFlow 的 chunk 函数。

```python
# ── Plan Handler 工厂（版本一：模型自主规划）─────────────────

def make_model_planner(user_input):
    """
    Plan Handler v1：模型自主规划（默认生产模式）

    工厂函数模式的关键：
    - 外层函数 make_model_planner 接收配置参数（user_input）
    - 内层函数 planner 是真正的 async handler，捕获外层的 user_input
    - 返回的是 planner 这个 async 函数本身，可以直接注入 TriggerFlow
    """
    async def planner(data):
        state = data.input
        tools_desc = build_tools_desc()        # 从注册中心获取工具描述
        history = state.get("tool_results", []) # 获取已执行的工具历史
        step = state.get("step", 0) + 1
        print(f"  [Plan Handler] Step {step} - 模型规划模式")

        # 构造模型请求的上下文信息
        info = {"可用工具": tools_desc}
        if history:
            info["已执行步骤"] = history

        # 使用 structured output 让模型输出结构化决策
        resp = (
            Agently.create_agent()
            .input({"用户需求": user_input})
            .info(info)
            .instruct([
                "根据用户需求和已执行步骤判断：是否需要调用工具？",
                "如果需要，输出 type='tool' 并指定工具名和参数",
                "如果已有足够信息，输出 type='final' 并给出回答",
            ])
            .output({
                "type":      ("'tool' | 'final'", "下一步动作类型"),
                "reasoning": ("str", "推理过程"),
                "tool_name": ("str | null", "type='tool' 时填工具名"),
                "tool_args": ("dict | null", "type='tool' 时填参数"),
                "answer":    ("str | null", "type='final' 时填最终回答"),
            })
            .get_response()
        )
        d = resp.get_data(ensure_keys=["type", "reasoning"])
        print(f"    推理：{d.get('reasoning', '')}")
        print(f"    决策：{d['type']}")

        state["step"] = step
        if d["type"] == "tool" and d.get("tool_name"):
            # 产出 action_calls[]——这是 Plan Handler 的标准输出格式
            state["action_calls"] = [{
                "id": f"call_{step:03d}",
                "function": {
                    "name": d["tool_name"],
                    "arguments": d.get("tool_args") or {},
                }
            }]
            data.emit_nowait("Execute", state)  # 触发执行阶段
        else:
            # 产出 final_answer——直接结束循环
            await data.async_set_state("answer", d.get("answer", "无法处理该请求"))
    return planner
```

| 行号范围 | 关键设计 | 解释 |
|---|---|---|
| 外层函数签名 | `def make_model_planner(user_input):` | 工厂函数：接收配置，返回 handler。`user_input` 被内层闭包捕获 |
| 内层函数签名 | `async def planner(data):` | 真正的 handler：签名符合 TriggerFlow chunk 标准（接收 data） |
| 工具描述注入 | `build_tools_desc()` | 每次 Plan 时动态从注册中心拉取工具列表，确保工具变更即时生效 |
| 历史上下文 | `state.get("tool_results", [])` | 将已执行的工具结果回传给模型，让模型做多步推理 |
| 标准输出 | `state["action_calls"] = [...]` | Plan Handler 的输出格式：action_calls 数组，每个元素含 id + function |
| 流程控制 | `data.emit_nowait("Execute", state)` | TriggerFlow 标准路由：产出决策后触发下一阶段 |

**Step 3: `make_fixed_planner`——固定规则规划**

这是替代 Plan Handler。它完全不调模型，直接按预设规则返回 `action_calls`。适用于审计模式、测试验证、紧急切换等场景。

```python
# ── Plan Handler 工厂（版本二：固定规则）─────────────────────

def make_fixed_planner(user_input):
    """
    Plan Handler v2：固定规则（审计/测试模式，只准查不准写）

    这个版本的 handler 完全不调用模型：
    - 第一次 Plan：强制返回 query_leave 的 action_call
    - 第二次 Plan（已有执行结果）：直接终止循环，返回固定回答
    - 注意：send_notify（写入工具）永远不被规划
    """
    async def planner(data):
        state = data.input
        # 如果已经有工具执行结果，说明第一轮已完成，直接返回最终回答
        if state.get("tool_results"):
            await data.async_set_state("answer", "查询完成（审计模式：已拦截写入操作）")
            return
        # 强制只规划 query_leave，忽略用户输入中可能请求的写入操作
        state["action_calls"] = [{
            "id": "call_001",
            "function": {"name": "query_leave", "arguments": {"employee_id": "EMP_001"}}
        }]
        print("  [Plan Handler] 审计模式：强制只允许 query_leave")
        data.emit_nowait("Execute", state)
    return planner
```

> 💡 **核心洞见**：`make_fixed_planner` 和 `make_model_planner` 的**函数签名完全一致**——都接收 `user_input`，都返回 `async def planner(data)`，产出的都是 `action_calls[]`。这就是"输入固定、输出固定、中间可替换"的具体体现。

**Step 4: 三种 Planning 替换策略对比**

实际业务中，替换 Plan Handler 有三种常见策略：

| 策略 | 做法 | 适用场景 | 示例 |
|---|---|---|---|
| **完全自定义** | 从头写一个新的 planner 函数 | 决策逻辑与模型完全无关 | 关键词匹配工具、配置中心下发策略 |
| **包裹修改（Wrap）** | 在原有 planner 基础上套一层，修改输入或输出 | 想在模型决策前后加钩子 | wrap prompt template 优化 user input、validate 门控过滤工具列表 |
| **关键词规则** | 用简单的字符串匹配替代模型推理 | 简单场景、低成本兜底 | `if "请假" in user_input → query_leave` |

```python
# ── 包裹模式示例：在模型 planner 基础上加 prompt 优化 ──
def make_wrapped_model_planner(user_input):
    """
    包裹模式（Wrap Pattern）：
    不重写 planner 逻辑，而是在原有 make_model_planner 的基础上：
    1. 对 user_input 套一层 prompt template 优化
    2. 对工具描述做门控（validate），过滤掉不允许的工具
    """
    # 优化用户输入：套一个 prompt 模板
    optimized_input = f"【安全审计模式】请只使用只读工具处理以下请求：{user_input}"

    # 复用原有的模型 planner，但传入优化后的输入
    return make_model_planner(optimized_input)
    # 注意：真实场景中还可以在 planner 内部对 tools_desc 做 validate 门控
```

> ⚠️ **注意事项**：包裹模式比完全重写更常用。因为大多数场景你并不想颠覆模型的决策能力，只是想在模型决策的上下游加点"佐料"——优化 prompt、过滤工具、记录日志。包裹模式让你复用原有逻辑，只改需要改的部分。

### 5.4 Execution Handler 工厂——执行层的可替换实现

#### 🎯 目标

实现两个 Execution Handler 的工厂函数：`make_real_executor`（真实执行）和 `make_mock_executor`（模拟执行），理解执行层如何做到"规划层不感知执行细节"。

#### 💻 Step-by-Step

**Step 1: `make_real_executor`——真实执行**

这是默认的 Execution Handler。它从 `state["action_calls"]` 中读取 Plan Handler 产出的工具调用列表，逐一通过注册中心查找并执行真实的 Python 函数。

```python
# ── Execution Handler 工厂（版本一：真实执行）─────────────────

def make_real_executor():
    """
    Execution Handler v1：真实执行（默认生产模式）

    职责：
    1. 从 state 中取出 Plan Handler 产出的 action_calls[]
    2. 遍历每个 action_call，通过注册中心查找对应的执行函数
    3. 执行函数，收集结果
    4. 将结果写入 state["tool_results"]，触发下一轮 Plan
    """
    async def executor(data):
        state = data.input
        results = []
        # 用 pop 取走 action_calls，避免重复执行
        for call in state.pop("action_calls", []):
            name = call["function"]["name"]
            args = call["function"]["arguments"]
            # 从注册中心获取真正的执行函数
            func = registry.get_func(name)
            if func is None:
                result = f"工具 {name} 未注册"
            else:
                result = func(**args)  # 真实调用
            print(f"  [Execute Handler] 真实执行 {name}({args}) → {result}")
            results.append({"tool": name, "args": args, "result": result})
        # 将结果追加到工具执行历史中
        state["tool_results"] = state.get("tool_results", []) + results
        await data.async_set_state("tool_results", state["tool_results"])
        data.emit_nowait("Reason", state)  # 回填结果，触发下一轮 Plan
    return executor
```

**Step 2: `make_mock_executor`——模拟执行**

Mock 执行器不真正调用函数，而是返回模拟数据。这在测试决策链路时非常有用——你关心的是"模型选了什么工具、传了什么参数"，而不是"工具执行的实际结果"。

```python
# ── Execution Handler 工厂（版本二：mock 执行）────────────────

def make_mock_executor():
    """
    Execution Handler v2：mock 执行（测试/验证模式）

    与 real_executor 的关键区别：
    - 不从注册中心查找函数，不执行真实逻辑
    - 返回格式化的模拟数据：f"[mock] 模拟执行 {name}"
    - 在每个调用处打印日志，方便追踪决策链路
    - 函数签名和输出格式与 real_executor 完全一致

    适用场景：
    - 测试 Plan Handler 的决策准确率
    - 验证工具调用链路是否按预期流转
    - 不想产生副作用的纯验证场景
    """
    async def executor(data):
        state = data.input
        results = []
        for call in state.pop("action_calls", []):
            name = call["function"]["name"]
            results.append({
                "tool": name,
                "args": call["function"]["arguments"],
                "result": f"[mock] 模拟执行 {name}"  # 模拟数据，不真调
            })
            print(f"  [Execute Handler] mock {name} → 返回模拟数据")
        state["tool_results"] = state.get("tool_results", []) + results
        await data.async_set_state("tool_results", state["tool_results"])
        data.emit_nowait("Reason", state)
    return executor
```

| 对比维度 | make_real_executor | make_mock_executor |
|---|---|---|
| 函数调用 | 通过 `registry.get_func()` 真实执行 | 不查注册中心，直接返回模拟字符串 |
| 副作用 | 可能产生真实副作用（发通知、写报告） | 零副作用 |
| 日志输出 | `真实执行 {name}({args}) → {result}` | `mock {name} → 返回模拟数据` |
| 使用场景 | 生产环境、需要真实结果 | 测试决策链路、验证工具选择 |
| 接口契约 | 输入 `data`，输出写入 `state["tool_results"]` | **完全相同** |

> 💡 **核心洞见**：`make_real_executor` 和 `make_mock_executor` 的函数签名和输出格式完全一致。这意味着你可以把 mock executor 注入到任何需要 Execution Handler 的地方——流水线不感知、不报错、不需要任何适配代码。这就是"接口契约"的力量。

**Step 3: 包裹模式——在真实 executor 前后注入副作用**

```python
# ── 包裹模式示例：在真实 executor 基础上加审计日志 ──
def make_audited_executor():
    """
    包裹模式（Wrap Pattern）：
    在真实 executor 的外层包裹审计日志逻辑。
    不修改原有 executor 的核心行为，只在前后加钩子。
    """
    real_executor = make_real_executor()

    async def audited_executor(data):
        state = data.input
        # 前置钩子：记录即将执行的操作
        calls = state.get("action_calls", [])
        print(f"  [Audit] 即将执行 {len(calls)} 个操作: {[c['function']['name'] for c in calls]}")

        # 执行原有的真实 executor
        await real_executor(data)

        # 后置钩子：记录执行结果
        results = state.get("tool_results", [])
        print(f"  [Audit] 执行完成，共 {len(results)} 条结果")

    return audited_executor
```

### 5.5 可插拔 ReAct Loop——工厂函数组装与四种组合演示

#### 🎯 目标

构建 `run_react_loop` 工厂函数，然后分别注入不同的 Plan Handler 和 Execution Handler，演示四种组合的实际运行效果。

#### 💻 Step-by-Step

**Step 1: `run_react_loop` 工厂函数——骨架固定，两端可注入**

```python
# ── 可插拔 ReAct Loop 工厂 ─────────────────────────────────

async def run_react_loop(plan_handler, exec_handler):
    """
    ReAct Loop 骨架工厂函数。

    骨架逻辑：
      PLAN（推理决策）→ EXECUTE（执行工具）→ 回填结果 → 继续 PLAN
    这个循环结构永远不会变。

    可变的是：
      - plan_handler：决定"调哪些、按什么顺序"
      - exec_handler：决定"怎么调起来"

    两个 handler 由调用方通过参数注入，替换时不需要修改任何 TriggerFlow 代码。
    """
    flow = TriggerFlow()

    async def reason(data):
        """Plan 阶段：直接委托给注入的 plan_handler"""
        await plan_handler(data)

    async def execute(data):
        """Execute 阶段：直接委托给注入的 exec_handler"""
        await exec_handler(data)

    # TriggerFlow 路由编排
    flow.to(reason)                    # 入口：先做 Plan
    flow.when("Execute").to(execute)   # Plan 产出 action_calls → 执行
    flow.when("Reason").to(reason)     # 执行完成 → 回到 Plan，循环

    result = flow.start({})
    return result.get("tool_results", []), result.get("answer", "")


# ── 测试输入 ──────────────────────────────────────────────
USER_INPUT = "查 EMP_001 的剩余假期，然后通知他"
```

> 💡 **核心洞见**：`run_react_loop` 的主流程只有两行实质代码——`await plan_handler(data)` 和 `await exec_handler(data)`。**主流程完全不知道 handler 是模型规划还是规则规划，是真实执行还是 mock 执行**。这就是插件化系统的终极形态：主流程像一列火车，Handler 像可更换的车厢——只要车厢的挂钩（接口）标准，什么车厢都能挂上去。

**Step 2: 四种组合演示**

```python

#### 🤔 如何选择？

```

    │
    └── 需要部分修改原有逻辑？
        └── 使用包裹模式（Wrap Pattern），不重写，只加钩子
```

### 5.6 插件化设计思想与架构洞察

#### 📖 详细解释

这一节我们一直在讲"可替换"，但可替换的前提是什么？答案是**接口契约**。没有明确的接口契约，替换就无从谈起。

```
┌──────────────────────────────────────────────────────────────┐
│                     插件化系统的三个条件                       │
│                                                              │
│  条件一：输入固定                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Plan Handler 的输入：data（含 user_input, history）    │   │
│  │ Exec Handler 的输入：data（含 action_calls[]）         │   │
│  │ 无论内部实现怎么变，拿到的 data 结构不变               │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  条件二：输出固定                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Plan Handler 的输出：action_calls[] 或 final_answer    │   │
│  │ Exec Handler 的输出：tool_results[]                   │   │
│  │ 下游不关心上游怎么算的，只看格式对不对                 │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  条件三：主流程不感知插件内部                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ run_react_loop 的 reason/execute chunk 只有一行：      │   │
│  │   await plan_handler(data)                            │   │
│  │   await exec_handler(data)                            │   │
│  │ 这行代码不知道、也不需要知道 handler 内部是什么        │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  三者具备 → 任意插件只要满足输入输出契约即可接入               │
└──────────────────────────────────────────────────────────────┘
```

> 💡 **核心洞见：Plan 层不应该关心 Execution 层的具体实现差异**

Plan Handler 的职责是"面对这个业务问题，我应该选择什么行为方案"。它输出的是一个**决策**（调哪个工具、传什么参数）。至于这个决策怎么被执行——是本地 Python 函数、HTTP 请求、MCP 协议调用、还是沙盒进程——Plan Handler 完全不需要知道。

这种分层隔离带来三个架构收益：

1. **独立演进**：Execution Handler 新增一种执行方式（比如接入 MCP 工具），Plan Handler 零改动。
2. **独立测试**：测试 Plan 逻辑时用 mock Executor，测试 Exec 逻辑时用固定 Planner。两个维度独立验证。
3. **独立替换**：生产环境出问题时，可以只替换 Plan Handler（切规则模式）而不影响 Execute，反之亦然。

```
Plan Handler（决策层）           Execution Handler（执行层）
┌─────────────────────┐         ┌──────────────────────────┐
│ "该查年假，用         │         │                          │
│  query_leave"        │────────▶│  可以是：                 │
│                      │         │  · 本地函数调用           │
│  不知道也不关心       │         │  · HTTP API 请求          │
│  下面怎么执行        │         │  · MCP 协议调用           │
│                      │         │  · 沙盒进程执行           │
└─────────────────────┘         │  · Skills 技能包          │
                                └──────────────────────────┘
```

> 📍 **系列定位**：这个分层架构直接为 第六章（工具类型扩展与归一化）铺路——正是因为在 Plan 层不关心执行细节，我们才能在 Execution 层自由扩展本地函数、HTTP、MCP、沙盒等多种执行方式，而不需要修改任何 Plan 层代码。

### 5.7 关键观察与总结

#### 💡 一句话总结

> ReAct Loop 的骨架是不变的（PLAN → EXECUTE → 回填），但 Plan 和 Execute 的具体实现通过工厂函数模式实现完全可替换——只要遵守"输入固定、输出固定"的接口契约，四种 Handler 组合可以自由搭配，覆盖生产、审计、测试、验证等全部场景。

### 6.1 工具类型全景——工具不只是本地函数

#### 🧠 直观理解

到目前为止，我们讨论的工具都是 Python 本地函数——定义一个 `query_leave`，注册到 `ToolRegistry`，ReAct Loop 里 `func(**args)` 一行调用。但在真实系统里，一个"工具"可能是本地函数，也可能是一个远程 HTTP 服务、一个 MCP 协议连接的工具、甚至是一个运行在沙盒里的脚本进程。它们的调用方式截然不同，但站在 Plan Handler 的视角，它只需要知道"这个工具叫什么、收什么、返回什么"——底层怎么"调起来"是 Execution Handler 的事。

就好比餐厅的菜单：食客（Plan Handler）只需要看菜单上的菜名和描述做选择，至于这道菜是厨房现炒（本地函数）、中央厨房配送（HTTP API）、外包供应商制作（MCP工具）、还是需要特殊设备处理（沙盒），那是后厨（Execution Handler）的事。

#### 📖 详细解释

工具不止一种形态，四种典型类型各有不同的调用方式和额外处理需求：

| 工具类型 | 怎么调 | 要额外处理什么 | 典型场景 |
|---|---|---|---|
| **本地函数** | 同步/异步 Python 调用 `func(**kwargs)` | 几乎不需要额外处理 | s01-s03 中的所有 demo 工具 |
| **HTTP API** | HTTP 请求（GET/POST），解析 JSON 响应 | 鉴权（Bearer Token / API Key）、超时控制、重试策略（指数退避）、HTTP 错误码 → 业务错误码转换 | 调用内部微服务、第三方开放平台 |
| **MCP 工具** | MCP 协议 `client.call_tool(name, arguments)` | 协议握手（initialize）、连接管理（长连接/重连）、工具发现（list_tools）、会话生命周期管理 | 标准化接入外部工具、IDE 插件生态 |
| **沙盒进程** | subprocess / worker 容器，stdin/stdout 通信 | 环境隔离（Docker/虚拟环境）、资源限制（内存/CPU/执行时间）、文件系统沙箱、审批流程（敏感操作确认） | 执行用户提交的代码、运行不受信任的脚本 |

> 🔗 **关联知识**：MCP 工具的具体实现详见 第10课：MCP 工具标准化协议；沙盒执行详见 第11课：沙盒与命令行执行。

> 💡 **核心洞见**：四种工具类型的差异只存在于"怎么调起来"这一层——即 Execution 层。Plan 层不需要也不应该关心这些差异。这就是"归一化"思想的起点：用统一接口屏蔽底层多样性。

### 6.2 归一化——Plan 层与 Execution 层的彻底分离

#### 🧠 直观理解

"归一化"的意思是：把不同工具类型的调用方式统一成同一个接口，让上层只面对一种标准格式。Plan Handler 看到的永远是 `ActionSpec`（工具叫什么、收什么、返回什么），Execution Handler 负责根据工具类型选择对应的 Executor 去执行。

#### 📖 详细解释

回看 React Loop 的工作流程：模型拿到工具描述 → 决定调哪个工具 → 传入参数 → 执行 → 拿到结果 → 回填上下文 → 继续循环。在这个流程里：

- **Plan Handler** 负责前半段：根据用户需求和工具描述（ActionSpec），决定"调哪些、按什么顺序"。它的输入是 ActionSpec 列表，输出是 `action_calls[]`。
- **Execution Handler** 负责后半段：接收 `action_calls[]`，根据每条调用的工具类型，选择对应的 Executor 去真正执行，输出 `ActionResult[]`。

Plan Handler 完全不需要知道 `query_leave` 是本地函数还是 HTTP API——它只需要知道"这个工具叫 `query_leave`，收一个 `employee_id` 参数，返回年假天数"。具体怎么调起来，是 Execution Handler 根据注册时标记的工具类型去分发的。

#### 🔗 推导链：为什么需要归一化

```
起点：s01-s03 只用了本地 Python 函数，一个简单的 func(**args) 搞定所有工具调用。
  │
  ├── 第1步：系统接入 HTTP API 类型的工具。
  │   为什么？内部微服务通过 RESTful API 暴露能力，不能直接 import 函数调用。
  │   如果 Plan Handler 的代码里写死了 func(**args)，接 HTTP API 就得改 Plan Handler，
  │   这违反了"Plan 层不关心执行细节"的原则。
  │
  ├── 第2步：系统又接入了 MCP 协议工具和沙盒执行。
  │   为什么？标准化协议和受限执行已成为企业级 AI 应用的标配需求。
  │   如果每加一种工具类型就要改 Plan Handler 和注册中心，架构就崩了。
  │
  ├── 第3步：抽象出统一的 Execution Handler 接口。
  │   为什么？让 Execution Handler 内部根据工具类型路由到不同的 Executor
  │   （LocalFunctionExecutor / HTTPExecutor / MCPClientExecutor / SandboxExecutor），
  │   Plan Handler 完全不感知这些差异。加新工具类型 = 加一个 Executor 实现，不改其他代码。
  │
  └── 结论：归一化的本质是"接口统一，实现多样"——Plan 层面对统一的 ActionSpec，
       Execution 层内部做类型分发。两层彻底解耦。
```

#### 🔀 对比辨析：归一化前 vs 归一化后

| 维度 | ❌ 归一化前（所有工具当函数处理） | ✅ 归一化后（统一 Execution Handler 接口） |
|---|---|---|
| 加新工具类型 | 改 Plan Handler + 改注册中心 + 改调用代码 | 只加一个 Executor 实现 |
| 测试/Mock | 需要 mock 每种工具类型的调用路径 | 统一替换 Execution Handler 为 Mock Executor |
| 工具调用代码 | `func(**args)` 硬编码 | `executor.execute(call)` 统一接口 |
| Plan Handler 关注点 | 既要选工具，又要知道怎么调 | 只选工具，不管执行 |

#### 💻 执行层归一化架构图

```
Plan Handler → 决定调什么（基于 ActionSpec）
      │
      │  产出 action_calls[]
      ▼
┌─────────────────────────────────────────────────┐
│         Execution Handler（可替换的统一接口）      │
│                                                  │
│  接收 action_calls[] → 按工具类型路由 → 返回结果   │
│                                                  │
│         ┌──────────────────────────┐            │
│         │    ActionDispatcher      │            │
│         │   （类型路由/分发器）      │            │
│         └──────────┬───────────────┘            │
│                    │                             │
│     ┌──────────────┼──────────────┐             │
│     ▼              ▼              ▼             │
│  ┌────────┐  ┌──────────┐  ┌──────────┐       │
│  │LocalFn │  │HTTP      │  │MCP       │  ...  │
│  │Executor│  │Executor  │  │Executor  │       │
│  └────────┘  └──────────┘  └──────────┘       │
│     │              │              │             │
│     ▼              ▼              ▼             │
│  func()       requests.     mcp.call_          │
│               post()        tool()             │
│                                                  │
│  沙盒 Executor 等其他类型按同样方式插入            │
└─────────────────────────────────────────────────┘
      │
      │  产出 ActionResult[]
      ▼
  回填进模型上下文，ReAct Loop 继续
```

> ⚠️ **注意事项**：归一化不是说所有工具必须用同一种底层技术去调——本地函数还是走 `func()`，HTTP 还是走 `requests`。归一化指的是**接口层面**的归一：Execution Handler 对外暴露统一的 `execute(action_calls) -> results` 接口，内部才根据工具类型做路由分发。

> 💡 **核心洞见**：抽象层级升级——从 `tool`（工具）到 `action`（行为）。`tool` 强调"这是个什么东西"（函数、API、MCP 连接），`action` 强调"它能做什么"（查询假期、发送通知）。对于 Plan Handler，重要的是行为语义而非工具形态。这就是为什么 Agently 框架用 `ActionSpec` 而不是 `ToolSpec` 来命名——它刻意强调了这一层抽象升级。

### 6.3 Agently 框架对照——手工实现的每一层，框架如何一行封装

#### 🧠 直观理解

s01-s03 我们用纯 Python + TriggerFlow 手工搭建了工具管理的三层架构：注册中心（s01）、动态注入（s02）、可替换 Handler（s03）。这些手工代码让你看清了架构的每一块砖是怎么垒起来的。现在，Agently 框架把这三层封装成了高层 API——**理解底层后，框架对你来说不是黑盒，而是已知原理的语法糖**。

#### 📖 详细解释

在你开始看 Agently 代码之前，先把手工实现和框架 API 的对应关系印在脑子里：

| 架构概念 | s01-s03 手工实现 | Agently API | 对应关系说明 |
|---|---|---|---|
| 能力声明 | `dict` 手动构造 `{"id":..., "desc":..., "input_schema":...}` | `ActionSpec` | 框架内部自动根据函数签名和 docstring 生成 |
| 注册中心 | 手写 `ToolRegistry` 类（~30 行） | `ActionRegistry` | `agent.action.action_registry` 可访问 |
| 注册工具 | `registry.register(id, desc, schema, func)` | `@agent.action_func` 装饰器 或 `agent.action.register_action()` | 两种注册方式，详见下方说明 |
| 动态注入 | `INTENT_TOOL_MAP` + `classify_intent()` + `build_tools_desc()` | `agent.use_actions([subset])` | 一行替代手工的意图分类 + 子集筛选 |
| Plan Handler | `make_model_planner()` / `make_fixed_planner()` | `agent.register_action_planning_handler(fn)` | 接口签名不同，功能等价 |
| Execution Handler | `make_real_executor()` / `make_mock_executor()` | `agent.register_action_execution_handler(fn)` | 接口签名不同，功能等价 |
| 执行结果 | `state["tool_results"]` 手动管理 | `ActionResult` 统一结构 | 框架自动回填上下文 |
| 策略约束 | `side_effect_level` 字段手动检查 | `ActionPolicy` | 框架内置副作用检查、审批流程等 |

> ⚠️ **注意事项**：Agently 提供了两种注册方式，它们的区别很重要——`@agent.action_func` 装饰器会自动将工具绑定到当前 agent（即自动加 agent tag），无需 `use_actions()` 显式激活；而 `agent.action.register_action()` 只向注册中心写入 spec 和 executor，不会自动绑定当前 agent，需要用 `use_actions()` 手动激活。这两者的差异直接影响到工具是否在 agent 的可用列表中。

#### 💻 代码示例：Agently 完整封装演示

以下代码将 s01-s03 的手工逻辑全部用 Agently 高层 API 重写。注意代码中的注释标注了每一处与手工实现的对应关系。

```python
# 文件名: s04_agently_encapsulation.py
# 功能: 用 Agently 高层 API 封装 s01-s03 的手工搭建逻辑
# 对应: s01 ToolRegistry → ActionRegistry
#       s02 INTENT_TOOL_MAP → use_actions()
#       s03 Handler Factories → register_action_planning/execution_handler()

import os, warnings

warnings.filterwarnings("ignore")
from dotenv import load_dotenv, find_dotenv
from agently import Agently

load_dotenv(find_dotenv())
Agently.set_settings(
    "OpenAICompatible",
    {
        "base_url": os.environ.get("DEEPSEEK_BASE_URL", "https://api.deepseek.com"),
        "model": os.environ.get("DEEPSEEK_DEFAULT_MODEL", "deepseek-v3"),
        "auth": os.environ.get("DEEPSEEK_API_KEY", ""),
    },
)


def make_agent_with_actions():
    """
    创建一个预装了两个工具的新 Agent。
    等价于 s01 中创建 ToolRegistry 并 register 两个工具。
    
    两种注册方式对比：
    - @agent.action_func: 装饰器方式，自动绑定到当前 agent，无需 use_actions() 显式激活
    - agent.action.register_action(): 底层 API，只向注册中心写入，需 use_actions() 手动激活
    """
    a = Agently.create_agent()
    a.set_agent_prompt("system", "你是企业内部助手，按需调用工具。")

    # 方式一：@agent.action_func 装饰器（推荐）
    # 自动加当前 agent 的 tag，注册后已在 agent 的 action 使用列表里
    @a.action_func
    def query_leave(employee_id: str) -> dict:
        "查询员工剩余年假天数"
        return {"employee_id": employee_id, "days_remaining": 12}

    # 方式二：agent.action.register_action() 底层 API
    # 只往 action registry 里注册 spec 和 executor，不会自动加 agent tag
    # 用这种方式注册的 action，需要 use_actions() 显式激活
    a.action.register_action(
        action_id="send_notify",
        desc="向员工发送通知消息",
        kwargs={"employee_id": (str, "员工ID"), "message": (str, "通知内容")},
        func=lambda employee_id, message: f"通知已发送给 {employee_id}",
        expose_to_model=True,
    )

    # 方式三（注释展示）：agent.register_action() 扩展方法
    # 属于 ActionExtension，会自动把 action 绑定到当前 agent
    # a.register_action(
    #     name="send_notify",
    #     desc="向员工发送通知消息",
    #     kwargs={"employee_id": (str, "员工ID"), "message": (str, "通知内容")},
    #     func=lambda employee_id, message: f"通知已发送给 {employee_id}",
    # )

    return a


# ── 对应关系展示：查看注册中心 ──────────────────────────────
agent = make_agent_with_actions()

# 查看注册中心 → 等价于 s01 的 registry.list_all()
print("── Agently 注册中心内容 ──")
for aid in agent.action.action_registry.list_action_ids():
    spec = agent.action.action_registry.get_spec(aid)
    if spec:
        desc = spec.get("desc", "")[:50]
        print(f"  {aid}: {desc}")

# s02 手工版: INTENT_TOOL_MAP + classify_intent → 动态选取工具子集
# s04 Agently 版: use_actions() 一行激活工具子集
# 注意: query_leave 已被 @a.action_func 自动激活，send_notify 需显式激活
agent.use_actions(["query_leave", "send_notify"])
print(f"\n激活子集：['query_leave', 'send_notify']（等价于 s02 的 INTENT_TOOL_MAP 筛选）\n")

# ── 场景一：默认模式 ──────────────────────────────────────
# 等价于 s03 的 make_model_planner + make_real_executor
print("── 默认模式：模型自主规划 + 真实执行 ──")
agent.set_action_loop(max_rounds=3)
agent.input("查 EMP_001 的剩余假期，然后通知他")
for r in agent.get_action_result():
    print(f"  [{r.get('action_id')}] → {r.get('result')}")
print(f"回答：{agent.get_text()[:100]}...\n")

# ── 场景二：替换 Plan Handler ─────────────────────────────
# 等价于 s03 的 make_fixed_planner 审计模式
print("── 审计模式：替换 Plan Handler ──")
a2 = make_agent_with_actions()
a2.use_actions(["query_leave", "send_notify"])
a2.set_action_loop(max_rounds=3)


async def audit_planner(context, request):
    """自定义 Plan Handler：审计模式，只允许读取操作"""
    done = context.get("done_plans", [])
    if done:
        return {"next_action": "response"}
    return {
        "next_action": "execute",
        "action_calls": [
            {
                "action_id": "query_leave",
                "purpose": "审计模式：只允许读取",
                "action_input": {"employee_id": "EMP_001"},
            }
        ],
    }


a2.register_action_planning_handler(audit_planner)
a2.input("查 EMP_001 的剩余假期，然后通知他")
for r in a2.get_action_result():
    print(f"  [{r.get('action_id')}] → {r.get('result')}")
print("注意：plan handler 拦截了写入操作\n")

# ── 场景三：替换 Execution Handler ─────────────────────────
# 等价于 s03 的 make_mock_executor 测试模式
print("── Mock 模式：替换 Execution Handler ──")
a3 = make_agent_with_actions()
a3.use_actions(["query_leave", "send_notify"])
a3.set_action_loop(max_rounds=3)


async def mock_executor(context, request):
    """自定义 Execution Handler：Mock 模式，不真正执行工具"""
    results = []
    for call in request.get("action_calls", []):
        results.append(
            {
                "ok": True,
                "status": "success",
                "action_id": call.get("action_id"),
                "result": f'[mock] 模拟执行 {call.get("action_id")}',
            }
        )

> 🐞 **常见坑**：`agent.action.register_action()` 和 `agent.register_action()` 是两个不同的方法。前者是底层 Action 管理器的方法，只注册不绑定 agent；后者是 ActionExtension 的扩展方法，会自动绑定到当前 agent。如果用了前者却没调 `use_actions()`，工具虽然在注册中心里，但模型看不到它。这是最常见的"注册了但模型不调"的原因。

**三层架构总结**：

| 层级 | 职责 | 特征 |
|---|---|---|
| **契约层**（注册中心） | 定义系统有哪些行为可用 | 稳定，随工具增长迭代 |
| **激活层**（动态注入） | 决定本次请求暴露哪些工具给模型 | 灵活，随场景动态切换 |
| **执行层**（Execution Handler） | 真正执行工具调用，屏蔽底层差异 | 可扩展，随新工具类型接入 |

> **手工搭建让你理解架构，框架封装让你高效开发。理解底层后，用框架才不会"瞎用"。**

如果你只学会 `@agent.action_func` 和 `use_actions()`，但不知道这些 API 背后是对应注册中心、动态注入、可替换 Handler 这套架构，那么当框架的行为不符合预期时，你只能猜测和试错。反过来，当你把 s01-s03 的手工代码吃透，再看 Agently 的任何封装——你看到的不再是魔术，而是你亲手搭过的架子披了一件语法糖的外衣。

#### ❓ 常见疑问

**Q: Skills 是不是就是一种复杂一点的工具？**

A: 不是。Skills（技能包）经常被误认为是"复杂一点的工具"，但它和工具根本不在同一层。工具（Action）是执行层的概念——它是一个可以被模型调用的原子行为。Skills 是资产层的概念——它包含 references、scripts、依赖和版本，是一个可以被分发和复用的能力包。Skills 的消费方式也不同于工具调用回路，它的复杂度更高，我们会在后续 Skills 章节单独展开讨论。这个可以作为你今天思考的延伸问题。

**Q: 如果不用 Agently，用 LangChain 或原生代码，这套架构还适用吗？**

A: 完全适用。注册中心、动态注入、Handler 可替换——这三个模块是架构层面的设计模式，不依赖任何特定框架。LangChain 的 `@tool` 装饰器就是注册中心的语法糖，`AgentExecutor` 里的 `plan` 和 `step` 就是 Plan/Execution Handler 的框架实现。理解这套架构后，换任何框架都只是换 API 语法，概念完全保留。

### 7.1 答疑：Agent找工具是通过向量匹配吗？

---

**🙋 问题原文**

> 张同学问："agent 找工具，主要是通过向量匹配吗？"

**📖 背景**

在第四章中，课程讨论了 Tag 过滤和 RAG 检索两种方案，用来从海量工具中筛选出当前任务最相关的工具。张同学由此产生一个非常自然的联想：Agent 在运行时找工具，是不是靠向量匹配来实现的？这个联想很合理——毕竟 RAG 方案确实用到了向量检索。但老师给出了一个让人意外的答案。

**💬 老师回答**

**结论先行：不是。** Agent 默认找工具的方式，**不是**通过向量匹配。

这里面有两个完全不同的机制，理解它们的区别至关重要。

---

**机制一：上下文学习（In-Context Learning）—— Agent 的默认方式**

当 Agent 拿到全量的工具列表（tool definitions，包含每个工具的 `name`、`description`、`parameters` 等字段）时，它会把这些工具定义信息**全部放入上下文（context）中**。然后，Agent 依赖大语言模型自身的语言理解和推理能力，从这些描述中直接判断哪个工具适合当前任务。

```
┌─────────────────────────────┐
│        用户输入               │
│    "帮我查一下今天天气"        │
└─────────────┬───────────────┘
              │
              ▼
┌──────────────────────────────────────────────────┐
│              上下文窗口 (Context Window)            │
│                                                   │
│  [System] 你可以使用以下工具：                       │
│                                                   │
│  tool_1: { name: "get_weather",                  │
│            description: "查询指定城市的实时天气" }    │
│                                                   │
│  tool_2: { name: "get_stock_price",              │
│            description: "查询股票实时价格" }         │
│                                                   │
│  tool_3: { name: "search_database",              │
│            description: "在内部数据库中搜索信息" }    │
│                                                   │
│  [User] 帮我查一下今天天气                           │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│              大语言模型 (LLM) 推理                   │
│                                                   │
│  "用户想查天气。tool_1 的描述是'查询天气'，          │
│   语义上完全匹配。应该调用 get_weather。"            │
│                                                   │
│  ▶ 模型通过「读懂」工具描述来做出选择                   │
│  ▶ 没有向量计算、没有索引、没有距离排序                │
└──────────────────────────────────────────────────┘
```

这种方式的本质是：**模型用自己的语义理解能力去"读"工具描述，然后做出判断。** 整个过程是模型在推理，而不是外部工程系统在计算。

---

**机制二：向量匹配（Vector Matching）—— 外挂的工程逻辑**

向量匹配则走了另一条完全不同的路。它完全不依赖模型的理解能力，而是依赖数学计算——在语义向量空间中计算两个向量之间的距离。

```
用户输入 → Embedding 模型 → 向量结果 → 与向量库中的向量做距离计算 → 排序 → 返回最相似的条目
```

最关键的区分在于：**向量匹配的整个过程，语言模型完全没有参与。** 它是在模型推理之"前"、之"外"完成的一个纯工程计算步骤。只有当向量检索完成、最相关的工具描述被筛选出来之后，这些内容才会被注入到上下文中，此时大模型才开始工作。

> 💡 **核心洞见**：上下文学习 vs 向量匹配，本质是"模型自己动脑子判断"和"数学帮你算出来"两条路径。前者依赖模型的语义理解，后者依赖向量空间中的距离度量。两者不是互斥关系——你完全可以在向量匹配筛选出候选工具后，再交给模型做最终决策。

**🤔 延伸思考**

当工具数量很少（几个到几十个）时，全量放入上下文的成本极低，上下文学习完全够用。但当工具数量膨胀到成百上千个时，全量放入上下文就不切实际了——不仅大幅消耗 token 预算，还可能超出模型的上下文窗口限制。此时，向量匹配就成为一种高效的"预筛选"机制：**先用向量检索从海量工具中筛出最相关的前 K 个，再把它们放入上下文让模型做最终选择。** 这正是第四章中 Tag 与 RAG 方案讨论的现实背景。

#### 二维坐标简化演示

上面展示的是高维空间中的向量计算，对于人脑来说很难直观想象。老师为了帮助理解核心原理，将向量降维到二维平面来做演示：

```
         y 轴
         ▲
         │
       6 ┤                           ● A (1, 5)
         │                             \
       5 ┤                              \
         │                               \  距离 ≈ 5.10
       4 ┤                    ● C (2, 4)
         │                      \
       3 ┤           ● B (1, 3)  \
         │            \           \  距离 ≈ 4.47
       2 ┤             \           \
         │              \ 距离≈3.16 \
       1 ┤               \           \
         │                \           \
       0 ┼────●────────────┼───────────┼──────────▶ x 轴
         │   Query                      \
         │   (0, 0)                      查询向量在原点
         │
         │   三个工具的 description 向量分别位于:
         │   A (1, 5)、B (1, 3)、C (2, 4)
         │
         ▼
```

在二维平面上，欧氏距离（Euclidean Distance）的计算公式为：

```
d = √[(x₂ - x₁)² + (y₂ - y₁)²]
```

将查询向量 (0, 0) 作为基准点，分别计算与三个工具向量的距离：

| 工具 | 坐标 | 距离计算式 | 距离值 | 排序 |
|------|------|-----------|--------|------|
| **工具 B** | (1, 3) | √(1² + 3²) = √10 | ≈ 3.16 | **第 1 近** |
| **工具 C** | (2, 4) | √(2² + 4²) = √20 | ≈ 4.47 | 第 2 |
| **工具 A** | (1, 5) | √(1² + 5²) = √26 | ≈ 5.10 | 第 3 远 |

因此，如果设定 `Top-1`（只取距离最近的那一个结果），则**工具 B** 胜出。它的向量与查询向量在语义空间中距离最近，这意味着它的描述（"这个工具是干什么的"）与用户的意图最接近。工具 B 对应的实际内容会被取出，注入到模型的上下文中。

---

#### 关键要点总结

> ⚠️ **关键区分一："整个过程没有语言模型参与"**
>
> 从用户输入被 Embedding 成向量，到与 Index 中的向量做距离计算，到排序取 Top-K——**这整段流程里，语言模型一步都没有参与**。这是一个纯粹的工程计算过程。许多初学者容易误以为向量检索是大模型"理解"了语义之后做出的匹配，但事实上，它完全是数学上的距离度量。大模型只在检索结果被注入到上下文之后，才开始它的推理工作。

> ⚠️ **关键区分二："空间换时间"——Embedding 发生在入库时，而非查询时**
>
> 向量数据库中 Index 里的所有向量（即所有工具 description 的向量化结果），是在**入库阶段**就已经提前通过 Embedding 模型完成向量化的。用户查询时，不需要再对库中的内容做 Embedding，只需要对「用户输入」这一个文本做一次 Embedding，然后直接进行距离计算。这就像图书馆的书在入库时就编好了分类号和架位，读者来借书时直接查目录就行，不需要每次重新把所有书分类一遍。

> 🔗 **关联知识**：向量检索是 RAG（检索增强生成）的工程核心。详见 [第四章：动态注入](#四动态注入意图分类tag标签与按需激活) 中 Tag 过滤与 RAG 检索方案的全面对比。

### 7.3 答疑：RAG技术栈组合方案

---

**🙋 问题原文**

> 同学问："RAG 可以这样组合吗？Embedding 用 OpenAI，数据库用 PostgreSQL 和 pgvector？"

**📖 背景**

在理解了向量库的工作原理之后，同学们开始关心实际技术选型。同学提出的这个组合将三个不同来源的技术组件拼在一起，需要确认这种混搭方案是否可行。

**💬 老师回答**

**完全可以，这是一个非常标准、广泛使用的 RAG 技术栈组合。**

RAG（检索增强生成）的核心，本质上就是解决两个独立的问题：

1. **Embedding（向量化）**：把文本转成向量——这个任务由 Embedding 模型完成
2. **向量存储与检索**：把向量存起来，查询时做距离计算——这个任务由向量数据库完成

这两个问题是**解耦**的，各自选型互不影响：

```
┌──────────────────────────────────────────────────────────────────┐
│                      标准 RAG 技术栈                               │
│                                                                  │
│  ┌──────────────────┐   ┌──────────────────┐   ┌──────────────┐ │
│  │   Embedding      │   │   向量数据库       │   │     LLM      │ │
│  │   模型            │──▶│   (存储 + 检索)    │──▶│   推理生成     │ │
│  │                  │   │                  │   │              │ │
│  │  可选方案：       │   │  可选方案：       │   │  可选方案：   │ │
│  │  · OpenAI        │   │  · PostgreSQL    │   │  · GPT-4     │ │
│  │    text-embedding│   │    + pgvector    │   │  · Claude    │ │
│  │  · 智谱 Embedding│   │  · Milvus        │   │  · DeepSeek  │ │
│  │  · MiniMax       │   │  · Pinecone      │   │  · 通义千问   │ │
│  │  · BGE (开源)    │   │  · Weaviate      │   │  · 等等       │ │
│  │  · 等等           │   │  · Chroma        │   │              │ │
│  └──────────────────┘   │  · Qdrant        │   └──────────────┘ │
│                          └──────────────────┘                    │
│                                                                  │
│  ⚠️ 三大组件各自独立选型，可自由混搭                                 │
│  Embedding 用 A 家、向量库用 B 家、LLM 用 C 家 —— 完全没问题        │
└──────────────────────────────────────────────────────────────────┘
```

PostgreSQL + pgvector 是一个特别受欢迎的组合，因为：
- PostgreSQL 是成熟的关系型数据库，运维生态完善
- pgvector 作为扩展插件，直接在 SQL 层面提供向量存储和检索能力
- 不需要引入新的数据库基础设施，降低运维复杂度

> 💡 **进阶优化：Rerank 重排序**
>
> 老师还特别提到了一个进阶技术——**Rerank（重排序）**。向量检索返回的结果（比如 Top-10）是基于数学距离排序的，但这个排序在精确度上还有优化空间。Rerank 的做法是：先用向量检索从海量数据中做**粗排**（召回 Top-K，比如 35 个候选），然后换一个专门的 Rerank 模型对这些候选结果做**精排**（二次排序），最终只取精排后的 Top-N。这个"粗排 + 精排"的两阶段策略，能显著提升最终结果的精度。
>
> ```
> 海量文档 → [向量检索粗排] → Top-35 → [Rerank模型精排] → Top-5 → 注入LLM
> ```

**🤔 延伸思考**

RAG 技术栈这种"组件解耦、自由组合"的架构设计，赋予了系统极大的灵活性。你可以随时更换 Embedding 模型（比如从 OpenAI 换到开源的 BGE）而不影响向量库和 LLM 的配置。这种解耦思维是 AI 工程架构设计的核心原则之一。

### 7.4 答疑：工具封装的判断标准

---

**🙋 问题原文**

> 同学问："什么样模块的功能是值得封装成工具的？它的判断标准是什么？"

**📖 背景**

这是一个非常务实、直击要害的问题。在 Agent 开发中，不是所有功能都适合封装为工具（Function/Tool）。如果封装标准不清晰，要么功能过度封装导致工具列表臃肿混乱，要么遗漏了真正需要的工具导致 Agent 能力不足。同学需要一个清晰、可操作的判断框架。

**💬 老师回答**

核心判断标准可以浓缩为一句话：

> **模型能否通过自身的知识储备和推理能力自行解决这个任务？如果不能，就需要封装为工具。**

这个判断标准的底层逻辑是：**工具是模型自身能力的「外部扩展」**。

| 任务类型 | 模型自己就能搞定 | 模型搞不定，需要封装工具 |
|----------|-----------------|------------------------|
| **通用知识** | "请解释光合作用的原理"（静态训练知识） | "今天英伟达股价是多少？"（实时数据，训练截止后发生） |
| **数据访问** | "写一个冒泡排序的实现"（公开算法知识） | "查询我公司内部订单系统的数据"（私有数据，模型无权访问） |
| **外部交互** | "写一封辞职信的邮件模板"（纯文本知识） | "帮我给老板发一封邮件"（需要调用真实邮件 API） |
| **精确计算** | "估算 12345 × 6789 大约是多少"（推理估算） | "计算 123456789 × 987654321 的精确结果"（需要计算器） |
| **系统操作** | "建议一个文件管理的方案"（建议类回答） | "读取 /data/config.json 文件的内容"（实际 I/O 操作） |
| **事实核查** | "介绍一下量子计算的基本概念"（概念讲解） | "验证这个论文引用是否真实存在"（需要访问外部数据库） |

进一步提炼，工具封装的核心判断可以走以下决策路径：

```
你的功能模块需要封装为工具吗？
    │
    ├── 模型仅凭训练数据中的知识就能高质量回答？
    │   ├── 是 → ❌ 不需要封装为工具（模型自己能搞定）
    │   └── 否 → 继续往下判断
    │
    ├── 该功能需要访问实时/外部/私密数据？
    │   ├── 是 → ✅ 必须封装为工具（模型不知道这些数据）
    │   └── 否 → 继续往下判断
    │
    ├── 该功能需要执行对外部世界产生真实影响的操作？
    │   │ （如：发邮件、写数据库、调API、操作文件系统）
    │   ├── 是 → ✅ 必须封装为工具（模型没有"手"去执行）
    │   └── 否 → 继续往下判断
    │
    └── 该功能需要远超模型推理能力范围的确定性精确计算？
        ├── 是 → ✅ 封装为工具（模型擅长推理，不擅长精确运算）
        └── 否 → ❌ 不需要封装，让模型自己处理
```

> 💡 **核心洞见**：工具的本质，是弥合「模型能力边界」与「任务需求」之间的鸿沟。模型的知识有截止日期（不知道最新股价）、模型没有访问权限（查不了你的私密数据库）、模型没有执行能力（发不了真实的邮件）——每当你遇到模型"我知道怎么做但我做不到"的情况，就是在说"这里需要一个工具"。

> 🔗 **关联知识**：这个判断标准与第二章中「什么样的东西应该是工具」的根本问题一脉相承。第二章从"工具是模型的行为载体"角度切入，本章进一步深化为「能力边界判断」——凡是模型能力边界之外的事情，都需要工具来弥合。

### 7.5 答疑：工具 vs Skills 的区分（概念引子）

---

**🙋 问题原文**

> 同学问："功能模块是封装成工具还是封装成 Skills？依据是什么？"

**📖 背景**

在讨论了"什么值得封装成工具"之后，同学进一步追问：Skills（技能）和 Tool（工具）之间有什么区别？什么时候该用 Skills？这是一个更高阶的问题，触及了 Agent 能力封装的层级差异。

**💬 老师回答**

**Skills 比工具复杂得多，两者根本不在同一个抽象层级上。**

老师没有直接给出"标准答案"，而是采用了启发式教学——引导同学去观察 Skills 的形态特征，自己从中归纳出差异：

**Tool（工具/Function）的典型形态：**

```
┌─────────────────────────────────────────┐
│               Tool (工具)                │
│                                         │
│  · name:        "get_weather"           │
│  · description: "查询指定城市的天气"       │
│  · parameters:  { city: string }        │
│                                         │
│  执行方式：一次函数调用，输入 → 返回结果    │
│  定位：     模型的外部「行为单元」          │
│  粒度：     单一职责、单一操作             │
│  复杂度：    低                          │
└─────────────────────────────────────────┘
```

**Skills（技能）的典型形态（预览）：**

```
┌─────────────────────────────────────────────────┐
│                Skill (技能)                       │
│                                                  │
│  · name:          "code-review"                  │
│  · description:   "代码审查与重构建议"              │
│                                                  │
│  包含以下组成部分：                                │
│  ┌──────────────────────────────────────────┐   │
│  │  references/    参考文档、规范、最佳实践      │   │
│  │  scripts/       可执行的自动化脚本            │   │
│  │  prompts/       结构化的提示词模板            │   │
│  │  dependencies   依赖声明与版本管理            │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  执行方式：整个资产包被加载，可能内含多个工具调用      │
│  定位：     可复用的「能力资产包」                   │
│  粒度：     复合功能，可能包含多个步骤和多个工具       │
│  复杂度：    高 —— 远高于单一 Tool                  │
└─────────────────────────────────────────────────┘
```

两者的层级关系：

```
              ┌─────────────────┐
              │     Skill        │  ← 能力资产层（最复杂）
              │  (能力资产包)      │
              └────────┬────────┘
                       │ 可包含多个
                       ▼
              ┌─────────────────┐
              │      Tool        │  ← 行为单元层
              │   (函数/工具)     │
              └────────┬────────┘
                       │ 可调用
                       ▼
              ┌─────────────────┐
              │      API         │  ← 底层接口层
              │   (外部服务)      │
              └─────────────────┘
```

> 💡 **老师的学习建议**：Skills 规范是由 Anthropic（Claude 的开发公司）发起的。老师建议感兴趣的同学直接去 Anthropic 官网查看 Skills 的官方规范和样例——通过观察官方提供的完整样例，来建立对 Skills 形态的直观认知，然后再抽象理解它与 Tool 的区别。这种"先看具体样例，再归纳抽象概念"的学习方法，本身也是值得借鉴的。

> ⚠️ **注意事项**：Skills 的深入讲解不在当前课程范围内。课程的安排是：下一节课会先讲 MCP（Model Context Protocol，模型上下文协议），它解决的是模型如何通过标准协议连接外部资源的问题——这是"连接层"的规范。而 Skills 涉及的是能力封装、资产管理、复用模式——这是更高层的抽象。只有在充分理解 Tool 和 MCP 的基础上，再引入 Skills 才不会产生混淆。Skills 会在后续课程中专门展开。

这个问题的本质是"海量功能模块的精确定位"——而这恰恰是本章核心主题（向量检索）的另一个应用场景。相比硬编码的字典映射或 Tag 体系，通过语义向量检索来匹配功能模块，是更灵活、更可扩展的方案。

### 8.1 答疑：Copilot 的思考过程是不是 Plan 模块？（CoT 深度解析）

---

**🙋 问题**：平时看到的 Copilot 思考过程，是不是就类似于课程里讲的 Plan 模块？

> 📖 **背景**：学员在学完 Plan Handler 与 Execution Handler 的设计后，观察到 Copilot 在回答前也会展示一段"思考过程"，自然将两者联系起来。这是一个非常常见且重要的误解。

**💬 老师回答**：

**结论先行：不是。这两者位于完全不同的抽象层级。**

Copilot 看到的"思考过程"，本质上是 **CoT（Chain of Thought，思维链）**，它发生在**单次推理请求内部**。而课程的 Plan 模块，是**多次请求之间的编排决策**——Plan 模块决定"这次调哪个工具、下次调哪个工具"，每一次工具调用都是一次独立的模型请求。

#### 🧠 直观理解

- **CoT（思维链）**：你在草稿纸上推导一道数学题的步骤。笔在纸上写、脑子在想——但这是一次完整的思考过程，你没有中途停下来去翻书或问别人。
- **Plan Handler**：你在做一个复杂项目时列出的任务清单，然后逐个执行——"先查天气，如果下雨就查交通，最后决定穿什么"。每完成一个任务，你停下来看看结果，再决定下一步做什么。

#### 📖 详细解释

**CoT 的原意**：CoT 来自 2023 年的一篇经典论文，研究的核心问题是：**为什么告诉模型 "step by step" 能显著提升推理准确率？**

要理解这个问题，必须先理解 **Transformers 架构的核心机制**：next token prediction。

```
用户输入：123+456

【不加 step by step 的情况】
  模型收到上下文："123+456"
  模型的 next token 是什么？
  → 模型没有做任何数学计算。在它眼里，"123" 是三个 token，"456" 是三个 token
  → 它只是从历史语料中理解：这是一个三位数开头的算式，结果大概在五百多
  → 它根据历史经验直接猜测：可能是 580、582 之类的数字
  → 然后就输出了

【加 step by step 的情况】
  指令告诉模型："请 step by step 地解决"
  模型理解了这个指令 → "我要展示推理过程"
  于是模型输出：
    Step one: 1+4=5   （百位相加）
    Step two: 2+5=7   （十位相加）
    Step three: 3+6=9 （个位相加）
    所以 123+456=579
  → 每一步都是基于"上文"在推"next token"
```

#### 🔗 推导链：CoT 为什么有效？

```
起点: 用户输入 "123+456"，无 step by step 指令
  │
  ├── 第1步: 模型推理 next token
  │   为什么？模型唯一会做的事就是根据所有上文推下一个 token
  │   上文只有 "123+456" → 没有引导它展开推理过程的信号
  │
  ├── 第2步: 模型直接输出一个近似答案（如 582）
  │   为什么？它在训练语料中见过类似"三位数+三位数"的模式
  │   它基于统计经验给出"看起来合理"的数，而非真正计算
  │
  └── 问题: 不准确的答案是因为模型"算错了"吗？
      不是。是模型根本没在计算——它在"回忆"见过的类似模式。

────────────────────────────────────────────

起点: 用户输入 "123+456"，加了 step by step 指令
  │
  ├── 第1步: 模型理解 "step by step" → "我要展示推理过程"
  │   为什么？因为指令在上文中，影响 next token 生成方向
  │
  ├── 第2步: 模型输出 "Step one: 1+4=5"
  │   为什么？因为它被引导"先展示步骤"
  │   P1（第一步输出）的输入是：原始 prompt + step by step 指令
  │
  ├── 第3步: P2（第二步输出）的输入是：原始 prompt + step by step + "Step one: 1+4=5"
  │   为什么？Transformers 架构：所有上文都参与 next token 预测
  │   前面写过的"1+4=5"变成了新的上文，帮助模型约束后续推理
  │
  ├── 第4步: 1+4=5 这个结果是怎么来的？
  │   关键：不是模型在"计算"！
  │   而是：10 以内的加减法在海量训练语料中有着极高的确定性
  │   模型在语料中见过无数次 "1+4=5" 这个模式
  │   所以当上文是 "1+4" 时，"5" 这个 token 的收敛概率极高
  │   → 它是统计确定性，不是算术能力
  │
  └── 结论: CoT 的本质 = 通过过程引导，用"每一步的高确定性收敛"来强化"最终答案的准确性"
```

> 💡 **核心洞见**：**1+4=5 不是模型在计算，而是语料中确定性收敛。** 10 以内的加减法在海量训练数据中重复了无数次，模型在这些 token 上有着近乎确定性的预测。CoT 把一个大问题拆成小步骤——每一步小步骤都在语料中有高确定性的答案——然后把小步骤串联起来，让最终结果也获得高确定性。CoT 不是在"教会模型算数"，而是在"引导模型走到确定性最高的那条路径上"。

> ⚠️ **注意事项**：现在很多新模型已经**内化了 CoT**。早期模型必须靠 prompt 中加 "step by step" 才能触发推理过程；后来的模型在训练阶段就大量使用了"Q→推理过程→A"格式的训练数据，于是模型天然就会"先推理再回答"。这就是为什么新模型的回答"废话变多了"——它被训练成了必须先展开推理链。Reason 模型（如 o1、DeepSeek-R1）更进一步，特别强化了 thinking 过程，其目的与 CoT 完全相同：通过扩展推理链来提升最终输出的准确性。

#### 🔀 CoT vs Plan Handler 对比

| 维度 | CoT（思维链） | Plan Handler（编排决策） |
|---|---|---|
| **发生层级** | 单次推理请求内部 | 多次请求之间 |
| **核心机制** | next token 逐步预测，上文影响下文 | 每次请求后分析结果，决定下一步动作 |
| **类比** | 你在草稿纸上推导一道题 | 你做项目时列任务清单、逐个执行 |
| **能否形成循环** | 不能（是线性推导） | 可以（ReAct Loop 就是循环） |
| **控制方式** | prompt 中的 "step by step" 指令 | 代码中的 Plan Handler 接口 |
| **典型产物** | 一段推理文字（thinking process） | 一个 action_calls[] 列表 |

#### 📊 执行流程对比（ASCII 图）

```
【CoT：单次推理内的思维链】
┌─────────────────────────────────────────────────────┐
│                  单次模型推理请求                       │
│                                                     │
│  用户输入 ──▶ P1（第一步推理）──▶ P2（第二步推理）       │
│                  │                   │              │
│                  ▼                   ▼              │
│            影响 P2 输出          影响最终答案           │
│                                                     │
│  所有步骤在同一个 response 里完成                      │
└─────────────────────────────────────────────────────┘

【Plan Handler：多次请求间的编排】
  请求1: 用户输入 ──▶ 模型决定调工具A ──▶ 执行工具A
                                          │
  请求2: 回填结果 ──▶ 模型决定调工具B ──▶ 执行工具B
                                          │
  请求3: 回填结果 ──▶ 模型判断完成 ──▶ 输出最终答案

  每次"决定调什么工具"都是一次独立请求
```

### 8.2 答疑：编排过程和结果做成训练集能否减少编排？（DAG 与编排自动化）

---

**🙋 问题**：把编排的过程和结果也做成训练集，是不是就能减少手工编写编排逻辑的工作量？

> 📖 **背景**：学员在理解手工编写 TriggerFlow 编排的复杂性后，自然地想到"能不能让模型自己学会编排"。这是从"手工编排"到"编排自动化"的思维跃迁。

**💬 老师回答**：

这个问题的答案，其实在第 8 课中已经给出了——**Todo List 模式**。

#### 🔗 回扣第 8 课：Todo List 模式的关键

第 8 课讲 Todo List 模式时，最重要的点**不是**模型生成 Todo List（那只是输出格式），而是**那个执行器（Executor）**。执行器消费 Todo List 这样的编排表达，自动完成对任务的对应处理。

但这里有一个**本质区别**需要理解：

| 编排方式 | 是否可成环 | 复杂度 | 典型场景 |
|---|---|---|---|
| **TriggerFlow 信号级编排** | **可以成环**（ReAct Loop 就是环） | 更高，程序级控制 | 需要多轮推理、动态决策的复杂任务 |
| **Todo List 类编排** | **不能成环** | 任务拆解 + 依赖声明 | 可以预先拆分为子任务的工作流 |

#### 🧠 DAG（有向无环图）详解

Todo List 类编排的核心数据结构叫做 **DAG（Directed Acyclic Graph，有向无环图）**。

**直观理解**：想象一个项目的任务依赖关系。写代码依赖写完设计文档，测试依赖写完代码。这是一个有方向的依赖链（"有向"），并且**不能出现循环依赖**——你不可能让 A 依赖 B、B 依赖 C、C 又依赖 A，那样整个系统就永远解不开了（"无环"）。

```
【合法的 DAG（不包含环）】
  [查天气] ──▶ [判断是否下雨]
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
  [查交通路况]          [直接出门]
        │
        ▼
  [综合建议]
  
  → 每个任务有明确的前置依赖，且不存在循环

【非法的依赖（成环 = 不是 DAG）】
  [查天气] ──▶ [判断是否下雨] ──▶ [查天气]
       ▲                              │
       └──────────────────────────────┘
  
  → A 依赖 B，B 依赖 A：这个系统永远无法执行
```

**DAG 的两个关键属性**：

1. **有向（Directed）**：边有方向。A → B 表示 B 依赖 A，A 必须先于 B 完成。
2. **无环（Acyclic）**：不存在从某个节点出发沿箭头走、最终回到该节点的路径。

> 💡 **核心洞见**：在执行这类自动化编排时，任务可以分解、依赖关系可以讲清楚，但**绝对不能成环**。一旦成环（A→B→C→A），执行器就会陷入死循环。这正是与 TriggerFlow 信号级编排的本质区别——TriggerFlow 允许在 ReAct Loop 中反复回到 "Reason" 节点，那是在程序控制下的有意循环。

**那么，训练集能否减少编排？**

思路是成立的：让人工把编排逻辑表达为 Todo List / DAG 格式，放入训练集，模型学习后面对类似任务时能自主生成合理的 DAG。但这和 TriggerFlow 的手工信号级编排是两条线——Todo List 模式擅长处理"可预拆解的任务"，信号级编排擅长处理"需要动态决策的不确定性任务"。**两者并不互斥，可以协同工作。**

### 8.3 答疑：模型生成时如何确保格式输出？

---

**🙋 问题**：模型本身声称有能力确保格式输出，为什么我们还要在工程层做结构化控制？

**💬 老师回答**：

模型提供商声称有能力、和工程层需要额外控制，**这两者根本不冲突**。

#### 🔀 类比：前端校验 + 后端校验

在 Web 开发中，前端对用户输入做校验（防止明显错误），后端同样要对所有输入做校验（安全性兜底）。没有人会觉得这是冗余的——因为**两端各司其职，负责不同的职责边界**：

| | 模型层（前端类比） | 工程层（后端类比） |
|---|---|---|
| **职责** | 在自身能力范围内尽量输出正确格式 | 确保"任意模型切换"时不冲击业务逻辑 |
| **失败代价** | 偶然的输出格式错误 | 下游业务逻辑崩溃 |
| **可控性** | 依赖模型提供商 | 开发者完全自主 |

> 💡 **核心洞见**：单模型声称有能力确保不代表可以放弃工程控制。我们的问题不是"这个模型能不能确保"，而是"**全天下所有模型在任意切换的情况下**，如何不影响业务逻辑"。模型会升级、会替换、可能出错——工程层的结构化控制是最后一道防线。

### 8.4 答疑：AI 编程助手开发中如何干净利落地进行架构重构？

---

**🙋 问题**：在 AI 编程助手的独立开发环境中，要涉及到对架构甚至底层设计的调整或重新设计，如何干净利落地完成？

**💬 老师回答**：

以下是架构重构的实践方法论：

1. **先让模型分析当前架构**。用 AI 助手输入现有代码，让模型理解当前结构、识别耦合点。
2. **定位需要改造的具体位置**。不是"感觉哪里不好就改"，而是有明确的改造理由和范围。
3. **出 spec 文档**。把改造方案写成可评审的 spec，包括：改造范围、新结构设计、迁移步骤、风险点。
4. **用 example 验证**。构造最小化的验证场景，确保新结构能跑通。
5. **出报告**。总结改造前后对比、收益量化。

> ⚠️ **注意事项**：避免过度设计。**系统正常在跑、用户没有诉求的时候，不应该做调整。** 架构改造的前提是"确实遇到了现有架构解决不了的问题"。只有让一个人用你的系统去做具体的场景落地，才能知道架构用起来到底怎么样。凭空设计的"完美架构"往往是最脆弱的。

│  意图分类 / 路由规则 / 向量检索 / 配置驱动                   │
│                                                          │
│  特点：灵活 —— 随场景调整，不同场景激活不同工具子集           │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│                      执行层（Handler）                      │
│                                                          │
│  Plan Handler（可替换）    Execution Handler（可替换）       │
│  决定"调哪些、什么顺序"     决定"怎么调起来"                 │
│                                                          │
│  特点：可扩展 —— 新工具类型只需增加 Executor 实现             │
│  LocalFunction │ HTTP │ MCPClient │ Sandbox │ ...         │
└──────────────────────────────────────────────────────────┘
```

#### 📊 四个模块回顾表

| 模块 | 解决的问题 | 核心设计 |
|---|---|---|
| **注册中心** | 工具散落各处，没有一处能回答"系统有哪些工具" | 集中声明，全局可见。工具的能力声明（desc + schema + func + side_effect_level）统一管理 |
| **动态注入** | 全量暴露给模型，token 成本高、模型选错概率大 | 注册一次，按需激活。意图分类/路由规则从注册中心筛选工具子集注入当前请求 |
| **Plan Handler** | 规划逻辑写死在代码里，无法切换决策模式 | 接口固定，实现可替换。模型自主规划 / 固定规则 / A/B 测试策略——随便换 |
| **Execution Handler** | 调用方式耦合，换一种执行方式要改大量代码 | 执行层统一接口。真实执行 / mock 执行 / 审计日志包裹 / 远程路由——随便换 |

#### 💡 核心价值

> **Plan 和 Execution 拆开后：Plan 不关心"执行细节"（是用本地函数、HTTP API、MCP 还是沙盒），Execution 不关心"为什么调这个"。执行层归一化后，新工具类型只需增加一个 Executor 实现——不改注册中心、不改 Plan Handler、不改循环骨架。**

#### 💡 一句话总结

> 工具管理的本质不是"把工具写对"，而是建立**契约层（稳定）→ 激活层（灵活）→ 执行层（可扩展）**的三层体系，让系统在工具数量和类型持续增长的条件下，仍能保持架构的清晰与可控。

### 8.6 课后思考

---

> 🎯 **思考题：Skills vs Tools vs MCP —— 它们的定位分别是什么？**

在后续课程中，我们会遇到三个容易混淆的概念：

| 概念 | 所处层级 | 核心特征 |
|---|---|---|
| **Tools（工具）** | 执行层 | 原子操作，如 `query_leave`、`send_notify`。注册在 ToolRegistry 中，由 Plan Handler 调度、Execution Handler 执行 |
| **MCP（模型上下文协议）** | 协议层 | 标准化的工具发现与调用协议。MCP 不是工具本身，而是"如何发现外部工具、如何调用外部工具"的一套规范 |
| **Skills（技能包）** | **资产层** | Skills 经常被误认为就是"复杂一点的工具"，但它和工具**根本不在同一层**。Skills 是资产包——包含 references、scripts、依赖和版本，其消费方式也不同于工具调用回路 |

**思考方向**：假如你要设计一个"代码审查 Skill"，它包含审查规则、示例代码、和审查脚本。这个 Skill 的消费方式为什么不能简单地用一个 `call_tool("code_review")` 来完成？Skill 的"资产属性"带来了哪些设计上的不同？

> 💡 **提示**：Skills 和工具不在同一层。工具是"调用一次，得到一个结果"；Skills 是"加载一个资产包，里面的内容被整合进多个环节"。这个区分会在后续 Skills 章节单独展开。
