## 一、开场与 Loop Engineering 概念引入

### 1.1 热词现象与正确认知

#### 🧠 直观理解

AI 领域的新词层出不穷，从 "Prompt Engineering" 到 "Context Engineering" 到 "Harness Engineering" 再到 "Loop Engineering"——这些词的变化速度已经远超普通从业者的消化速度。面对这种现象，有两种常见的态度：一是追着热词跑，了解每个词的定义就以为掌握了全部；二是不屑一顾，认为这不过是硅谷的营销话术。

老师的态度是第三种：**不要只看表面，要理解热词背后的成因——它为什么能被广泛接受和流传，底层支撑到底是什么。**

#### 📖 详细解释

老师从时间线切入：Harness Engineering 和 Loop Engineering 这两个词出现的时间非常近。Loop Engineering 大约在 2026 年 6 月才被提出，而 Harness Engineering 也不过早几个月。甚至有一些知识分享博主认为，硅谷这一轮"造词运动"有点过火——Harness 也好、Loop 也好，"讲的都是同一个事情：怎么控制好大模型"。

> 💡 **核心洞见**：老师说，"热词的出现当然有一部分营销的可能性，但它实实在在也需要有底层的支撑。" 正确的态度不是嘲笑硅谷造词——因为嘲笑完之后你什么都没学到。正确态度是追问：**这个词能火，它在描述一个什么新的工程实践？这个实践的底层逻辑是什么？** 这才是这堂课的核心方法论。

> ⚠️ **注意事项**：有同学在评论区问"AutoGPT 算不算 Loop Engineering？"老师没有当场回答，而是将其作为引子——"这是一个很好的思路"。这个问题会在后续模块（M06：React Loop 详解）中得到完整回答。

### 1.2 Loop Engineering 的正式定义

#### 🧠 直观理解

Loop Engineering 描述的是这样一种智能体行为模式：**给智能体一个目标，让它自己不断往前推进，直到——做完了（完成）、做不动了（受阻）、或者做得太过分了（越界）。**

#### 📖 详细解释

目前硅谷/北美比较广泛接受的 Loop Engineering 定义是：

> **让智能体持续追逐目标，直到完成、受阻或越界。**

注意，这个定义最重要的信息隐藏在"直到"后面的三个词里——**Loop Engineering 的终点不一定是"完成"**。传统软件工程中，一个任务的终点就是完成（要么成功，要么异常退出）。但 Loop Engineering 引入了两个新的终止条件：

| 终止条件 | 含义 | 示例 |
|---|---|---|
| **完成** | 目标已达成，任务结束 | 成功生成了一份完整的招聘市场调研报告 |
| **受阻** | 遇到不可突破的障碍，无法继续推进 | 反复尝试调用某个 API 但始终返回 403，确认无权限后停止 |
| **越界** | 超出权限范围或预设边界 | 智能体试图访问系统根目录来完成任务，被沙箱拦截 |

其中"完成"和"受阻"比较好理解。关键在于第三个——**越界**，这是 Loop Engineering 与简单自动化最本质的区别之一。

### 1.3 "越界"（Overstepping）深度解析

#### 🧠 直观理解

"越界"就像给一个助理布置任务："去帮我整理桌面上的文件"。助理完成了大部分整理，但看到你的抽屉有点乱，于是打开抽屉继续整理——此时他已经超出了你授权的范围。在智能体系统中，"越界"是一种**需要被识别和拦截的终止信号**，而不是一个 bug。

#### 📖 详细解释

老师用两个具体场景说明了"越界"在工程实践中的表现形式：

**场景一：沙盒（Sandbox）机制**

```
┌─────────────────────────────────────────┐
│              沙盒环境                     │
│  ┌───────────────────────────────────┐  │
│  │   智能体                           │  │
│  │   · 可以读/写沙盒内的文件            │  │
│  │   · 可以调用允许范围内的 API         │  │
│  │   · 可以创建子进程                  │  │
│  └───────────────────────────────────┘  │
│                                         │
│  ════════════ 权限边界 ═══════════════  │
│                                         │
│   ❌ 系统文件目录                        │
│   ❌ 用户隐私数据                        │
│   ❌ 网络外部访问                        │
└─────────────────────────────────────────┘
```

在执行任务时，我们通常不会让智能体直接在宿主系统环境中运行，而是放在一个沙盒里。沙盒本身就是一种权限约束——智能体可以做的事情被严格限定。

**场景二：根目录权限限制（Codex / Claude Code 实例）**

老师提到在使用 Codex 或 Claude Code 时的一个实际现象：智能体有一个根目录（启动时所在目录）。如果智能体在执行任务时说"我需要访问你的系统目录"——这就是越界。智能体需要的文件已经超出了当前赋予它的文件目录范围。

```
智能体当前可访问范围: /project/workspace/
智能体尝试访问: /Users/admin/.ssh/
                    ↑
                ← 这就是越界
```

> 💡 **核心洞见**：越界不是智能体的 bug，而是 Loop Engineering 中**有意识设计的终止条件**。它反映了一个设计哲学：智能体应该有边界意识。不是"能做就做"，而是"在授权范围内能做才做"。这种设计确保智能体在自主运行时不会失控。

---

#### 🔀 三种终止条件的对比

| 维度 | 完成 | 受阻 | 越界 |
|---|---|---|---|
| 智能体视角 | "我做完了" | "我做不了" | "我不能做" |
| 触发场景 | 目标达成 | 遇到不可突破的障碍 | 超出授权范围 |
| Loop 行为 | 停止，交付结果 | 停止，报告受阻原因 | 停止，触发权限警告 |
| 是否可恢复 | — | 可能需要人工介入 | 调整权限后可重试 |
| 设计意图 | 正常完成 | 容错机制 | 安全机制 |

### 1.4 Loop 的核心特征：持续推进模型

#### 🧠 直观理解

Loop 的核心动作不是"原地循环"，而是**不断往前推进**。就像一个玩闯关游戏的人，从 A 点到 B 点，到 B 点之后观察情况，再决定下一个目标点是哪里——每一步都在前进，而不是在原地打转。

#### 📖 详细解释

老师用了一个形象的描述来说明 Loop 的推进过程：

```
A —→ B —→ B₁ —→ B₂ —→ B₃ —→ ... —→ 目标完成点

每一段推进后：
  1. 观察当前状态（做了什么、结果如何）
  2. 判断是否需要调整方向（重规划）
  3. 确定下一个推进点（继续 or 停止）
```

这个"推进模型"有以下几个关键特征：

1. **持续性**：不是一次请求、一次回答就结束，而是一个持续的过程。
2. **目标导向**：始终朝着最初设定的目标前进（虽然路径可能调整）。
3. **自我感知**：每一步推进后都会评估——我离目标还有多远？我现在做的方向对吗？
4. **有终止条件**：不是无限循环，到达完成/受阻/越界三者之一就停下。

#### 🔗 推导链：为什么叫 Loop Engineering？

```
起点: 智能体需要完成一个长期的、复杂的任务
  │
  ├── 第1步: 简单的一次性请求无法完成
  │   为什么？因为任务太大，一次模型调用处理不了——
  │   信息不够（需要多次查）、步骤太多（需要分步做）、中间需要判断。
  │
  ├── 第2步: 需要一种"推进-观察-再推进"的循环机制
  │   为什么叫"Loop"？因为智能体不是在走直线，而是在走回环——
  │   做完一段，回头看看，调整方向，再做下一段。执行结果会回传到决策节点。
  │
  ├── 第3步: "Engineering"体现在哪里？
  │   为什么叫"Engineering"？因为这种循环不是靠"给模型写一个好 prompt"就能实现的——
  │   需要在外层构建完整的工程系统来管理循环的执行、状态的记录、权限的控制、错误的处理。
  │
  └── 结论: Loop Engineering = 让智能体以目标为驱动力，在有工程支撑的循环中持续向前推进。
```

```python
"""

# ------------------------------------------------------------
# Step 3: 定义结构化输出约束（Structured Output）
# ------------------------------------------------------------
output_schema = OutputSchema(
    format="markdown",                              # 输出格式要求
    sections=[                                      # 文档结构要求
        "概述",
        "核心概念",
        "对比分析",
        "应用场景",
        "总结",
    ],
)

# ------------------------------------------------------------
# Step 4: 流式执行任务
# ------------------------------------------------------------
for event in agent.stream(
    goal=goal,
    output_schema=output_schema,
):
    print(event.content)                            # 逐块打印中间和最终输出
```

**代码要素与 Loop 概念的对应关系**：

| 行号 | 代码要素 | 对应 Loop 概念 | 说明 |
|---|---|---|---|
| 7-10 | `LoopAgent(model=..., skills=[...])` | Agent 初始化 | 创建 Loop 执行体，配置驱动模型和可用技能 |
| 9 | `skills=[Skill("deflow")]` | 技能（Skills） | Agent 可调用的能力单元。deflow 提供研究式指导 |
| 13-17 | `goal = "..."` | 目标（Goal） | Loop 持续追逐的方向，不是一次性指令 |
| 20-28 | `output_schema = OutputSchema(...)` | 结构化输出约束 | 对最终产物的格式和内容框架提出工程化要求 |
| 33-37 | `agent.stream(goal=..., output_schema=...)` | 流式执行 | 逐步推进任务，实时输出中间结果 |
| 37 | `for event in ...` | 循环迭代 | Loop 的核心：不断检查状态、推进任务、处理异常 |

> 💡 **核心洞见**：从代码中可以看出，Loop Engineering 的编程模型与传统函数调用有着本质区别。你不是在调用一个「输入→输出」的函数，而是在启动一个「持续追逐目标」的自治过程。你给定的是目标（What）+ 约束（How），而非具体步骤（Steps）。

---

### 2.3 模式一：流式逐项推进

第一种运行模式是**流式逐项推进**。Agent 不预先做完整的全局规划，而是边执行边观察，根据每一步的结果动态调整后续行为。

#### 🔄 执行流程

```
Agent 启动
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 1: 规划（Planning）                  │
│   · 分析目标：研究 + 对比 + 生成文档       │
│   · 确定当前行为：先搜索相关资料            │
│   · 提出执行建议                          │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ Step 2: 执行（Execution）                  │
│   · 搜索 Loop Engineering 资料            │
│   · 搜索 FTD 资料                         │
│   · 整理对比分析                           │
│   · 尝试写入文件                           │  ◀── 可能出错！
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│ Step 3: 验证（Verification）              │
│   · 检查交付状态 ✅ 或 ❌                  │
│   · 判断：目标是否已达成？                  │
└──────────────┬──────────────────────────┘
               │
         ┌─────┴─────┐
         ▼           ▼
    ┌────────┐  ┌──────────────┐
    │ 已完成  │  │   未完成      │
    │   ✅   │  │              │
    └───┬────┘  │ 分析失败原因   │
        │       │ 调整执行策略   │
        ▼       │ 重新发起执行 ──┼──▶ 回到 Step 2
   任务结束     └──────────────┘
```

#### 🐛 排错实录：文件写入失败 → 自动检测并修复

在演示过程中，Agent 遇到了一个典型错误场景，这恰好完整展示了 Loop Engineering 「受阻但不停止」的核心特性。

**❌ 错误现象**

Agent 在尝试将研究结果写入本地文件时操作失败。

**🚨 系统行为时间线**

```
[T0] Agent 执行 write_file 操作，尝试保存研究结果
[T1] 操作报错：文件写入失败（权限不足 / Permission Denied）
[T2] Agent 进入验证阶段，检查交付状态
[T3] 验证结果：交付状态 = FAILED，目标未达成
[T4] Agent 分析失败原因：
      · 本地环境未授予文件写入权限
      · 当前路径可能需要调整
[T5] Agent 提出新的执行策略，调整写入方式或路径
[T6] Agent 重新执行写入操作...
[T7] 再次验证 → 如果仍失败 → 继续调整 → 直到成功或找到替代方案
```

**🔍 关键行为分析**

| 观察点 | 传统一次性请求 | Loop Engineering |
|---|---|---|
| **任务状态** | 报错后直接终止，返回 error | 报错后任务继续运行，不终止 |
| **失败检测** | 依赖异常抛出机制 | Agent 主动进入验证阶段，自行检查交付状态 |
| **恢复策略** | 无——需要人工介入 | Agent 自动分析原因，提出替代方案，重新尝试 |
| **循环次数** | 0（单次即终止） | 多次，直到目标达成或所有可行路径耗尽 |

**📝 经验总结**

1. **Loop 不以「操作成功」为判断标准，而以「目标达成」为标准**。单次文件写入成功不等于任务完成——Agent 会验证写入的内容是否完整、是否符合格式要求。
2. **验证环节是 Loop 的关键设计**。如果没有 Verify 步骤，Agent 将无法知道自己的执行是否真正达成了目标，Loop 也就不成其为 Loop。
3. **这个权限问题是真实演示中的意外**，老师提到"可能没给他开这个权限"，但这恰好成为理解 Loop 容错机制的最佳案例。

### 2.4 模式二：任务板（Task Board）驱动

老师"设置成另外一种模式"后，Agent 的行为发生了明显变化。这种模式下，Agent 先进行全局规划，生成一个任务清单（To-Do List），再按照清单逐项执行。

#### 📋 规划阶段：生成 Task Board

任务启动后，Agent 首先输出一个结构化的任务板，将大目标拆解为一系列可独立完成的子任务：

```
┌───────────────────────────────────────────────┐
│                 Task Board                      │
├───────────────────────────────────────────────┤
│  ☐ Task 1: 研究 Loop Engineering 定义与核心概念 │
│  ☐ Task 2: 研究 FTD 定义与核心概念              │
│  ☐ Task 3: 对比分析 Loop Engineering 和 FTD     │
│  ☐ Task 4: 整理典型应用场景与案例分析            │
│  ☐ Task 5: 生成结构化 Markdown 研究报告          │
│  ☐ Task 6: 验证文档完整性和格式合规性            │
└───────────────────────────────────────────────┘
```

> ⚠️ **已知问题**：此模式启动时会输出一个 warning，老师说明这是"一个已知的 bug，我们现在是在修复中"。这不影响对 Loop 执行逻辑的理解，可以暂时忽略。

#### 🔄 执行流程

与模式一的「边想边做」不同，模式二遵循「先规划后执行」的两阶段策略：

```
阶段一：任务拆解（Upfront Planning）
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. 分析目标的结构，识别可独立的子任务           │
│ 2. 确定子任务之间的依赖关系和执行顺序           │
│ 3. 生成 Task Board（事项板）                    │
│ 4. 将 Task Board 作为后续执行的「导航地图」      │
└──────────────┬───────────────────────────────┘
               │
               ▼
阶段二：逐项执行（Sequential Execution）
    │
    ├── Task 1: 研究 Loop Engineering ──────▶ ✅ 完成
    ├── Task 2: 研究 FTD ──────▶ ✅ 完成
    ├── Task 3: 对比分析 ──────▶ ✅ 完成
    ├── Task 4: 应用场景 ──────▶ ✅ 完成
    ├── Task 5: 生成文档 ──────▶ ✅ 完成
    └── Task 6: 验证完整性 ──────▶ ✅ 完成
               │
               ▼
       🎉 全部任务完成，交付最终文档
```

#### 🔗 与日常开发工具的类比

老师引导学员将这种体验与日常使用的 AI 工具进行关联：

| 工具 | 相似的行为模式 |
|---|---|
| **Codex** | 将复杂编程任务拆解为子任务序列，逐项实现，中间可调整 |
| **Claude Code** | 分析当前代码库状态 → 规划修改步骤 → 逐文件执行修改 |
| **GitHub Copilot Workspace** | 从 Issue 出发 → 拆解为 Spec → Plan → Task → 逐项落地 |

> 🔗 **关联知识**：任务板（Task Board）模式背后的拆解策略、任务间依赖管理、以及并行执行的可能性，将在 M10 中详细展开。

### 2.5 两种模式对比

| 维度 | 模式一：流式逐项推进 | 模式二：任务板驱动 |
|---|---|---|
| **规划时机** | 边执行边规划，逐步调整方向 | 执行前完成全局规划，生成完整任务清单 |
| **执行风格** | 流式输出，每步结果实时可见 | 按预设顺序逐项完成，有明确的进度感 |
| **灵活性** | 高——可根据执行中途的发现灵活调整 | 中——按既定计划推进，但支持中途修正 |
| **错误处理** | 单步出错即分析并尝试修复后继续 | 单任务失败时标记该任务，尝试绕过或重试 |
| **适用场景** | 探索性任务：目标明确但路径不确定 | 结构化任务：可以提前拆解为清晰的子任务 |
| **Loop 特性体现** | 执行→验证→调整→再执行的小循环 | 规划→拆解→逐项执行→总体验证的大循环 |

> ❓ **常见疑问**：是不是模式二一定比模式一好？
>
> **回答**：不是。老师演示这两种模式的目的，正是为了说明 Loop Engineering 提供了**灵活的执行策略选择**。如果任务探索性强、路径不确定，模式一的「边做边看」反而更高效；如果任务结构清晰、可分阶段执行，模式二的「先规划后执行」则更有序。

### 2.6 演示总结：对 Loop 的三个直观感受

#### 💡 感受一：Loop 不是「一次性请求」

传统 AI 编程体验是「提问 → 回答」的一次性交互。你问一个问题，模型给一个答案，交互结束。但 Loop Engineering 的体验完全不同——你设定一个目标，Agent 进入「持续追逐」状态，不断推进这个目标直到完成。

#### 💡 感受二：错误不是终点，而是信号的输入

本次演示最关键的观察发生在文件写入失败的场景。传统编程中，一次 IO 失败通常意味着流程中断、返回错误。但在 Loop 中：写入失败 → 验证发现目标未达成 → 分析原因 → 调整策略 → 重新执行。这体现了 Loop Engineering 与一次性请求的本质区别：**错误是流程的一部分，而非流程的终点**。

#### 💡 感受三：同一目标，多种执行路径

老师用同一个任务（研究概念并生成文档），演示了两种截然不同的执行路径——流式推进和任务板驱动。这说明 Loop 框架在目标确认的前提下，提供了灵活的执行策略选择。


## 三、单词请求 vs 持续目标追逐

### 3.1 一次行为（One-Shot Request）：单轮问答的极限

#### 🧠 直观理解

想象你去便利店买一瓶水：你走到收银台，说"给我一瓶水"，付钱，拿水，走人。这是一次完整的交互——一个输入，一个输出，结束了。这就是一次行为的模型请求。在 AI 编程的语境下，它对应的是我们最熟悉的模式：给出一个 prompt，模型返回一个结果，对话轮次为 1。

#### 📖 详细解释

一次行为（One-Shot Request）的本质是"单次模型请求"：给模型一组输入（指令 + 上下文），模型返回一组输出，这一轮交互就结束了。它隐含的假设是：**任务可以在一次往返中完成**。

一次行为的典型特征：

| 特征 | 说明 |
|---|---|
| 输入明确 | 用户描述清楚，不需要额外澄清 |
| 边界清晰 | 任务的起点和终点一目了然 |
| 无需中间决策 | 模型不需要在回答过程中停下来思考"我还要做什么" |
| 结果可直接消费 | 输出就是用户要的最终产物 |

查今天的天气、翻译一句话、问"React 是什么"——这些都是典型的一次行为。老师举了一个很具体的例子：**查今天天气**。你问一句"今天天气怎么样"，模型调用一次天气 API，返回结果，交互结束。

课堂上学员们也给出了自己的理解，包括：
- "一次行为是个 prompt"——抓住了问题的核心，一次输入。
- "单任务"——更抽象地指出了它的能力边界：只能处理单个、原子化的任务。

> ❓ **常见误解**：有人可能觉得"一次行为 = 只调用一次工具"。实际上，一次行为内部可能嵌套了多次搜索或多次内部推理，但对外部管理者（调用方）而言，它仍然是一次请求、一次返回。关键不在于内部做了什么，而在于**调用方是否需要介入管理**。如果不需要，它就是一次行为。

#### 💻 代码示例

一次行为在代码层面的经典形态：

```python
# 文件名: one_shot_example.py
# 功能: 演示一次行为（单次模型请求）的典型代码模式

import openai

client = openai.OpenAI()

# 一次行为：查天气
# 输入一次性给完，结果一次性拿到
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "user", "content": "今天北京的天气怎么样？"}
    ],
    tools=[{"type": "function", "function": {"name": "get_weather", /*...*/}}]
)

# 拿到结果就结束了——无需再管理
print(response.choices[0].message.content)
```

以上代码的典型特征：一次 `client.chat.completions.create()` 调用，得到结果后结束。调用方不需要关心"结果够不够好？要不要再试一次？要不要换一个方向继续查？"。这就是一次行为的边界。

---

### 3.2 持续目标追逐（Sustained Goal Pursuit）：多轮规划的必然

#### 🧠 直观理解

现在想象另一个场景：你要做一份"当前 AI 工程师就业市场调研报告"。你不能一次性完成——你需要先列出调研提纲，然后搜索招聘信息，阅读每个岗位描述，提炼薪酬水平、能力要求、工作地点分布，最后汇总成报告。这个任务天然需要多步走，而且**走在哪一步、下一步怎么走，需要不断观察和判断**。这就是持续目标追逐。

#### 📖 详细解释

持续目标追逐的核心特征是：**目标本身无法通过一次行为达成，需要多轮次的规划、执行、观察、调整**。它不是"把几个一次行为拼在一起"，而是需要一个**外层管理者**统筹这些行为，在每一步完成后做判断，并决定下一步。

#### 📋 案例分析：招聘市场调研任务

老师在课上做了一个非常详尽的拆解。目标任务是：调研当前 AI 工程师的招聘市场，汇总形成薪酬水平、能力要求、工作地点分布的完整报告。

如果把这个任务拆解成子任务链，会是什么样？

```
招聘市场调研任务
    │
    ├── Step 1: 生成规划任务板（Task Board）
    │   └── 先搞清楚要做哪些事，列一个 check list
    │
    ├── Step 2: 搞清楚 "AI Engineer" 的定义
    │   └── 检索相关概念，明确调研范围
    │
    ├── Step 3: 检索 AI Engineer 岗位
    │   └── 一次或多次搜索，找到岗位列表页面
    │
    ├── Step 4: 批量阅读岗位描述（JD）
    │   └── 并发打开多个页面，同时阅读，提取关键信息
    │
    ├── Step 5: 汇总信息
    │   └── 把 Step 4 中提取的零散数据合并成结构化内容
    │
    └── Step 6: 形成报告
        └── 按照薪酬、能力、地点等维度输出最终文档
```

这条链路非常长。关键的是，**链路中间的任何一步出了问题，都需要外层管理者去感知和处理**，而不是"把步骤 1-6 串在一起自动跑完"。

老师特别指出：

> 一次或者多次搜索，然后可能内部还会有搜索结果的一个整合——这对于我们的业务来讲，我们认为这个可能就是更小的一次行为。但外面这些阅读可能是一个并发的多次的多页面浏览行为。

也就是说，即便是"检索岗位"这一小步，内部可能已经包含了多次搜索和整合，但它仍然被视为一个"更小的一次行为"。而整个调研任务，是对很多很多这样的行为点**进行组装、整合、规划**。

> 💡 **核心洞见**：要完成一个"行为轮次显然应该大于单次"的任务，我们需要的不再是一个小的 function call，而是对大量行为点进行组织调度。这已经脱离了单次模型请求的范畴。

---

### 3.3 单次行为叠加不等于 Loop：缺失的"观察与重规划"环节

#### 🧠 直观理解

你可能本能地想到："那把多个一次行为串起来不就行了吗？先做完第一步，拿到结果传给第二步，然后第三步……"

老师在课上明确否定了这个直觉——**单次行为叠加，不等于 Loop**。

#### 🔗 推导链：从叠加到 Loop，缺了什么？

```
起点：多个一次行为，按顺序排列（Step 1 → Step 2 → Step 3 → ...）
    │
    ├── 问题 1：当 Step 2 的输入依赖于 Step 1 的输出时，Step 1 的输出质量能满足 Step 2 吗？
    │   为什么这很重要？因为 LLM 的输出具有不确定性——
    │   一次行为不会"自检"自己的结果是否足够好。
    │
    ├── 问题 2：如果 Step 1 失败了（或者产出不符合预期），谁来发现？
    │   为什么这很重要？叠加模式是"盲执行"——
    │   每一步只负责产出，不负责评估。
    │
    ├── 问题 3：任务进行到一半，发现最初的规划不合理，谁来调整？
    │   为什么这很重要？复杂任务在执行过程中会"长出"新信息——
    │   最初的规划是基于"未知"制定的，执行后获得新信息，可能推翻原计划。
    │
    └── 结论：叠加模式缺少一个关键角色——"观察-判断-决策"的外层循环。
        这个外层循环就是 Loop 的核心。
```

课堂上有一位学员的评论被老师重点肯定了："单次行为叠加不是 Loop。" 老师接着追问：**那缺了什么？**

答案是：**观察与重规划（Observation & Re-planning）**。

> ⚠️ **注意事项**：function calling 属于一次行为——模型在一次请求中决定调用哪个函数、传入什么参数，拿到结果后把结果返回。这是单次请求内的工具调用。而 Loop 是在 many many 次行为之间做组装整合，需要一个外层运行时去管理"下一步做什么"的决策。

#### 三种判断结果：Loop 中每一次观察后的岔路口

老师在课上详细解释了每一次观察和判断之后可能得到的三种结果：

```
┌─────────────────────────────────────┐
│  完成当前阶段 → 观察 → 判断         │
└──────────────┬──────────────────────┘
               │
       ┌───────┼───────┐
       ▼       ▼       ▼
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ 结果 1  │ │ 结果 2  │ │ 结果 3  │
  │ 重规划  │ │ 继续    │ │ 结束    │
  └────┬────┘ └────┬────┘ └────┬────┘
       │           │           │
       ▼           ▼           ▼
  当前规划不对    任务没完成    任务已完成
  需要调整方向    但方向正确    可以交付结果
  重新制定计划    执行下一步
```

| 判断结果 | 触发条件 | Loop 的动作 |
|---|---|---|
| 重规划（Re-plan） | 发现当前路线走不通、信息不足、方向错误 | 回到规划阶段，重新制定下一步目标 |
| 继续下一步（Continue） | 当前步骤完成，方向正确，任务未完成 | 推进到任务板中的下一个待办事项 |
| 结束（Done） | 所有子任务完成，最终报告已生成 | 终止运行，交付结果 |

> 🔗 **关联知识**：这三种判断结果是 M01 中三种终止条件（完成/受阻/越界）的进一步细化。这里的"重规划"对应的是"受阻但不终止"的情况——目标不变，但路径要调整。

---

### 3.4 从单次请求到系统工程的跨越

#### 📖 概念定位

我们可以把"一次行为"和"持续目标追逐"放到一个更大的坐标系里审视：

```
单次模型请求
    │ 能力边界：一次往返
    │ 管理复杂度：零
    │
    ▼
Function Calling（工具调用）
    │ 能力边界：一次请求内调用工具
    │ 管理复杂度：极低
    │
    ▼
行为叠加（Naive Chaining）
    │ 能力边界：多步串联，但盲执行
    │ 管理复杂度：低，但脆弱
    │
    ▼
Loop Engineering（持续目标追逐）
    │ 能力边界：多步 + 观察 + 重规划
    │ 管理复杂度：高，需要外层运行时
```

#### 🔀 对比表：一次行为 vs 持续目标追逐

| 维度 | 一次行为（One-Shot） | 持续目标追逐（Loop） |
|---|---|---|
| 典型任务 | 查天气、翻译一句话 | 市场调研、代码重构、系统诊断 |
| 行为轮次 | 1 次 | 不确定，通常 > 5 次 |
| 是否需要规划 | 否 | 是，且需要重规划能力 |
| 是否需要观察 | 否 | 是，每个阶段后必须观察判断 |
| 是否需要外层运行时 | 不需要调用方只是"发请求" | 需要调用方是"管理者" |
| 失败模式 | 返回错误或低质量结果 | 可能在中间某步发现路径错误，需要重试 |
| 代码模式 | 一次 `.create()` 调用 | 一个 while loop + 规划器 + 观察器 |

#### 🗺️ 层次图

```
           ┌────────────────────────────────────────┐
           │         智能体（Agent）                  │
           │  ┌──────────────────────────────────┐   │
           │  │       Loop（外层运行时管理）        │   │
           │  │  ┌──────────┐    ┌──────────┐    │   │
           │  │  │  规划器   │    │  观察器   │    │   │
           │  │  └─────┬────┘    └────┬─────┘    │   │
           │  │        │              │           │   │
           │  │        ▼              ▼           │   │
           │  │  ┌──────────────────────────┐     │   │
           │  │  │     任务板（Task Board）  │     │   │
           │  │  └──────────────────────────┘     │   │
           │  │                                   │   │
           │  │  ┌─────────┐ ┌─────────┐         │   │
           │  │  │行为点 A │ │行为点 B │  ...    │   │
           │  │  │(单次请求)│ │(单次请求)│         │   │
           │  │  └─────────┘ └─────────┘         │   │
           │  └──────────────────────────────────┘   │
           └────────────────────────────────────────┘
```

图中关键关系：**Loop 是外层管理者**，规划和观察是它的两只手；**行为点（一次行为）**是它调度的原子单元；**任务板**是它记录中间状态的共享内存。

> 💡 **核心洞见**：老师现场运行代码时，提到了一个非常具体的观察——执行过程中有 verify 失败的记录、有认为缺少证据源而修复的记录。这些都不是"一次行为"能处理的，因为它们需要 Loop 层面去感知问题、评估严重程度、决定修复策略。用老师的话说："整个这样的一个大的 Loop，我们才可以说是一个更接近于 Loop Engineering 的一个设计思路。"

---

> ⚠️ **注意事项**：老师特别强调，本节课的重点是讲 Loop Engineering 的设计思路，虽然课堂上有同学提到了 React 模式，但 Loop 与 React 有较大差异——这些差异会在 M06 中详细展开，本模块先锚定 Loop 的核心机制：**规划-执行-观察-重规划**的循环。

### 4.1 演进总览：从一次提问到一个系统

在进入今天的具体话题之前，我们需要先回答一个关键问题：**从最早的 Prompt Engineering 到今天的 Agent Engineering，中间到底发生了什么？**

这个问题之所以重要，是因为它回答的是"我们怎么走到这里的"。如果你只看当下的技术形态，很容易陷入"这东西不就是 XXX 吗"的误解 —— 只有把演进路径梳理清楚，才能真正理解每一层设计背后的"为什么"。

为了帮助大家建立整体框架，我们先看一张演进全景图：

```
时间线（简化）:
──────────────────────────────────────────────────────────────────────────────▶

┌─────────────────┐     ┌─────────────────────────┐     ┌───────────────────────┐
│ Prompt          │────▶│ Context Engineering     │────▶│ Tool/Function Calling │
│ Engineering     │     │ (Manus时代, 2024-2025)   │     │                       │
│                 │     │                         │     │                       │
│ 核心: 表达意图   │     │ 核心: 补充上下文信息       │     │ 核心: 让模型使用工具    │
│ 对象: 模型本身   │     │ 对象: 模型 + 外部信息      │     │ 对象: 模型 + 工具系统   │
│ 工程参与: 无     │     │ 工程参与: 信息检索层        │     │ 工程参与: 工具管理层    │
└─────────────────┘     └─────────────────────────┘     └───────────────────────┘
        │                         │                            │
        │       工程在模型之外       │                            │
        │       的分量逐步增大       │                            │
        ▼                         ▼                            ▼
               最终演进为完整 Agent System (Harness)
```

一条贯穿全课的线索是：**工程在模型之外的分量越来越大**。最早我们只需要写一段 prompt 丢给模型，而到了今天，模型周围包裹着一层又一层的工程系统。本模块聚焦于这条演进路径的前两步 —— 从 Prompt 到 Context。

### 4.2 第一阶段：Prompt Engineering

#### 🧠 直观理解

Prompt Engineering 就是"用自然语言把任务描述清楚，然后丢给模型"。你是一个用户，模型是一个回答者，你们之间只需要一段文字交互。

用生活类比：就像你给一个非常聪明但完全不了解你工作内容的同事发了一条微信，说"帮我调研一下 X 行业的情况"——你写得很认真，但同事只能凭自己已有的知识来回答你。他没法上网查，没法打电话问，没法打开你的内部系统看数据。

#### 📖 详细解释

Prompt Engineering 阶段的核心动作是：**把我们想让模型处理的任务，通过提示词（prompt）更清晰地表达出来**。这里的关注点全在"如何写好一段话"上 —— 如何表达意图、如何设定约束、如何引导输出格式。

当时市面上流行的说法是："我来教你怎么清晰地表达自己的诉求，然后模型就能帮你把任务做完。"大量自媒体博主、Prompt 工程师在做的事情，本质上就是教人写提示词。

这个阶段的交互模型非常简单：

```
┌──────────┐     一段 Prompt      ┌──────────┐     一段 Response     ┌──────────┐
│   用户    │─────────────────────▶│   模型    │─────────────────────▶│   用户    │
└──────────┘                      └──────────┘                      └──────────┘

整个过程只涉及两个实体：用户 和 模型。
模型能给出什么，完全取决于：
1. 训练数据里有什么
2. 你问的问题清不清晰
```

在这个阶段，prompt 的核心承接对象是**模型本身**，不是一套系统。用户写好 prompt，提交给模型，模型返回结果 —— 一条直线，没有中间环节。

#### ❌ Prompt Engineering 的三大局限

很快，大家就发现单纯依赖 prompt 是不够的。以"调研 Harness Engineering 的招聘市场情况"这个任务为例，Prompt Engineering 暴露了三个致命局限：

| 局限 | 具体表现 | 为什么是致命问题 |
|---|---|---|
| **拿不到外部事实** | 模型不知道当前招聘市场上到底有多少 Harness 工程师岗位、薪资范围是多少 | 模型的知识截止于训练数据，无法获取实时信息 |
| **无法主动获取信息** | 模型即使发现自己不知道，也没有办法去访问招聘网站获取数据 | 模型只有"嘴"没有"手"，不能行动 |
| **无法产生可溯源的结论** | 即使模型给出了看起来头头是道的分析报告，你无法验证其中的数字和结论是否真实 | 你不知道结论是从哪来的，缺乏可信度 |

一句话总结：**只做一次调用，模型能交付的就只能是一个"回答"**—— 不管这个回答多么流畅、多么像一个专业的分析报告，里面的事实性信息你都是无法核验的。

---

### 4.3 反思："模型吞噬一切"为什么是错的？

#### 🧠 背景

在 Prompt Engineering 兴起的时期，有一种非常流行的观点：

> "模型能力越强，它能处理的任务就越多。随着模型能力的持续提升，最终模型会吞噬一切（Model will eat everything）。"

老师在上课时直接指出：**这个观点是错误的**。不仅如此，老师还强调"这是本课程最核心的观点之一"，并且要求大家不能只记住结论——**必须理解为什么**，这样下次再有人抛出这个观点时，你才能有理有据地反驳。

#### 📖 推导链：为什么"模型吞噬一切"是错的？

```
起点: 我们不断给模型追加 prompt，认为模型能处理一切任务
  │
  ├── 第1步: 模型的能力瓶颈在哪里？
  │   我们给模型的任务"调研 Harness Engineering 招聘市场情况"
  │   为什么这个任务模型做不了？不是因为它不够聪明，而是它拿不到数据。
  │
  ├── 第2步: 模型能自动去获取数据吗？
  │   不能。模型只会根据 prompt 生成文本，它没有"访问外部系统"的能力。
  │   为什么不能？因为模型只是一个推理引擎，不是一个执行引擎。它可以通过训练知道"招聘网站"是什么，
  │   但它无法真正打开浏览器去爬数据。
  │
  ├── 第3步: 那模型能力再强一点，能不能自己获取数据？
  │   不能。这是架构层面的问题，不是能力层面的问题。
  │   为什么？因为"获取数据"需要的不是更强的推理，而是与外部世界交互的接口（API、浏览器、数据库连接）。
  │   再强的推理能力也无法替代一个不存在的接口。
  │
  └── 结论: 模型能力再强，也无法替代工程系统的作用。
      纯模型只能"思考"，不能"行动"。任务越复杂，越需要在模型之外构建工程支撑。
```

> 💡 **核心洞见**："模型吞噬一切"这个观点的根本问题在于：它混淆了"推理能力"和"执行能力"。模型可以做越来越好的推理，但它仍然需要工程系统来帮它连接外部世界、获取实时数据、产生可溯源的结论。**没有工程支撑，纯模型只能响应，不能行动。**

> ⚠️ **注意**：这不是说模型不重要——模型是核心引擎。但"核心引擎"不等于"全部"——就像汽车需要发动机，但只有发动机你开不走一辆车。

---

### 4.4 第二阶段：Context Engineering

#### 🧠 直观理解

既然纯 Prompt Engineering 的问题在于"模型拿不到外部信息"，那么很自然的下一步就是：**在提问的时候，把模型需要的外部信息也一起塞给它**。

Context Engineering 做的事情就是这样：除了你的原始问题（prompt），你还可以把相关的背景资料、最新数据、参考文档一起放进请求里。模型收到的不只是"帮我写个报告"，而是"帮我写个报告，这是相关的招聘数据……这是行业趋势……这是竞品分析……"。

这个阶段在 2024-2025 年 Manus 兴起时被广泛讨论，Manus 带火了一个词，就叫 "Context Engineering"。

#### 📖 详细解释

Context Engineering 与 Prompt Engineering 的最大区别在于：**在 prompt 之外，增加了一个信息补充层**。

```
Prompt Engineering 时代:
┌──────────┐    Prompt     ┌──────────┐
│   用户    │──────────────▶│   模型    │────▶ Response
└──────────┘               └──────────┘


Context Engineering 时代:
┌──────────┐    Prompt     ┌──────────┐
│   用户    │──────────────▶│          │
└──────────┘               │          │
                           │   模型    │────▶ Response (带有数据依据的回答)
┌──────────┐   Context     │          │
│ 外部信息   │──────────────▶│          │
│ (数据/文档)│              └──────────┘
└──────────┘
```

在 Context Engineering 阶段，我们不再只靠一段 prompt 打天下。我们开始意识到：要让模型的回答有数据依据，就必须让模型"看到"这些数据。

但这里立即暴露了一个新的问题，也是老师在教学现场引导大家思考的问题：

> "招聘信息是怎么进入 context 的呢？如果只有模型和 context 两个部分，谁来负责把招聘信息从招聘网站搬运到 context 里？"

**答案是什么？没有人 —— 或者更准确地说，在 Context Engineering 的最初形态中，这个角色还没有被清晰定义出来。** 这就为下一阶段的演进（Tool/Function Calling）埋下了伏笔 —— 我们需要一个机制来自动获取外部信息并放入 context。

#### 🔀 Context Engineering vs Prompt Engineering 对比

| 维度 | Prompt Engineering | Context Engineering |
|---|---|---|
| **传给模型的内容** | 只有原始需求描述 | 需求描述 + 补充信息/数据/文档 |
| **模型的回答依据** | 仅限训练数据中的知识 | 训练知识 + 本次请求中提供的上下文 |
| **核心承载对象** | 模型本身 | 模型 + 信息检索系统 |
| **工程参与程度** | 几乎为零 | 开始引入信息检索/注入层 |
| **回答的可溯源性** | 无法溯源 | 部分可溯源（依赖提供的上下文） |
| **典型问题** | "模型怎么说都对，但数字哪来的？" | "context 里的数据从哪来？谁负责放进去？" |

---

### 4.5 In-Context Learning 深度解析

#### 🧠 直观理解

Context Engineering 之所以能成立，不是因为我们在 prompt 里多写了几段话 —— 而是因为模型本身具备一项底层能力：**In-Context Learning（上下文中学习）**。

用白话讲：你告诉模型一些它原本不知道的事情，模型**不只是**机械地把这个信息塞进回答里 —— 它能够**理解、整合、推理**这些信息，然后产生超出简单拼凑的有意义的输出。

#### 📖 详细解释

In-Context Learning 是一个**模型层的底层能力**，不是工程方法。它的核心含义是：模型在单次请求的上下文中，能够学习并利用对话中提供的新信息。

为什么说这是一个"底层能力"？我们可以做一个思想实验：

> 假设模型不具备 In-Context Learning，所有响应只能从预训练语料中组织输出。那么，**不论你在 prompt 里塞多少额外信息，模型都无法将这些信息融汇到答案中**。你告诉它今天招聘市场的最新数据，它也只能用训练时的旧数据回答你。

正是因为在实践中我们发现模型具备这种能力，Context Engineering 才有了意义 —— 我们可以把模型原本不知道的信息提供给它，而模型不是照本宣科，而是对这些信息进行**组织、融合、推理**后再使用。

> ❓ **常见疑问**：In-Context Learning 会让模型的参数发生变化吗？

**不会。** In-Context Learning 影响的是单次请求的上下文窗口中的信息处理方式，**不改变模型的权重参数**。这也是它名称中 "In-Context"（在上下文中）的含义 —— 学习的效应仅限于本次对话。

#### 💡 案例：Harness 工程师 JD 识别

老师在上课时用了一个非常具体的例子来展示 In-Context Learning 的威力。这个例子值得完整展开，因为它清晰地展示了模型在上下文中做"语义推断"的能力。

**场景设定：**

你给模型提供了两条背景信息：

1. **第一条信息（概念定义）**：
   > "Harness 工程师"这个职位目前更多强调的是为模型提供外围工程性质的工作。但在这个词被定调之前，这类工程师也被称为 **AI Agent 工程师**、**AI 系统工程师**、或者**应用算法工程师**。

2. **第二条信息（待判断的数据）**：
   > 现在我手上有三份 JD（职位描述）：
   > - DeepSeek 的 Harness 工程师招聘 JD
   > - 华为的 AI Agent 工程师招聘 JD
   > - 某公司的模型智能系统工程师招聘 JD

然后你问模型：

> "这里面哪些岗位是 Harness 工程师的招聘 JD？"

**模型会怎么回答？**

模型会告诉你：三份 JD **全都是**。

这是怎么推理出来的？模型做了如下几步语义推断：

```
步骤1: 理解第一条背景信息
  ├── Harness 工程师 ≈ AI Agent 工程师
  ├── Harness 工程师 ≈ AI 系统工程师（或类似称呼）
  └── 这三者是同类项

步骤2: 对三条 JD 进行匹配
  ├── DeepSeek JD: 标题直接就是 "Harness 工程师" → 直接匹配 ✓
  ├── 华为 JD: 标题是 "AI Agent 工程师" → 根据步骤1，等同于 Harness 工程师 ✓
  └── 某公司 JD: 标题是 "模型智能系统工程师" → 根据步骤1的语义推断，也等同于 Harness 工程师 ✓

步骤3: 得出结论
  └── 三个 JD 全部符合要求
```

> 💡 **核心洞见**：在这个过程中，你**没有直接告诉**模型"华为的 JD 就是 Harness 工程师的 JD"。除了 DeepSeek 那份标题直接匹配，另外两份 JD 与 "Harness 工程师" 之间的关系是通过**语义推断**建立的，而不是硬编码的映射。模型做了一个"语义层面的桥接"——这是 In-Context Learning 区别于简单关键词匹配的本质所在。

#### ⚠️ In-Context Learning 的特点与注意事项

| 特点 | 说明 | 实践含义 |
|---|---|---|
| **不改变模型参数** | ICL 只在单次请求的上下文窗口内起作用，不会修改模型权重 | 你不需要担心一次错误的上下文"教坏"模型 |
| **影响范围限定单次请求** | 这次请求中输入的错误信息，只影响这次的输出 | 下一条请求中如果修正了错误信息，输出就不会再受之前错误的影响 |
| **上下文错误会污染输出** | 如果你给模型的上下文信息是错的，模型**会信任你的输入**并基于错误信息产生错误输出 | 模型在大多数情况下不会质疑你的输入，上下文信息的质量管理非常重要 |
| **模型不会主动质疑** | 即使你提供的上下文信息与模型的训练知识矛盾，模型通常也会优先采纳你提供的信息 | 你作为信息提供者，责任重大 |

> ⚠️ **注意**：正因为模型会信任你给的上下文，上下文的**质量管理**就变得极为关键。如果招聘 JD 数据本身有误、或者检索到的文档是过时的，这些错误会直接污染模型的输出 —— 这也是为什么后续演进中需要构建更完整的工程系统来管理信息质量。

### 4.6 上一代 NLP 系统 vs 现代大模型（小冰/小黄鸭类比）

> 📌 **Rule 3 提示**：这个类比在录音中持续近 3 分钟，是理解 In-Context Learning 独特性的关键对比。必须完整展开，不得压缩。

#### 🧠 为什么讲这个？

老师在解释 In-Context Learning 时，特意拉出了上一代 NLP 对话系统做对比。目的不是讲历史，而是通过对比让一个抽象概念变得**可感知**：当你看到一个不具备 In-Context Learning 的系统时，它的行为是什么样的？和现在的模型有什么区别？

#### 📖 上一代系统的工作方式

以微软"小冰"或者"小黄鸭"这样的第一代对话机器人为例。你如果玩过这些系统，典型的交互是这样的：

> 你："给我讲个笑话吧。"
> 小冰：[输出一个笑话]
>
> 你："小鸡为什么要过马路？"
> 小冰："因为它要去马路对面吃草莓。"

看起来还挺智能的，对吧？但它的底层逻辑和你想象的可能完全不同。

**上一代系统的核心流程：**

```
用户输入: "小鸡为什么要过马路？"
    │
    ▼
┌──────────────┐
│ 第1步: 分词    │  使用分词工具（如结巴/jieba）把句子切碎
│              │  结果可能是: "小鸡" / "为什么" / "要" / "过" / "马路"
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 第2步: 关键词  │  提取关键词，比如"小鸡 + 过马路"
│     提取     │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 第3步: 匹配    │  在预先构建的问答库中查找相关词条
│     答案库   │  QA Pair: "小鸡 + 过马路" → "因为它要去马路对面吃草莓"
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 第4步: 输出    │  直接输出匹配到的标准答案
│     标准答案  │  不做任何推理、不做任何整合
└──────────────┘
```

这个过程本质上是一个**规则引擎判断逻辑**：分词 → 匹配答案库 → 输出标准模板。整个系统就是一个大型的 QA 对的查找表。

> 💡 **关于结巴（jieba）分词**：老师在讲解时提到了 jieba 这个 Python 中文分词库。在上一代 NLP 系统中，分词是整个处理流程的入口步骤 —— 只有把句子正确切开，才能提取关键字去答案库匹配。到了现在的大模型时代，分词的重要性已经大幅下降，模型更多依赖子词（subword）级别的 tokenization，而不是传统的显式分词。

#### 📖 上一代系统的致命缺陷

现在我们把前面讲 ICL 时的那个测试搬过来，看看会发生什么：

> 你："我告诉你，小鸡过马路是为了到马路对面吃冰淇淋。所以现在我问你：小鸡为什么要过马路？"
>
> 上一代系统："因为它要去马路对面吃草莓。"（依然输出标准答案）

**为什么会这样？** 因为上一代系统的逻辑是：

```
1. 收到整个输入
2. 分词、提取关键词: "小鸡 + 过马路"
3. 在答案库中查找: "小鸡 + 过马路" → "因为它要去马路对面吃草莓"
4. 输出标准答案

至于你前面说的"吃冰淇淋"——系统完全没有理解，也不需要理解。
它只是在做关键词匹配，匹配到哪个就输出哪个。
```

**不管你在上文补充什么信息，只要答案库里对应的是"吃草莓"，它就会输出"吃草莓"。** 规则引擎不会因为你的上下文而改变自己的输出逻辑。

#### 🔀 对比：规则引擎 vs In-Context Learning

| 维度 | 上一代 NLP（规则引擎） | 现代大模型（具备 ICL） |
|---|---|---|
| **工作方式** | 分词 → 匹配答案库 → 输出标准模板 | 理解上下文语义 → 推理 → 生成回答 |
| **上下文利用** | 无法利用上下文补充信息 | 能理解并整合上下文中的新信息 |
| **回答来源** | 预先写死的 QA 对 | 综合训练知识 + 本次上下文动态生成 |
| **灵活度** | 答案库有什么就只能答什么 | 可以处理从未见过的问题组合 |
| **对"补充信息"的反应** | 完全无视，依然输出标准答案 | 会基于补充信息调整回答内容 |
| **典型局限** | "小鸡过马路"说了要吃冰淇淋，但它还是答"吃草莓" | 上下文给错了会带偏（但下次请求可修正） |

> 💡 **核心洞见**：这个对比揭示了一个本质区别 —— 上一代系统的"智能"是**静态固化**的（答案库定死了），而现代大模型的智能（在 ICL 的加持下）是**动态可塑**的 —— 你可以在单次交互中"注入"新的知识，模型能当场学会并运用。这种"当场学习"的能力，是 Context Engineering 得以存在的根本前提。

### 4.7 In-Context Learning ≠ RAG（重要辨析）

#### 🧠 直观理解

这是学员最容易混淆的一对概念，老师在上课时花了相当篇幅做辨析。

**简单总结：ICL 是"模型能做什么"，RAG 是"我们用这个能力做了什么"。**

ICL（In-Context Learning）是模型自身表现出的一种**能力** —— 它是模型层的东西，属于基础模型研究的范畴。

RAG（Retrieval-Augmented Generation）是一种**工程方法** —— 它是利用 ICL 这个能力，通过检索系统把外部信息注入上下文，从而提升模型回答质量的一种技术方案。

#### 📖 详细解释

```
┌─────────────────────────────────────────────────────┐
│                   RAG (工程方法)                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐  │
│  │  检索系统  │───▶│ 信息注入  │───▶│  模型生成回答  │  │
│  │ (向量匹配) │    │ (放入context)│   │              │  │
│  └──────────┘    └──────────┘    └──────┬───────┘  │
│                                         │           │
│                                依赖     │           │
│                                ┌───────▼───────┐   │
│                                │ In-Context    │   │
│                                │ Learning      │   │
│                                │ (模型能力)     │   │
│                                └───────────────┘   │
└─────────────────────────────────────────────────────┘
```

**为什么说 ICL 是 RAG 成立的前提？**

如果没有 ICL，RAG 会退化成什么样子？就是上一代 NLP 系统的样子 —— 通过向量匹配找到一个标准答案模板，然后输出。有没有价值？有，但很有限。

正是因为有了 ICL，模型能够**理解**注入到上下文中的信息并进行**语义层面的整合**，RAG 才从一个"高级一点的查找替换"变成了真正能提升回答质量的技术。

> 💡 **核心洞见**：老师明确说："In-Context Learning 不是 RAG，但 In-Context Learning 是 RAG 成立的最基本前提。" 这个关系就像"汽油发动机"和"汽车"的关系 —— 发动机不是汽车，但没有发动机，汽车就跑不起来。

#### 📖 RAG 技术的时间线

RAG 技术本身**早于大模型时代出现**。老师提到，RAG 的相关论文大概在 2020 年左右就已经发表，当时 NLP 领域有一些研究方向是利用向量检索来更好地找出应答所需的词条。

早期 RAG 的思路是：标准输入 → 向量匹配 → 找到标准模板 → 输出。这和上一代规则引擎的本质区别不大，只是匹配方式从"关键词匹配"变成了"向量相似度匹配"。

**大模型的 ICL 能力改变了 RAG 的格局**：有了 ICL 之后，RAG 不再只是"找到答案模板然后输出"，而是把检索到的信息作为上下文注入给模型，让模型去理解、整合、重新表达。这才是我们今天看到的 RAG 能产生巨大价值的原因。

> ❓ **常见误解**：以为 RAG 和 ICL 是同一回事，或者以为 RAG 必须先于 ICL 存在。
>
> **纠正**：ICL 是模型能力（底层），RAG 是工程方法（上层）。逻辑顺序是：先有 ICL 这个能力被发现 → 然后工程师利用这个能力构建了 RAG 这种方法。从时间线上看，RAG 的技术思路（检索+生成）确实早于大模型，但今天我们谈论的"有效 RAG"，其价值是由 ICL 赋予的。

### 4.8 工程角色的浮现

#### 🧠 直观理解

回顾整个演进过程，我们可以清晰地看到一条主线：**工程在模型之外的分量越来越大**。

在 Prompt Engineering 时代，工程几乎不参与 —— 写好 prompt，丢给模型，结束。

在 Context Engineering 时代，工程开始介入 —— 我们需要构建信息检索层，把外部数据注入到每一次请求的上下文中。

```
工程在模型之外的角色变化:

Prompt Engineering         Context Engineering         Tool/Function Calling
        │                        │                          │
  工程参与度: 0%          工程参与度: ~30%              工程参与度: ~60%+
        │                        │                          │
  只管把话说清楚           需要构建信息检索              需要构建工具管理、
                          和上下文注入系统              执行调度、结果验证
```

#### 📖 详细解释

老师强调了一个非常关键的观察：在 Context Engineering 时代，**除了模型之外，我们开始在外面加入工程了**。

这不是一个"锦上添花"的事，而是一个**质的变化**。因为一旦我们承认：

1. 模型需要外部信息才能高质量完成任务
2. 外部信息不会自动飞到模型面前
3. 需要有东西负责"获取信息 → 整理信息 → 注入上下文"这套流程

那么"这套流程由谁来做"就成了一个无法回避的工程问题。而且，随着任务的复杂度增加，这个工程系统的复杂度也会同步增长 —— 最终就演进成了我们今天讨论的 **Harness（完整 Agent 工程框架）**。

> 🔗 **前置知识**：详见 [三、单词请求 vs 持续目标追逐](#三单词请求-vs-持续目标追逐)—— 那里说明了为什么复杂任务需要超越单次模型请求。

> 🔗 **延伸阅读**：本文 [五、演进之路（下）：从 Tool 到 Workflow 到 Loop](#五演进之路下从-tool-到-workflow-到-loop)将继续讨论 Tool/Function Calling 到 Workflow 的演进，展示工程角色如何进一步扩大。

#### 🔗 推导链：从 Prompt 到工程系统的必然性

```
起点: 我们想让模型高质量地完成一个复杂任务
  │
  ├── 第1步: 只给 prompt，够吗？
  │   不够。模型拿不到外部事实，只能输出基于训练数据的推测。
  │   为什么不够？因为复杂任务需要实时数据（如招聘市场行情）做支撑。
  │
  ├── 第2步: 那加上 context（外部信息），够了吗？
  │   信息能力够了，但新的问题来了：谁来把信息放进 context？
  │   为什么这是个问题？因为信息不会自动出现——需要有人去检索、筛选、整合。
  │
  ├── 第3步: 所以必须构建工程系统
  │   在模型之外构建信息检索和上下文管理层。
  │   为什么"必须"？因为模型只是一个推理引擎——它需要工程系统扮演"眼睛"和"手"的角色。
  │
  ├── 第4步: 任务越复杂，工程越重
  │   当任务不仅需要"信息"还需要"行动"时，工程系统还要管理工具调用、执行调度、结果验证
  │   为什么？因为每个新增的需求都对应一种新的工程能力。
  │
  └── 结论: 从 Prompt 到 Agent，本质上就是工程在模型之外不断生长、不断膨胀的过程。
      Context Engineering 是这个过程中的第一个关键转折点。
```

---

### 💡 本节一句话总结

> **从 Prompt Engineering 到 Context Engineering 的演进，本质上是在模型这个"大脑"之外，逐步构建起一个能够为大脑提供信息、连接外部世界的"身体"——而 In-Context Learning，就是这个大脑能够"消化"外部信息的底层生理基础。**

## 五、演进之路（下）：从 Tool 到 Workflow 到 Loop

### 5.1 Context 时代的局限：信息仍是预处理的

#### 🧠 直观理解

在上一阶段（Context Engineering），我们通过 RAG、知识库等方式为模型补充信息。这就像请了一位专家顾问，但每次开会前你必须提前把相关资料放到他桌上——他只能看到你准备好的东西，无法自己去查资料，也无法主动去办事。

#### 📖 详细解释

在 Context 阶段，即使我们使用了 RAG 或直接的事实性补充，这些信息本质上仍是**预处理过程**的产物。无论是从向量数据库中检索到的文档，还是我们人为注入的上下文，都是「系统当前已知的、预先准备的」信息。

这就带来了两个核心局限：
1. **模型无法主动获取信息**：模型只能消费我们喂给它的信息。它不能自己决定「我需要更多信息，让我去查一下」。
2. **模型不能自动改变外界事件**：模型输出的只是一段文本响应，不会导致外部系统发生任何实际变化。它不能发邮件、不能创建工单、不能调用 API。

> 💡 **核心洞见**：在这个阶段，模型本质上还是一个「基于所见输入信息进行响应式输出的问答系统」。给它什么，它就根据什么回答；不给它的，它无从知晓；它想做的，没有渠道去执行。

### 5.2 行为系统 vs 响应系统

| 维度 | 响应系统 / 问答系统 | 行为系统 |
|---|---|---|
| 信息流 | 单向：用户输入 → 模型输出 | 双向：模型可向外获取信息，也可向外输出控制 |
| 模型角色 | 应答者 | 指令发出者 + 任务协调者 |
| 外部交互 | 无 | 有（通过工具/API） |
| 工程复杂度 | 低（维护 prompt + 上下文） | 高（需管理工具调用、状态、执行链） |
| 能力边界 | 受限于已知信息 | 可动态扩展 |

### 5.3 Tool / Action 时代：Function Calling 的登场

2023 年之后，Function Calling 的出现标志着 AI 应用进入全新阶段。核心突破是：**模型可以给出指令，而指令可以被进一步执行**。

工作机制：
1. 开发者定义工具的 schema
2. 模型在推理中判断需要调用工具时，输出结构化的 tool_call 指令
3. 外围工程系统实际执行函数调用
4. 执行结果返回模型，模型继续推理

> 💡 **核心洞见**：模型负责「决定做什么」，工程负责「实际去做」。

### 5.4 两层认知的协同扩展

| 维度 | Function Calling 之前 | Function Calling 之后 |
|---|---|---|
| 模型角色 | 应答式模块、问答机器人 | 指令发出者，可提出行为要求 |
| 输出类型 | 纯文本回答 | 文本回答 + 结构化工具调用指令 |
| 工程职责 | 维护模型所需信息 | 维护信息 + 执行模型指令 + 收集结果 |

### 5.5 Workflow 时代：多步工作的编排与管理

两个关键发现推动了 Workflow 时代：
1. **复杂任务不能一步完成，往往是多轮推进**
2. **多轮请求需要外层运行管理**（工作节点、连接关系、状态管理、条件分支、并发、循环）

```
用户请求
    │
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 搜索职位   │────▶│ 判断相关性 │────▶│ 合并去重   │
│(Tool Call)│     │(LLM Call) │     │(LLM Call) │
└──────────┘     └──────────┘     └──────────┘
    │                                  │
    │        ┌──────────┐              │
    └───────▶│ 状态存储  │◀─────────────┘
             └──────────┘
                  │
                  ▼
             ┌──────────┐
             │ 格式化输出 │
             │(LLM Call) │
             └──────────┘
```

### 5.6 案例分析：Coze 与 Dify 的落地困境

企业采购 Coze/Dify 的逻辑：模型够强 + 可视化编排 = 非工程师也能做 AI 应用。但实际结果是：
- **低估了使用者对模型能力的理解深度**——界面的低门槛 ≠ 认知的低门槛
- **没有人在乎工程那一层**——当时主流叙事是「强模型 + 多数据」，工程被边缘化

> 💡 **核心洞见**：Coze/Dify 的困境揭示了一个普遍规律——每当 AI 能力向前迈进一步，工程需求不是减少了，而是**转移并加深了**。

### 5.7 AI 应用的三个关键要素

2024 年主流说法：「AI 应用 = 更强大的模型 + 更多的私有数据」

老师从 2024 年开始补充第三要素：**对模型的控制能力**。到 2026 年，这个概念被概括为 **Harness Engineering**。

> 💡 **核心洞见**：Harness 这个词的出现，标志着一个重要的范式转变——AI 应用开发的核心挑战不再是「如何让模型更强」，而是「**如何有效地控制已经足够强的模型**」。

### 5.8 Engineering 含义的变迁

| 演进节点 | 人们关注的词 | Engineering 受到的质疑 | 工程变化 |
|---|---|---|---|
| 早期 | Prompt | 「哪里 Engineering 了？」 | 经验性、试错式 |
| Context 时代 | Context | 「也就加了个数据库」 | 维护数据库、检索策略 |
| Tool 时代 | Function | 「不就是 API 调用嘛」 | 注册工具、编排执行、回传结果 |
| Workflow 时代 | Workflow | 质疑开始减少 | 状态管理、节点编排、条件分支 |
| 现在 | **Engineering** | **质疑消失** | 工程深度成为核心竞争力 |

### 5.9 工程角色的进化链

```
无工程 ──▶ 维护接口 ──▶ 维护数据库 ──▶ 执行指令 ──▶ 缓存/状态存储 ──▶ ...
（纯Prompt） （给模型包接口） （Static Context） （FuncCall）  （Workflow）
```

> 💡 **核心洞见**：工程不只是在变大，而是在不断回应模型的需求。工程和模型就是这样一步一步相互演进过来的。

## 六、React Loop 详解与智能回路雏形

### 6.1 React 论文回顾：从"方法"到"回路"

#### 🧠 直观理解

React 不是一套固定的代码模板，而是一种**让模型在"思考"和"行动"之间循环推进**的方法论。你可以把它想象成一个边想边做的人：先想一想下一步该做什么，然后动手去做，做完之后观察结果，再根据结果决定下一步——如此循环往复，直到任务完成。

#### 📖 详细解释

React（Reasoning + Acting）是 2024 年左右提出的一种让大模型完成复杂任务的方法。它的核心流程可以概括为三段式循环：

```
思考 (Reasoning) → 行动 (Acting) → 观察 (Observation) → 思考 → 行动 → ...
```

**三个关键阶段：**

1. **思考/规划（Reasoning）**：模型分析当前状态，决定下一步该做什么。这一步会生成一个"执行指令"——一个具体的、可执行的行动描述。为什么这一步至关重要？因为这是整个回路中**唯一存在模型**的节点，所有的"智能"都集中在此处。

2. **行动（Acting）**：根据执行指令，在运行环境中实际执行操作——调用工具、运行代码、查询数据库等。注意：这一阶段**不涉及模型推理**，是纯执行层。它需要的是一个可靠的运行环境，而非智能。

3. **观察（Observation）**：获取行动的执行结果。这个结果会被记录下来，堆积形成**状态存储**，作为下一轮思考的输入——模型需要知道"刚才做了什么、结果如何"才能决定下一步。

> 💡 **核心洞见**：React 的规划**不等于**一开始就有一份完整的任务板（Task Board）。每一轮 React 的规划只负责"**下一步做什么**"——它是一步一步往前推进的，而不是预先规划好所有步骤。这一点是 React 和后续 Loop Engineering 的关键差异所在。Loop Engineering 会在 React Loop 之上增加全局的任务管理结构，而纯 React 没有这个层级。

**React 是方法，不是固定套路。** "React"这个词描述的是一种**方法论**，而不是某一套写死的代码流程。不同的工程实现可以有不同的 React 形态——你可以在 LangChain 的 AgentExecutor 里看到它，也可以在 AutoGPT 的自主循环里看到它——虽然实现方式不同，但核心的"思考→行动→观察"闭环是共通的。

**AutoGPT 时代的 React：缺少主任务板控制。** 在 AutoGPT 时期，React 表现为一个"自由循环"——模型不断推理、执行、观察、再推理。这种方式的问题在于：没有上层的任务管理机制来控制整体进度。模型可能在一个子任务上反复循环、迷失方向，导致任务无法收敛。这也正是后续 Loop Engineering 想要解决的问题：在 React Loop 之上，增加更复杂的外层工程结构来管理整体任务。

### 6.2 React Loop 架构详解

#### 🔄 React Loop 白板图还原

以下是老师在课堂上现场绘制的 React Loop 架构图。这张图是理解"智能回路"概念的基石——老师每次讲到 Agentic 系统架构时都会反复展示它。

```
                          ┌───────────────────────────┐
                          │      可用行为 / 工具         │
                          │    (Available Tools)       │
                          └─────────────┬─────────────┘
                                        │
                                        ▼
 ┌──────────────┐          ┌───────────────────────────┐
 │  用户目标输入  │─────────▶│  ① 理解和规划               │
 │  (User Goal)  │          │  Reasoning & Planning     │
 └──────────────┘          │  🧠 存在模型               │
                           └─────────────┬─────────────┘
                                         │
                                  🟡 中间产物
                                         │
                                         ▼
                           ┌───────────────────────────┐
                           │       执行指令              │
                           │     (Instruction)          │
                           └─────────────┬─────────────┘
                                         │
                                         ▼
                           ┌───────────────────────────┐
                           │  ② 指令执行                 │
                           │  Acting                    │
                           │  ⚙️ 不存在模型              │
                           │  需要运行环境               │
                           └─────────────┬─────────────┘
                                         │
                                         ▼
                           ┌───────────────────────────┐
                           │  ③ 执行结果                 │
                           │  Observation               │
                           │  结果堆积 → 形成状态存储     │
                           └──────┬────────────┬───────┘
                                  │            │
                                  │            ▼
                                  │   ┌───────────────────────────┐
                                  │   │       最终输出              │
                                  │   │      (Output)              │
                                  │   └───────────────────────────┘
                                  │
                                  │   ◀═══ 回路连线 (Loop) ═══
                                  │   执行结果回传规划节点
                                  │   形成闭环，驱动下一轮推理
                                  │
                                  ▼
                          回到 ① 理解和规划
```

#### 📖 各节点详解

| 节点 | 是否有模型 | 功能说明 |
|---|---|---|
| **用户目标输入** | — | 用户提出的最终目标，是整个回路的启动信号。所有的推理和执行都围绕这个目标展开 |
| **可用行为/工具** | — | 辅助输入：模型可以调用的工具、API、函数等行为集合。规划节点需要知道自己"有哪些武器可用"才能制定执行计划 |
| **① 理解和规划** | 🧠 有模型 | 核心智能节点。模型综合分析用户目标 + 可用行为 + 当前状态（含上轮执行结果），推理生成下一步的执行指令。这是整个回路中**唯一有模型参与决策**的地方 |
| **执行指令** | 🟡 中间产物 | 规划节点输出的具体行动描述。老师用黄色标注它，因为它是"思考"和"行动"之间的桥梁——它不是最终结果，而是驱动执行层工作的命令 |
| **② 指令执行** | ⚙️ 无模型 | 纯执行层。根据指令在运行环境中执行操作（调用工具、运行代码等）。不涉及任何模型推理，是机械化的执行过程 |
| **③ 执行结果** | — | 指令执行的产出物。每次执行的结果会堆积存储，形成系统中的**状态**。规划节点在做下一轮决策时，需要读取这个状态——"之前做了什么、结果如何" |
| **最终输出** | — | 当任务完成或满足终止条件时，将最终结果输出给用户 |
| **回路连线** | 🔄 核心特征 | 将执行结果**回传到规划节点**的那根连线。正是这根连线让系统能够"根据上一轮的结果，决定下一轮的行为"——这是智能回路区别于普通工作流的核心标志 |

> ⚠️ **注意事项**：老师特别强调了**有模型节点**和**无模型节点**的区分。理解和规划节点中存在模型（LLM），它是整个回路的"大脑"；指令执行节点中不存在模型，只是一个运行环境，是回路的"手脚"。这个区分对于理解"**智能到底体现在哪里**"至关重要——智能只在规划节点中，执行节点本身没有任何智能成分。

### 6.3 React Loop 也是一种 Workflow

#### 📖 详细解释

如果我们把 React Loop 和传统 Workflow 放在一起比较，会发现一个重要的结论：**它并没有脱离 Workflow 的基本结构**。

- **运行节点**：理解和规划是一个运行节点，指令执行也是一个运行节点——Workflow 的核心特征"有节点、有流转"在这里完全具备。
- **状态存储**：每次执行结果的堆积形成了系统中的状态。规划节点在做决策时，需要读取这个状态——这与传统 Workflow 中节点间的数据传递在结构上是一致的。
- **节点间的流转链路**：目标 → 规划 → 指令 → 执行 → 结果，这是一条清晰、标准的执行链路，与传统 Workflow 的 DAG（有向无环图）结构没有本质差别。

正是因为具备这些特征，React Loop 本质上也可以被视为一种 **Workflow**——只不过它的流转逻辑不是预先写死的，而是在每一步由模型动态决定的。

> 💡 **核心洞见**：吴恩达（Andrew Ng）老师曾提出一个观点：**任何已有的 Workflow，都有可能被进行 Agentic 改造**。也就是说，如果一个 Workflow 中的某些决策节点可以被替换为模型来判断和规划，它就具备了"智能化"的可能性。智能回路并没有脱离工作流的基本结构——它是在原有 Workflow 框架中**嵌入了模型的决策能力**。

#### 🔀 对比辨析：React Loop vs 传统固定 Workflow

| 维度 | 传统固定 Workflow | React Loop |
|---|---|---|
| 流程决定方式 | 预先写死的 if-else 逻辑 | 每步由模型动态决定 |
| 分支路径数量 | 有限的分支路径（工程师预先枚举） | 理论上无限——模型可以根据上下文自行推理出新的路径 |
| 对新情况的适应性 | 差——遇到未定义的分支会直接失败 | 强——模型可以根据上下文自行推理应对 |
| 状态管理方式 | 通常有明确的状态机（state machine） | 执行结果堆积形成状态，模型自行理解和利用这些状态 |
| 循环机制 | 通常是固定次数的循环或条件循环 | 模型**自主决定**是否继续循环、循环到何时停止 |
| 智能成分 | 无智能，纯逻辑驱动 | 有智能——模型参与决策，决定每一步的行动方向 |

### 6.4 智能回路的关键判断标准

#### 📖 详细解释

仅仅"有循环"并不等于"智能回路"。老师给出了一个**最关键、最核心的判断标准**：

> ⚠️ **智能回路的核心判断标准**：是否由**模型**来决定后续的行为——而不是按照写死的逻辑依次执行。

具体来说：

- **非智能回路**：虽然也有循环，但"下一步做什么"是由**工程师预定义的规则**决定的。比如："如果执行结果包含错误，则重试最多 3 次"——这是写好的逻辑，不是模型的自主判断。这种回路的本质是`for`循环或`while`循环，不会因为上下文变化而产生新的分支。

- **智能回路**：每一步"接下来该做什么"是由**模型根据当前上下文推理出来的**。模型会说："根据刚才的结果，现在需要查一下数据库"或者"任务已经完成，不需要继续了"——这些决策不是预先编码的，而是模型**实时推理**产生的。

为什么这个标准如此重要？因为它是区分"自动化"和"智能化"的分水岭。传统工作流也能实现"自动循环"，但那只是按照既定规则机械运转；而智能回路的本质是：**把决策权交给了模型**，让模型用自己的推理能力来判断每一步的方向。

> 💡 **核心洞见**：这个判断标准在后续的讨论中会反复出现。它直接回答了"AutoGPT 算不算 Loop Engineering"这个问题——AutoGPT 的循环中，模型确实在决定后续行为，因此它**具备智能回路的特征**；但它缺少上层任务板的控制，所以在工程结构上还有很大的提升空间。Loop Engineering 的目标不是否定 AutoGPT 的智能性，而是在其之上构建更完善的工程框架。

> ❓ **常见疑问**："模型决定后续行为"是不是意味着每次都在调用大模型？那成本会不会很高？
>
> **回答**：是的，模型参与决策确实会带来额外的调用成本和延迟。但这也是"智能"的代价——用模型的推理能力换取更强的灵活性和适应性。在实际工程中，可以通过结果缓存、选择合适的模型规格、设置合理的终止条件（如最大循环次数、超时控制）等方式来优化成本。这些成本优化策略本身也是 Loop Engineering 需要解决的核心问题之一。

### 6.5 "Loop"一词的命名由来

#### 📖 详细解释

为什么叫"Loop"？这个命名的来源非常直观，就是上面架构图中那条**从执行结果回传到规划节点的回路连线**。

老师讲道："因为往往我们会需要去让它尝试的，不断的去进行尝试，往前去推进，都会出现这样的一根连线，这根连线会把某一段的这个结果，往这一个规划的这个节点上面去进行引导。"

这根连线带来了闭环：

```
规划 → 指令 → 执行 → 结果 → 回到规划 → 再规划 → 再执行 → ...
```

每一次循环都在上一轮结果的基础上做出新的判断和行动。正是这个"回路"让系统具备了"**不断往前尝试**"的能力——不是一条直线走到底，而是在每一步都回头审视、调整方向、继续前进。

"Loop Engineering"这个名字，就是源于对这条回路连线的工程化——如何设计、管理、优化这条回路，让智能体能够高效、可靠地通过循环完成复杂任务。回路连线不是画在图上的装饰，它是整个系统能够"智能运转"的物理基础。

### 6.6 从小回路到复杂系统工程

#### 📖 详细解释

React Loop 给出了智能回路的基本框架，但老师明确指出：**仅有这样一个小小的智能回路是不够的**。

```
为什么 React Loop 不够用？
    │
    ├── 问题 1：缺乏整体任务管理
    │   React Loop 只知道"下一步做什么"，
    │   不知道"整个任务进展到什么程度了"。
    │   没有一个上层的任务板 (Task Board) 来追踪全局进度。
    │
    ├── 问题 2：缺乏错误恢复机制
    │   如果某一步执行失败了，回路能发现，
    │   但"要不要重试""换一种方式""直接放弃"——
    │   这些决策的工程化实现远比看起来复杂。
    │
    ├── 问题 3：缺乏资源管理
    │   每一轮循环都在消耗 token、时间、工具调用次数。
    │   没有外层机制来控制成本上限和防止无限循环。
    │
    ├── 问题 4：缺乏复杂任务分解
    │   一个复杂目标如何拆成多个子任务？
    │   子任务之间如何协调和依赖？
    │   这些都需要外层工程来支撑。
    │
    └── 问题 5：缺乏工具编排
        React Loop 假设工具列表是固定的，
        但实际场景中工具需要动态注册、
        权限控制、结果格式化等复杂处理。
```

正因为存在这些问题，老师指出："我们需要再做更多的事情"——模型已经被包裹在一个**相对来说比较庞大的工程**里面了。这个外层工程包括：任务管理、状态持久化、错误处理与重试策略、资源消耗控制、工具注册与调度、安全与权限管理等一系列基础设施。

> 💡 **核心洞见**：React Loop 是智能回路的**最小可行单元**（MVP）。它证明了"模型在循环中决策"这条路是可行的。但要让这条路走通、走稳、走远，还需要大量的工程实践——这就是 Loop Engineering 的全部内涵。从 React Loop 到 Loop Engineering，本质上是**从"能跑通"到"能工程化"的跨越**。

> 🔗 **关联知识**：本文接下来将展开 React Loop 不够用之后，我们需要什么样更复杂的外层工程来支撑真正可用的智能体系统——详见 Skills 与工程复杂度的持续提升章节。

### 7.1 回顾：React Loop 够用吗？

在上一章中，我们构建了一个最简 React Loop——模型在思考与执行之间反复循环，直到完成任务。但如果要使用一个外部提供的**工具集**（如 Skills）——它可能是一套脚本、一个 GitHub 仓库、一份指令文档，需要先安装、配置环境才能用——React Loop 能做这件事吗？

**答案是不能。** React Loop 缺少：外部存储空间、安装与寻路能力、信息筛选与管理能力。

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│1.读取仓库 │─▶│2.寻路    │─▶│3.安装下载 │─▶│4.任务触发 │─▶│5.间歇披露 │─▶│6.上下文  │
│  入口     │  │读README  │  │          │  │          │  │截取信息   │  │  注入    │
└──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

### 7.4 寻路机制

> 💡 **核心洞见**：寻路是 Skills 与 Function Calling 最本质的区别之一。Function Calling 中函数的签名预先定义好；而在 Skills 中，模型需要**自己找到**「该怎么用这个工具」。

以 Anthropic Skills 为例，"撰写 Word 文档"任务中：
- `document-convert.md` → 无关，丢弃
- `document-read.md` → 无关，丢弃
- `image-to-text.md` → 无关，丢弃
- `creating-new-doc.md` → **匹配！** 保留

### 7.5 间歇式披露与上下文注入

| 维度 | 全量注入 | 间歇式披露 |
|---|---|---|
| **注入时机** | 一开始全部注入 | 按任务阶段逐步披露 |
| **上下文占用** | 始终占用大量 token | 只占用当前阶段所需 token |
| **可扩展性** | Skills 越多越差 | Skills 数量不影响单次上下文大小 |

### 7.6 Skills 与 Function Calling 的辨析

| 维度 | Function Calling | Skills |
|---|---|---|
| **定义方式** | 预先声明函数签名 | 外部文档/脚本，需被「发现」 |
| **调用路径** | 固定：模型决定调 X → 系统执行 X | 不固定：寻路 → 安装 → 截取 → 注入 → 执行 |
| **复杂度来源** | 函数本身的逻辑 | 发现、装载、筛选、注入的全链路 |
| **Token 效率** | 函数描述全量在 context 中 | 按需截取，只保留相关部分 |

> 💡 **类比**：Function Calling 像手机内置功能（打电话）；Skills 像从应用商店下载 App——需要搜索、下载安装、打开需要的功能。

### 7.7 工程复杂度持续提升轨迹

```
M05: 单次推理 → M06: React Loop → M07: Skills → M08: Harness
 简单问答        思维+行动循环      外部工具集的      正式引入外层
                                   发现与装载        工程底座
```

### 7.8 Harness 概念引出

> 所有这些外部的——外部存储空间、安装寻路、信息管理——都是需要额外在**外层**构建的。所有这些外部的东西，就叫做 **Harness**。

> 🔗 **关联知识**：下一章（M08）将深入讨论 Harness 的定义及其与 Loop 的关系。

## 八、Harness Engineering 与 Loop Engineering 的关系

### 8.1 Harness Engineering：底座工程的定义

#### 🧠 直观理解

想象你要让一个机器人去完成复杂的家务——它需要记住自己做过什么（状态保存）、知道自己现在在做什么（运行支撑）、确保每一步不会搞砸（控制验证）。**Harness Engineering 就是为模型搭建这套"可以放心跑起来"的底座工程**。没有它，模型就像一个人站在流沙上干活，每一步都在下沉。

#### 📖 详细解释

Harness Engineering 的正式定义：

> **支撑模型运行行动、保存执行状态、接受控制验证的底座工程。**

| 维度 | 含义 | 为什么需要 |
|---|---|---|
| **支撑模型运行行动** | 提供模型执行 action 的运行时环境 | 模型不是凭空运行的，需要沙箱、进程管理、资源调度 |
| **保存执行状态** | 持久化记录模型每一步的上下文和中间结果 | 复杂任务跨多轮执行，状态丢失意味着任务不可恢复 |
| **接受控制** | 外部系统可以干预、暂停、重定向模型的行动 | 安全边界、人工审核节点、成本控制 |
| **验证** | 对模型的输出和执行结果进行校验 | 确保行动不越界、结果符合预期 |

#### 🔀 对比辨析：Harness 工程 vs Harness 产品

| 维度 | Harness 工程（底座工程） | Harness 产品（产品化产物） |
|---|---|---|
| **本质** | 一套工程架构设计 | 基于 Harness 工程思想构建的可用产品 |
| **典型例子** | 概念层面的模块拆解、状态管理设计 | Open Cloud、Codex、Claude Code |
| **源码可见性** | 产品源码中模块边界模糊，没有做清晰的工程化拆解 | 可以看到完整代码，但模块分散在产品执行流程中 |
| **是否可直接复用** | 目前缺乏标准的开源 Harness 框架 | 可以直接使用产品，但不能直接提取出 Harness 层 |

> ⚠️ **注意事项**：老师特别强调，如果你想通过阅读 Open Cloud 或 Codex 的源码来学习 Harness Engineering，你会失望。这些产品源码中，Harness 相关的模块并没有被清晰地拆解出来——很多状态管理、验证逻辑是**散落在产品执行的各个步骤中**的。

---

### 8.2 Loop Engineering：在底座之上运行的设计方案

Loop Engineering 的正式定义：

> **在 Harness 底座之上，利用其提供的运行环境，通过反馈对下一步行动进行改变和规划的设计方案。其目标就是让模型不断追逐更好的结果产出。**

用一句话概括 Harness 与 Loop 的关系：

```
Harness 保证环境状态能被不断地维护和读取。
Loop 把 Harness 维护的这些信息让模型去与目标成功标准进行比较，来决定接下来的行为方向。
```

---

### 8.3 关键辨析

#### 🔀 Loop vs React：形似而神不似

| 维度 | React（如 ReAct 模式） | Loop Engineering |
|---|---|---|
| **是否利用自查自纠正** | 是 | 是 |
| **是否有外层工程底座** | 否——运行在简陋环境中 | 是——建立在完整 Harness 底座之上 |
| **尺度** | 小型行为驱动（单轮思考→行动→观察） | 大型方案（跨多任务、长时间线、复杂状态） |
| **状态管理** | 依赖上下文窗口内的文本 | Harness 提供结构化的状态持久化 |

> 💡 **核心洞见**：React 给了我们一个关键启发——模型可以通过"自查→自纠正→再行动"的方式推进事项。但 React 本身是一个**算法模式**，而 Loop Engineering 是一个**系统架构方案**。

#### 🔀 Loop ≠ for/while 循环

> ⚠️ **注意事项**：老师特别指出，工程师听到 "Loop" 第一反应就是 `for` 循环或 `while` 循环。**大错特错。**

核心区别——Loop Engineering 的三段推进链：

```
行动产生新的状态
    →  新状态影响新的判断
        →  新判断改变下一步行为
            →  （事情不断向前推进，而不是原地打转）
```

| 维度 | for/while 循环 | Loop Engineering |
|---|---|---|
| **每次迭代的关系** | 每次迭代相互独立或仅通过计数器关联 | 上一次行动的结果**实质性改变**下一次行动 |
| **推进方向** | 原地重复（"循环"） | 向前推进（"前进"） |
| **决策逻辑** | 终止条件在循环开始时已确定 | 每一步的判断基于上一步产生的新状态动态调整 |

---

### 8.4 Loop 反馈机制的实例解析

```
任务启动: "调研目标行业各公司的薪酬水平"
    │
    ▼
┌──────────────────────────────────────────────────────┐
│ Step 1: 搜索 → 找到 20 个相关页面                      │
│ Step 2: 验证 → 92 个页面中可访问 12 个                  │
│ Step 3: 去重 → 12 个中有 3 个重复 → 最终 9 个独立样本    │
│ Step 4: 分析 → 9 个独立样本中有 2 个披露了薪酬信息       │
│ Step 5: 决策 → 样本量不足，继续扩大范围还是缩小？         │
└──────────────────────────────────────────────────────┘
```

> 💡 **关键观察**：每一步的行动都产生了新的状态，而新的状态又影响了下一步的判断。这就是"**行动→状态→判断→新行动**"的完整闭环。

### 8.5 本节总结

```
┌──────────────────────────────────────────────────────────┐
│                    Loop Engineering                         │
│          （设计方案：驱动模型不断追逐更好结果）                │
│   ┌─────────┐   ┌──────────┐   ┌──────────┐   ┌────────┐  │
│   │ 行动     │──▶│ 产生新状态 │──▶│ 比较目标  │──▶│ 调整方向 │  │
│   └─────────┘   └──────────┘   └──────────┘   └────────┘  │
└──────────────────────────┬───────────────────────────────┘
                           │ 依赖
                           ▼
┌──────────────────────────────────────────────────────────┐
│                   Harness Engineering                      │
│      （底座工程：支撑运行、保存状态、接受控制、验证结果）       │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐  │
│   │ 运行时环境 │  │ 状态持久化 │  │ 控制节点  │  │ 验证链路 │  │
│   └──────────┘  └──────────┘  └──────────┘  └─────────┘  │
└──────────────────────────────────────────────────────────┘
```

> 💡 **一句话总结**：没有 Harness 的 Loop 是纸上谈兵，没有 Loop 的 Harness 是空转的机器。两者结合，才能让模型在复杂任务中真正**有效推进**。

## 九、Loop 的两种尺度与无人值守困境

### 9.1 前置研究回顾：Loop 的学术脉络

#### 🧠 直观理解

Loop（循环）是 Agent 的核心工作机制——模型不是一次性给出答案，而是通过反复的「感知 → 思考 → 行动 → 观察 → 调整」来完成一个任务。这个思想在学术界早已有之，并不是工程上的凭空创造。

#### 📖 详细解释

在讨论 Loop Engineering 之前，有必要回顾几种已被广泛研究的前置范式，它们共同奠定了「Agent 通过循环完成任务」这一思想基础：

- **React（Reasoning + Acting）**：将推理（Reasoning）和行动（Acting）交错执行。模型先思考下一步该做什么，再执行，然后观察结果，继续思考。它不是一次推理到底，而是步进式地展开。
- **Reflection（反思）**：在执行若干步骤后，模型停下来对自己的行为和结果进行回顾，判断是否有偏差或遗漏，并据此调整后续策略。这相当于在 Loop 内部加入了一个「自检」环节。
- **Evaluate-Optimizer（评估-优化）**：模型生成一个候选输出，由一个评估器对其打分，然后优化器根据评分对下一次生成进行调整。这本质上是将「试错-改进」的循环机制形式化了。

> 💡 **核心洞见**：这些前置工作已经揭示了一个共性模式——在模型的工作流程中，通过循环回溯、自我审查、逐步优化，可以有效提升任务完成质量。Loop 不是一项发明，而是这些思想的工程化统一。

在此基础上，出现了更具体的执行范式（Execution Paradigm）：

| 范式 | 核心驱动力 | 关键特征 |
|---|---|---|
| **Turn-based**（轮次驱动） | 对话轮次 | 每轮一问一答，模型在每轮中根据上一轮的新信息做决策 |
| **Goal-based**（目标驱动） | 预设目标 | 以任务目标为锚点，每轮行动都朝向目标推进 |
| **Plan-based**（计划驱动） | 预先规划 | 先制定整体计划，然后按计划分步执行，中途根据反馈调整 |

> 🔗 **前置知识**：这三种范式的框架由 Harness 提供底座支持，详见 [八、Harness Engineering 与 Loop Engineering 的关系](#八harness-engineering-与-loop-engineering-的关系)

这些范式告诉我们：Loop 的设计不是随意的，它需要考虑驱动方式（轮次？目标？计划？）以及每一步如何衔接。这也是为什么后续讨论会聚焦于 Loop 的两种尺度——因为不同粒度的问题，适合用不同的范式去解决。

---

### 9.2 Loop 的两种尺度：微观与宏观

#### 🧠 直观理解

思考一个问题：当 Agent 接到「帮我做市场调研报告」这个任务时，它的内部到底在发生什么？

如果放大来看，你会发现**两套循环在同时运转**。小循环：Agent 在搜索某一类信息时，搜一下、看一眼结果、判断够不够、不够再搜——这是一个快速的、小粒度的反馈环。大循环：Agent 搜完所有信息后，停下来问你：「我理解的情况是这样，对吗？」得到确认后再进入下一阶段——这是一个慢速的、跨阶段的推进环。

这就是 Loop 的两种尺度：**微观 Loop** 解决「这一步做得对不对」，**宏观 Loop** 解决「整体方向走得对不对」。

#### 📖 详细解释

在任何具有一定复杂度的 Agent 任务中，Loop 的运行同时存在于两个层面：

```
宏观 Loop（跨阶段推进）
    │
    ├── 阶段一：前景调研
    │       ├── 微观 Loop：搜索 → 评估 → 不够 → 再搜（重复 N 次）
    │       └── 确认点：调研结果是否充分？
    │
    ├── 阶段二：信息确认（与用户交互确认点）
    │       └── 等待用户反馈 → 根据反馈调整
    │
    ├── 阶段三：制定计划
    │       ├── 微观 Loop：生成草案 → 自检 → 修改（重复 N 次）
    │       └── 向用户提交计划 → 等待确认
    │
    └── 阶段四：执行
            ├── 微观 Loop：执行某步骤 → 观察结果 → 调整参数（重复 N 次）
            └── 完成
```

#### 🔀 对比辨析

| 维度 | 微观 Loop（Micro Loop） | 宏观 Loop（Macro Loop） |
|---|---|---|
| **作用范围** | 单个阶段性目标内部 | 跨越多个阶段性目标 |
| **驱动方式** | 实时反馈驱动 | 目标/计划驱动 |
| **典型操作** | 执行一个动作、观察结果、调整参数 | 调研→确认→规划→确认→执行 |
| **人的参与** | 通常不需要，自动迭代 | **需要在关键节点确认** |
| **循环周期** | 秒级到分钟级 | 分钟级到小时级 |
| **失败后果** | 单步失败，可重试 | 方向错误，整阶段报废 |

> ❓ **常见疑问**：微观 Loop 和宏观 Loop 是不是同一个 Loop 的不同级别？
>
> 可以这样理解，但有一个本质区别：**微观 Loop 可以在信息闭环内自动收敛**——模型评估「信息够了、不够、再加一点」这个判断不需要人的参与。但**宏观 Loop 每进入一个新阶段之前，往往需要一个「外部信息注入」的节点**——比如用户确认调研方向是否正确、计划是否合理。这个「外部注入」的存在与否，恰恰是把两种 Loop 区分开的关键。

---

### 9.3 微观 Loop：阶段性目标内部的反馈闭环

#### 🧠 直观理解

想象你让一个助手查一家公司的基本信息。助手打开搜索引擎，输入关键词，看了前三条结果，发现其中两条是广告、一条数据太旧。于是它换了一个关键词再搜，这次找到了一份年报，它觉得信息够了，停下来把结果整理给你。

整个过程不到三十秒，但助手内部做了一个完整的微观 Loop：**执行 → 观察 → 判断 → 调整 → 再执行**。

#### 📖 详细解释

微观 Loop 的完整执行链如下：

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 读取当前 │───▶│ 选择下个 │───▶│ 执行具体 │───▶│ 观察环境 │───▶│ 记录证据 │
│  状态    │    │  任务    │    │  行动    │    │  变化    │    │  (日志)  │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └────┬────┘
                                                                  │
                                                                  ▼
                                                           ┌──────────┐
                                                           │ 验证目标  │
                                                           │ 达成情况  │
                                                           └─────┬────┘
                                                                 │
                                                    ┌────────────┼────────────┐
                                                    ▼                         ▼
                                              ┌──────────┐            ┌──────────┐
                                              │ 未完成：  │            │  已完成： │
                                              │ 修正策略  │            │  停止循环 │
                                              │ 重新循环  │            │  输出结果 │
                                              └──────────┘            └──────────┘
```

每一步拆解：

1. **读取当前状态**：回顾到哪了、上次做了什么、环境有什么反馈
2. **选择下个任务**：基于当前状态和目标，决定下一步该做什么
3. **执行具体行动**：调用工具、查询 API、生成内容等
4. **观察环境变化**：看工具返回了什么结果、环境状态有何变化
5. **记录证据**：把关键信息记下来（写入 workspace 或上下文），防止重复搜索或遗忘
6. **验证目标**：判断当前阶段的子目标是否已经达成
7. **修正或停止**：未达成则调整方向再试；已达成则输出并退出

> ⚠️ **注意事项**：微观 Loop 能够自动运行的前提是**每一步的判断依据都包含在已有信息之内**。也就是说，模型不需要外部注入新信息就能判断「够了」还是「不够」。一旦某个判断需要模型不知道的条件（比如「用户到底想要国内数据还是全球数据」），那就超出了微观 Loop 的自闭环能力，需要上升到宏观 Loop 的确认节点。

---

### 9.4 宏观 Loop：跨长程目标的递推机制

#### 🧠 直观理解

宏观 Loop 解决的不是「这一小步怎么走」，而是「这条路走不走得通」。它像一个项目经理，把一个复杂项目切成几段，每做一段就向你同步一次，确认方向正确后再继续下一段。

#### 📖 详细解释

以 **Superpowers** 这类 Agent 框架为例，它在处理一个复杂任务时的宏观 Loop 运作如下：

**完整的宏观 Loop 流程（Superpowers 案例）**：

```
用户发起任务
    │
    ▼
┌──────────────────────────────┐
│ 阶段一：前景调研               │
│   ├── 搜索相关领域信息         │
│   ├── 整理调研材料            │
│   └── [微观 Loop 内部迭代]    │
│       信息够了？没够→追加搜索   │
└──────────────┬───────────────┘
               │ 调研完成
               ▼
┌──────────────────────────────┐
│ 🔴 确认点 1：向用户确认理解   │
│   "我理解的情况是...对吗？"    │
│   └── 等待用户回答            │
│       ├── 用户说对 → 继续      │
│       └── 用户纠正 → 调整后继续│
└──────────────┬───────────────┘
               │ 确认通过
               ▼
┌──────────────────────────────┐
│ 阶段二：制定计划（Plan）       │
│   ├── 拆解任务为子步骤        │
│   ├── 预估时间/资源           │
│   └── [微观 Loop 内部迭代]    │
│       计划合理？不合理→修改     │
└──────────────┬───────────────┘
               │ 计划完成
               ▼
┌──────────────────────────────┐
│ 🔴 确认点 2：向用户确认计划   │
│   "这个计划你觉得合适吗？"     │
│   └── 等待用户回答            │
│       ├── 用户同意 → 执行     │
│       └── 用户不同意 → 重新规划│
└──────────────┬───────────────┘
               │ 确认通过
               ▼
┌──────────────────────────────┐
│ 阶段三：按计划逐步执行         │
│   每个子步骤内部有微观 Loop     │
│   执行 → 观察 → 判断 → 调整    │
└──────────────┬───────────────┘
               │
               ▼
            任务完成
```

> 💡 **核心洞见**：Superpowers 之所以在每个阶段节点停下来等用户输入，**不是因为技术做不到自动推进**，而是因为在这些节点上，模型缺少做出正确判断所需的**上下文信息**——这些信息只有用户知道（例如：调研方向对不对、计划是否合理、数据库权限有没有）。

每个确认点背后的逻辑是：

- **确认点 1（理解确认）**："我是把你当成一个公司的招聘负责人在调研，对吗？"——模型对用户身份和意图的猜测需要验证。
- **确认点 2（计划确认）**："这个计划你觉得合适吗？"——模型对任务拆解的取舍需要验证。
- **确认点 3（信息确认）**："我需要登录付费招聘数据库，你可以帮我吗？"——模型对资源可及性的假设需要验证。

**自主模式的变体**：

如果你不想在每个确认点都回复，Superpowers 也可以切换为「自主模式」：当模型在确认点询问用户但没有得到回应时，它会用**最可能的答案**来自动推进。例如，模型假设用户会同意按它猜的方向继续，然后就据此往下走。

> ⚠️ **注意事项**：自主模式的风险非常大。一旦模型在确认点猜错了方向（比如它以为你要找的是「技术岗招聘信息」，但你其实要找的是「销售岗招聘信息」），后续所有步骤都会在错误的方向上推进。而且因为用户没有介入纠偏，错误会累积到任务结束才被发现，造成的浪费是整环的。

### 9.5 无人值守困境：长程 Loop 的核心矛盾

#### 🧠 直观理解

一个很自然的想法是：「能不能让 Agent 全程自动跑，我只需要看最终结果就行了？」

老师的实践经验给出了一个直接的回答：**在长程任务中，完全无人值守的 Loop 目前没有解法（或者说，是一个非常困难且尚未解决的问题）。**

这个困境不是技术不够好，而是由任务本身的特性决定的。

#### 📖 详细解释

#### 🔗 推导链：为什么长程任务无法无人值守

```
起点：你给模型一个初始目标，比如「帮我做一份竞品分析报告」
  │
  ├── 第1步：模型开始搜集信息
  │   为什么不能自动跑完？
  │   → 你的初始目标包含了很多「你不知道你没说」的信息
  │      比如：竞品范围是国内的还是全球的？分析维度是价格还是功能？
  │      模型只能猜一个方向来搜。猜对了，第一步顺利；猜错了，后续全废。
  │
  ├── 第2步：模型在搜集过程中遇到付费数据库
  │   为什么需要人的输入？
  │   → 模型不知道你有没有权限、愿不愿意花钱、有没有替代数据源
  │      它可以猜「用户肯定不愿意花钱」就跳过这部分数据
  │      但也许这部分数据恰恰是你最需要的。跳过等于关键信息缺失。
  │      这些需要人参与来做判断。
  │
  ├── 第3步：模型基于搜集的信息制定分析框架
  │   为什么需要人确认？
  │   → 分析框架的选择本身就涉及价值判断
  │      比如选 SWOT 还是 Porter's Five Forces？
  │      模型不知道你的受众是谁、偏好哪种格式
  │
  └── 结论：初始目标（「做一份竞品分析报告」）本身不会创造未知事实。
      也就是说，目标文本里写不出来的信息，模型在运行过程中无法凭空获得。
      而这些缺失信息恰恰决定了任务能否正确完成。
      因此，人必须在过程中参与——要么提前写死在 prompt 里（但长程任务往往写不全），
      要么在确认节点介入回答。
```

#### 核心矛盾展开

**矛盾一：「初始目标不会创造未知事实」**

你写下的目标，再详细，也不可能预见到执行过程中所有需要决策的瞬间。目标定义的是「要什么」，它不包含「路径上每个路口怎么选」。而这些路口的选择信息，只有在任务推进过程中才会浮现——它们就是「执行过程中才暴露的未知事实」。

> 举例：你在 prompt 里写了「分析特斯拉的竞争对手」。但模型在执行中需要判断「竞争对手」的定义范围——传统车企算不算？中国新势力算不算？飞行汽车公司算不算？你不可能在 prompt 里穷举所有分类，但模型必须做出选择。

这意味着什么？意味着**你人需要盯住这些未知事实的出现节点**，在模型暴露出一项不确定性时，及时给予信息补充。

**矛盾二：「没有逐轮值守不等于系统不需要人的输入」**

这两个概念经常被混淆：

- **逐轮值守**：每一轮对话你都在旁边盯着，像直播一样参与每一个决策。
- **人的输入**：在关键决策节点上，系统需要你提供它无法自主获取的信息。

区分清楚这一点很重要。你不需要逐轮盯着，你只需要在确认点出现时给予反馈。但**没有确认点的自动推进**有一个前提——系统在整个执行过程中，不需要任何它本来不知道的信息。而长程任务恰好天然具备「不可预见的决策点」，所以这个前提不成立。

> 💡 **核心洞见**：完全无人值守的 Loop 和长程任务，在本质上是一对矛盾。长程任务的特点是「执行过程中会不断出现新的决策需求」，而无人值守意味着「执行过程中不接收任何新信息」。两个要求互相排斥——这就是为什么老师说「这本身是矛盾的」。

**矛盾三：模型错误判断时，无人纠偏**

即使不存在「未知信息的补充需求」，模型自己也会犯错。在微观 Loop 中，错误可以很快被发现和纠正（一个搜索不够，再搜一次就行了）。但在宏观 Loop 中，一个阶段性的方向判断错误会导致后续所有阶段在错误轨道上运行。

如果人在确认点介入，错误可以在阶段切换时被发现。但如果无人值守，错误会一路累积，最终输出一个完全偏离预期的结果。

#### 🔀 对比辨析

| 场景 | 能否无人值守 | 原因 |
|---|---|---|
| 短程单步任务（如「翻译这段话」） | 可以 | 所需信息一步到位，无中间决策点 |
| 有限范围的微观 Loop（如「搜到三条来源后整理」） | 可以 | 判断依据（信息量是否足够）在闭环内 |
| 长程多阶段任务（如「做竞品分析报告」） | **无法完全无人值守** | 确认点需要外部信息注入 + 需要纠偏机制 |

### 9.6 实践定位：有限范围内的 Loop Engineering

#### 🧠 直观理解

既然长程无人值守目前没有解法，那 Loop Engineering 讨论的到底是什么？老师的界定是：**在一个有限范围内，模型通过一定轮次不断把任务往前推进，能够把任务完成——这就够了。**

#### 📖 详细解释

Loop Engineering 的当前实践焦点，不是「怎么让一个 Agent 连续无人运行 24 小时」，而是：

1. **在有限范围内设计高效的 Loop 回路**：比如一个 React Loop 中，如何设计 workspace（工作空间）让状态传递更准确？如何设计 tool 调用让每一步的反馈更清晰？
2. **明确确认点的设置**：什么时候该停下来等用户输入？什么时候可以自动推进？确认点的设计本身就是 Loop Engineering 的核心课题。
3. **让微观 Loop 的收敛速度更快**：减少无效迭代、避免循环卡死、确保结果质量。

把期望锚定在「有限范围内保证完成质量」而不是「无限范围内追求无人化」，是更现实的工程路线。

> ⚠️ **注意事项**：L-- Evans 提出的 Loop Engineering 定义本身也强调了这是「讨论性的、实验性的」——它不是一份成品规范，而是一个正在演变的方向。在老师的实践经验中，这个方向的边界条件就是：**再长程的无人值守目前大家还没有解法**。

### 9.7 答疑：手写 Agent 还是用 Open Cloud？

---

**🙋 问题**：现在手写 Agent 的多，还是直接用 Open Cloud 的多？

> 📖 **背景**：学员听到 Loop 的机制讨论后，自然产生了工程实践方面的疑问——既然 Loop 设计这么复杂，能不能直接用封装好的产品而不自己写？

**💬 老师回答**：

Open Cloud 是一个**高封装的完整产品**。它把底层的 Loop 机制、工具调用、状态管理全部封装好了，你直接调用 API 就能获得 Agent 能力。

但**场景决定了你选什么**：

- 如果你的需求刚好落在 Open Cloud 封装的能力范围内（比如需要一个能做简单查询和整理的助手），用 Open Cloud 是最高效的选择。
- 但如果你需要在**你的系统里用到 Open Cloud 的局部能力**，或者需要**定制 Loop 的行为**，那就需要自己开发。

具体来说，以下场景需要手写：

| 场景 | 为什么需要手写 |
|---|---|
| 调整 React Loop 的回路上限或评估逻辑 | 封装产品不会暴露这个粒度的控制 |
| 让 Agent 与你的系统共享更多上下文 | 需要改造 workspace（工作空间）的数据结构 |
| 改变确认点的设置逻辑（比如从每阶段确认改为关键节点确认） | 确认点逻辑是 Loop Engineer 需要自定义的 |
| 接入企业内部的私有工具或数据源 | 封装产品无法感知你的内部系统 |

> 💡 **延伸**：这个问题本身揭示了一个重要原则——**选方案取决于你需要多大的控制粒度**。如果你只需要「有没有 Agent 能力」这个维度的答案，封装产品足够。但如果你需要「Agent 在什么条件下做什么决策」这个维度的控制力，那就必须深入 Loop 内部去调整。后续 [十、Loop Engineering 核心要素与任务板驱动架构](#十loop-engineering-核心要素与任务板驱动架构) 会具体展开 Loop Engineering 的核心要素。


## 十、Loop Engineering 核心要素与任务板驱动架构

### 10.1 Loop Engineering 的三大核心要素

#### 🧠 直观理解

如果把 Loop Engineering 比作一次长途自驾：你需要知道**目的地**（目标），需要仪表盘告诉你**到没到**（完成判断条件），还需要明确你是要**送人还是送货**（交付形态）。缺少任何一样，这趟旅程都不会有意义——要么没有方向，要么不知道什么时候停，要么交付了错误的东西。

#### 📖 详细解释

当我们要构建一个真正可执行的 Loop 工程时，输入侧必须准备好三个维度的信息。它们不是可有可无的辅助要素，而是决定了整个 Loop 能否成立的**前置条件**。

---

##### 要素一：目标（Goal）—— 追逐的方向

> 🔗 **推导链：为什么目标是第一要素**

```
起点: 一个没有目标的 Loop 系统
  │
  ├── 第1步: 系统启动循环
  │   为什么？Agent 进入自主执行模式，需要不断迭代
  │
  ├── 第2步: 系统在每次迭代中需要判断"接下来做什么"
  │   为什么？Loop 的本质是反复执行某个动作序列，动作选择依赖方向
  │
  ├── 第3步: 没有目标 → 方向缺失 → 动作选择无据可依
  │   为什么？Agent 不知道该优化什么、该往哪个方向前进
  │
  └── 结论: 目标定义了 Loop 的前进方向，是 Loop Engineering 的第一前置条件
```

目标的作用不是"给一个大致的方向"，而是**为每一轮迭代提供可参照的追逐方向**。在 Loop Engineering 的框架下，目标不是一个静态的输入，它会在整个执行过程中被反复读取、反复对照、反复评判。如果你连要追什么都不知道，Loop 本身的存在就失去了意义。

##### 要素二：完成判断条件（Completion Criteria）—— 反复评判的标准

> 💡 **核心洞见**：Codex 在设计上将目标从普通对话框中提取出来，放到**边栏（Sidebar）**中常驻。这不是一个 UI 细节，而是一个**架构决策**——它意味着"目标"在 Loop 执行期间承担的是一个**持续评判的锚点角色**，而不是一次性的输入。

为什么完成判断条件如此关键？因为如果你只告诉系统"去写一篇好文章"，系统永远不知道什么时候该停。你必须把"完成"的标准细化到可验证的程度——例如"文章超过 3000 字""覆盖了 5 个子话题""通过了拼写检查"。

在 Loop Engineering 中，最关键的工程动作不是"执行"，而是**把目标的判断条件写清楚**。因为：

- 完成条件定义了 Loop 的**退出机制**
- 完成条件是每一轮迭代后**评估是否继续**的判定依据
- 完成条件的粒度直接决定了 Loop 的**收敛速度**

##### 要素三：交付形态约束（Delivery Form）—— 产出物的形状

第三个容易被忽略但至关重要的要素是：**你想要什么形态的产出？** 同样的目标和判断条件，不同的交付形态会导致完全不同的执行路径：

| 交付形态 | 执行差异 | 典型场景 |
|---|---|---|
| 报告（Report） | 需要结构化分析、数据汇总、结论提炼 | 调研分析、竞品对比 |
| 文章（Article） | 需要叙事逻辑、读者视角、可读性优化 | 技术博客、产品文档 |
| 代码（Code） | 需要语法正确、可运行、通过测试 | 功能实现、Bug 修复 |
| 混合产出 | 需要多阶段编排，产物间有依赖关系 | 项目脚手架搭建 |

为什么交付形态必须提前约束？因为 Agent 在 Loop 执行中会自动选择工具链和执行策略。如果不约束交付形态，Agent 可能用"写文章"的方式去完成"写代码"的目标——结果是产出了一份代码说明文档而不是可运行的代码。这个约束本质上是在告诉 Loop 系统：**你最终的输出物应该是什么形状**。

#### 🔀 对比辨析：对话框目标 vs 边栏目标

| 维度 | ❌ 普通对话框输入 | ✅ 边栏锚定（Codex 模式） |
|---|---|---|
| 可见性 | 一次性，滚动后消失 | 常驻可见，始终参照 |
| 评判时机 | 依赖 Agent "记住"目标 | 每轮迭代可主动回读 |
| 偏离风险 | 高——长时间 Loop 容易漂移 | 低——锚点始终存在 |
| 工程含义 | 目标是"启动参数" | 目标是"持续约束" |

> ❓ **常见疑问**：为什么不能直接把目标写在 System Prompt 里？
>
> System Prompt 确实是承载目标的一种方式，但它的缺陷在于：LLM 的注意力在长上下文窗口下会逐渐衰减，System Prompt 中的目标信息在几十轮迭代后可能被"淹没"。而边栏锚定的设计提供了**显式的、可重复访问的目标存储位置**，每轮迭代都可以主动从边栏拉取目标，确保不偏离。

#### 💻 示意代码：三大要素的结构化表达

```python
# 文件名: loop_config.py
# 功能: Loop Engineering 三大核心要素的结构化配置

from dataclasses import dataclass
from typing import Literal

@dataclass
class Goal:
    """要素一：目标 —— 定义 Loop 的追逐方向"""
    description: str          # 目标描述，如"实现一个用户登录模块"
    priority: int = 1         # 优先级，多目标时用于排序

@dataclass
class CompletionCriteria:
    """要素二：完成判断条件 —— 定义 Loop 的退出标准"""
    conditions: list[str]     # 可验证的判断条件列表
    # 如: ["所有 API 返回 200", "代码通过 pylint 检查", "单元测试覆盖率 > 80%"]
    check_mode: Literal["all", "any"] = "all"  # all=所有条件满足才退出, any=任一满足即退出

@dataclass
class DeliveryForm:
    """要素三：交付形态约束 —— 定义产出物的形状"""
    form_type: Literal["report", "article", "code", "mixed"]  # 交付类型
    output_path: str          # 产出物存放路径
    template: str = ""        # 可选：产出物模板

@dataclass
class LoopConfig:
    """Loop Engineering 的完整输入配置"""
    goal: Goal
    criteria: CompletionCriteria
    delivery: DeliveryForm

# 使用示例：一个典型的代码生成 Loop 配置
config = LoopConfig(
    goal=Goal(description="实现用户注册 API 接口"),
    criteria=CompletionCriteria(
        conditions=[
            "接口 POST /register 返回 201",
            "用户名和邮箱格式校验通过",
            "密码经 bcrypt 加密存入数据库",
            "单元测试覆盖所有分支",
        ]
    ),
    delivery=DeliveryForm(
        form_type="code",
        output_path="./src/api/auth/register.py",
    )
)
```

> ⚠️ **注意事项**：三大要素不是"建议"而是**必要条件**。如果在工程实践中发现 Loop 不收敛或产出不符合预期，首先检查这三个要素是否被明确定义、是否粒度合适。

### 10.2 任务板驱动架构（Task Board）

#### 🧠 直观理解

如果把 Loop Engineering 比作一个建筑工地：三大核心要素是**工程蓝图**，而 Task Board 就是工地上的**施工看板**。看板上挂满了任务卡片，记录着每条任务的当前状态、执行历史、遇到的问题。工头（Agent）每完成一步，就在看板上更新状态，然后检查蓝图上的质量标准——合格就挂上下一步的卡片，不合格就继续修。

#### 📖 详细解释

Task Board（任务板）是 Loop Engineering 架构中最关键的**中间层结构**。它不是简单的 TODO 列表，而是一个**持久化的、可查询的工程状态机**。它在整个 Loop 生命周期中承担信息管理中枢的角色。

##### 七个外围行为与三类输出结果

在整个架构中，环绕在 Task Board 周围的是 **7 个核心行为**和 **1 个最终输出**：

| 编号 | 行为 | 类型 | 说明 |
|---|---|---|---|
| [1] | 目标输入 | 初始化 | 将 Goal、Criteria、Form 三大要素完整注入系统 |
| [2] | 读取 Task Board 当前状态 | 感知 | 查询每个任务卡片的执行状态、产物、错误记录 |
| [3] | 选择任务卡片 | 决策 | 根据状态选取一个"待执行"或"需重试"的任务 |
| [4] | 执行任务事项 | 执行 | 逐步推进任务，过程中记录环境反馈、产物、错误 |
| [5] | 回写 Task Board | 更新 | 将本轮执行产生的所有新信息写回对应卡片 |
| [6] | 按成功标准检查 | 评估 | 对照 Completion Criteria 判定任务是否完成 |
| [7] | 更新信息 + 重新选择 | 迭代 | 未完成时整理本轮信息，重新进入 [2]→[3] |
| [8] | 输出最终结果 | 终止 | 所有条件满足后，按 Delivery Form 输出最终产物 |

这 7 个行为构成一个**完整的闭环**：从初始化开始，经历感知-决策-执行-更新-评估的完整链路，未完成则回到感知阶段继续循环，完成则进入终止阶段。

##### Task Board 的核心角色

> 💡 **核心洞见**：Task Board 不是"记录本"而是"驱动引擎"。它不被动地存储信息，而是**主动参与每一轮迭代的决策**——Agent 每一轮都必须先从 Task Board 读取状态，再决定下一步动作。这意味着 Task Board 的状态质量直接决定了 Loop 的执行质量。

Task Board 同时维护两个维度的状态：

1. **工程状态（Engineering State）**：任务的结构化分解、卡片间的依赖关系、整体进度百分比——回答"项目做到哪了"。
2. **执行状态（Execution State）**：每张卡片的最新产物、错误列表、环境反馈日志——回答"这一步做得怎么样"。

这种双层状态管理使得 Agent 既能把握宏观进度，也能精准定位微观问题。

### 10.3 收敛性问题：从 React Loop 到 Task Board

#### 🧠 直观理解

一个没有收敛机制的 Loop，就像一个没有刹车的高速列车——它能跑起来，但永远停不下来。React Loop 的经典困境正是如此：Agent 在循环中不断执行，但没有可靠的机制告诉它"你已经完成了，可以停了"。

#### 📖 详细解释

> 🔗 **推导链：React Loop 为什么不收敛**

```
起点: React Loop 启动 —— Agent 接收一个模糊的目标
  │
  ├── 第1步: Agent 执行第一轮动作（如搜索、阅读、写作）
  │   为什么？React 架构的本质是 Thought→Action→Observation 循环
  │
  ├── 第2步: Agent 观察到结果后继续下一轮
  │   为什么？Agent 自动判断"还需要继续"，但缺乏明确的退出标准
  │
  ├── 第3步: Loop 持续执行 10 轮、20 轮、50 轮……
  │   为什么？没有外部结构告诉 Agent "这个已经足够好了"
  │   每一轮 Agent 都可能发现"还可以继续优化"，于是继续往前
  │
  ├── 第4步: 用户观察到 Agent 陷入无限循环
  │   为什么？Agent 在"完成"和"还可以更好"之间无法区分边界
  │
  └── 结论: React Loop 的收敛/发散完全依赖模型自身的判断力，
            缺乏工程级别的收敛控制机制
```

老师在课程中特别提到了一个实际场景：在做 GPT 复刻版或 React Loop 小项目时，最常见的问题就是"**不收敛**"——"这个循环会一直往前走，一直往前走，一直往前走"。这不是模型能力的问题，而是**架构设计的缺失**。

#### 🔀 对比辨析：React Loop vs Loop Engineering（Task Board）

| 维度 | ❌ React Loop（无 Task Board） | ✅ Loop Engineering（有 Task Board） |
|---|---|---|
| 状态管理 | 隐式——依赖模型自身的"记忆" | 显式——Task Board 持久化记录 |
| 收敛判定 | 模型自主判断，不可靠 | Completion Criteria 明确可验证 |
| 进度可见 | 用户只能看到最后一轮输出 | 每张卡片的状态变更可追溯 |
| 错误恢复 | 模型自己"想"怎么修 | 错误记录回写 → 下一轮针对性修复 |
| 多任务编排 | 模型自行记忆依赖关系 | Task Board 显式维护卡片依赖 |
| 退出机制 | 模型判断"可以了" 或 token 耗尽 | Criteria 条件全部满足 → 强制退出 |

Task Board 解决收敛性问题的核心逻辑是：**将"是否完成"的判断权从 Agent 的内部推理中剥离出来，交给一个外部可验证的结构化标准**。Agent 不需要自己去"感觉"是否完成——它只需要对照 Completion Criteria 逐条检查即可。

> ❓ **常见疑问**：Task Board 会不会本身也导致无限循环？比如 Criteria 写得不好，永远达不到？
>
> 这是一个合理的担忧。解决方案有两个层面：
> 1. **Criteria 设计规范**：每条判断条件必须是**客观可验证**的（如"代码通过 lint 检查"而非"代码写得不错"）
> 2. **超时/轮次上限**：即使 Criteria 设计有缺陷，也可以设置最大迭代轮次作为兜底保护——这不是 Task Board 的职责，而是 Harness 层的安全网（详见后续模块）

#### 💻 示意代码：收敛检查逻辑

```python
# 文件名: convergence_check.py
# 功能: 展示 Task Board 如何实现收敛判断

from dataclasses import dataclass, field

@dataclass
class TaskCard:
    """Task Board 上的单张任务卡片"""
    task_id: str                    # 卡片唯一标识
    description: str                # 任务描述
    status: str = "待执行"          # 当前状态: 待执行/进行中/已完成
    artifacts: list[str] = field(default_factory=list)  # 产生的中间产物
    errors: list[str] = field(default_factory=list)     # 记录的错误信息
    feedback: list[str] = field(default_factory=list)   # 环境反馈

@dataclass
class TaskBoard:
    """任务板 —— Loop Engineering 的核心状态管理结构"""
    cards: list[TaskCard]                                # 所有任务卡片
    criteria: list[str] = field(default_factory=list)    # 完成判断条件列表

    def get_pending_card(self) -> TaskCard | None:
        """[2] 读取状态 → [3] 选择任务: 找到第一个待执行或需重试的卡片"""
        for card in self.cards:
            if card.status in ("待执行", "进行中"):
                return card
        return None

    def update_card(self, task_id: str, **kwargs) -> None:
        """[5] 回写 Task Board: 将本轮执行信息写回卡片"""
        for card in self.cards:
            if card.task_id == task_id:
                for key, value in kwargs.items():
                    setattr(card, key, value)

    def check_convergence(self) -> bool:
        """[6] 按成功标准检查: 所有卡片是否都满足完成条件？
        
        这就是 Task Board 解决收敛性问题的关键方法：
        - 不再依赖 Agent 的主观判断
        - 而是对所有卡片进行客观的结构化验证
        """
        # 条件1: 所有卡片状态必须是"已完成"
        all_completed = all(card.status == "已完成" for card in self.cards)
        
        # 条件2: 每条 Criteria 必须被满足
        # (实际工程中会逐条验证，此处为示意)
        all_criteria_met = len(self.criteria) > 0  # 简化示意
        
        return all_completed and all_criteria_met


# === 收敛性对比: 无 Task Board vs 有 Task Board ===

# ❌ 无 Task Board 的 React Loop —— 收敛依赖模型自身
def react_loop_without_board(agent, goal: str, max_rounds: int = 50):
    """问题: 模型自己决定何时停止，缺乏可靠的退出标准"""
    for round_num in range(max_rounds):
        result = agent.think_and_act(goal)
        # 模型返回 "我认为完成了" —— 但真的完成了吗？
        # 没有客观标准可验证！
        if result.is_finished:
            break
    return result

# ✅ 有 Task Board 的 Loop Engineering —— 收敛依赖结构化标准
def loop_engineering_with_board(agent, board: TaskBoard):
    """改进: Task Board 提供可验证的收敛判断"""
    max_rounds = 50
    for round_num in range(max_rounds):
        # [2] 读取当前状态
        card = board.get_pending_card()
        if card is None:
            break  # 没有待处理任务
        
        # [3][4] 执行任务
        result = agent.execute_task(card)
        
        # [5] 回写 Task Board
        board.update_card(
            card.task_id,
            status=result.status,
            artifacts=result.artifacts,
            errors=result.errors,
        )
        
        # [6] 关键: 按成功标准检查（客观、可验证）
        if board.check_convergence():
            print("✅ 所有 Criteria 满足，Loop 收敛，退出循环")
            break  # 真正可靠的退出！
```

> ⚠️ **注意事项**：`check_convergence()` 的实现不能是简单的"所有卡片状态为已完成"。在真实工程中，每条 Criteria 都需要**可程序化验证**——例如运行 lint 工具检查代码质量、执行测试套件验证功能正确性。Task Board 的设计核心是将主观判断转化为客观验证。

### 10.4 Loop Engineering 与 Harness Engineering 的关系

#### 🧠 直观理解

Loop Engineering 定义了"**车该怎么开**"——路线规划、驾驶规则、到站判断。Harness Engineering 提供了"**路和基础设施**"——平坦的路面、加油站、交通信号灯。车没有路开不了，路没有车也没有意义。两者不是替代关系，而是**上下层支撑关系**。

#### 📖 详细解释

老师在课程中明确指出：**Loop Engineering 不是替代（replace）Harness Engineering**，新词不是用来代替旧词的。两者的关系是一个明确的分层：

- **Loop Engineering**：关注 Loop 本身的架构设计——目标如何定义、Task Board 如何驱动、收敛如何保证。它回答的是"**一个能收敛的自主循环应该长什么样**"。
- **Harness Engineering**：关注让 Loop 能够运行的**底层基础设施**——工具调用、沙箱环境、文件系统、权限控制、轮次上限、错误兜底。它回答的是"**现有的方案凭什么能成立**"。

#### 🔀 对比辨析

| 维度 | Loop Engineering | Harness Engineering |
|---|---|---|
| 关注层面 | 架构与流程设计 | 基础设施与支撑能力 |
| 核心问题 | 如何定义目标、如何收敛 | 如何安全执行、如何管理资源 |
| 关键产物 | Task Board、Criteria、收敛逻辑 | 沙箱、工具链、权限系统、限流器 |
| 类比角色 | 工程总监——设计施工方案 | 包工头——提供工具和场地 |
| 两者关系 | 定义"做什么、怎么判断做完" | 提供"在哪做、用什么做" |

> 💡 **核心洞见**：Harness Engineering 是 Loop Engineering 的**存在前提**。没有 Harness 提供的工具调用能力和安全执行环境，Task Board 再完美也无法落地。反过来，只有 Harness 而没有 Loop 的架构设计，系统就只能执行单步任务，无法形成持续收敛的自主循环。

用更直白的类比：**Loop Engineering 是 AI Agent 的"大脑"——决定做什么、何时停；Harness Engineering 是 AI Agent 的"身体"——提供手脚（工具）、感官（反馈通道）和生存环境（沙箱）**。两者缺一不可。

### 11.1 Harness 六大支撑模块总览

#### 🧠 直观理解

如果把 Loop Engineering（Task Board 驱动的任务循环）比作一辆在路上行驶的汽车，那么 Harness 六大支撑模块就是这辆车底盘上的六个核心子系统——发动机、变速箱、制动、转向、悬挂、电气。车能跑起来，不是因为有方向盘，而是因为方向盘后面的转向机构在工作。Loop 能运转，同样不是因为有 Task Board，而是因为 Task Board 之下有六大支撑模块在默默提供基础能力。

#### 📖 详细解释

在上一节（M10）中，我们建立了 Task Board 架构的心智模型：系统从任务板中读取状态和已有证据，选择一个任务卡，执行，回写结果，然后进入下一轮循环。这个描述听起来很顺畅，但每一个环节落地的背后，都对应着一个或多个 Harness 模块的支持：

```
                    ┌───────────────────────────────┐
                    │       Loop Engineering         │
                    │    (Task Board 驱动的回环)       │
                    └──────────────┬────────────────┘
                                   │
            ┌──────────────────────┼──────────────────────┐
            │                      │                      │
            ▼                      ▼                      ▼
   ┌────────────────┐   ┌────────────────┐   ┌────────────────┐
   │  读取任务板状态  │   │  选择 & 执行    │   │  回写状态更新    │
   └───────┬────────┘   └───────┬────────┘   └───────┬────────┘
           │                    │                    │
           └────────────────────┼────────────────────┘
                                │
              ┌─────────────────┴─────────────────┐
              │         六大 Harness 支撑模块        │
              ├─────────────────────────────────────┤
              │  ① 结构化请求与上下文                  │
              │  ② 复杂任务调度                       │
              │  ③ 工具/行动接口 + 沙盒                │
              │  ④ Workspace（工作空间/上下文存储）      │
              │  ⑤ 权限与预算控制                      │
              │  ⑥ Trace & Evaluate（追踪与评估）      │
              └─────────────────────────────────────┘
```

老师的原话是：「没有这个底层的一个基座，我们是不能去做出上面这个东西的。」这些模块在范畴上属于 **Harness Engineering**——它们是让你的 Agent 系统具备最基础支撑能力的工程模块，而非 Loop 本身。

> 💡 **核心洞见**：六大模块存在了，不代表 Loop 就自动成立。Loop 和模块之间是「支撑和被支撑」的关系。Loop 是在这些模块都已经就绪的前提之下，在上一层构建的长程任务管理环。

接下来，我们逐一展开这六个模块，理解它们各自解决什么问题、如何支撑 Loop 运行。

### 11.2 模块一：结构化请求与上下文管理

#### 🧠 直观理解

你让助手去超市买三样东西，如果你说「帮我买点菜」，助手可能买回来一堆你不想要的。但如果你给出一张购物清单，每项写清楚品名、数量、品牌偏好，执行结果就可靠得多。结构化请求就是这张「购物清单」——它确保模型和系统之间用确定性的格式而非模糊的自然语言来交换信息。

#### 📖 详细解释

这是六大模块中最基础的一个。模型的 Function Calling 能力要求系统发出的指令具备**结构化的清晰表达**——参数名、参数类型、必选/可选、返回值格式。如果指令结构混乱或存在歧义，模型输出将不可靠，整个 Loop 的第一步就会出错。

在 Loop Engineering 的语境下，这个模块的挑战不止于单次 Function Calling。Task Board 本身是一个信息管理的核心节点：

1. **Task Board 中的任务列表需要序列化**：每个任务卡不能只是一个自然语言句子（虽然它最初可能是自然语言生成的），它需要被组织成列表结构——每一项有 ID、状态、优先级、依赖关系等结构化字段。这样模型才能精确地「选择一个任务卡」而不是「大概挑一个」。

2. **状态更新需要反序列化**：模型执行完一个任务后，它生成的「回写」内容必须能被系统解析。如果模型返回「我觉得这个任务差不多完成了」，系统无法可靠地更新任务板。但如果模型返回 `{"task_id": "T-12", "new_status": "completed", "output_summary": "..."}`，系统可以精确更新。

3. **高可用性是前提**：老师特别强调「结构化请求的高可用性本身就是一个最基础的支撑」。如果模型在某一次调用中输出的结构不符合预期，整个 Loop 就可能断裂。这意味着系统需要对模型的输出做校验、容错、重试——而这些都是结构化请求管理模块的职责。

#### 💻 代码示例

```python
# 文件名: structured_request_example.py
# 功能: 展示 Task Board 中结构化任务项的序列化与反序列化

from dataclasses import dataclass, asdict
from enum import Enum
from typing import Optional
import json

class TaskStatus(Enum):
    PENDING = "pending"
    IN_PROGRESS = "in_progress"
    COMPLETED = "completed"
    FAILED = "failed"

@dataclass
class TaskCard:
    """Task Board 中每个任务卡的结构化表达"""
    task_id: str          # 唯一标识
    title: str            # 任务标题（自然语言）
    status: TaskStatus    # 当前状态
    priority: int         # 优先级（数字越大越优先）
    depends_on: list[str] # 依赖的其他任务 ID 列表
    evidence: str         # 执行该任务前收集到的证据/上下文

    def to_dict(self) -> dict:
        """序列化：将任务卡转为模型可消费的结构化数据"""
        return {
            "task_id": self.task_id,
            "title": self.title,
            "status": self.status.value,
            "priority": self.priority,
            "depends_on": self.depends_on,
            "evidence": self.evidence,
        }

    @classmethod
    def from_dict(cls, data: dict) -> "TaskCard":
        """反序列化：从结构化数据还原任务卡对象"""
        return cls(
            task_id=data["task_id"],
            title=data["title"],
            status=TaskStatus(data["status"]),
            priority=data["priority"],
            depends_on=data.get("depends_on", []),
            evidence=data.get("evidence", ""),
        )

# 模型回写的状态更新也需要结构化表达
@dataclass
class TaskUpdate:
    """模型执行完成后回写的结构化状态更新"""
    task_id: str
    new_status: TaskStatus
    output_summary: str          # 执行结果的文本摘要
    artifacts: list[str]         # 产出的文件/数据引用
    next_hint: Optional[str] = None  # 对后续任务的提示
```

| 行号 | 代码 | 解释 |
|---|---|---|
| 7-11 | `class TaskStatus(Enum)` | 任务状态使用枚举而非字符串常量，避免模型输出不可预期的状态值 |
| 13-21 | `class TaskCard` | 每个任务卡必须包含 ID、状态、优先级、依赖等结构化字段，不能只是一个自然语言句子 |
| 23-32 | `to_dict()` | 序列化方法确保交给模型的数据格式一致、可校验 |
| 34-42 | `from_dict()` | 反序列化方法将模型输出或存储数据恢复为对象，中间可加入校验逻辑 |
| 45-51 | `class TaskUpdate` | 模型回写的信息也必须结构化——系统需要精确更新任务板，不能依赖自然语言推断 |

> 🔗 **关联知识**：这个模块的思想与 Function Calling 的 JSON Schema 定义一脉相承。如果你之前接触过 OpenAI 的 function calling 或 MCP 协议中的 tool 定义，这里的结构化请求管理就是将这些定义从单次调用扩展到整个 Task Board 的生命周期管理。

### 11.3 模块二：复杂任务调度

#### 🧠 直观理解

你手上有十件事要做，但你不能同时做十件事。你需要判断先做什么、后做什么、哪件事做完才能做下一件——这就是调度。复杂任务调度模块就是 Agent 系统中的「任务分配大脑」，它把 Task Board 上一堆杂乱的任务项变成一个有可执行秩序的序列。

#### 📖 详细解释

老师用一句话概括了这个模块的核心：「复杂任务调度让我们的状态变成了一个可执行的秩序。」这句话蕴含了两层意思：

**第一层：状态到秩序的转换。** Task Board 维护的是一组任务的状态快照——哪些 pending、哪些 completed、哪些被阻塞。但「状态」本身不等于「下一步做什么」。调度模块需要根据状态信息做出决策：当前可执行的任务有哪些？它们的优先级如何排序？有没有依赖未满足的任务需要等待？这个决策过程从「状态描述」跨越到了「执行指令」。

**第二层：秩序必须是可执行的。** 「可执行」这个词很关键。调度结果不能只是一个建议——它必须是一个能被下游执行模块直接消费的指令。这意味着调度的输出也需要结构化（回到模块一），比如输出一个 `next_task_id` 和一个执行上下文，而不是一段「我觉得接下来可以做那个关于数据清洗的任务」的自然语言。

在 Loop Engineering 的架构中，调度模块处于 Task Board 和执行模块之间：读取状态 -> 评估可行性 -> 选择下一个任务 -> 下发执行指令。它是 Loop 中「选择下一步做什么」这个决策环节的工程实现。

#### 🔀 对比辨析

| 维度 | 简单任务调度（单次 API 调用） | Harness 级复杂任务调度 |
|---|---|---|
| 决策范围 | 单次调用内决定 tool 调用顺序 | 跨多轮 Loop 管理 Task Board 上的任务队列 |
| 状态感知 | 仅感知当前轮次上下文 | 感知全局任务板状态、历史执行结果、依赖关系 |
| 中断恢复 | 不支持或简陋 | 支持中断、暂停、依赖阻塞等待、人工介入后恢复 |
| 并发管理 | 不支持 | 可管理多个独立子任务并行执行 |
| 优先级策略 | 固定或无 | 可配置优先级规则、资源约束、截止时间 |

### 11.4 模块三：工具/行动接口 + 沙盒

#### 🧠 直观理解

模型说「我去查一下数据库」，但它不能自己动手操作数据库——它只能生成一个意图。工具接口就是这个「手」，它把模型的意图翻译成对真实环境的操作。沙盒则是给这个操作加了一个「安全围栏」——在围栏里你怎么折腾都行，但不会伤到外面的生产环境。

#### 📖 详细解释

老师将这个模块概括为：「工具让模型意图可以可靠地落到环境。」注意这里的用词——**可靠地**。模型产生意图（「我要读取这个文件」「我要执行这段代码」）只是第一步，真正困难的是确保这个意图在执行时：

1. **被正确翻译**：模型输出的参数（文件路径、命令参数）被准确映射到实际 API 调用或系统命令。
2. **被安全隔离**：沙盒机制确保模型的任何操作都在受控环境中执行。如果模型生成了一段有问题的代码，沙盒确保它不会影响宿主系统。
3. **有明确的反馈**：执行结果（成功/失败、输出内容、副作用）被结构化地回传给模型，构成下一轮决策的依据。

这个模块由两部分组成：

**工具/行动接口**：定义了模型可以调用哪些外部能力。类似于 MCP（Model Context Protocol）中的 tool 定义，但在 Harness 层面，它不仅是「定义一个 tool schema」，还包括 tool 的注册、发现、版本管理、执行生命周期管理。

**沙盒**：为工具执行提供隔离环境。在企业级场景中，沙盒不是可选的——它是安全底线。沙盒需要管理文件系统的隔离、网络访问的控制、资源限制（CPU/内存/时间）、执行超时等。

> ⚠️ **注意事项**：沙盒不是「跑代码的地方」那么简单。老师强调它要能「完成对应的工作」。如果你的 Agent 需要操作数据库，沙盒里就得有可访问的数据库实例；如果需要运行数据分析，沙盒里就得有对应的运行时环境。沙盒的能力边界决定了 Agent 系统的能力上限。

### 11.5 模块四：Workspace（工作空间/上下文存储）

#### 🧠 直观理解

你和朋友聊天，每次你说一句话，他只能记住你这「一句话」，之前聊过的全部忘了——这样的对话效率极低。Workspace 就是 Agent 的「记忆笔记本」，它让模型在不同的请求轮次之间能够记住之前发生过什么、产生过什么、当前的上下文是什么。

#### 📖 详细解释

老师对这个模块的表述非常精炼：「Workspace 让事实能够离开事实的上下文。」这句话需要拆开理解。

在传统的单次 LLM 调用中，所有的「事实」都存在于当次请求的上下文中——你把相关文档拼进 prompt，模型在上下文中处理。一旦请求结束，这些事实就消失了。下一次调用时，你必须重新把这些事实塞回上下文。

Workspace 模块改变了这个范式：

1. **事实持久化**：在一次任务的执行过程中产生的信息——代码执行结果、文件内容、数据查询结果、推理中间产物——被写入 Workspace 而非仅停留在当次请求的上下文中。

2. **跨轮次共享**：后续的任务卡在执行时，不需要在上游任务的自然语言描述中重新叙述所有历史信息。调度模块可以直接从 Workspace 中检索相关事实，组装到当前执行任务的上下文中。

3. **独立存储空间**：Workspace 是一个独立于模型调用上下文的外部存储。它有自己的命名空间、文件结构、访问控制。在 Task Board 架构中，Workspace 和 Task Board 是协作关系——Task Board 管理「要做什么」，Workspace 管理「已经知道什么和已经产出什么」。

```
┌─────────────────────────────────────────────────────┐
│                     Loop 回环                        │
│                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐       │
│  │ Task C   │───▶│ Task D   │───▶│ Task E   │  ...  │
│  └────┬─────┘    └────┬─────┘    └────┬─────┘       │
│       │               │               │              │
│       │  写入产物      │  读取上下文    │  写入新产物    │
│       ▼               ▲               ▼              │
│  ┌────────────────────────────────────────────────┐  │
│  │              Workspace（独立存储）                │  │
│  │  /outputs/task_c_result.json                    │  │
│  │  /data/cleaned_dataset.csv                      │  │
│  │  /evidence/user_intent_analysis.md              │  │
│  │  /context/session_state.yaml                    │  │
│  └────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

> ❓ **常见疑问**：「Workspace 和向量数据库/RAG 有什么区别？」
>
> Workspace 管理的是**当前任务的运行时上下文**（session-scoped），是任务执行过程中产生的实时数据和状态。RAG/向量数据库管理的是**长期知识库**（跨 session），是预先索引好的静态知识。两者的生命周期和用途不同——Workspace 随任务开始而创建、随任务结束而归档或销毁。

### 11.6 模块五：权限与预算控制

#### 🧠 直观理解

你请了一个实习生帮你处理工作，你会给他门禁卡（权限），也会告诉他「今天最多花 500 块预算」。Agent 系统也一样：它需要知道什么能做、什么不能做、做到什么程度该停。权限与预算控制就是 Agent 的「门禁卡 + 花费上限」。

#### 📖 详细解释

老师将这一模块的核心概括为：「权限预算让持续行动仍然可以被受控。」关键词是**持续行动**和**受控**——这两个词放在一起揭示了一个张力：Agent 系统的价值在于它能够自主地持续行动（Loop 的本质），但自主性越高，失控风险越大。权限与预算控制就是在这个张力中建立平衡。

这个模块包含两个核心能力：

**权限控制**：
- 定义 Agent 在系统中可以访问哪些资源（文件、API、数据库、网络）。
- 控制不同任务卡可以获取的权限级别（不是所有任务都需要最高权限）。
- 支持运行时权限校验——每次工具调用前检查权限，而非仅启动时检查一次。

**预算控制**：
- Token 预算：一次 Loop 任务最多消耗多少 token，防止模型失控消耗。
- 时间预算：一个任务最多执行多长时间，超时自动中断。
- 资金预算：如果涉及外部付费 API 或资源，设定花费上限。
- 步数预算：Loop 最多执行多少轮，防止无限循环。

**Human-in-the-Loop 中断**：老师特别提到「能不能支持我们在执行过程中间去进行中断，等待外界的回复，让 human in the loop 成为可行」。这意味着权限模块不仅要做「允许/拒绝」的判定，还要支持「暂停执行 -> 等待人工审批 -> 继续或终止」的工作流。比如 Agent 在执行过程中遇到了一个超出其权限范围的操作（删除生产数据库表），系统应该中断执行，推送审批请求给人类操作者，等待确认后再继续。

> 💡 **核心洞见**：权限与预算控制不是「限制 Agent 能力」的枷锁，而是「让 Agent 可以放心持续运行」的保障。正因为有了这些控制机制，我们才敢于让 Agent 进行长程自主任务——因为我们知道它不会越界、不会失控、不会无限消耗资源。

### 11.7 模块六：Trace & Evaluate（追踪与评估）

#### 🧠 直观理解

如果你的 Agent 系统完成了 100 个任务，你如何知道它做得对不对？哪些步骤出了问题？如何改进？Trace 就是 Agent 的「行车记录仪」——全程记录每一步发生了什么。Evaluate 就是「驾驶评估系统」——根据记录判断这次行程的优劣，并给出改进建议。

#### 📖 详细解释

老师在讲到这个模块时，特别做了一个区分：

> 「trace 这个是我们现在如果你去用很多的 Harness 工程的产品，这个东西对于你来讲可能是比较远的。但是我们如果真在企业内部想要去做一个类似的 Harness 工程，去支撑你企业内部有一个完整的智能系统的时候，你就会需要自己去完成 trace 和 evaluate。」

这个区分非常重要。它揭示了 Trace & Evaluate 模块的一个关键特性：**使用产品时它被封装好了，你感知不到；自建系统时它是必须品，没有它系统就是黑盒。**

**Trace（追踪/日志）** 的职责：
- 记录每一次模型调用的输入（system prompt、user prompt、上下文组装过程）和输出（原始响应、tool calls）。
- 记录每一次工具执行的参数、返回结果、耗时、错误信息。
- 记录 Task Board 的状态变更历史——哪个任务在什么时候从什么状态变成了什么状态。
- 记录 Loop 的决策过程——为什么在这一轮选择了任务 A 而不是任务 B。

**Evaluate（评估）** 的职责：
- 基于 Trace 数据对任务执行质量进行评估：任务是否真的完成了？输出质量如何？有没有遗漏？
- 识别执行过程中的异常模式：反复重试同一任务、执行路径偏离预期、输出质量下降。
- 为系统改进提供数据支撑：哪些 prompt 需要调整？哪些工具定义不清晰？哪些任务的执行策略需要优化？

```
Trace 数据流：
                                  ┌─────────────────────┐
  Model Call ──▶ Trace ──▶        │  Trace Storage       │
  Tool Exec  ──▶ Trace ──▶        │  (日志存储)          │
  State Change──▶ Trace ──▶       │                      │
  Decision    ──▶ Trace ──▶       └──────────┬──────────┘
                                             │
                                             ▼
                                  ┌─────────────────────┐
                                  │  Evaluate Engine     │
                                  │  (评估引擎)          │
                                  │                      │
                                  │  · 任务完成度评估     │
                                  │  · 路径偏离检测       │
                                  │  · 质量打分           │
                                  │  · 改进建议生成       │
                                  └─────────────────────┘
```

> ⚠️ **注意事项**：在产品的使用体验中，Trace & Evaluate 被产品层封装并提供了可视化面板，你只需要看面板上的指标。但在自建系统中，你需要自己设计 Trace 的数据结构、存储方案、评估逻辑。一个没有 Trace 的自建 Agent 系统在出现问题时，你将完全无法定位根因——你只能看到「任务失败了」，但不知道为什么失败。

### 11.8 Loop 与 Prompt 的关系

#### 🧠 直观理解

一个常见的误解是：「有了 Loop 就不需要 Prompt Engineering 了。」事实恰恰相反——Loop 没有消灭 Prompt，而是改变了 Prompt 的位置和形式。

#### 📖 详细解释

老师在录音中回应了学员的提问：「所以有了 loop 还需要 prompt 吗？」他的回答揭示了一个关键洞察：

**Loop 简化了用户侧的 Prompt。** 在没有 Loop 的系统中，用户需要在一个 prompt 里写清楚所有要求：要做什么、怎么做、遇到什么情况如何处理、输出的格式是什么。这个 prompt 又长又难写。有了 Loop 之后，用户只需要用自然语言描述目标，系统通过 Task Board 自动拆解任务、逐步执行——用户端的输入被大幅简化。

**但系统内部的 Prompt 管理依然存在，而且更复杂。** Loop 的每一次循环都涉及模型调用：
- 任务选择时：需要 prompt 来驱动模型从 Task Board 中选择下一步任务。
- 任务执行时：需要 prompt 来驱动模型调用工具、处理结果。
- 状态回写时：需要 prompt 来驱动模型输出结构化状态更新。
- 汇总输出时：需要 prompt 来驱动模型整合所有执行结果。

**Prompt Engineering 没有消失，而是「下沉」到了系统内部。** 以前是你手写一个长 prompt 给模型，现在是系统设计者在 Harness 层面管理一组精心设计的 prompt 模板，每个模板服务于 Loop 中的一个特定环节。这些系统级 prompt 的质量直接影响整个 Agent 系统的可靠性——它从「用户技能」变成了「系统工程能力」。

> 💡 **核心洞见**：Loop 工程解决的是「如何组织复杂任务」，而不是「如何让模型理解任务」。模型理解任务仍然依赖 Prompt——只是 Prompt 的位置从用户的输入框移到了系统的工程层。

### 11.9 使用者 vs 构建者

#### 🧠 直观理解

去餐厅吃饭的人不需要知道厨房里的灶台怎么点火、食材怎么切配——他只管点菜和吃。但如果你想在自己家开一个餐厅，你就需要知道厨房里的每一件设备怎么用。Agent 系统的世界同理：**用**和**建**是两个完全不同的角色，需要完全不同的知识储备。

#### 📖 详细解释

老师在录音中用了相当长的篇幅来区分这两个角色，这是整节课程的实践指导核心：

**使用者（User）**：
- 你使用的是别人已经构建好的 Harness 产品，如 Open Cloud、Codex 等。
- 你只需要关心输入什么、得到什么输出——底层的 Task Board、调度器、沙盒、Trace 都是产品内部的事。
- 你的技能重点是：如何写出好的指令、如何评估输出质量、如何在产品限制内最大化效用。
- **不需要**理解六大模块的内部实现。

**构建者（Builder）**：
- 你要在自己的系统中拥有类似上述产品的智能体能力。
- 你的需求可能是全量的（「我要一个完整的 Agent Loop 系统」），也可能是局部的（「我只需要一个沙盒执行能力」或「我只需要一个 Skills 执行器」）。
- 无论全量还是局部，你都需要理解六大模块，因为它们是你系统的基础组成单元。
- 你需要自己拼装这些模块——选择或实现结构化请求管理、任务调度器、沙盒、Workspace、权限控制、Trace 系统。

#### 🔀 如何判断自己的定位？

老师的原话给出了一个简洁的判断标准：

> 「如果你说你需要的，是我想要在我的系统里拥有类似上述这样的一个智能体系统——那你就需要把它拆开，你需要理解底下的这些东西。」

```
你面临的选择：
    │
    ├── 你只是想「使用」一个已有的 Agent 产品？
    │   └── 使用者路径 → 不需要学习六大模块的内部实现
    │       → 关注：产品功能、使用技巧、输出质量评估
    │
    ├── 你想在自己的系统里「拥有」Agent 能力（全量）？
    │   └── 构建者路径 → 六大模块需要全部理解
    │       → 关注：每个模块的设计原理、实现方案、模块间协作
    │
    └── 你只需要「局部能力」（沙盒/特定回路/Skills 执行器）？
        └── 构建者路径（局部）→ 聚焦你需要的模块
            → 关注：所需模块的接口定义、与现有系统的集成方式
```

> 💡 **核心洞见**：这个区分不仅影响你学什么，更影响你**学到什么深度**。使用者不需要碰底层，构建者需要把每个模块「拆开理解」。老师强调——「如果你希望你的系统里有，你就需要这些东西」。这不是可选项，而是工程上的硬性要求。

> 🔗 **前置知识**：Harness 六大模块的存在是为了支撑 M10 中建立的 Task Board Loop 架构。如果对 Loop 的核心流程还不熟悉，建议先回顾 Task Board 的读取-选择-执行-回写循环。

> 🔗 **延伸阅读**：关于使用者 vs 构建者的具体学习路径和技能树选择，在 M12 中有更详细的讨论。

### 12.1 答疑：Loop Coding vs Loop Engineering

---

**🙋 问题 1**：使用 CC（Claude Code）或 Codex 这样的工具时，只需要输入任务目标，它们就会自动完成。这背后是不是默认就是在使用 Loop 机制？这种使用方式能不能称为 Loop Engineering？

> 📖 **背景**
>
> 在前面的课程中，我们深入探讨了 Loop Engineering 的概念——如何在完备的 Harness 基础之上，构建能让模型持续追逐目标、完成产出的方案。但很多同学在实际使用 CC、Codex、Cursor 等 AI 编码工具时，发现这些工具本身"输入目标 → 自动完成任务"的行为模式看起来就像是某种 Loop。这引出了一个关键问题：**使用者和构建者对同一概念的理解是否有本质区别？**

**💬 老师回答**

##### 冷知识：Loop Engineering 概念的起源

在展开回答之前，先分享一个有趣的背景故事。Loop Engineering 这个概念最早的公开提及，来自于 Anthropic（Claude 模型背后的公司）的一位工程师在一次外部访谈中的发言。他说自己最近特别沉迷于使用 "goon loop" 这个指令。由此，"Loop" 作为一种让模型持续自主推进任务的方法论，开始进入开发者的视野，并逐渐演化成今天我们所讨论的 Loop Engineering 体系。

##### 核心辨析：Loop Coding 不等于 Loop Engineering

回到问题本身：用 CC 或 Codex 输入一段需求描述，等工具自动完成任务——这确实是 Loop 思想在用户侧的体现，但它应该被称为 **Loop Coding**，而非 Loop Engineering。

这两个概念的关键区别在于**关注层次的差异**：

```
         使用者视角                          构建者视角
    （Loop Coding）                    （Loop Engineering）
          │                                    │
    ┌─────┴─────┐                      ┌──────┴──────┐
    │ 输入需求    │                      │  设计 Harness │
    │ 等待产出    │                      │  编排回路逻辑 │
    │ 验收结果    │                      │  管理状态流转 │
    └───────────┘                      │  处理异常边界 │
                                       │  优化 token 消耗│
                                       └─────────────┘
```

| 维度 | Loop Coding（使用层） | Loop Engineering（构建层） |
|---|---|---|
| **核心动作** | 输入任务描述，获取结果 | 设计并实现持续追目标的系统方案 |
| **关注点** | 任务是否完成、结果是否正确 | 回路逻辑、状态管理、异常处理、工具编排 |
| **前置条件** | 有一个现成的 Agent 工具 | 有完备的 Harness 工程基础 |
| **典型场景** | 用 CC 修 bug、用 Codex 写功能 | 构建能自主规划→执行→验证→修正的智能系统 |
| **能力要求** | 能用自然语言描述需求 | 理解 Agent 架构、回路设计、工程落地 |

> ⚠️ **注意事项**
>
> 不要因为自己能用 CC 完成复杂任务，就认为自己"已经掌握了 Loop Engineering"。这就好比你会开车，不代表你懂得如何设计一辆汽车的发动机控制系统。Loop Engineering 要求你理解的是"为什么要这样设计"以及"如何在工程层面实现这个设计"。

##### Loop Engineering 的真正内涵

我们在课程中讨论的 Loop Engineering，指的是：**在工程基础已经齐备（拥有完备的 Harness）的前提下，如何构建一套方案，让模型能够持续不断地追逐一个目标，最终完成产出。** 这背后涉及的核心问题是：

1. **回路的触发条件**：什么情况下模型应该继续执行？什么情况下应该停止？
2. **状态的传递与管理**：每一轮迭代中，模型需要记住什么？传递什么？
3. **工具的调度策略**：何时调用何种工具？工具调用失败后如何降级？
4. **目标的拆解与追踪**：复杂目标如何分解？子目标完成度如何度量？
5. **异常的处理与恢复**：遇到意外情况时，回路如何自愈？

这些才是 Loop Engineering 真正要解决的问题，而不仅仅是"输入一段话，等模型自动完成"。

> 💡 **延伸思考**
>
> 如果你想从 Loop Coding 的层次提升到 Loop Engineering：
> - 下次使用 CC 时，观察它的思考过程（thinking trace），分析它是如何分解任务、何时决定调用工具、何时判断任务已完成的
> - 尝试回答：如果让你自己实现一个类似 CC 的 Agent，你需要处理哪些边界情况？
> - 对比 CC 和 Cursor 的 Loop 行为有什么不同——为什么会有这些差异？背后的设计取舍是什么？

### 12.2 答疑：Skills 与 Function Calling 的关系

---

**🙋 问题 2**：Skills 到底能不能理解成一种特殊形式的 Function Calling？它们之间是什么关系？

> 📖 **背景**
>
> 在 Harness 工程中，Skills 是一个非常重要的概念。但从表面上看，Skills 的执行过程——用户触发某个 Skill、系统执行一系列操作、返回结果——与 Function Calling（函数调用）的模式非常相似。这让很多同学产生疑问：Skills 是否只是换了个名字的 Function Calling？

**💬 老师回答**

##### 理论上的可能性：React Loop 足够好时可以模拟

从纯理论的角度来看，如果你的 React Loop（即 Reasoning + Acting 的循环）设计得足够好，或者你不太在意 token 消耗的话，你确实**可以**把 Skills 的整个生命周期——包括执行、安装、理解、装载过程——制作成一个不断推进的小回路。从这个意义上说，Skills 的执行过程在理论上可以被模拟为一种特殊的 Function Calling 回路。

##### 实践中的边界：Skills 是 Harness 工程的独立模块

但当我们真正去构建一个 Harness 工程时，会发现 Skills 需要被**单独拆开来处理**，它不能简单地等同于 Function Calling。原因在于：

1. **Skills 的执行逻辑更复杂**
   - Function Calling 通常是一次性的：**输入参数 → 执行函数 → 返回结果**
   - Skills 往往涉及多步骤流程：**加载 Skill 定义 → 理解上下文 → 分步执行 → 中间验证 → 汇总结果**
   - Skills 的执行过程中可能需要**动态决策**：根据中间结果决定下一步做什么，这比单纯的函数调用更复杂

2. **Skills 有独立的生命周期管理**
   - 安装/注册：Skill 需要被 Harness 系统识别和装载
   - 激活/触发：需要特定的条件或意图识别
   - 执行/监控：执行过程中需要状态追踪
   - 卸载/更新：需要支持热更新和版本管理

3. **Skills 需要上下文注入**
   - Function Calling 通常在模型的上下文窗口中传递函数签名
   - Skills 可能需要注入**完整的操作手册、规则集、领域知识**，远超普通函数描述的体量

##### 正确的定位：Skills 是 Harness 工程中的一个理论组成部分

```
┌─────────────────────────────────────────────────┐
│                 Harness 工程                      │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │  编排系统  │  │ Skills   │  │ 工具/Function │  │
│  │(Orchestr) │  │ 系统     │  │ Calling 系统  │  │
│  └─────┬────┘  └─────┬────┘  └──────┬───────┘  │
│        │             │              │           │
│        └──────┬──────┘              │           │
│               │                     │           │
│        ┌──────┴─────────────────────┴──────┐    │
│        │         底层 Loop 引擎              │    │
│        └───────────────────────────────────┘    │
│                                                  │
└─────────────────────────────────────────────────┘
```

Skills 和 Function Calling 都是 Harness 工程中的**并行组件**，而不是包含关系。它们各自解决不同层面的问题：

| 维度 | Skills | Function Calling |
|---|---|---|
| **粒度** | 粗粒度：一个完整的能力单元 | 细粒度：一个具体操作 |
| **状态管理** | 需要跨步骤状态保持 | 通常无状态或轻状态 |
| **上下文需求** | 需要大量领域知识和规则注入 | 仅需函数签名和参数描述 |
| **执行复杂度** | 多步骤、有分支、需决策 | 单次调用、确定性强 |
| **在 Harness 中的角色** | 能力扩展层 | 基础操作层 |

> 💡 **延伸理解**
>
> Skills 更像是一个"能力包"，它内部可能会用到 Function Calling 作为执行手段，但它本身所承载的远不止函数调用——它封装了领域知识、操作流程、判断规则和最佳实践。把 Skills 简单等同于 Function Calling，就如同把"一套完整的厨房操作规范"等同于"一把菜刀"一样，忽略了更高层次的编排和知识管理价值。

### 12.3 答疑：微调 vs Agent/RAG 的应用边界

---

**🙋 问题 3**：什么场景下，无法通过 Agent 的 RAG 或框架来解决，而必须通过模型微调（Fine-tuning）才能实现？

> 📖 **背景**
>
> 在前面课程中我们学习了 ICL（In-Context Learning）、RAG、Agent 框架等多种让模型获取知识和执行任务的技术手段。但学员在实际应用中会遇到一些场景，发现这些技术手段都不够用，最终不得不考虑微调。那么，什么时候是微调的"必选时刻"？

**💬 老师回答**

这个问题非常关键。微调和 Agent/RAG 之间的选择，本质上是两种不同的能力注入方式的对决。通过 ICL（In-Context Learning）把知识放到上下文窗口里，还是通过微调把知识"写进"模型参数里——这两者的边界在哪里？

##### 核心判断依据：ICL 无法改变模型的输出倾向

In-Context Learning 虽然强大，但它有一个根本的局限：**它只能在模型已有的能力边界内进行引导**，无法从根本上改变模型对于"什么样的输出是好的"这一内在判断倾向。

换句话说：
- **ICL/RAG** 告诉模型"关于这件事你应该知道什么"（知识注入）
- **微调** 改变了模型"面对这种情况时你倾向于怎么回答"（行为塑造）

##### 场景一：角色扮演——需要微调来塑造输出倾向

角色扮演是微调最经典的应用场景之一。

比如说你想让模型扮演一个特定的角色——一个有特定语气、特定说话风格、特定性格特征的角色。你会发现：

- **仅靠 System Prompt（属于 ICL 的范畴）**：可以约束一部分行为，但当对话变长或遇到边界情况时，模型很容易"跳出角色"，回复变得通用化
- **微调的效果**：对于同样的输入，模型的输出倾向会被调整——它"学会"了用特定的语气、特定的表达方式来回应。这种改变是**持久且稳定的**，不会因为对话变长而退化

为什么微调在角色扮演中如此有效？因为微调直接影响的是模型的**参数**（Weights），而不是临时的上下文。模型在微调过程中学习到的不是"现在你应该扮演这个角色"，而是"面对这类输入，这样回应是更自然的"。

##### 场景二：特定领域代码生成——需要微调注入领域表达

另一个典型场景是特定领域的代码生成。

以工业代码为例：通用大模型虽然在通用编程上表现优异（Coding Agent 能力很强），但在日常对话的处理上可能就没那么强。对于特定的工业代码场景：

- 工业代码有其**独特的语法规范、命名约定、架构模式、领域表达方式**
- ICL 可以在一两次对话中指导模型按照特定规范写代码，但**无法保证持续的一致性**——每次都需要在 prompt 中重新描述规范
- 通过微调，你可以让模型**内化这些领域规范**，使得它在面对工业代码任务时，"天然地"倾向于按照正确的领域表达进行输出

> 💡 **核心洞见**
>
> 微调影响的是模型参数的分布，因而改变了模型的"默认行为"。而 ICL/RAG 只能在已有参数分布的基础上提供临时的引导。当你需要改变的是"默认行为"而不仅仅是"当前回答"时，微调就是不可避免的选项。

##### 决策框架：何时选择微调？

```
遇到一个任务，判断是否需要微调：

      模型在 ICL 引导下能否稳定输出预期结果？
          │
    ┌─────┴─────┐
    │           │
   能稳定输出   不能稳定输出
    │           │
    ▼           ▼
  不需要微调    进一步判断：
              不能稳定的原因是什么？
                  │
          ┌───────┼───────┐
          │       │       │
     知识不足   格式/风格  输出倾向
      (RAG    不稳定     不对
      可解决)     │       │
          │      ▼       ▼
          ▼   可尝试优化   需要微调
       用RAG    Prompt    （根本原因是
       补充    但效果     模型参数的
              有限       "默认倾向"
                         不符合需求）
```

| 场景类型 | 推荐方案 | 原因 |
|---|---|---|
| 需要模型了解新的事实信息 | RAG / ICL | 知识注入用上下文窗口最直接高效 |
| 需要模型按特定格式/风格输出 | 先尝试 System Prompt，不稳定则微调 | 格式约束在 Prompt 层面有一定效果，但稳定性取决于任务复杂度 |
| 需要模型稳定扮演特定角色/人格 | 微调 | 角色一致性需要改变模型的输出倾向，ICL 难以长期维持 |
| 需要模型在特定领域持续产出高质量代码 | 微调 | 领域编码规范和表达方式需要内化到模型参数中 |
| 需要模型执行多步骤推理任务 | Agent + Loop | 推理能力是通用能力，微调通常不是首选 |

> 🔗 **关联知识**
>
> 在之前讨论的 ICL（In-Context Learning）和模型能力的课程中，我们提到过：模型的"能力"和"知识"是两个不同维度。微调更适合调整"能力倾向"（how to respond），而 RAG 更适合补充"知识内容"（what to know）。理解这个区分，就不会在微调和 RAG 之间产生混淆。

### 12.4 答疑：多模态处理中的 Loop 设计

---

**🙋 问题 4**：我们之前讨论的 Loop 机制，在多模态处理场景下也适用吗？比如短剧生成、语音应答这样的任务？

> 📖 **背景**
>
> 课程前半部分讨论的 Loop Engineering 主要以文本任务为例。但随着多模态模型（支持图像、视频、音频生成）的快速发展，学员自然会问：Loop 的思想在多模态场景中是否同样适用？有没有需要特别注意的地方？

**💬 老师回答**

##### Loop 机制在多模态场景完全适用

答案是肯定的——多模态处理中 Loop 机制完全适用。但需要注意一个关键的设计原则：**将智能模块与上游/下游模块分开**。

在多模态任务中，Loop 的复杂度往往不体现在单个模型的能力上，而体现在**多个异构模块之间的编排和协调**上。文本生成、图像生成、视频生成、语音合成——这些是不同的能力单元，需要被精心编排成一个协调的管线（Pipeline）。

##### 案例拆解：短剧生成管线

让我们以一个具体的短剧生成系统为例，来拆解多模态 Loop 的设计思路：

```
                        短剧生成系统
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ 剧本生成   │  │ 角色管理   │  │ 剧情管理   │
        │ (智能模块1) │  │ (智能模块2) │  │ (智能模块3) │
        └─────┬────┘  └─────┬────┘  └─────┬────┘
              │              │              │
              └──────────────┼──────────────┘
                             │
                      ┌──────┴──────┐
                      │  编排管线     │  ← Loop 的核心在这里
                      │ (Orchestr)  │
                      └──────┬──────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ 画面生成   │  │ VLM评估   │  │ 人工审核   │
        │(视频模型)  │  │(视觉评估)  │  │(质量把关)  │
        └─────┬────┘  └─────┬────┘  └─────┬────┘
              │              │              │
              └──────────────┼──────────────┘
                             │
                      ┌──────┴──────┐
                      │  是否通过？   │
                      └──────┬──────┘
                             │
                    ┌────────┼────────┐
                    ▼                 ▼
                 通过 ✅            不通过 ❌
             输出最终视频        回到编排管线
                                 重新生成/调整
```

##### 管线各环节详解

**第一层：智能模块层（上游）**

这一层负责"创造内容"，每个子模块都是相对独立的智能单元：

1. **剧本生成模块**：根据主题/大纲生成短剧的完整剧本。这是一个文本生成任务，可以用 LLM 完成。
2. **台词生成模块**：为每个角色生成符合其性格的台词。需要结合角色设定进行约束。
3. **人物行动描述模块**：生成每个场景中角色的动作、表情、走位等描述。这些描述将作为下游画面生成的输入。
4. **人物关系管理模块**：维护角色之间的关系状态（朋友/敌人/恋人等），确保剧情一致性。
5. **剧情管理模块**：管理剧情的节奏、冲突、转折等宏观结构。

> ⚠️ **注意事项**
>
> 这些智能模块本身是**可插拔**的。剧本生成可能需要较高的创造力，台词生成需要较强的角色理解能力，而人物关系管理可能更偏向于结构化数据管理。不同模块可能需要使用不同能力的模型，这是拆分的目的之一。

**第二层：编排管线层（中游）**

这是 Loop 机制发挥作用的核心层。编排管线负责：

- **任务分发**：将上游智能模块的产出分发到下游的生成模块
- **顺序控制**：决定各模块的执行顺序和依赖关系
- **回路控制**：当下游评估不通过时，决定回退到哪一步重新生成
- **状态管理**：维护整个流程中的状态信息

**第三层：生成与评估层（下游）**

1. **画面生成**：根据台词和人物行动描述，使用视频/图像生成模型（如各类视频生成模型）生成画面。这是一个典型的跨模态任务——文本描述 → 视觉内容。
2. **VLM 评估**：使用视觉语言模型（Vision Language Model，简称 VLM）对生成画面进行评估。VLM 的典型使用方式是：**针对生成画面提出评估问题 + 提供评估标准文档**，让 VLM 判断画面质量是否达标。例如："画面中角色的表情是否与台词描述的情感一致？"
3. **人工审核**：对于质量要求较高的场景，仍需要人工最终把关。

##### 多模态 Loop 的关键设计要点

| 设计要点 | 说明 | 为什么重要 |
|---|---|---|
| **模块解耦** | 智能模块与生成模块分离 | 允许独立迭代：换更好的剧本模型不影响画面生成管线 |
| **评估反馈** | VLM 或人工评估结果要能回传 | Loop 的本质在于"评估结果驱动下一次迭代" |
| **管线编排** | 统一的任务分发和流程控制 | 避免各模块各自为战，确保整体一致性 |
| **外部工作** | 剧本、角色、剧情的知识管理 | 这些不是模型能自动解决的，需要外部系统和人工参与 |

> 💡 **延伸思考**
>
> 多模态 Loop 的特殊之处在于：**评估环节的异构性**。纯文本场景下，你可以用同一个 LLM 来生成和评估。但在多模态场景下，你的评估器（如 VLM）和生成器（如视频模型）是完全不同的系统，它们之间的信息传递和反馈循环需要更精心的设计。这也是为什么在多模态场景下，"编排管线"的设计尤为重要——它负责在不同模态、不同模型之间架起桥梁。

### 12.5 答疑：工业场景中 Agent 的正确落地思路

---

**🙋 问题 5**：一个工厂有很多岗位，能不能为每个岗位做一个 Agent 来实现工厂的某种流程？有没有什么建议？应该怎么实现？

> 📖 **背景**
>
> 很多营销内容宣扬"用 Agent 替代员工"，这让不少同学在思考工业场景的 AI 落地时，会自然地沿着"一个岗位 → 一个 Agent"的思路去设计。但这种思路在真实的工业落地中是否可行？企业真正需要的是什么？

**💬 老师回答**

这个问题触及了当前 AI 落地的核心误区，老师的回答非常精彩且实务。

##### 核心观点：Agent 不应该按照"人的形象"来设计

这是最关键的认知转变。很多营销号会说"学会做 Agent，就能做出一个数字人替代一个员工"，但这种叙事很容易把我们的系统和具体的人的形象挂钩在一起。**在真实的企业落地中，我们几乎从来不会按照"人的形象"去落地 Agent。**

原因很简单：企业中的"岗位"是一个复杂的社会组织概念，它包含了：
- 明确的职责范围
- 与其他岗位的协作关系
- 隐含的决策权力
- 非正式的信息网络
- 经验和直觉判断

试图用一个 Agent 去"替代"这样一个复杂的岗位，不仅技术上极其困难，更重要的是——**这根本就不是正确的目标。**

##### 正确的思路：在具体环节中嵌入智能能力

从目前实际在开展的企业合作项目来看，真正的诉求不是"替代某个岗位"，而是在**特定的工作环节中嵌入智能能力**。以下是两个真实案例：

**案例一：硬件设备厂商 —— PCB 设计辅助**

某硬件设备厂商的真实需求：

- **不是**："帮我做一个能替代 PCB 设计工程师的 Agent"
- **而是**：
  1. "能不能根据初始的产品需求，自动生成原理图或 PCB 板的设计草图？"
  2. "在一个 PCB 设计完成后，能不能通过智能体帮我们检查设计中是否存在问题？"

注意这里的区别：Agent 不是在替代任何一个人，而是**进入了设计流程中的特定环节**——"辅助画图"和"辅助检查"。设计工程师仍然在岗位上，但他的工作方式变了：以前他需要从零开始画图、逐项检查，现在他可以先让 Agent 生成初稿、完成初步检查，然后自己专注于更高价值的决策和优化。

**案例二：质检抽检 —— 自动生成质检代码**

另一个典型案例来自产品质检场景：

- 企业有大量产品需要进行质检抽检
- 质检需要使用特定的质检代码，运行在单片机（MCU）上
- 传统方式：人工阅读产品说明书 → 手动编写质检代码 → 烧录到单片机 → 运行测试
- AI 嵌入方式：**根据产品说明书自动生成质检代码 → 直接注入到单片机 → 运行 → 获取结果**

这里的智能能力嵌入在"代码生成"这个环节，它影响的不是某个"质检员"的岗位，而是**整个质检工作流程的效率。**

##### Agent 真正影响的是工作流程，而非岗位

这是老师回答中最深刻的洞察：**Agent 这项新能力进入企业系统后，真正产生冲击的是工作流程（Workflow），而不是个别岗位。**

```
传统的思维方式（错误）：
┌──────────┐    ┌──────────┐    ┌──────────┐
│  岗位 A   │    │  岗位 B   │    │  岗位 C   │
│ (工种A)   │    │ (工种B)   │    │ (工种C)   │
└──────────┘    └──────────┘    └──────────┘
      ↓               ↓               ↓
   用Agent         用Agent         用Agent
   替代A           替代B           替代C


正确的思维方式：
┌─────────────────────────────────────────────┐
│                工作流程                       │
│                                             │
│  环节1 ──→ 环节2 ──→ 环节3 ──→ 环节4 ──→   │
│    │         │         │         │          │
│    ▼         ▼         ▼         ▼          │
│  [智能]    [传统]    [智能]    [传统]        │
│  嵌入      操作      嵌入      操作          │
│                                             │
│  智能能力分散嵌入在流程的各个环节中            │
│  而不是集中在某个"岗位"上                     │
└─────────────────────────────────────────────┘
```

##### 新能力带来新范式的冲击

这个观点的更深层含义是：**AI 模型能力进入企业信息系统，带来的是一场生产力范式的变化。**

- 以前我们设计信息系统时，是在"没有 AI 能力"的假设下进行的
- 当 AI 模型能力作为一个新的基本能力出现后，原有的信息系统架构受到了冲击
- 这个冲击不是在传统范式下能解决的问题——你必须**从新的视角去看待它**
- 你需要重新思考：**在这个新能力可用的前提下，我们的系统还能做哪些以前做不到的事情？**

> 💡 **核心洞见**
>
> 工业场景下的 Agent 落地，不是"用 AI 替换人"，而是"用 AI 重构流程"。一个环节一个环节地去想：在当前的流程中，哪里可以用 AI 能力来提升效率、降低错误率、或者实现以前做不到的事情？这才是务实且有效的落地思路。

##### 企业落地的建议框架

| 步骤 | 内容 | 示例 |
|---|---|---|
| 1. 流程梳理 | 画出完整的业务流程图，标注每个环节的输入输出 | PCB 设计流程：需求 → 原理图 → PCB 布局 → 审查 → 打样 |
| 2. 瓶颈识别 | 找到流程中耗时最多、错误率最高、或严重依赖专家经验的环节 | "原理图绘制耗时 3 天"、"审查靠资深工程师逐项检查" |
| 3. 可行性评估 | 判断该环节是否适合嵌入 AI 能力：输入是否结构化？评判标准是否明确？容错空间有多大？ | 产品说明书 → 质检代码：输入文本 + 输出代码，标准明确，适合 AI |
| 4. 模块拆分 | 不要想着做一个"全知全能的 Agent"，而是做一个"只做好这件事的智能模块" | PCB 检查 Agent 只负责检查设计问题，不负责画图 |
| 5. 流程再造 | 将智能模块嵌入流程后，重新设计上下游的衔接方式和人工介入点 | 自动检查 → 人工复核 → 修改 → 再次自动检查 |

> ⚠️ **注意事项**
>
> 工业场景的 AI 落地有一条铁律：**不要一开始就追求端到端的全自动化。** 先从最明确、最可控、价值最大的单个环节入手，验证可行后再逐步扩展。一上来就想用 Agent 替代整个岗位，十有八九会失败。

### 12.6 答疑：FDE 角色与工程素养

---

**🙋 问题 6**：学完这些内容后，能独立完成从需求到工程级的代码上线吗？老师提到的 FDE 是什么？

> 📖 **背景**
>
> 在课程最后的答疑环节，有同学对"学完之后能做什么"有切实的关切。老师用 FDE（Forward Deployed Engineer）这个概念来回答了这个问题，这既是一个角色定位，也是一种能力要求。

**💬 老师回答**

##### FDE 是什么？

**FDE** 全称 **Forward Deployed Engineer**（前向部署工程师），这是一个源自工程实践领域的概念，用来描述一种特殊的工程师角色。

FDE 的核心职责是：**弥合技术能力与业务场景之间的差距（Gap）。**

在 AI 大模型时代，这个差距尤为突出：

```
当前 AI 落地的核心 Gap：

┌─────────────────┐                    ┌─────────────────┐
│   模型侧（AI）    │                    │   企业侧（业务）   │
│                  │        Gap         │                  │
│ "我能力很强"      │ ←───────────────→ │ "我不知道 AI     │
│ "我能做很多事"    │                    │  能帮我做什么"    │
│                  │                    │                  │
│ 不懂业务逻辑      │                    │ 不懂模型能力      │
│ 不懂行业语境      │                    │ 不懂如何接入      │
│ 不懂企业流程      │                    │ 不懂限制与边界    │
└─────────────────┘                    └─────────────────┘

                    FDE 的工作就是在这个 Gap 上
                    架起一座桥
```

FDE 需要的核心能力：

1. **既懂业务**：理解企业的业务流程、行业语境、真实痛点
2. **又懂模型**：理解大模型的能力边界、使用方法、工程落地
3. **工程素养**：能设计系统架构、能写生产级代码、能处理边界情况

##### 为什么 FDE 这个角色如此重要？

因为当前的 AI 落地面临一个结构性问题：

- **企业的人**（业务专家、IT 部门、管理层）：了解自己的业务，但不懂 AI 到底能为他们带来什么实质性的改变。他们可能听说过 ChatGPT，但不知道如何将大模型嵌入到自己的 SAP/ERP/MES 系统里。
- **模型的人**（算法工程师、AI 研究员）：专注于提升模型能力，追求 benchmark 分数，但不太懂具体行业的业务逻辑。他们能做出很强的模型，却不知道这个模型在企业里到底该怎么用。

这两个群体之间存在一个巨大的鸿沟。FDE 就是这个鸿沟上的桥梁——他们能把 AI 模型的能力"翻译"成企业可以理解和使用的解决方案，也能把企业的需求"翻译"成技术层面可以去解决的问题。

##### 工程素养为什么依然关键？

即使有了 AI 编码工具（如 CC、Codex、Cursor），工程素养仍然不可替代。原因在于：

**AI 编码工具帮你写代码，但不帮你做决策。** 在面对一个复杂任务时，你仍然需要知道：

- 这个任务的系统架构应该如何设计？
- 应该关注哪些核心模块？
- 模块之间的依赖关系和数据流向是怎样的？
- 什么情况下应该拆分模块？什么情况下可以合并？
- 出现问题时，应该从哪个方向去排查？

这些都是 AI 工具目前无法替代的**工程判断力**。

> 💡 **核心洞见**
>
> 课程的核心目的不是教你如何使用某个具体的 AI 工具——这些工具可能半年后就变了。而是帮你建立一套**思维框架**：当面对一个复杂的 AI 落地任务时，你知道应该如何拆解、如何规划、如何设计回路、如何编排管线。这种框架性的理解，是 FDE 角色的根基，比掌握任何具体工具都重要。

##### Loop Engineering 能力与 FDE 的对应关系

FDE 需要掌握的课程核心内容：

| 课程核心内容 | FDE 的应用场景 |
|---|---|
| **多路编排** | 面对复杂任务时，能设计多步骤的智能管线，合理分发子任务 |
| **复杂回路设计** | 能设计评估→反馈→修正的闭环，让 Agent 能自我迭代优化 |
| **Harness 工程** | 理解底层工程支撑体系，能在此基础上扩展和定制 |
| **Skills 体系** | 能为不同业务场景封装可复用的智能能力包 |
| **微调 vs ICL 判断** | 能根据场景特性选择最合适的技术方案 |
| **工业落地思维** | 能从"流程嵌入"而非"岗位替代"的视角设计解决方案 |

##### 从需求到上线：FDE 的能力阶梯

```
能力层级                         能做什么
─────────────────────────────────────────────────
Level 1: 使用者                    用现成工具（CC/Codex）完成个人任务
（Loop Coding 层次）
─────────────────────────────────────────────────
Level 2: 单模块构建者              针对单一环节设计 Agent，解决特定问题
（基础 Agent 层次）
─────────────────────────────────────────────────
Level 3: 系统编排者                设计多模块协作的管线，处理复杂的编排逻辑
（编排工程层次）    ← 课程目标
─────────────────────────────────────────────────
Level 4: FDE                      贯通业务理解与技术方案，独立交付企业级项目
（全面工程层次）    ← 掌握课程内容后的可达目标
```

> 🔗 **关联知识**
>
> 这个能力阶梯的构建，与我们在前一个 Q&A（Q5）中讨论的工业落地思维是一脉相承的。FDE 不是替代某个岗位的人，而是提供了一种新的能力维度——将 AI 能力系统地嵌入到企业的工作流程中，从而带来生产力的跃升。

### 💡 一句话总结

> **Agent Engineering 的本质是在模型这个"大脑"之外，构建 Harness（底座工程）作为"身体"，再通过 Loop（设计方案）驱动这个身体持续追逐目标——从 Prompt 到 Context 到 Tool 到 Workflow，工程在模型之外的分量越来越大，直到今天 Harness + Loop 成为构建模型驱动的复杂任务系统的核心范式。**

### 练习 1: 识别 Loop 与 Harness

**题目**：打开 Claude Code 或 Codex，完成一个复杂任务（如"重构这个模块使其支持插件化"），观察其执行过程。完成以下分析：

**验收标准**：
- [ ] 识别出至少 3 个 Task Board 上的任务卡片（来自 Agent 的规划输出）
- [ ] 找出至少 1 个"观察→重规划"的实例（Agent 执行失败后调整策略）
- [ ] 列出对应的 Harness 支撑模块（这个操作由哪个 Harness 模块支撑？）
- [ ] 判断这属于微观 Loop 还是宏观 Loop

### 练习 2: 设计一个简易 Task Board

**题目**：为一个"自动生成周报"的 Agent 设计 Task Board 结构。周报需要：收集 Git 提交记录 → 汇总本周完成的任务 → 按项目归类 → 生成 Markdown 格式报告。

**验收标准**：
- [ ] 明确三大核心要素（目标、完成条件、交付形态）
- [ ] 设计至少 5 个任务卡片（含 ID、状态、依赖关系）
- [ ] 画出 Task Board 的状态流转图（ASCII）
- [ ] 写出至少 2 个"完成判断条件"的具体标准

### 练习 3: 演进之路梳理

**题目**：用自己的话画出 AI 应用的演进之路图（Prompt → Context → Tool → Workflow → React → Loop），并在每个阶段标注工程的参与程度和核心变化。

**验收标准**：
- [ ] 覆盖全部 6 个阶段
- [ ] 每个阶段有 1-2 句话说明"工程角色发生了什么变化"
- [ ] 标注出 In-Context Learning 和 RAG 的关系
- [ ] 标注出"模型吞噬一切"观点的错误之处
