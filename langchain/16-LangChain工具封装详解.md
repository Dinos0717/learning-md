## 一、课程背景与目标

### 1.1 前情回顾

在前面十五节课中，我们已经学会了如何使用 LangChain 调用大语言模型进行对话。但所有交互模式都是：

```
用户提问 → LLM 基于训练数据回答 → 返回结果
```

这种模式有一个根本性的限制：**LLM 只能基于训练数据运作**。

### 1.2 本节课的核心问题

> **LLM 不知道今天的天气、查不到数据库里的订单、调不了公司的内部 API——这些"外面的世界"LLM 接触不到。怎么办？**

```
用户："今天北京的天气怎么样？"
LLM： ???（训练数据是几个月前的，不知道今天天气）

用户："我最近的订单状态是什么？"
LLM： ???（无法访问你的订单数据库）

用户："帮我计算 12345 × 67890 的结果"
LLM： ???（可能给你编一个错误答案——幻觉）
```

**答案**：给 LLM 配备**工具**——让它能调用外部函数、API、数据库。

### 1.3 本课要掌握的目标


## 二、LLM 的三大局限性——为什么需要工具

### 2.1 局限性一：获取不到实时数据

> 大语言模型是通过**历史数据**训练出来的，它不知道训练之后发生的事情。

| 你能问的 | LLM 能做到吗？ |
|----------|:------------:|
| "Python 是什么？" | ✅ 知识在训练数据中 |
| "勾股定理是什么？" | ✅ 千年前的知识 |
| "今天北京天气怎么样？" | ❌ 需要实时数据 |
| "现在比特币价格多少？" | ❌ 需要实时数据 |
| "下周有什么电影上映？" | ❌ 需要实时数据 |

### 2.2 局限性二：无法与第三方应用交互

> LLM 是一个**文本模型**，它不能直接调用 API、查询数据库、操作文件系统。

| 你希望 LLM 做 | LLM 能做到吗？ |
|--------------|:------------:|
| "查一下我的订单状态" | ❌ 需要调用订单 API |
| "把结果存到数据库" | ❌ 需要数据库操作 |
| "给张三发一封邮件" | ❌ 需要调用邮件服务 |
| "读取这个 PDF 文件" | ❌ 需要文件系统操作 |

### 2.3 局限性三：幻觉问题

> 当 LLM 遇到不知道的数据时，它可能通过**推测或臆想**生成内容——这些内容是**没有经过验证的虚假信息**。

```
用户："张三今年多大了？"
LLM："张三今年32岁。" ← 实际上 LLM 根本不知道张三是谁，瞎编的！

用户："Python 4.0 有哪些新特性？"
LLM："Python 4.0 增加了..." ← Python 根本没有 4.0，纯属幻觉！
```

**幻觉产生的原因**：

```
LLM 遇到不知道的问题：
    ├── 不会说"我不知道"（大多模型没有这个"自觉"）
    ├── 根据历史数据中的模式进行推测
    └── 生成看起来合理、但实际上不存在的内容
```

> ⚠️ **工具不是万能的**——它能解决**提供了工具的领域**的幻觉问题。没提供工具的领域，幻觉依然存在。

## 三、工具的四大核心价值

### 3.1 解决 LLM 的局限性

| 局限性 | 工具如何解决 |
|--------|------------|
| 拿不到实时数据 | 提供"天气查询工具"→ 调用天气 API |
| 无法交互三方应用 | 提供"订单查询工具"→ 调用订单 API |
| 幻觉问题 | 提供"数据库查询工具"→ 基于真实数据回答 |

### 3.2 扩展 LLM 的能力边界

> 每增加一个工具，LLM 就多一项能力。

```
没有工具时的 LLM：只能对话
    ↓ 加一个工具
有"计算器"工具 → LLM 能精确计算
    ↓ 再加一个
有"天气查询"工具 → LLM 能报天气
    ↓ 再来一个
有"数据库查询"工具 → LLM 能查数据
    ↓ 继续...
```

### 3.3 模块化设计

> 工具是独立的模块——需要时加载，不需要时卸载。方便维护和复用。

```
项目 A 需要：天气工具 + 订单工具
项目 B 需要：天气工具 + 数据库工具 + 邮件工具
项目 C 需要：计算器工具

→ 工具就像乐高积木，按需组合
```

### 3.4 四大价值速查表

| 价值 | 说明 | 举例 |
|------|------|------|
| **突破局限** | 获取 LLM 拿不到的数据 | 实时天气、股票价格 |
| **扩展能力** | 赋予 LLM 新功能 | 计算、查询、发送邮件 |
| **减少幻觉** | 基于真实数据回答 | 从数据库查而非瞎编 |
| **模块复用** | 独立封装，按需组合 | 一个工具多个项目用 |

## 四、为什么需要自己封装工具

### 4.1 LangChain 提供的工具不够

LangChain 提供了一些内置工具，但**不可能覆盖所有业务场景**：

```
LangChain 内置工具（举例）：
├── 搜索工具
├── 计算器工具
├── Wikipedia 查询
└── ...

你的业务可能需要：
├── 调用公司内部的 CRM API
├── 查询自己的订单数据库
├── 访问内部的文档系统
└── ...
```

> LangChain 的开发团队**无法预知**每个企业、每个项目的具体业务需求。

### 4.2 业务需求的多样性

```
公司 A：电商平台 → 需要调用商品/订单/物流 API
公司 B：银行系统 → 需要调用账户/交易/风控 API
公司 C：医疗平台 → 需要调用患者/诊断/药品 API

→ 每家公司的 API 各不相同
→ LangChain 不可能为每家公司都预先做好工具
```

### 4.3 LangChain 的解决思路

> **"我告诉你怎么定义工具，具体需要什么工具——你自己封装。"**

```
LangChain 提供的：
├── @tool 装饰器 ← 把普通函数变成工具
├── 自动生成 Schema ← 让 LLM 理解工具的功能
└── 统一的调用接口 ← 方便 Agent 使用

你需要做的：
└── 写你自己的业务函数 + 加上 @tool 装饰器 = 工具封装完成
```

## 五、@tool 装饰器——核心机制详解

### 5.1 什么是 @tool 装饰器？

> `@tool` 是 LangChain 提供的一个 Python 装饰器。把它放在普通函数上面，就能将该函数转换为 LangChain 可识别的**工具（Tool）**。

```python
from langchain.tools import tool

@tool                    # ← 加上这个装饰器
def my_function(a: int, b: int) -> int:
    """两个数相加"""
    return a + b

# my_function 现在不仅是一个 Python 函数
# 还是一个 LangChain Tool 对象！
```

### 5.2 装饰前后对比

```python
# ── 装饰前：普通 Python 函数 ──
def add_numbers(a: int, b: int) -> int:
    """两个数相加"""
    return a + b

# 可以调用
result = add_numbers(3, 5)  # → 8
# 但没有 name、description、args_schema 等属性
# LLM 无法理解这个函数是干什么的

# ── 装饰后：LangChain 工具 ──
@tool
def add_numbers(a: int, b: int) -> int:
    """两个数相加"""
    return a + b

# 仍然可以调用
result = add_numbers(3, 5)  # → 8
# 而且多了很多属性！
print(add_numbers.name)         # → "add_numbers"
print(add_numbers.description)  # → "两个数相加"
print(add_numbers.args_schema)  # → {"a": int, "b": int, ...}
# LLM 可以通过这些信息理解工具的功能！
```

### 5.3 @tool 自动生成什么？

`@tool` 装饰器会自动从函数中提取和生成以下信息：

| 自动生成的内容 | 来源 | 用途 |
|:------------:|------|------|
| **工具名称** | 函数名 | LLM 通过名称识别工具 |
| **工具描述** | 函数的 docstring（注释） | LLM 判断这个工具能干什么 |
| **参数 Schema** | 函数的类型注解 | LLM 知道调用时需要传什么参数 |
| **参数是否必填** | 有没有默认值 | LLM 知道哪些参数必须传 |

### 5.4 工作原理

```
┌─────────────────────────────────────────────────────────┐
│                  @tool 装饰器工作原理                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  你写的代码：                                             │
│  @tool                                                   │
│  def add_numbers(a: int, b: int) -> int:                │
│      """两个数相加"""                                     │
│      return a + b                                       │
│                                                         │
│  @tool 自动提取：                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │ name: "add_numbers"           ← 函数名           │    │
│  │ description: "两个数相加"      ← docstring      │    │
│  │ args_schema: {                                  │    │
│  │     "a": {"type": "integer"},  ← 类型注解 int    │    │
│  │     "b": {"type": "integer"},  ← 类型注解 int    │    │
│  │ }                                              │    │
│  └─────────────────────────────────────────────────┘    │
│                         ↓                               │
│  LLM 收到这个 Schema → 理解工具功能 → 决定是否调用          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 六、代码实战——封装第一个工具

### 6.1 完整代码

```python
from langchain.tools import tool

# ============================================================
# 1. 定义普通函数 + @tool 装饰器
# ============================================================
@tool
def multiply(a: int, b: int) -> int:
    """两个数的乘法，返回乘积"""
    return a * b

# ============================================================
# 2. 验证：仍然可以正常调用
# ============================================================
result = multiply(6, 7)
print(f"6 × 7 = {result}")         # → 6 × 7 = 42

# ============================================================
# 3. 查看 @tool 自动生成的元信息
# ============================================================
print(f"工具名称：{multiply.name}")
print(f"工具描述：{multiply.description}")
print(f"参数信息：{multiply.args_schema}")
```

### 6.2 运行结果

```
6 × 7 = 42

工具名称：multiply
工具描述：两个数的乘法，返回乘积
参数信息：{
    'title': 'multiplySchema',
    'type': 'object',
    'properties': {
        'a': {'title': 'A', 'type': 'integer'},
        'b': {'title': 'B', 'type': 'integer'}
    },
    'required': ['a', 'b']
}
```

### 6.3 代码逐行解析

```python
from langchain.tools import tool
```
从 `langchain.tools` 导入 `tool` 装饰器。

```python
@tool
def multiply(a: int, b: int) -> int:
    """两个数的乘法，返回乘积"""
    return a * b
```

| 要素 | 作用 | 是否必须 |
|------|------|:------:|
| `@tool` | 将函数变为 LangChain 工具 | ✅ |
| 函数名 `multiply` | 工具的名称 | ✅ |
| `a: int, b: int` | 类型注解 → 自动生成参数 Schema | ⚠️ 强烈建议 |
| `-> int` | 返回值类型注解 | ⚠️ 建议 |
| `"""两个数的乘法..."""` | 工具的描述（docstring） | ⚠️ 强烈建议 |

> 📌 **最佳实践**：类型注解和 docstring 写得越完整，LLM 就越能准确地理解和选择你的工具。

## 七、@tool 自动生成的 Schema 详解

### 7.1 查看工具信息的三种方式

```python
@tool
def multiply(a: int, b: int) -> int:
    """两个数的乘法，返回乘积"""
    return a * b

# 方式一：逐项查看
print(multiply.name)           # multiply
print(multiply.description)    # 两个数的乘法，返回乘积
print(multiply.args_schema)    # 参数 Schema

# 方式二：查看 Schema 的字典形式
print(multiply.args_schema.model_json_schema())
# → JSON 格式的完整 Schema

# 方式三：查看完整的模型信息
print(multiply.model_dump())
# → 包含所有信息的字典
```

### 7.2 Schema 结构详解

```python
{
    'title': 'multiplySchema',     # Schema 标题
    'type': 'object',              # 参数类型：对象
    'properties': {                # 参数属性
        'a': {
            'title': 'A',          # 参数标题（从变量名自动大写）
            'type': 'integer'      # 参数类型（从 int 注解来）
        },
        'b': {
            'title': 'B',
            'type': 'integer'
        }
    },
    'required': ['a', 'b']         # 必填参数（没有默认值 = 必填）
}
```

| Schema 字段 | 含义 | 来源 |
|-----------|------|------|
| `title` | Schema 名称 | 函数名 + "Schema" |
| `properties` | 参数列表及类型 | 函数的类型注解 |
| `properties.X.title` | 参数名称 | 变量名首字母大写 |
| `properties.X.type` | 参数数据类型 | 类型注解（int→integer, str→string） |
| `required` | 必填参数列表 | 没有默认值的参数 = 必填 |

### 7.3 Schema 对 LLM 的意义

```
LLM 收到这个 Schema 后：

"哦，有一个工具叫 multiply"
"它的功能是两个数相乘"
"需要两个参数：a（整数）和 b（整数），都是必填的"
"返回值是一个整数"

→ LLM 知道了：
   何时该用这个工具（用户要求做乘法时）
   怎么调用这个工具（传 a 和 b 两个整数参数）
```

### 7.4 完整 vs 简略的函数定义

```python
# ❌ 不推荐的写法：缺少类型和注释
@tool
def f(x, y):
    return x * y
# LLM 看到：工具叫 f，不知道干什么，参数类型不明 → 不敢用！

# ✅ 推荐的写法：类型和注释完整
@tool
def multiply(a: int, b: int) -> int:
    """两个数的乘法，返回乘积"""
    return a * b
# LLM 看到：工具叫 multiply，做乘法，需要两个整数 → 清楚明了！
```

## 八、完整代码汇总

### 8.1 基础版——封装简单的计算工具

```python
from langchain.tools import tool

# ── 工具1：加法 ──
@tool
def add_numbers(a: int, b: int) -> int:
    """两个整数相加，返回和"""
    return a + b

# ── 工具2：乘法 ──
@tool
def multiply(a: int, b: int) -> int:
    """两个数的乘法，返回乘积"""
    return a * b

# ── 测试 ──
print(f"add_numbers: {add_numbers.name}")
print(f"  描述: {add_numbers.description}")
print(f"  调用: 3+5 = {add_numbers(3, 5)}")
print()

print(f"multiply: {multiply.name}")
print(f"  描述: {multiply.description}")
print(f"  调用: 6×7 = {multiply(6, 7)}")
```

### 8.2 进阶版——封装模拟的 API 调用工具

```python
from langchain.tools import tool
from typing import Optional

# ── 工具：天气查询 ──
@tool
def get_weather(city: str, date: Optional[str] = None) -> str:
    """查询指定城市的天气情况。
    
    参数:
        city: 城市名称，如"北京"、"上海"
        date: 日期，格式YYYY-MM-DD，不传则查询今天
    """
    # 模拟天气查询（实际项目中替换为真实 API 调用）
    weather_data = {
        "北京": "晴天，25°C",
        "上海": "多云，28°C",
        "深圳": "阵雨，30°C",
    }
    result = weather_data.get(city, f"暂无{city}的天气数据")
    return f"{city}天气：{result}"

# ── 工具：订单查询 ──
@tool
def query_order(order_id: str) -> str:
    """根据订单号查询订单状态。
    
    参数:
        order_id: 订单编号
    """
    # 模拟订单查询（实际项目中替换为数据库查询）
    orders = {
        "A001": "已发货，预计明日到达",
        "A002": "正在处理中",
        "A003": "已签收",
    }
    return orders.get(order_id, f"未找到订单 {order_id}")

# ── 验证信息 ──
print("=== 工具列表 ===")
for tool_func in [get_weather, query_order]:
    print(f"工具: {tool_func.name}")
    print(f"描述: {tool_func.description}")
    print(f"参数: {tool_func.args_schema}")
    print()

# ── 模拟调用 ──
print("=== 模拟调用 ===")
print(get_weather("北京"))       # → 北京天气：晴天，25°C
print(query_order("A001"))       # → 已发货，预计明日到达
```

## 九、不同工具封装方式对比

### 9.1 @tool vs 类继承 vs StructuredTool

| 方式 | 复杂度 | 灵活性 | 适用场景 |
|------|:-----:|:-----:|----------|
| **@tool 装饰器**（本课） | ⭐ 最简单 | ⭐⭐⭐ | ✅ 大多数场景，推荐 |
| 继承 `BaseTool` 类 | ⭐⭐ 中等 | ⭐⭐⭐⭐ | 需要复杂逻辑的工具 |
| `StructuredTool.from_function()` | ⭐⭐ | ⭐⭐⭐ | 需要手动指定 Schema |

### 9.2 本课定位

本课使用的是**最简单的 @tool 装饰器方式**——适合入门和大多数场景。后续课程会涉及 Agent 如何自动选择和调用这些工具。

## 十、常见问题与排错指南

### 10.1 错误速查表

| 错误/问题 | 原因 | 解决方法 | 优先级 |
|-----------|------|----------|:------:|
| `ImportError: cannot import name 'tool'` | 导入路径错误 | `from langchain.tools import tool` | 🔴高 |
| 工具没有 name/description 属性 | 装饰器没有正确使用 | 确保 `@tool` 放在函数定义上一行 | 🔴高 |
| LLM 不调用你的工具 | 工具描述不够清晰 | 补充详细的 docstring 和类型注解 | 🟡中 |
| args_schema 为空或不完整 | 函数没有类型注解 | 给所有参数加上类型注解 `: int`、`: str` | 🟡中 |
| 参数被误判为可选 | 给了默认值 | 不给默认值的参数 = required | 🟢低 |

### 10.2 写好工具函数的检查清单

```
□ 函数有清晰的函数名（LLM 通过名称识别）
□ 函数有详细的 docstring（LLM 通过描述理解功能）
□ 每个参数都有类型注解（int, str, bool...）
□ 返回值有类型注解
□ 不给不需要的参数设默认值（有默认值 = 可选）
□ @tool 装饰器在函数定义的正上方
```


```
│      └── LangChain 内置工具不全 → 需要自己封装               │
│                                                            │

│                                                            │
│  3️⃣  封装最佳实践                                           │
│      ├── 函数名：清晰描述功能                                │

│                                                            │
│  4️⃣  查看工具信息                                           │

│                                                            │
```

### 12.2 一句话总结

> **@tool 装饰器 = 把普通 Python 函数包装成 LLM 可理解、可调用的工具。** LLM 通过自动生成的 Schema（名称、描述、参数类型）来理解工具的功能，决定何时调用。写好类型注解和 docstring 是关键。

### 12.3 练习题答案

> **问**：哪一个**不是**工具调用的主要功能？
>
> - A. 解决大语言模型幻觉问题 ✅ 是主要功能
> - B. 扩展 Agent 实时数据的访问能力 ✅ 是主要功能
> - C. 直接提高大语言模型的语言生成质量 ❌ **不是**——工具只提供数据源，不改变模型本身的生成质量
> - D. 实现模块化设计，便于功能的维护和复用 ✅ 是主要功能
>
> **答案：C**
