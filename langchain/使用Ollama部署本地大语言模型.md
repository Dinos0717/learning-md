## 一、课程背景与目标

### 1.1 前情回顾

在前面几节课中，我们已经学会了：

- 如何通过 LangChain 接入**通义（Tongyi）**等云端大语言模型

### 1.2 本节课的新问题

> **如果不期望使用第三方供应商的服务，而是自己来提供模型服务，怎么办？**

```
之前的模式（依赖云端）：
    你的代码 → 互联网 → 阿里云/OpenAI 服务器 → 模型推理 → 返回结果
    ⚠️ 依赖网络、依赖供应商、可能产生费用

本节课的模式（本地部署）：
    你的代码 → 本地 Ollama 服务 → 本地 GPU/CPU 推理 → 返回结果
    ✅ 不需要网络、不依赖第三方、完全免费
```

## 二、为什么要使用本地模型

### 2.1 两种服务模式的本质区别

```
┌────────────────────────────────────────────────────────────┐
│                     大语言模型的使用方式                       │
├────────────────────────────┬───────────────────────────────┤
│     方式一：云端 API        │     方式二：本地部署             │
├────────────────────────────┼───────────────────────────────┤
│                            │                               │
│  模型运行在供应商的服务器    │  模型运行在你自己的电脑上         │
│  通过网络 API 调用          │  通过本地进程通信               │
│  按使用量付费               │  完全免费                      │
│  需要 API Key 认证          │  不需要 Key                   │
│  依赖网络连接               │  离线可用                      │
│  模型由供应商维护更新        │  模型由你自己管理               │
│  不能使用敏感数据（隐私顾虑）│  数据完全在本地（隐私安全）       │
│                            │                               │
│  代表：通义、OpenAI、智谱AI  │  代表：Ollama、LM Studio       │
│                            │                               │
└────────────────────────────┴───────────────────────────────┘
```

### 2.2 本地部署的典型使用场景

| 场景 | 说明 |
|------|------|
| **隐私敏感数据** | 公司内部文件、个人隐私数据不能上传到第三方服务器 |
| **离线环境** | 没有网络或网络不稳定的环境 |
| **成本控制** | 大量调用不想按 token 付费 |
| **学习研究** | 想深入了解模型运行机制 |
| **定制开发** | 需要对模型做微调或个性化配置 |
| **高频调用** | 每秒大量请求，云端 API 有并发限制 |

### 2.3 本节课的技术选型

> 不推荐自己从零写模型加载和服务代码（太复杂），而是使用成熟的软件来提供本地模型服务。推荐使用 **Ollama**——它简单易用，跨平台支持。

## 三、Ollama 介绍与安装

### 3.1 什么是 Ollama？

**Ollama** 是一个让你在本地电脑上轻松运行大语言模型的开源工具。它的核心特点：

```
Ollama 帮你做的事：
├── 模型下载管理        （一键下载各种开源模型）
├── 模型加载与推理       （自动处理模型加载、内存管理）
├── 提供本地 API 服务    （启动后提供 HTTP API，LangChain 可以直接调用）
├── 跨平台支持          （macOS / Windows / Linux）
└── 命令行交互          （下载后可以立刻在终端中对话）
```

### 3.2 下载 Ollama

**第一步：访问 Ollama 官网**

打开 Ollama 官方网站，首页会显示介绍信息和下载入口。

**第二步：选择对应平台下载**

官网首页有一个醒目的 **「Download」** 按钮，点击后进入下载页面。Ollama 支持三大平台：

| 平台 | 安装包格式 | 说明 |
|------|-----------|------|
| **Windows** | `.exe` 安装程序 | 视频中老师使用的是 Windows |
| **macOS** | `.dmg` 安装包 | Apple Silicon 和 Intel 都支持 |
| **Linux** | 命令行安装 | `curl -fsSL https://ollama.com/install.sh \| sh` |

**第三步：下载安装包**

点击对应平台的下载按钮，浏览器就会开始下载安装包。

> 视频中老师选择了 Windows 版本。下载过程视网络情况可能需要几分钟。如果下载速度慢，可能需要科学上网（Ollama 官网和模型下载服务器在海外）。

### 3.3 安装 Ollama（Windows 为例）

**安装过程：**

```
双击下载好的 OllamaSetup.exe
    ↓
弹出安装界面
    ↓
点击「Install」
    ↓
等待安装完成（进度条走完）
    ↓
显示「Welcome to Ollama」欢迎页面
    ↓
安装完成 ✅
```

**⚠️ 重要注意事项：**

> Ollama 安装过程中**不会询问安装路径**，默认安装到 C 盘：
> ```
> C:\Users\<你的用户名>\AppData\Local\Ollama
> ```
>
> - 如果你的 C 盘空间紧张，安装前请确保有足够的剩余空间
> - C 盘满了可能会导致系统卡顿
> - 模型的存储路径可以之后修改（见 3.4 节）

**安装完成后的验证：**

安装成功后，在终端输入：

```bash
ollama --version
# 预期输出：ollama version 0.x.x
```

同时，Windows 任务栏右下角会出现一个**羊驼图标**🦙，表示 Ollama 服务正在后台运行。

### 3.4 修改模型存储路径（重要！）

> 默认情况下，Ollama 下载的模型文件也存储在 C 盘。模型文件可能很大（几个 GB 到几十 GB），C 盘很可能不够用。**强烈建议修改存储路径。**

**操作步骤：**

```
第一步：找到任务栏右下角的 Ollama 羊驼图标 🦙

第二步：右键点击图标 → 选择「Settings」

第三步：弹出设置窗口，找到「Model Location」选项

第四步：点击选择按钮，选择一个空间充裕的盘
        例如：E:\ollama-models

第五步：确认修改 ✅
```

```
┌─────────────────────────────────────────┐
│          Ollama Settings                 │
│                                         │
│  Model Location:                        │
│  ┌─────────────────────────────────┐    │
│  │ E:\ollama-models        [选择]  │    │  ← 改到这里
│  └─────────────────────────────────┘    │
│                                         │
│  默认路径（不推荐）：                      │
│  C:\Users\xxx\AppData\Local\Ollama      │
│                                         │
└─────────────────────────────────────────┘
```

> 💡 修改后，所有新下载的模型都会存储到新路径。已经下载的模型需要手动移动。

## 四、模型的选择与评估

### 4.1 Ollama 支持的模型列表

Ollama 提供了丰富的开源模型。在 Ollama 官网上方有一个 **「Models」** 标签卡，点击进入可以浏览所有可用模型。

**部分可用模型：**

| 模型名称 | 出品方 | 说明 |
|----------|--------|------|
| `llama3.2` | Meta | 最新 Llama 系列 |
| `llama3.1` | Meta | Llama 3.1 系列 |
| `qwen3` | 阿里（通义） | 通义千问 3 开源版 |
| `qwen2.5` | 阿里（通义） | 通义千问 2.5 开源版 |
| `deepseek-r1` | DeepSeek | DeepSeek 推理模型 |
| `gemma3` | Google | Google 开源模型 |
| `mistral` | Mistral AI | 欧洲领先开源模型 |
| `phi4` | Microsoft | 微软小型高效模型 |
| `yi` | 零一万物 | 国产开源模型 |
| ... | ... | 还有更多 |

> 📌 视频中老师以 **deepseek-r1** 模型为例进行演示。这是 DeepSeek 的推理模型，擅长逻辑思考和逐步推理。

### 4.2 理解模型的参数规模

当你点击一个具体模型后，会发现它有多个版本。以 `deepseek-r1` 为例：

```
deepseek-r1 可用版本：
├── deepseek-r1:1.5b    ← 15亿参数
├── deepseek-r1:7b      ← 70亿参数
├── deepseek-r1:8b      ← 80亿参数
├── deepseek-r1:14b     ← 140亿参数
├── deepseek-r1:32b     ← 320亿参数
└── deepseek-r1:70b     ← 700亿参数
```

**冒号后面的数字是什么意思？**

> `1.5b` = 1.5 billion = **15亿个参数**
>
> 参数 = 模型内部的可调权重数量，类似于人脑的"神经元连接数"。

**参数规模与模型能力的关系：**

```
参数越少 → 模型"脑子"越小 → 能力越弱 → 但运行要求越低
参数越多 → 模型"脑子"越大 → 能力越强 → 但运行要求越高

1.5b  ←→  7b  ←→  14b  ←→  32b  ←→  70b
弱/快                           强/慢
```

> "你使用 1.5b 版本，发现它回复的内容都不称心意、不满意。这时候可以考虑使用更高的版本。"

### 4.3 模型文件的大小（Size）

每个模型版本旁边会标注文件大小：

| 版本 | 文件大小 | 含义 |
|:----:|:--------:|------|
| `1.5b` | ~1.1 GB | 占用硬盘约 1.1GB |
| `7b` | ~4.7 GB | 占用硬盘约 4.7GB |
| `8b` | ~5.2 GB | 占用硬盘约 5.2GB |
| `14b` | ~9 GB | 占用硬盘约 9GB |
| `70b` | ~40 GB | 占用硬盘约 40GB |

> ⚠️ 这个 Size 代表模型文件在硬盘上的存储空间。你下载的时候需要确保硬盘有足够的空间。

### 4.4 综合选择策略

选择模型时，需要**同时考虑**三个因素：

```
                    选哪个模型？

        ┌───────────────┼───────────────┐
        │               │               │
     硬盘空间         显存大小         你的需求
        │               │               │
   够放模型文件吗？   够跑推理吗？     只是体验还是
   1.5b→1.1GB      1.5b→2-4GB      生产使用？
   70b→40GB        70b→40GB+
```

**视频中老师的建议：**

> "如果仅仅是体验，我们一般跑 1.5b 就够了。我们也是案例演示，用 1.5b 比较稳定——占用的内存少、下载速度快、也能看见效果。"

## 五、硬件要求详解

### 5.1 两个关键硬件指标

运行本地模型需要考虑两个硬件指标，**缺一不可**：

| 指标 | 作用 | 不够会怎样 |
|------|------|-----------|
| **内存（RAM）** | 存放模型文件数据 | 模型加载失败或系统卡死 |
| **显存（VRAM）** | GPU 推理计算 | 推理极慢或无法运行 |

### 5.2 为什么必须考虑显卡和显存？

模型在回答问题时有一个**思考/推理的过程**。这个过程需要做大量数学计算，而显卡（GPU）就是专门做这种并行计算的硬件。

> "模型去理解你的需求，再去生成答案，有一个思考的过程。思考的过程是需要显卡来进行计算的。"

没有显卡（或显存不够）：
```
推理任务 → CPU 来做 → 非常慢（可能几分钟才回复一句）
```

有足够的显卡显存：
```
推理任务 → GPU 来做 → 很快（几秒内回复）
```

### 5.3 显存需求估算

视频中老师给出了一个**实用的估算方法**：

| 参数规模 | 大约需要的显存 | 说明 |
|:--------:|:-------------:|------|
| **1.5b** | 2 ~ 4 GB | ✅ 大多数电脑都能跑 |
| **7b** | 10 ~ 12 GB | 需要中端以上显卡 |
| **8b** | 12 ~ 16 GB | RTX 4060 Ti 16GB 或以上 |
| **14b** | 20 ~ 24 GB | RTX 4090 24GB 级别 |
| **32b** | ~30 GB+ | 需要专业级或多卡 |
| **70b** | ~40 GB+ | 需要高端服务器 |

> ⚠️ 这只是**大致估算**，实际需求还会受量化精度（int4/int8/fp16）等因素影响。

### 5.4 如何查看自己电脑的显存？

**Windows 系统：**

```
第一步：在任务栏空白处右键 → 选择「任务管理器」

第二步：点击「性能」标签

第三步：在左侧列表中找到「GPU」

第四步：查看右下方的「专用 GPU 内存」
        （Dedicated GPU Memory）
```

```
┌──────────────────────────────────────────┐
│            任务管理器 - 性能              │
│                                          │
│  CPU   内存   磁盘   GPU  ← 点这里        │
│                                          │
│  GPU 0: NVIDIA GeForce RTX 4070          │
│  ┌──────────────────────────────────┐    │
│  │         GPU 使用率图表            │    │
│  └──────────────────────────────────┘    │
│                                          │
│  专用 GPU 内存:  12.0 GB  ← 看这里！     │
│  共享 GPU 内存:   8.0 GB                 │
│                                          │
└──────────────────────────────────────────┘
```

**macOS 系统：**

```
点击左上角苹果图标 → 关于本机 → 查看「图形卡」信息
Apple Silicon（M1/M2/M3）的显存和内存是共享的
```

**视频中老师的例子：**

> "我这台电脑是 12GB 显存，最多只能跑 7b 版本。你说能不能跑 8b？能跑但有可能崩溃。所以说在我的电脑上，极限就是 7b。"

### 5.5 硬件决策流程

```
我要跑本地模型

第一步：查看显存
    ├── 显存 < 4GB  → 😞 只能跑 1.5b 或更小的模型
    ├── 显存 4-8GB  → 🙂 可以跑 1.5b-7b
    ├── 显存 8-12GB → 😊 可以跑 7b-8b
    ├── 显存 12-16GB→ 😄 可以跑 8b
    ├── 显存 16-24GB→ 🤩 可以跑 8b-14b
    └── 显存 > 24GB → 🚀 可以跑更大的模型

第二步：检查硬盘空间
    └── 确保有模型文件大小 × 1.5 的剩余空间

第三步：如果硬件不够
    ├── 选更小的参数版本
    ├── 换一个更轻量的模型
    └── 或者继续使用云端 API（也没什么不好）
```

## 六、下载并运行第一个本地模型

### 6.1 下载模型

Ollama 的 `run` 命令可以**同时完成下载和运行**。如果模型已经下载过，就直接运行；如果没下载过，会先下载再运行。

**基本语法：**

```bash
ollama run <模型名>:<版本>
```

**实战命令（视频中以 deepseek-r1 1.5b 为例）：**

```bash
ollama run deepseek-r1:1.5b
```

命令执行后：

```
pulling manifest
pulling 6e4c38d1a3e5... 100% ████████████ 1.1 GB/1.1 GB
pulling 56b8e48c1a3f... 100% ████████████  55 MB/55 MB
pulling ...
verifying sha256 digest
writing manifest
success ✅

>>> _   ← 现在可以在终端直接对话了
```

### 6.2 关于下载速度

> ⚠️ Ollama 的模型文件托管在海外服务器上。如果下载速度很慢或无法下载，可能需要科学上网。

**下载过程中会自动显示进度条**，对于 1.1GB 的 1.5b 模型，在网络良好的情况下几分钟就能下载完成。

### 6.3 在终端中测试模型

下载完成后，Ollama 会自动进入交互模式：

```
>>> 你好
你好！很高兴见到你，有什么我可以帮忙的吗？

>>> 今天天气怎么样？
今天是一个好天气！当然，具体天气情况因地区而异。
如果你想知道准确的天气信息，建议查看当地的天气预报。

>>> /bye
（退出对话）
```

**退出对话的命令：**

```bash
/bye
```

输入 `/bye` 后回到普通终端。但注意：**退出对话不等于关闭 Ollama 服务**。Ollama 服务仍在后台运行（任务栏有羊驼图标），可以被 LangChain 代码调用。

### 6.4 选择模型的经验总结

| 场景 | 推荐参数规模 | 理由 |
|------|:-----------:|------|
| 初次体验 / 学习 | `1.5b` | 下载快、占用少、马上看到效果 |
| 日常辅助 | `7b` ~ `8b` | 能力足够日常使用，大多数中端显卡能跑 |
| 开发测试 | `7b` ~ `14b` | 需要在能力和硬件之间平衡 |
| 生产使用 | `14b` ~ `32b`+ | 需要更好的回答质量 |
| 研究实验 | 最大你能跑动的 | 追求最好效果 |

## 七、用 LangChain 代码调用本地模型

### 7.1 整体思路

上一步我们已经在终端中测试了模型可用。现在要做的是——**用 Python 代码来调用本地 Ollama 服务**。

```
你的 Python 代码
    ↓ (LangChain 封装)
Ollama 本地服务（已经在后台运行）
    ↓ (加载模型文件)
本地 GPU/CPU 推理
    ↓
返回结果
```

### 7.2 第一步：安装 langchain-ollama

参考 LangChain 官方的集成文档，Ollama 有自己的独立包：

```bash
# 确保在正确的虚拟环境中
conda activate agent

# 安装 langchain-ollama（视频中版本为 0.3.3）
pip install langchain-ollama==0.3.3
```

> ⚠️ 如果包名中有连字符，某些终端不加引号可能报错，建议加引号：
> ```bash
> pip install "langchain-ollama==0.3.3"
> ```

**为什么安装这么快？**

## 八、不同模型的响应类型差异

### 8.1 核心认知

> **不同模型供应商的 `invoke()` 返回类型可能不同。** 这是视频中反复强调的一个重要知识点。

### 8.2 三种模型的响应对比

```python
# ============================================================
# 实验：同一个调用方式，不同模型的返回类型
# ============================================================

question = "夏天适合吃什么水果？"

# --- 通义（Tongyi）---
from langchain_community.chat_models import ChatTongyi
tongyi = ChatTongyi(api_key="...", model="qwen-turbo")
response1 = tongyi.invoke(question)
# response1 → 直接是字符串！
# print(response1) → "西瓜、葡萄、桃子、哈密瓜"

# --- 智谱AI（ZhipuAI）---
from langchain_community.chat_models import ChatZhipuAI
zhipu = ChatZhipuAI(model="glm-4")
response2 = zhipu.invoke(question)
# response2 → 是 AIMessage 对象！
# print(response2) → AIMessage(content="西瓜、葡萄...")
# print(response2.content) → "西瓜、葡萄、桃子、哈密瓜"

# --- Ollama 本地模型 ---
from langchain_ollama import ChatOllama
ollama = ChatOllama(model="deepseek-r1:1.5b")
response3 = ollama.invoke(question)
# response3 → 也是 AIMessage 对象！
# print(response3) → AIMessage(content="西瓜、葡萄...")
```

### 8.3 AIMessage 对象详解

当你直接输出 response（不加 `.content`）时，通义可能给你字符串，但智谱AI 和 Ollama 会给你一个 `AIMessage` 对象：

```python
# 直接输出 response（不用 print）
response
# 输出：
# AIMessage(
#     content='夏天适合吃的水果有：\n\n1. 西瓜...\n2. 桃子...',
#     response_metadata={...},
#     id='...'
# )
```

`AIMessage` 对象包含的常用属性：

| 属性 | 类型 | 说明 |
|------|------|------|
| `.content` | `str` | 模型回复的**文本内容**（最常用） |
| `.response_metadata` | `dict` | 响应的元数据（模型名、token 用量等） |
| `.id` | `str` | 消息的唯一 ID |

### 8.4 为什么要用 `print()` 而不是直接输出？

视频中演示了一个关键细节——**换行符的处理**：

```python
# 直接输出（Jupyter 中）
response.content
# 显示：'夏天适合吃的水果有：\n\n1. 西瓜\n2. 桃子\n3. 哈密瓜'
# ↑ \n 没有被解析为换行，显示为原始字符

# 用 print 输出
print(response.content)
# 显示：
# 夏天适合吃的水果有：
#
# 1. 西瓜
# 2. 桃子
# 3. 哈密瓜
# ↑ \n 被正确解析为换行，排版美观
```

> 📌 **结论**：获取内容用 `.content`，展示内容用 `print()`。这是实用的编码习惯，适用于所有模型。

### 8.5 不同模型返回类型的统一处理

为了代码的通用性，建议始终用 `.content` 提取文本内容：

```python
def ask_model(llm, question):
    """通用调用函数，无论什么模型都能正确处理"""
    response = llm.invoke(question)

    # 统一处理：如果 response 有 content 属性就用，否则直接转字符串
    if hasattr(response, 'content'):
        return response.content
    else:
        return str(response)

# 使用
result = ask_model(ollama_llm, "你好")
print(result)
```

### 8.6 响应类型速查表

| 模型 | `invoke()` 返回类型 | 获取文本的方式 |
|------|-------------------|---------------|
| 通义（ChatTongyi） | `str` | 直接使用 |
| 智谱AI（ChatZhipuAI） | `AIMessage` | `.content` |
| Ollama（ChatOllama） | `AIMessage` | `.content` |
| OpenAI（ChatOpenAI） | `AIMessage` | `.content` |
| Google（ChatGoogleGenerativeAI） | `AIMessage` | `.content` |

> 🎯 **最佳实践**：不管什么模型，一律用 `response.content` 获取文本内容，用 `print()` 输出。

## 十、云端 API vs 本地部署——全面对比

学完了云端调用和本地部署两种方式后，做一个系统性的对比：

| 对比维度 | 云端 API（通义、OpenAI…） | 本地部署（Ollama） |
|----------|:------------------------:|:------------------:|
| **模型能力** | ⭐⭐⭐⭐⭐ 商业级最强模型 | ⭐⭐⭐ 取决于参数规模和硬件 |
| **响应速度** | ⭐⭐⭐⭐ 专用服务器，通常很快 | ⭐⭐⭐ 取决于你的显卡 |
| **费用** | 按量付费 | **完全免费** |
| **隐私安全** | 数据上传到第三方服务器 | ⭐⭐⭐⭐⭐ **数据完全在本地** |
| **网络依赖** | 必须有网络 | **离线可用**（下载模型后） |
| **硬件要求** | 几乎无要求 | 需要较好的显卡和内存 |
| **安装配置** | ⭐⭐⭐⭐⭐ 装个包就行 | ⭐⭐⭐ 需要配置环境 |
| **模型选择** | 供应商提供什么就用什么 | ⭐⭐⭐⭐⭐ 开源模型随便选 |
| **稳定性** | ⭐⭐⭐⭐ 供应商保障 | 取决于你的电脑稳定性 |
| **并发能力** | 受 API 配额限制 | 受你的硬件限制 |
| **模型更新** | 自动（供应商更新） | 手动（ollama pull 更新） |
| **适用场景** | 快速开发、生产应用 | 学习研究、隐私数据、离线场景 |

### 10.1 什么时候用云端 API？

- 快速上手，不想折腾环境
- 需要最强的模型能力（GPT-4、Claude 等目前只能云端）
- 电脑配置有限，跑不动本地模型
- 开发阶段，快速验证想法

### 10.2 什么时候用本地部署？

- 数据敏感，不能上传到第三方
- 需要大量调用，不想付 API 费用
- 网络不稳定或离线环境
- 想学习模型运行原理
- 需要定制或微调模型

### 10.3 混合策略（推荐）

实际项目中，两种方式可以**结合使用**：

```
开发阶段：用云端 API 快速验证（通义/OpenAI）
    ↓ 方案确定后
敏感数据处理：用本地模型（Ollama）
    ↓
生产环境：云端 API（稳定性）或自建服务器（隐私+控制）
```

## 十二、常见问题与排错指南

### 12.1 错误速查表

| 错误/问题 | 原因 | 解决方法 |
|-----------|------|----------|
| Ollama 安装后打不开 | 安装不完整 | 重新下载安装，或检查杀毒软件是否拦截 |
| `ollama` 命令找不到 | 未添加到 PATH | 重启终端，或重新安装 |
| 模型下载速度极慢 | Ollama 服务器在海外 | 科学上网 |
| 下载到一半失败 | 网络不稳定 | 重新执行 `ollama pull`，支持断点续传 |
| `out of memory` | 显存或内存不足 | 换更小的模型版本（如从 7b 换到 1.5b） |
| 模型回复很慢（几分钟） | 没有 GPU 或显存不够 | 用 CPU 推理本身很慢，换小模型或用云端 API |
| `Connection refused`（代码调用时） | Ollama 服务未启动 | 检查任务栏是否有羊驼图标，没有则手动启动 Ollama |
| `model not found`（代码调用时） | 模型名写错或未下载 | `ollama list` 确认模型名，确保版本号匹配 |
| `ModuleNotFoundError: langchain_ollama` | 未安装包 | `pip install langchain-ollama` |
| C 盘空间不足 | 模型文件默认下载到 C 盘 | 修改 Model Location 到其他盘 |
| 安装时没提示选路径 | Ollama 不支持自定义安装路径 | 只能安装后修改模型的存储路径 |
| Jupyter 中 `\n` 不换行 | 直接输出不会解析转义字符 | 使用 `print(response.content)` |

### 12.2 排查流程

```
代码调用本地模型报错？
    ↓
第一步：检查 Ollama 服务是否在运行
    任务栏有羊驼图标？ → 没有 → 启动 Ollama 应用
    ↓ 有
第二步：检查模型是否已下载
    ollama list → 能看到你用的模型吗？
    ↓ 能
第三步：检查模型名是否匹配
    代码中的 model="deepseek-r1:1.5b"
    是否和 ollama list 中显示的名称完全一致？
    （包括冒号和版本号！）
    ↓ 一致
第四步：检查 langchain-ollama 是否安装
    pip list | grep ollama
    ↓ 安装了
第五步：检查显存是否足够
    任务管理器 → GPU → 专用 GPU 内存
    当前运行的模型是否超过了显存？
    ↓ 没超过
可能是其他问题，查看具体错误信息再排查
```

### 12.3 显存不足的典型表现

```
现象：
├── 模型加载后没有任何报错
├── 但提问后很久很久没有回复
├── 或者回复了几个字就卡住
├── 电脑变得非常卡
└── 最终可能程序崩溃

→ 大概率是显存不够了，换小模型！
```

```
│  2️⃣  Ollama 的作用                                         │
│      ├── 一键下载管理开源模型                                │
│      ├── 自动加载模型提供服务                                │
│      └── 提供本地 API 供 LangChain 调用                      │
│                                                            │
│  3️⃣  模型选择的三要素                                       │
│      ├── 参数规模（1.5b / 7b / 14b...）= 能力               │
│      ├── 文件大小（1.1GB / 4.7GB / 9GB...）= 硬盘占用       │
│      └── 显存需求（2-4GB / 10-12GB / 20-24GB...）= 能跑否   │
│                                                            │
│  4️⃣  显存估算法则                                          │
│      1.5b→2-4GB | 7b→10-12GB | 8b→12-16GB | 14b→20-24GB │
│                                                            │
│  5️⃣  用 LangChain 调用本地模型                              │
│      pip install langchain-ollama                          │
│      from langchain_ollama import ChatOllama               │
│      llm = ChatOllama(model="模型名")                       │
│      response = llm.invoke("问题")                          │
│      print(response.content)                               │
│                                                            │
│  6️⃣  不同模型的返回类型差异                                  │
│      通义 → str                                             │
│      智谱AI → AIMessage                                     │
│      Ollama → AIMessage                                     │
│      最佳实践：统一用 .content + print()                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 14.2 一句话总结

> 安装 Ollama → 根据自己的显存选模型 → `ollama run` 下载运行 → 终端测试通过 → `pip install langchain-ollama` → 代码调用 `ChatOllama` → 本地大模型应用搭建完成。全程不需要 API Key，不需要网络，不需要花钱。
