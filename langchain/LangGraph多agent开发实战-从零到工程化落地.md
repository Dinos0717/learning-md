## 一、课程背景——为什么还要从零学 Agent 开发

### 1.1 当前 Agent 开发的两种路径

```
现在构建智能体的两种方式：

方式一：用现成平台（Coze / 扣子 / Dify 等）
┌─────────────────────────────────────────────┐
│  优点：                                       │
│  • 拖拽式操作，上手极快                         │
│  • 写好提示词就能用                             │
│  • 内置插件市场，开箱即用                        │
│  • 不需要写代码                                │
│                                             │
│  局限：                                       │
│  • 受限于平台提供的能力边界                      │
│  • 底层逻辑是黑盒，出了问题无法深入排查           │
│  • 定制化能力有限                              │
│  • 数据安全和隐私不可控（数据要上传平台）          │
│  • 无法与企业现有系统深度集成                    │
└─────────────────────────────────────────────┘

方式二：从零编码构建（LangGraph + LangChain）
┌─────────────────────────────────────────────┐
│  优点：                                       │
│  • 完全掌控底层逻辑                            │
│  • 可以自由扩展和定制                          │
│  • 数据完全本地化，安全可控                     │
│  • 能与任何外部系统集成                         │
│  • 理解 Agent 的完整运行原理                    │
│                                             │
│  挑战：                                       │
│  • 需要编码能力                                │
│  • 架构设计需要经验                            │
│  • 学习曲线较陡                                │
│                                             │
│  → 本系列课程帮你跨越这些挑战！                  │
└─────────────────────────────────────────────┘
```

### 1.2 为什么选 LangGraph

| 评估维度 | LangGraph | 其他框架 |
|----------|:---------:|:-------:|
| **提出时间** | 2023年，起步早 | 多数更晚 |
| **生态基础** | 基于 LangChain，生态资源最广 | 各自独立生态 |
| **社区规模** | 最大，文档和案例丰富 | 较小 |
| **生产验证** | 多家知名公司已在生产环境使用 | 部分尚在早期 |
| **灵活性** | 底层编排框架，自由度最高 | 多数偏高层封装 |
| **多 Agent 支持** | 原生支持 Subgraph + 四种协作模式 | 部分支持或支持有限 |

### 1.3 本系列课程的定位

## 三、六大核心模块详解

### 3.1 模块一：大模型驱动模块（LLM Driver）

> Agent 的"大脑"——负责推理、规划、决策。

```
┌─────────────────────────────────────────────────────────────┐
│  LLM 驱动模块                                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  职责：                                                       │
│  • 封装大模型调用（ChatOpenAI / ChatTongyi / ChatZhipuAI）     │
│  • 模型参数管理（temperature, max_tokens, top_p）             │
│  • 工具绑定（bind_tools）                                    │
│  • 结构化输出（with_structured_output）                       │
│  • API Key 管理和切换                                        │
│                                                             │
│  依赖：几乎不依赖其他模块                                       │
│  → 只要有 API Key 和网络就能独立测试                            │
│                                                             │
│  测试文件：test_llm.py                                       │
│  → 验证模型能否正常调用，返回格式是否正确                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```python
# 模块示意：llm_driver.py
from langchain_openai import ChatOpenAI
from .config import MODEL_NAME, MODEL_BASE_URL, MODEL_API_KEY

def create_llm(tools=None, structured_output=None):
    """创建 LLM 实例的工厂函数"""
    llm = ChatOpenAI(
        model=MODEL_NAME,
        base_url=MODEL_BASE_URL,
        api_key=MODEL_API_KEY,
        temperature=0.7,
    )
    if tools:
        llm = llm.bind_tools(tools)
    if structured_output:
        llm = llm.with_structured_output(structured_output)
    return llm
```

### 3.4 模块四：Session 管理模块

> 多用户的"隔离墙"——保证不同用户的对话互不干扰。

```
┌─────────────────────────────────────────────────────────────┐
│  Session 管理模块                                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  核心问题：如何防止多用户记忆交叉混杂？                          │
│                                                             │
│  类比：                                                       │
│  像豆包/元宝的"新建对话"功能——                                  │
│  每个对话窗口是独立的，不会串台                                │
│                                                             │
│  实现机制：                                                   │
│  ┌──────────────────────────────────────────────┐           │
│  │  用户 A ──→ thread_id = "user-a-session-1"   │           │
│  │  用户 B ──→ thread_id = "user-b-session-1"   │           │
│  │  用户 C ──→ thread_id = "user-c-session-1"   │           │
│  │                                              │           │
│  │  每个 thread_id 对应独立的：                    │           │
│  │  • Checkpoint（状态快照）                      │           │
│  │  • 消息历史                                   │           │
│  │  • 长期记忆                                   │           │
│  └──────────────────────────────────────────────┘           │
│                                                             │
│  Session 生命周期管理：                                        │
│  • 创建：首次对话时自动分配 thread_id                           │
│  • 活跃：对话进行中，状态持续更新                               │
│  • 过期：长时间不活跃自动清理（可选）                            │
│  • 归档：重要会话可标记保留                                    │
│                                                             │
│  测试文件：test_session.py                                    │
│  → 两个不同 thread_id 的对话互不影响                            │
│                                                             │

            │   • 结果汇总          │

               ↓       ↓       ↓
        ┌─────────┐┌─────────┐┌─────────┐
        │Agent A  ││Agent B  ││Agent C  │
        │(Subgraph)│(Subgraph)│(Subgraph)│
        │         ││         ││         │
        │独立提示词││独立提示词││独立提示词│
        │独立工具集││独立工具集││独立工具集│
        │独立State ││独立State ││独立State │
        └────┬────┘└────┬────┘└────┬────┘
             │          │          │
             └──────────┼──────────┘
                        ↓
            ┌─────────────────────┐
            │   主 Agent（汇总）    │
            │   • 整合多份报告      │
            │   • 输出最终结论      │
            └──────────┬──────────┘
                       ↓
                    用户
```

### 4.2 与单 Agent 的本质区别

| 维度 | 单 Agent | 多 Agent（本项目） |
|------|----------|-------------------|
| **提示词** | 一份超长的万能提示词 | 每个 Agent 一份简短精准的提示词 |
| **工具** | 所有工具绑在一个 Agent | 每个 Agent 绑定自己需要的工具 |
| **记忆** | 共用一份记忆 | 每个 Agent 独立记忆 + 共享上下文 |
| **调度** | 不需要（就一个） | 主 Agent 负责分派和汇总 |
| **测试** | 测一个就行 | 每个 Agent 独立测，再测协作 |
| **扩展** | 加功能 = 改提示词 | 加功能 = 加一个 Agent |

### 5.2 测试顺序——从下往上

```
测试顺序 = 依赖从少到多

第一层（依赖最少，最先测试）：
    ├── test_llm_driver.py      → 大模型能否正常调用？
    └── test_data_structures.py → 数据结构定义是否正确？

第二层（依赖第一层）：
    ├── test_memory.py          → 短期/长期记忆是否正常？
    └── test_tools.py           → 工具函数能否正常执行？

第三层（依赖第二层）：
    ├── test_session.py         → 多用户隔离是否生效？
    └── test_single_agent.py    → 单个 Agent 能否完成任务？

第四层（依赖第三层）：
    ├── test_orchestration.py   → 多 Agent 协作是否正常？
    └── test_api.py             → HTTP 接口是否正常？

第五层（依赖所有下层）：
    ├── test_e2e.py             → 完整用户场景端到端测试
    └── test_production.py      → 生产环境部署验证
```

### 5.3 为什么用测试驱动

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  测试驱动开发的核心价值：                                       │
│                                                             │
│  1️⃣  快速验证                                                  │
│      写了一块代码 → 跑测试 → 立即知道对不对                      │
│      不用启动整个系统才能验证一个小改动                          │
│                                                             │
│  2️⃣  防止回退                                                  │
│      改模块 A 的代码 → 跑一遍所有测试                           │
│      → 发现改 A 把模块 C 搞坏了 → 立即修复                      │
│                                                             │
│  3️⃣  理解代码的捷径                                            │
│      拿到一个陌生的代码仓库 → 看 test_ 文件！                    │
│      测试文件告诉你：每个模块是怎么被调用的、期望的输出是什么       │
│      → 比读注释和文档更快理解代码                               │
│                                                             │
│  4️⃣  模块边界清晰                                              │
│      写测试时自然要思考：这个模块的输入是什么？输出是什么？         │
│      → 模块的 API 自然就定义清楚了                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 测试文件示例

```python
# ── test_llm_driver.py ──
"""测试大模型驱动模块——最底层的测试"""

from src.llm_driver import create_llm

def test_basic_invoke():
    """测试基本调用"""
    llm = create_llm()
    response = llm.invoke("你好，请用一句话介绍你自己")
    assert response.content is not None
    assert len(response.content) > 0
    print(f"✅ 基本调用测试通过：{response.content[:50]}...")

def test_streaming():
    """测试流式调用"""
    llm = create_llm()
    chunks = []
    for chunk in llm.stream("数到5"):
        chunks.append(chunk)
    assert len(chunks) > 0
    print(f"✅ 流式调用测试通过：共 {len(chunks)} 个 chunk")

def test_structured_output():
    """测试结构化输出"""
    from pydantic import BaseModel
    class TestOutput(BaseModel):
        answer: str
        confidence: float

    llm = create_llm(structured_output=TestOutput)
    result = llm.invoke("1+1等于几？")
    assert isinstance(result, TestOutput)
    assert result.answer is not None
    print(f"✅ 结构化输出测试通过：{result}")

# 运行：pytest test_llm_driver.py -v
```
