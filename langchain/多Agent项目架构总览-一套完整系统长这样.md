## 二、为什么架构分层至关重要

### 2.1 不分层的代价

```
一个不分层的 Agent 项目：

agent_all_in_one.py  (1500 行)
├── API Key 配置 (30行)
├── 数据模型定义 (100行)
├── LLM 调用封装 (200行)
├── 记忆管理 (150行)
├── Session 管理 (100行)
├── 路由逻辑 (200行)
├── Chat Agent 实现 (300行)
├── Meeting Agent 实现 (200行)
├── HTTP API (150行)
└── 入口函数 (70行)

问题：
😰 改 LLM 配置 → 翻到文件最开头
😰 加一个新 Agent → 不知道代码该插在哪里
😰 测试记忆模块 → 要启动整个系统
😰 排查一个 bug → 在 1500 行里大海捞针
😰 两个人协作 → 改同一个文件，天天冲突
```

### 2.2 分层的核心价值

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  架构分层的核心目的：                                         │
│                                                             │
│  1️⃣  可测试性                                                 │
│      每层独立测试 → 不用启动整个系统就能验证单个模块            │
│      → 依赖最少的先测，依赖多的后测                           │
│                                                             │
│  2️⃣  可维护性                                                 │
│      改 LLM 配置 → 只改 core/ 目录                           │
│      加新 Agent → 只改 agent/ 目录                           │
│      → 修改影响范围可控                                      │
│                                                             │
│  3️⃣  问题定位                                                 │
│      LLM 调用报错 → 看 core/ 层                              │
│      记忆丢失了 → 看 memory/ 层                               │
│      Agent 路由错了 → 看 agent/ 层                           │
│      → 精准定位，不用大海捞针                                 │
│                                                             │
│  4️⃣  团队协作                                                 │
│      张三负责 core/，李四负责 agent/，王五负责 memory/          │
│      → 各改各的目录，互不冲突                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 当前架构师的核心竞争力

> 为什么"会设计架构"仍然是工程师的核心竞争力？

```
现在：
    编程 Agent（如 Cursor、Copilot）可以写代码，
    但它们的架构设计要么过于复杂、要么过于简单，
    不完全符合你的项目需求。

    → 有经验的工程师来把控架构的层次和结构

未来：
    更好的编程 Agent 也许能做好架构设计。
    但今天——架构设计仍然是人的核心竞争力。
```

### 3.2 层间依赖关系

```
依赖方向：上层依赖下层，下层不依赖上层

  ↑          ↑              ↑                  ↑             ↑
  │          │              │                  │             │
CLI+HTTP   路由+子Agent   记忆+会话           Pydantic     LLM调用
                        管理+清理            模型定义      API配置


测试顺序（从下往上，依赖最少的最先测）：

1️⃣  核心驱动层  → test_core.py      只用 API Key，不依赖任何其他模块
2️⃣  数据结构层  → test_schemas.py   只用 Pydantic，不依赖其他模块
3️⃣  记忆层      → test_memory.py    依赖核心驱动层
4️⃣  Session层   → test_session.py   依赖记忆层
5️⃣  Agent层     → test_agent.py     依赖上面所有层
6️⃣  入口层      → test_api.py       依赖整个系统
```

## 四、路由 Agent——智能调度中枢

### 4.1 为什么需要一个路由 Agent

> 用户不需要知道系统有几个 Agent。他只需要正常输入，系统自动判断该由谁处理。

```
没有路由 Agent：                        有路由 Agent：

用户要先想清楚：                        用户正常输入即可：
"我这是聊天还是会议？                    "帮我整理今天开会的内容"
 我该选哪个？"                               ↓
    ↓                                 Router Agent 自动判断：
手动选择                               "这是会议纪要需求"
    ↓                                     ↓
┌──────────┐ ┌──────────┐             Meeting Agent 处理
│ 点 Chat   │ │点 Meeting │                 ↓
│ 按钮     │ │ 按钮      │             返回结构化纪要
└────┬─────┘ └────┬─────┘
     ↓            ↓
   Chat        Meeting

😰 用户要懂系统内部结构              😊 用户完全无感
```

### 4.2 路由 Agent 的工作流程

```
用户输入："帮我整理今天的会议记录，讨论了下季度产品规划和预算"
    ↓
┌─────────────────────────────────────────────────────────────┐
│                      路由 Agent (Router)                      │
│                                                             │
│  Step 1: 接收用户输入                                         │
│  Step 2: 调用 LLM 做意图识别                                  │
│  Step 3: 结构化输出 Intent 字段                                │
│                                                             │
│  提示词（简化）：                                              │
│  "分析以下用户输入，判断意图：                                  │
│   - 如果是日常对话、问答 → chat                                │
│   - 如果是会议记录、需要整理 → meeting                          │
│   - 如果无法判断 → unknown"                                   │
│                                                             │
│  输出结构：{ "intent": "meeting", "confidence": 0.92 }        │
│                                                             │
└─────────────────────────┬───────────────────────────────────┘
                          ↓
              intent = "meeting"
                          ↓
              ┌──────────────────────┐
              │   Meeting Agent      │
              │   处理会议纪要        │
              └──────────────────────┘
```

### 4.3 路由 Agent 的结构化输出

```python
# ── schemas/router.py ──
from pydantic import BaseModel, Field

class RouterDecision(BaseModel):
    """
    路由 Agent 的决策输出。
    不是自由文本，而是精确的结构化字段。
    """
    intent: str = Field(
        description="意图分类结果。可选值：chat / meeting / unknown"
    )
    confidence: float = Field(
        description="置信度 0-1，如 0.92 表示 92% 确定"
    )
    reason: str = Field(
        description="判断依据的简要说明"
    )

# ── 在路由 Agent 中使用 ──
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="...", ...)
router_llm = llm.with_structured_output(RouterDecision)

# 路由决策
decision = router_llm.invoke("帮我整理今天开会的内容")
# → RouterDecision(intent="meeting", confidence=0.95, reason="...")

# 根据 intent 字段决定路由方向
if decision.intent == "chat":
    result = chat_agent.process(user_input)
elif decision.intent == "meeting":
    result = meeting_agent.process(user_input)
else:
    result = "抱歉，我不太理解你的需求，能再说详细一点吗？"
```

### 4.4 路由 Agent 的关键设计决策

| 设计点 | 决策 | 理由 |
|--------|------|------|
| **有没有记忆？** | ❌ 不需要 | 路由只需要判断"这一次"的意图，历史不重要 |
| **用什么判断？** | LLM 语义理解 | 不用硬编码关键字——LLM 能理解"帮我理一下刚才的讨论"就是会议纪要 |
| **输出格式** | 结构化（Intent字段） | 下游代码需要精确的字段来做 if-else 路由 |
| **兜底策略** | unknown 分支 | LLM 无法判断时返回 unknown，提示用户 |

## 五、记忆策略——谁需要记忆，谁不需要

### 5.1 三个 Agent 的记忆需求分析

```
┌──────────────────────────────────────────────────────────────┐
│             三个 Agent 的记忆需求                               │
├────────────┬──────────┬─────────────────────────────────────┤
│   Agent    │ 需要记忆？ │              理由                    │
├────────────┼──────────┼─────────────────────────────────────┤
│ Router     │    ❌    │ 只需要判断"这一次"的意图              │
│ Agent      │          │ 上一次路由到哪里完全不重要             │
│            │          │ 直接调 LLM 就行                       │
├────────────┼──────────┼─────────────────────────────────────┤
│ Chat       │    ✅    │ 聊天需要上下文连贯性                   │
│ Agent      │          │ "我喜欢篮球"→ 下次问"我喜欢什么"      │
│            │          │ 如果记不住 → 前言不搭后语               │
├────────────┼──────────┼─────────────────────────────────────┤
│ Meeting    │    ❌    │ 上层已经把完整文本一次性传过来了         │
│ Agent      │ （可配） │ 每次处理是独立的                       │
│            │          │ 如果上层是一句句传的 → 需要记忆         │
│            │          │ 本项目中：不需要                       │
└────────────┴──────────┴─────────────────────────────────────┘
```

### 5.2 为什么 Chat Agent 需要记忆

```
没有记忆的 Chat Agent：                    有记忆的 Chat Agent：

用户："我叫张三"                           用户："我叫张三"
Agent："好的张三"                          Agent："好的张三"

用户："我做什么工作？"                      用户："我做什么工作？"
Agent："不知道，你没说过"                   Agent："你还没告诉我呢"
😰                                       （至少不瞎编）

用户："帮我推荐一个投资策略"                 用户："帮我推荐一个投资策略"
Agent："好的，推荐...(通用推荐)"             Agent："基于你是金融分析师，
😰 完全没用到"金融分析"这个信息                      推荐以下策略..."
                                            😊 用到了历史信息做个性化
```

### 5.3 为什么 Meeting Agent 不需要记忆（本项目）

```
场景A（本项目的设计）：上层一次性传来完整文本

客户端收集完整个会议记录
    ↓
一次性发送："以下是今天会议的全部内容：王经理说...张工说..."
    ↓
Meeting Agent 处理 → 返回纪要 → 结束
    ↓
下一次又是全新的会议记录 → 与上次无关
✅ 不需要记忆


场景B（如果交互方式不同）：上层逐句传来

客户端："王经理：Q2业绩增长了15%"
    ↓
Meeting Agent："收到，继续"
客户端："张工：新功能下周三上线"
    ↓
Meeting Agent："收到，继续"
...
客户端："没了，帮我整理纪要"
    ↓
Meeting Agent 需要记住前面收到的所有零散句子
⚠️ 这种情况下需要记忆！

但本项目选择了更简单的场景A
```

## 六、Session 生命周期管理

### 6.1 为什么不能无限存储

```
假设不做清理：

100 个用户 × 每人 10 个会话窗口 = 1000 个 Session
每个 Session 存储数千条消息
    ↓
内存/数据库无限增长
    ↓
系统越来越慢 → 最终崩溃 💥
```

### 6.2 本项目的清理策略

```
┌─────────────────────────────────────────────────────────────┐
│              Session 生命周期管理                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  策略：后台线程定期清理不活跃的 Session                         │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  清理线程（每小时执行一次）：                          │   │
│  │                                                     │   │
│  │  for each session:                                  │   │
│  │      if session 的最后活跃时间 > 24小时前:            │   │
│  │          清理该 session 的记忆上下文                  │   │
│  │                                                     │   │
│  │  判断依据：最后活跃时间（last_active_time）            │   │
│  │  清理内容：记忆数据（对话历史 + 摘要）                 │   │
│  │  清理周期：每小时检查一次                             │   │
│  │  超时阈值：24 小时不活跃                               │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  示例时间线：                                                 │
│                                                             │
│  10:00  用户对话 → 活跃，记录 last_active_time=10:00         │
│  11:00  后台检查 → 距上次活跃 1 小时 → 不清理                 │
│  22:00  后台检查 → 距上次活跃 12 小时 → 不清理                │
│  次日    后台检查 → 距上次活跃 25 小时 → 清理！               │
│  11:00                                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```python
# ── Session 清理示意 ──
import threading
import time
from datetime import datetime, timedelta

class SessionCleaner:
    """后台 Session 清理线程"""

    def __init__(self, sessions: dict, ttl_hours: int = 24):
        self.sessions = sessions
        self.ttl = timedelta(hours=ttl_hours)

    def start(self):
        """启动清理线程"""
        thread = threading.Thread(target=self._clean_loop, daemon=True)
        thread.start()

    def _clean_loop(self):
        """每小时检查一次，清理超时的 Session"""
        while True:
            time.sleep(3600)  # 1 小时
            now = datetime.now()
            expired = []
            for thread_id, session in self.sessions.items():
                if now - session.last_active > self.ttl:
                    expired.append(thread_id)

            for tid in expired:
                del self.sessions[tid]
                print(f"🧹 已清理不活跃 Session: {tid}")
```

## 十、架构设计的核心决策

### 10.1 关键设计决策汇总

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **分层数量** | 5 层（Core/Schemas/Memory/Agent/Entry） | 层数适中，既不过度设计也不过于扁平 |
| **路由方式** | LLM 语义路由（非关键字匹配） | 适应各种自然语言表达，不依赖固定模式 |
| **Router 记忆** | 无 | 只判断当次意图，历史不重要 |
| **Chat 记忆** | 有（近期完整+远期摘要） | 聊天需要上下文连贯 |
| **Meeting 记忆** | 无 | 上层一次传完整文本，每次处理独立 |
| **记忆绑定** | 与 Session 一对一绑定 | 一个窗口一个记忆，数据隔离 |
| **Session 清理** | 后台线程 + 24h 超时 | 防止内存无限增长 |
| **结构化输出** | Pydantic + with_structured_output | 下游代码需要精确字段，不是自然语言 |
| **多模型支持** | 统一的 create_llm() 工厂 | 切换模型只改配置，不改业务代码 |
| **双入口** | CLI + HTTP API | 开发用 CLI，生产用 API |

### 10.2 架构的扩展性

```
怎么加第三个 Agent？只需三步：

Step 1: 创建 agent/third_agent.py
    class ThirdAgent:
        def process(self, input_data): ...

Step 2: 在 schemas/router.py 中加一个 intent 值
    intent: Literal["chat", "meeting", "third_agent", "unknown"]

Step 3: 在 Router Agent 中加一条路由
    if decision.intent == "third_agent":
        result = third_agent.process(user_input)

✅ 完成！不需要改其他任何模块！
```

## 十一、常见问题与排错指南

### 11.1 架构相关 FAQ

| 问题 | 解答 |
|------|------|
| **为什么要分 5 层，3 层不行吗？** | 可以，但 memory 和 schemas 合到 agent 里会使 agent 文件过大。分层粒度取决于项目复杂度 |
| **为什么 Router Agent 不用记忆？** | 路由只需要判断"这一次"的意图。加记忆反而增加复杂度和成本，没有收益 |
| **Meeting Agent 真的不需要记忆吗？** | 取决于你的交互方式。如果客户端一次性传完整文本→不需要。如果逐句传→需要 |
| **24h 清理会不会太短？** | 可根据业务调整。聊天类应用建议 24h，业务系统可设更长 |
| **能支持多少个 Agent？** | 没有硬性限制。加 Agent 只需注册+加路由，不影响已有 Agent |

### 11.2 常见架构错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 所有 Agent 都加记忆 | Router 和 Meeting Agent 浪费资源 | 分析每个 Agent 是否真的需要记忆 |
| Session 不清理 | 内存无限增长→系统崩溃 | 启动后台清理线程 |
| 路由用关键字匹配 | "理一下刚才说的"无法识别为会议 | 用 LLM 做语义路由 |
| Agent 之间直接互相调用 | 循环依赖→代码耦合→难维护 | 统一通过 Router 分发 |
| schemas 定义在 agent 文件里 | 其他模块引用时产生循环导入 | 独立 schemas/ 目录 |

### 11.3 调试技巧

```
按层级排查问题：

1. LLM 调用失败 → 先看 core/config.py 的 API Key 是否正确
2. 结构化输出格式错误 → 看 schemas/ 中的 Pydantic 模型定义
3. 记忆丢失 → 看 memory/ 的 Session 是否正确创建和绑定
4. 路由错误（该走 Chat 走了 Meeting）→ 看 Router 的 Intent 输出
5. 某个 Agent 结果异常 → 单独测该 Agent，看其 prompt 和工具配置
```

- 每个模块的数据输入和输出
- 结构化输出的字段变化

**验收标准**：流程图完整，包含了 Router → Meeting Agent 的全链路。

### 练习四：扩展一个 Agent（⭐⭐⭐⭐）

**目标**：实践架构的扩展性

**要求**：为系统增加一个"翻译 Agent"。

**需完成**：

## 十三、本节小结

### 13.1 核心知识图谱

```
┌──────────────────────────────────────────────────────────────┐
│      多 Agent 项目架构总览——核心知识体系                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣  分层架构设计                                              │
│      ├── 5层：Entry → Agent → Memory/Session → Schemas → Core│
│      ├── 目的：可测试、可维护、可定位、可协作                    │
│      └── 原则：上层依赖下层，下层独立                           │
│                                                              │
│  2️⃣  路由 Agent                                               │
│      ├── 职责：意图识别 → 分发到子 Agent                        │
│      ├── 输出：结构化 RouterDecision（intent/confidence）      │
│      ├── 不需要记忆：只判断当次意图                             │
│      └── 不靠关键字：LLM 语义理解，识别各种自然表达             │
│                                                              │
│  3️⃣  记忆策略                                                 │
│      ├── Router Agent：❌（只判断一次）                         │
│      ├── Chat Agent：✅（上下文连贯性）                         │
│      ├── Meeting Agent：❌（一次性文本处理）                    │
│      └── 记忆与 Session 一对一绑定                             │
│                                                              │
│  4️⃣  Session 生命周期                                          │
│      ├── 每个 Session 独立记忆                                 │
│      ├── 后台线程每小时检查                                     │
│      └── 24h 不活跃 → 自动清理                                  │
│                                                              │
│  5️⃣  项目结构                                                 │
│      ├── src/：core/ + schemas/ + memory/ + agent/ + 入口    │
│      ├── tests/：与 src 一一对应                               │
│      └── 从下往上学习：Core → Schemas → Memory → Agent → Entry│
│                                                              │
│  6️⃣  两个执行流程                                              │
│      ├── Chat：路由→ChatAgent→思考→工具→观察→回复→存记忆       │
│      └── Meeting：路由→MeetingAgent→LLM分析→结构化输出→MD格式  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 13.2 一句话总结

> **本项目的架构像一栋五层大楼**——地基是 LLM 调用（Core）和数据契约（Schemas），中间层是记忆和会话管理（Memory/Session），顶楼是业务智能体（Agent），最外层是接入方式（CLI/HTTP）。每一层独立可测，层与层之间通过结构化数据通信，Router Agent 像大楼的前台，自动引导用户到对应的智能体——整个设计既保证了当前的功能完整性，也为未来的扩展留足了空间。
