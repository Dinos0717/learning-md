# 教学文档输出框架

> **本文档定义最终教学文档的严格输出格式。**
>
> 所有 Stage 2 撰写 Agent、Stage 3 审查 Agent、Stage 5 终审 Agent 必须以此文件为唯一格式标准。
>
> 本文档与 SKILL.md（流程编排）配合使用。

---

## 一、文档整体结构

```
[文档头部]
    ├── 标题行
    ├── 元数据行（日期、级别、录音时长、文档行数）
    └── 目录（锚点链接）

[正文：按章节排列]
    ├── 一、[大章标题]
    │   ├── 1.1 [小节标题]
    │   │   ├── 内容段落
    │   │   ├── 代码示例（完整可运行）
    │   │   ├── 排错实录（如适用）
    │   │   └── 对比表/决策树（如适用）
    │   └── 1.2 [小节标题]
    ├── 二、[大章标题]
    └── ...

[文档尾部]
    ├── 课程小结
    │   ├── 知识图谱（ASCII 图）
    │   ├── 一句话总结
    │   └── 系列定位
    └── 元数据行
```

## 二、文档头部格式

```markdown
# [课程标题]

> 📅 生成日期: YYYY-MM-DD | 🎯 级别: X 级 | ⏱️ 录音时长: ~XX 分钟 | 📏 文档规模: X,XXX+ 行
> 🏷️ 标签: `标签1` `标签2` `标签3`

---

## 📑 目录

- [一、章节标题](#一章节标题)
  - [1.1 小节标题](#11-小节标题)
  - [1.2 小节标题](#12-小节标题)
- [二、章节标题](#二章节标题)
  - ...

---
```

## 三、正文内容格式

### 3.1 章节标题层级

- 一级章节：`## 一、[章节标题]`
- 二级小节：`### 1.1 [小节标题]`
- 三级细节：`#### 1.1.1 [细节标题]`（仅在必要时使用）

### 3.2 每个核心概念的三层递进结构

> **必须**按「直观理解 → 详细解释 → 代码示例」三层展开。

```markdown
### 1.1 [概念名称]

#### 🧠 直观理解
[一句话说清楚这是什么，解决什么问题。用生活类比。]

#### 📖 详细解释
[原理、推导链、关键设计决策、为什么这样设计。每步带"为什么"。]

#### 💻 代码示例
[完整可运行代码。含 imports、中文注释。可附加行级解析表。]
```

### 3.3 标准 Callout 格式

```markdown
> ⚠️ **注意事项**：[老师强调"注意"/"一定要"的内容]

> 💡 **核心洞见**：[老师的关键洞察]

> 🔗 **关联知识**：[与前文/其他知识点的关联]

> 🐞 **常见坑**：[老师踩过的坑，含错误现象和原因]

> ❓ **常见疑问**：[学员常问的问题和回答]
```

### 3.4 表格格式

所有表格必须对齐、有表头分隔线：

```markdown
| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 方案A | ... | ... | ... |
| 方案B | ... | ... | ... |
```

### 3.5 ASCII 图格式

用于还原白板讲解、流程图、架构图：

```
┌──────────┐      ┌──────────┐
│  组件 A   │─────▶│  组件 B   │
└──────────┘      └──────────┘
       │
       ▼
┌──────────┐
│  组件 C   │
└──────────┘
```

### 3.6 代码块格式

```markdown
```python
# 文件名: example.py
# 功能: [一句话描述]

import xxx

# [步骤1说明]
def func():
    pass

# [步骤2说明]
# ...
```
```

复杂代码可附加**行级解析表**：

```markdown
| 行号 | 代码 | 解释 |
|---|---|---|
| 3 | `import xxx` | 导入 xxx 库，用于... |
| 6 | `def func():` | 定义核心函数，参数含义是... |
```

## 四、Rich Content Preservation Rules（富内容保留规则）

> **这些规则是子 Agent 撰写的硬性约束。审查 Agent 逐条核验。**

| Rule | 内容类型 | 触发条件 | 处理要求 |
|---|---|---|---|
| **Rule 1** | 老师踩坑过程 | 录音中出现"写错→报错→排查→修改→正确"流程 | 完整保留为**排错实录**段落：错误代码 → 报错信息 → 原因分析 → 修复代码 → 结果验证 |
| **Rule 2** | 推导链 | 录音中出现 A→B→C 的推理过程 | 每一步必须带「为什么」，**不允许**压缩成一句话结论 |
| **Rule 3** | 长类比 | 录音中 2 分钟以上的类比/比喻 | **完整展开**，不允许压缩。保留类比的起承转合 |
| **Rule 4** | Q&A 交流 | 录音中有学员提问和老师回答 | 每个问题独立展开：**问题原文** + 背景 + 详细回答 + 延伸建议 |
| **Rule 5** | 白板讲解 | 录音中有"大家看这个图"/"画一下" | 根据语言描述还原为 **ASCII 图** 或 **结构化表格** |
| **Rule 6** | 代码示例 | 录音中老师现场写代码 | 完整可运行（含 imports），中文注释，必要时附加行级解析表 |

### 排错实录格式（Rule 1）

```markdown
#### 🐛 排错实录：[问题简述]

**❌ 错误写法**
```[语言]
[有问题的代码]
```

**🚨 报错信息**
```
[完整的报错信息]
```

**🔍 原因分析**
[为什么会出错？根因是什么？]

**✅ 正确写法**
```[语言]
[修复后的代码]
```

**📝 经验总结**
[从这个坑里学到什么？如何避免？]
```

## 五、Reusable Content Patterns（可复用内容模式）

### Pattern 1: Learning Objectives Box（学习目标）

```markdown
> 🎯 **本节目标**
>
> 学完本节后，你将能够：
> - [ ] 目标 1（可验证的动作）
> - [ ] 目标 2
> - [ ] 目标 3
```

### Pattern 2: Before/After Comparison（前后对比）

```markdown
| 维度 | ❌ 不用此方案 | ✅ 使用此方案 |
|---|---|---|
| 代码量 | ... | ... |
| 可读性 | ... | ... |
| 性能 | ... | ... |
```

### Pattern 3: Knowledge Graph Box（知识图谱）

````markdown
### 🗺️ 知识图谱

```
            ┌─────────────────┐
            │    [核心概念]     │
            └────────┬────────┘
                     │
        ┌────────────┼────────────┐
        ▼            ▼            ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ [概念A]   │ │ [概念B]   │ │ [概念C]   │
  └──────────┘ └──────────┘ └──────────┘
```
````

### Pattern 4: Process Flow Diagram（流程时序图）

````markdown
### 🔄 执行流程

```
用户请求
    │
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Step 1   │────▶│ Step 2   │────▶│ Step 3   │
│ [说明]    │     │ [说明]    │     │ [说明]    │
└──────────┘     └──────────┘     └──────────┘
```
````

### Pattern 5: Decision Tree（决策树）

````markdown
### 🤔 如何选择？

```
遇到 [场景] 时：
    │
    ├── 条件 A 满足？
    │   ├── 是 → 使用 [方案1]
    │   └── 否 → 继续判断
    │
    ├── 条件 B 满足？
    │   ├── 是 → 使用 [方案2]
    │   └── 否 → 使用 [方案3]
```
````

### Pattern 6: Series Positioning（系列定位）

```markdown
> 📍 **系列定位**：本文是「[系列名称]」第 X 篇。
> - 上一篇：[标题] — [一句话概要]
> - 下一篇：[标题] — [一句话概要]
```

### Pattern 7: Derivation Chain（推导链）

````markdown
### 🔗 推导链：[推导主题]

```
起点: [初始状态/假设/问题]
  │
  ├── 第1步: [操作/推理]
  │   为什么？[原因]
  │
  ├── 第2步: [操作/推理]
  │   为什么？[原因]
  │
  └── 结论: [最终结果]
```
````

### Pattern 8: Cross-Reference（跨模块引用）

> 子 Agent 在撰写单个模块时，如需引用其他模块，使用占位符格式。
> Stage 4 合并时由主 Agent 替换为实际锚点链接。

```markdown
> 🔗 **前置知识**：详见 [第 X 章：章节标题]（← 合并时替换为锚点链接）

> 🔗 **延伸阅读**：本文 [第 Y 章：章节标题] 中有更详细的讨论（← 合并时替换）
```

## 六、Content Expansion Guidelines（内容展开指南）

> 录音稿是提纲，文档是完整教案。下表定义"录音稿提到 X → 文档应包含 Y"的展开标准。

| 录音稿提到 | 文档应包含 |
|---|---|
| 屏幕上展示的代码片段 | 完整可运行代码 + 行级注释 + 行级解析表 |
| 老师踩的坑 | 完整排错实录：错误→报错→原因→修复→经验 |
| "这个东西"/"那个东西" | 补齐正式术语，首次出现时标注英文 |
| 两种替代方案 | 完整对比表（优劣 + 场景 + 代码示例） |
| "注意"/"一定要" | ⚠️ callout 框 |
| 结尾测验/题目 | 题目 + 选项 + 正确答案 + 详细解析 |
| 老师的现场调试 | 完整踩坑旅程（每个调试步骤 + 推理过程） |
| 2+ 分钟类比/比喻 | 完整展开，保留起承转合 |
| 推理链 A→B→C | 每步记录"为什么"，用 Derivation Chain 模式 |
| Q&A 交流 | 每个问题完整保留（问题原文 + 背景 + 回答 + 延伸） |
| "大家看这个图" | ASCII 图 或 结构化表格还原 |
| 老师口头一笔带过但重要的概念 | 补充完整解释，不能跟着一笔带过 |
| 老师引用的外部资源/文档 | 补充链接或参考文献说明 |

## 七、模块类型差异化结构

> **子 Agent 在撰写前，先根据 `_chunks` 中的模块类型选择对应的结构模板。**
>
> 四种类型不是互斥的——一个模块可能混合多种，以主要类型为准。

### 7.1 概念讲解型（默认）

适用：介绍新概念、原理解释、设计思想。

```markdown
### X.X [概念名称]

#### 🧠 直观理解
[一句话 + 生活类比。让完全不懂的人也能建立心智模型。]

#### 📖 详细解释
[原理展开。推导链用 Derivation Chain 模式。关键设计决策带"为什么"。]

#### 🔀 对比辨析
[和相近概念的区别。用 Before/After 或对比表。]

> ❓ **常见误解**：[学员最容易搞混的地方 + 纠正]

#### 💻 代码示例
[完整可运行代码。展示概念在代码中的体现。]
```

### 7.2 代码实操型

适用：老师现场写代码、演示工具、命令行操作。

```markdown
### X.X [实操主题]

#### 🎯 目标
[本次实操要达成什么。用 Learning Objectives Box。]

#### 🏗️ 前置准备
[环境要求、依赖安装、初始状态。]

#### 💻 Step-by-Step

**Step 1: [操作名]**
```[语言]
# [这一步做什么]
[代码]
```
> [这一步的解释、为什么这样写、常见错误]

**Step 2: [操作名]**
...

#### ✅ 验证结果
[如何确认操作成功？预期输出是什么？]

#### 🐛 排错指南
[此实操中最常见的 2-3 个坑，用排错实录格式]
```

### 7.3 案例分析型

适用：老师分析实际案例、项目经验、业界事件。

```markdown
### X.X [案例名称]

#### 📋 背景
[案例发生的上下文：什么公司/项目、面临什么问题、约束条件]

#### ⚠️ 问题
[具体遇到了什么问题？为什么这是个问题？]

#### 🔍 分析过程
[老师如何分析这个问题？用了什么方法/框架？推理链。]

#### ✅ 解决方案
[最终采用了什么方案？为什么选这个而不是其他？]

#### 📊 结果与复盘
[方案实施后的结果。如果重来一次会怎么做？关键教训。]

#### 💡 迁移启示
[这个案例对我们有什么启发？类似场景如何复用这个经验？]
```

### 7.4 Q&A 答疑型

适用：录音中有学员提问和老师回答的段落。

```markdown
### X.X 答疑：[话题]

---

**🙋 问题 1**：[学员原话或问题概述]

> 📖 **背景**：[这个问题出现的上下文，为什么学员会这么问]

**💬 老师回答**：
[详细回答内容。保留老师的推理过程和举例。]

> 💡 **延伸**：[相关的知识点、进一步阅读、类似问题的处理思路]

---

**🙋 问题 2**：...
```

## 九、双角色质量检查清单

> Stage 5 终审 Agent 使用此清单逐项核验。

### 讲稿可用（Teacher-Facing）

- [ ] 目录清晰 → 5 分钟扫完知道今天讲什么
- [ ] 推导链完整 → 每一步有"为什么"，能照着讲
- [ ] 代码块完整可运行 → 能当堂粘贴运行
- [ ] 排错实录在手 → 能预判学员会踩什么坑
- [ ] 对比表在手 → 学员问"A 和 B 什么区别"能秒答

### 课件可用（Student-Facing）

- [ ] 结构清晰 → 能独立预习和复习
- [ ] 目录可跳转 → 锚点链接正确
- [ ] 图示直观 → ASCII 图和表格能看懂
- [ ] 排错指南 → 覆盖常见坑，能自查
- [ ] 课后练习 → 有明确验收标准
- [ ] 三层递进 → 每个核心概念按"直观理解→详细解释→代码示例"展开

## 十、禁止事项

| ❌ 禁止 | ✅ 正确做法 |
|---|---|
| 压缩推导链为一句话 | 每一步展开，带"为什么" |
| 保留口语填充词（"这个"、"那个"、"然后"） | 补齐正式术语 |
| 代码块缺少 imports 或语言标记 | 完整可运行代码 + 语言标记 |
| 合并多个独立主题 | 每个主题独立成章 |
| 省略老师踩坑过程 | 完整排错实录段落 |
| 压缩长类比 | 完整展开，保留起承转合 |
| 缺少章节编号 | 严格使用「一、1.1、1.1.1」编号体系 |
| 表格不对齐 | 所有表格列对齐 |
| ASCII 图边界不完整 | 检查每个框都闭合 |
| 缺少元数据行 | 文档头和尾各一行完整元数据 |
| 所有模块用同一结构 | 按模块类型选择差异化模板（见第七章） |
| 跨模块引用写死章节号 | 使用占位符格式（见 Pattern 8），由合并阶段替换 |

## 十一、子 Agent 使用本文档的方式

1. **开始撰写前**：通读第一~七章，了解输出结构和差异化模板
2. **判断模块类型**：根据 `_chunks` 中的模块类型，选择第七章对应的结构模板
3. **撰写时**：参照第三~七章的具体格式和模式
4. **自查时**：对照第九章的双角色清单和第十章禁止事项
5. **跨模块引用**：如需引用其他模块，使用 Pattern 8 的占位符格式
6. **交付前**：确认文档头尾格式（第二章 + 第八章）完整
